Start Here — What Even Is an API Gateway?

Before anything else — understand the problem it solves from scratch.

🤔 Imagine Your System Without a Gateway

You have built a microservices system. You have 8 services:

  • UserService running at http://user-service:8081
  • OrderService running at http://order-service:8082
  • ProductService running at http://product-service:8083
  • PaymentService running at http://payment-service:8084
  • InventoryService running at http://inventory-service:8085
  • NotificationService running at http://notification-service:8086
  • ReportService running at http://report-service:8087
  • SearchService running at http://search-service:8088

Now your mobile app needs to show a home screen. It needs user profile data, recent orders, and featured products. Without a gateway, the mobile app must:

  • Know the address of all 3 services
  • Make 3 separate network calls — one to each service
  • Handle auth separately for each call
  • Handle errors separately for each call
  • Every time a service moves to a new server or port, the app must be updated and redeployed

This is a maintenance nightmare. Now multiply this by 10 different client apps (iOS, Android, Web, Admin dashboard, 3rd party partners). Every one of them must know about every service. Every service change breaks every client.

Formal Definition

An API Gateway is a server that acts as the single entry point for all client requests into a microservices system. It receives all incoming requests, applies cross-cutting concerns (authentication, rate limiting, logging, routing), and forwards them to the appropriate downstream microservice.

The client only ever knows about one address — the gateway. The gateway knows about all the services. When services move, scale up, or change — only the gateway configuration changes. Clients are unaffected.

The Best Analogy — A Hotel Reception Desk

Imagine a large hotel with hundreds of rooms, a restaurant, a spa, a gym, a conference centre, and a parking garage. Guests never wander the building looking for things themselves. They go to reception.

Reception (the API Gateway):

Checks your identity (authentication — "can I see your booking?") → Checks what you are allowed to do (authorisation — "you have a standard room, not a suite") → Directs you to the right place (routing — "the restaurant is on floor 2, the spa is on floor 5") → Logs every interaction (audit trail) → Manages queue (rate limiting — "the spa is full, please wait or come back").

The guest never needs to know the internal layout of the hotel. If the restaurant moves to floor 3 tomorrow, reception updates their information — the guests still just ask reception.

What Does an API Gateway Actually Do?

A gateway is not just a proxy. It handles a whole layer of concerns so your services do not have to.

📋 The Full List of Responsibilities

Every request that enters your system passes through the gateway. Here is everything it does along the way:

CORR ID
1. Correlation ID Injection
The very first thing. If the request does not have an X-Correlation-ID header, the gateway generates a UUID and adds it. Every downstream service logs this ID. When something breaks in production, you can search all service logs for one ID and see the exact path the request took.
TLS
2. TLS Termination
The gateway decrypts incoming HTTPS traffic. Internal service-to-service calls then happen over plain HTTP — faster and no certificate management needed on every service. This is called TLS termination. The gateway is the only thing that handles SSL certificates.
AUTH
3. Authentication — Verify the Token
Checks the Authorization: Bearer <token> header. Validates the JWT signature. Checks expiry. If invalid, returns 401 immediately — the request never reaches your services. Extracts the user ID and roles from the token and adds them as headers for downstream services.
RATE
4. Rate Limiting / Throttling
Counts how many requests this user or IP has made in the current time window. If they have exceeded the limit, returns 429 Too Many Requests immediately. Protects your services from being overwhelmed by one bad actor or a traffic spike.
ROUTE
5. Routing — Send to the Right Service
Looks at the request path and maps it to the correct service. /api/orders/** goes to OrderService. /api/products/** goes to ProductService. The client sends one request to the gateway. The gateway forwards it to the right place. Client never needs to know internal service addresses.
LB
6. Load Balancing
If OrderService has 3 running instances, the gateway distributes requests across all 3. Client makes one request — gateway decides which instance handles it. If one instance is slow, fewer requests are sent to it. If one instance crashes, gateway removes it from rotation.
TRANS
7. Request / Response Transformation
Can add headers to the request before forwarding (like User-ID extracted from token). Can strip sensitive internal headers from the response before returning to the client. Can transform request formats if needed.
LOG
8. Logging and Metrics — Always Last
Records every request: timestamp, path, user, response code, response time in milliseconds. Even requests that were rejected at step 3 or 4 are logged. This gives you a complete picture of everything that entered your system.

How Routing Works — The Core Function

Routing is how the gateway decides where to send each request. Here is exactly how it works.

🤔 What Is Routing?

Routing is a set of rules that maps incoming request paths to downstream service addresses. Think of it as a phone directory — you look up "orders" and get the address of OrderService.

A routing rule has three parts:

  • Predicate (if): A condition that must be true to match this rule. Usually the URL path pattern. Can also be based on HTTP method, headers, query parameters, or time of day.
  • URI (go to): The destination — where to forward the request if the predicate matches.
  • Filters (do this along the way): Transformations to apply to the request or response for this specific route.
Concrete Example — What Routing Looks Like in Practice

Your gateway receives: GET https://api.myapp.com/api/orders/123

The gateway checks its routing rules in order:

  1. Does path match /api/users/**? → No, skip
  2. Does path match /api/orders/**? → Yes! Match found.
  3. Strip the /api prefix from the path → path becomes /orders/123
  4. Forward to http://order-service:8082/orders/123

The client sent to api.myapp.com/api/orders/123. OrderService received /orders/123. The client never knew the internal address. OrderService never knew about the gateway prefix.

# Spring Cloud Gateway — application.yml routing configuration spring: cloud: gateway: routes: # Route 1: anything starting with /api/orders goes to order-service - id: order-service-route uri: lb://ORDER-SERVICE # lb:// means use load balancer predicates: - Path=/api/orders/** # match condition filters: - StripPrefix=1 # remove /api from path before forwarding # Route 2: product service - id: product-service-route uri: lb://PRODUCT-SERVICE predicates: - Path=/api/products/** filters: - StripPrefix=1 # Route 3: public auth endpoints — no auth check needed - id: auth-service-route uri: lb://AUTH-SERVICE predicates: - Path=/api/auth/** filters: - StripPrefix=1
📌 What Does lb:// Mean?

lb://ORDER-SERVICE means "use the load balancer to find an instance of a service registered under the name ORDER-SERVICE". The gateway integrates with a Service Registry (like Eureka or Kubernetes DNS). Instead of hardcoding http://192.168.1.5:8082, you use the service name. The registry tells the gateway the current addresses of all running instances, and the gateway load-balances across them.

If you scale OrderService from 2 to 5 pods, the registry automatically updates. The gateway starts using all 5 pods without any configuration change.

Rate Limiting and Throttling — Protecting Your System

What throttling is, why you need it, and how it works under the hood.

🤔 What Is Rate Limiting and Why Do You Need It?

Imagine your API is free to use. A competitor writes a script that sends 50,000 requests per minute to your API. Without rate limiting:

  • Your services are overwhelmed — legitimate users cannot get responses
  • Your database gets hammered with queries — it slows down for everyone
  • Your infrastructure costs spike — you are paying for their abuse
  • Your system potentially crashes — a denial of service attack

Rate limiting says: "Each user can make a maximum of X requests per Y time window. After that, we return 429 Too Many Requests and they must wait."

Throttling is a related concept — instead of rejecting excess requests, you slow them down. The terms are often used interchangeably in practice.

Everyday Analogy — A Nightclub Bouncer

A nightclub has a maximum capacity. A bouncer stands at the door counting how many people are inside. When the club is full, the bouncer stops letting people in — they must wait outside. Regular guests get a VIP pass (higher limit). Unknown people (unregistered IPs) get a stricter limit.

The API gateway rate limiter is the bouncer. The time window is the "per hour" capacity. The Redis counter is the clicker the bouncer uses to count.

⚙️ How Rate Limiting Works Internally

When a request arrives, the gateway does this in order:

  1. Identify who is making the request: Extract the user ID from the JWT token (preferred), or use the IP address if not authenticated.
  2. Build a Redis key: Something like rate_limit:userId:123:minute:2024-01-15-14-35. The key includes the time window so it automatically resets every minute.
  3. Increment the counter in Redis: This is an atomic operation — INCR command in Redis. Returns the new count.
  4. Set TTL on the key: The Redis key expires after 60 seconds so counters automatically reset each minute. No manual cleanup needed.
  5. Check the count: If count is under the limit — allow the request. If count is over the limit — return 429 Too Many Requests with a Retry-After: 45 header telling the client how many seconds to wait.
📊 Why Redis? Why Not In-Memory?

Your gateway runs as 3 instances behind a load balancer for high availability. Each instance handles roughly one third of traffic.

With in-memory counter (wrong): User makes 30 requests to instance 1, 30 to instance 2, 30 to instance 3 = 90 total. Each instance thinks the user only made 30. Limit was 50. Rate limit never triggers. User effectively made 90 requests without restriction.

With Redis counter (correct): All 3 instances increment the same Redis key. Every instance sees the true total. When the user hits 50 total across all instances, all instances return 429. Rate limiting works correctly regardless of how many gateway instances you run.

This is called distributed rate limiting. It only works correctly with a shared external store — Redis is the standard choice because INCR is atomic and Redis is extremely fast (sub-millisecond).

🪣 Token Bucket vs Sliding Window — Two Algorithms

Token Bucket (most common):

  • Imagine a bucket that holds 100 tokens. Each request consumes one token.
  • Tokens refill at a steady rate (e.g. 10 tokens per second = 600 per minute).
  • If the bucket is empty, the request is rejected.
  • Allows bursting — a user can send 100 requests instantly if the bucket is full, then must slow down. Good for APIs where occasional bursts are acceptable.

Sliding Window (more precise):

  • Counts all requests in the last N seconds (not a fixed minute boundary).
  • At any point in time, the user cannot have more than X requests in the past 60 seconds.
  • No burst allowed — the limit is enforced continuously, not reset at the start of each minute.
  • Fairer but more complex and uses more Redis memory.

For most enterprise applications, token bucket is the right choice — it is simpler, efficient in Redis, and the burst allowance is actually a feature (allows users to do occasional heavy tasks).

# Rate limiting configuration in Spring Cloud Gateway spring: cloud: gateway: routes: - id: order-service-route uri: lb://ORDER-SERVICE predicates: - Path=/api/orders/** filters: - StripPrefix=1 - name: RequestRateLimiter args: # How many tokens refill per second redis-rate-limiter.replenishRate: 10 # Maximum tokens in the bucket (burst capacity) redis-rate-limiter.burstCapacity: 20 # How many tokens each request costs (default 1) redis-rate-limiter.requestedTokens: 1 # Which header/key to use to identify the caller key-resolver: "#{@userKeyResolver}"

Authentication at the Gateway — JWT Validation

How the gateway checks who you are before forwarding your request.

🤔 Why Handle Auth at the Gateway?

If you did not have a gateway, every single one of your 8 services would need to implement JWT validation. That means:

  • 8 copies of the same auth code — 8 places to update when the algorithm changes
  • 8 places where a bug could expose an endpoint
  • 8 services each making calls to the auth server to validate tokens
  • Inconsistency — one service might have a stricter validation than another

With a gateway, auth is implemented once. Any request that reaches a downstream service has already been validated. Services can trust the headers the gateway adds (X-User-Id, X-User-Roles) without re-validating.

⚙️ How JWT Authentication Works at the Gateway — Step by Step

A JWT (JSON Web Token) is a string in three parts separated by dots: header.payload.signature. The header says which algorithm was used. The payload contains claims (userId, email, roles, expiry time). The signature proves it was issued by your auth server and has not been tampered with.

Step 1 — Extract the token: The gateway reads the Authorization: Bearer eyJhbGci... header from the request.

Step 2 — Check if the route is public: Routes like /api/auth/login or /api/auth/register do not require a token. The gateway has a whitelist of public paths — it skips token validation for these.

Step 3 — Verify the signature: The gateway checks the token signature using the public key of your auth server. This is a local cryptographic check — no network call to the auth server is needed. If the signature is invalid, the token was either forged or tampered with. Return 401.

Step 4 — Check expiry: The payload contains an exp (expiry) timestamp. The gateway checks if the current time is past this timestamp. If expired, return 401.

Step 5 — Enrich the request: Extract userId and roles from the payload. Add them as request headers: X-User-Id: 456 and X-User-Roles: ADMIN,USER. Forward the enriched request to the downstream service.

Step 6 — Downstream service trusts the headers: OrderService receives the request. It reads X-User-Id to know whose order it is. It reads X-User-Roles to check permissions. It never validates the token itself — it trusts the gateway.

Important Security Note

Downstream services should only trust the X-User-Id and X-User-Roles headers if they come from the gateway. If a clever user manually adds these headers to their request trying to impersonate someone, the gateway must strip these headers from the incoming request before adding its own verified values. Never trust client-supplied identity headers — always use the gateway's validated version.

Correlation ID — How You Debug Production Problems

One of the most valuable things a gateway does — and often the most overlooked.

🤔 The Problem Without Correlation IDs

A user calls your support team saying their order failed at 2:47pm on Tuesday. You go to investigate. Your system has 8 services each writing logs. You search for this user's requests — you find 3,000 log entries across all services from that time window. You cannot tell which log entries belong to the same failed request versus other successful requests from the same user. You spend 2 hours trying to piece it together.

With Correlation IDs — The Same Investigation Takes 2 Minutes

The gateway assigned a UUID (req-7f3a-4b22-9c11) to the request. Every service that handled that request logged this ID alongside every log entry.

You open Kibana (or any log aggregator) and search: correlationId:"req-7f3a-4b22-9c11". You see every log entry from every service, in chronological order, for that exact request:

  1. Gateway received request at 14:47:23.041
  2. OrderService received forwarded request at 14:47:23.089
  3. OrderService called PaymentService at 14:47:23.201
  4. PaymentService called Stripe API at 14:47:23.350
  5. Stripe API timed out at 14:47:25.351 (exactly 2 seconds — the configured timeout)
  6. PaymentService returned 503 to OrderService at 14:47:25.400
  7. OrderService returned 503 to Gateway at 14:47:25.440
  8. Gateway returned 503 to client at 14:47:25.441

Root cause found: Stripe API was slow at that time. Total investigation time: 2 minutes.

How Would You Implement a Custom API Gateway?

Interviewers ask this to test whether you understand how gateways work internally.

🤔 Why Would You Build a Custom Gateway?

Most teams use Spring Cloud Gateway or Kong. But interviewers ask this to test your understanding of internals. Also: sometimes existing solutions are overkill for internal service meshes, or a company has specific compliance requirements that off-the-shelf solutions cannot satisfy.

The key insight is: a gateway is fundamentally a filter chain — the Chain of Responsibility design pattern. Every request passes through a series of filters. Each filter either processes the request and passes it to the next filter, or stops the chain and returns a response.

⚙️ The Filter Chain — Chain of Responsibility Pattern

The filters run in a specific order. The order matters enormously:

CorrelationIdFilter — Always First
If no X-Correlation-ID header exists, generate a UUID and add it. Every subsequent filter and every downstream service will log this ID. Must be first so all other filters can include it in their logs.
AuthenticationFilter — Second
Validate the JWT. If invalid → return 401 immediately, chain stops. If valid → extract userId and roles, add as headers, pass to next filter. Must run before rate limiting so we know which user to rate-limit.
RateLimitFilter — Third
Uses the userId (now available from step 2) to check Redis counter. If over limit → return 429. If under → increment counter and pass to next filter. Needs userId from auth filter, so must come after it.
RoutingFilter — Fourth
Look up which service this path maps to. Forward the request using WebClient (reactive) or RestTemplate (blocking). Wait for the response. Pass response through the remaining filters.
ResponseCleanupFilter — Fifth
Strip internal headers from the response before returning to client (X-Internal-Service-Name, X-Pod-Id, etc.). Clients should not see internal infrastructure details.
LoggingFilter — Always Last
Log the request: correlation ID, path, user, response status code, total time in milliseconds. Must be last so it captures the final response code and total end-to-end latency. Logs even rejected requests (401, 429) so you have a complete audit trail of everything that entered the system.
@Component public class AuthFilter implements GlobalFilter, Ordered { @Override public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) { String path = exchange.getRequest().getURI().getPath(); // Step 1: Skip auth for public routes like /api/auth/login if (isPublicPath(path)) { return chain.filter(exchange); // pass to next filter } // Step 2: Extract the token from Authorization header String authHeader = exchange.getRequest() .getHeaders().getFirst(HttpHeaders.AUTHORIZATION); if (authHeader == null || !authHeader.startsWith("Bearer ")) { exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED); return exchange.getResponse().setComplete(); // stop chain, return 401 } String token = authHeader.substring(7); // remove "Bearer " prefix // Step 3: Validate the JWT (local cryptographic check — no network call) if (!jwtService.isValid(token)) { exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED); return exchange.getResponse().setComplete(); // stop chain, return 401 } // Step 4: Extract claims and add as headers for downstream services String userId = jwtService.getUserId(token); String roles = jwtService.getRoles(token); ServerHttpRequest enrichedRequest = exchange.getRequest().mutate() .header("X-User-Id", userId) .header("X-User-Roles", roles) .build(); // Step 5: Pass enriched request to next filter in chain return chain.filter(exchange.mutate().request(enrichedRequest).build()); } // Order = -100 means this filter runs BEFORE Spring's built-in filters @Override public int getOrder() { return -100; } }

API Gateway vs Load Balancer — A Common Interview Trap

These two are often confused. Understanding the difference is a key interview signal.

🤔 What Is the Difference?

This is one of the most common interview follow-up questions. Many candidates confuse them or think they are the same thing.

AspectLoad BalancerAPI Gateway
Primary job Distribute traffic across multiple identical instances of the same service Route traffic to different services based on the request path, apply auth, rate limiting, logging
Awareness Has no knowledge of your application or business logic. Purely distributes requests. Application-aware. Understands paths, headers, JWT tokens, business rules.
Works at Network layer (Layer 4) or HTTP layer (Layer 7). Mostly traffic distribution. Application layer (Layer 7). Full HTTP awareness — reads and modifies requests and responses.
Example AWS ELB, NGINX upstream block, Kubernetes Service — all send traffic to 3 copies of the same pod Spring Cloud Gateway, Kong, AWS API Gateway — routes /orders to OrderService, /products to ProductService
Handles auth? No — no concept of JWT or user identity Yes — validates JWT and extracts user claims
Used together? Yes — the API Gateway is often itself behind a load balancer. External traffic hits the load balancer, which distributes it across 3 gateway instances. Each gateway instance routes to the appropriate services.
✅ API Gateway — Advantages
  • Single entry point — clients only need one address
  • Cross-cutting concerns in one place — auth, rate limiting, logging
  • Services are hidden from the outside world
  • Easy to add new services without changing clients
  • Can do request/response transformation
  • Simplifies client code — one call instead of many
❌ API Gateway — Disadvantages
  • Single point of failure if not made highly available
  • Added network hop — every request goes through it
  • Risk of becoming a bottleneck at very high traffic
  • Operational complexity — must be deployed, monitored, updated
  • Risk of "smart gateway" anti-pattern — business logic leaking in
  • Can become a deployment blocker if all teams share one gateway

Follow-Up Questions & Full Answers

The exact questions interviewers ask after you explain the API gateway. Click to expand.

Q1: Should authentication happen at the gateway or at each service? What are the trade-offs? +

Authentication at the gateway (recommended for most systems): The gateway validates the JWT once, extracts the user identity, and passes it downstream as trusted headers. Services do not need to implement JWT validation — they just read the X-User-Id header. This is the most common pattern for internal microservices behind a corporate API gateway.

Authentication at each service (defence in depth): Each service independently validates the JWT. Even if the gateway is bypassed or compromised, services still protect themselves. This is the pattern used for highly sensitive financial or healthcare systems where a single compromised gateway cannot expose all services.

Hybrid (best of both): The gateway handles JWT validation and adds identity headers. Services do not re-validate the JWT but they DO check authorisation — they verify the user has permission for the specific resource they are requesting. Authorisation is business-specific and belongs in the service. Authentication (proving who you are) can be centralised. Authorisation (proving what you can do) should stay in the service.

The practical rule: Centralise authentication at the gateway. Keep authorisation in each service. This gives you the simplicity of centralised auth without the risk of centralised authorisation (which would require every service to call a central auth service for every request).

Q2: What happens to in-flight requests when you deploy a new version of the gateway? +

This is a question about zero-downtime deployments for the gateway itself. The gateway is a microservice too — it has the same deployment challenges.

The solution is a rolling deployment: You run multiple instances of the gateway (at least 2, ideally 3) behind a load balancer. When you deploy a new version, you update one instance at a time. At any point, both old and new gateway versions are running.

The load balancer in front of the gateways detects when a gateway instance is being updated (via health checks) and stops sending it new traffic. In-flight requests on the old instance complete normally. Once the old instance drains (all in-flight requests finish), it shuts down and the new instance takes over. Then the next instance is updated the same way. This is called a rolling update with connection draining.

In Kubernetes, this is handled by the Deployment's rolling update strategy combined with a preStop lifecycle hook that waits for in-flight requests to drain before the pod terminates.

Important: New and old gateway versions must be backward compatible during the transition. Avoid making breaking changes to gateway configuration that would cause the old and new versions to behave incompatibly with the same backend services.

Q3: How do you prevent the gateway from becoming a single point of failure? +

The API gateway by definition is the entry point for all traffic — if it goes down, nothing works. Three strategies prevent this:

Strategy 1 — Multiple instances: Run at least 3 gateway instances in Kubernetes, each on a different node (using pod anti-affinity rules). A cloud load balancer distributes traffic across all 3. If one instance crashes or a node goes down, traffic automatically routes to the remaining 2. This eliminates single-instance failure.

Strategy 2 — Multi-region or multi-AZ: In cloud environments, run gateway instances across multiple Availability Zones (AZs). AWS, GCP, and Azure each have multiple physically separate data centres within a region. Traffic is load balanced across AZs. An entire AZ can fail and your gateway stays up.

Strategy 3 — Circuit breakers on the gateway itself: The gateway should have circuit breakers on its calls to downstream services. If OrderService goes down, the gateway should not keep making calls to it (wasting threads and connections). The circuit breaker opens, returns a fallback 503 response, and gives OrderService time to recover.

Strategy 4 — Stateless gateway: The gateway should not store any request state in memory. Rate limiting state goes to Redis. Route configurations come from a config server. This means any instance can handle any request — no sticky sessions, no in-memory data to lose if an instance crashes.

Q4: How do you implement A/B testing or canary releases at the gateway level? +

The gateway is the perfect place to implement traffic splitting because it sees every request before it reaches any service.

Canary release: You have OrderService v1 and OrderService v2. You want 5% of traffic to go to v2 as a test. In Spring Cloud Gateway, you can configure a WeightRoutePredicateFactory that assigns weights to routes. 95 weight to v1, 5 weight to v2. The gateway randomly routes 5% of requests to v2 matching the weight distribution.

A/B testing by user segment: Instead of random splitting, route specific users to the new version. Extract the user ID or a feature flag from the JWT header. Check if this user is in the experiment group (stored in Redis or a feature flag service). If yes, route to v2. If no, route to v1. This lets you test with specific user cohorts rather than random traffic.

Header-based routing: Developers or QA testers send a special header like X-Version: v2. The gateway routes requests with that header to the new version. Normal users without the header go to v1. This lets the team test v2 in production with real data without exposing it to users.

Tools like Argo Rollouts or Flagger automate this in Kubernetes — they progressively increase the canary percentage while monitoring metrics, and automatically roll back if error rates increase.

Q5: What is the "smart gateway, dumb services" anti-pattern? +

This is a well-known anti-pattern where business logic gradually migrates into the gateway, making it a monolith in disguise.

It starts innocuously. A developer adds a rule: "if the user is a premium member, route them to the fast endpoint." Then: "if the order total is over £100, add a free shipping header." Then: "if it is Black Friday, add a discount code to the request." Before long, the gateway contains business rules that belong in services, and the gateway has become a central intelligence layer that all services depend on.

Why this is harmful:

  • The gateway becomes a deployment bottleneck — every business change requires a gateway deployment
  • The gateway starts knowing about domain concepts (premium users, order totals) which violates its purpose
  • Business logic in the gateway is harder to test — it processes raw HTTP traffic, not domain objects
  • It reintroduces coupling — services become dependent on the gateway's transformations

The rule: The gateway should handle only cross-cutting infrastructure concerns — authentication, rate limiting, routing, logging, SSL, correlation IDs. Any logic that involves understanding your domain (users, orders, products, pricing) belongs in services.

Q6: How do you handle different rate limits for different tiers of users (free vs paid vs enterprise)? +

This is a very common real-world requirement. Most public APIs have tiered rate limits — free users get 100 requests/hour, paid users get 1,000, enterprise customers get unlimited.

Implementation approach:

When the gateway validates the JWT token, the payload includes the user's subscription tier ("tier": "PREMIUM"). The gateway's rate limiter reads this claim and applies tier-specific limits.

In Spring Cloud Gateway, you implement a custom KeyResolver that returns a key like userId:PREMIUM. The rate limiter config maps each key pattern to different limits: FREE = 100/hour, PREMIUM = 1000/hour, ENTERPRISE = no limit.

Alternatively, the tier information is stored in Redis alongside the counter. When the gateway looks up the rate limit, it first fetches the user's tier from Redis (cached from the last auth call), then applies the appropriate limit for that tier.

For enterprise customers: The recommended approach is a separate API key (not a JWT) that bypasses the standard rate limiter entirely and goes through a dedicated route with no limit. This also makes it easy to revoke access or apply custom per-customer limits.

Interview Tip — The Complete Answer Pattern

When asked "what API gateway did you use in your project and what did it do?", structure your answer in three parts:

1. What it is: "We used Spring Cloud Gateway as our single entry point for all client requests into our microservices system."

2. What it did: "It handled JWT authentication by validating tokens and extracting user identity, rate limiting via Redis counters to protect services from traffic spikes, request routing based on path predicates to 8 downstream services, correlation ID injection for distributed tracing, and TLS termination."

3. The trade-off you were aware of: "We ran 3 gateway instances to avoid a single point of failure, and we were careful not to put any business logic in the gateway — keeping it purely as an infrastructure concern."

This shows you understand the what, the how, and the why — and that you have thought about failure modes.
