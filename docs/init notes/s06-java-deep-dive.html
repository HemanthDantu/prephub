@Transactional — What It Is and How It Actually Works

Understanding @Transactional from the ground up — what a transaction is, what the annotation does, and all the ways it can go wrong.

🤔 Start Here — What Is a Database Transaction?

A database transaction is a group of operations that are treated as a single, indivisible unit. Either ALL operations in the group succeed and are committed to the database permanently, or ALL of them fail and are rolled back — as if they never happened.

The classic example: transferring money between two bank accounts. You need to:

  1. Debit £100 from Account A
  2. Credit £100 to Account B

If step 1 succeeds but step 2 fails (server crash), Account A has lost £100 with nothing going to Account B. This is a disaster. A transaction groups both steps — if step 2 fails, step 1 is automatically rolled back. Account A keeps its £100.

Transactions have four properties known as ACID: Atomicity (all or nothing), Consistency (data stays valid), Isolation (transactions don't see each other's partial results), Durability (committed data survives crashes).

What @Transactional Does

The @Transactional annotation tells Spring to wrap the method in a database transaction. Spring does this using a proxy — when you call the method through a Spring bean, Spring intercepts the call, begins a transaction, runs your method, and either commits (success) or rolls back (exception) the transaction when the method exits.

Without @Transactional, each individual repository call (each save(), each findById()) opens and closes its own tiny transaction independently. Multiple calls in the same method are NOT grouped together — if the second one fails, the first one is already committed and cannot be rolled back.

Concrete Example — Why @Transactional Matters

Without @Transactional — dangerous:

Your placeOrder() method: saves the order, saves the payment record, updates inventory. Each save is its own transaction. The order and payment save successfully. The inventory update fails. The order and payment are committed and cannot be undone. You have confirmed an order with no inventory reserved. Data is now inconsistent.

With @Transactional — safe:

Spring opens ONE transaction at the start of placeOrder(). All three saves happen inside it. When inventory update fails, Spring catches the exception and rolls back ALL three saves. Nothing is committed. Data stays consistent.

⚙️ What Triggers a Rollback?

By default, Spring rolls back the transaction when an unchecked exception (RuntimeException or Error) is thrown. It does NOT automatically roll back on checked exceptions (those that extend Exception but not RuntimeException).

This is a common source of bugs:

  • throw new RuntimeException() → transaction rolls back ✅
  • throw new NullPointerException() → rolls back ✅ (it's a RuntimeException)
  • throw new IOException() → does NOT roll back ❌ (it's a checked Exception)
  • throw new SQLException() → does NOT roll back ❌ (checked)

To roll back on a checked exception: @Transactional(rollbackFor = IOException.class)

To NOT roll back on a specific unchecked exception: @Transactional(noRollbackFor = OptimisticLockException.class)

Transaction Propagation — What Happens When Transactions Call Each Other
🤔 What Is Propagation?

Propagation controls what happens when a method marked @Transactional is called from another method that is already inside a transaction. Does the inner method join the outer transaction? Start a brand new one? Refuse to run without one?

Outer Transaction (OrderService.placeOrder)
REQUIRED (default) — joins outer transaction
PaymentService.charge() also has @Transactional(REQUIRED). It joins the existing transaction. Both live and die together — if placeOrder() rolls back, the charge rolls back too. Use for: normal business operations that must be atomic together.
REQUIRES_NEW — suspends outer, starts own transaction
AuditService.log() has @Transactional(REQUIRES_NEW). Spring SUSPENDS the placeOrder() transaction, starts a fresh one for the audit log, commits it immediately, then resumes placeOrder(). If placeOrder() later rolls back — the audit log is already committed and stays. Use for: audit logs, notifications — things that must persist even if the main operation fails.
SUPPORTS — uses outer if present, no transaction if not
If called within a transaction, participates. If called standalone, runs without a transaction. Use for: read methods that work fine either way.
MANDATORY — must be called within an existing transaction
Throws IllegalTransactionStateException if there is no active transaction. Use for: methods that MUST never run outside a transaction — enforces calling discipline.
// REQUIRES_NEW — real use case: audit log must persist even if main tx rolls back @Service public class AuditService { @Transactional(propagation = Propagation.REQUIRES_NEW) public void log(String action, Long entityId, String userId) { // This runs in its OWN transaction // Committed immediately, independent of caller's transaction auditRepo.save(new AuditLog(action, entityId, userId, Instant.now())); } } @Service public class OrderService { @Transactional // REQUIRED (default) public Order placeOrder(OrderRequest req) { Order order = orderRepo.save(new Order(req)); auditService.log("ORDER_ATTEMPTED", order.getId(), req.getUserId()); // Audit log committed in its own transaction — already in DB inventoryService.reserve(order); // throws if out of stock // If this throws, placeOrder tx rolls back, order is deleted // But audit log stays — we know someone tried to place this order return order; } } // readOnly = true — three real benefits @Transactional(readOnly = true) public List<Order> getOrderHistory(Long userId) { // Benefit 1: Hibernate skips dirty checking (no scan for changes to save) // Benefit 2: Routing datasource sends this to read replica automatically // Benefit 3: DB can optimise — no write locks needed return orderRepo.findByUserId(userId); }
The Most Common @Transactional Mistake — Self-Invocation

@Transactional works through a Spring proxy. When you call a @Transactional method from OUTSIDE the class (through the injected Spring bean), the proxy intercepts the call and wraps it in a transaction.

When you call a @Transactional method from within the same class using this.myMethod(), you call the real object directly — bypassing the proxy entirely. The @Transactional annotation has no effect.

  • ✅ Works: orderService.placeOrder() called from another class through the injected bean
  • ❌ Does NOT work: this.placeOrder() called from within OrderService itself

Fix: Extract the method into a separate Spring bean and inject it, or inject the service into itself using @Autowired OrderService self (Spring 4.3+ supports self-injection).

📋 From Your Notes: "If I Don't Use @Transactional on a GET Method — Is It a Problem?"

Direct answer: it depends on what the method does.

Simple single repository call (usually fine without it): return orderRepo.findById(id) — Spring Data repository methods have their own built-in transaction per call. This works fine without an explicit @Transactional on the service method.

Multiple repository calls (potentially a problem without it): If your method calls orderRepo.findById(id) and then accesses order.getItems() (a lazy-loaded collection), the JPA session closes after the first call. The second access throws LazyInitializationException because there is no open session.

Best practice: Always add @Transactional(readOnly = true) to service methods that read data — it is free performance (no dirty checking) and prevents lazy loading exceptions. The cost of adding it is zero. The cost of missing it can be subtle bugs.

@Autowired vs @Component — Spring Dependency Injection Explained

What each annotation does, which classes are eligible, and the right way to inject dependencies.

🤔 The Big Picture — Inversion of Control (IoC)

In traditional Java, you create objects: OrderService service = new OrderService(new OrderRepository(), new KafkaTemplate());. You are in control of creating and wiring everything together.

Inversion of Control (IoC) flips this. You declare what you need, and Spring creates and manages the objects for you. You give up control of object creation to the Spring container (also called the Application Context). Spring is now in control.

Why is this better? Testing becomes trivial — inject a mock instead of a real object. Swapping implementations is easy — change which bean gets injected without changing the caller. Lifecycle management is automatic — Spring creates beans at startup, manages their scope, and cleans them up.

@Controller
Web Layer — HTTP Request Handlers
Handles incoming HTTP requests. Maps URL paths to methods. Parses request body, path variables, query params. Returns responses. Specialisation of @Component — adds request mapping scanning on top. Use @RestController (= @Controller + @ResponseBody) for REST APIs.
@RestController @RequestMapping @GetMapping
@Service
Business Logic Layer
Specialisation of @Component — functionally identical, but semantically signals "this class contains business logic." Orchestrates calls to repositories and other services. Applies @Transactional boundaries. Has no technical difference from @Component — it is a documentation annotation that makes architecture explicit.
@Service @Transactional
@Repository
Data Access Layer
Specialisation of @Component with one technical difference: Spring adds exception translation. Vendor-specific exceptions (e.g. HibernateException, SQLException) are automatically translated to Spring's DataAccessException hierarchy. This means your service layer does not need to know about Hibernate-specific exceptions.
@Repository JpaRepository
📋 Which Classes Are Eligible for @Autowiring?

From your notes — this is a direct interview question. The answer: any class that is registered as a Spring bean.

A class becomes a Spring bean in two ways:

  • Component scanning: Annotate the class with @Component, @Service, @Repository, or @Controller. Spring scans the classpath for these annotations and registers them automatically.
  • @Bean method: In a @Configuration class, write a method annotated with @Bean that returns an instance. The returned object is registered as a bean. Use this for third-party classes you cannot annotate.

A plain POJO (a class with no annotations) is NOT a Spring bean and cannot be @Autowired. A third-party class (RestTemplate, ObjectMapper) is NOT a Spring bean until you create a @Bean method for it.

// COMPONENT SCANNING — annotate your own classes @Service // Registers OrderService as a Spring bean public class OrderService { // CONSTRUCTOR INJECTION — the recommended way private final OrderRepository orderRepo; // final = cannot change after init private final NotificationService notifier; // depends on interface, not impl // No @Autowired needed — Spring injects when there's only one constructor public OrderService(OrderRepository orderRepo, NotificationService notifier) { this.orderRepo = orderRepo; this.notifier = notifier; } } // @Bean METHOD — for third-party classes you cannot annotate @Configuration public class AppConfig { @Bean public RestTemplate restTemplate(RestTemplateBuilder builder) { // RestTemplate is not your class — you can't put @Component on it // @Bean method makes it available for @Autowiring everywhere return builder .setConnectTimeout(Duration.ofSeconds(3)) .setReadTimeout(Duration.ofSeconds(10)) .build(); } @Bean @Primary // This bean wins when multiple of same type exist public ObjectMapper objectMapper() { return JsonMapper.builder() .configure(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES, false) .build(); } } // FIELD INJECTION — works but avoid it (hard to test, hides dependencies) @Service public class BadExample { @Autowired private OrderRepository orderRepo; // Can't make final. Hides that this dep exists. // To test: must start Spring context. Can't do: new BadExample(mockRepo) }
✅ Constructor Injection — Why It Is Best
  • Dependencies are explicit — visible in the constructor signature
  • Fields can be final — immutable, safer
  • Testable without Spring — new OrderService(mockRepo)
  • Forces you to think about too many dependencies (large constructor = design smell)
  • No @Autowired annotation needed in modern Spring
❌ Field Injection — Why to Avoid It
  • Cannot make fields final
  • Dependencies are hidden — not obvious from the API
  • Cannot test without starting Spring context
  • Allows creating objects in an invalid state
  • Harder to detect circular dependencies at compile time

SOLID Principles — With Real Code Examples

Each principle explained, what it prevents, and a concrete before/after code example.

S
Single Responsibility Principle
"A class should have only one reason to change."
What it prevents: The "God class" — one enormous class that does validation, database access, email sending, and event publishing. When the email template changes, you edit the same class that handles DB logic. Any change risks breaking something else.

In microservices: Each service should own one business domain. OrderService should not also handle shipping logic.
O
Open/Closed Principle
"Open for extension, closed for modification."
What it prevents: Endless if-else chains that grow with every new requirement. Adding a new payment provider means editing existing code — risking regressions in CreditCard and PayPal processing you already tested.

The fix: Interface + implementations. New providers add a new class — existing code untouched.
L
Liskov Substitution Principle
"Subtypes must be substitutable for their base type."
What it prevents: Surprises when you use a subclass. If Square extends Rectangle but breaks the contract (setting width changes height), code that works with Rectangle breaks when given a Square. Classic LSP violation.

Rule of thumb: If you find yourself checking instanceof to handle subclass behaviour differently, LSP is violated.
I
Interface Segregation Principle
"Clients should not depend on methods they do not use."
What it prevents: Fat interfaces that force implementors to provide dummy implementations for methods they do not need. A ReadOnlyOrderService forced to implement delete() from a general OrderService interface.

The fix: Split into OrderReader, OrderWriter, OrderDeleter. Clients depend only on what they use.
D
Dependency Inversion Principle
"Depend on abstractions, not concretions."
What it prevents: High-level business logic depending on low-level implementation details. If OrderService directly instantiates new EmailNotificationService(), you cannot swap it for SMS without editing OrderService. You also cannot test OrderService without a real email server running.

The fix: OrderService depends on a NotificationService interface. Spring injects the implementation at runtime. To test: inject a mock. To swap: change the bean configuration. OrderService code never changes.

Concurrent Updates — "Two Users Updating the Same Object"

From your notes: "I have a user 1 object, you are also updating it, I am also trying to update the same object." Here is the complete solution.

🤔 The Lost Update Problem

Two users open the same record simultaneously. Both read it. Both modify it. Both save it. The second save overwrites the first — silently, with no error. The first user's changes are gone forever. This is called the lost update problem.

👩 Alice (Thread 1)
1. Reads order: status=PENDING, items=5
2. Changes status to SHIPPED
3. Saves: status=SHIPPED, items=5 ✅
 
😱 Her SHIPPED status is GONE — Bob's save overwrote it
👨 Bob (Thread 2)
1. Reads order: status=PENDING, items=5
 
 
2. Changes items to 3
3. Saves: status=PENDING, items=3 ← overwrites Alice!
🛡️ Solution 1 — Optimistic Locking (@Version)

Add a @Version field to your entity. Hibernate automatically manages this field. Every time the entity is saved, the version number increments. When you try to save, Hibernate adds the version to the WHERE clause: UPDATE orders SET status=? WHERE id=? AND version=?.

If Alice read version=1 and Bob also read version=1, Alice saves first (version becomes 2). When Bob tries to save with version=1, the WHERE clause matches zero rows. Hibernate detects this and throws OptimisticLockingFailureException. Bob must re-read the data (now at version=2) and retry his change.

Best for: User-facing edits where conflicts are rare. Low-contention scenarios. Financial read-and-display workflows.

🔐 Solution 2 — Pessimistic Locking (SELECT FOR UPDATE)

Lock the database row the moment you read it. Other transactions trying to read the same row are blocked until you release the lock. This prevents any other transaction from even reading the data while you are working with it.

Uses SQL SELECT ... FOR UPDATE under the hood. The lock is released when your transaction commits or rolls back.

Best for: Inventory decrement on Black Friday (cannot afford a single oversell), financial account balance updates (absolutely cannot lose a debit/credit), any scenario where concurrent modification is likely and the cost of conflict is high.

// ── Optimistic Locking — @Version ────────────────── @Entity public class Order { @Id private Long id; private String status; private int itemCount; @Version private Long version; // Hibernate manages this — do not set manually // Hibernate generates: UPDATE orders SET status=?, version=2 // WHERE id=? AND version=1 // If version=1 row doesn't exist → OptimisticLockingFailureException } // Handle conflict — retry with fresh data public void updateStatus(Long orderId, String newStatus) { int maxRetries = 3; for (int i = 0; i < maxRetries; i++) { try { Order order = orderRepo.findById(orderId).orElseThrow(); order.setStatus(newStatus); orderRepo.save(order); return; // success } catch (OptimisticLockingFailureException e) { if (i == maxRetries - 1) throw e; // give up after 3 attempts // Retry: re-read fresh data and try again } } } // ── Pessimistic Locking — SELECT FOR UPDATE ──────── @Repository public interface ProductRepository extends JpaRepository<Product, Long> { @Lock(LockModeType.PESSIMISTIC_WRITE) // SELECT ... FOR UPDATE @Query("SELECT p FROM Product p WHERE p.id = :id") Product findByIdForUpdate(@Param("id") Long id); } @Transactional public void decrementStock(Long productId, int qty) { // Row is LOCKED — other transactions wait here until we commit Product product = productRepo.findByIdForUpdate(productId); if (product.getStock() < qty) { throw new InsufficientStockException("Cannot sell more than available"); } product.setStock(product.getStock() - qty); productRepo.save(product); // Lock released when @Transactional method exits } // ── Redis Distributed Lock — for cross-instance coordination ── // When you have multiple application instances and need locking across all RLock lock = redissonClient.getLock("product-lock:" + productId); try { lock.lock(5, TimeUnit.SECONDS); // auto-release after 5s (prevents deadlock) // ... update product ... } finally { lock.unlock(); // always release in finally }

Multithreading & Async — Which Process Did You Use?

From your notes: "Multithreading — which multithreading process are you used in your project?"

🤔 The Core Decision: CPU-Bound vs I/O-Bound Work

Before choosing an async approach, identify what kind of work you are parallelising:

  • I/O-bound work (waiting for DB, HTTP calls, file reads): The thread spends most time waiting, not computing. Many threads are fine because they mostly sleep. Virtual threads (Java 21) excel here.
  • CPU-bound work (calculations, data transformation, image processing): Threads actively use the CPU. Too many threads = context switching overhead slows everything down. Rule: threads ≈ CPU core count.

Local vs Static vs Instance Variables — Print Them

From your notes: "Local variable, static variable, non-static variable — print them." Here is the complete explanation with memory locations.

📋
Stack Memory
Local variables Method parameters Return addresses

Created when method starts. Destroyed when method returns. Each thread has its own stack — inherently thread-safe. Very fast (just a pointer move).
🏗️
Heap Memory
Instance variables Objects (new X()) Arrays

Lives inside each object on the heap. Exists as long as the object exists. Shared between threads that hold a reference to the same object. Garbage collected when no references remain.
🏛️
Metaspace
Static variables Class metadata Static methods

ONE copy per class. Loaded when class loads, unloaded when class is unloaded. Shared by ALL instances of the class and ALL threads. Not garbage collected in the normal way.

Follow-Up Questions & Full Answers
Q1: What are the transaction isolation levels and when would you use each? +

Isolation levels control how much one transaction can see the uncommitted changes of other concurrent transactions. From weakest (most concurrency but most anomalies) to strongest (least concurrency but no anomalies):

READ UNCOMMITTED: You can read another transaction's uncommitted data (dirty reads). If that transaction rolls back, you read data that never actually existed. Almost never used in production — the risk is too high. Only useful for approximate reporting where accuracy does not matter.

READ COMMITTED (PostgreSQL default): You only see data that has been committed. No dirty reads. But two reads of the same row in the same transaction might return different values if another transaction commits between them (non-repeatable read). Good for most OLTP applications.

REPEATABLE READ (MySQL default): Within a transaction, the same row always returns the same value even if another transaction commits a change. Prevents non-repeatable reads. But two queries with different WHERE clauses might return different row counts if another transaction inserts rows matching your criteria (phantom read). Good for financial report generation within a single transaction.

SERIALIZABLE: Transactions behave as if they ran one after another, not concurrently. No dirty reads, no non-repeatable reads, no phantom reads. But very poor performance — extensive locking. Use only when absolute correctness is required: bank account balances, inventory that absolutely cannot oversell.

In Spring: @Transactional(isolation = Isolation.REPEATABLE_READ). In practice, most applications use the database default (READ COMMITTED) and handle edge cases with optimistic or pessimistic locking on specific operations rather than blanket SERIALIZABLE isolation.

Q2: What is circular dependency in Spring and how do you fix it? +

A circular dependency is when Bean A depends on Bean B, and Bean B depends on Bean A (directly or through a chain). Spring cannot create either bean because each one requires the other to already exist.

Example: OrderService has a constructor parameter of type PaymentService. PaymentService has a constructor parameter of type OrderService. Spring tries to create OrderService first, but needs PaymentService. It tries to create PaymentService, but needs OrderService. Infinite loop. Spring throws BeanCurrentlyInCreationException at startup.

Fix 1 — Refactor to remove the circular dependency (best): A circular dependency almost always means your design has a problem. OrderService and PaymentService probably share some responsibility that should be extracted into a third service. Extract the shared logic into OrderPaymentCoordinator or similar. Both services depend on it but not on each other.

Fix 2 — Use @Lazy on one injection point: @Autowired public OrderService(@Lazy PaymentService paymentService). Spring injects a lazy proxy instead of the real bean. The real bean is only created when a method is first called. This breaks the circular startup dependency.

Fix 3 — Use setter injection instead of constructor injection for one: Spring creates both beans without their circular dependency, then injects them via setter after creation. Works, but setters cannot be final and this is a code smell.

Important note: In Spring Boot 2.6+, circular dependencies are detected and throw an error by default. In older versions, they could silently work with field injection (using proxies) which hid the design problem. The strict detection in modern Spring is a feature — it forces you to fix the design.

Q3: What is dirty checking in Hibernate and when does it cause unexpected UPDATE statements? +

Dirty checking is Hibernate's mechanism for detecting which entities have been modified within a transaction and need to be saved. When a transaction commits, Hibernate compares every managed entity's current state against the snapshot it took when it was loaded. If anything changed, Hibernate generates and executes an UPDATE statement — automatically, without you calling save().

Why this is surprising: You load an entity to read it. Your code accidentally modifies a field (common with getters that have side effects, or collections that get populated). You never call save(). But Hibernate detects the change and issues an UPDATE anyway when the transaction commits.

Common unexpected UPDATE scenario:

  • You load a User entity to check their email for a notification
  • Your code calls user.getLastAccessTime() — and that getter updates the lastAccessTime field as a side effect
  • You never call save()
  • Transaction commits — Hibernate dirty check detects lastAccessTime changed — issues UPDATE
  • In logs, you see unexpected UPDATE statements for read operations

Fix 1 — Use @Transactional(readOnly = true): Hibernate skips dirty checking entirely for read-only transactions. No matter what you accidentally modify, no UPDATE is generated. This is why readOnly=true is not just a hint — it has real behaviour effects.

Fix 2 — Use DTO projections: Instead of loading a full entity (which becomes managed by Hibernate), load a DTO using a constructor expression. DTOs are not managed by Hibernate — no dirty checking, no unexpected updates.

Fix 3 — Use @Immutable on entities that should never be updated: Tells Hibernate to never issue UPDATE for this entity, even if it detects changes.

Q4: What is the difference between a race condition and a deadlock? +

These are two different types of concurrency problems and they require different solutions.

Race condition: Two or more threads access shared data concurrently, and the final result depends on the timing and order of execution. The outcome is non-deterministic — it "races" to see which thread finishes first.

Example: Two threads both read a counter value (10), both increment it, both write back (11). The counter should be 12 but is 11 — one increment was lost. The "lost update" problem is a race condition.

Fix: synchronisation (synchronized block, ReentrantLock), atomic operations (AtomicInteger), or database locking (optimistic/pessimistic).

Deadlock: Two or more threads are each waiting for a resource held by the other — neither can proceed. Both threads are stuck forever (or until a timeout kills them).

Example: Thread 1 holds lock on Table A and needs Table B. Thread 2 holds lock on Table B and needs Table A. Thread 1 waits for Thread 2 to release B. Thread 2 waits for Thread 1 to release A. Neither ever releases. The database detects this and kills one transaction (the "deadlock victim") with an error.

Fix: always acquire locks in the same order across all code paths (Thread 1 and Thread 2 should both lock A then B). Use short transactions to reduce the window for deadlocks. Use optimistic locking where possible to avoid holding DB locks at all.

Q5: Give a real example of an LSP violation and how to fix it. +

The classic example is the Square-Rectangle problem, but here is a more realistic Java example:

The violation: You have a Bird base class with a fly() method. You create an Ostrich subclass (it is a bird). But ostriches cannot fly. Your Ostrich.fly() throws UnsupportedOperationException. Any code that works with Bird objects and calls fly() will now break when given an Ostrich. The subtype is not substitutable for the base type without surprises.

The fix — Interface Segregation instead of deep inheritance: Remove fly() from the Bird base class. Create a Flyable interface with fly(). Make Eagle, Parrot, Sparrow implement Flyable. Ostrich does not implement Flyable. Code that needs flying behaviour depends on Flyable, not Bird. Ostriches can safely be Birds without breaking anything.

A Spring-specific example: You have a Repository base that includes a delete() method. You create a ReadOnlyRepository subclass for audit tables that should never be deleted. Your ReadOnlyRepository.delete() throws UnsupportedOperationException. Code that accepts a Repository and calls delete() will break when given a ReadOnlyRepository — LSP violation.

Fix: Separate the interfaces. ReadableRepository only has find methods. WritableRepository extends ReadableRepository and adds save and delete. The audit table repository only implements ReadableRepository. Code that needs delete depends on WritableRepository — it can never accidentally receive a ReadableRepository.

Q6: What is a memory leak in Java and what are the common causes? +

Java has garbage collection — objects are automatically freed when there are no more references to them. A memory leak in Java is when objects accumulate in memory and are never collected because references to them are inadvertently kept alive — even though they will never be used again.

Common cause 1 — Static collections that grow unboundedly: static Map cache = new HashMap() in a long-running application. Entries are added on every request but never removed. The map grows indefinitely. Fix: use a bounded cache with eviction (Caffeine, Guava Cache) with a max size and TTL.

Common cause 2 — ThreadLocal not cleaned up: ThreadLocal stores per-thread data. In a thread pool, threads are reused. If you set a ThreadLocal value and the thread is returned to the pool without clearing it, the value persists for the next request that reuses the thread. Over time, each thread's ThreadLocal grows. Fix: always call threadLocal.remove() in a finally block or servlet filter after each request.

Common cause 3 — Listeners not deregistered: You register an event listener or callback. The object holding the listener is garbage collected but the listener registration keeps a reference to it. The object cannot be collected. Fix: deregister listeners when they are no longer needed.

Common cause 4 — JPA entities held in memory: Loading large numbers of entities into a List in a long-running service method. The entire result set sits in heap until the method returns. Fix: use streaming (Stream<Entity>) and process row by row instead of loading all at once.

How to detect: Monitor heap usage over time with JVM metrics (Micrometer + Prometheus). A slow, steady upward trend that never decreases is the signature of a memory leak. Use heap dumps (jmap, VisualVM, or Java Flight Recorder) to identify which object types are accumulating.

Interview Tip — Answering Java Deep-Dive Questions

When asked about @Transactional, never just say "it wraps the method in a transaction." Show you understand the proxy mechanism: "@Transactional works through a Spring AOP proxy. When another bean calls a @Transactional method on my service, Spring intercepts the call through the proxy, begins a JDBC transaction, executes my method, and commits or rolls back based on whether an exception was thrown. One important subtlety: if I call a @Transactional method from within the same class, it bypasses the proxy and has no effect — this is a common source of bugs."

For concurrent updates: "We use optimistic locking via @Version for user-facing edits where conflicts are rare. For inventory operations during high-traffic events, we use pessimistic locking with SELECT FOR UPDATE because the cost of overselling is much higher than the performance cost of blocking. For distributed locking across multiple application instances, we use a Redis Redlock."
