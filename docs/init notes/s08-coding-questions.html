How to Approach Coding Questions in an Interview

The process matters as much as the solution. Interviewers watch how you think, not just what you produce.

🧠 The Five Steps — Always Follow This Order
  1. Clarify the problem. Ask: what is the input type? Can there be nulls? What do we do with duplicates? What is the expected output format? This shows maturity and prevents solving the wrong problem.
  2. Talk through the naive approach first. Even if it is O(n²), say it out loud. "The naive approach is a nested loop — O(n²). That is probably too slow. Let me think about how to do better."
  3. Explain your optimised approach before coding. "I can convert list B to a HashSet. HashSet.contains() is O(1) instead of O(n). That brings the overall complexity from O(n²) to O(n+m)."
  4. Code it. Talk while you code. "Here I am creating the set from list B. Now I stream list A and filter using set.contains..."
  5. Analyse complexity and test with examples. State the time and space complexity. Walk through your example to verify correctness. Mention edge cases: empty lists, all duplicates, no common elements.

Question 1 — List Intersection (Java 8)

From your notes: ArrayList l1 = ["a","b","bc"] and l2 = ["cc","dd","aa"] — retrieve all from both lists. Also: list intersection using Java 8 streams.

🤔 Clarify First — What Does "Retrieve All From Both" Mean?

Before writing code, ask the interviewer which operation they want:

  • Intersection — elements that appear in BOTH lists: [1,2,3] ∩ [2,3,4] = [2,3]
  • Union — all unique elements from EITHER list: [1,2,3] ∪ [2,3,4] = [1,2,3,4]
  • Difference — elements in A but NOT in B: [1,2,3] - [2,3,4] = [1]

From your notes, the question asks for intersection — elements common to both lists. Always clarify this upfront in an interview.

Why the Naive Approach (Nested Loop) Is Wrong

The first instinct is: for each element in list A, loop through list B and check if it exists. This works, but:

For list A with n elements and list B with m elements, this does n × m comparisons. With n=10,000 and m=10,000 — that is 100,000,000 comparisons. Slow.

The key insight: List.contains() is O(n) — it scans the entire list for each lookup. But HashSet.contains() is O(1) — it uses a hash function to jump directly to the answer. Convert list B to a HashSet once, then check each element of list A against the set. Total: O(n+m) instead of O(n×m).

Question 2 — Minimum Merge Cost / Connect Sticks

From your notes: Array of integers [5,3,1,2] — find the lesser sum / minimum cost to connect all sticks. LeetCode 1167.

Problem Statement

Given an array of stick lengths, connect all sticks into one stick. The cost of connecting two sticks is the sum of their lengths. Return the minimum total cost to connect all sticks into one.

Example: sticks = [5, 3, 1, 2]
Connect 1+2=3, cost=3. Now sticks=[5,3,3].
Connect 3+3=6, cost=6. Now sticks=[5,6].
Connect 5+6=11, cost=11. Total cost = 3+6+11 = 20.

🤔 Why Does the Order of Merging Matter?

Every time you merge two sticks, the result gets merged again in future steps. A small stick that you merge late in the process gets added into every subsequent merge — it contributes to the total cost many times.

A small stick merged early contributes to fewer merges. Therefore: always merge the two smallest sticks first. This greedy choice guarantees the minimum total cost.

Intuition — Why Smallest First Is Always Optimal

Think of it as building a tree from the bottom up. Each merge is a level in the tree. A stick at depth 3 gets counted 3 times in the total cost (once in each merge level). You want the smallest sticks to be deepest in the tree — counted the most times, but since they are small the impact is minimal. Large sticks should be near the root — counted few times.

This is exactly how Huffman encoding works — the most frequent characters (analogous to smallest sticks) get shorter codes (fewer merge levels), minimising total bit usage.

✓
Answer: 20
✅ Greedy + Min Heap — O(n log n)
Always merge the two smallest. Use a PriorityQueue (min heap) to efficiently get the two smallest in O(log n). Total: O(n log n) for n-1 merges.
❌ Wrong: Merge Largest First
sticks=[5,3,1,2]: 5+3=8(cost 8), 8+2=10(cost 18), 10+1=11(cost 29). Total=29 vs optimal 20. Large values repeated in too many merges.

// ── Solution: Greedy + Priority Queue (Min Heap) ───── public int connectSticks(int[] sticks) { // PriorityQueue in Java is a MIN heap by default // poll() returns the SMALLEST element in O(log n) PriorityQueue<Integer> minHeap = new PriorityQueue<>(); // Add all sticks to the heap — O(n) for (int stick : sticks) { minHeap.offer(stick); } int totalCost = 0; // Keep merging until only one stick remains while (minHeap.size() > 1) { int first = minHeap.poll(); // smallest — O(log n) int second = minHeap.poll(); // 2nd smallest — O(log n) int merged = first + second; totalCost += merged; // this merge costs first+second minHeap.offer(merged); // put merged stick back — O(log n) } return totalCost; } // ── Test it ────────────────────────────────────────── // connectSticks([5,3,1,2]) → 20 ✓ // connectSticks([1,8,3,5]) → 30 ✓ (see walkthrough: 1+3=4, 4+5=9, 8+9=17 → 4+9+17=30) // connectSticks([1]) → 0 ✓ (only one stick — no merges needed) // connectSticks([3,3,3,3]) → 24 ✓ (3+3=6, 3+3=6, 6+6=12 → 6+6+12=24) // ── Why PriorityQueue is a Min Heap ────────────────── // Java's PriorityQueue uses natural ordering (Comparable) by default // For integers: smallest integer = highest priority // poll() always removes and returns the minimum element // For MAX heap — reverse the comparator: PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Collections.reverseOrder()); // or: new PriorityQueue<>((a,b) -> b - a); // ── Complexity Analysis ─────────────────────────────── // Building the heap: O(n) // n-1 iterations, each with 2 polls + 1 offer: // each poll/offer is O(log n) // so (n-1) * 3 * O(log n) = O(n log n) // Total: O(n log n) time, O(n) space for the heap
🔗 Connection to Huffman Encoding

This algorithm is identical to building a Huffman encoding tree — the algorithm used in ZIP file compression, JPEG compression, and many other places.

In Huffman encoding, you have characters with frequencies (how often they appear in text). Frequent characters get short binary codes. Rare characters get long codes. You build the code by always merging the two least-frequent characters — exactly like merging the two smallest sticks.

The total cost in our problem = the total number of bit operations in Huffman encoding. Minimising it gives you the most efficient compression. Same greedy algorithm, same proof of correctness, same O(n log n) complexity.

Follow-Up Questions & Full Answers
Q1: What is the time and space complexity of each list intersection approach? +

Let n = size of list A, m = size of list B.

Nested loop: For each of n elements in A, scan all m elements in B. Time: O(n×m). Space: O(n) for the result list. At n=m=10,000 this is 100,000,000 operations — unacceptably slow.

retainAll with ArrayList as argument: Internally, ArrayList.contains() is O(m). Called n times. Time: O(n×m). Just as slow as the nested loop despite being a single method call. The method name is misleading — always pass a Set to retainAll.

retainAll with HashSet as argument: HashSet.contains() is O(1) average. Called n times. Building the HashSet: O(m). Time: O(n+m). Space: O(n) for the copy + O(m) for the set = O(n+m). BUT it mutates the list — dangerous if you need the original.

Stream + HashSet: Building set: O(m). Streaming A with filter: n × O(1) = O(n). Collecting result: O(k) where k is the intersection size. Total time: O(n+m). Space: O(m) for the set + O(k) for the result. Best balance of performance and immutability.

Sort + Two Pointers: Sort both: O(n log n + m log m). Merge step: O(n+m). Total: O(n log n + m log m). Space: O(1) extra (if sorting in place). Use this when you cannot afford extra space for a HashSet but can afford to sort. Also useful when the lists are already sorted.

Q2: How would you handle null values in the lists? +

Null handling depends on the business requirements — always clarify with the interviewer whether nulls are valid inputs.

Option 1 — Filter out nulls before processing:

Set<Integer> setB = listB.stream().filter(Objects::nonNull).collect(toSet());

List<Integer> result = listA.stream().filter(x -> x != null && setB.contains(x)).collect(toList());

This is the safest approach — nulls are silently excluded from both sides.

Option 2 — Include nulls in intersection: A null in list A that also appears in list B should be in the result. HashSet handles null correctly — you can add null to a HashSet and contains(null) returns true if null is in the set.

Set<Integer> setB = new HashSet<>(listB); // null is allowed in HashSet

List<Integer> result = listA.stream().filter(setB::contains).collect(toList());

This works because HashSet.contains(null) does not throw — it returns true if null was added to the set.

Option 3 — Throw on null: If nulls should never appear: add a validation step at the start that throws IllegalArgumentException if either list contains null. Use Objects.requireNonNull() or a stream with .peek(x -> Objects.requireNonNull(x, "null values not allowed")).

Q3: How do you find elements in list A but NOT in list B (set difference)? +

Set difference is a common follow-up. The approach is identical to intersection but with a negated filter:

Set<Integer> setB = new HashSet<>(listB);

List<Integer> onlyInA = listA.stream().filter(x -> !setB.contains(x)).collect(toList());

Time: O(n+m). Space: O(m) for the set.

Symmetric difference (elements in A or B but NOT in both):

This requires two passes — once for elements in A not in B, once for elements in B not in A, then combine.

Set<Integer> setA = new HashSet<>(listA);

Set<Integer> setB = new HashSet<>(listB);

Q4: What if you can only merge 3 sticks at a time instead of 2? +

This is a great extension question that tests whether you understand the algorithm deeply.

The answer changes significantly. With 2-way merges, you can always merge the two smallest and it works out. With 3-way merges (merge 3 sticks at a time), there is an additional constraint: you can only perform a merge if you have at least 3 sticks. After each merge, you go from n sticks to n-2 sticks.

This means the algorithm only works if you can reduce to 1 stick via 3-way merges. Starting with n sticks, each merge reduces by 2. You need (n-1) to be divisible by 2. If (n-1) % 2 != 0, you cannot form valid 3-way merges with all sticks — some merge will have fewer than 3 sticks. In that case, you would pad with dummy zero-length sticks to make it work.

The algorithm itself is the same: use a min heap, poll 3 smallest at a time instead of 2, merge them, push the merged value back, add to total cost.

Generalised to k-way merges: each merge reduces stick count by k-1. You need (n-1) % (k-1) == 0. If not, pad with (k-1 - ((n-1)%(k-1))) zero-length sticks.

Q5: Can you solve connect sticks with a sorted array instead of a heap? What is the trade-off? +

Yes, but it requires careful thought about where to insert the merged value.

Attempt with simple sort: Sort the array, always merge the first two elements. After merging, you need to insert the merged value in the correct sorted position to maintain the sorted order for the next step. Inserting into a sorted array is O(n) (shifting elements). For n merges: O(n²) total. Worse than the heap approach.

Better approach — two queues: Sort the original array into Queue 1. Create empty Queue 2 for merged values. At each step, find the two smallest by comparing the fronts of both queues (each front access is O(1)). Merge them, push result into Queue 2. Since Queue 2 is always in sorted order (each merged value is larger than or equal to previous merged values), this gives O(n) merges each O(1) = O(n) merge phase after the O(n log n) initial sort. Total: O(n log n).

This two-queue approach achieves the same O(n log n) as the heap but avoids the heap overhead (no sifting). In practice the heap is simpler to code and equally fast for most inputs. The two-queue approach is worth knowing as a follow-up demonstration of algorithmic depth.

Q6: How do you prove the greedy algorithm is optimal? +

This is a harder follow-up for senior positions. The proof uses an exchange argument.

Claim: An optimal solution always merges the two smallest sticks first.

Proof by contradiction: Suppose we have an optimal solution that does NOT merge the two smallest sticks (call them a and b, where a ≤ b) at the first step. Instead, it merges some other pair (say c and d, where at least one of them is not the minimum). We will show we can replace the first merge with (a+b) without increasing the total cost.

In the optimal solution, at some point a and b are both merged into larger sticks. The cost of merging a is at least a (it contributes to at least one merge). The cost of merging b is at least b. If we merge them first, we "spend" (a+b) for their first merge, but then the merged value (a+b) participates in subsequent merges instead of a and b separately. Because a and b are the smallest values, (a+b) is ≤ any other merged value at the same level. Swapping the first merge to (a,b) either keeps the total cost the same or reduces it.

This exchange argument shows that deferring the merge of the two smallest can never improve the solution. Therefore, always merging the two smallest is optimal. This is the same proof used to show Huffman encoding is optimal.

Interview Tip — Talking Through Your Code

For the intersection question, always clarify first: "Before I code — are we looking for intersection (common elements), union (all elements), or difference? And should duplicates be preserved or deduplicated?"

Then explain your approach before typing: "I'll convert list B to a HashSet. That costs O(m) time and space, but each lookup drops from O(m) to O(1). Then I stream list A and filter using the set. Total time is O(n+m) vs O(n×m) for nested loops. For large lists the difference is enormous — at 10,000 elements each: 20,000 operations vs 100,000,000."

For the connect sticks question: "The greedy insight is that small sticks merged early contribute to fewer total merges — this minimises cost. I'll use a Java PriorityQueue which is a min heap — poll() gives the smallest in O(log n). The algorithm runs in O(n log n) — n-1 merges, each merge is O(log n) for the two polls and one offer."

Always finish with: "Let me verify with the example: [5,3,1,2] → merge 1+2=3 (cost 3), heap=[3,3,5] → merge 3+3=6 (cost 9), heap=[5,6] → merge 5+6=11 (cost 20). Answer: 20."
