Kafka — From Scratch to Production

Everything you need to know: what Kafka is, how it works internally, and how to configure it for production.

🤔 Why Does Kafka Exist? What Problem Does It Solve?

Before Kafka, large systems used traditional message queues (RabbitMQ, ActiveMQ). These worked but had critical limitations at scale:

  • Messages deleted after consumption: Once consumed, the message was gone. If a new service was added later, it could not read historical messages.
  • One message, one consumer: If two services needed the same message, you needed two separate queues with duplicate messages.
  • Limited throughput: Traditional queues struggled above tens of thousands of messages per second.
  • No replay: If a consumer had a bug and processed messages incorrectly, you could not re-process them — they were gone.

LinkedIn built Kafka in 2011 to handle their activity stream — billions of events per day. The core innovation: treat messages as an immutable, ordered log rather than a queue. Messages stay in the log for a configurable retention period. Any consumer can read from any point in the log at any time.

Formal Definition

Apache Kafka is a distributed event streaming platform. It stores events in an ordered, durable, replayable log. Producers append events to topics. Consumers read from topics at their own pace, tracking their position (offset) independently. Multiple consumer groups can each read the full topic independently without interfering with each other.

The Best Analogy — A Newspaper Archive

A newspaper publisher (producer) prints the daily paper. Every edition is stored in a library archive (Kafka topic). The archive keeps editions for 7 days (retention period).

Any subscriber (consumer) can come and read any edition, starting from today or going back to any day within the retention window. If a new subscriber joins today, they can still read every edition from the past week. Multiple subscribers read independently — one reading today's paper does not prevent another from reading yesterday's.

The library has multiple shelves (partitions) — the Tuesday edition is on shelf 1, Wednesday on shelf 2. You can read shelf 1 and shelf 2 simultaneously for speed. But articles in shelf 1 are always in the order they were published on that shelf.

Topics and Partitions — The Core Structure
What Is a Topic?

A topic is a named category of events. Think of it as a folder. order-events contains all events about orders. payment-events contains all events about payments. Services publish to topics. Services subscribe to topics.

A topic is split into one or more partitions. Each partition is a separate, ordered, append-only log. Messages within a partition are strictly ordered. Messages across partitions are not guaranteed to be ordered.

🔑 Message Keys — How Kafka Routes to Partitions

When a producer sends a message, it can include a message key. Kafka hashes the key to determine which partition receives the message. All messages with the same key always go to the same partition.

Why this matters for ordering: Messages within one partition are strictly ordered. If you use orderId as the message key, all events for order #123 (ORDER_CREATED, PAYMENT_DONE, ORDER_SHIPPED) will always land on the same partition — in order. The ShippingService consumer processing that partition sees them in the correct sequence.

Without a key: Messages are distributed round-robin across partitions. You get maximum throughput but no ordering guarantee — ORDER_SHIPPED might be processed before ORDER_CREATED by a consumer that only reads one partition.

Choosing a good key: Use the entity ID (orderId, userId, productId) as the key. This ensures all events for one entity are ordered correctly, while events for different entities (different orderIds) can be processed in parallel across partitions.

👥
Consumer Groups — The Killer Feature
🤔 What Is a Consumer Group and Why Does It Matter?

A consumer group is a set of consumers that cooperate to consume a topic. Within a group, each partition is assigned to exactly one consumer. This gives you parallel processing — 6 partitions processed by 6 consumers simultaneously.

The game-changing property: multiple independent consumer groups can each read the full topic. They do not share partitions. They do not share offsets. One group consuming messages has zero effect on another group.

Real Example — One Topic, Five Consumer Groups

Your order-placed topic receives every order event. Five different services each need to react:

  1. ShippingService — consumer group shipping-group — schedules delivery for each order
  2. EmailService — consumer group email-group — sends confirmation email
  3. InventoryService — consumer group inventory-group — reduces stock count
  4. AccountingService — consumer group accounting-group — records the revenue
  5. AnalyticsService — consumer group analytics-group — updates sales dashboard

All 5 groups independently read every order event. If AnalyticsService falls 10,000 messages behind (consumer lag), it has zero effect on any other service. Each service processes at its own speed, completely independently. This is impossible with a traditional queue — reading a message from a queue removes it. Kafka keeps it for all groups.

⚠️ Consumer Lag — The Most Important Kafka Metric

Consumer lag is the difference between the latest offset in a partition and the offset your consumer has processed. It tells you how far behind your consumer is.

If the order-placed topic has 100,000 messages and your shipping-group has processed 95,000, the consumer lag is 5,000.

Why it matters: Growing consumer lag means your consumer is falling behind the producer. This leads to:

  • Increasing latency — orders placed now will not be processed for minutes
  • Risk of exceeding retention period — if messages expire before being consumed, they are lost
  • Memory/disk pressure on Kafka brokers

How to fix high consumer lag: Add more consumer instances (up to the number of partitions — you cannot have more active consumers than partitions in a group). Or reduce processing time per message. Or increase partition count (requires careful planning — cannot easily be reduced).

How to monitor: Use kafka-consumer-groups.sh --describe CLI, or Confluent Control Center, or Burrow (LinkedIn's consumer lag monitoring tool).

Production Topic Configuration — Every Setting Explained
📋 The Three Most Important Production Decisions

Decision 1 — Partition count: Partitions determine parallelism. More partitions = more consumers can run simultaneously = higher throughput. But too many partitions have downsides: more file handles on brokers, longer leader election time if a broker fails, higher memory use by consumers tracking offsets. Rule of thumb: start with max(target_throughput / single_partition_throughput, number_of_consumer_instances). A single partition handles roughly 10MB/s or 5,000 simple messages/second.

Decision 2 — Replication factor: How many copies of each partition exist across brokers. Production minimum is 3 — this survives 2 broker failures. One copy is the leader (handles reads and writes). Others are followers (replicate from leader). If the leader crashes, a follower is promoted automatically.

Decision 3 — min.insync.replicas: The minimum number of replicas that must acknowledge a write for it to be considered successful. Set to 2 with replication factor 3. If only 1 replica is in sync (network issue), writes will fail rather than succeed and potentially lose data. Safety over availability.

📋 Log Compaction — A Special Retention Mode

Normal retention deletes messages after a time period (7 days). Log compaction is different — Kafka keeps only the most recent message per key, regardless of age.

Use case — materialised views: You publish a user profile update every time it changes. After 1,000 updates, you only care about the current state, not all the history. With compaction, Kafka keeps only the latest update per userId. Any consumer that joins later (or needs to rebuild its state) reads the compacted log and gets the current state of every user profile.

Use case — tombstone messages: To delete a record from the compacted log, publish a message with the key but a null value. Kafka treats null-value messages as tombstones — they indicate "delete this key." After compaction, the entry is removed entirely.

When NOT to use: Do not use compaction when you need full event history (event sourcing). Use compaction for current-state topics, not event-history topics.

Deployment Process for Microservices

From code push to production — the complete CI/CD pipeline and Kubernetes deployment strategies.

🤔 Why Deployment Strategy Is as Important as Code Quality

You can write perfect code but deploy it badly and cause more downtime than a bug would. Facebook's 6-hour outage in 2021 was caused by a configuration change pushed incorrectly — not bad code. Deployment strategy determines how your system behaves during the 5-minute window when old and new versions coexist side by side.

In microservices, deployment is more complex than a monolith because:

  • Services have independent deployment cycles — you cannot deploy everything atomically
  • API changes must be backward compatible during transition (old and new run simultaneously)
  • Database migrations must run before the new code that uses them
  • Service dependencies might call each other — version mismatches must be handled
🔄
The Complete CI/CD Pipeline
1
Code Push — Developer pushes to Git
Developer pushes a feature branch. Raises a pull request. Code review by team members. After approval, merged to main branch. The merge triggers the CI pipeline automatically.
2
CI — Build, Test, Scan
Automated pipeline runs: compile the code, run all unit and integration tests, run static analysis (SonarQube), run security vulnerability scan (Snyk/OWASP). If any step fails, the pipeline stops — no artifact is produced. This is the safety net that prevents bad code from progressing.
3
Docker Image Build
Build a Docker image from the Dockerfile. Tag it with the Git commit SHA (e.g. order-service:abc1234). Never use the :latest tag in production — it makes rollbacks impossible because you cannot tell which version :latest points to.
4
Push to Container Registry
Push the tagged image to a container registry (AWS ECR, GCP Artifact Registry, Docker Hub). Kubernetes will pull from here during deployment. Scan the image for vulnerabilities before pushing — use Trivy or AWS ECR native scanning.
5
Deploy to Staging — Helm or kubectl
Update the Kubernetes Deployment manifest with the new image tag. Apply via helm upgrade or kubectl apply. Helm charts package all Kubernetes resources (Deployment, Service, ConfigMap, HPA) into a versioned, reusable package. Kubernetes handles the rolling update.
6
Smoke Tests and Automated Checks
Run a suite of smoke tests against the staging environment. Check key user journeys work end-to-end. Monitor error rates and latency for 5 minutes after deployment. Auto-rollback if error rate exceeds threshold.
7
Production Deployment — Same Process
Same pipeline runs against production. May require manual approval gate. Production deployments should be boring — identical process to staging, same image tag, same automation. If production feels special and nerve-wracking, your deployment process needs improvement.
🎯
Kubernetes Deployment Strategies
Default
Rolling Update

Replace pods one by one. Old and new run simultaneously during rollout. Zero downtime if health checks configured.

Best for: Most deployments. Simple and safe.

Risk: Old and new APIs coexist — must be backward compatible.

Zero Risk
Blue-Green

Two identical environments. Switch traffic instantly. Instant rollback by switching back.

Best for: DB schema changes. High-risk releases.

Risk: Double infrastructure cost during transition.

Gradual
Canary

Route 5% of traffic to new version. Increase gradually as confidence grows. Auto-rollback on error.

Best for: High-risk features. A/B testing in production.

Risk: Requires traffic splitting infrastructure (Istio/Argo).

Before You Deploy — The Production Checklist

From your notes: "Before deploying a new microservice, do we need to do something?" Yes. Here is the complete checklist.

📝
Run Database Migrations First
Run Flyway or Liquibase migrations before deploying the new application version. Never rely on the app to run migrations on startup in production — if 3 pods start simultaneously, all 3 try to migrate at once, causing conflicts. Use Kubernetes init containers or a separate migration job that must complete before the Deployment starts.
🔑
Create Secrets and ConfigMaps Before Pods Start
Pods that reference a Secret or ConfigMap that does not exist yet will fail to start with CreateContainerConfigError. Apply all Secrets (DB passwords, API keys) and ConfigMaps before applying the Deployment. Use Kubernetes Secrets for sensitive values — never put passwords in environment variables in the Deployment YAML.
🏥
Configure Liveness and Readiness Probes
Without a readiness probe, Kubernetes sends traffic immediately when the pod starts — before Spring Boot has finished initialising (DB connection pool, caches). Add initialDelaySeconds appropriate for your startup time. Without a liveness probe, Kubernetes never restarts a deadlocked pod.
📦
Set Resource Requests and Limits
Without requests, the Kubernetes scheduler cannot make good placement decisions. Without limits, one pod with a memory leak can consume all node memory and cause other pods to be OOMKilled. Set both on every container. A good starting point: requests = 50% of limits.
🔄
Ensure Backward API Compatibility
During a rolling update, old and new pod versions run simultaneously. If you removed a field from your API response, old consumers calling the new pods will break. Follow the expand-contract pattern: first deploy the new version that still supports the old API (expand), then update all consumers, then remove the old API (contract).
📊
Set Up Monitoring and Alerting First
Deploy the new service with dashboards and alerts configured before it goes live. You should be able to see error rate, latency, and pod health the moment traffic starts. Discovering problems through customer complaints 30 minutes after deployment is too late.
🏷️
Register with Service Discovery
In Kubernetes, the Service object must be applied before the Deployment — otherwise pods start but have no Service routing traffic to them. Apply Service YAML first. In Eureka/Consul-based systems, verify the service registers on startup before routing traffic to it.

Follow-Up Questions & Full Answers

Questions interviewers ask about Kafka and deployment. Click to expand.

Q1: How do you decide how many partitions to create for a topic? Can you change it later? +

Partition count is one of the most consequential decisions you make for a Kafka topic and it deserves careful thought upfront.

How to calculate the right number:

  • From the consumer side: You cannot have more active consumers in a group than partitions. If you plan to scale ShippingService to 6 instances, you need at least 6 partitions. Start with your expected maximum consumer count.
  • From the throughput side: A single partition handles roughly 10MB/s write throughput. If your topic needs 100MB/s, you need at least 10 partitions.
  • Practical rule: Take the larger of these two numbers, then round up to a number that is convenient (divisible by the number of brokers for even distribution). Start slightly higher than you need — it is easier to start with 12 partitions than to increase later.

Can you change it? You can increase partitions but cannot decrease them without recreating the topic. The critical problem with increasing: if you are using a message key for ordering (e.g. orderId), adding partitions changes which partition each key maps to. A key that previously went to partition 2 might now go to partition 5. In-flight messages on partition 2 and new messages on partition 5 for the same orderId will be processed out of order. This breaks ordering guarantees. For this reason, plan partition count carefully upfront, especially for topics where message ordering matters.

Q2: What is the difference between acks=1 and acks=all? When would you use each? +

The acks setting controls how many broker acknowledgements the producer waits for before considering a message successfully written.

acks=0 (fire and forget): Producer sends the message and does not wait for any acknowledgement. Maximum throughput, but if the broker is down or the message is lost — you never know. Use only for non-critical metrics or telemetry where losing an occasional message is acceptable.

acks=1 (leader only): Producer waits for the partition leader to acknowledge the write. The leader writes to its local log and responds immediately without waiting for followers. Faster than acks=all, but if the leader crashes after acknowledging but before followers replicate, the message is lost. The leader failover will promote a follower that does not have the message.

acks=all (all in-sync replicas): Producer waits for ALL in-sync replicas to write the message to their local logs before acknowledging. Combined with min.insync.replicas=2, at least 2 replicas must have the message before success is returned. If the leader crashes now, any follower has the message and can serve it. No data loss. Slightly higher latency.

When to use each: Use acks=0 only for truly disposable data (click tracking, approximate metrics). Use acks=1 for important but not critical data where throughput matters more than guaranteed delivery. Use acks=all (the default with enable.idempotence=true) for all financial, order, payment, and other business-critical events. The latency difference is usually under 5ms — for most use cases, always use acks=all.

Q3: What happens during a Kafka consumer group rebalance? Why is it disruptive? +

A rebalance happens when the assignment of partitions to consumers in a group changes. This is triggered when: a consumer joins the group (new pod starts), a consumer leaves the group (pod stops), a consumer is considered dead (heartbeat timeout), or partition count changes.

During a rebalance: ALL consumers in the group stop processing. The group coordinator (a Kafka broker) reassigns all partitions across the current consumers. Once assignment is complete, consumers resume processing from where they left off. This stop-the-world behaviour is what makes rebalances disruptive — during the rebalance (which can take seconds to minutes), no messages are being processed.

Real impact: If you deploy a new version of ShippingService and do a rolling update (pods come up and down one by one), each pod start and stop triggers a rebalance. With 6 pods and 6 partitions, you might trigger 12 rebalances during a rolling deployment (6 old pods stopping + 6 new pods starting). At 10 seconds per rebalance, that is 2 minutes of zero processing.

Modern solution — cooperative rebalancing (Kafka 2.4+): Instead of stopping all consumers during rebalance, only the partitions that are being reassigned are paused. Consumers keep their existing partitions during the rebalance. This is called incremental cooperative rebalancing. Enable it with: partition.assignment.strategy=CooperativeStickyAssignor. This makes rolling deployments far less disruptive to Kafka processing.

Q4: What is a Helm chart and why is it used instead of raw YAML? +

A Helm chart is a package of Kubernetes YAML manifests with templating support. Think of it as npm for Kubernetes — a package manager for deploying applications.

The problem with raw YAML: You have 5 microservices. Each needs a Deployment, Service, ConfigMap, HorizontalPodAutoscaler, and Ingress. That is 25 YAML files. Most of them are nearly identical — only the service name, image tag, port, and a few config values differ. Maintaining 25 files means updating 5 files every time you deploy a new image tag.

What Helm adds:

  • Templating: Use variables like {{ .Values.image.tag }} that are filled in at deploy time. One template file generates the correct YAML for any service.
  • Values files: Separate configuration from templates. values-staging.yaml has staging-specific values (smaller replicas, different secrets). values-prod.yaml has production values. One chart, multiple environments.
  • Release management: Helm tracks what was deployed, with what values, at what time. helm rollback order-service 2 rolls back to the previous release. Atomic installs — if any resource fails to apply, Helm rolls back the entire release.
  • Dependency management: A chart can declare dependencies on other charts (e.g. your app depends on a PostgreSQL chart). Helm installs them all.

When NOT to use Helm: For very simple single-service deployments, raw kubectl apply is simpler. For large teams with complex GitOps workflows, tools like ArgoCD or Flux manage Helm releases automatically from a Git repository — the Git state defines what is deployed.

Q5: How do you do zero-downtime database schema changes during a rolling deployment? +

This is one of the hardest problems in microservice deployments. During a rolling update, old pods and new pods run simultaneously and BOTH hit the same database. A schema change must work with both old and new application code simultaneously.

The expand-contract pattern (the safe way):

Phase 1 — Expand (backward compatible addition): Add the new column or table as nullable with no constraints. Deploy new application code that WRITES to both old and new columns/tables but only READS from old ones. All pods (old and new) work with the schema at this point.

Phase 2 — Migrate: Run a background job to backfill data from old column to new column for existing records. This can run while the application is live.

Phase 3 — Switch reads: Deploy application code that reads from the new column/table. Still writes to both for backward compatibility. All pods are now on this version.

Phase 4 — Contract (cleanup): Remove the old column from writes. Deploy code that only uses the new column. Only do this once 100% of pods are on the new version.

Phase 5 — Drop old column: The old column/table is now safe to drop from the database.

This is always multiple deployments spread over days or weeks, not one big bang. This is why database changes feel slower in microservices — correctness requires multiple steps.

Q6: What is the difference between OOMKilled and CrashLoopBackOff in Kubernetes? +

These are two of the most common pod failure states you will encounter in production.

OOMKilled (Out of Memory Killed): The pod exceeded its memory limit. The Linux kernel's OOM killer terminated the process. The exit code is 137 (128 + signal 9). This is Kubernetes enforcing your memory limits — the pod tried to use more memory than you allowed.

Common causes: Your memory limit is set too low for your workload. A memory leak — the application allocates memory but never frees it, growing over time. A sudden large load — processing a large file or large query result set spikes memory use.

How to fix: First, check if it is a genuine limit issue (increase the limit) or a leak (fix the code). Use JVM flags like -XX:MaxRAMPercentage=75.0 to ensure the JVM heap respects container limits. Monitor memory over time — a steady upward trend is a leak.

CrashLoopBackOff: The container keeps crashing and Kubernetes keeps restarting it. After each crash, Kubernetes waits longer before the next restart — 10s, 20s, 40s, 80s, 160s (exponential backoff). The status shows CrashLoopBackOff once the waiting time reaches 5 minutes.

Common causes: Application exception on startup (misconfigured bean, DB not reachable, missing environment variable, bad configuration value). Missing required secret — if a required Secret does not exist, Spring Boot fails to start. Liveness probe too aggressive — K8s kills the pod before Spring Boot finishes starting. Port conflict — another process is using the same port.

How to diagnose: kubectl logs pod-name --previous shows the logs from the previous (crashed) container. The stack trace there is your root cause. Also check kubectl describe pod pod-name for events and exit codes.

Interview Tip — Answering "How Do You Create a Kafka Topic in Production?"

Do not just say "kafka-topics.sh --create". Walk through the decisions:

Strong answer: "When creating a production Kafka topic, we consider three key settings. Partition count is set based on our expected consumer count — if we plan to run 6 ShippingService instances, we create 6 partitions so each instance gets one. Replication factor is always 3 in production — one leader and two followers, survives 2 broker failures. We set min.insync.replicas to 2 and acks=all on the producer to guarantee no data loss even if the leader crashes right after acknowledging a write. We also set the retention period based on our recovery window — 7 days is standard, enough time to replay events if a consumer has a bug and needs to reprocess."

Then mention monitoring: "After creating the topic, we monitor consumer lag for each consumer group using our Kafka monitoring dashboard. A growing lag is our early warning system that a service is falling behind."
