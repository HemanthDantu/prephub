Start Here — The Most Important Question

Before choosing REST, gRPC, or Kafka — you must answer one question first.

The Fundamental Decision

When Service A needs to talk to Service B, you must first decide: does Service A need an answer before it can continue?

If YES — use synchronous communication. Service A sends a request and WAITS for the response. Nothing happens until the answer comes back.

If NO — use asynchronous communication. Service A sends a message and IMMEDIATELY moves on. Service B processes it in its own time. Service A does not wait.

Getting this decision wrong is one of the most common architectural mistakes. Choosing synchronous when you should be async creates tight coupling and cascading failures. Choosing async when you need sync creates data consistency problems.

The Phone Call vs Text Message Analogy

Synchronous = Phone call. You call someone, you wait on the line while they find the answer, you get it instantly. If they do not pick up — you are stuck. Your whole conversation is blocked until they respond.

Asynchronous = Text message. You send the message and put your phone away. You do other things. They read it and reply when they are ready. If their phone is off — the message sits in their inbox and gets delivered when they come back online. You were never blocked.

🛒 Real Example — An E-Commerce Order Flow

When a customer clicks "Place Order", here is what must happen synchronously (customer is waiting, needs an answer):

  • Check stock — is the item actually available? Must know before confirming.
  • Validate payment method — does this card exist? Must know before charging.
  • Charge the card — must succeed before showing "order confirmed".
  • Reserve inventory — lock the stock so nobody else buys it simultaneously.

And what happens asynchronously (customer does NOT need to wait for these):

  • Send confirmation email — can arrive 2 seconds later, nobody minds.
  • Notify the warehouse — they have minutes to start picking the order.
  • Update recommendation engine — "user bought X" can be processed later.
  • Update loyalty points — can calculate in the background.
  • Log to analytics — real-time analytics can afford a few seconds lag.

This is why Amazon shows "Order Confirmed!" instantly — the async events happen in the background. The checkout only waits for the 4 synchronous steps.

REST over HTTP — The Most Common
What Is REST?

REST (Representational State Transfer) is a style of building APIs over HTTP. It uses standard HTTP methods to describe what action you want to perform on a resource.

  • GET /orders/123 — retrieve order 123. Reading, no side effects.
  • POST /orders — create a new order. Body contains order data.
  • PUT /orders/123 — replace order 123 entirely with new data.
  • PATCH /orders/123 — update part of order 123 (e.g. just the status).
  • DELETE /orders/123 — delete order 123.

Responses come back as JSON. Every developer, every framework, every language understands JSON. That universality is REST's biggest strength.

How Service A Calls Service B via REST in Spring Boot

Spring Boot provides WebClient (reactive, non-blocking — modern) and RestTemplate (blocking — older but simpler). WebClient is the current recommendation.

// WebClient — the modern way (reactive, non-blocking) @Service public class OrderService { private final WebClient webClient; public OrderService(WebClient.Builder builder) { // base URL — in production this comes from config, not hardcoded this.webClient = builder.baseUrl("http://inventory-service").build(); } public StockResponse checkStock(String productId) { return webClient .get() .uri("/api/stock/{id}", productId) // path variable .retrieve() .bodyToMono(StockResponse.class) // deserialise JSON to object .timeout(Duration.ofSeconds(3)) // ALWAYS set a timeout .block(); // wait for response (blocking) } } // Inventory Service — the endpoint being called @RestController @RequestMapping("/api/stock") public class StockController { @GetMapping("/{productId}") public StockResponse getStock(@PathVariable String productId) { return stockService.getStock(productId); } }
✅ REST — Advantages
  • Universal — every language and framework supports it
  • Human readable — easy to debug with curl or Postman
  • Stateless — each call is self-contained
  • Easy to document (Swagger/OpenAPI)
  • Works through firewalls and proxies
  • Simple mental model — request, response
❌ REST — Disadvantages
  • JSON serialisation overhead on every call
  • No strict contract — clients can break silently
  • Chattier than binary protocols
  • No native streaming support
  • Caller is blocked waiting for response
  • Creates runtime coupling — if Service B is down, Service A fails

gRPC — When Speed Matters Internally
What Is gRPC?

gRPC (Google Remote Procedure Call) is a high-performance communication framework created by Google. Instead of sending JSON text, it uses Protocol Buffers (Protobuf) — a binary format that is much smaller and faster to serialise and deserialise than JSON.

You define your API in a .proto file — a schema that both the client and server must follow. The protobuf compiler generates code for both sides. If the schema changes in a breaking way, compilation fails — you cannot silently break the contract like you can with REST.

How much faster? A typical JSON payload of 1KB serialises in ~50 microseconds. The same data as Protobuf serialises in ~5 microseconds and is typically 3-5x smaller. At 100,000 requests per second internally, this adds up to significant CPU and bandwidth savings.

When to Use gRPC vs REST
  • External-facing API (mobile app, web app, 3rd party): Use REST. Everyone understands JSON. Clients do not want to manage .proto files.
  • Internal service-to-service with high throughput: Use gRPC. Your payment service calling your fraud service 10,000 times per second — binary protocol pays off.
  • Real-time streaming: gRPC supports server streaming, client streaming, and bidirectional streaming. REST does not natively. Use gRPC for live dashboards, real-time price feeds, log tailing.
  • Polyglot teams: gRPC has code generators for Java, Go, Python, Node, C++, C#, Ruby. One .proto file generates consistent client/server code in all languages.

Kafka — Asynchronous Event-Driven Communication
What Is Kafka and How Does It Work?

Apache Kafka is a distributed event streaming platform. Think of it as a highly durable, ordered, replayable message bus. Services publish events to Kafka topics. Other services subscribe to those topics and process the events at their own pace.

Key concept — the Producer: The service that creates and publishes the event. It sends the event to Kafka and immediately moves on. It does not know or care who reads the event.

Key concept — the Topic: A named channel in Kafka. Like a folder. "order-events" topic contains all events related to orders. Topics are split into partitions for parallelism.

Key concept — the Consumer: A service that subscribes to a topic and processes events. Multiple services can subscribe to the same topic independently. Each gets every event — they do not share events.

Key concept — Consumer Group: Multiple instances of the same service form a consumer group. Kafka distributes partitions across the group — each instance gets a subset of partitions. This is how you scale consumers horizontally.

The Newspaper Publisher Analogy

A newspaper publisher (producer) prints the daily paper and puts it on newsstands (the Kafka topic). Anyone who wants to read it (consumers) buys a copy. The publisher does not know who reads it. Subscribers (multiple consumer groups) can each independently subscribe — the morning paper subscriber, the archive library, the recycling company — each gets their own copy and does their own thing with it.

If a subscriber misses yesterday's paper — Kafka can replay it. The newspaper remains on the shelf for 7 days (retention period). You can always go back and read what you missed.

// PRODUCER — OrderService publishes an event when order is placed @Service public class OrderService { private final KafkaTemplate<String, OrderEvent> kafkaTemplate; public Order placeOrder(OrderRequest request) { Order order = orderRepository.save(new Order(request)); // Publish event — fire and forget, no waiting OrderEvent event = new OrderEvent(order.getId(), order.getUserId(), order.getItems()); kafkaTemplate.send("order-placed", order.getId().toString(), event); // execution continues immediately — we do NOT wait for consumers return order; } } // CONSUMER — EmailService listens for ORDER_PLACED and sends confirmation @Service public class EmailNotificationConsumer { @KafkaListener(topics = "order-placed", groupId = "email-service-group") public void handleOrderPlaced(OrderEvent event, Acknowledgment ack) { try { emailService.sendConfirmation(event.getUserId(), event.getOrderId()); ack.acknowledge(); // commit offset ONLY after successful processing } catch (Exception e) { // do NOT acknowledge — Kafka will re-deliver this message log.error("Failed to send email for order {}", event.getOrderId(), e); } } } // ANOTHER CONSUMER — WarehouseService also listens to the SAME topic @KafkaListener(topics = "order-placed", groupId = "warehouse-service-group") public void handleOrderForWarehouse(OrderEvent event, Acknowledgment ack) { warehouseService.prepareShipment(event); ack.acknowledge(); }
Why Does Manual Offset Commit Matter?

Kafka tracks which messages each consumer group has processed using an offset — a number indicating how far through the topic the consumer has read. When you commit an offset, you tell Kafka "I have successfully processed up to this message."

Auto-commit (dangerous): Kafka auto-commits the offset a few seconds after the message was received — BEFORE your processing code has finished. If your service crashes after the commit but before processing completes, the message is considered processed but never actually was. Silent data loss.

Manual commit (safe): You call ack.acknowledge() ONLY after your processing code has successfully completed. If your service crashes before acknowledging, Kafka re-delivers the message to another consumer instance. You might process it twice (at-least-once) but you will never lose it. Handle duplicates with idempotency keys.

✅ Kafka — Advantages
  • Producer and consumer are completely decoupled
  • If consumer is down, messages wait durably — nothing is lost
  • Multiple independent consumers can read the same events
  • Messages can be replayed — reprocess old events with a new service
  • Scales horizontally — add more partitions, add more consumers
  • Absorbs traffic spikes — consumers process at their own pace
❌ Kafka — Disadvantages
  • Eventual consistency — consumers lag behind producers
  • No direct response — harder to know if processing succeeded
  • More infrastructure to operate (Kafka cluster, Zookeeper/KRaft)
  • Harder to debug — must trace events across services
  • Schema changes require careful coordination
  • Overkill for simple point-to-point messaging

K8s Service-to-Service Debugging — The "App Not Found" Problem

From your notes: "Deployed to Kubernetes, all pods are up and running, but another service calling yours gets App Not Found or URL Not Found." Here is the complete systematic debug guide.

🏗️ How K8s Service Discovery Works — The Foundation

In Kubernetes, services talk to each other using internal DNS — not IP addresses. When you create a Kubernetes Service object, the cluster DNS automatically registers a hostname.

The format is: service-name.namespace.svc.cluster.local

  • Within the same namespace: just use order-service — DNS resolves it automatically.
  • Across namespaces: must use the full name order-service.orders.svc.cluster.local

A Kubernetes Service is NOT the same as your application. It is a virtual load balancer inside the cluster. It has a selector that matches pod labels. When a pod matches the selector, the Service adds it to its endpoint list and starts sending traffic to it.

The Most Important Concept — Service vs Pod

Your pod can be Running but your Service has no endpoints. This is the most common cause of "app not found". The pod is alive but the Service cannot find it because the selector does not match the pod's labels. They must match exactly, including case sensitivity.

Service selector: app: order-service
Pod label: app: orderservice (missing the dash) ← Service cannot find this pod!

Follow this exact sequence when debugging. Do not skip steps — they build on each other.

1
Check Pod Status
First verify the pod is actually running AND ready. STATUS=Running is not enough — READY must show 1/1 (or 2/2 if you have a sidecar).
kubectl get pods -n your-namespace -o wide
🔴 If STATUS is CrashLoopBackOff → your app is crashing on startup. Check logs with kubectl logs pod-name --previous
🔴 If READY is 0/1 → pod is running but readiness probe is failing. Your app started but is not healthy yet.
✅ Expected: STATUS=Running, READY=1/1, RESTARTS=0 (or low)
2
Check if the Service Exists
Verify the Service object itself exists in the right namespace. Sometimes the Service YAML was never applied, or was applied to the wrong namespace.
kubectl get svc -n your-namespace
🔴 If service does not appear → the Service was never deployed. Apply the Service YAML first.
✅ Expected: your service name appears with a ClusterIP assigned and the correct port.
3
Check Endpoints — THE MOST REVEALING STEP
Endpoints show which pod IPs the Service is currently routing to. If this shows <none>, the Service has no healthy pods matching its selector. This is the most common root cause.
kubectl get endpoints order-service -n your-namespace
🔴 If output shows <none> → selector mismatch or pods are not ready. Go to step 4.
✅ Expected: shows pod IPs like 10.244.1.5:8080,10.244.2.3:8080
4
Check Label Mismatch — If Endpoints Shows <none>
Compare the Service's selector with the pod's actual labels. They must match exactly.
kubectl describe svc order-service -n your-namespace | grep Selector kubectl get pods -n your-namespace --show-labels
🔴 If Selector shows app=order-service but pod labels show app=orderservice → fix the mismatch in either the Service YAML or the Deployment YAML.
✅ The selector key=value must exactly match one or more labels on the pod.
5
Check Readiness Probe — If Pod is Running but Endpoints is Still <none>
Kubernetes only adds a pod to Service endpoints when the readiness probe passes. If your Spring Boot app takes 30 seconds to start but the readiness probe checks every 10 seconds with no initialDelaySeconds — K8s marks the pod as not ready before it finishes starting.
kubectl describe pod order-service-abc -n your-namespace
Look in the Events section at the bottom for "Readiness probe failed". Look at the Conditions section for Ready: False.
🔴 Readiness probe failing → check your /actuator/health/readiness endpoint. Is your DB connected? Is your app still starting up?
✅ Fix: add initialDelaySeconds: 30 to your readiness probe to give Spring Boot time to start.
6
Test DNS Resolution from Inside the Cluster
Run a debug pod inside the cluster and test if DNS resolves correctly. This eliminates whether the issue is DNS or the service itself.
kubectl run debug --image=busybox --restart=Never -n your-namespace -- sleep 3600 kubectl exec -it debug -n your-namespace -- nslookup order-service
🔴 nslookup fails → DNS resolution is broken. The Service may not exist or may be in a different namespace.
🔴 Calling from a different namespace → use full DNS: nslookup order-service.orders.svc.cluster.local
✅ nslookup returns an IP → DNS is working. Test the actual HTTP endpoint next.
7
Test HTTP Connectivity
From the debug pod, actually call the service endpoint.
kubectl exec -it debug -n your-namespace -- wget -qO- http://order-service/actuator/health
🔴 Connection refused → pod is running but not listening on the expected port. Check your spring.server.port matches the Service's targetPort.
🔴 404 Not Found → the endpoint path is wrong. Your app may have a context path configured.
✅ Returns {"status":"UP"} → the service is reachable. The problem is in the calling service's URL configuration.
8
Check NetworkPolicy — Often Overlooked
In clusters with network policies enabled, traffic between namespaces or even between pods in the same namespace may be blocked by default. A NetworkPolicy that says "only allow ingress from pods with label app=frontend" will silently block your new service.
kubectl get networkpolicies -n your-namespace kubectl describe networkpolicy policy-name -n your-namespace
🔴 A NetworkPolicy exists with restrictive ingress rules → add a rule to allow traffic from your calling service.
✅ No NetworkPolicies or a permissive policy → network is not the issue.

The most important thing: the Service selector must exactly match the Deployment's pod labels.

The Cascading Failure Problem — Why Sync Is Dangerous at Scale

Understanding why synchronous communication can bring down your entire system.

What Is a Cascading Failure?

Service A calls Service B synchronously. Service B is slow — it takes 10 seconds instead of 200ms. Service A's thread is blocked for 10 seconds waiting. Service A has 20 threads in its pool. After 20 simultaneous slow calls to B, all 20 threads are blocked. Service A now cannot process any new requests. Other services calling Service A get timeouts. THEY run out of threads too. Within minutes, services that had nothing to do with the original slowness are all down.

This is a cascading failure — one slow service brings down the entire system, like dominoes.

Real War Story — Amazon 2013

A single service experienced a memory leak. It started responding slowly. Services that depended on it kept waiting for responses with no timeout. Their thread pools filled up. They became slow. Services depending on THEM became slow. Within 20 minutes, dozens of services that had nothing to do with the original memory leak were also timing out.

The fix that came from this: Strict timeouts on every external call (never wait more than X seconds), circuit breakers (stop calling a slow service after N failures), and bulkheads (separate thread pools per downstream dependency — if one is full, others are unaffected).

Netflix built Hystrix specifically because of cascading failure events. Today Resilience4j is the modern replacement. These tools exist because of real production disasters like this.

The Rule — Always Set Timeouts

Every synchronous call to another service MUST have a timeout. Without a timeout, your thread waits forever. "Forever" is how cascading failures start.

Good timeout values: connect timeout = 1-2 seconds (how long to wait for the connection to be established). Read timeout = 3-10 seconds (how long to wait for the response once connected). These values depend on your SLA — if your API must respond in under 500ms, your internal calls must time out in under 400ms.

Q1: If ServiceA calls ServiceB synchronously and ServiceB is slow — what happens to ServiceA? +

This is the cascading failure question. Here is the exact sequence of what happens:

Step 1: ServiceB becomes slow (GC pause, DB slowdown, traffic spike, memory leak). Instead of responding in 200ms, it takes 10 seconds.

Step 2: ServiceA's thread is blocked for 10 seconds per call. Each slow call holds a thread for 10 seconds instead of 200ms — that is 50x more threads consumed for the same traffic.

Step 3: ServiceA has a thread pool of 20 threads. With normal 200ms response times, each thread handles 5 requests/second = 100 requests/second total. With 10-second response times, each thread handles 0.1 requests/second = 2 requests/second total. 50x less throughput from the same thread pool.

Step 4: All 20 threads are now blocked waiting for ServiceB. New requests to ServiceA queue up. After a few seconds, the queue is full. New requests get rejected with connection timeout.

Step 5: ServiceA appears to be down. Services that call ServiceA have the same problem. The failure cascades.

The prevention: Set a timeout of 2 seconds on calls to ServiceB. After 2 seconds with no response, release the thread. Add a circuit breaker — after 50% of calls timeout, stop calling ServiceB entirely and return a fallback. Add a bulkhead — put ServiceB calls in a dedicated thread pool of 5 threads, separate from the main pool. Even if all 5 are blocked waiting for ServiceB, the other 15 threads keep serving other requests.

Q2: How do you handle a schema change in a Kafka event without breaking consumers? +

Schema changes in Kafka events are one of the trickiest operational challenges in event-driven systems. The problem is that producers and consumers deploy independently. When the producer adds a new field, some consumers may still be running old code that does not know about it.

Strategy 1 — Backward-compatible changes only: Only ever ADD new optional fields to events. Never remove fields. Never rename fields. Never change a field's type. Old consumers simply ignore unknown fields (if using JSON with Jackson, this is the default behaviour with DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES = false). New consumers use the new fields. Both coexist safely.

Strategy 2 — Schema Registry (Confluent Schema Registry): A central registry stores all event schemas and their versions. Producers register their schema before publishing. Consumers validate incoming events against the registered schema. The registry enforces compatibility rules — it rejects a new schema if it breaks backward compatibility. This is the production-grade approach used by large Kafka deployments.

Strategy 3 — Event versioning: Include a version field in every event: {"version": "2", "data": {...}}. Consumers check the version and use version-specific deserialisers. This allows completely different event shapes across versions but requires maintaining multiple deserialisers.

Strategy 4 — Consumer-driven contracts: Before changing an event schema, check which consumers depend on it. Update all consumers to handle both old and new formats (making consumers backward compatible). Then update the producer. Then remove the old format handling from consumers. This "expand-contract" migration is the safest but most labour-intensive approach.

Q3: What is the difference between Kafka and RabbitMQ? When would you choose each? +

This is a very common interview question. They are both messaging systems but with fundamentally different designs and trade-offs.

Kafka — Log-based, pull model: Kafka stores messages in an immutable, ordered log. Messages are retained for a configurable period (days to weeks) regardless of whether they have been consumed. Consumers pull messages at their own pace. Multiple independent consumer groups can each read the full topic independently. Messages can be replayed — if you add a new service, it can read events from the beginning of time.

RabbitMQ — Queue-based, push model: RabbitMQ pushes messages to consumers. Once a consumer acknowledges a message, it is deleted (by default). No replay capability. Better suited to task queues where each message is consumed once by exactly one consumer. More complex routing options (exchanges, bindings, routing keys).

Choose Kafka when: High throughput (millions of messages/day), multiple independent consumers need the same events, you need replay capability (audit logs, event sourcing, rebuilding read models), you need long retention, or you want to decouple producers and consumers completely.

Choose RabbitMQ when: Task distribution (one job processed by one worker), complex routing rules (route based on message content), lower volume with simpler setup, priority queues (process urgent tasks first), or you need request-reply patterns.

Simple rule of thumb: If you are building event streaming where multiple services react to the same events — Kafka. If you are building a task queue where jobs are distributed to workers — RabbitMQ.

Q4: What is a Service Mesh and how does it differ from the application handling communication? +

A service mesh is an infrastructure layer that handles service-to-service communication transparently, without any application code changes. The most popular is Istio.

Without a service mesh, your application code handles retries, timeouts, circuit breaking, TLS, and distributed tracing. Each service implements these separately, leading to inconsistency across services.

With a service mesh, a sidecar proxy (Envoy) is automatically injected into every pod. All inbound and outbound network traffic is intercepted by the sidecar. The sidecar handles retries, timeouts, circuit breaking, mTLS encryption, and tracing — without your application code knowing it exists.

Concrete difference: With application-level resilience (Resilience4j), each service must add the dependency, write annotations, configure retry policies, and maintain the configuration. If you want to change the retry policy for all services, you must update every service. With Istio, you update one Kubernetes CRD (VirtualService) and the change applies to all services instantly.

Trade-off: A service mesh adds operational complexity (Istio is notoriously complex to configure and debug), adds memory and CPU overhead (Envoy sidecar per pod), and adds latency (extra network hop through the proxy). Use it when you have many services and the consistency benefit outweighs the complexity cost. For teams with fewer than 10 microservices, application-level resilience is usually simpler and sufficient.

Q5: What is the liveness probe vs readiness probe in Kubernetes? Why do both exist? +

These are two separate health checks that serve completely different purposes. Confusing them is one of the most common Kubernetes mistakes.

Liveness probe — "Is this pod still alive? Should Kubernetes kill and restart it?"

The liveness probe checks whether the application is still functioning at a basic level. If it fails, Kubernetes kills the pod and restarts it. Use it to detect: deadlocks (the JVM is running but no threads can make progress), infinite loops consuming 100% CPU with no output, or any state where restarting would fix the problem.

Example: check if the JVM is responsive to a basic HTTP ping at /actuator/health/liveness.

Readiness probe — "Is this pod ready to receive traffic? Should Kubernetes include it in load balancing?"

The readiness probe checks whether the application is ready to handle requests. If it fails, Kubernetes removes the pod from the Service's endpoint list — no new requests are routed to it, but the pod is NOT killed. Use it to detect: the application is still starting up (DB connection pool not yet established, caches not yet warmed up), the application is temporarily overloaded and needs to shed load, or a dependency is temporarily unavailable.

Why both exist — the key scenario: Imagine your pod is running but its database connection pool is temporarily exhausted. The pod is alive (liveness passes) but it cannot handle new requests (readiness fails). Kubernetes should NOT kill and restart it — that would not fix the DB connection issue. It should just stop sending new requests to it until it recovers. Readiness handles this. Liveness would restart it unnecessarily.

In Spring Boot: /actuator/health/liveness returns UP as long as the JVM is running normally. /actuator/health/readiness returns UP only when all dependencies are connected and the app is ready to serve traffic.

Q6: How does service discovery work in Kubernetes? How does one service find another? +

Service discovery in Kubernetes is built into the platform via DNS. Unlike traditional systems where you maintain a service registry (Eureka, Consul), Kubernetes handles discovery automatically.

How it works step by step:

  • You create a Kubernetes Service object for your microservice.
  • The cluster's internal DNS server (CoreDNS) automatically creates a DNS record: service-name.namespace.svc.cluster.local
  • Any pod in the cluster can resolve this DNS name to the Service's ClusterIP.
  • The Service's ClusterIP is a virtual IP (not a real server). kube-proxy translates it to one of the matching pod IPs using iptables or IPVS rules.
  • The request is load-balanced across all healthy pods that match the Service selector.

Within the same namespace: Just use the service name. http://order-service/api/orders. DNS resolves order-service to the ClusterIP.

Across namespaces: Must include the namespace: http://order-service.orders.svc.cluster.local/api/orders

Client-side load balancing (Spring Cloud LoadBalancer): If you use Spring Cloud with Eureka or Consul, client-side load balancing is done in the application. The service registry tells the caller about all available instances, and the caller chooses one. In Kubernetes, server-side load balancing via the Service's kube-proxy rules is typically sufficient and simpler.

Interview Tip — How to Answer "How Do Services Communicate?"

Do not just say "we used REST". Walk through the decision:

Strong answer: "Our services communicate using a combination of approaches. For synchronous calls where the caller needs an immediate answer — like checking stock availability before confirming an order — we use REST via WebClient with a 3-second timeout and Resilience4j circuit breaker. For operations where the caller does not need to wait — like sending confirmation emails or notifying the warehouse — we publish events to Kafka topics. Each downstream service consumes events at its own pace, independently. This means if the email service is slow, it does not affect the checkout flow at all."

Then be ready for: "What happens if Kafka is down?" Answer: the producer call to send the event will fail. We handle this with the Outbox pattern — we first write the event to a DB table in the same transaction as the business data, then a relay process publishes to Kafka when it recovers. We never lose events.
