Why isolation exists: without it, transactions can quietly lie to each other

Isolation is about making concurrent work feel predictable. The big idea: multiple transactions can overlap in time, but your application still wants answers that make sense. This section gives you the anomaly vocabulary interviewers love to test.

Dirty read Non-repeatable read Phantom read Lost update Write skew
What you should be able to say after this section “Isolation levels are trade-offs between correctness and concurrency. Each level blocks specific anomalies, and the hard part is recognizing which anomaly matters for the business rule you are protecting.”
Section 1 of 5 · ~6 mins
🧒 ELI5 Isolation in kid language

Think of two kids editing the same sticker chart at once

Imagine you and your brother both grab the family chore sticker chart. One of you erases a star but has not decided yet. The other peeks mid-erase and believes the star is really gone. That is how apps get confused: they react to work that was not actually finished.

Isolation levels are like house rules for the chart. Weak rules say, “Anyone can peek anytime.” Stronger rules say, “Only look at finished updates,” or even, “Pretend people took turns one by one.”

Section 1 of 5 · ~6 mins
Technical Anomaly vocabulary you must know

The five anomalies that make isolation levels real

Dirty read: Transaction B reads a value written by transaction A before A commits. If A rolls back, B observed fiction.

Non-repeatable read: Transaction B reads the same row twice and gets different committed values because another transaction changed that row in between.

Phantom read: Transaction B reruns a predicate query like WHERE amount > 100 and sees a different set of rows because another transaction inserted or deleted matching rows.

Lost update: Two transactions read the same old value, both compute a new value, and one overwrites the other’s work.

Write skew: Two transactions read overlapping facts, each updates a different row, and together they break a cross-row rule even though no single row looks invalid.

Anomaly What changes? Interview memory hook Severity
Dirty read You read uncommitted data “I saw work that never became real.” Very risky
Non-repeatable read Same row, different committed value later “I reread one row and it changed.” Common
Phantom read Same query, different row set later “The population changed, not just one row.” Set-level issue
Lost update One write silently replaces another “Last writer wins by accident.” Business bug
Write skew Different rows updated, shared rule broken “Individually valid, jointly invalid.” Subtle

🧠 Quiz · Which anomaly is this?

A doctor-on-call app requires at least one doctor on duty. Transaction A sees Dr. Lee on duty and takes Dr. Kim off. Transaction B sees Dr. Kim on duty and takes Dr. Lee off. Both commit. Now nobody is on duty. What happened?

Correct answer: Write skew. Each transaction updated a different row, so there was no direct overwrite. The failure came from breaking a cross-row rule: “at least one doctor must remain on duty.”
Section 2 of 5 · ~6 mins
🧒 ELI5 Read Uncommitted vs Read Committed

Peeking at homework before your friend is done

Read Uncommitted is like copying your friend’s homework while they are still erasing and rewriting answers. You might copy something they later throw away.

Read Committed says you can only look after they put the pencil down and say, “Done.” That prevents the most embarrassing kind of mistake: believing half-finished work.

Section 2 of 5 · ~6 mins
Warning Important PostgreSQL nuance

PostgreSQL does not really expose dirty reads

In SQL theory, Read Uncommitted allows dirty reads. In PostgreSQL specifically, requesting Read Uncommitted behaves like Read Committed, because PostgreSQL’s MVCC design does not expose uncommitted row versions the way people often imagine.

This is a great interview answer: “Read Uncommitted is the weakest standard level, but in PostgreSQL it maps to Read Committed.”

Section 2 of 5 · ~6 mins
🔗 ANALOGY Restaurant menu board

Only trust dishes after the chef says “send it”

If servers can read the kitchen board while dishes are still being crossed out, they can promise the wrong food to customers. Read Committed makes them wait for the chef’s final send-off. But the board can still change again before the server looks a second time.

🧠 Quiz · Read Committed intuition

Inside one transaction, you run the exact same SELECT twice. Another transaction commits a row change between those two statements. Under Read Committed, what can happen?

Correct answer: You can see a different committed value on the second read. Read Committed gives each statement its own fresh committed view, so repeated statements can observe newer committed data.
Section 3 of 5 · ~6 mins
🧒 ELI5 Repeatable Read

You take a photo of the room and keep using that photo

Repeatable Read is like walking into a toy room, snapping a photo, and deciding, “For this whole game, I will trust only this photo.” If someone moves a toy later, your game still uses the original photo. So when you look at the same toy twice, it has not changed for you.

Section 3 of 5 · ~6 mins
🔗 ANALOGY Frozen scoreboard snapshot

One stable scoreboard for the whole play review

A referee reviews an entire drive using the same frozen scoreboard image. Even if the stadium scoreboard changes later, the referee’s review does not. That is the key: one transaction-wide snapshot instead of one snapshot per statement.

Section 3 of 5 · ~6 mins
Technical Stable row reads, snapshot semantics

What Repeatable Read buys you

Repeatable Read is stronger than Read Committed because the transaction keeps a consistent snapshot for the whole transaction, not a fresh one per statement. That means rereading the same row gives the same visible version.

Standard shorthand: dirty reads and non-repeatable reads are blocked, but phantom reads may still exist.

PostgreSQL nuance: PostgreSQL’s Repeatable Read is stronger than the minimum SQL-standard promise and also avoids phantom reads, but it can still allow serialization anomalies such as write skew. That is exactly why Serializable still exists.

First query establishes snapshot
→

→

// JDBC example: Repeatable Read Connection conn = dataSource.getConnection(); conn.setAutoCommit(false); conn.setTransactionIsolation( Connection.TRANSACTION_REPEATABLE_READ ); // First read establishes a stable snapshot PreparedStatement ps = conn.prepareStatement( "select balance from account where id = ?" );
Section 3 of 5 · ~6 mins
Warning Stronger does not mean perfect

Repeatable Read can still fail cross-row rules

You may block simple reread surprises but still lose correctness on business rules involving multiple rows or predicates. Write skew is the classic trap: each transaction sees a stable snapshot, but the two snapshots together still permit a jointly invalid outcome.

Interview line “Repeatable Read is snapshot stability, not full serial equivalence.”
Section 3 of 5 · ~6 mins
Technical Phantom intuition

Row-level stability vs set-level stability

People often memorize the difference like this: non-repeatable read is about one row changing, while phantom read is about the set of rows matching a predicate changing. That is a good mental model even though some engines, including PostgreSQL, provide stronger guarantees than the minimum standard requires.

Question Think row or set? Example
Did one known row change? Row-level Employee #7 salary changed from 100k to 110k
Did the population matching my query change? Set-level Orders over $100 suddenly include 3 extra rows
🧠 Quiz · What is the best summary?

Which statement best describes Repeatable Read in an interview?

Correct answer: one stable transaction snapshot. That is the cleanest way to describe Repeatable Read. It is stronger than Read Committed, but it is not the same as full serial execution.
Section 4 of 5 · ~6 mins
🧒 ELI5 Serializable

Everyone lines up and pretends they took turns

Serializable is like kids taking turns at a toy box so neatly that, even if they were actually moving around at the same time, the final result looks exactly like they went one after another. No weird overlap stories are allowed.

Section 4 of 5 · ~6 mins
🔗 ANALOGY Bank teller queue

As if one teller finished each customer before the next began

Even if multiple tellers worked in parallel behind the scenes, Serializable guarantees the outcome is equivalent to some safe queue order. That is why it is the easiest level to reason about and one of the most expensive to sustain under load.

Section 4 of 5 · ~6 mins
Technical Full isolation

Transactions must appear sequential

Serializable is the strongest standard isolation level. The database must ensure the final committed outcome is consistent with some one-by-one ordering of the transactions. That blocks dirty reads, non-repeatable reads, phantom reads, and deeper serialization anomalies like write skew.

The catch: stronger guarantees usually mean more coordination, more aborts/retries, less concurrency, or higher latency. That is why engineers do not blindly use Serializable everywhere.

Section 4 of 5 · ~6 mins
Technical When it shines

Best fit scenarios

  • Money movement or inventory invariants
  • Cross-row rules that must never drift
  • Scheduling/booking systems with strict availability guarantees
  • Complex reporting transactions that must reflect one coherent world
Section 4 of 5 · ~6 mins
Warning “Expensive” does not only mean CPU

Why Serializable costs more

Cost can show up as contention, waiting, reduced throughput, more deadlock/abort handling, or retry logic in the application. In PostgreSQL’s serializable mode, you should expect possible serialization failures and write code that can safely retry entire transactions.

🧠 Quiz · Serializable intuition

What is the best mental model for Serializable?

Correct answer: the result must look sequential. That is the essence of serializability. The engine may still execute work concurrently internally, but the committed outcome must be equivalent to a safe serial order.
Section 5 of 5 · ~6 mins
🧒 ELI5 Picking the right level

Choose the house rule that matches the game

If kids are just counting toys, a simple “only trust finished changes” rule may be enough. But if they are deciding whether the last toy can be borrowed, you may need the strictest rule. Isolation level choice is about what kind of mistake your business can and cannot tolerate.

Section 5 of 5 · ~6 mins
🔗 ANALOGY Door locks

Not every room needs the same lock

You do not put a bank vault lock on a pantry cabinet. But you also do not put a pantry latch on a vault. Read Committed is a practical everyday lock. Serializable is the vault lock for invariant-heavy workflows.

Section 5 of 5 · ~6 mins
Technical 30-second interview answer

Simple answer you can say aloud

“Isolation levels define how much concurrent transactions are allowed to interfere with each other. Read Uncommitted is the weakest and can allow dirty reads. Read Committed blocks dirty reads but the same query can return different results later. Repeatable Read gives a stable transaction snapshot, so rereading the same row stays consistent. Serializable is the strongest and makes the final result equivalent to running transactions one by one, but it usually costs more in coordination and retries.”

Section 5 of 5 · ~6 mins
Warning Memorization trap

Do not blindly memorize engine behavior as universal truth

Interviewers sometimes ask for the textbook SQL-standard ladder, and sometimes they ask for your database engine’s real behavior. Say both when relevant: “Standard says X; PostgreSQL specifically behaves like Y.” That answer sounds senior because it shows you know both theory and implementation nuance.

🧠 Quiz · Picking a level

You are implementing a booking workflow where subtle concurrent decisions must not violate availability rules. Which level is the safest default answer in an interview, assuming correctness matters more than throughput?

Correct answer: Serializable. For strict invariants and availability rules, Serializable is the cleanest isolation answer because it prevents anomalies that snapshot-based lower levels may still allow.

Here is the fast-scan cheat sheet. Read this once before an interview and you will sound much more deliberate when the conversation shifts from textbook definitions to real concurrency trade-offs.

Dirty read

  • You observed uncommitted data.
  • If the writer rolls back, your read was fake.
  • Main level associated with it: Read Uncommitted.

Non-repeatable read

  • Same row read twice, different committed value later.
  • Typical under Read Committed.

Phantom read

  • Same predicate query, different matching row set later.
  • Think set-level drift, not just one row drift.

Lost update

  • Two writers start from the same old value.
  • One write silently wipes out the other.

Write skew

  • Different rows updated, shared rule breaks.
  • Classic reason Serializable still matters.

Read Uncommitted

  • Weakest standard level.
  • Dirty reads possible in theory.
  • Almost never the right production answer.

Read Committed

  • Prevents dirty reads.
  • Each statement gets a fresh committed view.
  • Default in PostgreSQL.

Repeatable Read

  • Stable transaction snapshot.
  • Same row reread gives same result.
  • In PostgreSQL, stronger than many simplified textbook summaries.

Serializable

  • Strongest standard level.
  • Outcome must look like transactions ran one by one.
  • Expect coordination cost and possible retries.

Best interview framing

  • Start with anomaly names.
  • Then explain which level blocks what.
  • Finish with the trade-off: correctness vs concurrency.
