🔗 ANALOGY

Library catalog

What it means in real code

Definition: B-tree indexes support range queries and ordered lookups; hash indexes support equality.

Why it matters

  • Faster queries
  • Less I/O
  • Better performance

Common pitfall

  • Full table scans
  • Slow filters
  • Missing indexes on FK

Section 2 of 4 · ~7 mins

Composite and Covering Indexes

Composite index order matters; covering indexes can satisfy queries alone.

🧒 ELI5

Two-level index

You first look by last name, then first name.

🔗 ANALOGY

Phonebook order

Order determines how fast you find entries.

Technical

What it means in real code

Definition: Composite index uses column order; covering index includes all needed columns.

Why it matters

  • Faster multi-column filters
  • Avoid table lookups
  • Better performance

Common pitfall

  • Wrong column order
  • Index not used
  • Too wide indexes
Order mattersCoveringOver-indexing
Choose columns→Order by selectivity→Create index→Verify plan

Interview answer

“I order composite indexes by the most selective leading column.”

IndexBenefit
CompositeMulti filters
CoveringIndex-only scan
Composite index
CREATE INDEX idx_orders_user_status ON orders(user_id, status);
Warning

Column order changes usage

Queries must match the leading index columns to benefit.

Mini takeaway
  • Order columns carefully.
  • Covering indexes reduce lookups.
  • Avoid bloated indexes.
Quiz

Why does column order matter in a composite index?

Correct answer: B The leading column drives index usage.

EXPLAIN and Query Plans

EXPLAIN shows how the database executes a query.

🧒 ELI5

Route map

You check the map before driving.

🔗 ANALOGY

Factory blueprint

A blueprint shows how work flows.

Technical

What it means in real code

Definition: EXPLAIN reveals join types, scans, and index usage.

Why it matters

  • Detect slow paths
  • Validate indexes
  • Optimize queries

Common pitfall

  • Seq scans on large tables
  • Huge sort cost
  • Nested loops on big joins
PlanCostActual time
Run EXPLAIN→Interpret plan→Optimize→Re-test

Interview answer

“I use EXPLAIN ANALYZE to confirm indexes are used.”

Plan nodeMeaning
Seq ScanNo index
Index ScanIndex used
EXPLAIN ANALYZE
EXPLAIN ANALYZE
SELECT * FROM orders WHERE user_id = 42;
Warning

Read plans carefully

Costs are estimates; compare with actual time.

Mini takeaway
  • Use EXPLAIN ANALYZE.
  • Check index usage.
  • Fix large scans.
Quiz

What does Seq Scan indicate?

Correct answer: B Seq Scan means a full table scan.
Section 4 of 4 · ~7 mins

Index Maintenance

Indexes require upkeep and add write overhead.

🧒 ELI5

Clean your bookshelf

If you add too many books, it becomes hard to manage.

🔗 ANALOGY

Tool maintenance

More tools require more upkeep.

Technical

What it means in real code

Definition: Each index adds storage and write cost; remove unused indexes.

Why it matters

  • Keeps writes fast
  • Reduces storage
  • Avoids bloat

Common pitfall

  • Unused indexes
  • Slow inserts
  • Index bloat
Write costBloatCleanup
Check usage→Drop unused→Reindex if needed→Monitor

Interview answer

“I monitor index usage and remove unused ones to reduce write overhead.”

ActionWhy
Drop unusedLess overhead
ReindexRemove bloat
List indexes
SELECT indexname, idx_scan FROM pg_stat_user_indexes;
Warning

Too many indexes slow writes

Each write updates every index.

Mini takeaway
  • Monitor index usage.
  • Remove unused indexes.
  • Balance read/write.
Quiz

What is a downside of many indexes?

Correct answer: B Indexes add write overhead.
