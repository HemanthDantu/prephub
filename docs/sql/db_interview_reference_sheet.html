Keys & Relationships

4 questions

Primary key uniquely identifies each row, cannot be NULL, and there is only one primary key definition per table. It usually becomes the table’s main identifier.

Unique key also enforces uniqueness, but a table can have multiple unique constraints and databases generally allow NULL values depending on engine rules.

CREATE TABLE users (
  id INT PRIMARY KEY,
  email VARCHAR(255) UNIQUE
);

An index is mainly for faster lookup. A primary key is a constraint that guarantees unique row identity and usually creates an index behind the scenes.

Use an index to speed access patterns. Use a primary key to define the row’s canonical identifier.

A foreign key links one table to another and enforces referential integrity. It ensures a child row points to a valid parent row.

CREATE TABLE orders (
  id INT PRIMARY KEY,
  user_id INT,
  FOREIGN KEY (user_id) REFERENCES users(id)
);

A composite key uses multiple columns together to identify a row. It is common in join tables and many-to-many relationships.

CREATE TABLE enrollment (
  student_id INT,
  course_id INT,
  PRIMARY KEY (student_id, course_id)
);

Schema & Modeling

3 questions

Normalization reduces duplicate data and improves consistency by splitting data into related tables.

Denormalization intentionally duplicates some data to reduce joins and improve read performance.

Most production systems use a balance: normalize for correctness, denormalize selectively for read-heavy paths.

A view is a saved SQL query that behaves like a virtual table. It simplifies reuse, access control, and reporting queries.

CREATE VIEW active_users AS
SELECT id, email
FROM users
WHERE active = TRUE;

SQL databases use structured schemas, strong relational modeling, joins, and transactional guarantees.

NoSQL databases trade some relational structure for flexible schemas, horizontal scale, and simpler access for specific workloads.

Choose based on access patterns, consistency requirements, schema flexibility, and operational complexity.

Transactions & Consistency

7 questions

Atomicity: all operations in the transaction succeed or all are rolled back.

Consistency: data moves from one valid state to another valid state.

Isolation: concurrent transactions do not interfere in unsafe ways.

Durability: once committed, the data survives failures.

A transaction groups multiple database operations into one unit of work. It either commits fully or rolls back fully.

BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

DELETE removes selected rows and is typically logged row by row.

TRUNCATE removes all rows quickly and usually bypasses row-by-row logging.

DROP removes the table itself, including structure and data.

Use DELETE when you need row-level filtering, TRUNCATE for fast table reset, and DROP when the object should no longer exist.

Locks coordinate concurrent access so reads and writes do not corrupt data. Common lock scopes are row-level and table-level.

Lock contention becomes a performance problem when transactions stay open too long or many operations target the same rows.

A deadlock happens when two transactions each hold resources the other needs, so both wait indefinitely until the database aborts one of them.

Reduce deadlocks with short transactions, consistent lock order, and narrower update scope.

Isolation levels define how much one transaction can see another transaction’s changes.

READ UNCOMMITTED allows dirty reads. READ COMMITTED prevents dirty reads. REPEATABLE READ prevents non-repeatable reads. SERIALIZABLE gives the strongest isolation but has the highest concurrency cost.

Optimistic locking assumes conflicts are rare and detects them at update time, typically with a version column.

Pessimistic locking locks data before updating so others must wait.

-- Pessimistic locking example
SELECT *
FROM accounts
WHERE id = 1
FOR UPDATE;

SQL & Querying

7 questions

INNER JOIN returns only rows with matches on both sides.

LEFT JOIN returns all rows from the left table and matching rows from the right table; non-matches on the right come back as NULL.

SELECT u.id, o.id
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;

The N+1 problem happens when one query loads a parent list and then additional queries run for each row to load related data. That causes excessive round trips and poor performance.

// 1 query for users, then N queries for orders
List<User> users = userRepo.findAll();
for (User user : users) {
  user.getOrders();
}

Typical fixes are fetch joins, entity graphs, batch fetching, or DTO projections.

A query execution plan is the database optimizer’s chosen strategy for running a query. It shows whether the engine will use an index scan, full table scan, hash join, nested loop, sort, and more.

Use EXPLAIN or the database-specific plan tools to diagnose performance issues and verify whether your indexes are actually being used.

A full table scan happens when the optimizer decides reading every row is cheaper than using an index.

Common reasons: no usable index, low selectivity, function-wrapped predicates, or small tables where sequential scan is faster anyway.

Slow joins are commonly caused by missing indexes on join columns, large intermediate result sets, bad join order, stale statistics, or over-fetching too many columns.

Typical fixes: add the right indexes, reduce result shape, filter earlier, review join order, and inspect the execution plan.

Selectivity measures how well a column filters rows. High-selectivity columns like email are more useful for indexes than low-selectivity columns like status or gender.

If many rows share the same value, the optimizer may ignore the index and scan instead.

Having an index does not guarantee usage. The optimizer may skip it if the predicate is non-sargable, the leading column in a composite index is missing, selectivity is low, or the query returns a large portion of the table.

-- Index on email may not help here
SELECT *
FROM users
WHERE LOWER(email) = 'test@example.com';

Always confirm with an execution plan instead of assuming the index is helping.

Indexing & Storage

5 questions

An index is a separate data structure that helps the database locate rows faster without scanning the whole table.

Indexes are most useful for columns used in WHERE, JOIN, ORDER BY, and sometimes GROUP BY.

Indexes speed reads, but they slow down INSERT, UPDATE, and DELETE because each write must also maintain the index structure.

They also consume storage and can increase optimizer complexity. Good indexing is workload-specific, not blanket coverage.

A B-Tree is the most common index structure in relational databases. It keeps keys sorted in a balanced tree so lookups, inserts, deletes, and range scans remain efficient.

It works well for equality filters, prefix matches, sorting, and range queries like >, <, and BETWEEN.

Clustered index defines how table rows are physically ordered. Because data itself is stored in that order, a table can have only one clustered index.

Non-clustered index is a separate structure that stores indexed values plus a row locator pointing to the actual data. A table can have multiple non-clustered indexes.

Indexes hurt when the write cost outweighs the read benefit, such as on heavily updated columns, low-selectivity columns, very small tables, or tables with too many overlapping indexes.

They also hurt when query patterns prevent index use, such as applying functions to indexed columns or skipping the leading column in a composite index.

CREATE INDEX idx_users_email ON users(email);

Hibernate / JPA / Spring

8 questions

Lazy loading defers loading related entities until the relation is accessed.

Eager loading loads related entities immediately with the parent.

Lazy is usually safer as a default, but it must be managed carefully to avoid N+1 queries and session-boundary issues.

LazyInitializationException occurs when code tries to access a lazily loaded association after the Hibernate session is already closed.

Common fixes are fetch joins, DTO projections, explicit transaction boundaries, or loading required associations before leaving the service layer.

@Query("SELECT u FROM User u JOIN FETCH u.orders")
List<User> findUsersWithOrders();

Open-In-View keeps the persistence context open through the web layer, which can hide lazy loading problems and trigger unexpected queries during serialization or view rendering.

It often masks N+1 issues instead of forcing explicit data loading where it belongs.

DTOs let you return only the fields the client needs, which avoids exposing internal entity graphs, reduces over-fetching, and prevents lazy loading surprises at the API boundary.

They are also better for versioning and contract control.

In Spring, @Transactional is applied through proxies, so it typically works on public methods invoked through the Spring-managed bean, not on private self-invoked methods.

Wrong placement can silently result in no transaction being created at all.

@Service
public class PaymentService {
  @Transactional
  public void transfer() {
    // commit or rollback as one unit
  }
}

Batch inserts reduce database round trips by sending multiple insert statements together. That can significantly improve throughput for bulk writes.

In Hibernate, batching usually requires configuration and avoiding patterns that force flushes too often.

# Example Hibernate settings
spring.jpa.properties.hibernate.jdbc.batch_size=50
spring.jpa.properties.hibernate.order_inserts=true

Unexpected SQL usually comes from entity state transitions, cascade settings, flush timing, lazy associations, bidirectional relationships, and default fetching behavior.

Turn on SQL logging and inspect generated queries instead of assuming the ORM is doing what you intended.

First-level cache is session-scoped and built into Hibernate. Repeated reads of the same entity in one session can reuse it without another database hit.

Second-level cache is optional, shared across sessions, and useful for stable, read-heavy data when cache invalidation is managed carefully.
