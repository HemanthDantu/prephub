Atomicity — all or nothing

Atomicity means a transaction should feel indivisible: either every change in the unit of work becomes real, or none of it does. This is the property behind safe money transfers, inventory updates, and multi-step business operations.

🧒 ELI5

Imagine moving a Lego bridge from one table to another. You cannot leave half the bridge on one table and half on the other and call the move “done.” Either the whole bridge gets moved, or you put it back exactly where it started.

That is atomicity. A database transaction is saying: treat all these little changes like one single move.

🔗 ANALOGY

Think of online flight booking. One “booking” may involve:

  • reserve a seat
  • charge a card
  • create a ticket record

If the card charge fails, you do not want the seat reservation to remain stuck forever. The system should cancel the whole thing.

Atomicity at the engine level

What rollback means inside the database

The interview answer: Atomicity guarantees that a transaction’s changes are committed as a unit. On failure, the database rolls them back so partial effects do not remain.

Conceptually, databases keep enough information to undo or discard incomplete work. In traditional engine explanations, that is described as an undo log / rollback log. In MVCC systems, old row versions and transaction metadata also play a big role in making aborted work invisible and restoring a clean state.

BEGIN
→
Update A
→
Update B
→
Error happens
→
ROLLBACK
→
Original state preserved

Mental model: the database must never leave your business object halfway updated after a transaction aborts.

Java example · Spring transaction rollback

@Transactional
public void transfer(Long fromId, Long toId, BigDecimal amount) {
    accountRepository.debit(fromId, amount);
    accountRepository.credit(toId, amount);

    // Simulate a failure after one step
    if (amount.compareTo(new BigDecimal("10000")) > 0) {
        throw new IllegalStateException("Risk check failed");
    }
}

Because the method is transactional, the debit does not remain committed by itself when the exception is thrown.

⚠️ Warning

Atomicity only covers transactional database work. External side effects such as sending an email, calling another service, or publishing to a queue are not automatically rolled back unless you design for that explicitly.

Quiz — what is the strongest atomicity statement here?

A transaction updates 3 tables and crashes after 2 updates. Which statement best matches atomicity?

Correct answer: B. Atomicity is the all-or-nothing guarantee. The database must not leave business state partially applied after transaction failure.
Section 2 of 5 · ~6 mins

Consistency — rules stay true before and after

Consistency means every committed transaction must move the database from one valid state to another valid state. If a change would break declared rules, the database rejects it.

🧒 ELI5

Think of a playground with rules: no one can go down the slide backward, and only kids wearing socks can enter the trampoline area. A game can be fun and busy, but those rules still have to hold.

In a database, those playground rules are things like NOT NULL, FOREIGN KEY, UNIQUE, and CHECK.

🔗 ANALOGY

Imagine airport security. A boarding pass must match the passenger, the gate must exist, and the flight must still be open. If any rule fails, you do not “sort of” board the flight. You are stopped before the final step.

Consistency is constraint enforcement

Where the rules fire

Interview answer: Consistency means the database enforces integrity rules so committed data cannot violate defined constraints.

App sends INSERT / UPDATE
→
NOT NULL check
→
UNIQUE / PK check
→
FK validation
→
CHECK expressions
→
Commit only if valid
Constraint What it prevents Example
NOT NULL Missing required values email cannot be empty for a user
FOREIGN KEY Dangling references An order cannot point to a customer that does not exist
CHECK Invalid domain values balance >= 0
UNIQUE Duplicate business identifiers Only one row per email address

Java example · letting the DB enforce consistency

public void createOrder(Long customerId, BigDecimal total) {
    jdbcTemplate.update(
        "insert into orders(customer_id, total_amount) values (?, ?)",
        customerId,
        total
    );
}

If customer_id has a foreign key to customers(id) and that customer does not exist, the insert fails. The database stops invalid state from being committed.

⚠️ Warning

Consistency does not mean “your business logic is perfect.” It means the database protects the rules you declared. If a rule matters, encode it in constraints or transaction logic.

Quiz — which scenario is really about consistency?

Which example best shows the database enforcing consistency?

Correct answer: C. That is a classic foreign-key consistency check. The other options map to isolation, durability, and atomicity.
Section 3 of 5 · ~6 mins

Isolation — concurrent work should not see chaos

Isolation is the tricky one. It controls what one transaction can observe while other transactions are still running. The goal is to avoid nonsense results caused by overlapping reads and writes.

🧒 ELI5

Imagine two kids editing the same sticker chart. If one child is still moving stickers around, the other child should not peek in the middle and think the half-finished chart is the real final version.

Isolation decides when other people are allowed to see your changes.

🔗 ANALOGY

Think of a Google Doc with revision history. A weak isolation world is like reading a sentence while someone is halfway through deleting it. A stronger isolation world is like getting a stable snapshot of the document for your whole review.

The anomalies isolation tries to block

Three core things interviewers expect you to name

Anomaly Meaning Why it is bad
Dirty read Transaction A reads uncommitted data from transaction B A might act on data B later rolls back
Nonrepeatable read A reads the same row twice and sees different committed values Your logic may depend on a value staying stable during the transaction
Phantom read A reruns a predicate query and sees new or missing rows Set-based decisions can break

Concurrent timeline

Tx A
SELECT balance = 100 ...still running... SELECT balance again
Tx B
UPDATE balance = 70 COMMIT
Result
If Tx A now sees 70, that is a nonrepeatable read

This is why isolation levels exist: they define how stable a transaction’s view should be while other transactions are committing changes around it.

Java example · choosing an isolation level

@Transactional(isolation = Isolation.READ_COMMITTED)
public OrderSummary loadOrderSummary(Long orderId) {
    Order order = orderRepository.findById(orderId).orElseThrow();
    List<LineItem> lines = lineItemRepository.findByOrderId(orderId);
    return new OrderSummary(order, lines);
}

At READ_COMMITTED, each statement sees committed data as of that statement’s start. Two SELECTs inside one transaction can still see different values if another transaction commits in between.

⚠️ Warning

Isolation is not “the database being slow.” It is about visibility and correctness under concurrency. Stronger isolation can mean more waiting or retries, but the reason is correctness.

Quiz — identify the anomaly

Transaction A reads a row. Transaction B commits an update to that same row. Transaction A reads the same row again and gets a different value. What is that called?

Correct answer: A. A dirty read is reading uncommitted data. Here the second value came from another transaction that already committed, so this is a nonrepeatable read.
Section 4 of 5 · ~6 mins

Isolation levels in practice — what PostgreSQL gives you

Once you know the anomalies, the next interview step is mapping them to isolation levels. PostgreSQL is especially important because it uses MVCC snapshots, so some guarantees are stronger than what people casually assume.

Interview-ready comparison

PostgreSQL isolation levels, simplified

Level Dirty reads Nonrepeatable reads Phantoms Good mental model
Read Committed Blocked Possible Possible Fresh snapshot per statement
Repeatable Read Blocked Blocked Blocked in PG Stable snapshot for the transaction
Serializable Blocked Blocked Blocked Behave as if transactions ran one-by-one

Important PostgreSQL detail: asking for READ UNCOMMITTED behaves like READ COMMITTED, because PostgreSQL’s MVCC model does not expose dirty reads.

🔗 ANALOGY

Read Committed is like taking a new photo every time you look.

Repeatable Read is like keeping one frozen photo for the whole meeting.

Serializable is like having a referee ensure the final result matches some one-at-a-time order, even if that means asking one player to retry.

Row locks still matter

MVCC helps readers and writers coexist, but it does not eliminate locking. Commands such as SELECT ... FOR UPDATE take row locks so concurrent writers do not modify the same row at the same time.

SELECT ... FOR UPDATE
→
Row lock acquired
→
Other writer waits
→
Commit / rollback releases lock

Java example · strong isolation plus retry

public void reserveInventoryWithRetry(Long skuId, int qty) {
    for (int attempt = 1; attempt <= 3; attempt++) {
        try {
            inventoryService.reserveSerializable(skuId, qty);
            return;
        } catch (CannotSerializeTransactionException ex) {
            // retry whole transaction
        }
    }
    throw new IllegalStateException("Could not reserve inventory safely");
}

@Transactional(isolation = Isolation.SERIALIZABLE)
public void reserveSerializable(Long skuId, int qty) {
    // read current stock, validate, update
}

Serializable gives the strongest correctness story, but your app must be ready to retry transactions that fail due to serialization conflicts.

⚠️ Warning

Higher isolation is not always “better.” It can reduce concurrency or produce retry errors. Pick the weakest level that still protects your business invariant.

Quiz — best PostgreSQL mental model

Which statement is correct for PostgreSQL?

Correct answer: D. PostgreSQL does not implement a distinct dirty-read mode, and Serializable can reject some transactions so the application can retry safely.
Section 5 of 5 · ~6 mins

Durability — committed means it survives a crash

Durability answers the scary question: after the database says “commit successful,” what if power dies immediately? In PostgreSQL, the answer is built around WAL — Write-Ahead Logging.

🧒 ELI5

Imagine you are cleaning your room, but before moving each toy you first write the move into a notebook. If the lights go out halfway, you can reopen the notebook and replay the moves you promised were done.

That notebook is the basic idea of WAL.

🔗 ANALOGY

A theater crew keeps the official script in a binder backstage. The set pieces on stage can be rearranged later, but the script is the source of truth. If something crashes mid-performance, the crew uses the binder to rebuild the right scene.

PostgreSQL durability model

How WAL guarantees durability

Interview answer: PostgreSQL guarantees durability by writing change records to the WAL and flushing those WAL records to durable storage before reporting commit success. After a crash, it replays WAL from the last checkpoint to recover committed work.

Transaction changes buffers
→
WAL record generated
→
WAL flushed to disk
→
COMMIT success returned
→
Data pages can flush later
Normal path
Write WAL fsync WAL reply COMMIT OK
Crash later
Read last checkpoint Replay WAL forward Restore committed state

Java example · commit boundary

Connection con = dataSource.getConnection();
con.setAutoCommit(false);

try {
    // multiple SQL statements
    con.commit();   // success returned only after DB commit rules are satisfied
} catch (Exception ex) {
    con.rollback();
    throw ex;
}

Application code does not manually manage WAL. The engine does it under the hood so that a successful commit survives a crash.

⚠️ Warning

If PostgreSQL is configured for asynchronous commit, the server can return success before the latest WAL is fully flushed. That improves latency, but the newest transactions can be lost in a crash.

💡 Extra detail worth saying in interviews

  • WAL is stored in PostgreSQL’s pg_wal area.
  • Crash recovery replays WAL entries from the latest checkpoint forward.
  • PostgreSQL also uses full-page writes to protect against torn page writes during crashes.
  • Durability is about surviving crashes after commit acknowledgement, not merely after the SQL statement ran in memory.

Quiz — what does WAL really buy you?

Which option best explains PostgreSQL durability?

Correct answer: C. WAL is the crash-recovery backbone. The log is forced out first, and recovery replays it to rebuild committed state after failure.

Module Complete · ACID cheat sheet

You now have the interview version of ACID in your head

Use this page as your 2-minute review right before an interview or system design round.

Fast recap
  • Atomicity = all-or-nothing transaction unit. On failure, partial work is rolled back.
  • Consistency = the database enforces rules like NOT NULL, FK, UNIQUE, and CHECK so only valid states are committed.
  • Isolation = controls what concurrent transactions can see, preventing anomalies like dirty reads, nonrepeatable reads, and phantoms.
  • Durability = once commit succeeds, data survives crashes.

Atomicity

  • Business unit must not split into partial commits.
  • Rollback cancels the whole transaction.
  • Classic explanation: undo / rollback log.

Consistency

  • Moves DB from one valid state to another.
  • Constraints are the key mechanism.
  • Consistency is not the same thing as “bug-free app logic.”

Isolation

  • Read Committed = fresh snapshot per statement.
  • Repeatable Read = stable snapshot for transaction.
  • Serializable = strongest guarantee, may require retries.

Durability + WAL

  • Write-ahead means log first, data page later.
  • Commit success normally waits for WAL flush.
  • Crash recovery replays WAL from the last checkpoint.
One clean interview answer

“ACID is the set of transaction guarantees that make database operations reliable. Atomicity gives all-or-nothing execution, Consistency enforces integrity rules, Isolation controls what concurrent transactions can observe, and Durability ensures a committed transaction survives crashes. In PostgreSQL, durability is provided through write-ahead logging, where WAL records are flushed before commit is acknowledged and replayed during crash recovery.”
