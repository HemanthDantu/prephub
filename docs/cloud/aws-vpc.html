Networking Basics — Start Here

Before VPCs make sense, you need a clear mental model of IP addresses, networks, and why isolation matters. This section assumes you know nothing about networking.

What is an IP Address?

Every device connected to a network — your laptop, a server, a phone — has an IP address (Internet Protocol address). Think of it like a postal address for a machine. When your browser requests a webpage, it sends a packet to the server's IP address. The server sends a response back to your IP address.

An IPv4 address looks like this: 192.168.1.105 — four numbers separated by dots, each between 0 and 255. Under the hood, it's a 32-bit binary number:

# IP Address: 192.168.1.105
192  .  168  .   1   .  105
↓        ↓       ↓       ↓
11000000.10101000.00000001.01101001  (binary — how computers see it)

# IP addresses range from 0.0.0.0 to 255.255.255.255
# That's 4,294,967,296 (≈4.3 billion) possible addresses

Public vs Private IP Addresses

Not all IP addresses can be reached from the internet. There are two types:

Public IP addresses
Globally unique — routable on the internet. Any machine anywhere in the world can send a packet to a public IP and it will arrive.

Examples: 8.8.8.8 (Google DNS), 54.239.28.85 (Amazon)

Your home router has a public IP. Web servers have public IPs.
Private IP addresses (RFC 1918)
Reserved for internal networks. NOT routable on the public internet. Millions of private networks can all use the same private IP ranges without conflict.

Reserved private ranges:
10.0.0.0 – 10.255.255.255 (10.x.x.x)
172.16.0.0 – 172.31.255.255
192.168.0.0 – 192.168.255.255
Analogy — Office Building
Public vs Private addresses are like building addresses vs room numbers

Your office building has one public address: "100 Main Street, New York" — anyone in the world can send mail here. Inside the building, rooms are numbered 101, 102, 103. Room numbers are private — you can't mail a letter to "Room 103" without specifying the building. Similarly, your EC2 instance might have private IP 10.0.1.45 (internal) and public IP 54.x.x.x (internet-facing).

Why Do We Need Networks at All?

A network groups machines together so they can communicate directly (fast, private, cheap) while remaining isolated from machines outside the network. This matters for three reasons:

Security
Your database should NOT be accessible from the internet. By putting it on a private network, only machines inside that network (like your app servers) can connect to it. The internet can't even see it exists.
Performance
Traffic between machines on the same network stays local — low latency (< 1ms), high bandwidth. Your app server calling your database over a private network is orders of magnitude faster than going over the internet.
Cost
Data transfer between machines in the same AWS availability zone is free. Data leaving AWS (to the internet) costs money per GB. Keeping traffic private saves significant money at scale.

What Happens When a Packet Travels

Understanding packet routing is the foundation for everything in this module. When your EC2 instance sends a request, here's what actually happens:

Example: EC2 instance calls an external API (api.stripe.com)
1
EC2 creates a packet: Source IP = 10.0.1.45 (EC2's private IP), Destination IP = 54.187.174.169 (Stripe's IP). Sends it to its default gateway.
2
Route table consulted: EC2's subnet has a route table. The packet's destination (54.x.x.x) doesn't match any local route, so it matches the default route 0.0.0.0/0 → NAT Gateway.
3
NAT Gateway translates: Replaces source IP 10.0.1.45 with its own public IP 52.22.x.x. Remembers this translation. Forwards to Internet Gateway.
4
Internet Gateway routes: Packet leaves AWS and reaches Stripe's servers at 54.187.174.169.
5
Response returns: Stripe responds to 52.22.x.x. NAT Gateway looks up its translation table, replaces destination with 10.0.1.45, delivers to EC2.
Key networking insight: Every networking decision in AWS (VPC, subnets, security groups, NAT) is about controlling which packets are allowed to travel where. Once you understand packets and routes, VPC becomes completely logical rather than magical.

What is a VPC?

A Virtual Private Cloud is your own logically isolated section of the AWS cloud. It's your private data centre inside AWS — you control the networking completely.

The Problem Without a VPC

Imagine AWS was one giant flat network — all customers' servers in one shared space. Your EC2 instance and a random stranger's EC2 instance could reach each other directly. A database server would be on the same network as public web servers. This would be a security nightmare. AWS solves this with VPCs.

What a VPC Actually Is

A VPC (Virtual Private Cloud) is a logically isolated virtual network that you define within an AWS Region. When you create a VPC, you get your own private IP address space, your own route tables, your own gateways, your own security rules. Other AWS customers cannot see or reach resources inside your VPC — even though you're all on the same physical hardware.

Analogy
VPC is like buying a plot of land in a city

AWS is the city. When you create a VPC, you're buying a plot of land. You choose its size (IP range). You decide what buildings go on it (EC2, RDS, Lambda). You build roads inside it (subnets and route tables). You decide if there's a gate to the outside world (Internet Gateway). Your neighbour's plot is completely separate — they can't drive onto your land unless you create a road connecting your plots (VPC Peering).

VPC — The Fundamental Properties

What you get when you create a VPC
CIDR block: The IP address range for your VPC, e.g. 10.0.0.0/16 — this gives you 65,536 private IP addresses to assign to resources inside.

Implicit router: Every VPC has a virtual router. All traffic between resources in the VPC goes through it. You control the router via route tables.

Default route table: Controls where packets go. By default, resources in the VPC can reach each other (local route). No internet access by default.

Default security group: Every VPC gets a default SG. Allows all outbound, allows inbound from the same SG. Replace with custom SGs in production.

DNS resolution: AWS provides a DNS server at the VPC base address + 2 (e.g., 10.0.0.2). Resolves AWS service endpoints and private hostnames automatically.

Default VPC — The One That Already Exists

When you create a new AWS account, AWS automatically creates a Default VPC in each region. This is so you can launch EC2 instances immediately without thinking about networking. The default VPC:

# Default VPC configuration (AWS creates this for you)
CIDR block:       172.31.0.0/16     # 65,536 IP addresses
Default subnet:   172.31.0.0/20     # one subnet per AZ, each /20 = 4,096 IPs
Internet Gateway: attached          # everything has internet access by default
Route table:      0.0.0.0/0 → IGW   # all subnets are public
DNS:              enabled

# Everything in the default VPC gets a public IP by default
# This is convenient for learning but BAD for production
# Production: create a custom VPC with private subnets
Default VPC is for learning only. In production, always create a custom VPC. The default VPC has every subnet public (internet-accessible) by default, no meaningful segmentation, and shared DNS namespaces. You'd never put a production database in the default VPC.

CIDR & IP Addressing

CIDR notation is how you define a range of IP addresses. It appears everywhere in AWS — VPC creation, subnet design, security group rules, route tables. You must be able to read it fluently.

What is CIDR?

CIDR stands for Classless Inter-Domain Routing. It's a compact way of writing a range of IP addresses using just two pieces of information: a starting IP address and a number that tells you how many addresses are in the range.

Format: IP_ADDRESS / PREFIX_LENGTH

The prefix length (the number after the slash) tells you how many bits of the IP address are "fixed" (the network part). The remaining bits can vary — giving you the range of host addresses.

Reading CIDR — The Mental Model

# CIDR: 10.0.0.0/16
# IP in binary: 00001010.00000000.00000000.00000000
# /16 means: first 16 bits are FIXED (the network)
#            last 16 bits are FREE (the hosts)

# Fixed:  00001010.00000000 = 10.0
# Free:   xxxxxxxx.xxxxxxxx = 0.0 to 255.255

# So 10.0.0.0/16 covers: 10.0.0.0 → 10.0.255.255 = 65,536 addresses
# Formula: 2^(32-prefix) = 2^(32-16) = 2^16 = 65,536

Designing Your VPC IP Scheme — Real Planning

Before creating subnets, plan your IP space carefully. You can't change a subnet's CIDR after creation. Here's how to think about it:

Subnets — Public vs Private

A subnet divides your VPC into smaller networks. The public vs private distinction is the most critical architectural decision you make — and it's all about routing, not magic keywords.

What is a Subnet?

A subnet (short for sub-network) is a subdivision of your VPC's IP range. You create subnets within your VPC, each assigned to a specific Availability Zone. Every EC2 instance, RDS database, Lambda function (in VPC) — everything gets placed in a specific subnet and receives an IP address from that subnet's range.

Analogy — Office Building Floors
Subnets are like floors in a building

Your VPC is the entire building. Subnets are the floors. Floor 1 (10.0.1.0/24) might be the public-facing reception area — visitors can come in. Floor 11 (10.0.11.0/24) is the private office space — only employees with a badge can access it. Floor 21 (10.0.21.0/24) is the secure data room — only certain employees, no visitors ever. The building address (VPC CIDR) is shared, but the floors are separate spaces.

The Critical Truth: Public vs Private is Just a Route

This is the most important thing to understand about subnets. There is no "public" flag on a subnet. A subnet is public or private based entirely on one thing: whether its route table has a route to an Internet Gateway.

Public Subnet Route Table
Destination     Target
10.0.0.0/16     local        ← VPC-internal traffic
0.0.0.0/0       igw-0abc123  ← ALL other traffic → Internet Gateway

That 0.0.0.0/0 → igw-xxx line is what makes a subnet public. Resources here can send and receive traffic to/from the internet.

Private Subnet Route Table
Destination     Target
10.0.0.0/16     local        ← VPC-internal traffic only
0.0.0.0/0       nat-0def456  ← outbound only via NAT
(or no 0.0.0.0/0 at all)

No route to IGW = private subnet. Resources here are NOT reachable from the internet. They can reach out (via NAT) but nothing can reach in.

What Goes in Each Subnet Type

Creating Subnets

# Create public subnets (one per AZ)
aws ec2 create-subnet \
  --vpc-id vpc-0abc123def456789 \
  --cidr-block 10.0.1.0/24 \
  --availability-zone us-east-1a \
  --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=public-subnet-1a}]'

aws ec2 create-subnet \
  --vpc-id vpc-0abc123def456789 \
  --cidr-block 10.0.2.0/24 \
  --availability-zone us-east-1b \
  --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=public-subnet-1b}]'

# Auto-assign public IPs to instances launched in public subnets
# (so you don't have to manually request a public IP each time)
aws ec2 modify-subnet-attribute \
  --subnet-id subnet-0pub1a \
  --map-public-ip-on-launch

# Create private app subnets
aws ec2 create-subnet \
  --vpc-id vpc-0abc123def456789 \
  --cidr-block 10.0.11.0/24 \
  --availability-zone us-east-1a \
  --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=private-app-subnet-1a}]'

# Create private database subnets
aws ec2 create-subnet \
  --vpc-id vpc-0abc123def456789 \
  --cidr-block 10.0.21.0/24 \
  --availability-zone us-east-1a \
  --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=private-db-subnet-1a}]'

# RDS requires a "DB Subnet Group" — a named group of subnets in multiple AZs
aws rds create-db-subnet-group \
  --db-subnet-group-name production-db-subnet-group \
  --db-subnet-group-description "Production database subnets" \
  --subnet-ids subnet-0db1a subnet-0db1b subnet-0db1c
Three-tier subnet pattern — the standard for production: Public tier (ALB, NAT), App tier/Private (EC2, ECS, Lambda), Data tier/Private (RDS, ElastiCache). This creates defence in depth — even if an attacker breaches a public subnet, they hit another firewall before reaching the database.

Route Tables — The Traffic Rulebook

Route tables tell traffic where to go. Every packet leaving a subnet is matched against the route table — the most specific match wins. Understanding routes means understanding how traffic actually flows.

What is a Route Table?

A route table is a set of rules (routes) that determine where network traffic is directed. Every subnet must be associated with exactly one route table. When a packet leaves an EC2 instance in a subnet, the VPC router looks up the destination IP in the route table and sends the packet to the matching target.

Route Table Rules

Each route has two parts: Destination (what IP range does this rule cover?) and Target (where should I send matching traffic?). The router uses longest prefix match — the most specific route wins.

## Public Subnet Route Table — for web servers and ALB
Destination      Target           What this means
10.0.0.0/16      local            Traffic to ANY IP in the VPC → stay local (internal)
0.0.0.0/0        igw-0abc123      Everything else → Internet Gateway

## Why "local" wins over 0.0.0.0/0 for internal traffic:
# Longest prefix match: 10.0.0.0/16 is more specific than 0.0.0.0/0
# A packet to 10.0.11.45 (app server) matches 10.0.0.0/16 → local → stays in VPC
# A packet to 8.8.8.8 (Google) matches only 0.0.0.0/0 → IGW → internet

## Private App Subnet Route Table — for EC2 app servers
Destination      Target
10.0.0.0/16      local            All VPC traffic → local
0.0.0.0/0        nat-0def456      Outbound internet → NAT Gateway (NOT IGW)
# Notice: same 0.0.0.0/0 but target is NAT (not IGW)
# This means: can reach internet outbound, but NOT reachable from internet

## Private Database Subnet Route Table — for RDS, ElastiCache
Destination      Target
10.0.0.0/16      local            Only VPC-internal traffic allowed
# NO 0.0.0.0/0 route at all — databases should NEVER reach the internet
# If a DB packet tries to leave the VPC → no matching route → dropped

Why Longest Prefix Match Matters — A Concrete Example

Real Example — Traffic Decision
EC2 app server (10.0.11.45) sends a packet to 10.0.1.100 (the ALB)

Route table has: 10.0.0.0/16 → local and 0.0.0.0/0 → nat-xxx

The destination 10.0.1.100 matches BOTH routes: it falls within 10.0.0.0/16 AND within 0.0.0.0/0. Which wins? The more specific one wins — /16 is more specific than /0. Traffic stays local inside the VPC. The NAT Gateway is never involved.

Now the same EC2 sends a packet to 54.187.x.x (Stripe API). Only 0.0.0.0/0 matches. Traffic goes to NAT Gateway → internet.

Main Route Table vs Custom Route Tables

Every VPC has one main route table — subnets that don't have an explicit route table association use this one. In production, never use the main route table for anything important. Create explicit custom route tables for each tier.

# Create separate route tables for each tier
aws ec2 create-route-table \
  --vpc-id vpc-0abc123def456789 \
  --tag-specifications 'ResourceType=route-table,Tags=[{Key=Name,Value=public-rt}]'

# Add route to IGW for public subnets
aws ec2 create-route \
  --route-table-id rtb-0public123 \
  --destination-cidr-block 0.0.0.0/0 \
  --gateway-id igw-0abc123456

# Associate public subnets with this route table
aws ec2 associate-route-table \
  --route-table-id rtb-0public123 \
  --subnet-id subnet-0pub1a

aws ec2 associate-route-table \
  --route-table-id rtb-0public123 \
  --subnet-id subnet-0pub1b

# Create private route table (with NAT)
aws ec2 create-route-table \
  --vpc-id vpc-0abc123def456789 \
  --tag-specifications 'ResourceType=route-table,Tags=[{Key=Name,Value=private-app-rt-1a}]'

aws ec2 create-route \
  --route-table-id rtb-0private123 \
  --destination-cidr-block 0.0.0.0/0 \
  --nat-gateway-id nat-0def456789  # → NAT Gateway, not IGW

# Associate private app subnets
aws ec2 associate-route-table \
  --route-table-id rtb-0private123 \
  --subnet-id subnet-0app1a
One NAT Gateway per AZ. In production with 3 AZs, you need 3 separate NAT Gateways (one in each public subnet per AZ). Why? If your single NAT Gateway in 1a fails, private instances in 1b and 1c lose internet access. Each AZ's private subnets should have their own route table pointing to a NAT in the same AZ.

Internet Gateway & NAT Gateway

Two gateways, completely different purposes. IGW connects public subnets bidirectionally to the internet. NAT Gateway allows private subnets to reach out, but blocks the internet from reaching in.

Internet Gateway (IGW) — The Front Door

An Internet Gateway is a horizontally scaled, redundant, highly available VPC component that allows communication between your VPC and the internet. You attach one IGW to your VPC — there's no bandwidth limit, no availability risk.

The IGW does two things:

Outbound (EC2 → Internet)
For instances with public IPs: when a packet leaves with source IP 10.0.1.45 (private), IGW translates it to the instance's public IP (e.g., 54.x.x.x). This is called 1:1 NAT — each private IP maps to exactly one public IP.

The instance doesn't even know its own public IP — IGW manages the translation transparently.
Inbound (Internet → EC2)
When the internet sends traffic to 54.x.x.x (the public IP), IGW translates destination to 10.0.1.45 (private IP) and delivers to the EC2 instance in the public subnet.

This is why web servers can be reached from the internet — IGW bridges the public ↔ private IP translation. 
# Create and attach Internet Gateway — one per VPC
aws ec2 create-internet-gateway \
  --tag-specifications 'ResourceType=internet-gateway,Tags=[{Key=Name,Value=prod-igw}]'

# Attach to VPC (one IGW per VPC)
aws ec2 attach-internet-gateway \
  --internet-gateway-id igw-0abc123456 \
  --vpc-id vpc-0abc123def456789

# Now add the route to your public subnet route table:
aws ec2 create-route \
  --route-table-id rtb-0public123 \
  --destination-cidr-block 0.0.0.0/0 \
  --gateway-id igw-0abc123456

# For an EC2 instance to be internet-accessible, it ALSO needs:
# 1. A public IP (or Elastic IP) — assigned at launch or manually
# 2. The subnet must auto-assign public IPs (or assign manually)
# 3. A security group rule allowing inbound traffic on port 80/443/22
# 4. The subnet's route table must point to the IGW

NAT Gateway — The One-Way Door

NAT stands for Network Address Translation. A NAT Gateway sits in a public subnet and allows instances in private subnets to connect to the internet for outbound traffic (software updates, calling external APIs) — but prevents the internet from initiating connections into those private instances.

Why is this the right pattern for most servers?

Think about your app servers — they need to reach out: pull package updates, call Stripe APIs, send emails via SES, download S3 files. But they should NOT be directly reachable from the internet. Anyone on the internet should only be able to reach your load balancer (in the public subnet), not your app servers directly.

How NAT Gateway works — outbound from private subnet
1
Private EC2 (10.0.11.45) sends packet to api.stripe.com (54.x.x.x). Route table says: 0.0.0.0/0 → nat-0def456.
2
NAT Gateway receives the packet. Replaces source IP (10.0.11.45) with NAT Gateway's own Elastic IP (52.22.x.x). Records the translation in a connection table: "10.0.11.45:58234 → 52.22.x.x:58234".
3
Packet leaves via Internet Gateway to Stripe. Stripe sees the source as 52.22.x.x (NAT's public IP) — never knows 10.0.11.45 exists.
4
Stripe responds to 52.22.x.x. NAT Gateway looks up connection table, translates destination back to 10.0.11.45:58234, delivers to the EC2 instance.
5
Unsolicited traffic from internet trying to reach 52.22.x.x? NAT Gateway checks its connection table — no existing connection, packet dropped. Private EC2 is protected.
# Create NAT Gateway — must be in a PUBLIC subnet, needs an Elastic IP

# First, allocate an Elastic IP (public IP that you own)
aws ec2 allocate-address --domain vpc
# Returns: AllocationId=eipalloc-0abc123, PublicIp=52.22.x.x

# Create NAT Gateway in the PUBLIC subnet (not private!)
aws ec2 create-nat-gateway \
  --subnet-id subnet-0pub1a \             # PUBLIC subnet in us-east-1a
  --allocation-id eipalloc-0abc123 \     # The Elastic IP
  --tag-specifications 'ResourceType=nat-gateway,Tags=[{Key=Name,Value=nat-gw-1a}]'

# NAT Gateway takes a few minutes to become Available
aws ec2 describe-nat-gateways --filter Name=tag:Name,Values=nat-gw-1a

# Add route in private subnet route table pointing to NAT
aws ec2 create-route \
  --route-table-id rtb-0private-app-1a \
  --destination-cidr-block 0.0.0.0/0 \
  --nat-gateway-id nat-0abc123def456     # NOT igw — nat!

# Repeat for each AZ: NAT-1b in public-subnet-1b, NAT-1c in public-subnet-1c
# Each AZ's private route table points to its own NAT Gateway

Security Groups — The Instance Firewall

Security groups are virtual firewalls attached to individual EC2 instances, RDS databases, Lambda functions in VPC — any resource that gets a network interface. They are stateful — the most important word for the exam and real world.

What is a Security Group?

A Security Group (SG) acts as a virtual firewall for your instances. It controls inbound and outbound traffic at the instance level (technically, the Elastic Network Interface level). Every EC2 instance must have at least one security group. If you don't specify one, the default SG is used.

Key properties you must understand:

Stateful
This is the most important property. If you allow inbound traffic from IP X on port 80, the return traffic (response packets) is automatically allowed — even if there's no explicit outbound rule for it.

The SG remembers connection state. You only need to write the rule for the initiating direction.
Allow only — no deny rules
Security groups only have Allow rules. You cannot explicitly deny traffic in a security group. Everything is denied by default — you only whitelist what's allowed.

This means: if you want to block a specific IP that another rule would allow, you can't do it with a SG alone — use a NACL.

Default Security Group Behaviour

# Default SG (the one AWS creates for every new VPC)
Inbound:
  All traffic  |  All  |  All  |  sg-default (source = the SG itself)
  → Allows traffic FROM instances that are also in this same SG

Outbound:
  All traffic  |  All  |  All  |  0.0.0.0/0
  → Allows ALL outbound traffic

# What this means in practice:
# If EC2-A and EC2-B both have the default SG, they can talk to each other
# But nothing from the internet can reach them
# They can reach the internet (outbound is open)

# NEVER use the default SG in production
# Create specific SGs for each tier with exact rules needed

NACLs — Network Access Control Lists

NACLs are subnet-level firewalls. Unlike security groups, they are stateless, have numbered rules evaluated in order, and can explicitly deny traffic. They're the second layer of network defence.

What is a NACL?

A Network Access Control List (NACL) is a numbered list of rules that control traffic in and out of a subnet. Every subnet must be associated with one NACL. The default NACL allows all traffic — you only add rules when you need restrictions.

NACLs operate at the subnet level, not the instance level. An NACL rule filters ALL traffic entering or leaving a subnet, regardless of which instance it's going to. Security groups filter traffic per-instance. Both layers work together.

NACL Rules — Numbered and Ordered

NACL rules are evaluated in order from lowest to highest number. The first matching rule is applied. Rule 100 is evaluated before rule 200 before rule 32766. There is always a final * rule that denies everything not matched.

## Example NACL for a public subnet (inbound rules)
Rule#   Type      Protocol  Port Range  Source           Action
100     HTTP      TCP       80          0.0.0.0/0        ALLOW
110     HTTPS     TCP       443         0.0.0.0/0        ALLOW
120     SSH       TCP       22          203.0.113.0/24   ALLOW  # office IP range
130     Custom    TCP       1024-65535  0.0.0.0/0        ALLOW  # ephemeral ports!
*       All       All       All         0.0.0.0/0        DENY   # implicit deny

## Example NACL for a public subnet (outbound rules)
Rule#   Type      Protocol  Port Range  Destination      Action
100     HTTP      TCP       80          0.0.0.0/0        ALLOW
110     HTTPS     TCP       443         0.0.0.0/0        ALLOW
120     Custom    TCP       1024-65535  0.0.0.0/0        ALLOW  # ephemeral ports!
*       All       All       All         0.0.0.0/0        DENY

Stateless — The Critical Difference From Security Groups

NACLs are stateless. They do not track connection state. Every packet — both request AND response — is evaluated independently against the rules. This leads to the most common NACL mistake:

❌ Broken NACL — no ephemeral ports
Inbound:
  100: TCP 443 0.0.0.0/0 ALLOW
  *: DENY

Outbound:
  100: TCP 443 0.0.0.0/0 ALLOW
  *: DENY

Result: HTTPS requests arrive (inbound 443 ✓)
But responses can't leave!
Why? The response goes out on an ephemeral
port (e.g., 49200) — not port 443.
Outbound NACL doesn't allow 49200 → blocked.
✅ Correct NACL — ephemeral ports allowed
Inbound:
  100: TCP 443 0.0.0.0/0 ALLOW
  110: TCP 1024-65535 0.0.0.0/0 ALLOW
  *: DENY

Outbound:
  100: TCP 443 0.0.0.0/0 ALLOW
  110: TCP 1024-65535 0.0.0.0/0 ALLOW
  *: DENY

The 1024-65535 range covers all ephemeral
ports (client-side random ports).
Both request and response can flow.

What are ephemeral ports?

When a client connects to a server (e.g., browser → web server port 443), the client uses a random ephemeral (short-lived) port as its source port — typically 1024–65535. The server's response goes back to that ephemeral port. Since NACLs are stateless, you must explicitly allow these ephemeral ports in your rules, or response traffic gets dropped.

# Ephemeral port ranges by OS:
Linux:   32768 – 60999
Windows: 49152 – 65535
NAT GW:  1024 – 65535  (AWS recommends allowing this full range)

# Rule of thumb for NACLs:
# Inbound: allow the protocol ports you care about + 1024-65535 for responses
# Outbound: allow the protocol ports you care about + 1024-65535 for outbound responses

When to Actually Use NACLs

Most production setups use the default NACL (allow all) and rely entirely on security groups. NACLs are most useful for:

Blocking a specific IP
You discover an IP address is attacking your application. Security groups can't deny — they only allow. NACL can explicitly deny: Rule 90: DENY all traffic from 192.0.2.200/32 (above all other rules).
Subnet-level egress control
You want to prevent an entire subnet from reaching the internet (belt and suspenders with route tables). Add an outbound NACL deny for 0.0.0.0/0 as a backup to the missing route.
Compliance requirements
Some compliance frameworks (PCI-DSS) require network-level controls independent of instance-level controls. NACLs provide this — even if someone misconfigures a security group, NACL is still there.
Emergency incident response
During a security incident, quickly add a NACL deny rule at the subnet level to cut off all traffic to/from a compromised subnet. Faster than modifying all instance security groups.

Memory Trick

Security Group = Bodyguard
Stands at the door of each individual room (instance). Checks everyone coming in. Remembers who it let in (stateful), so when they leave it lets them out without checking again. Can only say YES (allow) — can't explicitly reject anyone.

Applies only to rooms it's assigned to. 5 rooms → 5 bodyguards.
NACL = Building Security Checkpoint
Checks ALL traffic entering and leaving the whole floor (subnet). Has a numbered checklist — goes through rules in order. Can say YES or NO explicitly. Forgets everyone immediately (stateless) — same person leaving is checked again.

One checkpoint covers all rooms on the floor automatically.

The Both-Layer Check — How They Work Together

When an internet user sends a packet to your EC2 instance, it must pass both NACL and Security Group:

Inbound packet: Internet → EC2 instance in public subnet
1
Internet Gateway: Receives packet from internet destined for the EC2's public IP. Translates to private IP (10.0.1.45). Sends into VPC.
2
NACL evaluated (inbound): Packet enters the subnet. NACL checks: does rule 100 (TCP 80 ALLOW) match? Yes → allowed into subnet.
3
Security Group evaluated (inbound): Packet arrives at instance's network interface. SG checks all inbound rules. Rule "TCP 80 0.0.0.0/0 ALLOW" matches → allowed to instance.
4
EC2 processes and responds. Response packet (source: 10.0.1.45, dest: user's IP, source port: 49200) needs to leave.
5
Security Group (outbound): Stateful — remembers the inbound connection. Return packet is automatically allowed regardless of outbound rules. ✓
6
NACL (outbound): Stateless — doesn't remember anything. Evaluates the outbound packet fresh. Port 49200 is an ephemeral port. If your NACL allows TCP 1024-65535 outbound ALLOW → allowed. If not → BLOCKED. This is why forgetting ephemeral ports in NACLs breaks things.
In practice: 95% of your security work is done with Security Groups. Use the default NACL (allow all) unless you have a specific need — like blocking an attacking IP or meeting a compliance requirement that mandates network-layer controls.

VPC Endpoints — Private AWS Access

Your EC2 instance in a private subnet needs to call S3 or DynamoDB. Without VPC endpoints, it goes via NAT Gateway → Internet → AWS public endpoints. With VPC endpoints, it stays entirely inside AWS's private network. Faster, cheaper, more secure.

The Problem VPC Endpoints Solve

S3, DynamoDB, SQS, Secrets Manager, and most AWS services are accessed via public endpoints (e.g., s3.amazonaws.com). Without VPC endpoints, traffic from your private EC2 instance to S3 goes: EC2 → NAT Gateway → Internet Gateway → S3 public endpoint. This means:

  • Your traffic leaves AWS's network and traverses the public internet
  • You pay NAT Gateway data processing charges (~$0.045/GB)
  • Slightly higher latency (internet routing)
  • No way to restrict which S3 buckets are accessible

VPC Endpoints create a private connection between your VPC and AWS services that stays entirely within AWS's network.

Two Types of VPC Endpoints

Gateway Endpoint — free, limited services
Works as a route table entry. Traffic destined for S3 or DynamoDB is routed via the gateway instead of internet.

Supports: S3 and DynamoDB only
Cost: Free (no hourly charge, no data charge)
How it works: Adds a route to your route table: pl-68a54001 (S3 prefix) → vpce-xxx
Does not: use an IP in your subnet, need DNS changes
Interface Endpoint (AWS PrivateLink) — paid, all services
Creates an Elastic Network Interface (ENI) with a private IP in your subnet. DNS resolution for the service endpoint resolves to this private IP instead of the public one.

Supports: 100+ services (EC2, SSM, Secrets Manager, SQS, SNS, Lambda, API Gateway, etc.)
Cost: ~$0.01/hour per AZ + $0.01/GB
How it works: DNS override — service endpoint resolves to private IP

Gateway Endpoint — S3 (Most Common)

# Create a Gateway Endpoint for S3
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0abc123def456789 \
  --service-name com.amazonaws.us-east-1.s3 \
  --vpc-endpoint-type Gateway \
  --route-table-ids rtb-0private-app-1a rtb-0private-app-1b rtb-0private-app-1c

# AWS automatically adds a route to the specified route tables:
# Destination: pl-68a54001 (S3 prefix list — a list of S3's IP ranges)
# Target: vpce-0abc123456

# Now when your EC2 calls S3:
# - Packet destined for S3 matches the prefix list route
# - Traffic goes via endpoint (stays on AWS network)
# - No NAT Gateway, no internet, free data transfer

# Verify the route was added
aws ec2 describe-route-tables --route-table-id rtb-0private-app-1a
# Look for routes with GatewayId starting with "vpce-"

# Your application code changes NOTHING — same S3 endpoint
s3 = S3Client.create()                          # same as before
s3.getObject(bucket="my-bucket", key="file")    # traffic now stays private

Interface Endpoint — Secrets Manager (Typical Use)

# Create Interface Endpoint for Secrets Manager
# Allows Lambda/EC2 in private subnet to call Secrets Manager without internet

aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0abc123def456789 \
  --service-name com.amazonaws.us-east-1.secretsmanager \
  --vpc-endpoint-type Interface \
  --subnet-ids subnet-0app1a subnet-0app1b \     # one per AZ
  --security-group-ids sg-0endpoint \            # SG to control who can use the endpoint
  --private-dns-enabled                          # CRITICAL: DNS override

# Security group for the endpoint itself:
# Inbound: TCP 443 from sg-app (only app servers can reach Secrets Manager endpoint)
# Outbound: empty (interface endpoints don't send outbound)

# With --private-dns-enabled:
# secretsmanager.us-east-1.amazonaws.com resolves to a PRIVATE IP (e.g., 10.0.11.x)
# WITHOUT the flag: resolves to public IP, traffic goes to internet
# Your application code STILL calls secretsmanager.us-east-1.amazonaws.com
# DNS magic handles the routing

# Verify it works from EC2 in private subnet:
nslookup secretsmanager.us-east-1.amazonaws.com
# Should return: 10.0.11.x (your VPC IP, not public AWS IP)
# Then:
aws secretsmanager get-secret-value --secret-id my-app-db-password
# Success — never left your VPC

Endpoint Policies — Security Control

# VPC Endpoints can have their own policies to restrict what's allowed through them
# This is a powerful data exfiltration prevention tool

# Example: S3 endpoint policy — only allow access to OUR buckets
# Even if an attacker gets into our network, they can't copy data to their S3 bucket
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::my-company-*",       // our buckets
        "arn:aws:s3:::my-company-*/*"
      ]
    },
    {
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::aws-ssm-*/*"   // AWS Systems Manager needs S3
    }
  ]
}

VPC Peering & Transit Gateway

How multiple VPCs communicate with each other — point-to-point peering for simple connections, Transit Gateway for hub-and-spoke at scale.

Why Connect VPCs?

You have a production VPC and a development VPC. Your security team's VPC needs to scan all other VPCs. Your shared services VPC (with Active Directory, internal tools) needs to be reachable by all application VPCs. VPC isolation is great for security, but sometimes services in different VPCs need to talk.

VPC Peering — Point-to-Point

VPC Peering creates a direct private network connection between two VPCs. Traffic between peered VPCs stays on AWS's private network — never goes through the internet.

# VPC Peering: prod-vpc (10.0.0.0/16) ↔ dev-vpc (10.1.0.0/16)

# Step 1: Create peering connection (from prod account/VPC)
aws ec2 create-vpc-peering-connection \
  --vpc-id vpc-0prod123 \                    # requester VPC
  --peer-vpc-id vpc-0dev456 \               # accepter VPC
  --peer-region us-east-1                   # same region (or specify for cross-region)

# Step 2: Accept the peering (from dev account/VPC)
aws ec2 accept-vpc-peering-connection \
  --vpc-peering-connection-id pcx-0abc123def

# Step 3: Add routes in BOTH VPCs (peering is not automatic in route tables)

# In prod-vpc route table: add route to dev-vpc CIDR via peering
aws ec2 create-route \
  --route-table-id rtb-0prod-private \
  --destination-cidr-block 10.1.0.0/16 \    # dev-vpc CIDR
  --vpc-peering-connection-id pcx-0abc123def

# In dev-vpc route table: add route to prod-vpc CIDR via peering
aws ec2 create-route \
  --route-table-id rtb-0dev-private \
  --destination-cidr-block 10.0.0.0/16 \    # prod-vpc CIDR
  --vpc-peering-connection-id pcx-0abc123def

# Step 4: Update security groups to allow traffic from peered VPC's CIDR
# RDS in prod: add inbound rule TCP 5432 10.1.0.0/16 (dev-vpc's IP range)

VPC Peering Limitations

No transitive routing
If A peers with B, and B peers with C — A cannot reach C. Each pair needs its own peering connection. With 10 VPCs wanting full mesh: 10×(10-1)/2 = 45 peering connections to manage. This doesn't scale.
No overlapping CIDRs
Both VPCs must have non-overlapping CIDR blocks. If prod is 10.0.0.0/16 and dev is also 10.0.0.0/16 — you can't peer them. This is why you plan CIDRs before building.

Transit Gateway — Hub and Spoke at Scale

AWS Transit Gateway (TGW) is a cloud router that acts as a central hub connecting multiple VPCs, on-premises networks (via VPN or Direct Connect), and other AWS accounts. Instead of managing N×(N-1)/2 peering connections, every VPC connects to one TGW.

# Without TGW (5 VPCs = 10 peering connections, each needing route table entries)
prod ↔ dev       prod ↔ staging     prod ↔ shared-services     prod ↔ security
dev ↔ staging    dev ↔ shared-services   ...  (6 more pairs)

# With TGW (5 VPCs = 5 attachments to 1 TGW)
prod ──┐
dev  ──┤
staging ─── TGW ─── on-premises (VPN)
shared ──┤
security─┘

# Create Transit Gateway
aws ec2 create-transit-gateway \
  --description "Central network hub" \
  --options AmazonSideAsn=64512,AutoAcceptSharedAttachments=disable \
  --tag-specifications 'ResourceType=transit-gateway,Tags=[{Key=Name,Value=main-tgw}]'

# Attach each VPC to the TGW
aws ec2 create-transit-gateway-vpc-attachment \
  --transit-gateway-id tgw-0abc123456789 \
  --vpc-id vpc-0prod123 \
  --subnet-ids subnet-0prod1a subnet-0prod1b \  # one subnet per AZ
  --tag-specifications 'ResourceType=transit-gateway-attachment,Tags=[{Key=Name,Value=prod-attachment}]'

# TGW has its own route table — configure routing
# By default: all attached VPCs can reach each other (if you use default route table)
# For isolation: create separate TGW route tables per attachment

# Restrict so dev VPCs can reach shared-services but not prod:
# Create two TGW route tables: "prod-rt" and "dev-rt"
# prod-rt: route to shared-services, on-premises
# dev-rt: route to shared-services only (no route to prod VPC)

Beginner

What is a VPC and why do you need one? ▼
A VPC (Virtual Private Cloud) is a logically isolated section of the AWS cloud where you can launch AWS resources in a virtual network you define. You need one because: (1) Isolation — your resources are completely separated from other AWS customers, even though you share physical hardware. (2) Security — you control network access. Your database can be in a private subnet with no internet access at all. (3) Control — you define your own IP address ranges, subnets, route tables, and gateways. Without a VPC, all EC2 instances would be on the same flat network with internet access by default — a security nightmare.
What is the difference between a public and a private subnet? ▼
The difference is entirely in the route table — there is no "public" flag on a subnet. A public subnet has a route in its route table pointing to an Internet Gateway (0.0.0.0/0 → igw-xxx). Resources in a public subnet CAN be reached from the internet (if they have a public IP) and can reach the internet directly. A private subnet has no route to an Internet Gateway. Resources here are unreachable from the internet. They may have a route via NAT Gateway for outbound-only internet access, or no internet access at all (database subnets). The distinction is purely routing — the same subnet could be "made public" by adding a route to an IGW.
What is the difference between an Internet Gateway and a NAT Gateway? ▼
Internet Gateway (IGW): Bidirectional — allows both inbound from the internet AND outbound to the internet. Attached to the VPC itself. Used by public subnets. Resources need a public IP address. Does 1:1 NAT (private IP ↔ public IP per instance). Free.

NAT Gateway: Outbound only — allows private subnet instances to reach the internet (for software updates, API calls) but prevents the internet from initiating connections back in. Lives in a public subnet (uses a public/Elastic IP itself). Serves as the single outbound point for all private instances (many private IPs → one public IP). Costs ~$0.045/hour plus data charges. You need one per AZ in production for availability.
What is the difference between a Security Group and a NACL? ▼
Four key differences: (1) Level: Security Groups operate at the instance (ENI) level. NACLs operate at the subnet level. (2) Stateful vs Stateless: SGs are stateful — return traffic is automatically allowed. NACLs are stateless — every packet is evaluated independently, so you must allow ephemeral ports in both directions. (3) Allow vs Allow+Deny: SGs only have allow rules; everything else is denied by default. NACLs can explicitly deny — useful for blocking specific IP addresses. (4) Rule evaluation: SGs evaluate all rules (any match = allow). NACLs evaluate in numbered order (first match wins). In practice: use Security Groups as your primary firewall for everything. Use NACLs only when you need to explicitly deny specific IPs or add a network-level audit requirement.

Intermediate

Your Lambda function needs to access an RDS database in a private subnet. What VPC configuration is required? ▼
Lambda must be configured to run inside the VPC (by specifying VPC config with subnet IDs and security group). Steps: (1) Lambda VPC config: Specify the private subnets and a security group for Lambda. Lambda creates ENIs in those subnets — this is how it gets a VPC-local IP. (2) Security groups: Create sg-lambda for the Lambda function. Create/modify sg-rds on RDS to allow inbound TCP on the DB port (5432 for PostgreSQL) from sg-lambda. (3) Route tables: Lambda's subnets need routes to the internet (via NAT) if Lambda calls external services. Without NAT, Lambda in VPC cannot reach the internet OR AWS public endpoints. (4) VPC Endpoints: If Lambda only needs to call AWS services (S3, Secrets Manager, DynamoDB), add VPC Endpoints instead of NAT — cheaper and more secure. Important caveat: Lambda in VPC has cold start overhead due to ENI creation. Lambda NOT in VPC cannot access resources in a private subnet.
What are VPC Endpoints and when would you use them? What's the difference between Gateway and Interface endpoints? ▼
VPC Endpoints create a private connection between your VPC and AWS services without going through the internet. Traffic stays on AWS's private network. Use when: you have resources in private subnets that need to access AWS services (S3, DynamoDB, Secrets Manager, etc.) and want to avoid internet routing for security (no traffic on public internet), cost (NAT Gateway charges ~$0.045/GB), or performance. Gateway Endpoint: Works via route table entries. Only for S3 and DynamoDB. Free. Traffic is routed via a prefix list in the route table — no ENI, no IP in your subnet. Interface Endpoint (PrivateLink): Creates an ENI with a private IP in your subnet. Works for 100+ services (EC2, SQS, SNS, Secrets Manager, SSM, Lambda, etc.). Costs ~$0.01/hour per AZ. Requires enabling private DNS so service endpoints resolve to the private IP. Controlled by a security group on the endpoint itself.
Why do NACLs require rules for ephemeral ports? Give a concrete example. ▼
NACLs are stateless — they evaluate every packet independently without remembering connection state. When a client (browser) connects to your EC2 web server on port 443, the client randomly selects a source port from the ephemeral range (e.g., 54321). The server's response goes TO port 54321 on the client. Concrete example: browser at 1.2.3.4:54321 → EC2:443 (request). EC2:443 → 1.2.3.4:54321 (response). NACL inbound rule says "allow TCP 443 from 0.0.0.0/0" — request comes in fine. NACL outbound rule says "allow TCP 443 to 0.0.0.0/0" — response goes to port 54321, NOT 443. Outbound rule doesn't match → response is DROPPED → connection appears broken. Fix: add outbound NACL rule "allow TCP 1024-65535 to 0.0.0.0/0". Security groups don't have this problem because they're stateful — they remember "I allowed this inbound connection" and automatically permit the return traffic.
Explain the longest prefix match rule in routing with a practical example. ▼
When a packet's destination IP matches multiple routes, the router picks the MOST SPECIFIC route (the one with the longest prefix/highest CIDR number). Example route table: 10.0.0.0/16 → local, 10.0.1.0/24 → tgw-xxx, 0.0.0.0/0 → nat-xxx. Packet to 10.0.1.50: matches all three (10.0.0.0/16, 10.0.1.0/24, and 0.0.0.0/0 technically doesn't but let's say 10.0.0.0/8 → another route). Winner: 10.0.1.0/24 is /24 (most specific), so packet goes to Transit Gateway. Packet to 10.0.5.100: matches 10.0.0.0/16 and 0.0.0.0/0. Winner: /16 beats /0, so packet goes local. Packet to 8.8.8.8: only 0.0.0.0/0 matches → goes to NAT Gateway. This is why the "local" route (covering your VPC CIDR) always wins for internal traffic, even though 0.0.0.0/0 technically covers everything.

Advanced / Architect

Design the VPC architecture for a high-availability three-tier web application. What components are needed and where does each go? ▼
Production VPC (10.0.0.0/16) in us-east-1 with 3 AZs. Public subnets (10.0.1-3.0/24, one per AZ): Application Load Balancer (spans all three AZs), NAT Gateways (one per AZ — critical for AZ fault tolerance). Private app subnets (10.0.11-13.0/24): EC2 Auto Scaling Group or ECS Fargate tasks. These never have public IPs — only ALB can reach them. Private data subnets (10.0.21-23.0/24): RDS with Multi-AZ (no internet route at all), ElastiCache Redis cluster. Security groups: sg-alb (allows 443/80 from internet, outbound to sg-app), sg-app (allows 8080 from sg-alb, outbound to sg-db/sg-cache/443 to internet), sg-db (allows 5432 from sg-app only). VPC Endpoints: Gateway endpoint for S3 (free), Interface endpoints for Secrets Manager and SSM (so app servers don't need NAT just for AWS APIs). Internet Gateway: one, attached to VPC. Route tables: public-rt (→ IGW), private-app-rt-1a/1b/1c (each → own NAT GW), private-db-rt (local only). This provides: no database exposure to internet, app servers unreachable from internet directly, multi-AZ fault tolerance at every layer.
You have 8 VPCs across 3 AWS accounts that all need to communicate with a shared services VPC. How would you architect this? ▼
Use AWS Transit Gateway (TGW) with AWS Resource Access Manager (RAM). Architecture: (1) Create TGW in the shared services account. (2) Share the TGW via RAM to all 3 accounts — no need to create separate TGWs. (3) Each VPC in each account creates a TGW attachment (costs ~$0.05/hr per attachment). (4) TGW route tables control connectivity — create two route tables: "shared-services-rt" (routes to all 8 app VPCs — so shared services can reach all) and "app-rt" (routes to shared-services VPC only — app VPCs can NOT reach each other, preventing lateral movement if compromised). (5) Update each VPC's route table to point to TGW for the relevant CIDRs. Why not VPC Peering: would need 8 peering connections per pair = 28+ connections, each needing route table updates in both VPCs. TGW is 9 attachments total, centrally managed. Ensure all VPCs have non-overlapping CIDRs — this is why planning CIDR ranges before building is critical.
What is a CIDR block of 0.0.0.0/0 vs 10.0.0.0/16 vs 10.0.1.0/24? How many IP addresses does each contain? ▼
0.0.0.0/0: /0 means ZERO fixed bits — all 32 bits are free. 2^32 = 4,294,967,296 addresses. This covers ALL IP addresses — literally "the entire internet." Used in route tables as the default route (catch-all) and in security groups to mean "from anywhere." 10.0.0.0/16: /16 means 16 bits fixed. 2^(32-16) = 2^16 = 65,536 addresses. Covers 10.0.0.0 → 10.0.255.255. Standard size for a production VPC — large enough for many subnets, small enough to remain manageable. 10.0.1.0/24: /24 means 24 bits fixed. 2^(32-24) = 2^8 = 256 addresses. Covers 10.0.1.0 → 10.0.1.255. AWS reserves 5, so 251 usable. Standard size for a single-purpose subnet (one tier, one AZ). In security group rules: 0.0.0.0/0 = allow from internet, /32 = allow from one specific IP, /24 = allow from a whole subnet.
