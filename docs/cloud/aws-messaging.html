Async vs Sync — Why Messaging Exists

Before touching SQS or SNS, you need to understand the fundamental problem they solve. Without this context, messaging feels like unnecessary complexity. With it, it becomes the obvious solution.

The Synchronous Service Problem

When Service A calls Service B directly over HTTP, they are tightly coupled. Service A waits for Service B to respond before continuing. This works fine until it doesn't — and the failures are spectacular.

Real Production Failure
Direct HTTP coupling causes cascade failure

Your e-commerce checkout works like this: user places order → OrderService calls PaymentService → calls InventoryService → calls EmailService → calls AnalyticsService → returns success. All synchronous, all in sequence.

What actually happens: EmailService is slow today (email provider having issues). InventoryService waits 30 seconds for EmailService. OrderService waits 30 seconds. User sees a spinner. They abandon. You lose the sale. And EmailService being slow has nothing to do with whether the order was successfully placed — it's a downstream notification that has zero impact on the order itself.

Cascade: If EmailService goes down, orders stop processing. If AnalyticsService has a bug and throws an exception, orders fail. Services that should be independent are coupled together by synchronous calls.

Asynchronous Messaging Decouples Everything

With messaging, Service A puts a message in a queue and returns immediately. Service B reads from the queue when it's ready. Service A doesn't wait, doesn't care if Service B is slow, doesn't fail if Service B is down. They are loosely coupled.

❌ Synchronous (direct calls)
OrderService.placeOrder()
  ↓ HTTP call (waits 200ms)
  PaymentService.charge()
  ↓ HTTP call (waits 50ms)
  InventoryService.reserve()
  ↓ HTTP call (waits 3,000ms!!!)
  EmailService.sendConfirmation()
  ↓ HTTP call (waits 100ms)
  AnalyticsService.track()
  ← DONE (total: 3,350ms)

If EmailService dies → orders fail
If it's slow → users wait
Tight coupling = fragile system
✅ Asynchronous (messaging)
OrderService.placeOrder()
  ↓ HTTP call (waits 200ms)
  PaymentService.charge()
  ↓ HTTP call (waits 50ms)
  InventoryService.reserve()
  ↓ SQS.sendMessage() (waits 5ms)
  "order-placed" event queued
  ← DONE (total: 255ms!) ✓

EmailService reads queue when ready
AnalyticsService reads queue when ready
If EmailService dies → messages queue up
  → delivered when it recovers
Loose coupling = resilient system

The Three Messaging Patterns

Queue (Point-to-Point)
One sender, one receiver. Message is consumed once and deleted. Order between sender and consumer. Used for work distribution — divide tasks among multiple workers.

AWS service: SQS
Pub/Sub (Broadcast)
One sender (publisher), many receivers (subscribers). Every subscriber gets a copy. Publisher doesn't know who is subscribed. Used for event notification — "something happened, tell everyone who cares."

AWS service: SNS, EventBridge
Streaming (Replay)
Ordered, partitioned log of records. Multiple consumers can read at their own pace. Records retained for days/weeks — can replay from the beginning.

AWS service: Kinesis, Kafka (MSK)

What is SQS?

Simple Queue Service — a fully managed message queue. One of the oldest AWS services (2004) and still one of the most used. Understanding SQS deeply is essential because it's used in nearly every distributed AWS application.

What a Queue Actually Does

A queue is a buffer between a producer (something creating work) and a consumer (something doing the work). The producer adds messages to the queue. The consumer picks messages off the queue, processes them, and deletes them. The queue holds messages if consumers are busy or down — nothing is lost.

Analogy
SQS is like a ticket number system at a bank

Customers (producers) take a numbered ticket when they arrive. Tellers (consumers) call the next number when they're free. If all tellers are busy, customers wait. If the bank suddenly gets 100 customers, tickets are issued immediately — customers don't get turned away, they just wait. If a teller is sick, their tickets get processed by other tellers or when they return. The ticket system (queue) absorbs the mismatch between arrival rate and service rate.

SQS Core Properties

At-least-once delivery
SQS guarantees every message is delivered at least once. In rare cases, a message might be delivered twice. Your consumer code must be idempotent — processing the same message twice should produce the same result as processing it once.
Unlimited throughput
Standard queues handle unlimited messages per second. No capacity planning. Send 1 message or 1 million messages per second — SQS handles it. You're billed per message, not per capacity unit.
Retention period
Messages persist in the queue for 1 minute to 14 days (default: 4 days). If no consumer processes a message within the retention period, it's deleted. Set longer retention for fault-tolerant systems.

Message Lifecycle — What Actually Happens

Complete SQS message lifecycle
1
Producer sends: OrderService calls sqs.sendMessage(). SQS stores the message and returns a MessageId. Message is now visible in the queue.
2
Consumer receives: Worker calls sqs.receiveMessage(). SQS returns up to 10 messages and makes them invisible for the visibility timeout period (default 30s). This prevents other workers from seeing and processing the same message.
3
Consumer processes: Worker processes the message (writes to DB, calls an API, sends an email). This should complete within the visibility timeout.
4a — Success:
Worker calls sqs.deleteMessage() with the receipt handle. Message is permanently removed from the queue. Processed successfully.
4b — Failure:
Worker crashes or throws an exception. It never calls deleteMessage. Visibility timeout expires. Message becomes visible again and is redelivered to another worker. This is SQS's built-in retry mechanism.
5
Dead Letter Queue: After N retry attempts (configured by maxReceiveCount), message is moved to the DLQ for investigation. Never silently lost.

SQS in Practice — Sending and Receiving

# Create a queue
aws sqs create-queue \
  --queue-name orders-processing \
  --attributes '{
    "VisibilityTimeout": "60",
    "MessageRetentionPeriod": "86400",
    "ReceiveMessageWaitTimeSeconds": "20",
    "RedrivePolicy": "{
      \"deadLetterTargetArn\": \"arn:aws:sqs:us-east-1:123:orders-dlq\",
      \"maxReceiveCount\": \"3\"
    }"
  }'

# Returns: QueueUrl = https://sqs.us-east-1.amazonaws.com/123456789012/orders-processing
SqsClient sqs = SqsClient.create();
String queueUrl = "https://sqs.us-east-1.amazonaws.com/123456789012/orders-processing";

// PRODUCER: Send a message
SendMessageResponse sent = sqs.sendMessage(SendMessageRequest.builder()
    .queueUrl(queueUrl)
    .messageBody(objectMapper.writeValueAsString(order))
    .messageAttributes(Map.of(
        "orderType", MessageAttributeValue.builder()
            .dataType("String").stringValue("PHYSICAL").build()))
    .delaySeconds(0)    // optionally delay delivery by 0-900 seconds
    .build());
log.info("Sent: {}", sent.messageId());

// CONSUMER: Poll for messages (long polling — waits up to 20s for messages)
while (true) {
    ReceiveMessageResponse response = sqs.receiveMessage(
        ReceiveMessageRequest.builder()
            .queueUrl(queueUrl)
            .maxNumberOfMessages(10)     // up to 10 per receive call
            .waitTimeSeconds(20)          // long polling (cheaper, fewer empty receives)
            .visibilityTimeout(60)         // 60s to process each message
            .build());

    for (Message msg : response.messages()) {
        try {
            Order order = objectMapper.readValue(msg.body(), Order.class);
            orderProcessor.process(order);

            // SUCCESS: delete message from queue
            sqs.deleteMessage(DeleteMessageRequest.builder()
                .queueUrl(queueUrl)
                .receiptHandle(msg.receiptHandle())  // unique handle for this receive
                .build());
        } catch (Exception e) {
            log.error("Failed to process {}", msg.messageId(), e);
            // DON'T delete — visibility timeout expires, message redelivered
        }
    }
}

Long Polling vs Short Polling

Short Polling (default, avoid)
SQS returns immediately — even if the queue is empty. You get back an empty response and immediately call again. Creates thousands of empty API calls per minute. Costs more, wastes CPU. WaitTimeSeconds=0
Long Polling (always use this)
SQS holds the connection open up to WaitTimeSeconds (max 20) waiting for a message. Returns immediately if a message arrives. Dramatically reduces empty responses. Cheaper and more efficient. WaitTimeSeconds=20

Batch Operations — Process Multiple Messages at Once

# Send up to 10 messages in one API call (batch send)
aws sqs send-message-batch \
  --queue-url https://sqs.us-east-1.amazonaws.com/123/orders \
  --entries '[
    {"Id":"1","MessageBody":"{\"orderId\":\"ord-001\"}"},
    {"Id":"2","MessageBody":"{\"orderId\":\"ord-002\"}"},
    {"Id":"3","MessageBody":"{\"orderId\":\"ord-003\"}"}
  ]'
# Cost: one API call vs three = 66% cost reduction at scale

# Delete batch (always batch deletes after processing)
sqs.deleteMessageBatch(DeleteMessageBatchRequest.builder()
    .queueUrl(queueUrl)
    .entries(processedMessages.stream()
        .map(m -> DeleteMessageBatchRequestEntry.builder()
            .id(m.messageId())
            .receiptHandle(m.receiptHandle())
            .build())
        .collect(Collectors.toList()))
    .build());

Standard vs FIFO Queues

Two fundamentally different queue types with different guarantees, throughput limits, and cost. Choosing wrong causes either data loss, duplicate processing, or ordering bugs.

Standard Queue — Maximum Throughput, Best-Effort Ordering

Standard queues offer unlimited throughput and at-least-once delivery. Messages are generally delivered in the order sent but there's no guarantee — SQS distributes messages across many servers for throughput, and occasionally this causes out-of-order delivery. Occasionally a message may be delivered more than once.

Use Standard when: order doesn't matter, occasional duplicates are fine (your processing is idempotent), you need maximum throughput. Examples: sending emails (duplicate email is annoying but not catastrophic), triggering image processing, logging events, fan-out workloads.

FIFO Queue — Exactly-Once, In-Order

FIFO queues guarantee: messages are delivered in the order sent, each message is processed exactly once (no duplicates). They achieve this via a content-based deduplication ID — SQS deduplicates messages with the same ID within a 5-minute window.

Trade-off: FIFO queues have a throughput limit — 3,000 messages/second with batching (300/sec without). Much lower than Standard. You also pay ~10% more per message.

Use FIFO when: order matters (financial transactions, sequential state machine transitions), duplicates are unacceptable (charge a credit card exactly once), workflow that must execute in steps.

# Create a FIFO queue (name MUST end in .fifo)
aws sqs create-queue \
  --queue-name payments-processing.fifo \
  --attributes '{
    "FifoQueue": "true",
    "ContentBasedDeduplication": "true",  
    "DeduplicationScope": "messageGroup",
    "FifoThroughputLimit": "perMessageGroupId"
  }'

# FIFO messages require MessageGroupId (like a partition key)
# Messages in the same group are ordered relative to each other
# Messages across groups can be processed in parallel
// Send to FIFO queue — must include MessageGroupId
sqs.sendMessage(SendMessageRequest.builder()
    .queueUrl("https://sqs.us-east-1.amazonaws.com/123/payments-processing.fifo")
    .messageBody(serialise(paymentEvent))
    .messageGroupId("user-" + userId)     // same user = same group = ordered
    .messageDeduplicationId(paymentId)     // prevents duplicate processing
    .build());

// Why per-user grouping?
// User Alice makes payments: Payment A1 then A2 then A3
// They MUST process in order (A1 before A2 before A3)
// User Bob makes payments: B1, B2
// Bob's payments can process in parallel with Alice's
// MessageGroupId per user achieves this

FIFO Message Group ID — The Parallelism Key

This is the subtlety that most people miss about FIFO queues. Messages in the same group are strictly ordered and processed one at a time. Messages in different groups can be processed in parallel. Design your group IDs around what needs to be ordered relative to each other.

## Good MessageGroupId choices:
userId       → per-user ordering (most common)
orderId      → per-order operations ordered
accountId    → per-account financial operations

## Bad: everything in one group
MessageGroupId = "all"  ← everything serialised, 300 msg/sec throughput
                            no benefit from FIFO, bottleneck

## Bad: too granular
MessageGroupId = paymentId  ← each payment is its own group, no ordering
                               between related operations, might as well use Standard

Visibility Timeout & Dead Letter Queues

Two SQS mechanisms that are critical to understand for building reliable systems. Get visibility timeout wrong and messages are processed twice. Ignore DLQs and failing messages silently disappear.

Visibility Timeout — How SQS Prevents Double Processing

When a consumer receives a message, SQS makes that message invisible to all other consumers for the VisibilityTimeout duration. This prevents two consumers from processing the same message simultaneously while still allowing retry on failure.

Why This Exists
The problem without visibility timeout

Without invisibility: Consumer A receives "Process Order 123" and starts working. Three seconds later, Consumer B also receives "Process Order 123" (queue thinks A didn't get it). Now both A and B are processing the same order — double charge, double shipment. Disaster.

With visibility timeout: Consumer A receives the message. SQS hides it for 60 seconds. Consumer B polls the queue and sees nothing. A processes successfully and deletes the message. Message gone. If A crashes mid-processing, the 60s expires, the message reappears, B picks it up and processes it. Reliability without double processing.

## Key rules for setting VisibilityTimeout:
# Must be > time it takes to process the message
# If processing takes 45s and timeout is 30s → message reappears WHILE being processed
# → double processing even though consumer is alive!

# Safe formula: VisibilityTimeout = max_processing_time × 6
# (gives plenty of headroom)

# For variable processing times: extend visibility timeout dynamically
ChangeMessageVisibilityResponse r = sqs.changeMessageVisibility(
    ChangeMessageVisibilityRequest.builder()
        .queueUrl(queueUrl)
        .receiptHandle(msg.receiptHandle())
        .visibilityTimeout(120)  // extend by 2 more minutes
        .build());
// Call this when you know processing will take longer than expected
// Common pattern: extend every 30 seconds in a background thread

## Visibility timeout settings reference:
# Queue-level default: 0-43200 seconds (12 hours)
# Per-receive override: can set in receiveMessage call
# Maximum: 43200 seconds (12 hours)

Dead Letter Queue (DLQ) — Never Lose a Failed Message

A DLQ is a regular SQS queue where SQS automatically sends messages that have failed processing too many times. Without a DLQ, a consistently failing message would cycle through your queue forever, consuming resources and preventing other messages from being processed (in FIFO). With a DLQ, problematic messages are isolated for investigation.

DLQ Flow — Message repeatedly fails processing
Producer
→
Main Queue
→
Consumer tries (fail #1)
↺
Message re-visible after timeout
→
Consumer tries (fail #2)
↺
Consumer tries (fail #3 = maxReceiveCount)
→
Dead Letter Queue
→
Alert! Developer investigates
# Step 1: Create the DLQ
aws sqs create-queue --queue-name orders-processing-dlq

# Step 2: Configure the main queue to use it
aws sqs set-queue-attributes \
  --queue-url https://sqs.us-east-1.amazonaws.com/123/orders-processing \
  --attributes '{
    "RedrivePolicy": "{
      \"deadLetterTargetArn\": \"arn:aws:sqs:us-east-1:123:orders-processing-dlq\",
      \"maxReceiveCount\": \"3\"
    }"
  }'

# maxReceiveCount=3 means:
# Attempt 1: consumer receives, fails, doesn't delete
# Attempt 2: message re-visible, consumer receives, fails again
# Attempt 3: consumer receives, fails again
# SQS moves to DLQ — consumer won't see it again in main queue

# Step 3: Set up CloudWatch alarm on DLQ depth
aws cloudwatch put-metric-alarm \
  --alarm-name orders-dlq-not-empty \
  --namespace AWS/SQS \
  --metric-name ApproximateNumberOfMessagesVisible \
  --dimensions Name=QueueName,Value=orders-processing-dlq \
  --threshold 0 \
  --comparison-operator GreaterThanThreshold \
  --evaluation-periods 1 \
  --period 60 \
  --statistic Sum \
  --alarm-actions arn:aws:sns:...:engineering-alerts
# Alert fires the moment any message lands in DLQ

# After fixing the bug: redrive messages from DLQ back to main queue
aws sqs start-message-move-task \
  --source-arn arn:aws:sqs:us-east-1:123:orders-processing-dlq \
  --destination-arn arn:aws:sqs:us-east-1:123:orders-processing

Making Your Consumer Idempotent

Since SQS delivers at-least-once, your consumer must handle the same message arriving twice without bad effects. This is called idempotency.

public class OrderProcessor {

    public void processOrder(String sqsMessageId, Order order) {

        // Check if we've already processed this message
        // Store processed messageIds in DynamoDB with TTL
        boolean alreadyProcessed = processedMessageRepo
            .exists(sqsMessageId);  // check DynamoDB

        if (alreadyProcessed) {
            log.info("Duplicate SQS message {}, skipping", sqsMessageId);
            return;  // idempotent — safe to skip
        }

        // Process the order
        orderRepository.save(order);
        inventoryService.reserve(order.getItems());

        // Record that we processed this message (TTL: 7 days)
        processedMessageRepo.save(sqsMessageId, Instant.now().plusDays(7));
    }
}

SQS + Lambda Integration

The most powerful and common SQS pattern — Lambda automatically polling the queue, scaling to match message volume, handling partial failures, and reporting results back.

How Lambda Polling Works with SQS

When you configure an SQS queue as a Lambda trigger, AWS creates an Event Source Mapping (ESM). The ESM is an internal process that polls the SQS queue on your behalf, batches messages, and invokes your Lambda function. You never write polling code — AWS does it for you.

Lambda + SQS — internal flow
1
You create an Event Source Mapping: "Poll this SQS queue and invoke this Lambda function."
2
ESM continuously polls the queue using long polling. When messages arrive, it batches them (up to your configured batch size, e.g. 10 messages).
3
ESM invokes Lambda with the batch as the event payload. Lambda scales automatically — if 1,000 messages are queued, ESM invokes many Lambda concurrencies simultaneously.
4
Lambda processes the batch. If ALL messages succeed, ESM deletes them all from SQS.
5
If Lambda throws an exception (fails the whole batch), ESM does NOT delete any messages. All messages become visible again after visibility timeout. The whole batch retries.
6
With Partial Batch Response: Lambda returns which specific message IDs failed. ESM deletes successful ones, retries only failed ones.
# Create Event Source Mapping — Lambda polls SQS automatically
aws lambda create-event-source-mapping \
  --function-name OrderProcessor \
  --event-source-arn arn:aws:sqs:us-east-1:123:orders-processing \
  --batch-size 10 \
  --maximum-batching-window-in-seconds 5 \   # wait up to 5s to fill a batch
  --function-response-types ReportBatchItemFailures  # partial failure support
public class OrderHandler implements
    RequestHandler<SQSEvent, SQSBatchResponse> {

    public SQSBatchResponse handleRequest(SQSEvent event, Context ctx) {
        List<SQSBatchResponse.BatchItemFailure> failures = new ArrayList<>();

        for (SQSEvent.SQSMessage msg : event.getRecords()) {
            try {
                Order order = objectMapper.readValue(msg.getBody(), Order.class);
                orderService.process(order);
                log.info("Processed order: {}", order.getId());

            } catch (NonRetryableException e) {
                // Bug in the message data — don't retry, send to DLQ
                log.error("Non-retryable error for {}: {}", msg.getMessageId(), e.getMessage());
                failures.add(SQSBatchResponse.BatchItemFailure.builder()
                    .withItemIdentifier(msg.getMessageId())
                    .build());

            } catch (TransientException e) {
                // Downstream service is slow — retry this message
                log.warn("Transient error for {}, will retry", msg.getMessageId());
                failures.add(SQSBatchResponse.BatchItemFailure.builder()
                    .withItemIdentifier(msg.getMessageId())
                    .build());
            }
        }

        // Failed messages go back to queue (or DLQ if maxReceiveCount exceeded)
        // Successful messages are automatically deleted by Lambda ESM
        return SQSBatchResponse.builder()
            .withBatchItemFailures(failures)
            .build();
    }
}

Scaling Behaviour

## How Lambda scales with SQS:
# Lambda adds concurrency based on queue depth:
# - Starts with 5 concurrent invocations
# - Adds 60 more per minute until queue clears or concurrency limit hit
# - With a 10,000 message burst: full concurrency in ~3 minutes

## Scaling control:
# Reserved concurrency on the Lambda function limits max concurrency
# This protects downstream services (your DB can't handle 1000 parallel workers)
aws lambda put-function-concurrency \
  --function-name OrderProcessor \
  --reserved-concurrent-executions 50  # max 50 concurrent Lambda workers

What is SNS?

Simple Notification Service — a fully managed pub/sub messaging service. Publishers send one message to a topic; every subscriber gets a copy. Used for broadcast notification and fan-out.

The Pub/Sub Mental Model

In SQS, there's one queue, one producer, and one (or more competing) consumers — a message is processed by exactly one consumer. In SNS, there's one topic, and any number of subscribers. Every subscriber gets their own copy of every message published to the topic. The publisher has no idea who is subscribed.

Real-World Example
Order placed event — 5 systems need to know

When an order is placed, five different systems need to act: EmailService sends confirmation, InventoryService reserves items, AnalyticsService records the event, FraudService checks for fraud, and RecommendationService updates the model.

Without SNS: OrderService makes 5 separate calls — it's tightly coupled to all five downstream services. Add a 6th service? Modify OrderService. Remove one? Modify OrderService again.

With SNS: OrderService publishes ONE message to the order-placed topic. All 5 services subscribe to the topic — each gets their own copy. Add/remove subscribers without touching OrderService. Zero coupling between producer and consumers.

SNS Core Concepts

Topic
A named channel. Publishers send messages to it. Subscribers receive from it. Topics have an ARN: arn:aws:sns:us-east-1:123:order-placed
Subscription
The connection between a topic and an endpoint. When you subscribe SQS to SNS, SNS will automatically push messages to that SQS queue. Protocol can be SQS, Lambda, HTTP, email, SMS.
Delivery protocols
SQS (push to queue), Lambda (invoke function directly), HTTP/S (POST to your endpoint), Email (human-readable), SMS (text message), Mobile Push (iOS/Android notifications).
# Create a topic
aws sns create-topic --name order-placed
# Returns: TopicArn=arn:aws:sns:us-east-1:123:order-placed

# Subscribe an SQS queue to the topic
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123:order-placed \
  --protocol sqs \
  --notification-endpoint arn:aws:sqs:us-east-1:123:email-service-queue

# Subscribe a Lambda function directly
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123:order-placed \
  --protocol lambda \
  --notification-endpoint arn:aws:lambda:us-east-1:123:function:FraudChecker

# Subscribe an HTTPS endpoint (your service's webhook URL)
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123:order-placed \
  --protocol https \
  --notification-endpoint https://analytics.internal.acme.com/events

# Publish a message — delivered to ALL subscribers simultaneously
aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:123:order-placed \
  --message '{"orderId":"ord-abc123","userId":"user-456","total":99.99}' \
  --subject "New Order Placed"
// Publishing from Java SDK
SnsClient sns = SnsClient.create();

PublishResponse response = sns.publish(PublishRequest.builder()
    .topicArn("arn:aws:sns:us-east-1:123:order-placed")
    .message(objectMapper.writeValueAsString(orderEvent))
    .messageAttributes(Map.of(
        "orderType", MessageAttributeValue.builder()
            .dataType("String")
            .stringValue(order.getType().name())
            .build()))
    .build());

log.info("Published to {} subscribers, messageId={}",
    // SNS simultaneously pushes to all subscriptions
    response.messageId());

The SQS Queue Policy — Required for SNS to Push

# When SNS pushes to SQS, it must have permission to write to the queue
# Add this policy to the SQS queue (not IAM — this is a resource policy)
aws sqs set-queue-attributes \
  --queue-url https://sqs.us-east-1.amazonaws.com/123/email-service-queue \
  --attributes '{
    "Policy": "{
      \"Version\": \"2012-10-17\",
      \"Statement\": [{
        \"Effect\": \"Allow\",
        \"Principal\": {\"Service\": \"sns.amazonaws.com\"},
        \"Action\": \"sqs:SendMessage\",
        \"Resource\": \"arn:aws:sqs:us-east-1:123:email-service-queue\",
        \"Condition\": {
          \"ArnEquals\": {
            \"aws:SourceArn\": \"arn:aws:sns:us-east-1:123:order-placed\"
          }
        }
      }]
    }"
  }'

Fan-Out Pattern & Message Filtering

The most important SNS pattern: SNS fan-out to multiple SQS queues. Combined with subscription filters, this lets each consumer receive only the messages relevant to them.

Fan-Out — The Production Pattern

Direct Lambda subscriptions to SNS work for simple cases, but for reliable production systems you want: SNS → SQS → Lambda. The SQS queue adds durability (if Lambda has a cold start or is throttled, messages queue up and nothing is lost) and retry logic (DLQ on the SQS queue).

Fan-out: SNS → Multiple SQS queues → Multiple Lambdas
OrderService
publishes
SNS: order-placed
SQS: email-queue
↓
Lambda: EmailSender
SQS: inventory-queue
↓
Lambda: InventoryReserver
SQS: analytics-queue
↓
Lambda: AnalyticsTracker

When OrderService publishes one message, SNS delivers a copy to all three queues simultaneously. Each queue has its own Lambda consumer, its own DLQ, its own retry logic. Completely independent. If InventoryService's Lambda is throttled and messages queue up, EmailService is unaffected.

Message Filtering — Subscribe to Only What You Need

By default, every SNS subscriber receives every message published to the topic. Message filtering lets each subscription specify a filter policy — a JSON condition on message attributes. The subscription only receives messages that match its filter.

Real Example
One topic, different consumers for different order types

You have one order-placed topic. Digital orders (ebooks, software) don't need InventoryService (nothing physical to reserve). Physical orders don't need DigitalFulfillmentService. Without filtering: all services receive all messages and must check and ignore irrelevant ones. With filtering: each service only receives what it cares about.

# Subscribe InventoryService queue but only receive PHYSICAL orders
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123:order-placed \
  --protocol sqs \
  --notification-endpoint arn:aws:sqs:us-east-1:123:inventory-queue \
  --attributes '{
    "FilterPolicy": "{
      \"orderType\": [\"PHYSICAL\"]
    }"
  }'

# Subscribe DigitalFulfillment queue but only receive DIGITAL orders

What is EventBridge?

EventBridge is a serverless event bus that connects AWS services, SaaS applications, and your own services using events. It's SNS's more powerful sibling — with richer routing, schema registry, and native AWS service integration.

Why EventBridge Exists Beyond SNS

SNS is excellent for simple pub/sub with message attribute filtering. EventBridge goes further — it receives events natively from 200+ AWS services (CloudTrail API calls, EC2 state changes, RDS events, etc.) and third-party SaaS apps, provides content-based routing on the full event body (not just attributes), includes a schema registry, and has more sophisticated retry and error handling.

SNS strengths
Simple pub/sub. High throughput (millions/sec). SMS and mobile push delivery. Fan-out to SQS/Lambda/HTTP. Message filtering on attributes. Simpler to set up.
EventBridge additional power
Content-based routing on full JSON body. Native AWS service events (200+ services). SaaS integration (Zendesk, Stripe, Datadog). Schema registry. Event replay. Pipes for point-to-point enrichment. Cross-account and cross-region delivery.

What is Kinesis?

Kinesis is AWS's real-time data streaming platform. Unlike SQS (queue — processed once, deleted) Kinesis is a log — data is retained, multiple consumers can read it at different positions, and you can replay historical data.

The Fundamental Difference: Queue vs Stream

This is the most important concept to grasp before Kinesis makes sense. SQS is a queue — process a message, delete it, next. Kinesis is a log — messages are written sequentially, retained for days, and multiple independent consumers each have their own read position.

SQS — Queue Model

Message received → processed → deleted. Next consumer gets the next message. Only one consumer processes each message. No replay. Messages gone after processing.

Good for: work queues, task distribution, retry handling

Kinesis — Stream Model

Records written to a numbered position in the stream. Retained for 24h–365 days. Consumer A reads position 1-100. Consumer B independently reads position 1-100. Consumer C starts fresh. All read the same data.

Good for: real-time analytics, audit logs, event sourcing, replay scenarios

Real Example — Why You Need Kinesis
Website clickstream — three systems need the same data, at different speeds

Your website generates 10 million clicks/day. Three systems need this data:

Real-time fraud detection: Must process within 100ms. Checks patterns against known fraud signatures. Fast consumer.

Analytics dashboard: Aggregates clicks per page per minute. Processes every 60 seconds. Medium speed consumer.

ML training pipeline: Batches a day's data and trains a recommendation model overnight. Slow consumer.

With SQS: only one consumer gets each click — you'd need three separate queues and three producers. With Kinesis: one stream, three independent consumers, each reading at their own pace from their own position.

Kinesis Data Streams Components

Stream
The top-level resource. Made up of shards. You specify the number of shards (or use on-demand). Data is retained 24h–365 days.
Shard
The unit of capacity. Each shard handles: 1MB/s writes, 2MB/s reads, 1000 records/s writes. Add more shards to scale throughput. More shards = more parallelism for consumers.
Record
The data unit. Up to 1MB. Has a partition key (determines which shard), a sequence number (position in shard), and the data blob. Records with the same partition key always go to the same shard.
# Create a Kinesis Data Stream
aws kinesis create-stream \
  --stream-name website-clickstream \
  --shard-count 4  # 4 shards = 4MB/s write, 8MB/s read

# Or use On-Demand mode (auto-scales, simpler)
aws kinesis create-stream \
  --stream-name website-clickstream \
  --stream-mode-details StreamMode=ON_DEMAND

Shards and Partition Keys — The Core Mechanics

When you write a record to Kinesis, you specify a partition key (a string). Kinesis hashes this key to determine which shard receives the record. Records with the same partition key always land on the same shard — preserving ordering for that partition.

    .data(SdkBytes.fromUtf8String(clickEventJson))

// Returns: ShardId, SequenceNumber (position in the shard)
log.info("Written to shard {} at sequence {}",
    response.shardId(), response.sequenceNumber());

PutRecordsResponse batchResponse = kinesis.putRecords(PutRecordsRequest.builder()
    .streamName("website-clickstream")
    .records(entries)
    .build());

Lambda Consumer — The Simplest Consumer

# Create Event Source Mapping — Lambda processes Kinesis records
aws lambda create-event-source-mapping \
  --function-name ClickstreamProcessor \
  --event-source-arn arn:aws:kinesis:us-east-1:123:stream/website-clickstream \
  --batch-size 100 \
  --starting-position LATEST \          # LATEST or TRIM_HORIZON (from beginning)
  --bisect-batch-on-function-error \    # on error, split batch and retry halves separately
  --function-response-types ReportBatchItemFailures
public class ClickstreamProcessor implements RequestHandler<KinesisEvent, StreamsEventResponse> {

    public StreamsEventResponse handleRequest(KinesisEvent event, Context ctx) {
        List<StreamsEventResponse.BatchItemFailure> failures = new ArrayList<>();

        for (KinesisEvent.KinesisEventRecord record : event.getRecords()) {
            try {
                String shardId = record.getEventID().split(":")[0];
                String sequenceNumber = record.getKinesis().getSequenceNumber();
                String data = record.getKinesis().getData().toString(StandardCharsets.UTF_8);

                ClickEvent click = objectMapper.readValue(data, ClickEvent.class);
                clickAnalytics.process(click);

            } catch (Exception e) {
                failures.add(StreamsEventResponse.BatchItemFailure.builder()
                    .withItemIdentifier(record.getKinesis().getSequenceNumber())
                    .build());
            }
        }

        return StreamsEventResponse.builder()
            .withBatchItemFailures(failures)
            .build();
    }
}

Kinesis Data Firehose

Firehose is not a stream — it's a fully managed delivery pipeline that automatically batches, optionally transforms, and delivers streaming data to destinations like S3, Redshift, or OpenSearch. Zero consumer code to write.

Firehose vs Data Streams

Kinesis Data Streams
Low latency (~200ms). Multiple independent consumers. Replay support. You write consumer code. Consumer manages checkpoints. For real-time processing where you need custom logic.
Kinesis Data Firehose
60s–900s buffer (near-real-time). One destination per Firehose. No consumer code — Firehose handles delivery automatically. For loading streaming data into S3/Redshift/OpenSearch/Splunk without building a pipeline.

SQS vs SNS vs EventBridge vs Kinesis

The most-asked question when it comes to AWS messaging. These services overlap but serve distinct purposes. A clear mental model prevents choosing wrong.

One-Line Summary Each

SQS — Work queue
"I have work to distribute. One worker processes each task. I need retry, DLQ, and backpressure."
SNS — Broadcast notification
"Something happened. Everyone who cares should know. I want fan-out to many consumers simultaneously."
EventBridge — Smart routing
"I want to react to events across AWS services and applications with content-based routing and rich patterns."
Kinesis — Real-time streaming log
"High-volume, ordered records. Multiple independent consumers. I need replay. Analytics pipeline."

FeatureSQSSNSEventBridgeKinesis

Decision Tree

1
Do you need to distribute work to competing workers?
Yes → SQS. Each message processed by exactly one worker. Retry, DLQ, scaling by queue depth. Classic work queue.
2
Do you need to notify multiple independent services about an event?
Simple fan-out (same message to all) → SNS. Complex routing (different services get different events based on content) → EventBridge.
3
Do you need to react to AWS service events?
Yes → EventBridge (default bus). CloudTrail, EC2 state, RDS events, CodePipeline — all land on the default bus automatically.
4
High volume data stream with multiple consumers or replay needed?
Yes → Kinesis Data Streams. Clickstream, IoT telemetry, log aggregation, event sourcing, any scenario needing independent consumers at different speeds or replay.
5
Just need to load streaming data into S3/Redshift/OpenSearch?
Yes → Kinesis Firehose. No consumer code. Automatic batching, format conversion, delivery. Near-real-time (60s+ buffer).

Interview Questions — Messaging

16 questions from beginner to architect. Messaging patterns are core to system design interviews.

Beginner

Why would you use SQS instead of calling a service directly over HTTP? ▼
Direct HTTP coupling means: (1) Caller waits for the callee to respond — if callee is slow, caller is slow. (2) If callee is down, caller fails. (3) If callee is overwhelmed, caller gets errors. (4) Traffic spikes directly hit the callee. SQS decouples them: (1) Caller sends to SQS and returns immediately — milliseconds instead of seconds. (2) If callee is down, messages queue up and are processed when it recovers — nothing is lost. (3) Callee processes at its own pace — natural rate limiting. (4) SQS absorbs traffic spikes — messages queue up and process when there's capacity. You accept a trade-off: the result is asynchronous (you won't know immediately if processing succeeded), but the system is more resilient, scalable, and performant.
What is the visibility timeout in SQS and what happens if it's too short? ▼
The visibility timeout is the period during which a message received by a consumer is hidden from other consumers. The purpose: prevent two consumers from processing the same message simultaneously. If too short: your consumer is still processing when the timeout expires. SQS makes the message visible again. Another consumer picks it up and starts processing. Now two consumers are processing the same message concurrently — double processing. For orders: two emails sent, two charges made, two inventory deductions. Fix: set the timeout longer than your maximum processing time. Formula: VisibilityTimeout = expected_max_processing_time × 6 for safety margin. For variable-duration processing: use ChangeMessageVisibility to extend dynamically while processing.
What is the difference between SNS and SQS? ▼
SQS (queue): One producer sends messages, one or more competing consumers each process a different message. Each message is consumed exactly once (by one consumer), then deleted. Use for: distributing work among workers, task queues, retries with DLQ. SNS (pub/sub): One publisher sends a message, EVERY subscriber gets their own copy. Use for: notifying multiple systems about an event. Fan-out. The canonical production pattern is SNS → SQS (fan-out): SNS broadcasts the event, multiple SQS queues subscribe (each service has its own queue), each service processes from its own queue at its own pace. This gives you: fan-out (SNS), retry + DLQ + rate-limiting (SQS), complete decoupling.
What is a Dead Letter Queue and why is it essential? ▼
A Dead Letter Queue (DLQ) is a separate SQS queue where messages are automatically moved when they fail to be processed successfully after N attempts (maxReceiveCount). Without a DLQ: a "poison pill" message (malformed data that always causes exceptions) loops in the queue forever, consuming retry attempts and potentially blocking other messages (especially in FIFO queues). Developers never know it's happening. With a DLQ: after 3 failures, the problematic message is isolated in the DLQ and stopped retrying. A CloudWatch alarm on DLQ depth fires immediately, alerting developers. Developers can inspect the message, understand what went wrong, fix the bug, then redrive the message back to the main queue for reprocessing. Nothing is ever silently lost. Production rule: every SQS queue that processes business-critical data must have a DLQ with a CloudWatch alarm.

Intermediate

When would you choose Kinesis over SQS? ▼
Kinesis over SQS when: (1) Multiple independent consumers need the same data: With SQS, each message goes to one consumer. With Kinesis, Consumer A, B, and C can all read the same records independently from their own position. (2) You need to replay data: SQS deletes messages after processing. Kinesis retains records for 24h–365 days. You can rewind to position 0 and reprocess. If a bug corrupted your database, replay from the stream to rebuild. (3) Order within a partition matters: Kinesis guarantees ordering per shard per partition key. SQS only guarantees FIFO in FIFO queues. (4) Real-time analytics pipeline: High-volume data where you need multiple analytics systems processing concurrently — clickstream, IoT, log aggregation. Choose SQS when: work distribution to competing workers, retry handling with DLQ, simpler setup, order doesn't matter, each task done by one worker.
Explain the SNS fan-out to SQS pattern and why it's used instead of direct Lambda subscriptions. ▼
SNS fan-out to SQS (SNS → multiple SQS queues → Lambda per queue) vs direct SNS → Lambda: Why SQS in the middle: (1) Durability: If the Lambda is throttled or has a cold start, messages queue up in SQS and aren't lost. Direct SNS → Lambda delivery failures go to DLQ but can be lost if retry exhausted. (2) Retry logic: SQS + Lambda has configurable retry with exactly the DLQ handling you need. Direct SNS → Lambda has limited retry control. (3) Rate control: SQS naturally limits how fast Lambda processes — the batch size and reserved concurrency control throughput. Protects downstream services from being overwhelmed. (4) Visibility: SQS metrics (depth, age, DLQ count) give clear operational visibility into each service's queue health. (5) Message filtering: SNS filter policies control which messages each SQS queue receives — each service only processes what it cares about. Use direct SNS → Lambda only for very simple, high-reliability functions where the Lambda is always available.
How do you make an SQS consumer idempotent? Why is it necessary? ▼
Idempotency means processing the same message twice produces the same result as processing it once. Necessary because SQS standard queues offer at-least-once delivery — a message may occasionally be delivered more than once. Implementation approaches: (1) Idempotency key in DynamoDB: Store the SQS messageId in DynamoDB with a TTL (7 days). Before processing, check if this ID exists. If yes: skip. If no: process and record it. (2) Conditional DB updates: Use database conditions to make updates idempotent. UPDATE orders SET status='SHIPPED' WHERE status='PACKED' — running this twice is safe; second run does nothing. (3) Upsert pattern: Use INSERT ON CONFLICT DO NOTHING or equivalent. Writing the same record twice = same result. (4) FIFO queue: For critical workflows, use SQS FIFO with message deduplication ID — SQS itself deduplicates within a 5-minute window, so duplicates never reach your consumer.

Advanced

What is the difference between SNS and EventBridge? When would you choose EventBridge over SNS? ▼
Both are pub/sub, but EventBridge is significantly more powerful: Filtering: SNS filters on message attributes only. EventBridge filters on the full JSON body using rich operators (numeric comparisons, prefix matching, anything-but, exists). AWS service events: EventBridge's default bus receives events from 200+ AWS services automatically (EC2 state changes, CloudTrail API calls, RDS events). SNS cannot. Schema registry: EventBridge discovers and versions event schemas, generates SDK code bindings. SNS has nothing. Replay: EventBridge can archive events and replay them. SNS: fire-and-forget. SaaS integration: EventBridge connects to Zendesk, Stripe, Datadog, etc. as event sources. Choose SNS when: Simple fan-out, SMS/mobile push delivery (EventBridge doesn't support these), maximum throughput (SNS handles more), lower per-event cost. Choose EventBridge when: Need to react to AWS service events, content-based routing on event body, schema governance, event replay, complex enterprise integration.
Design an event-driven architecture for an e-commerce platform where placing an order must trigger: payment processing (synchronous), inventory reservation, email confirmation, analytics recording, and fraud checking. ▼
Architecture: Synchronous path (must succeed for order to be placed): User → API Gateway → OrderService → PaymentService (direct HTTP, synchronous, 200ms timeout). Both complete → write order to Aurora → return orderId to user. Only these two are synchronous because order placement requires payment confirmation. Async fan-out (once payment confirmed): OrderService publishes OrderConfirmed event to EventBridge custom bus. Rules and targets: (1) Rule: all OrderConfirmed → SQS inventory-queue → Lambda InventoryReserver. DLQ for failures. (2) Rule: all OrderConfirmed → SQS email-queue → Lambda EmailSender. DLQ. (3) Rule: PHYSICAL orders only → SQS fraud-queue → Lambda FraudChecker. (4) Rule: all events → Kinesis Firehose → S3 (analytics data lake, Athena queries). Reliability: Each SQS queue has maxReceiveCount=3 and a DLQ with CloudWatch alarm. All Lambda consumers are idempotent (check eventId in DynamoDB before processing). FIFO queue for inventory reservation (prevent double reservation). EventBridge retry policy: 3 retries, 24h max age, DLQ for undeliverable. Monitoring: CloudWatch dashboard showing SQS queue depths, Lambda error rates, DLQ message counts. PagerDuty integration for DLQ alarms.
