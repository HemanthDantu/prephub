Module 8 · Section 0

Why API Gateway Exists

API Gateway is the front door to your backend services. Understanding what it replaces and what it adds makes every other section in this module click instantly.

What You'd Have to Build Without It

Imagine you have a Lambda function (or an ECS service) that processes orders. You want to expose it as an HTTPS API. Without API Gateway, you must build and maintain:

Infrastructure
An EC2/ECS server to receive HTTP requests, TLS certificate management, a load balancer for HA, auto-scaling that load balancer, monitoring and alarming on it.
Security middleware
Authentication (verify JWTs, validate API keys), rate limiting (throttle abusive callers), IP allowlisting, DDoS protection, WAF integration — all custom code in your app.
API management
URL routing (which path goes to which function), request validation, response formatting, versioning, CORS headers, custom domains, API documentation generation.

API Gateway provides all of this as a managed service, configured declaratively. Your Lambda function only sees processed, validated, authenticated requests — and returns plain JSON. API Gateway handles everything else.

What API Gateway Actually Does

Complete API Gateway request flow — every step
1
Client sends HTTPS request to your API: POST https://api.acme.com/v1/orders with JWT in Authorization header.
2
TLS termination: API GW terminates the TLS connection. Your backend never handles certificates.
3
Authentication: API GW validates the JWT against Cognito (or calls your Lambda Authorizer). If invalid → 401 Unauthorized immediately, Lambda never invoked.
4
Throttling check: Is this caller within their rate limit? If they've exceeded 1000 req/sec → 429 Too Many Requests, Lambda never invoked.
5
Request validation: Does the request body match the defined model? Required fields present? If not → 400 Bad Request, Lambda never invoked.
6
Lambda invocation: API GW transforms the HTTP request into a structured JSON event and invokes your Lambda synchronously. Waits for response.
7
Lambda processes: Your function does its work (reads DB, calls services) and returns a structured JSON response object.
8
Response transformation: API GW takes Lambda's response, sets HTTP status code, adds CORS headers, and sends the HTTP response to the client.
Key insight: Steps 2–5 are all handled by API Gateway before your Lambda is ever invoked. Unauthenticated requests, rate-limited clients, and invalid requests never cost you a Lambda invocation. Your Lambda only runs for valid, authenticated, rate-allowed requests. This is both a security and cost benefit.

What API Gateway Is Not

Not a load balancer for stateful services
For routing to ECS/EC2 backends, an ALB is simpler and cheaper for most cases. API Gateway shines for Lambda backends and APIs that need rich auth, throttling, and management features. For ECS microservices talking to each other internally, use ALB or Service Connect.
Not a message queue
API Gateway is synchronous by default — it waits for your Lambda to respond (max 29 seconds). For async workloads, have your Lambda immediately enqueue to SQS and return 202 Accepted. Don't use API Gateway's 29-second timeout for long processing.

Three API Types

API Gateway offers three fundamentally different API types. Each is designed for a different use case. Choose the wrong type and you'll either pay too much or miss features you need.

REST API — Feature-Rich, Full Control

The original and most powerful API Gateway type. Supports the complete feature set: custom request/response transformations using Velocity Template Language (VTL), API keys, usage plans, caching, request validation with models, WAF integration, private APIs in VPC. More expensive per request.

REST API — when to use
Use REST API when you need: request/response body transformation (change the shape of data between client and backend), API keys and usage plans for external developers, response caching (reduce Lambda invocations), private APIs accessible only from VPC, WAF integration for web application firewall, or AWS service proxy (call SQS, DynamoDB, Step Functions directly without Lambda).

Cost: $3.50/million requests (first 333M/month) in us-east-1

HTTP API — Simple, Fast, Cheap

Launched in 2020 as a simpler, lower-cost alternative. Up to 71% cheaper than REST API. Supports Lambda integration, JWT authorizers, OIDC, and basic CORS. Missing many REST API features but that's fine for most modern APIs that use Lambda for all logic.

HTTP API — when to use
Use HTTP API for: CRUD APIs backed by Lambda, APIs using JWT/OIDC authentication (Cognito, Auth0, etc.), microservices routing to Lambda or HTTP backends, APIs where you want minimal cost. This is the right choice for most new Lambda-backed APIs.

Cost: $1.00/million requests in us-east-1 (3.5× cheaper than REST)

WebSocket API — Persistent Bidirectional Connections

Manages persistent WebSocket connections. Clients connect once and maintain an open connection. The server can push messages to clients at any time. API Gateway handles the connection management; your Lambda receives events for connect, disconnect, and messages.

WebSocket API — when to use
Use for: real-time applications (live dashboards, collaborative editing), chat applications, live trading feeds, gaming, any use case where the server needs to push updates to clients without polling. Not for: standard request-response APIs (REST/HTTP API are better and cheaper).

Cost: $0.25/million connection minutes + $1.00/million messages

REST vs HTTP API — Which to Use?

The most common question when starting with API Gateway. The answer is almost always HTTP API for new projects — here's exactly why and when REST API is still the right choice.

Start with HTTP API

For most modern backends: your Lambda function receives a request, does some processing, and returns a response. You want Cognito JWT authentication, CORS, and Lambda routing. HTTP API does all of this for 3.5× less cost with simpler configuration.

Use REST API when you need
Request body transformation (VTL)
Response body transformation
API Keys + Usage Plans (external devs)
Response caching
Private API (VPC-only access)
Direct service integrations (SQS/DDB)
Request validation with JSON schemas
WAF integration (HTTP API also supports WAF now)
HTTP API is enough for
Lambda proxy integration (all logic in Lambda)
JWT auth (Cognito, Auth0, Okta, custom)
CORS (easier to configure)
Route to multiple Lambda functions
Private integrations via VPC Link
Low-latency proxy to HTTP backends
Auto-deploy on route changes
90% of real-world API use cases

The Decision in Practice

Decision Examples
What type to pick for common scenarios

Building a REST API for a mobile app (Cognito auth, Lambda functions): HTTP API. All auth, routing, and CORS work natively. Your Lambda handles all business logic.

Building a public API for third-party developers with rate limiting per customer: REST API. Need API keys and usage plans to assign different rate limits per customer/tier.

Receiving webhooks from Stripe and publishing to SQS (no Lambda needed): REST API with direct SQS integration. API GW receives the webhook and puts it in SQS directly — no Lambda cold starts, cheaper.

Internal microservice-to-microservice API: Neither. Use ALB (cheaper, faster, no per-request cost for ECS backends). API Gateway adds cost for internal traffic with no benefit.

Live chat feature / collaborative editor: WebSocket API. Needs persistent connections and server push.

Lambda Integration Deep Dive

The most used integration type. Understanding exactly what the event payload looks like and what your Lambda must return is essential — a mis-shaped response and clients get a mysterious 502.

Lambda Proxy Integration — What Gets Passed

In proxy integration (the default and recommended mode), API Gateway packages the entire HTTP request — method, path, headers, query params, body, auth context — into a single JSON event and passes it to Lambda. Your Lambda receives this structured object and must return a specific structured response. There's no magic — you see and control everything.

The Complete Event Object (REST API / HTTP API v1 format)

{
  "version": "1.0",
  "resource": "/users/{userId}",            // route template
  "path": "/users/alice",                  // actual request path
  "httpMethod": "GET",

  "headers": {
    "Authorization": "Bearer eyJhbGci...",
    "Content-Type": "application/json",
    "X-Forwarded-For": "203.0.113.15",    // client IP
    "Host": "api.acme.com"
  },

  "queryStringParameters": {
    "page": "2",
    "limit": "25"
  },

  "pathParameters": {
    "userId": "alice"                     // extracted from {userId} in route
  },

  "requestContext": {
    "authorizer": {
      "claims": {                           // filled by Cognito authorizer
        "sub": "user-uuid-123",
        "email": "alice@example.com",
        "cognito:groups": "[\"admin\",\"users\"]"
      }
    },
    "identity": {
      "sourceIp": "203.0.113.15"
    },
    "requestId": "abc-123-def",
    "stage": "prod"
  },

  "body": "{\"name\":\"Alice\",\"email\":\"alice@example.com\"}",
  // body is a STRING (even if JSON was sent) — you must parse it

  "isBase64Encoded": false
}

The Response Your Lambda Must Return

{
  "statusCode": 200,              // HTTP status code — required

  "headers": {                      // optional response headers
    "Content-Type": "application/json",
    "X-Request-Id": "abc-123"
  },

  "body": "{\"userId\":\"alice\",\"name\":\"Alice Smith\"}"
  // body MUST be a string — serialise your object to JSON first
}

Java Lambda Handler — Complete Working Example

public class UserHandler implements
    RequestHandler<APIGatewayProxyRequestEvent, APIGatewayProxyResponseEvent> {

    private static final ObjectMapper mapper = new ObjectMapper();
    private static final UserService service = new UserService();

    public APIGatewayProxyResponseEvent handleRequest(
            APIGatewayProxyRequestEvent event, Context ctx) {

        try {
            // Extract path parameter
            String userId = event.getPathParameters().get("userId");

            // Extract query parameters (nullable — use getOrDefault)
            Map<String,String> query = event.getQueryStringParameters();
            int page = query != null
                ? Integer.parseInt(query.getOrDefault("page", "1"))
                : 1;

            // Extract authenticated user from Cognito claims
            Map<String,Object> claims = event.getRequestContext()
                .getAuthorizer().getClaims();
            String callerEmail = (String) claims.get("email");

            // Parse request body (only for POST/PUT/PATCH)
            UserUpdateRequest body = null;
            if (event.getBody() != null) {
                body = mapper.readValue(event.getBody(), UserUpdateRequest.class);
            }

            // Execute business logic
            User user = service.getUser(userId);

            // Return success response
            return response(200, user);

        } catch (UserNotFoundException e) {
            return errorResponse(404, "USER_NOT_FOUND", e.getMessage());
        } catch (ValidationException e) {
            return errorResponse(400, "VALIDATION_ERROR", e.getMessage());
        } catch (Exception e) {
            log.error("Unexpected error", e);
            return errorResponse(500, "INTERNAL_ERROR", "An internal error occurred");
        }
    }

    private APIGatewayProxyResponseEvent response(int status, Object body) {
        try {
            return new APIGatewayProxyResponseEvent()
                .withStatusCode(status)
                .withHeaders(Map.of("Content-Type", "application/json"))
                .withBody(mapper.writeValueAsString(body)); // must be a String!
        } catch (JsonProcessingException e) {
            throw new RuntimeException(e);
        }
    }

    private APIGatewayProxyResponseEvent errorResponse(int status, String code, String msg) {
        return response(status, Map.of(
            "error", code,
            "message", msg,
            "timestamp", Instant.now().toString()
        ));
    }
}

Request & Response Transformation (REST API)

REST API's unique power: reshape requests before they reach your Lambda and responses before they reach the client — using Velocity Template Language (VTL) mapping templates. No Lambda code needed.

What Transformation Solves

Sometimes your backend expects a different format than what clients send. Or your backend returns more data than clients need (over-fetching). Or you want to add context (like the caller's identity) to the request body. Transformation lets you do this at the API Gateway layer — without a Lambda function.

Authentication Options

API Gateway supports multiple authentication mechanisms. Choosing the right one depends on who's calling your API — end users, other services, or external developers.

Four Authentication Methods

Cognito User Pool Authorizer
API GW validates the JWT token from Cognito User Pool. No code needed — just configure which Cognito pool to validate against. Returns claims (user's email, groups, custom attributes) to Lambda in the event context.

Use for: End-user authentication (web app, mobile app users logging in).
Lambda Authorizer
A Lambda function receives the token/header and returns an IAM policy (Allow/Deny). Maximum flexibility — validate any token format, call any auth service, implement custom logic.

Use for: Custom JWT validation, API key validation against a database, third-party auth providers, complex authorization rules.
IAM Authorization (AWS SigV4)
Callers must sign requests with AWS SigV4 signature. Only IAM identities (roles, users) can call. API GW validates the signature automatically.

Use for: Service-to-service calls within AWS where calling services have IAM roles. No JWT handling required.
API Keys (REST API only)
Static keys in the x-api-key header. Combined with usage plans to assign different rate limits per key.

Use for: External developer access where you want per-customer rate limiting. Not for authentication — keys are not user-specific and easy to leak. Use in combination with another auth method.

Choosing the Right Auth Method

ScenarioMethod
Mobile/web app users logging in via CognitoCognito Authorizer
Third-party OAuth (Auth0, Okta, Google)Lambda Authorizer or HTTP API JWT authorizer
Service A calling Service B within your AWS accountIAM auth (SigV4) or Lambda Authorizer
External developers using your public APIAPI Keys + usage plans + Cognito/Lambda auth
Internal admin API (no external access)IAM auth + VPC Link (only accessible inside VPC)
Webhook receiver (Stripe, GitHub)Lambda Authorizer (validate webhook signature)

Cognito Authorizer

The simplest and most common auth setup for end-user APIs. Cognito handles user sign-up, sign-in, MFA, and token issuance. API Gateway validates the tokens automatically.

How It Works End to End

Cognito Authentication Flow — from user login to API call
1
User logs in via your web/mobile app. App calls Cognito InitiateAuth with username and password.
2
Cognito validates credentials and returns three tokens: IdToken (user identity claims), AccessToken (API authorisation), RefreshToken (gets new tokens when expired).
3
Client stores tokens (usually in memory or secure storage). For every API call, the client includes the token in the Authorization: Bearer {idToken} header.
4
API Gateway receives the request, extracts the token, validates it against Cognito: checks signature (using Cognito's public keys), expiry, issuer, audience. No call to Cognito at this point — validation is local using Cognito's published JWKS.
5
If valid: API GW adds the decoded claims to the Lambda event's requestContext.authorizer.claims. Lambda can read the user's email, groups, and custom attributes directly.
6
If invalid or expired: API GW returns 401 Unauthorized immediately. Lambda is never invoked.

Setting Up Cognito Authorizer

# Step 1: Create Cognito User Pool (if not already existing)
aws cognito-idp create-user-pool \
  --pool-name myapp-users \
  --auto-verified-attributes email \
  --policies '{"PasswordPolicy":{"MinimumLength":8,"RequireUppercase":true}}'

  --user-pool-id us-east-1_abc123 \
  --client-name myapp-web-client \

Lambda Authorizer

Maximum flexibility: a Lambda function decides whether a request is allowed. Used for custom token validation, third-party auth providers, webhook signature verification, or any auth logic that doesn't fit a standard pattern.

How Lambda Authorizer Works

When a request arrives, API Gateway calls your authorizer Lambda before calling your main Lambda. The authorizer Lambda examines the token (or other headers) and returns an IAM policy document saying Allow or Deny. API Gateway caches the result (by token value) for a configurable TTL — so the authorizer Lambda isn't called on every request.

# Create Lambda Authorizer on REST API
aws apigateway create-authorizer \
  --rest-api-id abc123xyz \
  --name CustomJwtAuthorizer \
  --type TOKEN \                    # TOKEN = reads from Authorization header
  --authorizer-uri arn:aws:apigateway:us-east-1:lambda:path/functions/arn:aws:lambda:us-east-1:123:function:JwtAuthorizer/invocations \
  --identity-source method.request.header.Authorization \
  --authorizer-result-ttl-in-seconds 300   # cache valid result for 5 min
  # Same token = cached policy = authorizer not called again for 5 min
public class JwtAuthorizer implements
    RequestHandler<APIGatewayCustomAuthorizerEvent, APIGatewayCustomAuthorizerResponse> {

    public APIGatewayCustomAuthorizerResponse handleRequest(
            APIGatewayCustomAuthorizerEvent event, Context ctx) {

        String token = event.getAuthorizationToken();
        // token = "Bearer eyJhbGci..." — strip "Bearer " prefix
        String jwt = token.replaceFirst("^Bearer ", "");

        try {
            // Validate JWT — check signature, expiry, issuer
            DecodedJWT decoded = jwtValidator.validate(jwt);
            String userId = decoded.getSubject();
            String email = decoded.getClaim("email").asString();
            List<String> roles = decoded.getClaim("roles").asList(String.class);

            // Return ALLOW policy with context passed to downstream Lambda
            return buildPolicy(userId, "Allow",
                event.getMethodArn(),              // Allow the specific route called
                Map.of(
                    "userId", userId,
                    "email", email,
                    "roles", String.join(",", roles)
                ));

        } catch (JWTVerificationException e) {
            // Return DENY — API GW sends 403 Forbidden
            return buildPolicy("anonymous", "Deny",
                event.getMethodArn(), Map.of());
        }
    }

    private APIGatewayCustomAuthorizerResponse buildPolicy(
            String principalId, String effect,
            String methodArn, Map<String,Object> context) {

        Statement stmt = Statement.builder()
            .effect(effect)
            .resources(List.of(methodArn))
            .build();

        return APIGatewayCustomAuthorizerResponse.builder()
            .principalId(principalId)
            .policyDocument(PolicyDocument.builder().statements(List.of(stmt)).build())
            .context(context)      // passed to downstream Lambda in requestContext.authorizer
            .build();
    }
}

// Downstream Lambda reads authorizer context:
Map<String,Object> auth = event.getRequestContext().getAuthorizer();
String userId = (String) auth.get("userId");   // set by authorizer Lambda
String roles  = (String) auth.get("roles");

Webhook Signature Verification (e.g., Stripe, GitHub)

public class StripeWebhookAuthorizer implements
    RequestHandler<APIGatewayCustomAuthorizerRequestEvent, APIGatewayCustomAuthorizerResponse> {
    // REQUEST type authorizer — receives full request, not just token

    public APIGatewayCustomAuthorizerResponse handleRequest(
            APIGatewayCustomAuthorizerRequestEvent event, Context ctx) {

        String signature = event.getHeaders().get("stripe-signature");
        String body = event.getBody();
        String webhookSecret = getSecret("stripe/webhook-secret");

        if (verifyStripeSignature(signature, body, webhookSecret)) {
            return buildPolicy("stripe", "Allow", event.getMethodArn(), Map.of());
        }
        return buildPolicy("unknown", "Deny", event.getMethodArn(), Map.of());
    }
}

Throttling & Usage Plans

Rate limiting protects your backend from overload and enables fair usage. API Gateway has two levels of throttling — account-wide defaults and per-API, per-stage, per-key plans.

Throttling Levels

Account-Level (AWS default)
AWS sets a default of 10,000 requests/second across all APIs in your account per region, with a burst limit of 5,000. This is a hard limit — request a service quota increase if needed.
API Stage-Level (your limit)
Set per-stage limits on a specific API. Example: prod stage allows 2,000 req/sec, dev stage allows 100 req/sec. Prevents dev traffic from consuming prod's quota.
Usage Plan + API Key (per customer)
Assign API keys to usage plans with different rate limits. Basic tier: 10 req/sec. Premium tier: 500 req/sec. Enterprise: unlimited. Per-customer throttling.

Creating Usage Plans for External APIs

Two Throttling Concepts: Rate and Burst

## Rate (steady-state): requests per second sustained
## Burst: how many requests can spike above the rate simultaneously

## Example: rate=100, burst=500
# Normal operation: up to 100 req/sec handled fine
# Sudden burst of 500 requests: first 500 handled immediately (bucket full)
# After initial burst: back to 100/sec rate (token bucket refills at 100/sec)
# If 600 requests arrive simultaneously: 500 handled, 100 get 429

## This is the token bucket algorithm:
# Bucket size = burst limit
# Bucket refills at rate/second
# Each request takes one token
# No tokens = 429 Too Many Requests

CORS — Complete Guide

CORS (Cross-Origin Resource Sharing) is the most common source of confusion when building web APIs. Understanding exactly why it exists and how to configure it correctly saves hours of debugging.

Why CORS Exists — The Security Model

Browsers enforce a security policy: JavaScript on https://app.acme.com cannot make requests to https://api.acme.com unless api.acme.com explicitly says it's okay. Without this policy, a malicious website could make requests to your bank's API using your logged-in browser session (cross-site request forgery).

CORS is the mechanism for your API to say: "Yes, requests from https://app.acme.com are allowed." Your API must include specific Access-Control-* headers in every response.

The Two CORS Request Types

Simple Request (no preflight)
GET or POST with simple headers (Content-Type: application/json is fine). Browser sends the request directly and checks the response headers. If CORS headers are present and allow the origin: success. If missing: browser blocks the response (request was sent, but JS can't read the response).
Preflight Request (OPTIONS first)
DELETE, PUT, PATCH, or any request with custom headers (like Authorization). Browser first sends an HTTP OPTIONS request asking "can I do this?". Your API must respond to OPTIONS with CORS headers. Only then does the browser send the actual request. Two network round-trips.

Configuring CORS on HTTP API (Easiest)

# HTTP API: configure CORS at the API level — applies to all routes
aws apigatewayv2 update-api \
  --api-id abc123xyz \
  --cors-configuration '{
    "AllowOrigins": ["https://app.acme.com", "https://admin.acme.com"],
    "AllowMethods": ["GET", "POST", "PUT", "PATCH", "DELETE", "OPTIONS"],
    "AllowHeaders": ["Content-Type", "Authorization", "X-Api-Key", "X-Request-Id"],
    "ExposeHeaders": ["X-Request-Id", "X-Rate-Limit-Remaining"],
    "MaxAge": 86400,
    "AllowCredentials": true
  }'

# HTTP API handles OPTIONS requests automatically
# No Lambda invocation needed for preflight
# Response headers added automatically to all responses

# For development (allow any origin — NOT in production!):
"AllowOrigins": ["*"]

Configuring CORS on REST API (More Manual)

# REST API: must configure OPTIONS method on each resource AND Lambda must add headers

# Step 1: Enable CORS on resource (adds OPTIONS method via console or API)
# This creates an OPTIONS method with a MOCK integration that returns CORS headers

# Step 2: Lambda must include CORS headers in EVERY response
private static final Map<String, String> CORS_HEADERS = Map.of(
    "Access-Control-Allow-Origin", "https://app.acme.com",
    "Access-Control-Allow-Methods", "GET,POST,PUT,DELETE,OPTIONS",
    "Access-Control-Allow-Headers", "Content-Type,Authorization",
    "Access-Control-Allow-Credentials", "true"
);

private APIGatewayProxyResponseEvent response(int status, Object body) {
    return new APIGatewayProxyResponseEvent()
        .withStatusCode(status)
        .withHeaders(CORS_HEADERS)   // MUST include on every response
        .withBody(mapper.writeValueAsString(body));
}

VPC Link — Private Backends

VPC Link connects API Gateway to resources inside your VPC — ECS services, ALBs, EC2 instances — that don't have public IPs. Traffic never leaves AWS's private network.

Why VPC Link

Your ECS microservices run in private subnets with no public IPs — good security practice. But API Gateway is a public AWS service. How does it reach your private ECS service? VPC Link creates a private connection from API Gateway into your VPC via AWS PrivateLink.

Without VPC Link — public exposure required
Your ECS service needs a public-facing ALB, public IPs, and internet-accessible endpoints. Anyone can try to access your service. Not ideal for internal services.
With VPC Link — fully private
API Gateway → VPC Link → Private ALB (in your VPC) → ECS tasks (private subnets). Traffic never leaves AWS network. No public IPs needed on your ECS service. Internet cannot reach your service directly.
# Step 1: Create an NLB (Network Load Balancer) pointing to your ECS service
# VPC Link requires NLB for REST API, ALB or NLB for HTTP API
aws elbv2 create-load-balancer \
  --name myapp-nlb-private \
  --type network \
  --scheme internal \                # private, no public IP
  --subnets subnet-0app1a subnet-0app1b

# Step 2: Create VPC Link
aws apigateway create-vpc-link \
  --name myapp-vpc-link \
  --target-arns arn:aws:elasticloadbalancing:...:loadbalancer/net/myapp-nlb/abc123

# Returns: id=vpc-link-id (takes a few minutes to become AVAILABLE)

# Step 3: Configure integration to use VPC Link
# In REST API integration: type=HTTP_PROXY, connectionType=VPC_LINK
# Integration URI: http://internal.nlb.dns.name/{proxy}
# connectionId: vpc-link-id

# For HTTP API (simpler — ALB supported directly):
aws apigatewayv2 create-integration \
  --api-id abc123xyz \
  --integration-type HTTP_PROXY \
  --integration-method ANY \
  --integration-uri arn:aws:elasticloadbalancing:...:listener/app/myapp-alb/abc/def \
  --connection-type VPC_LINK \
  --connection-id vpc-link-id \
  --payload-format-version 1.0

Step 2: Lambda Handler — Handles All Methods

public class UserHandler implements
    RequestHandler<APIGatewayV2HTTPEvent, APIGatewayV2HTTPResponse> {

    private static final ObjectMapper json = new ObjectMapper();
    private static final UserService users = new UserService();

    public APIGatewayV2HTTPResponse handleRequest(
            APIGatewayV2HTTPEvent event, Context ctx) {

        // Extract authenticated user from JWT claims
        Map<String,String> claims = event.getRequestContext()
            .getAuthorizer().getJwt().getClaims();
        String callerId = claims.get("sub");
        String callerEmail = claims.get("email");

        // Route based on HTTP method
        String method = event.getRequestContext().getHttp().getMethod();

        return switch (method) {
            case "GET"    -> getUser(event);
            case "POST"   -> createUser(event, callerId);
            case "PUT"    -> updateUser(event, callerId);
            case "DELETE" -> deleteUser(event, callerId);
            default      -> error(405, "Method not allowed");
        };
    }

    private APIGatewayV2HTTPResponse getUser(APIGatewayV2HTTPEvent event) {
        String userId = event.getPathParameters().get("userId");
        try {
            User user = users.getById(userId);
            return ok(user);
        } catch (NotFoundException e) {
            return error(404, "User not found: " + userId);
        }
    }

    private APIGatewayV2HTTPResponse createUser(APIGatewayV2HTTPEvent event, String callerId) {
        try {
            CreateUserRequest req = json.readValue(event.getBody(), CreateUserRequest.class);
            validate(req);
            User created = users.create(req, callerId);
            return created(created, "/users/" + created.getId());  // 201 + Location header
        } catch (ValidationException e) {
            return error(400, e.getMessage());
        } catch (DuplicateException e) {
            return error(409, "Email already exists");
        }
    }

    // Helper methods
    private APIGatewayV2HTTPResponse ok(Object body) {
        return APIGatewayV2HTTPResponse.builder()
            .withStatusCode(200)
            .withHeaders(Map.of("Content-Type", "application/json"))
            .withBody(json.writeValueAsString(body))
            .build();
    }

    private APIGatewayV2HTTPResponse created(Object body, String location) {
        return APIGatewayV2HTTPResponse.builder()
            .withStatusCode(201)
            .withHeaders(Map.of(
                "Content-Type", "application/json",
                "Location", location))
            .withBody(json.writeValueAsString(body))
            .build();
    }

    private APIGatewayV2HTTPResponse error(int status, String message) {
        try {
            return APIGatewayV2HTTPResponse.builder()
                .withStatusCode(status)
                .withHeaders(Map.of("Content-Type", "application/json"))
                .withBody(json.writeValueAsString(Map.of("error", message)))
                .build();
        } catch (Exception e) { throw new RuntimeException(e); }
    }
}

Interview Questions — API Gateway

14 questions covering the breadth of API Gateway for backend developer and solutions architect interviews.

Beginner

What is API Gateway and what problems does it solve? ▼
API Gateway is a fully managed service that acts as the "front door" for your APIs. It handles: (1) TLS termination — your backend never manages certificates. (2) Authentication — validates JWT tokens, API keys, or calls a custom Lambda authorizer before your backend is invoked. (3) Throttling — rate limits requests per second at the account, API, stage, or per-key level. Invalid, unauthenticated, and throttled requests never reach your Lambda. (4) Routing — maps HTTP methods and paths to specific backend integrations. (5) CORS — handles cross-origin request headers. (6) Monitoring — built-in CloudWatch metrics (latency, error rates, request counts). Without API Gateway, you'd build and maintain all of this yourself in your application code.
What is the difference between HTTP API and REST API in API Gateway? ▼
HTTP API: Simpler, cheaper ($1/million requests vs $3.50/million). Supports Lambda proxy integration, native JWT/OIDC authorizers, basic CORS, and VPC Link. Missing features: request/response body transformation, API keys with usage plans, response caching, WAF integration (HTTP API now has WAF), private VPC APIs. Best for: Lambda-backed APIs with JWT auth — covers 90% of real use cases. REST API: Full-featured, more expensive. Adds: request/response transformation using VTL mapping templates, API keys with usage plans (per-customer rate limiting), response caching, private VPC-only APIs, direct AWS service integrations (call SQS/DynamoDB without Lambda). Use REST API when you specifically need those advanced features.
What must a Lambda function return when invoked by API Gateway proxy integration? ▼
Lambda must return a JSON object with: (1) statusCode (integer) — the HTTP status code, required. (2) headers (object, optional) — response headers like Content-Type, CORS headers. (3) body (string, required for responses with content) — the response body MUST be a string. If your body is an object, you must serialise it to JSON first. If body is missing, API GW returns no body. The #1 cause of 502 Bad Gateway is returning a non-string body (returning an object directly instead of calling JSON.stringify/ObjectMapper.writeValueAsString) or missing statusCode entirely.
What is CORS and why do you need to configure it on API Gateway? ▼
CORS (Cross-Origin Resource Sharing) is a browser security mechanism. When JavaScript on https://app.acme.com calls https://api.acme.com, the browser requires api.acme.com to explicitly permit this via Access-Control-* response headers. Without CORS headers, the browser blocks the response even though the request was sent and processed. API Gateway must: (1) Handle OPTIONS preflight requests (browser sends OPTIONS before PUT/DELETE/custom-header requests to check permissions). (2) Include Access-Control-Allow-Origin (what origins are allowed), Allow-Methods, and Allow-Headers in every response. HTTP API handles this automatically if you configure CORS at the API level. REST API requires configuring an OPTIONS method on each resource AND your Lambda must include CORS headers in every response it returns.

Intermediate

How does Cognito Authorizer work with API Gateway? Walk through the full flow. ▼
(1) User logs in via Cognito (web/mobile app calls InitiateAuth). Cognito returns an IdToken (JWT). (2) Client stores the IdToken and includes it in every API request: Authorization: Bearer {idToken}. (3) Request arrives at API Gateway. API GW extracts the token from the Authorization header. (4) API GW validates the JWT locally using Cognito's published public keys (JWKS endpoint — fetched once and cached). Validates signature, expiry, issuer, and audience. No call to Cognito for each request — validation is cryptographic. (5) If valid: API GW decodes the token, adds all claims (sub, email, groups, custom attributes) to the Lambda event's requestContext.authorizer.claims. Invokes Lambda. (6) Lambda reads claims directly from the event — no JWT parsing needed. Can implement fine-grained authorization (only admin group can DELETE). (7) If invalid/expired: API GW returns 401 immediately. Lambda never invoked.
When would you use a Lambda Authorizer instead of a Cognito Authorizer? ▼
Use Lambda Authorizer when: (1) Third-party auth provider (Auth0, Okta, Google) — need to call their token introspection endpoint or validate against their JWKS. (2) Custom token format — your tokens aren't standard JWTs (legacy systems, proprietary formats). (3) Webhook signature verification — verify Stripe, GitHub, or other webhook signatures using HMAC. The authorizer checks the webhook signature header. (4) Database-backed API keys — validate API keys against your own database (not Cognito), check if key is active, check subscription tier. (5) Complex authorization rules — multiple factors beyond a JWT claim (check subscription in DB, verify IP whitelist, check feature flags). Use Cognito Authorizer when: you're already using Cognito for user management and just need JWT validation — simpler, no extra Lambda, automatic result caching.
What is API Gateway throttling? What's the difference between rate and burst? ▼
Throttling is rate limiting — API Gateway rejects requests that exceed defined limits with 429 Too Many Requests. Two parameters: Rate (steady-state): sustained requests per second the API can handle. Example: 100 req/s. Burst: maximum simultaneous requests before throttling kicks in. Uses the token bucket algorithm: burst limit is the bucket size (tokens available for spikes), rate is how fast tokens refill. Example: rate=100, burst=500 means: up to 500 simultaneous requests are fine, then it throttles at 100/sec. With a sudden spike of 700 requests: first 500 handled (burst), next 100 throttled with 429. Three levels: AWS account default (10,000 req/s, 5,000 burst — shared across all APIs), API stage level (your specific limits), and usage plan level (per-API-key limits for external developers).
What is a stage in API Gateway? How would you use stages for dev/staging/prod environments? ▼
A stage is a named snapshot of your API configuration with its own URL, settings, and stage variables. Each stage gets its own URL: api-id.execute-api.region.amazonaws.com/{stage}/. For dev/staging/prod: create three stages. Each stage has stage variables that parameterise the Lambda alias. Stage variable lambdaAlias=dev → calls Lambda:dev alias (latest code, may be unstable). lambdaAlias=staging → Lambda:staging (RC build). lambdaAlias=prod → Lambda:prod (tested release). Same API definition, different Lambda targets. Stage-level throttle settings: prod stage gets 2,000 req/s, dev gets 100 req/s (prevents dev testing from consuming prod quota). Stage-level logging: prod logs at ERROR level only, dev logs everything. Custom domains: api.acme.com maps to prod stage. dev-api.acme.com maps to dev stage. Deployments are promoted: changes go dev → staging → prod via API GW deployments (or CI/CD pipeline).

Advanced

Design an API Gateway architecture for a B2B SaaS platform where different enterprise customers need different rate limits and you need to track usage per customer. ▼
Use REST API with API keys and usage plans: (1) Create usage plan tiers: Starter (100 req/day, 5/sec), Professional (10,000/day, 50/sec), Enterprise (unlimited, 500/sec). (2) For each onboarded enterprise customer, create an API key (aws apigateway create-api-key). Associate the key with their usage plan (create-usage-plan-key). The customer includes their key in x-api-key header. (3) API Gateway automatically enforces rate limits per key, rejects with 429 when exceeded. (4) Usage tracking: API Gateway logs to CloudWatch with API key context. Query CloudWatch Logs Insights to get per-key usage. Or enable detailed metrics per API key. (5) Lambda receives the API key in the event (event.requestContext.identity.apiKeyId) — use this for per-customer data isolation. (6) For auth: add Cognito or Lambda Authorizer on top of API keys. API keys alone aren't auth — they're rate limiting identifiers. (7) Custom domain: api.acme.com with path mappings per API version. (8) WAF integration: block malicious traffic patterns, geographic restrictions, SQLi/XSS protection at the API GW level.
Your API is returning 502 errors. What do you check? ▼
502 Bad Gateway means API Gateway received a response from Lambda that it couldn't parse into a valid HTTP response. Check in this order: (1) Lambda response format: Does your Lambda return an object with statusCode (integer), headers (object), and body (STRING)? The most common mistake: returning a plain object or list as body instead of serialising to JSON string first. (2) Lambda timeout: API Gateway has a 29-second integration timeout. If Lambda takes longer: 504 (not 502). But a timeout can sometimes manifest as 502. Check Lambda execution duration in CloudWatch Logs. (3) Lambda execution errors: If Lambda throws an unhandled exception, API GW gets an error response. Check CloudWatch Logs for the Lambda function — look for stack traces. (4) Lambda concurrency limits: If Lambda hits reserved concurrency = 0 or account limit, it returns a throttle response, which API GW turns into 502 or 503. Check Lambda throttle metrics. (5) VPC connectivity: If Lambda is in VPC and tries to pull code from ECR or access internet, check ENI limits, subnet routing, security groups. (6) Enable API Gateway execution logging to CloudWatch — it shows the exact Lambda response that caused the 502.
