IAM Fundamentals

IAM is the identity and authorization layer behind almost every safe AWS design decision.

🧒 ELI5

Badges and keycards

In a big office, not everyone should open every room. A janitor, accountant, and visitor need different badges, even though they all walk through the same building.

If you hand out master keys to everyone just because it is easier, one mistake can affect the whole office.

IAM exists so cloud access works like properly scoped badges instead of universal master keys.

🔗 ANALOGY

Hotel front desk and temporary room keys

A hotel does not issue one permanent master key to every guest. It hands out temporary keys that only open the right room for the right time period.

AWS roles exist for the same reason: short-lived, scoped access is safer than long-lived shared credentials.

Technical

What it means in real code

Definition: IAM defines who or what can call AWS APIs and what actions they are allowed to perform against which resources.

A request is evaluated against identity-based and resource-based policies, plus explicit denies, conditions, and service control boundaries if present. Roles are assumed identities that produce temporary credentials, which is why EC2, ECS, and Lambda commonly use roles instead of embedded access keys.

In production, IAM is one of the highest-leverage places to reduce blast radius. Least privilege, temporary credentials, and explicit scoping make credential leaks and operator mistakes far less dangerous.

Why it matters

  • Reduces blast radius
  • Supports auditable, policy-driven access
  • Removes the need to hard-code long-lived secrets in applications

Common pitfall

  • Using the root account for daily work
  • Granting Action:* on Resource:*
  • Embedding access keys in code or AMIs
Least privilegeRole assumptionPolicy conditionsLong-lived keys
Authenticate→Assume role or use identity→Evaluate policies→Allow or deny request

Interview-ready view

💬 Model Interview Answer

IAM is the access control layer for AWS. In interviews I usually say users are mainly for humans, roles are mainly for workloads, and policies define what actions are allowed on which resources. The common gotcha is people using long-lived access keys in apps instead of IAM roles, which creates secret rotation and leak risk they did not need to accept.

Identity typeBest use
IAM userHuman access with MFA and limited use of access keys
IAM roleWorkload or federated identity with temporary credentials
PolicyStatement of allowed or denied API actions
❌ Wrong: static credentials inside the app
export AWS_ACCESS_KEY_ID=AKIA...
export AWS_SECRET_ACCESS_KEY=very-secret
java -jar app.jar
✅ Right: attach a role to the workload
# EC2 / ECS / Lambda receives temporary credentials automatically
# Policy example:
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject"],
      "Resource": "arn:aws:s3:::reports-bucket/*"
    }
  ]
}

Temporary role credentials remove secret distribution from the application startup path.

⚠️ Warning

Over-permissioning is the quiet IAM failure mode

The most dangerous IAM mistake is broad permissions that “just make it work.” They are easy to forget and hard to reason about later. Scope actions and resources tightly, prefer roles over keys, and use explicit conditions when access should depend on tags, source VPC endpoints, or principals.

Mini takeaway
  • Prefer roles for workloads
  • Use least privilege
  • Enable MFA for humans
  • Avoid wildcard permissions unless truly justified

EC2 Compute Basics

EC2 is the core “rent a VM” model in AWS, but the interview value comes from understanding when that flexibility is helpful and what you must manage yourself.

🧒 ELI5

Renting an apartment instead of a hotel room

If you rent an apartment, you have more freedom to arrange things how you want, but you also have to manage more yourself. You choose the furniture, handle repairs, and keep track of the keys.

EC2 gives that same flexibility compared with more managed platforms. You control the operating system and runtime, but you also own patching, capacity, and health.

That is why EC2 is powerful but never “free” from an operational perspective.

🔗 ANALOGY

Leasing a car fleet

A company that leases cars can choose exactly which models it wants, but it also has to track maintenance, fuel, and replacement schedules.

With EC2, instance families, autoscaling groups, and AMIs are your fleet management tools.

Technical

What it means in real code

Definition: EC2 provides virtual machines with configurable CPU, memory, storage, networking, and boot images so you can run general-purpose workloads on AWS-managed infrastructure.

You launch instances from AMIs, select an instance family and size, attach security groups and storage, then often place them behind a load balancer or in an Auto Scaling Group. Unlike Lambda or many PaaS offerings, the guest OS, patching strategy, and many runtime details remain your responsibility.

In production, EC2 matters when you need OS-level control, long-running processes, special networking, or workloads that do not fit serverless or container platforms cleanly. The trade-off is that you gain flexibility while inheriting more operations work.

Why it matters

  • Useful for custom runtime and OS control
  • Supports long-lived services and background workers
  • Integrates cleanly with load balancers and autoscaling groups

Common pitfall

  • One large instance with no scaling plan
  • SSH-ing into instances for manual fixes
  • No AMI or bootstrap standardization
AMI + autoscalingSecurity groupsUser dataSnowflake server
Choose AMI→Launch instance or ASG→Attach security/networking→Monitor and replace

Interview-ready view

💬 Model Interview Answer

I explain EC2 as the flexible compute primitive: you rent VMs, choose the OS and instance size, and build the operational model you need. It is a good fit when you need more control than serverless or managed platforms provide. The gotcha is that teams sometimes treat EC2 instances as pets and patch them manually instead of rebuilding from AMIs and replacing them.

Decision pointTypical guidance
Need full OS controlEC2 is reasonable
Need zero-server managementPrefer a more managed platform
Need elasticityUse Auto Scaling Groups and load balancing
❌ Wrong: manual pet server workflow
# Launch one EC2 instance, SSH in, install packages by hand,
# edit config manually, and hope no one terminates it.
✅ Right: replaceable instance pattern
# Build an AMI or use startup automation
# Place instances in an Auto Scaling Group behind an ALB
# Use user data or configuration management for bootstrapping

The right pattern treats instances as replaceable capacity rather than unique snowflakes.

⚠️ Warning

Manual instance repair creates hidden production debt

The biggest EC2 misunderstanding is thinking operational maturity comes later. If your instances are patched and configured manually, disaster recovery and scaling become human-memory exercises. Standardize bootstrapping, monitor health, and prefer replacement over manual repair whenever possible.

Mini takeaway
  • Use AMIs or automated bootstrap
  • Prefer autoscaling over one big server
  • Standardize security groups and instance profiles
  • Treat instances as replaceable

S3 Storage Essentials

S3 is object storage, not a general POSIX filesystem, and that difference matters in both design and interview answers.

🧒 ELI5

Warehouse shelves with labels

Imagine a giant warehouse where every box has a label and you retrieve boxes by label rather than by walking folders like a normal office cabinet.

You can store huge amounts safely and cheaply, but the warehouse is designed for labeled objects, not for the kind of tiny file-editing patterns you might do on your laptop.

That is why S3 is fantastic for static assets, backups, logs, and data files, but not a drop-in replacement for a local disk.

🔗 ANALOGY

Package shipping center

A shipping center is excellent at holding and moving packages identified by tracking numbers. It is not designed for you to open a package in place, change one line, and save it back like a text document.

S3 works the same way: whole-object operations are natural, filesystem semantics are not.

Technical

What it means in real code

Definition: S3 is highly durable object storage where data is stored as objects inside buckets and addressed by keys rather than mounted as a normal block or file device.

Each object has metadata, a key, and content. Access is controlled by IAM or bucket policies, and common patterns include pre-signed URLs, lifecycle rules, versioning, and event notifications to trigger downstream processing when objects are created or changed.

In production, S3 is a default answer for static files, data exports, logs, backups, and artifacts because it is durable, scalable, and cheap relative to always-on disks. The gotcha is forgetting that S3 APIs and consistency semantics are object-oriented, not POSIX-style shared-disk behavior.

Why it matters

  • Excellent durability and scale for object workloads
  • Supports event-driven workflows and lifecycle rules
  • Pairs well with CloudFront, backups, and data pipelines

Common pitfall

  • Trying to use S3 as a low-latency database
  • Assuming it behaves like a mounted local filesystem
  • Overexposing buckets or objects publicly
VersioningLifecycle rulesPre-signed URLsPublic bucket by accident
Write object→Store in bucket→Authorize read/write→Optionally trigger downstream event

Interview-ready view

💬 Model Interview Answer

I usually summarize S3 as durable object storage, not a general filesystem. It is ideal for assets, backups, logs, and large files that you access by key. The trade-off is that you design around object operations and permissions, and the gotcha is teams accidentally making buckets public or building filesystem-style assumptions into application code.

FeatureTypical use
VersioningProtect against accidental overwrite or delete
Lifecycle policyMove or expire old objects
Pre-signed URLGrant temporary controlled download/upload access
❌ Wrong: treating S3 like a shared local disk
# App expects to append lines to the same file constantly
# and assumes normal file locking semantics on S3 paths.
✅ Right: object-centric usage
# Example pattern:
# 1. Upload generated report as one object
# 2. Store the object key in the database
# 3. Return a pre-signed URL for download

This pattern matches S3 strengths: whole-object storage, simple retrieval, and controlled access.

⚠️ Warning

Public exposure is the easiest S3 mistake to regret

The dangerous misunderstanding is assuming bucket privacy is obvious and permanent. Public ACLs, bucket policies, or sloppy URL-sharing can expose sensitive data quickly. Block public access by default, use least privilege, and prefer pre-signed URLs when humans or clients need temporary object access.

Mini takeaway
  • Think in objects and keys
  • Use versioning and lifecycle rules
  • Guard public access carefully
  • Use pre-signed URLs for temporary access
Quiz

Correct answer: BB is correct because S3 is object storage optimized for durable, scalable object workloads. A is a database problem, C is a cache problem, and D confuses object storage with block or filesystem storage.

VPC & Networking Basics

This section gives the interview-level networking vocabulary behind secure cloud deployments: subnets, routing, and ingress/egress boundaries.

🧒 ELI5

Neighborhood roads and gated areas

A city has neighborhoods, roads, gates, and rules about who can enter which area. A school bus route is different from a delivery truck route, even though both move through the same city.

A VPC is your private cloud neighborhood. Subnets divide it into zones, route tables decide traffic paths, and security groups act like gate rules at each building.

Without that structure, everything sits on the same road and security becomes guesswork.

🔗 ANALOGY

Office building floors and reception desk

Sensitive teams sit on different floors, the reception desk controls entry, and hallways direct people to the right rooms.

That is the mental model for public and private subnets, internet gateways, and internal-only services.

Technical

What it means in real code

Definition: A VPC is an isolated virtual network in AWS where you define IP ranges, subnets, routes, and traffic controls for your resources.

Public subnets typically have routes to an internet gateway, while private subnets do not and may reach outbound internet through NAT. Security groups are stateful virtual firewalls attached to resources, and network ACLs operate at the subnet boundary with stateless rules.

In production, VPC design matters because exposure mistakes happen at the network edge long before application code can save you. Placing databases in private subnets, limiting inbound ports, and understanding egress paths are foundational cloud hygiene.

Why it matters

  • Separates internet-facing and internal resources
  • Supports least-exposed network design
  • Makes routing and inspection explicit

Common pitfall

  • Putting databases in public subnets without need
  • Opening security groups broadly to 0.0.0.0/0
  • Confusing security groups with route tables
Private subnetSecurity groupNAT egressOpen world ingress
Define CIDR→Create subnets→Attach routes/gateways→Apply security groups

Interview-ready view

💬 Model Interview Answer

My short version is that a VPC is the private network boundary for your AWS resources. I usually explain public versus private subnets and mention that security groups are stateful allow-lists attached to instances or services. The common gotcha is thinking a security group can make up for a bad subnet or route design when those are separate layers of the network model.

ComponentRole
SubnetIP range and routing domain inside the VPC
Security groupStateful traffic filter attached to a resource
Internet gatewayEnables public internet routing for suitable subnets
❌ Wrong: overexposed network design
# Web app and database both placed in a public subnet
# Security group allows 0.0.0.0/0 to ports 80, 443, 5432
✅ Right: layered network boundary
# ALB in public subnet
# App instances in private subnet
# Database in private subnet
# Security groups allow only ALB -> app and app -> db traffic

The improved design narrows exposure and preserves a clear trust boundary between internet traffic and internal data services.

⚠️ Warning

Public reachability is often accidental, not intentional

The networking trap is focusing only on inbound web ports and forgetting the rest of the topology. Put public entry points where they belong, keep data services private, and review routes, subnet placement, and security groups together rather than one at a time.

Mini takeaway
  • Separate public and private layers
  • Keep databases private by default
  • Use security groups as scoped allow-lists
  • Review routes and exposure together
Quiz

Correct answer: BB is correct because it limits public exposure to the intended entry point while keeping stateful backend resources private. A increases risk unnecessarily. C is wrong because private IP space alone does not replace traffic controls. D ignores the fact that network boundaries and identity controls solve different problems.

Observability & Cost Basics

Good AWS usage is not only about launching resources; it is also about knowing what is happening and what it is costing.

🧒 ELI5

Driving with a dashboard

A driver needs a speedometer and fuel gauge, not just an engine. Otherwise the car may still move, but the trip becomes risky and expensive.

Cloud systems are the same: if you do not measure latency, errors, and spend, you are driving blind.

Observability and cost visibility keep cloud architecture from quietly becoming both unreliable and overpriced.

🔗 ANALOGY

Utility bill plus smoke alarm

A utility bill tells you what the house costs to run, and a smoke alarm tells you when something is going wrong right now. You need both to manage the home responsibly.

CloudWatch and cost tools fill those two roles for AWS workloads.

Technical

What it means in real code

Definition: Observability in AWS means collecting logs, metrics, traces, and alarms, while cost basics means understanding which resources drive spend and how to attribute and control that usage.

CloudWatch captures metrics and logs, alarms evaluate thresholds, and tagging plus cost reports help attribute spend by team or environment. The engineering discipline is to define what “healthy” means in numbers and what budget or utilization signals should trigger action.

In production, teams that skip observability and cost hygiene often discover problems only after customer impact or billing shock. Basic alarms, structured logs, and tags are low-effort guardrails with large payoff.

Why it matters

  • Enables faster incident detection
  • Supports capacity and cost planning
  • Creates accountability through tagging and alarms

Common pitfall

  • No alarms on error rate or CPU
  • Resources without ownership tags
  • Keeping oversized instances forever because nobody reviews utilization
CloudWatch alarmTaggingRetention settingsNo cost ownership
Emit metrics/logs→Set alarms→Review utilization/spend→Tune or clean up resources

Interview-ready view

💬 Model Interview Answer

I usually keep this answer simple: if you cannot see health and cost, you do not really control the platform. I mention CloudWatch metrics and alarms for operational visibility and tagging plus regular reviews for cost hygiene. The gotcha is that cost optimization is not only a finance task; right-sizing and cleanup are engineering decisions too.

SignalWhy it matters
LogsExplain what happened during failures
Metrics and alarmsSurface unhealthy behavior early
Tags and cost reportsShow who owns and pays for what
❌ Wrong: deploy and forget
# Launch instances, buckets, and load balancers
# No alarms, no tags, no log retention policy, no budget alerts
✅ Right: minimal visibility baseline
# Example baseline:
# - Emit application logs to CloudWatch
# - Alarm on 5xx rate, CPU, and queue depth
# - Tag resources with service, environment, and owner
# - Review rightsizing and idle resources regularly

A minimal baseline dramatically reduces both blind spots and mystery spend.

⚠️ Warning

Untagged resources become orphaned costs quickly

The operational mistake here is treating tagging and alarms as optional cleanup. If nobody owns a resource and no alert fires when it misbehaves, the cloud account slowly accumulates both hidden risk and hidden spend. Start with a small baseline and enforce it consistently.

Mini takeaway
  • Ship logs and metrics intentionally
  • Alarm on a few meaningful signals
  • Tag resources with owner and environment
  • Review spend and utilization regularly
Quiz

Correct answer: BB is correct because it combines runtime visibility with ownership and cost attribution. A is incomplete because logs without metrics and alarms slow detection. C just hides issues by spending more. D is too late and too narrow to be operationally effective.
