Three Layers — The Mental Model First

Docker, Kubernetes, and Auto Scaling Groups feel confusing together because they operate at different levels. Once you understand the three distinct layers they each live on, the whole picture snaps into focus.

The Confusion Explained

People hear: "We run Kubernetes on AWS with auto scaling." And wonder: does Kubernetes scale things? Does AWS scale things? What's Docker doing? Who's in charge? The answer: they each handle a completely different concern at a different layer. They don't compete — they compose.

🐳 Layer 1 — PACKAGING: Docker
Solves: "My app works on my machine but not on the server."
Packages your application + all its dependencies into a portable, isolated unit called a container. Runs identically everywhere — laptop, CI server, production.
Question it answers: "How do I package and run my application consistently?"
↑ gives containers to ↓
☸ Layer 2 — ORCHESTRATION: Kubernetes or ECS
Solves: "I have 50 containers. How do I run them, keep them healthy, balance traffic, deploy updates, and restart failed ones — automatically?"
Manages the lifecycle of containers: scheduling, health monitoring, scaling containers up/down, rolling deployments.
Question it answers: "How do I run and manage many containers reliably?"
↑ deploys containers onto ↓
🖥 Layer 3 — INFRASTRUCTURE: EC2 + ASG (or Fargate)
Solves: "I have 100 containers to run. Where do the actual physical/virtual machines come from? How do I provision more servers when load increases?"
Provides the compute capacity that the orchestrator places containers onto.
Question it answers: "What machines are the containers actually running on, and how do I scale the fleet?"
The core insight: Kubernetes/ECS scales containers (the application layer). ASG scales EC2 instances (the infrastructure layer). These are different things. You might go from 5 containers to 50 containers without adding any EC2 instances (if existing instances have spare capacity). And you might add EC2 instances without changing the container count (if containers need more headroom). Fargate removes Layer 3 entirely — you never think about machines.

The Analogy That Makes It Click

Analogy — Restaurant Chain
Docker = Standardised meal kits. Orchestration = Restaurant manager. Infrastructure = Kitchen tables/stations.

Docker (packaging): Each dish is prepared as a sealed, standardised meal kit (container image). The kit has everything needed — ingredients, recipe, tools. The chef (your app) works the same way whether in London or New York because everything they need is in the kit. No "it works in the test kitchen but not the main kitchen" problem.

Orchestration (Kubernetes/ECS): The restaurant manager decides which chef works which station, ensures every station is staffed, notices when a chef is sick and calls in a replacement, handles rush hour by opening more stations, manages shift rotations during deployments.

Infrastructure (EC2/ASG): The physical kitchen stations themselves — tables, stoves, equipment. When demand is high, you rent more kitchen space. When quiet, you close some stations to save money. The manager doesn't care about this — they just need stations to exist.

Fargate: A catering company that provides kitchen stations on demand. "I need 5 stations now, 50 in an hour." The stations just appear. No building, no maintenance — you just use them.

The Four Paths — What "Using Containers on AWS" Actually Means

Path A: ECS + Fargate (most common for new projects)
AWS manages everything below the container. You define your container, resources, and desired count. AWS provisions, scales, and manages the infrastructure automatically. Zero server management.
Path B: EKS + Fargate
Kubernetes orchestration (portable, standard API) + Fargate infrastructure (no server management). Best of both worlds: K8s portability without EC2 cluster maintenance.
Path C: EKS + EC2 + ASG (maximum control)
Kubernetes orchestration + your own EC2 fleet managed by Auto Scaling Groups. Full control over instance types, placement, Spot Instances. Most complex to operate.
Path D: ECS + EC2 + ASG
ECS orchestration + your own EC2 fleet. Middle ground: AWS-native simplicity with EC2 cost optimisation (Spot Instances, reserved pricing, GPU instances).

The Problem Docker Solves

"It works on my machine" — the most infamous phrase in software development. Understanding why this problem exists makes Docker's solution obvious.

Why "Works on My Machine" Happens

Your Spring Boot app needs: Java 21, specific JVM flags, environment variables pointing to your local database, a config file at a specific path, and a specific version of a native library. Your colleague has Java 17, different JVM flags, different env vars, and a newer version of the library. The app behaves differently — sometimes doesn't run at all.

In production: the server has its own Java version, its own library versions, a different OS, different environment variables. Every deployment is a game of "what's different this time?"

Before Docker — Environment Dependency Hell
Developer machine:
  Java 21, Spring Boot 3.2
  MySQL 8.0 on localhost
  .env file with local configs
  /opt/ssl-certs/ (local path)

CI server:
  Java 17 (wrong version!)
  No MySQL (mock only)
  Different config location

Production:
  Java 11 (very wrong!)
  MySQL on RDS
  Secrets in /etc/app.conf
  SSL certs in /etc/ssl/

Every environment is different.
Deployment = debugging environment
differences. "Works on my machine"
is a constant reality.
With Docker — Identical Environment Everywhere
Docker image contains:
  Java 21 (exact version)
  Your compiled app
  All dependencies
  Config file structure
  SSL cert locations
  Everything needed

Developer machine: runs the image
CI server: runs the image
Production: runs the image

The environment IS the image.
The image is identical everywhere.
"Works on my machine" = works
everywhere the image runs.

Container vs Virtual Machine

Containers are often compared to VMs but they work very differently. Understanding this distinction matters because it explains why containers start in seconds, not minutes.

VM vs Container — What's Actually Running
Virtual Machine
Your App (Spring Boot)
Guest OS (full Ubuntu)
Hypervisor (VMware/KVM)
Physical Hardware
Startup: 30–90 seconds
Image size: 1–10 GB
Isolation: hardware-level
Container
Your App (Spring Boot)
App libs (JRE + deps only)
Container Runtime (Docker)
Host OS kernel (shared!)
Physical Hardware
Startup: 0.5–3 seconds
Image size: 50 MB – 500 MB
Isolation: OS-level namespaces

Containers share the host OS kernel — they don't boot an entire OS. This is why they start so fast and are so much smaller. The trade-off: containers are isolated at the OS process level (namespaces, cgroups), not the hardware level. VMs have stronger isolation for multi-tenant environments; containers are slightly less isolated but vastly more efficient.

AWS uses Firecracker microVMs for Lambda and Fargate containers. A microVM is a middle ground — it boots a minimal Linux kernel (like a container) but uses hardware virtualisation (like a VM). Sub-second startup, near-zero overhead, yet true hardware isolation. This is how AWS can safely run your containers alongside other customers' containers on the same physical hardware.

How Docker Actually Works

Images, containers, layers, registries — the mechanics of Docker that let you build once and run anywhere.

Image vs Container — The Fundamental Distinction

Image — The Blueprint (read-only)
A Docker image is a read-only template. It contains your application code, runtime, libraries, and config, frozen at build time. Like a class in OOP — defines the structure but isn't running yet.

Stored in a registry (ECR, Docker Hub). Tagged with a version. myapp:v1.2.3
Container — The Running Instance
A container is a running instance of an image. Like an object instantiated from a class. Has its own process space, network interface, filesystem (writable layer on top of the image).

Run 1 image → 1 container. Run the same image 10 times → 10 independent containers, each isolated.

Layered Images — Why Docker is Fast and Efficient

Docker images are built in layers. Each instruction in a Dockerfile adds a new layer. Layers are cached — if a layer hasn't changed, Docker reuses the cached version instead of rebuilding. This makes builds fast and images efficient to transfer.

# Dockerfile builds layers from bottom to top
# Layer 1: Base image (downloaded once, cached forever)
FROM amazoncorretto:21-alpine3.19    ← pulled from Docker Hub: ~120MB

# Layer 2: Rarely-changing system dependencies
RUN apk add --no-cache curl          ← only rebuilt if this line changes

# Layer 3: Application dependencies (changes occasionally)
COPY pom.xml .
RUN mvn dependency:go-offline -q     ← downloads dependencies

# Layer 4: Application code (changes on every commit)
COPY src ./src
RUN mvn package -DskipTests -q       ← only this rebuilds on code changes

# Configuration (not a layer — just metadata)
EXPOSE 8080
ENV JAVA_OPTS="-Xmx512m -Xms256m"
ENTRYPOINT ["java", "-jar", "target/app.jar"]

The smart ordering above is critical: put things that change rarely at the top, things that change often at the bottom. If only your code changes (Layer 4), Docker reuses the cached Layers 1-3 and only rebuilds Layer 4. Build time goes from minutes to seconds.

Working with Docker — Essential Commands

# Build an image from Dockerfile in current directory

Docker Compose — Running Multiple Containers Locally

Your app needs the app container, a database, and a Redis cache — all running locally for development. Docker Compose defines and runs this multi-container setup with one command.

# docker-compose.yml — local dev environment
version: '3.9'
services:
  app:
    build: .
    ports: ["8080:8080"]
    environment:
      DB_URL: jdbc:postgresql://db:5432/mydb
      REDIS_URL: redis://cache:6379
    depends_on: [db, cache]

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: mydb
      POSTGRES_PASSWORD: localpassword
    volumes:
      - postgres_data:/var/lib/postgresql/data

  cache:
    image: redis:7-alpine
    ports: ["6379:6379"]

volumes:
  postgres_data:

# Start everything:
docker compose up -d

# Tear everything down:
docker compose down
Docker Compose is for local development only. It runs on one machine — no HA, no auto-restart if the machine fails, no scaling across multiple hosts. For production multi-container workloads: use ECS or Kubernetes. They are the production equivalent of Compose, but distributed.

Production-Grade Dockerfile

A Dockerfile that's fast to build, small in size, and secure. The gap between a "works" Dockerfile and a production Dockerfile is significant.

The Multi-Stage Build — Build Once, Ship Small

The most important Dockerfile pattern. Use a large build image (with Maven, JDK, all build tools) to compile your app, then copy only the compiled output into a minimal runtime image. The final image has no build tools, no source code — only what's needed to run.

# Multi-stage Dockerfile for a Spring Boot application

# ====== STAGE 1: BUILD ======
# Use full Maven + JDK image for building (this is large — ~600MB)
FROM maven:3.9-amazoncorretto-21 AS builder

WORKDIR /build

# Copy pom.xml first — download dependencies as a separate layer
# This layer is cached until pom.xml changes (not on every code commit)
COPY pom.xml .
RUN mvn dependency:go-offline -q

# Now copy source and build
COPY src ./src
RUN mvn package -DskipTests -q

# ====== STAGE 2: RUNTIME ======
# Minimal JRE-only image (no JDK, no Maven, no build tools)
FROM amazoncorretto:21-alpine3.19 AS runtime

# Run as non-root user (security best practice)
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

WORKDIR /app

# Copy ONLY the compiled JAR from the builder stage
COPY --from=builder --chown=appuser:appgroup \
  /build/target/myapp.jar ./app.jar

# Document what port the app listens on
EXPOSE 8080

# Health check — orchestrators use this
HEALTHCHECK --interval=30s --timeout=10s --start-period=60s --retries=3 \
  CMD wget -qO- http://localhost:8080/actuator/health || exit 1

# JVM tuning for containers (respect container memory limits)
ENV JAVA_OPTS="-XX:+UseContainerSupport -XX:MaxRAMPercentage=75.0 -XX:+UseG1GC"

ENTRYPOINT ["sh", "-c", "java $JAVA_OPTS -jar app.jar"]

# Result: Final image is ~150MB instead of 600MB
#         No Maven, no JDK, no source code in the final image
#         Security: runs as non-root user
#         Efficiency: dependency layer cached separately

Important JVM Flags for Containers

# Before Java 10: JVM didn't understand container memory limits
# It saw HOST machine RAM → allocated too much heap → OOMKilled
# Fix: UseContainerSupport (default from Java 11+)

-XX:+UseContainerSupport          # JVM reads cgroup limits (not host RAM)
-XX:MaxRAMPercentage=75.0         # use 75% of container RAM for heap
                                   # (remaining 25% for metaspace, threads, etc.)

# Example: Container has 1GB RAM
# Without flags: JVM might allocate 4GB heap (sees host RAM) → crash
# With flags: JVM allocates 750MB heap → runs fine

-XX:InitialRAMPercentage=50.0     # start at 50% heap, grow to MaxRAM
-XX:+ExitOnOutOfMemoryError       # crash cleanly (let orchestrator restart)
                                   # instead of limping with corrupted state

The Problem Orchestration Solves

Docker solves packaging. But once you have 50 containers running across 10 servers — who keeps them running, distributes the load, replaces failed ones, and deploys updates without downtime? That's orchestration.

What Goes Wrong Without Orchestration

You have 10 EC2 instances and 5 microservices, each needing 3 replicas = 15 containers total. You manually deploy them by SSHing into servers. Here's what you must handle manually:

Container crashes
Container crashes at 3am. No one restarts it. Service is down until someone manually SSHes in and runs docker start. Users see errors for hours.
Deployments cause downtime
To deploy a new version, stop the old container, pull new image, start new container. Gap between stop and start = downtime. Manual process = human error.
Load distribution
Server 1 has 10 containers running. Server 5 has 2. Unbalanced. You'd need to manually move containers around. Who tracks what's running where?
Server failures
Server 3 dies. 3 containers gone. Someone must manually restart those containers on surviving servers. Meanwhile, those services are under-replicated.
Scaling
Traffic spikes. You need 5 more containers of the API service. Which server has capacity? SSH in, run docker commands, manually register with load balancer...
Resource management
How much CPU and RAM does each container get? If one container uses all RAM on a server, others get OOMKilled. Manual limits = human error = production incidents.

What an Orchestrator Does

An orchestrator (Kubernetes, ECS) is a control plane that continuously watches the state of your containers and takes automated action to keep them in the desired state. You declare what you want ("5 replicas of my API container, each with 0.5 CPU and 1GB RAM") and the orchestrator makes it happen and keeps it that way.

1
Scheduling — where does each container run?
Orchestrator picks which server (node) each container runs on based on available CPU/RAM, affinity rules, and spread requirements. You never manually decide "container goes on server 3."
2
Health monitoring — is each container healthy?
Continuously runs health checks. Failed container → immediately replace it on a healthy server. Unhealthy container removed from load balancer before being restarted. Zero-downtime recovery.
3
Desired state reconciliation
"I declared 5 replicas, I see only 3 running." → Start 2 more. Server died taking 2 containers with it → start them elsewhere. This loop runs constantly, every few seconds.
4
Rolling deployments — update without downtime
Deploy new version: start 1 new container → wait for it to pass health checks → remove 1 old container → repeat. Traffic always flows to healthy containers. Zero downtime.
5
Service discovery and load balancing
When a new container starts, it's automatically registered for traffic. When it stops, automatically removed. No manual load balancer configuration.

Core Kubernetes Objects

Pod — The Smallest Unit

A Pod is the smallest deployable unit in Kubernetes — one or more containers that share a network namespace and storage. Containers inside a Pod talk to each other via localhost. Most of the time: one container per Pod.

# pod.yaml — the most basic K8s object
apiVersion: v1
kind: Pod
metadata:
  name: myapp-pod
  labels:
    app: myapp
spec:
  containers:
  - name: myapp
    image: 123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp:v1.2.3
    ports:
    - containerPort: 8080
    resources:
      requests:                 # minimum guaranteed resources
        memory: "512Mi"
        cpu: "250m"             # 250 millicores = 0.25 vCPU
      limits:                   # maximum allowed — exceed = throttle/OOM
        memory: "1Gi"
        cpu: "1000m"            # 1000m = 1 vCPU
    env:
    - name: DB_URL
      valueFrom:
        secretKeyRef:
          name: db-credentials  # K8s Secret
          key: url
    livenessProbe:              # if this fails, restart the container
      httpGet: {path: /health, port: 8080}
      initialDelaySeconds: 60
      periodSeconds: 30
    readinessProbe:             # if this fails, remove from load balancer
      httpGet: {path: /ready, port: 8080}
      initialDelaySeconds: 10
      periodSeconds: 5

Deployment — Manages Pods at Scale

You never create Pods directly in production. You create a Deployment — a higher-level object that manages a set of identical Pods (a ReplicaSet). The Deployment controller ensures the desired number of Pods are always running and handles rolling updates.

# deployment.yaml — manage 5 replicas of your app
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-deployment
spec:
  replicas: 5                    # desired number of running pods
  selector:
    matchLabels:
      app: myapp
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 0          # never go below 5 pods during update
      maxSurge: 2                # allow up to 7 pods during update
  template:                      # pod template — every pod looks like this
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: myapp
        image: myapp:v1.2.3
        ... (same as pod spec above)

# Deploy a new version:
kubectl set image deployment/myapp-deployment myapp=myapp:v1.3.0
# K8s gradually replaces v1.2.3 pods with v1.3.0 pods
# Watches readinessProbe — only sends traffic to ready pods

# Rollback:
kubectl rollout undo deployment/myapp-deployment

Service — Stable Network Endpoint

Pods are ephemeral — they start and stop, get new IP addresses each time. A Service provides a stable DNS name and IP that load-balances across all matching Pods. Other services call the Service's DNS name — they never care which specific Pod responds.

# service.yaml — stable endpoint for the myapp pods
apiVersion: v1
kind: Service
metadata:
  name: myapp-service
spec:
  selector:
    app: myapp                   # matches all pods with label app=myapp
  ports:
  - port: 80                     # service listens on 80
    targetPort: 8080             # forwards to port 8080 on pods
  type: ClusterIP                # internal only (LoadBalancer = external ALB)

# Other services call this via DNS: http://myapp-service/api/users
# K8s DNS resolves myapp-service → Service IP → load balances across pods
# If pods restart with new IPs, Service automatically picks them up

HorizontalPodAutoscaler — Auto Scale Containers

# hpa.yaml — scale pods based on CPU
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp-deployment
  minReplicas: 3
  maxReplicas: 50
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60   # scale when avg CPU > 60%

# This scales PODS (the application layer)
# It does NOT scale EC2 nodes (infrastructure layer)
# If pods need more capacity than nodes have → scheduler can't place them
# → pods stay "Pending" until a node with capacity exists
# → THIS is where Cluster Autoscaler or Karpenter (node scaling) comes in

ECS — AWS's Orchestrator

ECS (Elastic Container Service) is AWS's proprietary orchestrator. It solves the same problems as Kubernetes but with simpler concepts, deeper AWS integration, and less operational overhead.

ECS vs Kubernetes — Same Layer, Different Tools

ECS and Kubernetes are both orchestrators — they both sit at Layer 2. They solve the same problem: running and managing containers reliably at scale. The difference is in complexity, portability, and trade-offs.

ConceptKubernetesECS Equivalent

ECS is Simpler — Here's Why

Kubernetes: You manage the control plane (on EC2)
If you self-host K8s: you provision master nodes, manage etcd backups, handle K8s version upgrades, configure RBAC, manage certificate rotation, deal with networking plugins (CNI), and install add-ons. Even with EKS (managed K8s), you still manage worker nodes and many components.
ECS: AWS manages everything above your containers
ECS control plane is 100% managed by AWS. No master nodes. No etcd. No networking plugins. Deep native integration: IAM roles per task, native CloudWatch Logs, ALB integration, Secrets Manager injection — all configured declaratively, no add-ons needed. 
## ECS Service = Kubernetes Deployment + Service combined

# ECS scales containers (application layer) via Application Auto Scaling
aws application-autoscaling put-scaling-policy \
  --service-namespace ecs \
  --resource-id service/my-cluster/myapp-service \
  --scalable-dimension ecs:service:DesiredCount \
  --policy-type TargetTrackingScaling \
  --target-tracking-scaling-policy-configuration '{
    "TargetValue": 60.0,
    "PredefinedMetricSpecification": {
      "PredefinedMetricType": "ECSServiceAverageCPUUtilization"
    }
  }'

# This scales TASKS (containers) — same as K8s HPA
# When a task needs resources and no capacity exists:
#   - Fargate: AWS automatically creates a new microVM (no worries)
#   - EC2 launch type: task stays PENDING until a node has capacity
#     → Capacity Provider with ASG triggers EC2 scale-out

Who Provides the Machines?

Layer 3 is the infrastructure — the actual compute that containers run on. Two options: manage EC2 instances yourself (via ASG) or let Fargate do it automatically.

The Infrastructure Challenge

Kubernetes or ECS knows you want to run 50 containers. But those containers need to actually run on physical/virtual machines. Someone has to: provision EC2 instances, install the container runtime, join them to the cluster, ensure there's enough total capacity for all containers, remove instances when containers scale down (to save money).

Option A: EC2 + Auto Scaling Group (you manage nodes)
You create an ASG with EC2 instances pre-configured as cluster nodes. When the orchestrator can't schedule containers (not enough capacity on existing nodes), something must tell the ASG to add more EC2 instances.

For ECS: Capacity Providers + ASG
For EKS: Cluster Autoscaler or Karpenter + ASG
Option B: Fargate (AWS manages nodes)
AWS provisions an isolated microVM for each task/pod. You don't see the underlying instances. No ASG to configure. No node groups to manage. Just declare your container's CPU and RAM requirements. Fargate provisions exactly what's needed.

Works with both ECS and EKS.

EC2 Capacity Provider — How ECS Scales Infrastructure

When you use ECS with EC2 launch type, a Capacity Provider bridges the orchestrator and the infrastructure. When ECS has more tasks to place than existing instances can handle, the Capacity Provider triggers the ASG to add instances.

## Flow: ECS detects insufficient capacity → Capacity Provider scales ASG

Step 1: ECS Service wants 20 tasks running
Step 2: Current EC2 nodes can only fit 14 tasks
Step 3: 6 tasks stay in PROVISIONING state
Step 4: Capacity Provider sees unplaceable tasks
Step 5: Sends scale-out signal to ASG
Step 6: ASG launches 2 more EC2 instances (~3 minutes)
Step 7: Instances join ECS cluster
Step 8: 6 pending tasks placed on new instances

# Create Capacity Provider linked to an ASG
aws ecs create-capacity-provider \
  --name my-ec2-capacity-provider \
  --auto-scaling-group-provider '{
    "autoScalingGroupArn": "arn:aws:autoscaling:...:autoScalingGroup:my-asg",
    "managedScaling": {
      "status": "ENABLED",
      "targetCapacity": 80,
      "minimumScalingStepSize": 1,
      "maximumScalingStepSize": 10
    },
    "managedTerminationProtection": "ENABLED"
  }'

Karpenter — Modern Node Scaling for EKS

For EKS, Karpenter is the modern alternative to Cluster Autoscaler. Instead of scaling ASG instance counts, Karpenter directly provisions EC2 instances of exactly the right size for pending pods. More efficient, faster (30s vs 3min), and supports diverse instance types including Spot.

## Karpenter flow:
# 1. Pod declared: needs 3 vCPU, 6GB RAM
# 2. No existing node has that capacity
# 3. Karpenter checks: what's the cheapest/fastest EC2 type to fit this?
# 4. Karpenter directly launches a c6i.xlarge (4vCPU, 8GB)
# 5. Node joins cluster in ~30s
# 6. Pod scheduled immediately

## Unlike Cluster Autoscaler:
# - Karpenter doesn't need pre-configured ASG node groups
# - Picks optimal instance type per workload (bin packing)
# - Consolidates nodes when underutilised (saves money)
# - Supports multiple instance types and Spot in one NodePool

Two Scaling Dimensions

The most important thing to understand about container scaling: there are TWO completely independent dimensions of scaling that must work together. Confusing them leads to bad architecture and production incidents.

Dimension 1: Application Scaling (Horizontal Pod/Task Autoscaler)

This scales the number of containers. More containers = more application capacity to handle requests. Managed by the orchestrator (ECS Application Auto Scaling, K8s HPA). Reacts to: CPU, memory, request rate, queue depth, custom metrics.

Dimension 2: Infrastructure Scaling (Cluster Autoscaler / ASG)

This scales the number of EC2 instances. More instances = more raw capacity that containers can be scheduled on. Managed by ASG, Cluster Autoscaler, or Karpenter. Reacts to: unschedulable containers, node CPU/memory usage.

Why Teams Skip ASG (or Work Around It)

You asked about this specifically. Here's the honest picture of why mature engineering teams often bypass or supplement ASG with custom logic.

The Core Problems with Pure ASG for Container Workloads

Problem 1: 3-5 Minute Scale-Out Lag

The biggest limitation. When traffic spikes, the chain is: CloudWatch alarm (1-2 min) → ASG decision → EC2 launch → AMI boot → OS startup → ECS/K8s agent registers → orchestrator places containers → health check → traffic. Total: 3-8 minutes. By then, your users are already seeing errors.

## How teams work around this:

# 1. Over-provisioning baseline
# Keep nodes at 40-50% capacity instead of 70-80%
# Extra capacity absorbs spikes without needing new nodes
# Cost: you're paying for idle capacity, but cheaper than user-facing outages

# 2. Container-level pre-scaling
# Scale CONTAINERS first (seconds), not nodes (minutes)
# If nodes have spare capacity, containers scale immediately
# Only need new nodes if all existing capacity is exhausted

# 3. Scheduled pre-scaling
aws autoscaling put-scheduled-update-group-action \
  --auto-scaling-group-name app-nodes-asg \
  --scheduled-action-name morning-scale-up \
  --recurrence "0 7 * * MON-FRI" \
  --desired-capacity 12     # scale up BEFORE users arrive (8am traffic)

aws autoscaling put-scheduled-update-group-action \
  --scheduled-action-name night-scale-down \
  --recurrence "0 20 * * MON-FRI" \
  --desired-capacity 4      # scale down after hours

# 4. Use Fargate for burst capacity
# EC2 for baseline (cheaper), Fargate for overflow (fast scale)
# ECS Capacity Providers support mixed strategies:
{
  "capacityProviders": [
    {"capacityProvider": "my-ec2-cap-provider", "base": 5, "weight": 80},
    {"capacityProvider": "FARGATE", "weight": 20}
  ]
}
# Base 5 tasks on EC2 always. Additional tasks: 80% EC2, 20% Fargate.
# During sudden spikes, Fargate handles the overflow while EC2 catches up.

Problem 2: ASG Doesn't Know About Container Bin-Packing

ASG scales based on CPU/memory of EC2 instances. But if you have 4 EC2 instances each at 30% CPU and 90% memory, ASG thinks everything is fine (CPU is low). But the orchestrator can't place new containers (memory full). The signals don't align.

## Solution: Capacity Provider's own utilisation metric
# ECS Capacity Providers use "CapacityProviderReservation" metric
# This counts: (reserved capacity / available capacity) across the cluster
# Scale-out when this > 80% (regardless of CPU/memory split)

## Solution: Karpenter bin-packing
# Karpenter provisions nodes based on actual pod requests, not instance CPU%
# "This pod needs 2.5 vCPU and 5GB — provision a c6g.xlarge"
# Much smarter than ASG target tracking on instance CPU

Problem 3: Slow Scale-In (Overspend)

After traffic drops, ASG is conservative about terminating instances — it doesn't want to accidentally under-provision. Default scale-in cooldown is 300 seconds. Karpenter consolidates more aggressively — if two nodes could be merged into one, it does it automatically, saving money.

ECS vs EKS — Decision Guide

Both are AWS container orchestrators. This is one of the most common architecture questions. Here's a clear framework.

The Core Trade-off

ECS — AWS-Native Simplicity

Interview Questions — Containers & Scaling

12 questions covering the concepts from this module.

Conceptual

What is the difference between a Docker image and a container? ▼
An image is a read-only, immutable template — a frozen snapshot of your application and all its dependencies, built by a Dockerfile. It's stored in a registry (ECR, Docker Hub) and can be versioned and tagged. A container is a running instance of an image — it has its own process space, network interface, and a thin writable layer on top of the read-only image. You can run many containers from the same image simultaneously — they're all independent. Analogy: the image is a class in OOP; a container is an object instance. Multiple objects, same class.
Why are containers faster to start than virtual machines? ▼
VMs virtualise the entire hardware stack — they boot a full OS kernel, run an init system, start system services. This takes 30-90 seconds minimum. Containers share the host OS kernel — no kernel boot needed. The container runtime (Docker) uses Linux kernel features (namespaces for isolation, cgroups for resource limits) to create an isolated process. The container starts as fast as your application process starts — typically 1-5 seconds. The file system is assembled from image layers, which are cached and don't need to be copied. The trade-off: containers have weaker isolation (shared kernel) compared to VMs (hardware-level). AWS Fargate uses Firecracker microVMs — boots a minimal kernel in ~125ms, giving near-container speed with hardware-level isolation.
Explain the three layers of a container platform and what each one does. ▼
Layer 1 — Packaging (Docker): Packages your application and all its dependencies (JRE, libraries, config) into a portable container image. Solves "works on my machine." The image runs identically everywhere. Layer 2 — Orchestration (ECS, Kubernetes): Manages containers at scale. Decides which container runs on which machine (scheduling), monitors health (restarts failed containers), handles rolling deployments (zero downtime updates), auto-scales container count based on load, provides service discovery (stable DNS names across container restarts). Layer 3 — Infrastructure (EC2 + ASG, or Fargate): The actual compute capacity that containers run on. ASG provisions and scales EC2 instances. Capacity Providers/Karpenter scale nodes when orchestrator can't place containers. Fargate abstracts this layer entirely — AWS provisions microVMs on demand, invisible to you.
What is the difference between scaling containers and scaling nodes? Why does it matter? ▼
Scaling containers (application scaling): orchestrator increases the number of running containers for your service. Takes seconds. Limited by available capacity on existing nodes. Managed by Application Auto Scaling (ECS) or HPA (K8s). Scaling nodes (infrastructure scaling): ASG or Karpenter provisions more EC2 instances. Takes 3-5 minutes. Needed when no existing node has sufficient CPU/memory for new containers. These are independent. You can scale containers without scaling nodes (if nodes have spare capacity). You can scale nodes without scaling containers (pre-warming infrastructure). The latency difference matters: application scaling is instant; infrastructure scaling takes minutes. This is why teams use Fargate (removes node scaling delay) or maintain over-provisioned nodes (containers always have room to scale).

Practical

What is Docker layer caching and how do you optimise a Dockerfile for it? ▼
Docker builds images layer by layer. Each instruction creates a layer. A layer is only rebuilt if it changes OR if any layer above it changes. Optimisation: put instructions that change rarely at the top, instructions that change often at the bottom. For a Java app: (1) FROM base image — rarely changes. (2) Install system deps (apk/apt) — rarely changes. (3) COPY pom.xml and download dependencies (mvn dependency:go-offline) — changes when you add/update dependencies, not on every code commit. (4) COPY src and compile — changes on every commit. Without this ordering: every code change would retrigger dependency download (minutes). With correct ordering: only the compilation layer rebuilds (seconds), because layers 1-3 are cached. Multi-stage builds further optimise: final image only contains runtime artifacts (JAR + JRE), not build tools (Maven, JDK), reducing image size from ~600MB to ~150MB.
What is a Kubernetes Deployment and how does a rolling update work? ▼
A Deployment is a K8s controller that manages a ReplicaSet — ensuring a specified number of identical Pods are always running. Rolling update flow: you change the Deployment's image tag. K8s creates a new ReplicaSet with the new version. It starts bringing up new pods one by one (or a configured batch). Each new pod must pass its readinessProbe before K8s considers it healthy. As new pods become ready, K8s removes old pods. With maxUnavailable=0 and maxSurge=2: you always have at least the original replica count running (100% capacity maintained), temporarily running at 100%+surge. If new pods fail readiness: K8s stops the rollout (Deployment circuit breaker). kubectl rollout undo to revert. At no point during a successful rolling update are all instances the old version or all the new version — there's a gradual transition while traffic flows to healthy pods only.
When would you choose ECS over EKS? ▼
Choose ECS when: (1) Building on AWS long-term without multi-cloud requirements — ECS is deeply integrated with AWS services (IAM, CloudWatch, Secrets Manager, ALB) natively, no add-ons needed. (2) Team has limited or no Kubernetes experience — ECS has fewer concepts and less YAML, faster to become productive. (3) Fargate is your primary launch type — Fargate integration is simpler in ECS. (4) Cost sensitivity at small scale — EKS charges $0.10/hour (~$72/month) per cluster for the control plane; ECS control plane is free. (5) Simplicity over ecosystem — ECS deploys and scales reliably without managing add-ons (CoreDNS, load balancer controller, metrics server, etc.). Choose EKS when your team has K8s expertise, you need portability across clouds, you need advanced K8s features (CRDs, complex scheduling), or you need the K8s ecosystem (Helm, ArgoCD, Istio, Prometheus operator).

Advanced

A company's ECS service isn't scaling fast enough during traffic spikes. Walk through your diagnosis and solutions. ▼
Diagnosis: (1) Check what's slow — Application Auto Scaling taking too long to fire? (CloudWatch metrics granularity, alarm evaluation period). Container scaling fast but nodes not available? (ECS task PROVISIONING vs RUNNING states). (2) Check capacity: are tasks stuck in PROVISIONING? If yes: capacity problem — nodes full or no nodes. If no: containers are scaling but app is still slow — different problem. Solutions by root cause: (A) Container scaling too slow: reduce CloudWatch alarm evaluation period (minimum 1 period of 1 min), use target tracking instead of step scaling (more responsive), lower target value (60% → 50% CPU triggers sooner). (B) Node capacity bottleneck with EC2: Use Fargate for overflow (mixed Capacity Provider strategy). Pre-scale nodes via scheduled action before expected peaks. Use Karpenter (30s node provisioning vs 3-5min ASG). Over-provision nodes to 50% utilisation baseline. (C) Best long-term: Migrate burst capacity to Fargate. EC2 for predictable baseline (cost-efficient), Fargate for elastic overflow (fast).
Explain how you'd run a Java microservice on ECS Fargate — from code to production traffic. ▼
(1) Write a Dockerfile: multi-stage (Maven builder → JRE runtime), non-root user, HEALTHCHECK, UseContainerSupport JVM flag, MaxRAMPercentage=75. (2) CI/CD: on git push, run tests, docker build for linux/arm64 (Graviton), push to ECR. (3) Task Definition: image from ECR, 0.5 vCPU, 1GB RAM, port 8080, env vars from Secrets Manager via ARN reference, awslogs log driver (CloudWatch), IAM task role with DynamoDB/S3 permissions, execution role with ECR pull + Secrets Manager access. (4) ECS Service: desired count=3, Fargate launch type, private app subnets, sg-app security group, target group registered on ALB port 8080, health check on /actuator/health, deployment circuit breaker enabled with rollback, Application Auto Scaling target tracking 60% CPU. (5) ALB: HTTPS listener port 443, forwards to target group, health check /actuator/health 2xx. (6) Deploy new version: update task definition with new image tag, ecs update-service, rolling update with zero downtime. Traffic flows within minutes.
What is Karpenter and how is it different from Cluster Autoscaler for EKS node scaling? ▼
Cluster Autoscaler (CA): Traditional approach. Scales pre-defined ASG node groups up or down by count. When pods are unschedulable, CA picks which node group to scale up. Scale-out: ~3-5 minutes (EC2 boot + join cluster). Reactive — waits for pending pods. Can only use the instance types defined in its node groups. Scale-in: conservative, slow. Karpenter: Modern alternative. Instead of incrementing a node group count, Karpenter directly calls EC2 APIs to launch exactly the right instance type for pending pods. Looks at pending pod's CPU/memory requests and picks the optimal (cheapest/fastest) instance type. No pre-defined node groups needed. Scale-out: ~30 seconds (Karpenter skips ASG overhead). Consolidation: proactively merges underutilised nodes (evicts pods, terminates node, reschedules pods on remaining nodes) → saves money. Spot diversity: can use 20+ instance types simultaneously to minimise interruptions. Karpenter is AWS-maintained and is now the recommended approach for EKS node scaling.
