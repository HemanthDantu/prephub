Section 1 of 4 · ~6 mins

Images vs Containers

This section builds the core mental model: an image is the packaged artifact, and a container is a running instance of that artifact.

🧒 ELI5

Recipe card vs lunch box

A recipe tells you exactly how to make a dish, but you cannot eat the recipe itself. You need to actually cook it to get food on the table.

If you change the lunch after it is served, that does not change the recipe for the next lunch. The next meal still starts from the same original instructions.

Docker works the same way: the image is the repeatable recipe, and the container is the one running copy created from it.

🔗 ANALOGY

Movie master file vs screening

A studio keeps one master copy of a movie, then runs many screenings from it. If one screening projector breaks, you start another screening from the same master.

That is why containers are treated as disposable runtime instances rather than precious pets you patch by hand.

Technical

What it means in real code

Definition: An image is an immutable, versioned filesystem snapshot plus metadata, while a container is a runtime process that adds a temporary writable layer on top of that image.

When you run docker run, the container runtime mounts the image layers read-only, adds a thin writable layer, then starts the configured ENTRYPOINT or CMD as PID 1 inside isolated namespaces and cgroups.

In production this matters because reproducibility comes from rebuilding and redeploying images, not from logging into a running container and “fixing” it. Teams that mutate containers create drift between environments and make rollbacks unpredictable.

Why it matters

  • Keeps environments reproducible
  • Makes rollback a tag change instead of a manual repair
  • Supports fast scale-out because containers are disposable

Common pitfall

  • SSH-ing into containers and patching them live
  • Using latest tags everywhere
  • Treating container filesystem changes as persistent state
Immutable imageDisposable runtimeWritable layerPet container
Write Dockerfile→Build image→Run container→Replace on change

Interview-ready view

💬 Model Interview Answer

An image is the immutable artifact and a container is the running process created from it. I usually explain that images should be versioned and containers should be replaceable. The common gotcha is teams debugging by changing files inside a live container, which fixes one instance but destroys reproducibility.

ConceptWhat to remember
ImageImmutable template with layers and runtime metadata
ContainerOne running instance with a writable layer
Best practiceRebuild and redeploy instead of patching live
❌ Wrong: patching a running container
docker run -d --name api myorg/api:latest
docker exec -it api sh
# Edit files inside the container to "fix" production
vi /app/config.yml
✅ Right: rebuild and replace
docker build -t myorg/api:1.4.2 .
docker run -d --name api-v2 -p 8080:8080 myorg/api:1.4.2
# After validation, stop and remove the old container
docker stop api && docker rm api

The right pattern preserves a versioned artifact and makes rollback a simple image tag swap.

⚠️ Warning

Container drift is the real trap

The most dangerous misunderstanding is thinking container filesystem changes are durable application state. They disappear when the container is recreated, and worse, they make the running instance different from the source-controlled build. Persist data in volumes or external systems, and rebuild images for code changes.

Mini takeaway
  • Images are immutable artifacts
  • Containers are replaceable runtime instances
  • Persist state outside the container
  • Version image tags explicitly
Quiz

Correct answer: BB is correct because images are the repeatable build artifact and containers are runtime instances created from them. A is backwards: the container should not become the source of truth. C creates drift and breaks rollback discipline. D misses the key operational difference that makes Docker useful in real deployment pipelines.
Section 2 of 4 · ~6 mins

Dockerfile and Layers

This section explains how image layers are built, cached, and optimized for smaller, faster builds.

🧒 ELI5

Stacking transparent sheets

Imagine drawing a picture on clear plastic sheets stacked on top of each other. If you change only the top sheet, you do not redraw the whole stack.

That is why ordering matters. Put the parts that change rarely near the bottom, and the parts that change often near the top.

Docker build caching works the same way, so the order of instructions affects build speed a lot.

🔗 ANALOGY

Packing a suitcase in the right order

If you put daily-use items on top, you can swap them quickly without repacking the whole bag. If you bury them at the bottom, every tiny change forces a full repack.

A badly ordered Dockerfile destroys cache reuse and makes CI slow for every developer.

Technical

What it means in real code

Definition: A Dockerfile builds an image as an ordered sequence of cached layers, where each instruction can reuse or invalidate the cache of the instructions that follow.

Each RUN, COPY, or ADD can create a new layer. If a layer changes, Docker must rebuild that layer and every subsequent layer, which is why copying the whole source tree before dependency installation is often expensive.

In production and CI, careful layer ordering reduces build time, registry bandwidth, and attack surface. Multi-stage builds also let you compile with heavy tooling but ship a smaller runtime image.

Why it matters

  • Faster CI builds through cache reuse
  • Smaller images with multi-stage builds
  • Lower vulnerability surface by shipping only runtime dependencies

Common pitfall

  • COPY . . before installing dependencies
  • Using one huge base image for convenience
  • Leaving package manager caches and build tools in the final image
Multi-stage buildLayer cacheInstruction orderBloated image
Choose base image→Install deps with cache-friendly order→Copy source→Build final runtime image

Interview-ready view

💬 Model Interview Answer

Dockerfile optimization is mostly about cache boundaries and image size. I usually install dependencies before copying the rest of the application if the dependency manifest changes less often than the source. A common gotcha is copying the full repo too early, which invalidates the cache and makes every commit rebuild from scratch.

PatternEffect
Multi-stage buildKeeps build tools out of the final image
Stable dependency layerSpeeds up rebuilds
Small base imageReduces pull time and security noise
❌ Wrong: cache-unfriendly Dockerfile
FROM eclipse-temurin:21-jdk
WORKDIR /app
COPY . .
RUN ./mvnw clean package -DskipTests
CMD ["java","-jar","target/app.jar"]
✅ Right: multi-stage and cache-friendly
FROM maven:3.9-eclipse-temurin-21 AS build
WORKDIR /workspace
COPY pom.xml .
RUN mvn -q -DskipTests dependency:go-offline
COPY src ./src
RUN mvn -q -DskipTests package

FROM eclipse-temurin:21-jre
WORKDIR /app
COPY --from=build /workspace/target/app.jar app.jar
ENTRYPOINT ["java","-jar","/app/app.jar"]

The second version preserves cache reuse on dependencies and ships only the JRE plus the built jar.

⚠️ Warning

Layer order is the easiest place to waste CI time

The common failure is not understanding that one changed layer invalidates the ones after it. If you COPY the full repository before dependency install, even a README change can force a full Maven or npm reinstall. Move stable inputs earlier, use multi-stage builds, and keep the final image minimal.

Mini takeaway
  • Instruction order controls cache reuse
  • Multi-stage builds shrink runtime images
  • Copy stable files first
  • Ship only runtime artifacts
Quiz

Why does copying the whole source tree before dependency installation often slow builds?

Correct answer: BB is correct because Docker layer caching is sequential: when an earlier layer changes, later layers must be rebuilt. A is false because RUN layers are cached normally. C is obviously not true in production systems. D mixes unrelated instructions and does not explain cache invalidation.
Section 3 of 4 · ~6 mins

Volumes and Networking

This section covers where persistent data belongs and how containers communicate without hard-coding host assumptions.

🧒 ELI5

Notebook vs locker

A student may carry a notebook all day, but if the backpack is lost, the notes go too. Important belongings belong in a locker, not only in the bag.

Container filesystems are like the backpack: temporary. Volumes are like the locker: created to survive container replacement.

Networking is similar: people in the same school can talk by room number instead of memorizing street addresses.

🔗 ANALOGY

Apartment building utilities

Every apartment is separate, but they share plumbing and a building intercom. Residents talk through stable apartment numbers, not by guessing where someone parked today.

In Docker, service discovery through network aliases is much safer than hard-coding container IPs.

Technical

What it means in real code

Definition: Volumes provide persistent storage outside the container writable layer, and Docker networks provide name-based communication between containers.

A named volume is managed by Docker and mounted into the container filesystem. User-defined bridge networks create DNS entries so one container can reach another by service name, which is how Compose stacks resolve api -> db reliably.

This matters in production-like local development because databases must keep data across container recreation and services must connect without relying on unstable IP addresses. If you persist data in the container layer or hard-code addresses, your environment becomes fragile.

Why it matters

  • Keeps database data across restarts
  • Lets services discover each other by name
  • Makes local stacks closer to real deployments

Common pitfall

  • Storing Postgres data in the container filesystem
  • Using localhost incorrectly between containers
  • Hard-coding container IP addresses
Named volumeBridge network DNSPort publishingHard-coded IP
Create network→Start containers→Resolve service names→Persist data in volume

Interview-ready view

💬 Model Interview Answer

Volumes solve persistence and networks solve service-to-service connectivity. I normally explain that a container should be replaceable, so durable data belongs in a volume or external database. The gotcha is that localhost inside one container points to itself, not to a sibling container.

NeedCorrect tool
Keep database filesNamed volume
Connect api to dbUser-defined network + service name
Expose service to hostPublished port
❌ Wrong: ephemeral data and wrong hostname
docker run --name db postgres:16
docker run --name api -e DB_HOST=localhost myorg/api:1.0
✅ Right: persistent volume and network alias
docker network create app-net
docker volume create pg-data

docker run -d --name db --network app-net               -v pg-data:/var/lib/postgresql/data postgres:16

docker run -d --name api --network app-net               -e DB_HOST=db -p 8080:8080 myorg/api:1.0

The API now reaches the database by container name on the shared network, and the database data survives recreation.

⚠️ Warning

Localhost confusion breaks many Docker setups

The most common networking mistake is using localhost from one container to reach another container. localhost always means “this container.” Put both containers on the same user-defined network and use the target service name, and use volumes for state you cannot lose.

Mini takeaway
  • Use volumes for durable state
  • Use service names on shared networks
  • localhost is container-local
  • Publish ports only when host access is needed
Quiz

Your API container cannot reach Postgres running in another container. Which fix is most correct?

Correct answer: AA is correct because user-defined networks give containers DNS-based discovery by name. B is wrong because localhost points back to the API container itself. C confuses runtime connectivity with image packaging. D is unreliable because container IPs are not a safe discovery mechanism.
Section 4 of 4 · ~6 mins

Docker Best Practices

This section turns the basics into operational habits: security, observability, startup behavior, and image hygiene.

🧒 ELI5

Camping with a checklist

When you go camping, you pack only what you need, label everything clearly, and check that the flashlight works before leaving. That discipline prevents surprises in the dark.

Containers work the same way. Small images, health checks, and non-root execution reduce the number of bad surprises after deployment.

Best practices are boring when they work, and very expensive when they are skipped.

🔗 ANALOGY

Opening a restaurant safely

A restaurant opens only after the kitchen is ready, the food is stored correctly, and staff know who can access what. Skipping those basics creates health and safety issues immediately.

In Docker, health checks, least privilege, and explicit configuration are the equivalent safety rules.

Technical

What it means in real code

Definition: Docker best practices are the set of image, runtime, and security habits that keep containers predictable, lightweight, and safe to operate.

A good container image uses a minimal base, runs as a non-root user, has a clear ENTRYPOINT, externalizes configuration, and optionally exposes a health signal so orchestrators know when to restart or stop routing traffic.

In production, the biggest gains come from reducing attack surface and reducing mystery. Small images pull faster, health checks improve recovery, and non-root containers reduce the blast radius of compromise.

Why it matters

  • Improves security posture
  • Makes failures easier to detect and replace
  • Reduces build and deploy time

Common pitfall

  • Running as root by default
  • Stuffing secrets into the image
  • No health checks or readiness strategy
Non-root userHealth checkConfig via envSecrets baked in
Pick minimal base image→Create non-root user→Expose config via env→Add health signal

Interview-ready view

💬 Model Interview Answer

The best Docker habits are small images, non-root execution, explicit config, and replaceability. I also mention health checks because orchestration systems need a signal before sending traffic or deciding a container is unhealthy. A classic gotcha is baking secrets into images, which turns every registry pull into a secret-distribution event.

PracticeReason
Run as non-rootReduces privilege if the process is compromised
Externalize configPromotes one image across environments
Health checksImproves restart and routing decisions
❌ Wrong: oversized and over-privileged image
FROM ubuntu:latest
WORKDIR /app
COPY . .
ENV DB_PASSWORD=supersecret
CMD ["bash","start.sh"]
✅ Right: safer runtime image
FROM eclipse-temurin:21-jre
WORKDIR /app
RUN useradd -r -u 1001 appuser
COPY app.jar app.jar
USER 1001
ENV JAVA_OPTS=""
ENTRYPOINT ["sh","-c","java $JAVA_OPTS -jar /app/app.jar"]

The second version avoids root, avoids secrets in the image, and keeps the runtime surface narrower.

⚠️ Warning

Baking secrets into images creates a quiet security disaster

People often focus on image size and forget secret handling. If credentials are in ENV lines, shell history, or copied config files, anyone with image access may have them forever. Inject secrets at runtime through a secret manager or orchestrator integration, and run the process as a non-root user.

Mini takeaway
  • Prefer minimal runtime images
  • Run as non-root
  • Do not bake secrets into images
  • Expose health and config explicitly
Quiz

Which practice most directly reduces blast radius if the application process is compromised?

Correct answer: BB is correct because dropping root privileges limits what the process can do if exploited. A is about versioning convenience, not privilege boundaries. C increases exposure instead of reducing it. D is a security anti-pattern because it spreads secrets with the artifact.
