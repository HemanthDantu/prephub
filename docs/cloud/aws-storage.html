The Three Fundamental Storage Types

Before diving into S3, EBS, or EFS, you need a mental model for the three fundamentally different ways data can be stored. Every storage service in AWS — and in computing generally — falls into one of these three categories.

The Three Types — What They Actually Are

Object Storage
Stores data as flat objects — a blob of bytes with a unique key and metadata. No folders, no hierarchy, no concept of "edit a part of the file." Read the whole thing or write a new version.

Think: A giant key-value store where values are files.

AWS service: S3
Access: HTTP API (GET, PUT, DELETE by key)
Block Storage
Stores data as fixed-size blocks on a virtual disk. The OS manages the filesystem on top. You can read/write at any byte offset — overwrite individual blocks without rewriting the whole file.

Think: A hard drive attached to a server.

AWS service: EBS
Access: Mount as disk, OS reads/writes blocks
File Storage
Stores data as files in a hierarchy (folders within folders). Multiple servers can mount and share the same filesystem simultaneously — shared access with POSIX semantics.

Think: A shared network drive (NFS).

AWS service: EFS
Access: Mount as NFS, standard filesystem calls

The Analogy That Makes It Stick

Analogy — A Library
Object vs Block vs File Storage

Object storage (S3) = A warehouse full of sealed boxes. Each box has a unique barcode (key). You can only get a whole box out or put a whole box in. You can't open a box and change just one item inside — you take the box out, unpack it, modify it, and put an entirely new box back.

Block storage (EBS) = A personal bookshelf. Your OS organises the books (files) however it wants. You can open a book to any page and read or write just that page without touching other pages. The shelf is dedicated to you — no one else can use it simultaneously.

File storage (EFS) = A shared library. Multiple people can walk in and browse the same collection simultaneously. Files are in organised folders. Everyone sees the same data. Change a file — everyone reading it sees the change.

What is S3?

Simple Storage Service — the most used AWS service. Infinite capacity, 99.999999999% (11 nines) durability, and HTTP access from anywhere. Understanding S3 deeply is fundamental to almost every AWS architecture.

S3 From First Principles

S3 is a global object store accessible via HTTP. It was one of the first AWS services launched (2006) and is arguably the most important. Almost every AWS service either reads from or writes to S3: Lambda packages live there, CloudFormation templates, EC2 AMI snapshots, CloudWatch Logs archives, Athena query results, ML training data.

The most important things to understand about S3 before going deeper:

Infinite scale
S3 can store an unlimited number of objects. There is no capacity planning. A single bucket can hold a single file or 100 trillion files. Each object can be up to 5 TB. You never provision storage — it grows automatically.
11 nines durability
99.999999999% — this means if you stored 10 million objects, you'd expect to lose one object every 10,000 years. S3 achieves this by storing data redundantly across at least 3 Availability Zones, checksumming data, and continuously verifying integrity.
HTTP access
Every S3 object has a URL. Read, write, and delete via standard HTTP GET/PUT/DELETE requests. No database driver, no protocol to learn. Any language, any tool that can make HTTP calls can use S3.

The S3 Data Model — What You're Actually Working With

S3 has a deliberately simple data model. Understanding it prevents confusion about "folders" and helps you design efficient key structures.

S3 Data Hierarchy
S3 (Global Service)
└── Bucket (globally unique name, tied to a specific region)
    └── Object (key + value + metadata + version ID)
        ├── Key: "photos/users/alice/profile.jpg"
        ├── Value: raw bytes (the actual image data)
        ├── Size: up to 5 TB per object
        ├── Metadata: Content-Type, custom headers, user-defined
        └── Version ID: if versioning is enabled
There are no real folders in S3. S3 is a flat key-value store. "Folders" are an illusion — the console just groups objects whose keys share the same prefix before a / character. The key photos/users/alice/profile.jpg is one long string — not a file in three nested folders. This matters for performance: all objects with the same prefix go to the same partition. Heavy traffic to one prefix can cause throttling.

S3 Consistency Model

Since December 2020, S3 provides strong read-after-write consistency for all operations in all regions. Before this, there was eventual consistency for overwrites and deletes. This change means:

# Before Dec 2020 (eventual consistency):
s3.putObject(bucket, "config.json", newData)
Thread.sleep(500)  // had to wait — immediately reading might return old data
s3.getObject(bucket, "config.json")  // might still get old version!

// After Dec 2020 (strong consistency):
s3.putObject(bucket, "config.json", newData)
s3.getObject(bucket, "config.json")  // GUARANTEED to return new data
// No sleep needed. Any list operation will also include the new object.

Buckets, Keys & Objects

The mechanics of working with S3 — naming rules, key design patterns, how to structure your data, and the operations you'll use daily.

Buckets — The Container

A bucket is a container for objects. You create a bucket in a specific region and give it a name. That name must be globally unique across all AWS accounts — not just your account. If someone else has a bucket called my-photos, you can't create one with the same name.

# Bucket naming rules:
# - 3 to 63 characters
# - Lowercase letters, numbers, hyphens only
# - Must start with a letter or number
# - Cannot be formatted as an IP address (e.g., 192.168.0.1)
# - Globally unique across ALL AWS accounts

# Good bucket names (company-environment-purpose pattern):
acme-production-user-uploads
acme-dev-lambda-artifacts
acme-prod-data-lake-raw
acme-prod-static-website

# Create a bucket
aws s3api create-bucket \
  --bucket acme-production-user-uploads \
  --region us-east-1

# For regions OTHER than us-east-1, must specify LocationConstraint
aws s3api create-bucket \
  --bucket acme-eu-user-uploads \
  --region eu-west-1 \
  --create-bucket-configuration LocationConstraint=eu-west-1

# Block all public access immediately (always do this first)
aws s3api put-public-access-block \
  --bucket acme-production-user-uploads \
  --public-access-block-configuration \
    "BlockPublicAcls=true,IgnorePublicAcls=true,\
     BlockPublicPolicy=true,RestrictPublicBuckets=true"

Object Keys — Design Them Carefully

The key is the full path of the object within the bucket. How you design keys affects performance, cost, and queryability. S3 partitions data by key prefix — high-traffic prefixes can hit S3's per-prefix throughput limit (3,500 PUT/s, 5,500 GET/s per prefix partition).

## Pattern 1: Date-based partitioning (for time-series data)
logs/2024/01/15/app-server-1.log
logs/2024/01/15/app-server-2.log
logs/2024/01/16/app-server-1.log
# Good for: archiving, querying by date range with Athena
# Bad for: high-volume writes to the same date prefix cause throttling

## Pattern 2: ID-based (for user/entity data)
users/a3f7b291/profile-photo.jpg
users/b8d2c490/profile-photo.jpg
users/c1e9f344/profile-photo.jpg
# User IDs as UUIDs naturally distribute across partitions
# S3 hashes key prefixes to determine partition
# Random-looking prefixes = better distribution

## Pattern 3: Environment separation
production/uploads/users/photo.jpg
staging/uploads/users/photo.jpg
# One bucket per environment vs prefix-based separation
# Security: prefix-based is harder to lock down per team
# Recommend: separate buckets per environment

## Pattern 4: Old anti-pattern (pre-2018 S3) — random prefix
# Before S3 automatic partitioning improvements, teams used to add
# random prefix to avoid hot partitions:
3a7f/logs/2024/01/15/app.log  ← random hash prefix
# No longer needed. Modern S3 handles this automatically.
# Don't add random prefixes — they hurt readability for no benefit

Core S3 Operations — SDK Examples

S3Client s3 = S3Client.builder().region(Region.US_EAST_1).build();

// Upload an object
s3.putObject(
    PutObjectRequest.builder()
        .bucket("acme-production-user-uploads")
        .key("users/a3f7b291/profile-photo.jpg")
        .contentType("image/jpeg")
        .serverSideEncryption(ServerSideEncryption.AWS_KMS)
        .metadata(Map.of("userId", "a3f7b291", "uploadedAt", Instant.now().toString()))
        .build(),
    RequestBody.fromBytes(imageBytes)
);

// Download an object
ResponseBytes<GetObjectResponse> result = s3.getObjectAsBytes(
    GetObjectRequest.builder()
        .bucket("acme-production-user-uploads")
        .key("users/a3f7b291/profile-photo.jpg")
        .build()
);
byte[] imageData = result.asByteArray();
String contentType = result.response().contentType(); // "image/jpeg"

// List objects with a prefix
ListObjectsV2Response listing = s3.listObjectsV2(
    ListObjectsV2Request.builder()
        .bucket("acme-production-user-uploads")
        .prefix("users/a3f7b291/")  // list all of user a3f7b291's objects
        .maxKeys(100)
        .build()
);
listing.contents().forEach(obj ->
    System.out.println(obj.key() + " - " + obj.size() + " bytes"));

// Delete an object
s3.deleteObject(DeleteObjectRequest.builder()
    .bucket("acme-production-user-uploads")
    .key("users/a3f7b291/profile-photo.jpg")
    .build());

// Check if object exists (without downloading it)
try {
    HeadObjectResponse head = s3.headObject(HeadObjectRequest.builder()
        .bucket("acme-production-user-uploads")
        .key("users/a3f7b291/profile-photo.jpg")
        .build());
    long size = head.contentLength(); // Just metadata, no download
} catch (NoSuchKeyException e) {
    // Object doesn't exist
}
Use HeadObject to check existence, not GetObject. HeadObject fetches only metadata headers — no body download. GetObject downloads the entire object. If you're checking whether a 5GB file exists, HeadObject costs ~$0 extra. GetObject would charge you for 5GB of data transfer.

Storage Classes & Costs

S3 offers 7 storage classes at dramatically different price points. Choosing correctly can reduce storage costs by 80–95%. Each class trades retrieval speed and cost against storage price.

Why Storage Classes Exist

Not all data needs to be instantly accessible. A user profile photo that's loaded on every page view needs millisecond retrieval. A compliance audit log from 5 years ago that no one has opened in 3 years could take hours to retrieve — and that's fine. S3 charges less per GB for less immediately accessible data.

The Six Storage Classes — Detailed

S3 Standard
~$0.023/GB/month · Retrieval: milliseconds · Min duration: none
The default. Data stored in 3+ AZs. Designed for frequently accessed data that needs immediate retrieval. No minimum storage duration — delete after 1 second and you pay for 1 second. Use for: website assets, active user data, application data accessed daily.
S3 Intelligent-Tiering
~$0.023/GB (frequent) → $0.0125/GB (infrequent) · Monitoring fee: $0.0025/1000 objects
AWS automatically moves objects between Frequent Access and Infrequent Access tiers based on usage patterns (30 days without access → moves to IA tier, restoring to Frequent on next access). Use for: unpredictable access patterns where you don't know which objects will be hot vs cold. Good default for most production data.
S3 Standard-IA (Infrequent Access)
~$0.0125/GB/month storage · $0.01/GB retrieval · Min: 30 days · Min object: 128KB
Lower storage price but you pay per retrieval. 3 AZs. 30-day minimum billing — if you delete an object in 5 days, you're charged for 30. Use for: backups, disaster recovery data, files accessed once a month or less. Trap: if you access data frequently, retrieval fees make it MORE expensive than Standard.
S3 One Zone-IA
~$0.01/GB/month · $0.01/GB retrieval · Min: 30 days · Only 1 AZ
Like Standard-IA but stored in ONE AZ only. 20% cheaper. If that AZ fails, your data could be lost. Use for: re-creatable data (thumbnails, transcoded video, data that can be regenerated), secondary backups. Never for: primary data you can't afford to lose.
S3 Glacier Instant Retrieval
~$0.004/GB/month · Retrieval: milliseconds · Min: 90 days
Archive storage with instant retrieval. Cheapest option that still provides millisecond access. 90-day minimum. Use for: medical images, news media, data accessed once per quarter but needing immediate access when needed.
S3 Glacier Flexible Retrieval (formerly Glacier)
~$0.0036/GB/month · Retrieval: minutes to hours · Min: 90 days
Archival storage — retrieval takes 1–12 hours (Expedited: 1–5 min, Standard: 3–5 hours, Bulk: 5–12 hours). Use for: annual financial data, compliance records, data you access once per year. Classic Glacier use case.
S3 Glacier Deep Archive
~$0.00099/GB/month · Retrieval: 12–48 hours · Min: 180 days
Cheapest AWS storage tier. Magnetic tape equivalent. Retrieval takes 12–48 hours. 180-day minimum. Use for: data that must be retained for legal/compliance reasons (7-10 years) but will almost never be accessed. Medical records, financial records required by regulation.

Cost Comparison — Real Numbers

Standard-IA:            $0.0125 × 100,000 × 12 = $15,000/year (+ retrieval)
One Zone-IA:            $0.01 × 100,000 × 12   = $12,000/year (+ retrieval, one AZ)
Glacier Instant:        $0.004 × 100,000 × 12   = $4,800/year (+ retrieval)
Glacier Flexible:       $0.0036 × 100,000 × 12  = $4,320/year (+ retrieval)

# Saves $26,412/year — but only valid if you access it < once/year

Setting the Storage Class

# At upload time
aws s3api put-object \
  --bucket acme-production-user-uploads \
  --key logs/2022/report.csv \
  --body report.csv \

Versioning & Lifecycle Policies

Versioning keeps every version of every object — protecting against overwrites and deletes. Lifecycle policies automatically transition or expire objects — managing cost without manual work.

Versioning — Every Change Preserved

When versioning is enabled on a bucket, S3 preserves every version of every object. Overwriting an object creates a new version — the old one is retained. "Deleting" an object creates a delete marker — the object isn't actually removed, just hidden. You can restore any previous version.

Real-World Use Case
Developer accidentally overwrites a production config file

Without versioning: the previous config is gone. You're recovering from memory or a separate backup. With versioning: the previous version is right there. Restore with one API call. Takes seconds.

Other use cases: document management (keep all revisions of a contract), audit requirements (prove what a file contained at a specific point in time), website assets (roll back a CSS file that broke the site).

# Enable versioning on a bucket
aws s3api put-bucket-versioning \
  --bucket acme-production-user-uploads \
  --versioning-configuration Status=Enabled

# Now uploading the same key creates a NEW version:
aws s3api put-object --bucket acme-prod --key config.json --body v1.json
# VersionId: 111aaa
aws s3api put-object --bucket acme-prod --key config.json --body v2.json
# VersionId: 222bbb

# List ALL versions of an object
aws s3api list-object-versions \
  --bucket acme-prod \
  --prefix config.json
# Returns both versions with their IDs and LastModified timestamps

# Get a specific version
aws s3api get-object \
  --bucket acme-prod \
  --key config.json \
  --version-id 111aaa \    # explicitly request v1
  config-v1-restored.json

# "Delete" with versioning enabled → creates delete marker
aws s3api delete-object --bucket acme-prod --key config.json
# DeleteMarkerVersionId: 333ccc — object appears deleted but both versions exist

# Restore: delete the delete marker
aws s3api delete-object \
  --bucket acme-prod \
  --key config.json \
  --version-id 333ccc      # delete the DELETE MARKER — object is restored

# Permanently delete a specific version
aws s3api delete-object \
  --bucket acme-prod \
  --key config.json \
  --version-id 111aaa      # actually removes v1 permanently
Versioning and cost. With versioning enabled, every version of every object takes up storage and is billed. A frequently updated 1GB file with 100 versions = 100GB billed. Combine versioning with lifecycle policies to expire old versions automatically — keep the latest 5 versions, delete the rest after 30 days.

Lifecycle Policies — Automated Object Management

Lifecycle policies define rules that S3 applies automatically to objects: transition them to cheaper storage classes as they age, or expire (delete) them after a set time. Set it once, S3 runs it every day.

# Complete lifecycle policy — real production example
aws s3api put-bucket-lifecycle-configuration \
  --bucket acme-production-logs \
  --lifecycle-configuration '{
    "Rules": [
      {
        "ID": "application-logs-tiering",
        "Status": "Enabled",
        "Filter": {"Prefix": "app-logs/"},
        "Transitions": [
          {
            "Days": 30,
            "StorageClass": "STANDARD_IA"
          },
          {
            "Days": 90,
            "StorageClass": "GLACIER"
          },
          {
            "Days": 365,
            "StorageClass": "DEEP_ARCHIVE"
          }
        ],
        "Expiration": {
          "Days": 2555    
        }
      },
      {
        "ID": "delete-old-versions",
        "Status": "Enabled",
        "Filter": {},
        "NoncurrentVersionTransitions": [
          {
            "NoncurrentDays": 30,
            "StorageClass": "STANDARD_IA"
          }
        ],
        "NoncurrentVersionExpiration": {
          "NoncurrentDays": 90
        }
      },
      {
        "ID": "clean-up-incomplete-multipart",
        "Status": "Enabled",
        "Filter": {},
        "AbortIncompleteMultipartUpload": {
          "DaysAfterInitiation": 7
        }
      }
    ]
  }'
## What this policy does:
#
# Rule 1 — app-logs/ prefix:
#   Day 0:    Upload → S3 Standard ($0.023/GB)
#   Day 30:   → Standard-IA ($0.0125/GB) — probably not checking logs daily
#   Day 90:   → Glacier ($0.0036/GB) — unlikely to review 3-month-old logs
#   Day 365:  → Deep Archive ($0.00099/GB) — compliance retention only
#   Day 2555: DELETE (7 years — meets most compliance requirements)
#
# Rule 2 — Old versions:
#   After 30 days: non-current versions → Standard-IA
#   After 90 days: non-current versions DELETED
#   Prevents versioning from accumulating unlimited storage cost
#
# Rule 3 — Incomplete multipart uploads:
#   Abandon incomplete uploads after 7 days
#   Without this, partial uploads accumulate forever and are billed

S3 Security — Policies, Encryption & Block Public Access

S3 security has multiple overlapping layers. Understanding which layer controls what — and how they interact — prevents the most common AWS security breaches.

The Three Access Control Layers

S3 has three mechanisms for controlling access. They work together — a request must pass all applicable checks.

IAM Policies (identity)
Attached to a user, role, or group. Controls what that identity can do across all S3 buckets. Covered in Module 2. "This role can read from any bucket with the 'data-science' tag."
Bucket Policies (resource)
Attached to the bucket itself. Controls who can access THIS bucket. Can grant access to other AWS accounts, specific roles, or even public access. Includes a Principal element (unlike IAM policies).
Block Public Access (account/bucket)
A hard override that prevents any public access regardless of bucket policies. Set at account level to protect all buckets, or per bucket. Most important safety setting.

Bucket Policy — Real Production Examples

## Example 1: Allow only a specific IAM role to read objects
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowAppRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:role/AppServerRole"
      },
      "Action": ["s3:GetObject", "s3:ListBucket"],
      "Resource": [
        "arn:aws:s3:::acme-production-data",
        "arn:aws:s3:::acme-production-data/*"
      ]
    },
    {
      "Sid": "DenyHTTP",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::acme-production-data",
        "arn:aws:s3:::acme-production-data/*"
      ],
      "Condition": {
        "Bool": {"aws:SecureTransport": "false"}
      }
    }
  ]
}

## Example 2: Cross-account access — allow Account B to write logs
{
  "Statement": [{
    "Effect": "Allow",
    "Principal": {"AWS": "arn:aws:iam::ACCOUNT-B-ID:root"},
    "Action": "s3:PutObject",
    "Resource": "arn:aws:s3:::central-logs/account-b/*"
  }]
}

## Example 3: Public static website (specific use case — CloudFront OAC preferred)
{
  "Statement": [{
    "Effect": "Allow",
    "Principal": "*",
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::acme-public-website/*"
  }]
}
# This requires Block Public Access to be disabled — only for truly public sites

Block Public Access — The Safety Net

Block Public Access is a feature that overrides bucket and object ACL settings to prevent public access — even if a bucket policy accidentally grants it. It exists because S3 bucket misconfigurations have caused some of the largest data breaches in history.

# 4 settings — apply at account level for organisation-wide protection
aws s3control put-public-access-block \
  --account-id 123456789012 \
  --public-access-block-configuration \
    "BlockPublicAcls=true,\    # rejects PUT with public ACL
     IgnorePublicAcls=true,\   # ignores any existing public ACLs
     BlockPublicPolicy=true,\  # rejects bucket policies granting public access
     RestrictPublicBuckets=true" # restricts access to buckets with public policies

# Account-level BPA overrides per-bucket settings
# If you need ONE public bucket (e.g., website), disable BPA at bucket level
# while keeping it on at account level is NOT possible
# Solution: use CloudFront as the public-facing layer — S3 stays private

Encryption — At Rest and In Transit

## Three encryption options for data at rest:

# 1. SSE-S3 (simplest) — AWS manages keys, AES-256, no extra cost
aws s3api put-object \
  --bucket acme-data \
  --key file.csv \
  --body file.csv \
  --server-side-encryption AES256

# 2. SSE-KMS (most control) — your KMS key, audit trail in CloudTrail
aws s3api put-object \
  --server-side-encryption aws:kms \
  --ssekms-key-id arn:aws:kms:us-east-1:123:key/mrk-abc123 \
  ...
# Every read/write creates a CloudTrail entry → full audit of who accessed what
# KMS has a request rate limit — for very high throughput, use SSE-S3

# 3. SSE-C (client-provided key) — you manage keys, you send key on every request
# Rarely used — you must handle key management entirely

## Enforce encryption via bucket policy (deny unencrypted puts)
{
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:PutObject",
  "Resource": "arn:aws:s3:::acme-sensitive-data/*",
  "Condition": {
    "StringNotEquals": {
      "s3:x-amz-server-side-encryption": "aws:kms"

  }
}

Presigned URLs & S3 Event Notifications

Two of the most important S3 features for building real applications — presigned URLs for secure direct client access, and events for reactive, event-driven architectures.

Presigned URLs — Secure Time-Limited Access

A presigned URL is an S3 URL that grants temporary permission to perform a specific action (upload or download) on a specific object, without requiring the requester to have AWS credentials. The permissions and expiry are cryptographically embedded in the URL itself.

Upload flow — browser directly to S3 (the right way)

User uploads a profile photo — frontend never touches your server's bandwidth
1
User selects photo in browser. JavaScript calls your backend API: POST /api/upload-url with filename and content type.
2
Your backend (Lambda/ECS) calls S3Presigner to generate a presigned PUT URL valid for 5 minutes for that specific key. Returns the URL to the frontend.
3
JavaScript does a direct HTTP PUT to the presigned S3 URL. The photo goes straight from the user's browser to S3. Your server is not involved. Zero bandwidth on your servers.
4
S3 validates the signature, accepts the upload. Your backend is notified via an S3 Event Notification that the upload completed.
// Backend: generate presigned upload URL
public class UploadController {

    private static final S3Presigner presigner = S3Presigner.create();

    public UploadUrlResponse getUploadUrl(String userId, String contentType) {
        String key = "users/" + userId + "/profile." + getExtension(contentType);

        PresignedPutObjectRequest presigned = presigner.presignPutObject(
            PutObjectPresignRequest.builder()
                .signatureDuration(Duration.ofMinutes(5))
                .putObjectRequest(r -> r
                    .bucket("acme-production-user-uploads")
                    .key(key)
                    .contentType(contentType)
                    .metadata(Map.of("userId", userId)))
                .build()
        );

        return new UploadUrlResponse(
            presigned.url().toString(),   // The presigned PUT URL
            key,                             // So frontend knows the final key
            Instant.now().plusSeconds(300) // Expiry time
        );
    }

    public String getDownloadUrl(String userId) {
        // Generate time-limited download URL for private document
        return presigner.presignGetObject(
            GetObjectPresignRequest.builder()
                .signatureDuration(Duration.ofHours(1))  // 1 hour download window
                .getObjectRequest(r -> r
                    .bucket("acme-private-docs")
                    .key("contracts/" + userId + "/agreement.pdf"))
                .build()
        ).url().toString();
    }
}

// Lambda handler receiving S3 event
public class CSVProcessor implements RequestHandler<S3Event, Void> {

    private static final S3Client s3 = S3Client.create();
    private static final DynamoDbClient ddb = DynamoDbClient.create();

    public Void handleRequest(S3Event event, Context ctx) {
        event.getRecords().forEach(record -> {
            String bucket = record.getS3().getBucket().getName();
            String key = URLDecoder.decode(
                record.getS3().getObject().getKey(), StandardCharsets.UTF_8);

            log.info("Processing: {}/{}", bucket, key);

            // Download CSV
            ResponseInputStream<GetObjectResponse> stream =
                s3.getObject(GetObjectRequest.builder()
                    .bucket(bucket).key(key).build());

            // Parse and write to DynamoDB
            try (CSVReader csv = new CSVReader(new InputStreamReader(stream))) {
                csv.readAll().forEach(row -> ddb.putItem(buildRequest(row)));
            }

            // Move processed file to archive prefix
            s3.copyObject(CopyObjectRequest.builder()
                .sourceBucket(bucket).sourceKey(key)
                .destinationBucket(bucket)
                .destinationKey(key.replace("uploads/", "processed/"))
                .build());
            s3.deleteObject(DeleteObjectRequest.builder()
                .bucket(bucket).key(key).build());
        });
        return null;
    }
}

What is EBS?

Elastic Block Store — network-attached block storage for EC2. It's the virtual hard drive your EC2 instance boots from and your databases write to. Understanding EBS is essential for any stateful workload.

What Makes EBS Different From S3

EBS gives you a raw block device — like a hard drive. You format it with a filesystem (ext4, xfs, NTFS), mount it on an EC2 instance, and use it like a local disk. Your database writes to it, your OS boots from it, your application writes log files to it. The key properties:

Network-attached but feels local
EBS volumes are physically separate from your EC2 instance — they connect over AWS's high-speed internal network. But from the OS perspective, they look and feel like a local disk device (/dev/xvda, /dev/nvme0n1). You format it, mount it, run ls — it's just a disk.
AZ-specific
An EBS volume exists in exactly one Availability Zone. An EC2 instance in us-east-1a can only attach an EBS volume also in us-east-1a. To move a volume to another AZ: take a snapshot, create a new volume from the snapshot in the target AZ.
Persistent across instance stop/start
When you stop an EC2 instance, EBS volumes persist. The data is still there when you restart. This is the fundamental difference from instance store — EBS data survives instance lifecycle events.
Detachable and reattachable
You can detach an EBS volume from one instance and attach it to another (within the same AZ). Useful for: disaster recovery (attach failed instance's disk to a new instance), moving data between instances, forensic analysis.

The EBS Volume Lifecycle

# Create a volume
aws ec2 create-volume \
  --availability-zone us-east-1a \    # must match the EC2 instance's AZ
  --size 100 \                        # GB
  --volume-type gp3 \
  --encrypted \                       # always encrypt
  --kms-key-id arn:aws:kms:...:key/xxx
# Returns: VolumeId=vol-0abc123def456789

# Attach to a running EC2 instance
aws ec2 attach-volume \
  --volume-id vol-0abc123def456789 \
  --instance-id i-0abc123def456789 \
  --device /dev/sdf                   # name the device

# On the EC2 instance — format and mount the new volume
lsblk                               # see: nvme1n1 (the new volume)
sudo mkfs.xfs /dev/nvme1n1          # format with XFS filesystem
sudo mkdir -p /data                 # create mount point
sudo mount /dev/nvme1n1 /data       # mount it

# Make it persist across reboots (/etc/fstab)
UUID=$(sudo blkid -s UUID -o value /dev/nvme1n1)
echo "UUID=$UUID  /data  xfs  defaults,nofail  0  2" | sudo tee -a /etc/fstab

# Resize an existing volume online (no downtime!)
aws ec2 modify-volume \
  --volume-id vol-0abc123def456789 \
  --size 200 \          # increase from 100GB to 200GB
  --volume-type gp3     # can also change type online

# After AWS completes the modification, resize the filesystem too
sudo growpart /dev/nvme1n1 1   # extend the partition
sudo xfs_growfs /data          # extend the filesystem (XFS)
# ext4: sudo resize2fs /dev/nvme1n1
# No reboot required — filesystem now shows 200GB

EBS Volume Types & When to Use Each

Five volume types with dramatically different performance characteristics. Picking wrong costs money or hurts performance. Understanding IOPS, throughput, and latency is essential.

Key Metrics Before We Start

IOPS (I/O Operations Per Second)
How many read/write operations per second. Critical for databases doing many small random reads/writes (8KB-16KB). A MySQL server doing heavy transactional work needs high IOPS.
Throughput (MB/s)
How many megabytes per second can flow. Critical for sequential workloads — streaming large files, log processing, data warehouse scans. Hadoop reading 1GB files cares about throughput, not IOPS.
Latency
Time for a single I/O operation to complete. Critical for latency-sensitive databases (Redis on disk, PostgreSQL with write-ahead log). io2 Block Express: <1ms latency. gp3: single-digit milliseconds.

The Five Volume Types

gp3 — General Purpose SSD (the default, recommended)
3,000 IOPS baseline · Up to 16,000 IOPS · 125 MB/s baseline · Up to 1,000 MB/s · $0.08/GB-month
The best choice for most workloads. IOPS and throughput can be configured independently — you can increase throughput without adding more IOPS (unlike gp2). 20% cheaper than gp2. Use for: OS boot volumes, dev/test databases, small-to-medium production databases, ECS task storage.
io2 Block Express — Provisioned IOPS SSD (maximum performance)
Up to 256,000 IOPS · Up to 4,000 MB/s · Sub-millisecond latency · $0.125/GB + $0.065/provisioned IOPS
Maximum performance for the most demanding databases. 99.999% durability (vs 99.8-99.9% for other types). Sub-millisecond latency. Can be attached to multiple instances simultaneously (Multi-Attach). Use for: large production databases (Oracle, SAP HANA, large PostgreSQL/MySQL), I/O-intensive workloads where performance directly impacts revenue.
st1 — Throughput Optimised HDD (large sequential reads)
Max 500 IOPS · Up to 500 MB/s · $0.045/GB-month · Cannot be boot volume
Magnetic HDD — low IOPS but high throughput for big sequential reads. Much cheaper per GB than SSDs. Use for: log processing, data warehousing, Hadoop HDFS, Kafka brokers, any workload that reads/writes large files sequentially. Not for: any database that does random access.
sc1 — Cold HDD (archived, rarely accessed)
Max 250 IOPS · Up to 250 MB/s · $0.015/GB-month · Cheapest EBS option
Lowest cost EBS option. For cold data that is accessed infrequently. Use for: archives, compliance data, rarely accessed backups on EC2. If data is accessed more than a few times per month, st1 or gp3 is better.

Choosing the Right Type — Decision Guide

WorkloadTypeWhy
EC2 OS boot volumegp3Cost-effective, sufficient IOPS for OS
MySQL/PostgreSQL (medium)gp3 (16,000 IOPS)Random I/O, sufficient for most
MySQL/PostgreSQL (large prod)io2 Block ExpressHigh IOPS, sub-ms latency, 99.999% durability

gp2 is legacy — always use gp3. gp3 replaced gp2 as the default SSD type. gp3 is 20% cheaper and lets you scale IOPS and throughput independently. In gp2, IOPS scales with volume size (3 IOPS/GB). In gp3, you configure IOPS independently — you can have a 10GB volume with 16,000 IOPS. Always choose gp3 for new volumes.

Snapshots, Encryption & Multi-Attach

EBS snapshots are incremental backups stored in S3. Encryption protects data at rest. Multi-Attach enables high-availability cluster patterns.

EBS Snapshots — Incremental Backups to S3

Snapshots capture the state of an EBS volume and store it in S3 (you can't access the S3 bucket directly — AWS manages it). They're incremental: the first snapshot copies all data, subsequent snapshots copy only changed blocks. You pay only for the changed data each snapshot.

# Take a snapshot
aws ec2 create-snapshot \
  --volume-id vol-0abc123def456789 \
  --description "prod-db-before-major-update-2024-01-15" \
  --tag-specifications 'ResourceType=snapshot,Tags=[{Key=Name,Value=prod-db-backup}]'
# Returns: SnapshotId=snap-0abc123 (initially "pending", becomes "completed")

# Create volume from snapshot (same or different AZ/region)
aws ec2 create-volume \
  --snapshot-id snap-0abc123 \
  --availability-zone us-east-1b \   # can be different AZ from source
  --volume-type gp3 \
  --size 100                         # can be larger than original but not smaller

# Copy snapshot to another region (for DR)
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id snap-0abc123 \
  --destination-region eu-west-1 \
  --description "DR copy for production database" \
  --encrypted                        # re-encrypt with destination region's KMS key

# Automate snapshots with Data Lifecycle Manager (DLM)
aws dlm create-lifecycle-policy \
  --description "Daily snapshots, keep 7 days" \
  --state ENABLED \
  --execution-role-arn arn:aws:iam::123:role/DLMRole \
  --policy-details '{
    "PolicyType": "EBS_SNAPSHOT_MANAGEMENT",
    "ResourceTypes": ["VOLUME"],
    "TargetTags": [{"Key": "Backup", "Value": "daily"}],
    "Schedules": [{
      "Name": "Daily",
      "CreateRule": {"Interval": 24, "IntervalUnit": "HOURS", "Times": ["03:00"]},
      "RetainRule": {"Count": 7},
      "CopyTags": true
    }]
  }'

# Tag your volumes and DLM automatically backs them up daily, keeps 7
aws ec2 create-tags \
  --resources vol-0abc123def456789 \
  --tags Key=Backup,Value=daily

EFS — Elastic File System

A fully managed NFS (Network File System) that multiple EC2 instances and ECS containers can mount simultaneously. Elastic capacity — grows and shrinks automatically.

What Problem EFS Solves

EBS volumes can only be attached to one EC2 instance at a time (except Multi-Attach which is complex). S3 is object storage — no filesystem semantics. If you need multiple EC2 instances to share the same filesystem simultaneously with standard POSIX file operations — that's EFS.

Real Use Cases
When you actually need EFS

Web server fleet sharing content: 10 EC2 instances serving a WordPress site. All need read/write access to the same uploads folder. EFS mounts as /var/www/html/uploads on all 10 — they all see the same files simultaneously.

ECS container shared storage: Multiple ECS tasks need access to shared configuration files or a shared cache directory on disk. EFS provides an NFS mount that all containers can access.

Dev environments: Multiple developers' Lambda functions or ECS tasks share a common code repository or ML model files during development.

EFS Architecture

# EFS creates "mount targets" in each AZ of your VPC
# EC2 instances mount the filesystem via NFS from the mount target in their AZ

# Create EFS filesystem
aws efs create-file-system \
  --performance-mode generalPurpose \  # generalPurpose or maxIO
  --throughput-mode elastic \          # elastic (automatic) or provisioned
  --encrypted \
  --tags Key=Name,Value=shared-content
# Returns: FileSystemId=fs-0abc123456

# Create mount targets (one per AZ)
aws efs create-mount-target \
  --file-system-id fs-0abc123456 \
  --subnet-id subnet-0app1a \
  --security-groups sg-0efs123
# Repeat for subnet-0app1b and subnet-0app1c

# Security group for EFS mount target:
# Inbound: TCP 2049 (NFS) from sg-app (EC2 app servers)
# Outbound: none needed

# Mount on EC2 (using AWS EFS mount helper — handles TLS and retries)
sudo dnf install -y amazon-efs-utils
sudo mkdir -p /mnt/shared
sudo mount -t efs -o tls fs-0abc123456:/ /mnt/shared
# tls = encrypt in transit
# All instances in all AZs connecting to the same fs-id see the same files

# Persistent mount in /etc/fstab
echo "fs-0abc123456:/ /mnt/shared efs _netdev,tls 0 0" | sudo tee -a /etc/fstab

Instance Store — Ephemeral Local Storage

Physical NVMe SSDs soldered directly to the host machine your EC2 runs on. Blazing fast — and completely gone when your instance stops.

What is Instance Store?

Instance store volumes are physical storage devices attached directly to the host server your EC2 instance runs on. Unlike EBS (network-attached), instance store is truly local. The data travels microseconds, not milliseconds. But it's tied to the physical machine — stop, hibernate, or if the underlying host fails, the data is gone permanently.

Instance Store — Extreme Speed, Ephemeral

i3en.6xlarge: 2 × 7,500 GB NVMe SSDs · 1,600,000 IOPS · 16 GB/s throughput

Data is LOST when instance stops, hibernates, terminates, or the underlying host fails. Data SURVIVES reboots (same physical host, same drives).

EBS — Persistent, Network-Attached

io2 Block Express: up to 256,000 IOPS · 4 GB/s throughput

Data PERSISTS across all instance states. Data survives instance failure. Can be detached and reattached to another instance.

When to Use Instance Store

Good use cases
Buffer/cache: Memcached cache that can be cold-started from the database

Temporary data: ML training intermediate results, video transcoding scratch space

Replicated data: Cassandra or HDFS where data is replicated across multiple nodes — losing one node's disk is tolerable

Streaming logs: Buffer before shipping to S3/CloudWatch
Never use for
Databases (primary): Any data you can't afford to lose

Application state: Session data, shopping carts, anything a user cares about

Anything requiring persistence: If the host fails you lose data with no recovery path

Module 5 · Interview Q&A

Interview Questions — Storage

16 questions from beginner to architect. Storage questions appear in almost every AWS interview. Click to reveal answers.

Beginner

What is the difference between S3, EBS, and EFS? When would you use each? ▼
S3: Object storage. Key-value store for files. Access via HTTP API. Infinite scale. 11 nines durability. Use for: static assets, backups, data lakes, anything you access via URL. No filesystem — can't run a database on it.

EBS: Block storage. Virtual hard drive attached to one EC2 instance. Filesystem on top (ext4, xfs). High IOPS, low latency. Use for: database storage, EC2 root volumes, any app needing a traditional filesystem with random read/write. AZ-specific — can't share between instances (except Multi-Attach io2).

EFS: File storage. NFS-based shared filesystem. Multiple EC2 instances and ECS containers can mount it simultaneously. Elastic capacity. More expensive than EBS per GB. Use for: shared content (WordPress uploads), ML model sharing, developer shared environments. Not for databases (too slow for random I/O).
What are S3 storage classes and why do they exist? ▼
S3 storage classes exist because different data has different access patterns, and charging the same price for all data wastes money. Standard (~$0.023/GB) is for frequently accessed data. Standard-IA (~$0.0125/GB + retrieval fee) for monthly access. Glacier ($0.0036/GB) for annual access, retrieving in minutes-hours. Deep Archive (~$0.001/GB) for 7-10 year compliance archives with 12-48h retrieval. Intelligent-Tiering automatically moves objects between tiers based on actual usage. The key insight: a 5-year-old audit log taking hours to retrieve is fine. A product image that loads on every page view must return in milliseconds. Matching storage class to access pattern can reduce storage costs by 90%.
What is an EBS snapshot and how does incremental snapshotting work? ▼
An EBS snapshot is a point-in-time backup of an EBS volume stored in S3 (AWS-managed, not your own bucket). Incremental means: the first snapshot copies all data blocks. Each subsequent snapshot copies ONLY the blocks that changed since the last snapshot. All snapshots appear complete — when you restore from any snapshot, AWS assembles all the blocks needed. You only pay for what changed. Example: a 100GB volume with 5% daily change rate — first snapshot: 100GB. Each daily snapshot after: ~5GB. Delete old snapshots without affecting newer ones. Use cases: pre-change backup ("snapshot before this database migration"), disaster recovery (cross-region copy), creating new volumes from a known-good state.
What is a presigned URL in S3 and when would you use it? ▼
A presigned URL grants temporary permission to perform a specific S3 operation (GET or PUT) without requiring the requester to have AWS credentials. The permissions, expiry time, and specific resource are cryptographically embedded in the URL — anyone with the URL can perform the operation until expiry. Use cases: (1) Direct browser-to-S3 uploads: your backend generates a presigned PUT URL, returns it to the frontend, the browser uploads directly to S3 without your server handling any file data. Saves bandwidth and cost. (2) Secure file sharing: generate a presigned GET URL for a private document, email it to a client. They can download it without having AWS access. URL expires in 1 hour. (3) Authorised downloads from private buckets: your app server authorises the download request, generates a short-lived presigned URL, redirects the client to it.

Intermediate

What is S3 versioning and what problem does it solve? What's the cost implication? ▼
S3 versioning preserves every version of every object. When you PUT an object that already exists, instead of overwriting it, a new version is created and the old one is retained. Deleting an object creates a "delete marker" — the object appears deleted but all versions are still stored. The problem it solves: (1) accidental overwrites — restore previous version in seconds, (2) accidental deletes — remove the delete marker to restore, (3) audit trail — prove what a file contained at any point in time, (4) rollback capability — revert a changed config or document. Cost implication: every version takes up storage and is billed. A 1MB file updated 100 times = 100MB billed. Best practice: combine versioning with a lifecycle policy to expire non-current versions after 30-90 days to avoid unbounded storage growth.
How would you securely allow users to upload files directly to S3 from a React web app? ▼
Never expose AWS credentials to the browser. The correct pattern: (1) User selects a file in the browser. (2) Browser calls your backend API (e.g., POST /api/presigned-upload with filename and content type). (3) Backend validates the request (authentication, file type check, size limits), generates a presigned PUT URL using S3Presigner with a short expiry (5 minutes), returns the URL and the expected S3 key. (4) Browser does a direct PUT to the presigned URL with the file as the body. (5) S3 validates the signature and accepts the upload — no data ever goes through your backend. (6) S3 event notification triggers a Lambda to validate the uploaded file (virus scan, image validation), then moves it from a staging prefix to the final location. This pattern: keeps your servers free of file upload bandwidth, is scalable to any number of concurrent uploads, and keeps credentials server-side.
A DBA complains that the database is running slow. You suspect it's an I/O bottleneck on EBS. How do you diagnose and fix it? ▼
Diagnose: (1) CloudWatch EBS metrics: VolumeReadOps/WriteOps (IOPS consumed), VolumeReadBytes/WriteBytes (throughput), VolumeQueueLength (should be near 0, queuing = saturated). (2) If VolumeQueueLength is consistently > 1 or IOPS consumed = provisioned maximum → I/O bottleneck confirmed. (3) On the EC2 instance: iostat -x 1 to see real-time I/O utilisation, %util near 100% confirms bottleneck. Fix options: (1) gp3 volume — increase provisioned IOPS up to 16,000 (free up to 3,000 IOPS base, ~$0.005/IOPS/month above that). No downtime — modify the volume in place, filesystem recognises it immediately. (2) If you need >16,000 IOPS — migrate to io2 Block Express (up to 256,000 IOPS, higher cost). (3) Also check query patterns — often adding a database index reduces I/O more than any hardware change. (4) If reads are the bottleneck — add a read replica (RDS) or Redis cache to offload read traffic.
What are S3 lifecycle policies and how would you design one for an application log bucket? ▼
Lifecycle policies automate transitions between storage classes and object expiration. For application logs: logs from today are frequently accessed for debugging. After 30 days they're rarely needed. After a year, probably just for compliance. Design: (1) Day 0-30: Standard — active debugging, fast access needed. (2) Day 31-90: Standard-IA — monthly access pattern, retrieval fee acceptable. (3) Day 91-365: Glacier Flexible — quarterly access at most, hours to retrieve is fine. (4) Day 366-2555 (7 years): Deep Archive — compliance retention only, 48-hour retrieval acceptable. (5) Day 2555+: Delete — regulatory retention period met. Additionally: configure NoncurrentVersionExpiration to delete old versions after 90 days (prevent versioning cost explosion), and AbortIncompleteMultipartUpload after 7 days (clean up abandoned uploads). Result: 90%+ cost reduction vs keeping everything in Standard.

Advanced

What is the S3 consistency model and how did it change in December 2020? ▼
Before December 2020, S3 had eventual consistency for overwrite PUTs and DELETEs. This meant: if you wrote a new version of an object and immediately read it, you might get the old version. If you deleted an object and listed the bucket, the deleted object might still appear. Application code needed defensive sleeps or retries. After December 2020: S3 provides strong read-after-write consistency for ALL S3 operations: PUTs, DELETEs, and list operations. After a successful PUT, any subsequent GET will return the new object. After a successful DELETE, the object won't appear in lists. This is now a fundamental guarantee — no more defensive retries or sleep calls for consistency. The implementation is backed by an internal consensus layer that S3 added. Performance is unchanged — there's no latency cost for strong consistency.
How does Block Public Access interact with bucket policies? What's the difference between account-level and bucket-level BPA? ▼
Block Public Access (BPA) is a hard override for public access settings, evaluated before bucket policies. If BPA is enabled, even a bucket policy that says "Allow Principal: *" (everyone) is blocked. The four BPA settings: BlockPublicAcls (rejects requests with public ACLs), IgnorePublicAcls (ignores existing public ACLs), BlockPublicPolicy (rejects bucket policies that grant public access), RestrictPublicBuckets (restricts access even if policy allows public). Account-level BPA applies to ALL buckets in the account — even new ones created later. Bucket-level BPA applies to a specific bucket only. Important interaction: account-level BPA cannot be overridden by bucket-level settings. If account-level BPA is on, you cannot make any bucket public regardless of bucket-level settings. If you NEED one public bucket (e.g., a website), the recommended pattern is: disable account-level BPA for just that capability, keep bucket-level BPA on all private buckets, and use CloudFront with Origin Access Control as the public-facing layer so the S3 bucket itself stays private.
Design a complete media storage system for a video platform with 100 million users. Include storage strategy, lifecycle management, and delivery architecture. ▼
Architecture: (1) Ingestion: Users upload via presigned PUT URL to S3 raw-uploads/ prefix. S3 event triggers Lambda for validation. Lambda starts an MediaConvert transcoding job to create multiple quality levels (1080p, 720p, 480p, 360p) in HLS/DASH format. (2) Storage tiers: transcoded/ — S3 Standard (actively streamed, 2-3 years). archive/ — S3 Glacier Instant Retrieval for videos with views < 100/month. S3 Intelligent-Tiering auto-manages the Standard↔Archive boundary. (3) Lifecycle: Videos not viewed in 12 months → Glacier. Not viewed in 3 years → Deep Archive. Deleted by user → delete marker + 30-day expiry. (4) Delivery: CloudFront CDN in front of S3. HLS manifest (.m3u8) served from CloudFront edge. Signed cookies for subscriber-only content (not presigned URLs — cookies work for multi-file HLS streams). Custom origin for adaptive bitrate. (5) Cost optimisation: S3 Intelligent-Tiering for transcoded videos (most only popular for weeks). EFS for shared ML thumbnail generation pipeline. 6-region replication for top 1% of videos by views. (6) Metadata: DynamoDB for video metadata (title, duration, views, state). OpenSearch for search. CloudWatch + custom metrics for view counts.
