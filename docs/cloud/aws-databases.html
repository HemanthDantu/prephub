SQL vs NoSQL — The Mental Model You Need First

Before picking a database, you need to understand what each type fundamentally is, what trade-offs they make, and why both exist. This isn't a debate — they solve different problems.

What a Database Actually Does

At its core, a database stores data and lets you retrieve it reliably — even after crashes, power failures, and concurrent access from thousands of users. The hard part is not storing bytes (that's just a file) — it's the guarantees: consistency, durability, and efficient querying at scale.

SQL (Relational) Databases — Structured, Consistent, Flexible Queries

SQL databases organise data into tables with rows and columns. Tables relate to each other via foreign keys. You query using SQL — a powerful language that lets you ask arbitrary questions: joins, aggregations, filters, subqueries. The schema is defined upfront and enforced by the database.

What SQL databases are optimised for
Complex queries: "Give me all orders placed in the last 30 days for customers in the UK who haven't ordered in the previous 90 days, grouped by product category." SQL handles this elegantly. NoSQL would require multiple queries and application-side joining.

Relationships: A user has many orders, each order has many items, each item references a product. SQL enforces these relationships via foreign key constraints — the database prevents orphaned records.

ACID transactions: Transfer £100 from Alice's account to Bob's — either both the debit and credit happen, or neither does. SQL gives you true multi-row, multi-table atomicity.

NoSQL Databases — Flexible Schema, Massive Scale, Specific Access Patterns

NoSQL is an umbrella term for databases that don't use the SQL relational model. They trade SQL's flexibility for one or more of: massive horizontal scale, flexible schema, specific data model optimisations (key-value, document, graph, time-series).

What NoSQL databases are optimised for
Known access patterns: You always look up a user by their ID, and all their data is returned in one call. NoSQL optimises for this one query — data is stored together on disk so retrieval is one operation.

Horizontal scale: Relational databases are hard to shard across multiple servers. NoSQL (like DynamoDB) distributes data automatically across many servers based on partition key. Add more capacity = add more partition nodes.

Variable schema: User records might have different sets of attributes. NoSQL stores each item independently — no migrations needed when adding new fields.

The Spectrum — Not a Binary Choice

The cardinal rule: SQL when you need flexible queries and complex relationships. NoSQL when you have known access patterns and need to scale horizontally. Most production systems use both — SQL for core business data, NoSQL for high-throughput read/write paths.

OLTP vs OLAP — Another Critical Distinction

OLTP — Online Transaction Processing
Many concurrent users doing small, fast operations — insert an order, update a balance, look up a product. Row-oriented storage. Optimise for fast individual record access. RDS, Aurora, DynamoDB are OLTP.
OLAP — Online Analytical Processing
Few users running complex aggregations over large datasets — "total revenue by month by product line for the last 5 years." Column-oriented storage (faster for aggregations). Redshift, Athena are OLAP. Don't run analytics directly on your OLTP database.

Multi-AZ vs Read Replicas

The most commonly confused RDS concepts. They look similar — both involve a second database — but they solve completely different problems. Get this wrong and you'll either lose data or wonder why your read replica isn't a failover target.

The Core Difference — One Sentence Each

Multi-AZ = High Availability
A standby replica in a different AZ that takes over automatically if the primary fails. It's a failover target, not a read target. You cannot send queries to the standby.
Read Replica = Horizontal Read Scale
An asynchronous copy you send read queries to. Offloads read traffic from the primary. It's a query target, not a failover target — it won't automatically take over if primary fails.

Multi-AZ — Deep Dive

With Multi-AZ enabled, RDS maintains a synchronous standby replica in a different AZ. Every write to the primary is synchronously replicated to the standby before acknowledging success to your application. This means zero data loss on failover.

Multi-AZ — What happens when the primary fails
1
Primary DB instance in us-east-1a fails (hardware failure, AZ issue, patching).
2
RDS detects the failure within 10-30 seconds (health checks every few seconds).
3
RDS flips the DNS CNAME for your endpoint from the failed primary to the standby. This is the key — your application connects to the endpoint hostname, not a specific IP.
4
DNS propagates. New connections go to the former standby in us-east-1b — now promoted to primary.
5
Total time: typically 20-60 seconds. Existing connections break — application must reconnect. Connection pools handle this automatically.
## Key facts about Multi-AZ:
# - Synchronous replication = 0 data loss on failover
# - Standby is NOT accessible for reads (only for failover)
# - Automatic failover — no human action needed
# - The endpoint URL stays the same — app doesn't need to change
# - Failover also triggered by: OS patching, minor version upgrade, storage scaling

# Endpoint your app connects to (never changes during failover):
prod-postgres-primary.c1abcdef2ghi.us-east-1.rds.amazonaws.com

# RDS changes where this DNS name POINTS — primary vs standby — transparently

Read Replicas — Deep Dive

Read Replicas use asynchronous replication. Your application writes to the primary. Changes are streamed to replicas asynchronously. This means there's a small replication lag (usually milliseconds, can be seconds under heavy load). Replicas have a separate endpoint — your application must explicitly route read queries to the replica endpoint.

# Create a read replica
aws rds create-db-instance-read-replica \
  --db-instance-identifier prod-postgres-replica-1 \
  --source-db-instance-identifier prod-postgres-primary \
  --db-instance-class db.r6g.xlarge \   # can be smaller than primary
  --availability-zone us-east-1b \
  --publicly-accessible false \
  --multi-az false                       # replicas can also be Multi-AZ

# Replica endpoint (different from primary!)
Primary:  prod-postgres-primary.c1abc.us-east-1.rds.amazonaws.com

Backups, Security & Connection Management

RDS automated backups, point-in-time recovery, security hardening, and managing database connections correctly — all the operational topics that matter in production.

Automated Backups and Point-in-Time Recovery

RDS automatically backs up your database every day during your specified backup window, and continuously archives transaction logs to S3. This enables Point-in-Time Recovery (PITR) — restore to any second within your backup retention period.

# Automated backup configuration
--backup-retention-period 14   ← keep 14 days of backups (max: 35, min: 0 = disable)
--preferred-backup-window "03:00-04:00"   ← UTC backup window (low traffic time)

# Point-in-time restore: restore to 3 days ago at 14:30:00 exactly
aws rds restore-db-instance-to-point-in-time \
  --source-db-instance-identifier prod-postgres-primary \
  --target-db-instance-identifier prod-postgres-restored \
  --restore-time "2024-01-12T14:30:00Z" \
  --db-instance-class db.r6g.2xlarge \
  --db-subnet-group-name prod-db-subnet-group

# This creates a NEW instance — it doesn't overwrite the existing one
# After restore: verify data, then update application to point to restored instance
# Or use it as a recovery reference and copy specific data out

# Manual snapshots (persist beyond retention period, until you delete them)
aws rds create-db-snapshot \
  --db-instance-identifier prod-postgres-primary \
  --db-snapshot-identifier prod-postgres-pre-migration-2024-01-15

# Copy snapshot to another region (DR)
aws rds copy-db-snapshot \
  --source-db-snapshot-identifier \
    arn:aws:rds:us-east-1:123:snapshot:prod-postgres-pre-migration-2024-01-15 \
  --target-db-snapshot-identifier prod-postgres-dr-copy-2024-01-15 \
  --region eu-west-1

RDS Security — Four Layers

1
Network isolation — VPC + private subnets
Always deploy RDS in private subnets with no internet route. Use a DB Subnet Group spanning multiple AZs. The database should only be reachable from within your VPC.
2
Security group — allow only app servers
Security group on RDS: inbound TCP 5432 (PostgreSQL) from sg-app only. Nothing else. Not your developer's laptop, not a bastion host directly — use RDS Proxy or SSM Session Manager for developer access.
3
Database credentials — Secrets Manager
Never put database passwords in environment variables or code. Use Secrets Manager with automatic rotation. Your app fetches the current password at startup using an IAM role — if someone reads your environment variables, they can't connect to the DB.
4
Encryption — at rest and in transit
Enable storage encryption with KMS. Enforce SSL/TLS connections with the rds.force_ssl parameter. RDS provides certificates — configure your JDBC URL with ?sslmode=require&sslrootcert=....
// Connecting to RDS from Java — complete, secure setup
public DataSource createSecureDataSource() {

    // Fetch password from Secrets Manager (not env vars)
    SecretsManagerClient sm = SecretsManagerClient.create();
    String secret = sm.getSecretValue(r -> r
        .secretId("prod/postgres/master"))
        .secretString();
    Map<String, String> creds = objectMapper.readValue(secret, new TypeRef<>(){});

    // HikariCP connection pool — production config
    HikariConfig config = new HikariConfig();
    config.setJdbcUrl("jdbc:postgresql://prod-postgres.c1abc.us-east-1.rds.amazonaws.com:5432/mydb"
        + "?ssl=true&sslmode=verify-full&sslrootcert=/opt/rds-ca.pem");
    config.setUsername(creds.get("username"));
    config.setPassword(creds.get("password"));

    // Connection pool sizing — critical for RDS performance
    config.setMaximumPoolSize(20);          // max connections to DB
    config.setMinimumIdle(5);              // keep 5 connections open
    config.setConnectionTimeout(3000);     // 3s to get a connection from pool
    config.setIdleTimeout(300000);         // close idle connections after 5min
    config.setMaxLifetime(1800000);        // replace connections after 30min (handles DB restarts)
    config.setLeakDetectionThreshold(60000); // warn about connections held >1min

    return new HikariDataSource(config);
}

RDS Proxy

A managed connection pooler that sits between your application and RDS. Solves the connection exhaustion problem for serverless and microservice architectures.

The Problem — Connection Limits and Lambda

Every connection to PostgreSQL or MySQL consumes memory on the database server. A db.r6g.large PostgreSQL can handle ~87 connections. Now consider a Lambda function with 500 concurrent executions — each trying to open a database connection. That's 500 connections from Lambda alone, crashing through the limit and generating errors for real users.

Even without Lambda: 10 microservices × 10 instances × 5 pool connections = 500 connections. Add read replicas, staging environments, monitoring tools — you hit limits fast.

How RDS Proxy Solves This
RDS Proxy sits between your application and the database. Your application connects to the proxy (up to 60,000 connections from the app side). The proxy maintains a small pool of actual connections to RDS and multiplexes them. 10,000 Lambda function connections to the proxy = maybe 50 connections actually held on the RDS instance.

Additional benefits: automatic failover is faster (proxy holds connections during failover, no reconnect needed from app), IAM authentication for the proxy, Secrets Manager integration. 
# Create an RDS Proxy
aws rds create-db-proxy \
  --db-proxy-name prod-postgres-proxy \
  --engine-family POSTGRESQL \
  --auth '[{
    "AuthScheme": "SECRETS",
    "SecretArn": "arn:aws:secretsmanager:...:secret:prod/postgres/master",
    "IAMAuth": "REQUIRED"

  --vpc-subnet-ids subnet-0app1a subnet-0app1b \
  --vpc-security-group-ids sg-0proxy123

# Register the RDS instance with the proxy
aws rds register-db-proxy-targets \
  --db-proxy-name prod-postgres-proxy \

Aurora Architecture

Aurora is not just "a faster RDS." It's a fundamentally different database engine built from the ground up for cloud infrastructure. Understanding why it's faster and more resilient than standard RDS is essential.

The Core Aurora Innovation — Shared Distributed Storage

Standard RDS (MySQL, PostgreSQL) separates compute from storage — the database engine runs on EC2 and writes to EBS. EBS is in one AZ. Replication (for Multi-AZ) writes the data twice — to primary EBS, then over the network to a standby EBS in another AZ.

Aurora completely redesigns this. The storage layer is a shared, distributed, self-healing storage cluster that spans 3 AZs with 6 copies of your data. The database engine doesn't write to EBS — it writes to this distributed storage layer.

Aurora Architecture
Writer Instance
Aurora MySQL/PostgreSQL engine. Handles all writes. us-east-1a
Reader Instance 1
Read queries only. Shares storage. us-east-1b
Reader Instance 2
Read queries only. Shares storage. us-east-1c
Distributed Storage Layer (6 copies across 3 AZs)
us-east-1a
Storage node × 2
Copies 1 & 2
us-east-1b
Storage node × 2
Copies 3 & 4
us-east-1c
Storage node × 2
Copies 5 & 6
Quorum write: 4 of 6 must acknowledge. Quorum read: 3 of 6.

Why This Architecture Wins

Aurora vs RDS Performance
Aurora MySQL: 5x faster than RDS MySQL. Why? RDS MySQL writes data twice (redo log + double-write buffer). Aurora's storage does the redo log only — storage takes care of applying it. Less I/O from compute to storage.

Aurora PostgreSQL: 3x faster than RDS PostgreSQL. Aurora only sends WAL log entries to storage — not full page writes. Dramatically reduces write amplification.
Resilience
Can lose 2 AZs (out of 3) and still serve reads. Can lose 1 AZ and still serve writes (quorum of 4 out of 6 needed). Storage self-heals automatically — if a node fails, new node re-synced from remaining nodes automatically.

Aurora replicas have near-zero lag (milliseconds) because they read from the same storage tier — no replication over the network.

Aurora Serverless v2 & Global Database

Two Aurora features that address completely different problems — Serverless for variable workloads, Global Database for multi-region active-passive.

Aurora Serverless v2 — Instantly Scalable Compute

Standard Aurora has fixed compute (you choose the instance class). Aurora Serverless v2 scales the database engine compute up and down automatically in response to your actual load — in increments as fine as 0.5 ACU (Aurora Capacity Units). Scale from dev-level to full production without changing anything.

# Create Aurora Serverless v2 cluster
aws rds create-db-cluster \
  --db-cluster-identifier prod-aurora-serverless \
  --engine aurora-postgresql \
  --engine-version 15.4 \
  --serverless-v2-scaling-configuration \
    MinCapacity=0.5,MaxCapacity=128 \   # 0.5 to 128 ACUs
  --master-username dbadmin \
  --master-user-password-secret-id prod/aurora/master \
  --db-subnet-group-name prod-db-subnet-group \
  --vpc-security-group-ids sg-0db123 \
  --storage-encrypted \
  --deletion-protection

# Create the writer instance (also serverless):
aws rds create-db-instance \
  --db-instance-identifier prod-aurora-serverless-writer \
  --db-cluster-identifier prod-aurora-serverless \
  --db-instance-class db.serverless \   # special serverless class
  --engine aurora-postgresql

# ACU sizing:
# 1 ACU ≈ 2 GB RAM + proportional CPU
# 0.5 ACU = minimum (dev/test idle)
# 1 ACU = ~$0.12/hour per ACU
# Scales in milliseconds — no cold start lag like Aurora Serverless v1

Aurora Global Database — Multi-Region Active-Passive

Aurora Global Database replicates an Aurora cluster to up to 5 secondary regions with <1 second replication lag using Aurora's dedicated replication infrastructure (not standard database replication). If your primary region fails, promote a secondary region to primary in under a minute.

What is DynamoDB?

A fully managed NoSQL database with single-digit millisecond performance at any scale. No servers to manage, no schema migrations, no capacity planning beyond choosing a billing mode.

DynamoDB From First Principles

DynamoDB is AWS's proprietary NoSQL database, built for internet-scale workloads. It powers Amazon's own infrastructure — shopping cart, retail recommendations, Prime Video metadata. The design goals: predictable single-digit millisecond latency at any scale, fully managed with no operational overhead, automatic multi-AZ replication.

What Makes DynamoDB Fundamentally Different

Horizontal scale by default
Data is automatically distributed across many partitions based on the partition key. Adding more capacity = AWS adds more partitions. No sharding configuration, no manual redistribution. A single DynamoDB table can handle millions of reads/writes per second.
Schemaless (except keys)
Only the primary key (partition key + optional sort key) is defined at table creation. Every other attribute is completely flexible — each item can have different attributes. Add new fields to some items without any migration.
Key-based access
All queries must include the partition key. You cannot do "SELECT * WHERE status = 'pending'" — unless you create a Global Secondary Index. Access patterns must be designed upfront. This is the trade-off for the scale.

Global Secondary Indexes (GSI) — Query on Non-Key Attributes

A GSI lets you query DynamoDB using attributes that are NOT your primary key. It's a separate copy of your data, with a different key scheme, maintained automatically by DynamoDB. You pay for the extra storage and write capacity.

Real Problem GSI Solves

Your table has PK=userId, SK=orderId. You want all orders with status='PENDING' across all users for the warehouse dashboard. Without a GSI, you'd have to Scan the entire table — expensive and slow. With a GSI on status as the partition key: instant lookup of all PENDING orders.

      "IndexName": "status-createdAt-index",

        {"AttributeName": "status",    "KeyType": "HASH"},

Capacity Modes, Reads & Writes

How DynamoDB billing works, what Read Capacity Units and Write Capacity Units actually mean, and when to use provisioned vs on-demand mode.

Two Billing Modes

On-Demand Mode (Pay-Per-Request)
DynamoDB auto-scales instantly to any load. You pay per read/write request. No capacity planning. Good for: unpredictable traffic, new applications, spiky workloads, dev/test.

Cost: ~$1.25 per million writes, ~$0.25 per million reads. More expensive at steady high load than provisioned.
Provisioned Mode + Auto Scaling
You provision RCU (Read Capacity Units) and WCU (Write Capacity Units). Enable Auto Scaling and DynamoDB adjusts within min/max bounds automatically. Good for: predictable, steady traffic patterns. 60-70% cheaper than on-demand at consistent load.

Cost: ~$0.65/RCU-month, ~$0.25/WCU-month (provisioned)

RCU and WCU — What They Actually Mean

## Read Capacity Unit (RCU):
# 1 RCU = 1 strongly consistent read per second for items up to 4KB
#       = 2 eventually consistent reads per second for items up to 4KB
# For items larger than 4KB, multiple RCUs are consumed

# Examples:
# GetItem of a 1KB item, strong consistency:    1 RCU
# GetItem of a 1KB item, eventual consistency:  0.5 RCU
# GetItem of a 5KB item, strong consistency:    2 RCU  (ceil(5/4) = 2)
# Query returning 40KB total, eventually cons.: 5 RCU  (ceil(40/4) × 0.5 = 5)

## Write Capacity Unit (WCU):
# 1 WCU = 1 write per second for items up to 1KB
# Writes are more expensive — 1/1KB vs 1/4KB for reads

# Examples:
# PutItem of 1KB item:   1 WCU
# PutItem of 3.5KB item: 4 WCU  (ceil(3.5/1) = 4)
# DeleteItem of any size: same as write to same item

## Transactional reads/writes cost 2x:
# TransactGetItems: 2 RCU per item read
# TransactWriteItems: 2 WCU per item written

Hot Partitions — The Most Common DynamoDB Problem

DynamoDB distributes data across partitions by hashing the partition key. If all your writes go to the same partition key value, all traffic goes to one partition — a "hot partition." DynamoDB limits throughput per partition.

## ❌ Hot partition example
Table: SensorReadings, PK: sensorType

Every 10ms: PutItem(sensorType="TEMPERATURE", timestamp=now, value=22.5)
→ All writes to the SAME partition key
→ One partition handles ALL writes
→ That partition throttles
→ ProvisionedThroughputExceededException

## ✅ Fix: add randomisation to partition key
sensorType-0: "TEMPERATURE-0"
sensorType-1: "TEMPERATURE-1"
...spread writes across 10 partition keys, aggregate on read

## ✅ Or use a better primary key design
PK: sensorId (unique per sensor) + SK: timestamp
→ Writes distributed across all sensor IDs = no hot partition

Streams, Transactions & DAX

DynamoDB Streams — Change Data Capture

DynamoDB Streams is a time-ordered sequence of item-level changes in a table — a change log. Every insert, update, and delete is recorded. Records are retained for 24 hours. Lambda can poll this stream and react to changes.

Real Use Cases
What you can build with DynamoDB Streams

Replicate to OpenSearch: Order is inserted in DynamoDB → Stream → Lambda → OpenSearch. Now you can full-text search orders without polluting DynamoDB with scan operations.

Send notifications: Order status changes to SHIPPED → Stream → Lambda → SNS → push notification to user.

Cross-region replication (DynamoDB Global Tables): Under the hood, Global Tables uses Streams to replicate changes across regions.

Audit log: Any change to sensitive data → Stream → Lambda → append to immutable audit log in S3.

# Enable streams on a table
aws dynamodb update-table \
  --table-name Orders \
  --stream-specification \
    StreamEnabled=true,StreamViewType=NEW_AND_OLD_IMAGES
# StreamViewType options:
# KEYS_ONLY         — only PK/SK of changed item
# NEW_IMAGE         — item after the change
# OLD_IMAGE         — item before the change
# NEW_AND_OLD_IMAGES — both (most useful, highest cost)

# Create Lambda trigger from the stream
aws lambda create-event-source-mapping \
  --function-name OrderStreamProcessor \
  --event-source-arn arn:aws:dynamodb:us-east-1:123:table/Orders/stream/xxx \
  --batch-size 100 \
  --starting-position TRIM_HORIZON   # or LATEST for new records only
public class OrderStreamProcessor implements RequestHandler<DynamodbEvent, Void> {

    public Void handleRequest(DynamodbEvent event, Context ctx) {
        for (DynamodbStreamRecord record : event.getRecords()) {
            String eventType = record.getEventName(); // INSERT, MODIFY, REMOVE

            if ("MODIFY".equals(eventType)) {
                Map<String, AttributeValue> newImage = record.getDynamodb().getNewImage();
                Map<String, AttributeValue> oldImage = record.getDynamodb().getOldImage();

                String newStatus = newImage.get("status").s();
                String oldStatus = oldImage.get("status").s();

                if ("SHIPPED".equals(newStatus) && !"SHIPPED".equals(oldStatus)) {
                    // Order just shipped — send notification
                    String userId = newImage.get("userId").s();
                    notificationService.sendShippingNotification(userId);
                }
            }
        }
        return null;
    }
}

The difference between a DynamoDB system that handles millions of requests efficiently and one that collapses under load is almost always in the design.

Patterns — What Works

Pattern 1: TTL for Automatic Expiry

// DynamoDB automatically deletes this item after ttl timestamp
// No Lambda, no scheduled cleanup job needed

Pattern 2: Optimistic Locking (Version Control)

// Prevent concurrent writes from overwriting each other
// Add a version attribute; increment on every write with condition
ddb.updateItem(UpdateItemRequest.builder()
    .tableName("Products")
    .key(Map.of("productId", AttributeValue.fromS("prod-123")))
    .updateExpression("SET price = :price, version = :newV")
    .conditionExpression("version = :currentV")  // fails if another writer changed it
    .expressionAttributeValues(Map.of(
        ":price",   AttributeValue.fromN("29.99"),
        ":newV",    AttributeValue.fromN("6"),       // increment version
        ":currentV",AttributeValue.fromN("5")        // expected current version
    ))
    .build());

Anti-Patterns — What Breaks DynamoDB

Scan instead of Query
Scan reads the entire table — every item — and filters in memory. On a 10GB table, scan reads 10GB every call. Extremely expensive and slow.

Fix: Design your schema so every access pattern can be answered with a Query (uses PK) or GetItem (uses full key). If you can't avoid scanning, use a GSI or export to Athena/S3 for analytics.
Using DynamoDB like a relational DB
Storing normalised data in separate tables and doing multiple queries to "join" them in the application. You lose DynamoDB's speed advantage and make it more complex than just using RDS.

Fix: Denormalise into single-table design. Accept some data duplication in exchange for single-query access patterns.
Monotonically increasing sort keys
Using orderId = autoincrement or timestamp as sort key. All new writes go to the same "end" of the partition — a hot partition at the leading edge.

Fix: Use UUID sort keys, or add randomisation. If you need time-based ordering, use a partition key that distributes writes.
Storing large items
Items >4KB consume multiple RCUs per read, and items >400KB aren't supported at all. Storing full HTML pages, binary data, or large JSON blobs in DynamoDB is wasteful.

Fix: Store large data in S3, store the S3 key reference in DynamoDB.

Caching Patterns in Depth

How data flows between your application, cache, and database — the four patterns with their trade-offs, when each is appropriate, and what happens when things fail.

Pattern 1: Cache-Aside (Lazy Loading) — Most Common

The application checks the cache first. On a miss, it reads from the database and writes to the cache. The cache only contains data that has been requested at least once.

Cache-Aside — Read Flow
public Product getProduct(String id) {
    String cacheKey = "product:" + id;

    // 1. Check cache first
    Product cached = redis.get(cacheKey);
    if (cached != null) return cached;    // Cache HIT — return immediately

    // 2. Cache MISS — go to database
    Product product = db.findById(id);

    // 3. Populate cache for future requests
    redis.setex(cacheKey, 3600, product);  // TTL: 1 hour
    return product;
}
✓ Cache-Aside Advantages
  • Cache only contains requested data — no waste
  • Cache failure is non-fatal — fall through to DB
  • Resilient — if Redis goes down, app still works
  • Simple to implement and reason about
✗ Cache-Aside Disadvantages
  • First request always hits DB (cold start)
  • Cache stampede: many requests miss simultaneously, all hit DB
  • Stale data: TTL-based, not event-driven invalidation
  • Extra code for cache population logic

Pattern 2: Write-Through — Always Fresh Cache

Write to the cache and the database simultaneously on every write. Cache is always up to date.

Write-Through — Write Flow
public Product updateProduct(String id, ProductUpdate update) {
    Product updated = db.save(id, update);     // Write to DB first
    redis.set("product:" + id, updated);       // Immediately update cache
    return updated;
    // Future reads will find fresh data in cache
}

Database Selection Guide

Choosing the wrong database is one of the most expensive architectural mistakes. Here's the framework to make the right decision every time.

The Decision Tree

1
Do you need complex queries (JOINs, aggregations, ad-hoc SQL)?
Yes → SQL database (RDS or Aurora). No → Consider NoSQL.
2
Is your access pattern well-defined and key-based?
Yes → DynamoDB. Queries like "get user by ID, get user's orders" map perfectly. No defined access patterns → SQL (you can evolve queries freely).
3
Do you need millisecond latency AND infinite scale?
Yes → DynamoDB (guaranteed single-digit ms, horizontal scaling). Can tolerate 10-50ms and moderate scale? → RDS/Aurora is fine and much easier to query.
4
For SQL: how large will your dataset grow?
<1TB, standard OLTP → RDS PostgreSQL/MySQL (simpler, cheaper). >1TB or need >16,000 IOPS or sub-30s failover → Aurora. Multi-region active-passive → Aurora Global Database.
5
Is your workload read-heavy with repeated data?
Yes → Add ElastiCache (Redis) in front of any database. Cache frequently-accessed, rarely-changing data to reduce database load and improve latency.

Interview Questions — Databases

20 questions from beginner to architect. Databases are the most asked-about topic in system design interviews. Click to reveal answers.

Beginner

What does RDS actually manage for you? Why not just run MySQL on EC2? ▼
RDS manages: OS installation, database engine installation and patching, automated backups with point-in-time recovery, multi-AZ replication and automatic failover, read replica creation and management, storage auto-scaling, CloudWatch monitoring integration, SSL certificate management, and minor version updates. Running MySQL on EC2: you do all of this yourself. The engineering time to implement and maintain production-grade automated failover, backup/restore testing, and patch management is enormous. RDS trades some configuration flexibility for freedom from undifferentiated operational work. Most teams can't build better automated failover than AWS's 20+ years of distributed systems engineering.
What is the difference between RDS Multi-AZ and Read Replicas? ▼
Multi-AZ: For high availability. Synchronous standby replica in a different AZ. On primary failure, RDS automatically flips the DNS endpoint to the standby in 20-60 seconds. Zero data loss. The standby is PASSIVE — you cannot read from it. Your application uses one endpoint and failover is transparent. Cost: 2x instance price. Read Replicas: For read scaling. Asynchronous copies of the primary. You can send read queries to them (your application must explicitly route to their endpoint). There will be small replication lag (usually milliseconds). Can be in different regions. Multiple replicas possible (up to 5 for MySQL, 15 for Aurora). They do NOT automatically take over if the primary fails — you must manually promote a replica. Use together: Multi-AZ primary for HA + read replicas for offloading reports and read-heavy features.
What is the difference between SQL and NoSQL? When would you choose each? ▼
SQL (Relational): Structured tables with enforced schema. Relationships via foreign keys. Powerful SQL for any query pattern — joins, aggregations, subqueries. ACID transactions across multiple tables. Best for: complex queries you can't fully define upfront, data with complex relationships, financial/transactional data requiring strict consistency.

NoSQL (DynamoDB): Flexible schema per item. Access via primary key (partition key + sort key). No joins — denormalise. Horizontal auto-scaling. Guaranteed single-digit millisecond latency. Best for: defined, key-based access patterns, massive scale (millions of requests/second), uniform low latency, serverless architectures.

Choose SQL when you need to ask flexible questions of your data and relationships matter. Choose NoSQL when you know your access patterns upfront and need infinite scale or guaranteed millisecond latency.
What is a cache and why would you put it in front of a database? ▼
A cache is an in-memory store that holds the results of expensive operations so subsequent identical requests return instantly without re-running the operation. You add a cache (Redis via ElastiCache) in front of a database because: (1) In-memory reads are 100-1000x faster than database queries — microseconds vs milliseconds. (2) Repeated queries (same product viewed by 10,000 users) only hit the database once; subsequent requests get cached results. (3) Reduces database connection and query load — a cache-heavy architecture might reduce DB queries by 80%+, allowing a smaller/cheaper database to handle the same traffic. Trade-off: you introduce a new layer of complexity, must handle cache invalidation, and cache misses still hit the database.

Intermediate

Why is Aurora faster than standard RDS MySQL/PostgreSQL? ▼
Aurora's speed comes from its custom distributed storage engine. Standard MySQL/PostgreSQL write data twice — a redo log AND the full page (double-write buffer for MySQL). Aurora's database engine only writes redo log entries to the distributed storage layer — the storage nodes apply those log entries to materialise the data pages themselves. This "log is the database" approach dramatically reduces write amplification: Aurora writes ~7x less data to the storage layer per transaction. Additionally, Aurora's reader instances share the same storage as the writer — they don't receive replicated data over the network, they read directly from the shared storage tier. This is why Aurora read replicas have near-zero lag and failover is extremely fast (no need to catch up on replication backlog). Result: Aurora MySQL is ~5x faster than RDS MySQL, Aurora PostgreSQL is ~3x faster than RDS PostgreSQL on IOPS-intensive workloads.
Explain DynamoDB partition key, sort key, and GSI. When would you add a GSI? ▼
Partition key (PK): Hashed to determine which partition holds an item. Items with the same PK are stored together. To retrieve items efficiently, you must know the PK. Sort key (SK): Optional. Orders items within a partition. Enables range queries, begins_with queries, and sorting. Composite key (PK+SK) lets you store multiple related items together (e.g., PK=userId, SK=ORDER#timestamp → all of a user's orders). GSI: An additional index with its own PK and SK, automatically maintained by DynamoDB. Add a GSI when you have a query pattern that doesn't include your table's partition key. Example: your table has PK=orderId (for order lookups), but you need to query by status='PENDING' for warehouse dashboard. Create a GSI with PK=status, SK=createdAt. Now you can query all pending orders efficiently without a full table scan. GSIs cost extra (storage + write capacity consumed for index maintenance) but are often essential for NoSQL flexibility.
What is RDS Proxy and when do you need it? ▼
RDS Proxy is a managed database connection pooler between your application and RDS. It multiplexes thousands of application connections into a much smaller pool of actual database connections. You need it when: (1) Lambda functions connect to RDS — Lambda can create hundreds of concurrent executions each wanting their own DB connection. A db.r6g.large PostgreSQL allows ~87 connections. 500 concurrent Lambdas = 500 connections = crash. RDS Proxy handles 60,000 application connections and keeps only ~50-100 actual DB connections. (2) Many microservices — 10 services × 10 instances × 5 connections = 500 connections from the application side. (3) Faster failover — RDS Proxy holds connections during Multi-AZ failover, reducing failover time from 60+ seconds to 28 seconds as the proxy reconnects transparently. Additional benefit: supports IAM authentication for RDS, meaning your application never handles database passwords.
What are DynamoDB Streams and what can you build with them? ▼
DynamoDB Streams is a change log for a DynamoDB table — a time-ordered sequence of every item-level modification (insert, update, delete). Records are retained for 24 hours. A Lambda function polls the stream (Event Source Mapping) and processes changes in batches. You can build: (1) Real-time notifications — order status changes to SHIPPED → Lambda → SNS push notification. (2) Search index sync — new/updated product in DynamoDB → Lambda → OpenSearch for full-text search, without polluting DynamoDB with expensive Scans. (3) Audit log — any change to sensitive data → Lambda → append to immutable audit log in S3. (4) Aggregations — order placed → Lambda updates daily revenue counter in a separate DynamoDB summary table. (5) Cross-service events — user registered → Lambda → sends welcome email, creates default preferences, triggers ML personalization. DynamoDB Global Tables uses Streams internally for cross-region replication.

Advanced

How do you design DynamoDB for a social media application where you need: user profile, user's posts, all posts in chronological order, and posts by hashtag? ▼
Design access patterns first. Using single-table design: User profile: PK=USER#userId, SK=PROFILE. User's posts (newest first): PK=USER#userId, SK=POST#reversedTimestamp (use MAX_LONG - timestamp for newest-first sort). Timeline: Global feed needs a GSI or separate table. Add GSI: PK=entityType (value: "POST"), SK=createdAt. Query GSI for all posts, sorted by time. Posts by hashtag: For each hashtag in a post, write an item: PK=HASHTAG#tag, SK=POST#createdAt, with post details denormalised. Querying hashtag #aws → Query PK=HASHTAG#aws, sorted by SK descending = all posts with that hashtag. Key decisions: (1) Denormalise aggressively — duplicate data so each query can be answered independently. (2) Use begins_with("POST#") to distinguish post items from profile items in user partition. (3) GSI for chronological feed (entityType as PK would cause hot partition — consider sharding: entityType = "POST#0" to "POST#9" and fan-out the query).
Explain the cache stampede problem and how you'd prevent it in production. ▼
Cache stampede (dog-pile): when a popular cached item expires and many concurrent requests all simultaneously miss the cache, all flooding the database. Example: product page cached for 1 hour, 10,000 concurrent users. At expiry time, all 10,000 requests miss simultaneously → 10,000 identical database queries → database overwhelmed → response time spikes → timeout cascade. Solutions: (1) Mutex/Single-flight pattern: First miss acquires a distributed lock (Redis SETNX), fetches from DB, populates cache. Other misses wait for the lock, re-check cache, find it populated. Only one DB query per key. (2) Probabilistic early expiration: When checking cache, occasionally "expire" items early based on how close to expiry and how long the DB fetch takes. Refresh the item before the deadline. (3) Background refresh: Cache items never expire; a background job proactively refreshes them before expiry. (4) TTL jitter: Add ±10% randomisation to TTL so items don't all expire simultaneously. (5) Stale-while-revalidate: Return stale cached data immediately while triggering an async background refresh. User gets fast response, cache updates in the background.
Design the database layer for an e-commerce order management system handling 100,000 orders/day with requirements: strong consistency for payments, fast order lookup by order ID and by user, weekly analytics, 7-year compliance retention. ▼
Multi-database architecture: Aurora PostgreSQL (Multi-AZ + 1 reader): Core order and payment data. Strong ACID transactions for payment processing — debit customer, create order, update inventory atomically. Multi-AZ for HA (payments can never be down). Read replica for reporting queries without affecting transactional performance. Schema: orders, order_items, payments, customers tables with proper foreign keys. RDS Proxy for connection pooling. DynamoDB: Order status lookups (customer-facing). PK=orderId for instant O(1) lookup. GSI on userId+createdAt for "my orders" page. Copy order status here from Aurora via DynamoDB Streams→Lambda sync. DynamoDB serves the 100K daily reads from customers checking status at millisecond latency without hitting Aurora. ElastiCache Redis: Cache frequently-viewed active orders, product catalogue, user profile. 1-hour TTL for product data, 5-minute TTL for order status. Analytics: Daily job exports Aurora data to S3 (Parquet format). Athena or Redshift for weekly analytics queries — never run analytics on Aurora directly. 7-year retention: Lifecycle policy moves exported data: S3 Standard (1yr) → Standard-IA (3yr) → Glacier (7yr). Cost: pennies per GB per year for archival.
