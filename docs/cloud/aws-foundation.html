What is AWS & Cloud Computing?

The fundamental shift — from owning hardware to renting capability. Why cloud exists, what it truly solves, and the mental model every developer needs before writing a single line of AWS code.

The World Before Cloud — The Real Pain

To understand AWS deeply, you first need to feel the pain it solved. Imagine it's 2004 and you're building a web application. Here's what stood between your idea and production:

Physical procurement
You submit a PO for servers. Finance approves it. Vendor ships in 4-6 weeks. You rack and cable it yourself or pay someone to. You've spent $100K before writing a line of production code.
Capacity guessing
How much traffic will you get in 18 months? Over-provision and you've wasted money. Under-provision and your site crashes on launch day. You're always wrong. The cost of being wrong is enormous.
Undifferentiated work
Your engineers spend time on network switches, RAID configuration, OS patching, physical security, data center contracts. None of this helps your customers. It's pure overhead.

Amazon built AWS initially to solve its own problem — every internal team at Amazon was reinventing the same infrastructure. In 2006, they opened it to the public. The core insight: infrastructure is a utility, like electricity. You don't generate your own power — you plug into the grid.

The Cloud Value Proposition — Precisely

Cloud isn't just "someone else's computers." That misses the point entirely. Here's what actually changed:

Old World (On-Prem)Cloud (AWS)Why It Matters
Capital expense upfrontOperational expense, pay-per-useLower barrier to experiment; no sunk cost
6–8 week provisioningSeconds via APIShip faster; respond to demand in real time
Fixed capacityElastic — scale up or downMatch cost to actual demand
Manage everythingChoose your abstraction levelFocus engineering on business logic
One location (or expensive DR)Global in minutesLow latency everywhere; built-in resilience
Your security expertiseAWS + your shared responsibilityLeverage AWS's full-time security teams

The Three Service Models — With Real AWS Examples

This distinction matters because it tells you exactly how much you're responsible for:

IaaS — Infrastructure as a Service
AWS gives you raw infrastructure. You control everything from the OS up.

AWS examples: EC2 (virtual machines), EBS (block storage), VPC (networking)

You manage: OS patching, runtime installation, app deployment, security configuration

Use when: You need full control, have specific OS requirements, or running legacy software
PaaS — Platform as a Service
AWS manages the platform — OS, runtime, patching. You deploy code and configure.

AWS examples: RDS (managed DB), Elastic Beanstalk, Lambda

You manage: Application code, data, configuration

Use when: You want to focus on business logic and let AWS handle operational overhead
SaaS — Software as a Service
Fully managed end-to-end product. You configure and use it.

AWS examples: Amazon WorkMail, AWS Chime, Cognito (as a service)

You manage: Data and configuration only

Use when: The service does exactly what you need and flexibility isn't required

Five Key Cloud Characteristics (NIST Definition)

The National Institute of Standards and Technology defines cloud computing with five characteristics. Understanding these helps you reason about any cloud service:

1
On-demand self-service
Provision compute, storage, databases — without human interaction from the provider. You click or call an API; it's done in seconds. No sales calls, no tickets, no waiting.
2
Broad network access
Access over standard networks (internet or private connections) from any device. AWS services are accessible via HTTPS from anywhere.
3
Resource pooling
AWS serves thousands of customers from the same hardware using multi-tenancy. This is how AWS achieves economies of scale — your S3 bucket and Netflix's S3 bucket run on the same physical infrastructure, completely isolated.
4
Rapid elasticity
Capacity scales up or down automatically or on-demand. From the outside it appears infinite. You never think "do we have enough servers" — you think "what should the auto-scaling policy be."
5
Measured service
Usage is monitored, measured, and reported. You pay for exactly what you consume — GB-hours of storage, request counts, compute-seconds. No flat fee for unused capacity.

AWS Account — The Container for Everything

Every resource in AWS lives inside an AWS Account. Think of it as a completely isolated container with its own resources, billing, and identity boundaries.

AWS Account — What it contains
Root user: The email/password combination used to create the account. Has unrestricted access to everything. Should be locked away — use IAM users for daily work.

Resources: All your EC2 instances, S3 buckets, databases, Lambda functions, etc. live inside the account.

Billing: All charges accrue to the account. One credit card, one invoice.

IAM principals: Users, roles, and groups that can access the account's resources.

Default limits (Service Quotas): Every service has default limits (e.g. 5 VPCs per region). You can request increases.
Root user warning: The root user can do anything — including actions that cannot be done by any IAM user, like closing the account or changing support plans. Best practice: create root user, immediately create an IAM admin user, enable MFA on root, then never use root again. Lock the root credentials in a vault.

How You Interact With AWS

Every interaction with AWS — whether from the console, CLI, or SDK — ultimately becomes an HTTPS API call to an AWS endpoint. There is no magic. The console is just a web UI making API calls on your behalf.

Global Infrastructure

Regions, Availability Zones, Edge Locations — the physical backbone of AWS and why this architecture is the foundation of every resilient system you'll build.

The Three Layers of AWS Infrastructure

AWS physical infrastructure is organised into three layers, each serving a different purpose. Understanding these is critical because almost every architectural decision you make depends on it.

Regions
Geographic area containing multiple data centres. Completely isolated from other regions — no data crosses region boundaries unless you explicitly configure it. 33+ regions globally as of 2024. You choose which region to deploy to.
Availability Zones (AZs)
One or more discrete data centres within a Region, each with independent power, cooling, and networking. Connected to each other via low-latency, high-bandwidth private fibre links. Typically 3–6 per Region. This is where fault isolation lives.
Edge Locations
Points of Presence (PoPs) at internet exchange points globally — 400+ locations. Used by CloudFront (CDN), Route 53 (DNS), WAF. Serve cached content close to users. Not full regions — just content delivery and DNS.

Regions — Deep Dive

A Region is the highest-level unit of AWS infrastructure. Each region is completely independent — it has its own control plane, its own endpoints, and its own copy of every service. AWS never moves your data between regions automatically.

Why Regions Exist
Data residency, latency, and fault isolation

Data residency: GDPR requires EU citizen data to stay in the EU. You deploy to eu-west-1 (Ireland) or eu-central-1 (Frankfurt). AWS guarantees your data never leaves that region without your explicit action.

Latency: Your users are in Tokyo. You deploy to ap-northeast-1 (Tokyo). 30ms round-trip instead of 180ms from us-east-1. The difference is noticeable to users.

Fault isolation: If a catastrophic event (hurricane, power grid failure, fibre cut) hits us-east-1, your ap-southeast-1 deployment is completely unaffected.

Availability Zones — The Key to High Availability

This is where the real engineering happens. AZs are isolated failure domains within a region. If data centre A catches fire, data centres B and C in the same region continue running. The AZ names (us-east-1a, us-east-1b) are logical names — the actual physical mapping varies per account (AWS shuffles them to prevent everyone deploying to "1a").

❌ Single-AZ deployment

All resources in us-east-1a. If that AZ has a power failure, cooling issue, or network partition, your entire application is down. AWS SLA doesn't cover single-AZ failures — those happen.

✅ Multi-AZ deployment

App servers in 1a, 1b, and 1c. Load balancer spans all three. RDS standby in 1b. If 1a fails, traffic automatically routes to 1b and 1c — zero downtime. This is the minimum for any production system.

AZ selection in practice

# Always spread across AZs. Example with EC2 Auto Scaling:
aws autoscaling create-auto-scaling-group \
  --auto-scaling-group-name my-app-asg \
  --availability-zones us-east-1a us-east-1b us-east-1c \  # spread instances
  --min-size 3 --max-size 12 --desired-capacity 6

# RDS Multi-AZ — primary in 1a, standby in 1b, automatic failover
aws rds create-db-instance \
  --db-instance-identifier mydb \
  --multi-az \           # this flag is all you need
  --db-instance-class db.t3.medium \
  --engine postgres

Edge Locations & CloudFront

Edge Locations are not full regions — they don't run EC2 or RDS. They serve one purpose: get content close to users. When a user in Mumbai requests your website hosted in us-east-1, CloudFront serves cached content from the Mumbai Edge Location — ~10ms instead of ~200ms.

What runs at Edge Locations
CloudFront (CDN) — cached static assets, APIs
Route 53 — DNS resolution worldwide
AWS WAF — web application firewall
Lambda@Edge — code executed at edge
AWS Shield — DDoS protection
What doesn't run at Edge
EC2 instances — compute only in regions
RDS databases — only in regions
Lambda (standard) — only in regions
S3 buckets — only in regions (though cached by CF)
VPCs — only in regions

Regional vs Global Services

This is a common source of confusion. Some AWS services are regional (you specify a region when you create resources), while others are global (one instance serves everywhere).

Global ServicesRegional Services
IAM (users, roles, policies)EC2 (instances per region)
Route 53 (DNS)VPC (one per region)
CloudFront (distributions)S3 (buckets in specific regions)
AWS OrganizationsRDS, DynamoDB, Lambda
AWS Billing & Cost ExplorerCloudWatch (per region)
Practical implication: When you switch regions in the AWS Console, your EC2 instances, RDS databases, and VPCs change. But your IAM users and Route 53 hosted zones stay the same — they're global. This confuses many beginners who can't find their EC2 instance after accidentally switching regions.

Local Zones & Wavelength Zones

Beyond standard Regions and AZs, AWS has two extensions for specialised use cases:

Local Zones
Extensions of a Region placed closer to large population centres (e.g., Los Angeles, Chicago, Boston). Run EC2, EBS, RDS with single-digit millisecond latency to that metro area. Useful for video production, gaming, financial trading near specific cities.
Wavelength Zones
AWS infrastructure embedded within telecom providers' 5G networks. Sub-millisecond latency to mobile devices. Used for AR/VR, real-time gaming, autonomous vehicle data processing directly at the 5G edge.

Shared Responsibility Model

The most important mental model in AWS security — knowing exactly what AWS owns vs what you own. Every security incident involves someone misunderstanding this boundary.

The Core Concept

AWS does not secure everything for you. Security in the cloud is a shared responsibility. AWS secures the infrastructure. You secure everything built on top of it. This isn't fine print — it's the foundational principle you must understand before building anything production-ready.

THE SHARED RESPONSIBILITY MODEL
YOU ARE RESPONSIBLE FOR (Security IN the cloud)
Customer Data Encryption (your keys) IAM configuration OS patching (EC2) Network / Firewall config Application security Platform config
AWS IS RESPONSIBLE FOR (Security OF the cloud)
Physical datacentres Hardware Network infrastructure Hypervisor / virtualisation Managed service software Global network

How the Boundary Shifts With Service Type

The critical insight is that the boundary moves depending on which type of service you use. The more managed the service, the more AWS takes responsibility for:

ServiceAWS ManagesYou Manage
EC2 (IaaS)Physical host, hypervisor, network fabricOS, patches, runtime, app code, IAM, security groups, data encryption
RDS (PaaS)+ OS, database engine, automated backups, patchesDB configuration, access control, schema, data encryption, instance type, VPC placement
Lambda (Serverless)+ Runtime, OS, scaling, execution environment isolationFunction code, IAM permissions, memory/timeout, environment variables, data
DynamoDB (Fully Managed)+ Hardware provisioning, data replication, availabilityTable design, access policies, encryption settings, data in tables
S3 (Object Storage)Durability (11 9s), hardware, replicationBucket policies, access control, encryption keys, public access settings, versioning

Real-World Violations — Where Things Go Wrong

Every major AWS security breach in the news violated the customer's side of the shared responsibility model. AWS's infrastructure was never compromised:

S3 bucket exposed publicly
Company stored customer data in S3. Bucket policy allowed * (everyone) to read. Data leaked. AWS's fault? No — AWS gave you the tool (bucket policies). You misconfigured it. AWS has "Block Public Access" settings specifically to prevent this.
IAM credentials in GitHub
Developer committed AWS access keys to a public GitHub repo. Attacker found them via automated scanning, spun up 10,000 EC2 instances for crypto mining in minutes. AWS's fault? No — AWS provided IAM. Developer leaked their own credentials.
EC2 instance compromised
EC2 running old, unpatched OS. Attacker exploited known vulnerability via public IP. AWS's fault? No — AWS manages the hypervisor. The OS on the instance is your responsibility to patch. (This would be different with RDS — AWS patches the DB engine.)
Unencrypted sensitive data
Company stored PII in DynamoDB without encryption. AWS's fault? No — DynamoDB offers encryption at rest, but you must enable it. If you don't configure encryption, AWS doesn't force it (though modern DynamoDB enables encryption by default now).

Your Key Responsibilities — Checklist Mindset

1
Identity & Access (IAM)
Who can access what? Least privilege principle. No wildcard (*) permissions in production. Rotate credentials. Use roles not users for services. Enable MFA everywhere.
2
Data protection
Encrypt data at rest (KMS) and in transit (TLS). You choose encryption — AWS provides the tools. You own the decision to use them.
3
OS & patch management
For EC2: your OS, your patches. Keep AMIs current. Use AWS Systems Manager Patch Manager for automation. For managed services (RDS, Lambda): AWS handles this for you.
4
Network controls
Security Groups, NACLs, VPC design — you configure these. Block all inbound by default. Only allow what's needed. No 0.0.0.0/0 inbound on SSH/RDP in production.
5
Application security
Your application code is entirely your responsibility. SQL injection, XSS, insecure deserialization — AWS cannot protect you from bugs in your own code (though WAF can help at the perimeter).
Key insight for architects: When designing a system, ask "which tier of service gives me the right responsibility balance?" If your team has security expertise and needs control → EC2 (IaaS). If you want to focus on product → Lambda/RDS (PaaS). The less you manage, the faster you move — but you give up some flexibility.

Every AWS Interaction is an HTTP Request

This is the most important mechanical fact about AWS. There is no special protocol. The AWS Console, CLI, and SDKs all make signed HTTPS requests to AWS service endpoints. Understanding this unlocks your ability to debug any AWS issue.

AWS Service Endpoints

Each AWS service in each region has a predictable DNS endpoint. All follow the same pattern:

Signature Version 4 (SigV4) — How Authentication Works

AWS uses Signature Version 4 (SigV4) to authenticate every request. This is a HMAC-SHA256 signing process that proves the request came from someone with valid credentials — without ever sending the secret access key over the wire.

Why not just send username + password?

Passwords over HTTP are interceptable. SigV4 works differently: you sign the request with your secret key, AWS verifies the signature. The secret key never travels. Also: the signature includes the request body and timestamp, so signatures can't be replayed or tampered with.

The 4 steps of SigV4 signing

1
Create a canonical request
Normalise the HTTP request into a consistent string: HTTP method + URI + query string + headers + hash of body. This string is deterministic — the same request always produces the same canonical form.
2
Create a string to sign
Combine: algorithm name (AWS4-HMAC-SHA256), timestamp, credential scope (date/region/service), and SHA256 hash of the canonical request from step 1.
3
Calculate the signing key
Derive a signing key from your secret access key + date + region + service + "aws4_request". This key is unique to this day/region/service — even if one derived key leaked, it couldn't be used for other services.
4
Calculate the signature
HMAC-SHA256(signing key, string to sign). Add this signature to the Authorization header. AWS performs the same calculation on their side with the secret key stored for your access key ID. If signatures match → authenticated.
# Pseudocode for SigV4 (the SDKs do this automatically)
secret_key = "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY"

# Step 3: Derive signing key
date_key    = HMAC("AWS4" + secret_key, "20240115")
region_key  = HMAC(date_key, "us-east-1")
service_key = HMAC(region_key, "s3")
signing_key = HMAC(service_key, "aws4_request")

# Step 4: Final signature
signature = HMAC(signing_key, string_to_sign)

# Authorization header format
Authorization: AWS4-HMAC-SHA256
  Credential=<AccessKeyId>/<Date>/<Region>/<Service>/aws4_request,
  SignedHeaders=<list of headers included in signature>,
  Signature=<hex-encoded signature>
You almost never implement SigV4 manually. AWS SDKs sign every request automatically using your configured credentials. Understanding SigV4 matters for: debugging auth failures, building custom HTTP clients, understanding what IAM actually validates, and implementing presigned URLs.

Credentials — Where They Come From

The SDK needs credentials (Access Key ID + Secret Access Key, optionally + Session Token) to sign requests. It looks for them in this priority order:

1
Environment variables
AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_SESSION_TOKEN. Highest priority — useful for CI/CD pipelines.
2
Java system properties (Java SDK only)
aws.accessKeyId, aws.secretAccessKey. Passed via -D flags.
3
AWS credentials file
~/.aws/credentials — set by aws configure. Profiles let you have multiple credential sets. Use with profile selection.
4
AWS config file / Instance profile / IAM role
When running on EC2, ECS, Lambda — the SDK automatically fetches temporary credentials from the instance metadata service (IMDS). This is the preferred method for production. No credentials stored anywhere.
# ~/.aws/credentials (created by 'aws configure')
[default]
aws_access_key_id = AKIAIOSFODNN7EXAMPLE
aws_secret_access_key = wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY

[production]
aws_access_key_id = AKIAI44QH8DHBEXAMPLE
aws_secret_access_key = je7MtGbClwBF/2Zp9Utk/h3yCo8nvbEXAMPLEKEY

# Use a specific profile
aws s3 ls --profile production
export AWS_PROFILE=production  # or set for session

# On EC2/Lambda — SDK calls this URL automatically for creds
http://169.254.169.254/latest/meta-data/iam/security-credentials/{role-name}
# Returns temporary AccessKeyId + SecretAccessKey + SessionToken
# Auto-refreshed before expiry — you never manage rotation

Request Errors — Reading AWS Error Responses

AWS returns errors as structured HTTP responses. Learning to read them saves hours of debugging:

# 403 Forbidden — authentication or authorisation failure
HTTP/1.1 403 Forbidden
{
  "Code": "AccessDenied",
  "Message": "User: arn:aws:iam::123456789:user/alice is not authorized
             to perform: s3:GetObject on resource: arn:aws:s3:::my-bucket/file.txt"
}
# This error tells you EXACTLY what: which principal, which action, which resource
# → Debug: check Alice's IAM policies, check bucket policy, check S3 Block Public Access

# 401 — Invalid credentials
{"Code":"InvalidClientTokenId", "Message":"The security token included in the request is invalid."}
# → Credentials are wrong or expired

# 400 — Request validation failure
{"Code":"ValidationError", "Message":"Value 'xyz' at 'instanceType' failed to satisfy constraint..."}
# → Your request parameters are wrong

# Throttling
{"Code":"ThrottlingException", "Message":"Rate exceeded"}
# → Implement exponential backoff and retry

Module 1 · Interview Q&A

Interview Questions — Foundation

Questions you'll face for any AWS role — from junior cloud developer to senior architect. Click to reveal detailed answers.

Conceptual / Beginner

What is the difference between a Region, Availability Zone, and Edge Location? ▼
Region: A geographic area (e.g., us-east-1, eu-west-1) containing multiple data centres. Completely isolated — data never crosses regions automatically. You deploy resources to a region. AZ: One or more physical data centres within a region, each with independent power/cooling/networking. AZs within a region are connected via low-latency private fibre. You deploy across multiple AZs for high availability. Edge Location: Points of Presence (400+) at internet exchange points globally. Used only for CDN (CloudFront), DNS (Route 53), and WAF. Not full regions — you can't run EC2 there. They serve cached content close to users for low latency.
Why would you use multiple AWS accounts instead of one? ▼
Multiple accounts provide: (1) Blast radius isolation — if prod account is compromised, dev account is unaffected. Resources are completely isolated by default. (2) Billing separation — easy to see exactly what each team/environment costs. (3) Permission isolation — different teams can't accidentally access each other's resources. (4) Service limit isolation — dev workloads can't exhaust prod's EC2 limits. Typical pattern: separate accounts for prod, staging, dev, security, logging, shared services. Managed via AWS Organizations with consolidated billing.
What is the AWS Shared Responsibility Model? Give a concrete example. ▼
AWS is responsible for security of the cloud — physical hardware, hypervisor, global network, managed service software. You are responsible for security in the cloud — data, IAM configuration, network controls, OS patching (for EC2), application security. Example: You run an RDS PostgreSQL database. AWS manages: the hardware, the host OS, the PostgreSQL software updates, automated backups, multi-AZ replication. You manage: who can connect (security groups, IAM), what data is stored, encryption keys, instance sizing, parameter groups. If someone accesses your database because you misconfigured the security group to allow 0.0.0.0/0 — that's your fault, not AWS's.
What is the difference between IaaS, PaaS, and SaaS? Give AWS examples. ▼
IaaS (Infrastructure as a Service): You get raw infrastructure. AWS manages hardware/network/virtualisation. You manage everything from OS up. Example: EC2, EBS, VPC. You install the OS, install your runtime, deploy your app, patch the OS. PaaS (Platform as a Service): AWS manages OS, runtime, patching. You deploy code and configure. Example: Lambda (no OS at all), RDS (AWS patches the DB engine), Elastic Beanstalk. SaaS (Software as a Service): Fully managed software product. You configure and use it. Example: Amazon WorkMail, Cognito, SES (you just call the email API). Key insight: as you move from IaaS → PaaS → SaaS, you give up control but gain operational simplicity.
What is SigV4 and why does AWS use it instead of just an API key in the header? ▼
Signature Version 4 (SigV4) is AWS's request authentication method. It works by signing the request using HMAC-SHA256 with your secret access key — the key itself is never sent. Why not just send an API key? Three reasons: (1) No secret transmission — if someone intercepts the request, they see only the signature, not the secret key that generated it. (2) Request integrity — the signature covers the request body and headers, so any tampering makes the signature invalid. (3) Replay protection — the signature includes a timestamp; requests older than 15 minutes are rejected, so a captured request can't be replayed later. AWS SDKs handle SigV4 automatically — you only need to understand it for debugging auth failures, implementing presigned URLs, or building custom HTTP clients.

Intermediate

How does the AWS SDK credential provider chain work? In what order does it look for credentials? ▼
The SDK checks these sources in order, using the first one that provides valid credentials: (1) Environment variables — AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_SESSION_TOKEN. (2) Java system properties (Java SDK) — aws.accessKeyId, aws.secretAccessKey. (3) AWS credentials file — ~/.aws/credentials, configured via `aws configure`. (4) ECS container credentials — for ECS tasks with a task role assigned. (5) EC2 instance metadata service (IMDS) — for EC2 instances with an IAM instance profile. The IMDS approach is best practice for production: no credentials stored on disk or in environment variables. The SDK calls http://169.254.169.254/latest/meta-data/iam/security-credentials/{role-name} automatically and refreshes temporary credentials before they expire.
You run an EC2 instance in us-east-1 and need to access S3 in the same region. Should you route through the public internet or use a VPC Endpoint? Why? ▼
Use a VPC Endpoint (Gateway Endpoint for S3). Reasons: (1) Security — traffic stays within AWS's private network, never touches the public internet. No exposure to internet threats. (2) Cost — data transfer to/from S3 via internet has outbound data transfer charges. VPC Gateway Endpoints are free. (3) Performance — private network is lower latency and more reliable than routing through internet gateway. (4) No need for NAT Gateway — instances in private subnets can reach S3 without a NAT Gateway (which costs money per GB). A Gateway Endpoint is a route table entry — traffic destined for S3's prefix list is routed via the endpoint instead of the internet gateway. Zero configuration in your application code.
What's the difference between an S3 bucket in us-east-1 and one in eu-west-1? Can an app in us-east-1 access both? ▼
Both are S3 buckets — same service, same API. The differences: (1) Data residency — data in eu-west-1 physically stays in Ireland. GDPR-relevant. Data in us-east-1 stays in N. Virginia. (2) Latency — an app in us-east-1 will have much lower latency accessing the us-east-1 bucket (~1-5ms) vs the eu-west-1 bucket (~80-100ms). (3) Data transfer costs — accessing the eu-west-1 bucket from us-east-1 incurs inter-region data transfer fees (~$0.02/GB). Same-region transfers are free. Yes, an app in us-east-1 can absolutely access both buckets — S3 is accessible over HTTPS from anywhere. The API call just goes to different endpoints: s3.us-east-1.amazonaws.com vs s3.eu-west-1.amazonaws.com.

Advanced / Architect Level

How would you design an AWS architecture to ensure zero data loss and <1 minute RTO (Recovery Time Objective) if an entire AWS Region goes down? ▼
This requires an active-active or active-passive multi-region architecture: Database: Aurora Global Database (replication lag <1s) or DynamoDB Global Tables (multi-master, active-active). RDS cross-region read replica with manual promotion for active-passive. Application: Deploy identical stacks in two regions (e.g., us-east-1 and us-west-2). Use Auto Scaling Groups and identical infrastructure in both. Traffic routing: Route 53 with health checks and failover routing policy. Health checks detect regional failure and automatically update DNS within 30-60 seconds. State/Sessions: ElastiCache cross-region replication or DynamoDB for session state — nothing stored in local application memory. Storage: S3 Cross-Region Replication (CRR) with same-object names so no app code changes needed. Secrets: Secrets Manager with cross-region replication. Key trade-off: this architecture costs 2x+ and significantly complicates operations. Ensure the business actually needs this SLA before building it.
Explain what happens at the infrastructure level when you click "Launch Instance" in the EC2 console. ▼
At a high level: (1) Console → API: Browser sends a signed HTTPS POST to ec2.{region}.amazonaws.com with a RunInstances API call, including your AMI ID, instance type, subnet, security groups, key pair, and IAM instance profile. (2) IAM authentication & authorisation: AWS authenticates the SigV4 signature (you are who you claim). IAM evaluates your policies to check you have ec2:RunInstances permission on the requested resources. (3) EC2 control plane: EC2 scheduling system selects a physical host in the AZ you specified (or across AZs if you didn't specify). The host must have capacity for your instance type. (4) Hypervisor: AWS's Nitro hypervisor creates an isolated virtual machine on the selected physical host. Nitro is custom hardware — a dedicated card handles virtualisation, giving near-bare-metal performance. (5) Networking: A virtual network interface (ENI) is created and attached. The physical host's NIC is configured with your VPC routing rules via the Nitro card. Security group rules are implemented at the hypervisor level — not in the OS. (6) AMI boot: The root EBS volume is created from your AMI snapshot. Instance boots, runs user data script if provided. (7) IMDS: The instance metadata service is available at 169.254.169.254. If you assigned an IAM role, the role's temporary credentials are available here. Total time: typically 30-90 seconds.
