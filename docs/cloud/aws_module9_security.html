The AWS Shared Security Model

Security on AWS is a shared responsibility. Understanding exactly where AWS's responsibility ends and yours begins is the foundation of every security decision you make.

The Dividing Line

AWS and you each own different layers of security. Misunderstanding this line is how data breaches happen — companies assume AWS protects something that they're actually responsible for.

AWS is responsible for (Security OF the Cloud)
Physical data centre security
Hardware (servers, networking, storage)
Hypervisor and virtualisation layer
Managed service infrastructure (RDS engine patches, S3 durability)
AWS global network
Regions, AZs, Edge Locations
You are responsible for (Security IN the Cloud)
IAM users, roles, and policies
What data you store and how you encrypt it
Network configuration (Security Groups, NACLs, VPC)
OS-level security on EC2 (patching, hardening)
Application code vulnerabilities
S3 bucket policies and public access settings
The most common misunderstanding: "AWS stores my data in S3, so AWS protects it." AWS guarantees S3's durability and availability. But if you accidentally set your bucket to public, or have an IAM misconfiguration that lets anyone read it — that's your responsibility. S3 data breaches are almost always IAM or bucket policy misconfigurations, not AWS infrastructure failures.

Security Shifts With the Service Model

RDS (Managed PaaS)DB credentials, network access, data, app queriesOS patches, DB engine patches, hardware
Lambda (Serverless)Function code, IAM roles, environment variablesOS, runtime patches, underlying infrastructure
S3 (Object Storage)Bucket policies, object ACLs, encryption settingsStorage infrastructure, durability replication

Encryption Fundamentals

Before KMS makes sense, you need a clear mental model of symmetric vs asymmetric encryption, what keys actually are, and why managing keys is the hard part.

Why Encrypt?

Encryption transforms readable data (plaintext) into unreadable data (ciphertext) using a mathematical algorithm and a key. Without the key, the ciphertext is computationally infeasible to reverse. Even if an attacker gets your S3 data or your database backup, encrypted data is useless to them without the key.

Encryption at Rest
Data encrypted when stored on disk. S3 objects, EBS volumes, RDS databases, DynamoDB tables can all be encrypted at rest. If someone steals the hard drive from a data centre, the data is unreadable without the decryption key.
Encryption in Transit
Data encrypted when travelling over a network. Always use TLS (HTTPS). Enforce HTTPS on S3 bucket policies. Use ssl=require on database connections. If someone intercepts network traffic, they see only encrypted ciphertext.

Symmetric vs Asymmetric Encryption

Symmetric — Same key encrypts and decrypts
AES-256 is the most common symmetric algorithm. Extremely fast — can encrypt gigabytes per second. Used for bulk data encryption (files, database records, S3 objects).

Problem: How do you securely share the key with someone else? If you email it, attackers can intercept it. This is the key distribution problem.
Asymmetric — Public/private key pair
RSA, ECC algorithms. Generate a key pair: public key (share freely) and private key (never share). Encrypt with public key, only the private key can decrypt. Or sign with private key, verify with public key.

Slower than symmetric — not practical for bulk data. Used for key exchange, digital signatures (TLS certificates, JWT signing), SSH authentication.

The Key Problem — Why KMS Exists

Encryption is only as strong as your key management. Storing your encryption key next to your encrypted data defeats the purpose — if an attacker gets both, they can decrypt everything. The hard questions are:

  • Where do you store encryption keys securely?
  • How do you rotate keys without re-encrypting all your data?
  • How do you control which services/people can use which keys?
  • How do you audit who decrypted what and when?
  • What happens if a key is compromised?

This is exactly what AWS KMS (Key Management Service) solves.

KMS — Key Management Service

The central encryption service in AWS. KMS manages encryption keys in hardware security modules (HSMs), integrates with almost every AWS service, and provides a full audit trail of every cryptographic operation.

What KMS Is

KMS is a managed service for creating and controlling cryptographic keys. Keys stored in KMS never leave the KMS service — you never download or handle the raw key material for symmetric keys. Instead, you call KMS to encrypt or decrypt, and KMS does the operation inside its secure HSM hardware. The key material never exists in your application's memory.

Key never leaves KMS
For symmetric KMS keys, the raw key material never leaves the KMS HSM. You call Encrypt/Decrypt APIs — the data goes to KMS, KMS returns the encrypted/decrypted result. The key itself is never exposed.
Full audit trail
Every KMS API call (Encrypt, Decrypt, GenerateDataKey) is logged in CloudTrail. You can see exactly who encrypted what, when, using which key. Essential for compliance (HIPAA, PCI-DSS, SOC2).
Fine-grained access control
KMS key policies control who can use or manage each key. Your Lambda can encrypt but not delete the key. Your DBA can decrypt but not re-share the key. Granular per-key IAM-based control.

Creating and Using a Customer Managed Key

# Create a CMK with a descriptive alias
aws kms create-key \
  --description "Production orders database encryption key" \
  --key-usage ENCRYPT_DECRYPT \
  --key-spec SYMMETRIC_DEFAULT \
  --policy '{
    "Version": "2012-10-17",
    "Statement": [
      {
        "Sid": "Enable account root to manage key",
        "Effect": "Allow",
        "Principal": {"AWS": "arn:aws:iam::123456789012:root"},
        "Action": "kms:*",
        "Resource": "*"
      },
      {
        "Sid": "Allow orders Lambda to encrypt/decrypt",
        "Effect": "Allow",
        "Principal": {"AWS": "arn:aws:iam::123456789012:role/OrdersLambdaRole"},
        "Action": ["kms:Encrypt", "kms:Decrypt", "kms:GenerateDataKey*"],
        "Resource": "*"
      },
      {
        "Sid": "Allow RDS to use key",
        "Effect": "Allow",
        "Principal": {"Service": "rds.amazonaws.com"},
        "Action": ["kms:Decrypt", "kms:GenerateDataKey*", "kms:CreateGrant"],
        "Resource": "*"
      }
    ]
  }'
# Returns: KeyId=mrk-abc123, Arn=arn:aws:kms:us-east-1:123:key/mrk-abc123

# Create a human-readable alias
aws kms create-alias \
  --alias-name alias/prod-orders-key \
  --target-key-id mrk-abc123

# Enable automatic key rotation (every 365 days — AWS handles it)
aws kms enable-key-rotation --key-id mrk-abc123

# Direct encrypt/decrypt (for small data < 4KB — e.g., secrets, tokens)
# Encrypt
aws kms encrypt \
  --key-id alias/prod-orders-key \
  --plaintext "my-sensitive-value" \
  --query CiphertextBlob --output text
# Returns base64-encoded ciphertext

# Decrypt
aws kms decrypt \
  --ciphertext-blob fileb://encrypted.bin \
  --query Plaintext --output text | base64 --decode

KMS Key Policy vs IAM Policy

KMS has its own resource-based policy — the key policy. Both must allow an action for it to succeed (same as cross-account S3). The key policy controls who can manage and use the key. IAM policies on roles control what the role can do with KMS.

## To allow an IAM role to use a KMS key, you need BOTH:

## 1. Key policy on the KMS key (allows the role):
{
  "Sid": "Allow Lambda to use key",
  "Effect": "Allow",
  "Principal": {"AWS": "arn:aws:iam::123:role/OrdersLambdaRole"},
  "Action": ["kms:Decrypt", "kms:GenerateDataKey"],
  "Resource": "*"
}

## 2. IAM policy on the role (allows KMS calls):
{
  "Effect": "Allow",
  "Action": ["kms:Decrypt", "kms:GenerateDataKey"],
  "Resource": "arn:aws:kms:us-east-1:123:key/mrk-abc123"
}

## If either is missing → AccessDenied

Envelope Encryption Deep Dive

The technique used by every AWS service that encrypts data with KMS. Understanding this explains why KMS isn't slow for large files and how encryption at scale actually works.

The Problem with Direct KMS Encryption

KMS's Encrypt API has a 4KB limit. You can't call KMS.Encrypt with a 10GB S3 object. And even if you could, sending gigabytes to KMS for every encryption operation would be extremely slow and expensive.

Envelope encryption solves this: use KMS to protect a symmetric data key, then use that data key locally to encrypt the actual data.

How Envelope Encryption Works

Encrypting data with envelope encryption
1
Generate Data Key: Call kms:GenerateDataKey with your KMS key ID. KMS returns TWO things: a plaintext data key (used immediately, then discarded) and an encrypted data key (encrypted with your KMS key — safe to store).
2
Encrypt your data locally: Use the plaintext data key to encrypt your actual data with AES-256 locally in your application. This is extremely fast — no network calls, no KMS involvement.
3
Discard plaintext key: Zero out the plaintext data key from memory. The plaintext key should never be stored anywhere.
4
Store together: Store the encrypted data + the encrypted data key together (e.g., in S3 as object metadata). The KMS key ID is also stored so you know which KMS key to use for decryption later.
Decrypting data with envelope encryption
1
Retrieve encrypted data key: Read the encrypted data key from storage (stored alongside the encrypted data).
2
Call KMS Decrypt: Pass the encrypted data key to kms:Decrypt. KMS decrypts it using the KMS key and returns the plaintext data key. KMS validates your identity and permissions before returning anything.
3
Decrypt data locally: Use the plaintext data key to decrypt the actual data locally in your application. Fast, no network calls.
4
Use and discard: Use the decrypted data. Zero out the plaintext data key from memory when done.
Why this is brilliant: (1) The KMS key never touches your data — only the small data key (32 bytes) goes to KMS. (2) Encrypting/decrypting large files is done locally at AES-256 speed — gigabytes per second. (3) Key rotation: to re-key all your data, you only need to re-encrypt the data keys — not the data itself. (4) The encrypted data key is useless without the KMS key — you've used the HSM to protect your data encryption key.

Envelope Encryption in Java Code

// AWS SDK handles envelope encryption automatically via the Encryption SDK
// But here's the manual approach to understand exactly what happens
KmsClient kms = KmsClient.create();

// Step 1: Generate data key (for encrypting data)
GenerateDataKeyResponse dkResp = kms.generateDataKey(
    GenerateDataKeyRequest.builder()
        .keyId("alias/prod-orders-key")
        .keySpec(DataKeySpec.AES_256)
        .build()
);

byte[] plaintextDataKey = dkResp.plaintext().asByteArray();  // use then discard
byte[] encryptedDataKey = dkResp.ciphertextBlob().asByteArray(); // store this

// Step 2: Encrypt data locally with the plaintext data key (AES-256)
byte[] encryptedData = aesEncrypt(myLargeData, plaintextDataKey);

// Step 3: Discard plaintext key (zero out memory)
Arrays.fill(plaintextDataKey, (byte) 0);

// Step 4: Store encryptedData + encryptedDataKey together
// (e.g., in S3 with encrypted data key as metadata)

// ============ DECRYPTION ============
// Step 1: Call KMS to decrypt the data key
DecryptResponse decryptResp = kms.decrypt(
    DecryptRequest.builder()
        .ciphertextBlob(SdkBytes.fromByteArray(encryptedDataKey))
        .build()
);

byte[] recoveredDataKey = decryptResp.plaintext().asByteArray();

// Step 2: Decrypt data locally
byte[] decryptedData = aesDecrypt(encryptedData, recoveredDataKey);
Arrays.fill(recoveredDataKey, (byte) 0); // discard after use

// In practice: use AWS Encryption SDK which handles all of this automatically
// aws-encryption-sdk-java: one call to encrypt, one to decrypt, handles envelope internally

KMS in Practice — Every Service

How KMS integrates with every major AWS service you've already learned about. Most services just need a KMS key ARN — AWS handles the envelope encryption transparently.

S3 — Object Encryption

# SSE-S3: AWS manages the key (free, no audit trail)
# SSE-KMS: Your CMK, full CloudTrail audit, $0.03/10K operations

# Encrypt specific object with SSE-KMS
aws s3api put-object \
  --bucket production-data \
  --key sensitive/report.pdf \
  --body report.pdf \
  --server-side-encryption aws:kms \
  --ssekms-key-id alias/prod-data-key

# Enforce SSE-KMS on all uploads via bucket policy
{
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:PutObject",
  "Resource": "arn:aws:s3:::production-data/*",
  "Condition": {
    "StringNotEquals": {
      "s3:x-amz-server-side-encryption": "aws:kms"
    }
  }
}
# Any upload without SSE-KMS is rejected

# Default encryption on bucket (all new objects use this key)
aws s3api put-bucket-encryption \
  --bucket production-data \
  --server-side-encryption-configuration '{
    "Rules": [{
      "ApplyServerSideEncryptionByDefault": {
        "SSEAlgorithm": "aws:kms",
        "KMSMasterKeyID": "alias/prod-data-key"
      },
      "BucketKeyEnabled": true
    }]
  }'
# BucketKeyEnabled=true: reduces KMS API calls by 99% (uses per-bucket data key)
# Without it: every S3 object read/write = one KMS API call = $0.03/10K

Secrets Manager

The right way to handle passwords, API keys, and other credentials in AWS. No more hardcoded secrets in code, environment variables, or config files.

The Secrets Problem

Applications need secrets to function: database passwords, third-party API keys, encryption keys, OAuth client secrets. These end up in the worst places:

❌ Where secrets should NOT live
Source code (committed to Git → accessible to everyone forever)
Environment variables (visible in process list, logs, crash dumps)
Config files on disk (anyone with filesystem access can read)
Docker image layers (docker inspect reveals env vars)
CloudFormation templates (stored in CloudFormation history)
✅ Secrets Manager — centralised, encrypted, rotated
Encrypted at rest with KMS (your CMK)
Access controlled by IAM (principle of least privilege)
Full CloudTrail audit of every access
Automatic rotation (Lambda rotates on schedule)
Versioning — roll back if rotation fails

Creating and Using Secrets

# Create a secret (JSON format for structured credentials)
aws secretsmanager create-secret \
  --name prod/orders/db-credentials \
  --description "Production orders database credentials" \
  --kms-key-id alias/prod-secrets-key \
  --secret-string '{
    "username": "orders_app",
    "password": "xK9#mN2$pL8@vQ5",
    "host": "prod-postgres.c1abc.us-east-1.rds.amazonaws.com",
    "port": 5432,
    "dbname": "orders"
  }'

# Create a simple string secret (API key)
aws secretsmanager create-secret \
  --name prod/payments/stripe-secret-key \
  --secret-string "sk_live_abc123..."

# Read a secret value
aws secretsmanager get-secret-value \
  --secret-id prod/orders/db-credentials \
  --query SecretString --output text | jq .password

# IAM policy for Lambda to read this specific secret
{
  "Effect": "Allow",
  "Action": "secretsmanager:GetSecretValue",
  "Resource": "arn:aws:secretsmanager:us-east-1:123:secret:prod/orders/db-credentials-*"
  // Note: secret names get a random suffix — use wildcard or exact ARN
}

Reading Secrets in Java — Production Pattern

@Component
public class SecretsCache {
    // Cache secret to avoid calling Secrets Manager on every DB connection
    // Secrets Manager has a soft limit of 10,000 API calls per second
    // At 1000 Lambda invocations/sec each calling SM → you'd hit limits

    private static final SecretsManagerClient sm = SecretsManagerClient.create();
    private static final Map<String, CachedSecret> cache = new ConcurrentHashMap<>();

    public static String get(String secretName) {
        CachedSecret cached = cache.get(secretName);
        if (cached != null && !cached.isExpired()) {
            return cached.value();  // Return cached value (TTL: 5 minutes)
        }

        String secretValue = sm.getSecretValue(r -> r
            .secretId(secretName))
            .secretString();

        cache.put(secretName, new CachedSecret(secretValue, Duration.ofMinutes(5)));
        return secretValue;
    }
}

// Usage in Lambda handler (outside handler method for warm start reuse)
public class OrderHandler {
    // Fetched once at cold start, cached for 5 minutes
    private static final String DB_CREDS = SecretsCache.get("prod/orders/db-credentials");
    private static final DbCredentials CREDS = parse(DB_CREDS);

    public void handleRequest(...) {
        // CREDS already loaded — no SM call per invocation
    }
}

Automatic Rotation — The Killer Feature

Secrets Manager can rotate secrets automatically on a schedule using a Lambda function. AWS provides pre-built rotation Lambdas for RDS, Redshift, DocumentDB, and ElastiCache. For other secrets, you write a Lambda following the rotation contract.

# Enable automatic rotation for RDS password
aws secretsmanager rotate-secret \
  --secret-id prod/orders/db-credentials \
  --rotation-lambda-arn arn:aws:lambda:us-east-1:123:function:SecretsManagerRDSPostgreSQLRotationSingleUser \
  --rotation-rules AutomaticallyAfterDays=30

# What happens during rotation:
# 1. Secrets Manager calls rotation Lambda with AWSPENDING version stage
# 2. Lambda creates new password in the database (ALTER USER ... PASSWORD ...)
# 3. Lambda sets AWSPENDING secret to new credentials
# 4. Lambda tests new credentials (connects to DB with new password)
# 5. If test passes: Lambda promotes AWSPENDING to AWSCURRENT
# 6. Old password is still in AWSPREVIOUS for 30 days (in case of issues)
# 7. Applications fetching the secret automatically get new credentials

# Your application must re-fetch the secret on rotation:
# Use the caching pattern above with short TTL (5 minutes)
# Or catch 401/authentication errors and re-fetch immediately

SSM Parameter Store

A lighter-weight, cheaper alternative to Secrets Manager for storing configuration and non-sensitive parameters. Part of AWS Systems Manager.

What SSM Parameter Store Is

SSM Parameter Store is a hierarchical key-value store for configuration data and secrets. It has two tiers: Standard (free, 10KB limit per param) and Advanced (paid, 8KB per param, higher throughput, TTL). Parameters can be encrypted with KMS (SecureString type) or stored as plain text.

Parameter Types

String
Plain text. Use for: non-sensitive config like environment name, feature flags, URLs. Example: /myapp/prod/environment = production
StringList
Comma-delimited list. Use for: allowed IP ranges, allowed domains, lists of values. Example: /myapp/prod/allowed-origins = https://app.acme.com,https://admin.acme.com
SecureString (encrypted)
Encrypted with KMS. Use for: passwords, API keys, sensitive config that doesn't need rotation. Cheaper than Secrets Manager for static secrets.

Hierarchical Path Structure

# Parameters organised as a path hierarchy
# Retrieve entire "folder" of params in one call

/myapp/
  prod/
    db-url = jdbc:postgresql://prod-db.../orders
    db-user = orders_app
    db-password = (SecureString, KMS encrypted)
    redis-url = redis://prod-redis.../
    feature-flags/
      new-checkout = true
      dark-mode = false
  staging/
    db-url = jdbc:postgresql://staging-db.../orders
    db-password = (SecureString)

# Create a SecureString parameter
aws ssm put-parameter \
  --name "/myapp/prod/db-password" \
  --value "xK9#mN2$pL8@vQ5" \
  --type SecureString \
  --key-id alias/prod-secrets-key \
  --description "Production database password"

# Get a parameter
aws ssm get-parameter \
  --name "/myapp/prod/db-password" \
  --with-decryption \             # required for SecureString
  --query "Parameter.Value" --output text

# Get all parameters in a path (one API call)
aws ssm get-parameters-by-path \
  --path "/myapp/prod" \
  --recursive \
  --with-decryption

Reading SSM in Lambda / Java

SsmClient ssm = SsmClient.create();

// Read single parameter
String dbPassword = ssm.getParameter(r -> r
    .name("/myapp/prod/db-password")
    .withDecryption(true))
    .parameter()
    .value();

// Read all prod configuration at once
Map<String, String> config = ssm.getParametersByPath(r -> r
    .path("/myapp/prod")
    .recursive(true)
    .withDecryption(true))
    .parameters().stream()
    .collect(Collectors.toMap(
        p -> p.name().replaceFirst("/myapp/prod/", ""),
        p -> p.value()
    ));

CloudTrail — Every API Call Logged

CloudTrail records every AWS API call made in your account — who called what, when, from where, and what the result was. It's the forensic record of everything that happens in your account.

What CloudTrail Captures

Every AWS API call — from the console, CLI, SDK, or service-to-service — generates a CloudTrail event. The event captures: the identity (who), the action (what), the resource (on what), the timestamp (when), the source IP (from where), and the request/response parameters.

Real Use Cases
When CloudTrail saves the day

Incident investigation: "Someone deleted the production S3 bucket at 2am." CloudTrail tells you exactly which IAM user/role called DeleteBucket, from which IP, at what time, and whether it was via console or CLI.

Compliance audit: "Prove that no one accessed patient records in HIPAA data between Jan 1 and Mar 31." Query CloudTrail logs for GetObject events on the sensitive bucket for that date range.

Security alerts: "Alert me when anyone calls CreateUser or AttachAdminPolicy." EventBridge rule watches CloudTrail → triggers Lambda → PagerDuty alert.

Cost analysis: "What Lambda functions were invoked more than expected last month?" CloudTrail has all Lambda:InvokeFunction events.

CloudTrail — WHO did WHAT

"This S3 bucket currently has public access enabled — it shouldn't." Resource configuration snapshots. Answers: what is the current state of my resource? did it change? does it comply?

WAF — Web Application Firewall

WAF protects your web applications from common web exploits at the HTTP layer — before requests reach your Lambda, ECS service, or ALB backend. Think of it as a security bouncer for your API.

What WAF Protects Against

Injection attacks
SQL injection (SQLi): ' OR 1=1-- in query params. XSS (Cross-Site Scripting): <script>steal_cookies()</script> in form fields. WAF detects and blocks these patterns.
Bot and scraping
Credential stuffing (trying thousands of stolen passwords). Content scraping. Inventory hoarding bots. AWS Managed Rules include known bad bot signatures.
Rate-based attacks
DDoS at HTTP layer (Layer 7). Block IPs that send more than N requests per 5 minutes. Geographic restrictions — block traffic from specific countries.

WAF Attaches to These Services

WAF is a web ACL (Access Control List) that attaches to: CloudFront (global edge — stops attacks at the edge before they reach your region), ALB (regional, protects traffic to ECS/EC2), API Gateway (REST API or HTTP API), and AppSync (GraphQL).

Setting Up WAF with Core Rule Sets

Cognito — User Identity Management

Cognito handles user sign-up, sign-in, MFA, social logins, and token management for your applications. Build complete auth without building auth infrastructure.

Cognito's Two Services

User Pools — User Directory
A user directory for your application. Handles sign-up, sign-in, password policies, MFA, forgot password, email/phone verification, social login (Google, Facebook, Apple). Returns JWT tokens (IdToken, AccessToken, RefreshToken) after successful authentication.

Cognito + API Gateway — The Complete Auth Flow

End-to-end: user login → API call with auth
1
User enters email/password in web app. Amplify calls Cognito InitiateAuth.
2
Cognito validates credentials. If MFA enabled: challenges for TOTP code. Returns IdToken (JWT), AccessToken, RefreshToken.
3
Frontend stores tokens (memory for IdToken/AccessToken, secure storage for RefreshToken). AccessToken expires in 1 hour.
4
Frontend calls API with IdToken in Authorization header. API Gateway validates token against Cognito's JWKS (cached). Extracts claims, passes to Lambda.
5
Lambda reads user's email, orgId, groups from event claims. No DB lookup needed for basic auth — it's in the token. Implements fine-grained authorisation.
6
When token expires: Amplify automatically uses RefreshToken to get new AccessToken/IdToken. User doesn't see the refresh — seamless.

Interview Questions — Security

16 questions from beginner to architect level. Security is tested in every AWS interview.

Beginner

What is the AWS Shared Responsibility Model? ▼
The Shared Responsibility Model divides security duties between AWS and the customer. AWS is responsible for security OF the cloud: physical data centres, hardware, networking, hypervisors, and the underlying managed service infrastructure. The customer is responsible for security IN the cloud: IAM users and policies, data encryption, network configuration (Security Groups, VPCs), OS-level security on EC2, application code security, and how data is stored. The boundary shifts based on service type. For EC2 (IaaS): you manage the OS and above. For RDS (PaaS): AWS manages the database engine and OS, you manage data and network access. For Lambda (Serverless): AWS manages everything below your function code. The most common mistake is assuming AWS protects your S3 bucket contents — AWS guarantees durability and availability, but access control and encryption are your responsibility.
What is KMS and why use it instead of your own encryption? ▼
AWS Key Management Service (KMS) is a managed service for creating and controlling cryptographic keys stored in hardware security modules (HSMs). Use it instead of rolling your own encryption because: (1) Key security: For symmetric keys, the key material never leaves the KMS HSM — you call Encrypt/Decrypt APIs and KMS does the operation inside its secure hardware. You never have raw key bytes in your application. (2) Audit trail: Every KMS API call (Encrypt, Decrypt, GenerateDataKey) is logged in CloudTrail. Prove who decrypted what data and when — critical for HIPAA, PCI-DSS, SOC2. (3) Access control: Key policies define who can use or manage each key — independent of application-level access. Your Lambda can decrypt but not delete the key. (4) Automatic rotation: Enable annual key rotation — AWS generates new key material, all previous encryptions still work. (5) Integration: Every major AWS service (S3, EBS, RDS, DynamoDB, SQS, Lambda) integrates natively with KMS.
What is envelope encryption? Why does it exist? ▼
Envelope encryption solves two problems: KMS's 4KB encryption limit and the performance cost of sending large data to KMS. Process: (1) Call GenerateDataKey — KMS returns a plaintext data key AND an encrypted copy of that key. (2) Use the plaintext data key to encrypt your actual data locally with AES-256 — very fast, no network calls. (3) Discard the plaintext data key from memory. (4) Store the encrypted data + encrypted data key together. To decrypt: send the encrypted data key to KMS.Decrypt (gets plaintext key back), use it to decrypt data locally, discard. Why it exists: (1) KMS can only encrypt up to 4KB directly — a 10GB file needs local encryption. (2) Local AES-256 is orders of magnitude faster than sending data to KMS over the network. (3) Key rotation: re-key all data by only re-encrypting the small data keys, not the data itself.
Where should you store database passwords in AWS? ▼
Use AWS Secrets Manager. Store the database credentials as a JSON secret: username, password, host, port, dbname. Your Lambda/ECS role has secretsmanager:GetSecretValue permission on that specific secret ARN. The application fetches the secret at startup (outside the handler for Lambda, for warm start reuse) and caches it for 5 minutes. Secrets Manager encrypts the secret with KMS, provides an audit trail in CloudTrail, and — most importantly — supports automatic rotation. For RDS, AWS provides a pre-built rotation Lambda that: creates a new DB password, updates the database, tests the new credentials, then promotes the new version. Rotating every 30-90 days. Alternatives: SSM Parameter Store SecureString is cheaper ($0) vs Secrets Manager ($0.40/month) but has no automatic rotation. For non-rotating static secrets, SSM is fine. Never store credentials in environment variables, config files, source code, or Docker images.

Intermediate

What is CloudTrail and how is it different from CloudWatch? ▼
CloudTrail records every AWS API call — who called what, when, from where, on which resource. It's a forensic audit log of account activity. "Who deleted the S3 bucket?" → CloudTrail. "Who changed this security group?" → CloudTrail. Logs are stored as JSON files in S3 and optionally streamed to CloudWatch Logs for querying. Essential for: security investigations, compliance audits, change tracking. CloudWatch collects operational metrics and logs from AWS services and your applications. CPU utilisation, request latency, error rates, application logs. Used for: monitoring performance, alerting on thresholds, dashboards, debugging. The distinction: CloudTrail = management/audit plane (API calls). CloudWatch = data/operations plane (metrics and application logs). Many security setups use both together: CloudTrail events → CloudWatch Logs → CloudWatch alarms for suspicious activity (like root account usage).
What is AWS Config and what does it add beyond CloudTrail? ▼
CloudTrail records API calls (actions): "the security group was modified at 3pm." AWS Config records resource state: "here is what the security group's rules look like RIGHT NOW, and here is what they looked like every time they changed." Config continuously evaluates resources against compliance rules. If a rule says "no public RDS instances" and someone creates a public RDS → Config immediately marks it NON_COMPLIANT and can trigger automatic remediation (run an SSM automation document to fix it). Config also maintains a complete configuration history — you can see a timeline of every change to any resource and who caused it (via CloudTrail correlation). Use cases unique to Config: "Was this S3 bucket public at any point in the last 6 months?" → Config history answers this. "Alert me whenever any security group is modified." → Config evaluates on every change, CloudWatch Events fires.
What is WAF and when should you use it? What's the difference between Count and Block mode? ▼
AWS WAF (Web Application Firewall) is a Layer 7 (HTTP) firewall that sits in front of your web application (on CloudFront, ALB, or API Gateway). It inspects HTTP requests and blocks or allows them based on rules. Protects against: SQL injection, XSS (cross-site scripting), known bad bots, volumetric request attacks, requests from suspicious IPs/geos. Should use whenever: you have a public-facing web application or API, you're handling sensitive data, compliance requirements (PCI-DSS requires WAF), or you're a likely attack target. Count vs Block mode: Count mode logs matching requests and increments a CloudWatch metric but takes no action — traffic still reaches your backend. Use Count for 1-2 weeks when first deploying WAF to see which legitimate requests match the rules (false positives). Once confident the rules won't block legitimate traffic, switch to Block mode. Block mode rejects matching requests with 403 before they reach your Lambda or ECS service. Start in Count, transition to Block — prevents WAF from breaking your own application on day one.
How does GuardDuty work? What does it detect that CloudTrail alone can't? ▼
GuardDuty is a managed threat detection service that continuously analyses three data sources: CloudTrail API logs, VPC Flow Logs, and DNS logs. It uses machine learning baselines, anomaly detection, and AWS threat intelligence feeds to identify malicious behaviour. CloudTrail tells you what API calls were made — you'd have to manually write rules to detect "this looks suspicious." GuardDuty applies ML baselines and threat intelligence automatically. Examples of what GuardDuty detects that raw CloudTrail can't: (1) EC2 instance credentials used from an external IP — if the Instance Metadata IP credential is suddenly called from a non-EC2 IP, GuardDuty recognises this pattern (stolen credentials). (2) EC2 instance communicating with known malware C2 server — GuardDuty matches DNS queries against threat intelligence. You'd never write a CloudWatch rule for this. (3) Bitcoin mining — EC2 connecting to known mining pools. (4) Anomalous API call rate — if a role suddenly calls CreateRole 50x more than its baseline, GuardDuty flags it. Enable GuardDuty in all regions — it's essentially free to start and provides continuous threat detection with zero ongoing configuration.

Advanced

Design a secrets management strategy for a microservices application with 20 services, each needing database credentials, third-party API keys, and configuration. ▼
Strategy layered by secret type: (1) Database credentials (Secrets Manager): One secret per service per environment — prod/payments/db-creds, prod/orders/db-creds. 20 services × 3 envs = 60 secrets at $0.40/month = $24/month. Enable automatic rotation with RDS rotation Lambda on 30-day schedule. Each service's IAM role has GetSecretValue only on its own secrets (not other services). (2) Third-party API keys (Secrets Manager): prod/payments/stripe-key — typically shared fewer, maybe 5-10 secrets total. Rotation Lambda calls Stripe API to rotate keys on a schedule. (3) Configuration (SSM Parameter Store, free): /myapp/prod/service-urls, /myapp/prod/feature-flags — all non-sensitive config. Hierarchical structure allows GetParametersByPath to load all service config in one call. (4) Caching: Lambda: fetch outside handler, cache 5 minutes. ECS: ECS-injected as env vars at container startup, restart containers after rotation. (5) KMS: One CMK per environment (prod/staging/dev) — all secrets encrypted with it. Key policy restricts to specific roles. Annual rotation enabled. (6) Access control: Each service role has explicit Allow for only its own secrets. No wildcard ARNs. Config rule monitors for over-broad SecretManager permissions. Drift detection: CloudTrail alarm when any service accesses another service's secret.
Walk through how you'd investigate a suspected account compromise in AWS. ▼
(1) Immediate containment: If GuardDuty or Security Hub fired, identify the affected resource (EC2 instance, IAM role). For EC2: change security group to deny all inbound/outbound, preserve instance for forensics. For IAM: attach a deny-all policy to the user/role temporarily. (2) Identify the entry point (CloudTrail): Query CloudTrail for the time window around the alert. Filter by the affected principal. Look for: unusual API calls, calls from unexpected IPs, new user/role creations, policy changes, data access (S3 GetObject, Decrypt). CloudWatch Logs Insights: filter userIdentity.arn = "compromised-arn" | sort eventTime desc. (3) Blast radius assessment: What resources did the compromised identity have access to? Check IAM policies. What API calls were made? What data was accessed? Were any new IAM users/roles created? Were any data stores exfiltrated? (4) Eradicate: Rotate all credentials the compromised identity could access. Remove any backdoor IAM users/roles created. Patch the vulnerability that allowed initial compromise. (5) Recover: Restore from known-good backups if data was tampered. Deploy patched version from clean CI/CD pipeline. Re-enable access gradually with enhanced monitoring. (6) Lessons learned: Which Config rules/GuardDuty findings missed this? What monitoring gaps existed? Update incident runbooks.
