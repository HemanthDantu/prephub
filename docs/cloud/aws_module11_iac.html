Why IaC? The Problem it Solves

Infrastructure as Code (IaC) treats your cloud infrastructure — VPCs, Lambda functions, DynamoDB tables, IAM roles — as code that lives in Git, gets reviewed, tested, and deployed just like application code.

The Problem: Manual Infrastructure

You've been clicking through the AWS console to set things up. That works for one environment. Now consider:

Reproducibility
You spent 2 hours clicking through the console to set up your production environment. Now create an identical staging environment. Can you? Did you document every setting? What about the 47 small configuration decisions you made?
Drift
Developer Alice added a security group rule in production to fix an incident. Bob changed a Lambda memory setting. Carol created an S3 bucket. Six months later, prod, staging, and dev are completely different — "snowflake servers." No one knows what the true configuration is.
Disaster recovery
Production region has an outage. You need to spin up in a new region in 30 minutes. Without IaC, this could take days of re-clicking the console. With IaC: run a single command and infrastructure is recreated in minutes.
Manual infrastructure — what goes wrong
Environments differ from each other
No history of what changed when
No code review for infrastructure changes
Hard to recreate after disaster
"Works in prod, fails in staging"
Manual, error-prone, slow to deploy
No rollback mechanism
Knowledge lives in one person's head
Infrastructure as Code — what you get
Identical environments (same template)
Full Git history of every change
Infrastructure PRs with review
Recreate any environment in minutes
Staging = prod in code, tested before prod
Automated, consistent, fast deployment
Roll back by redeploying previous version
Knowledge lives in the repository

The Two AWS-Native IaC Tools

CloudFormation — Declarative YAML/JSON
You declare the desired end state in a YAML/JSON template. CloudFormation figures out how to create, update, or delete resources to reach that state. The most direct AWS IaC tool — has been around since 2011. Deep integration with every AWS service.
CDK — Code That Generates CloudFormation
Write infrastructure in Java, TypeScript, Python, or C#. CDK compiles your code to a CloudFormation template. Gives you loops, conditions, abstractions, and type safety that plain YAML can't provide. Deployed via CloudFormation under the hood.

CloudFormation Core Concepts

How CloudFormation works — templates, stacks, resources, and the deployment lifecycle.

The Key Concepts

Template
A YAML or JSON file that declares what AWS resources you want. Stored in Git and S3. The template is the source of truth — what you intend your infrastructure to look like.
Stack
A running instance of a template. When you deploy a template, CloudFormation creates a Stack containing all the declared resources. One template → many stacks (prod stack, staging stack, dev stack).
Resource
An AWS resource within a stack — a Lambda function, SQS queue, IAM role, S3 bucket, etc. Each resource has a logical ID (your name) and a physical ID (what AWS calls it).

How CloudFormation Deploys

1
You write/update a template and submit it
via CLI, console, or CI/CD pipeline. CloudFormation receives the template.
2
CloudFormation builds a dependency graph
Resources that depend on each other (Lambda needs IAM Role's ARN) are created in the right order. Independent resources are created in parallel.
3
Resources are created/updated/deleted
CloudFormation calls the AWS APIs on your behalf. For updates, it computes the diff between the current stack state and the new template — only changed resources are touched.
4
Stack reaches CREATE_COMPLETE or ROLLBACK
If any resource fails, CloudFormation automatically rolls back — deletes or reverts all resources created in this deployment. You never end up in a half-deployed state.

Stack States

StateMeaning
CREATE_IN_PROGRESSStack is being created — resources being provisioned
CREATE_COMPLETEAll resources successfully created
CREATE_FAILEDAt least one resource failed — stack rolled back
UPDATE_IN_PROGRESSStack is being updated
UPDATE_COMPLETEUpdate successful
UPDATE_ROLLBACK_COMPLETEUpdate failed, rolled back to previous state
DELETE_IN_PROGRESSStack being deleted — all resources being removed
ROLLBACK_COMPLETEInitial creation failed and was rolled back — stack is empty
# Deploy a CloudFormation template
aws cloudformation deploy \
  --stack-name prod-orders-service \
  --template-file infrastructure/orders-service.yaml \
  --capabilities CAPABILITY_NAMED_IAM \   # required if template creates IAM resources
  --parameter-overrides \
    Environment=prod \
    DatabasePassword=/prod/db/password \
  --tags \
    Project=orders \
    CostCentre=engineering

# Check stack status
aws cloudformation describe-stacks \
  --stack-name prod-orders-service \
  --query "Stacks[0].StackStatus"

# List all resources in the stack
aws cloudformation list-stack-resources \
  --stack-name prod-orders-service \
  --query "StackResourceSummaries[].{Type:ResourceType,LogicalId:LogicalResourceId,PhysicalId:PhysicalResourceId}"

# Delete the entire stack (removes ALL resources)
aws cloudformation delete-stack \
  --stack-name dev-orders-service
# Deleting a stack removes every resource it created — great for ephemeral dev envs
# Resources with DeletionPolicy: Retain are kept after stack deletion

Template Anatomy Deep Dive

A CloudFormation template is a YAML document with a defined structure. Understanding each section and writing real resource definitions is the core CloudFormation skill.

Template Top-Level Sections

---
AWSTemplateFormatVersion: "2010-09-09"   # always this value

Description: |
  Orders Service infrastructure.
  Lambda + SQS + DynamoDB for order processing.

Parameters:      # inputs — values passed at deploy time
  ...

Mappings:        # static lookup tables (e.g., region → AMI ID)
  ...

Conditions:      # boolean conditions used to conditionally create resources
  ...

Resources:       # REQUIRED — the AWS resources to create
  ...

Outputs:         # values to export for other stacks or display
  ...

A Complete Real-World Template

---
AWSTemplateFormatVersion: "2010-09-09"
Description: "Orders processing service — Lambda, SQS, DynamoDB"

Parameters:
  Environment:
    Type: String
    AllowedValues: [prod, staging, dev]
    Default: dev
  LambdaMemoryMB:
    Type: Number
    Default: 1024

Conditions:
  IsProduction: !Equals [!Ref Environment, prod]

Resources:

  # ── DynamoDB Table ──────────────────────────────────
  OrdersTable:
    Type: AWS::DynamoDB::Table
    DeletionPolicy: Retain   # CRITICAL: keep data if stack deleted
    UpdateReplacePolicy: Retain
    Properties:
      TableName: !Sub "${Environment}-orders"
      BillingMode: PAY_PER_REQUEST
      AttributeDefinitions:
        - AttributeName: orderId
          AttributeType: S
        - AttributeName: userId
          AttributeType: S
        - AttributeName: createdAt
          AttributeType: S
      KeySchema:
        - AttributeName: orderId
          KeyType: HASH
      GlobalSecondaryIndexes:
        - IndexName: userId-createdAt-index
          KeySchema:
            - AttributeName: userId
              KeyType: HASH
            - AttributeName: createdAt
              KeyType: RANGE
          Projection:
            ProjectionType: ALL
      PointInTimeRecoverySpecification:
        PointInTimeRecoveryEnabled: true
      SSESpecification:
        SSEEnabled: true
        SSEType: KMS
        KMSMasterKeyId: !Ref OrdersKMSKey
      Tags:
        - Key: Environment
          Value: !Ref Environment

  # ── SQS Queue + DLQ ────────────────────────────────
  OrdersDLQ:
    Type: AWS::SQS::Queue
    Properties:
      QueueName: !Sub "${Environment}-orders-dlq"
      MessageRetentionPeriod: 1209600   # 14 days
      KmsMasterKeyId: !Ref OrdersKMSKey

  OrdersQueue:
    Type: AWS::SQS::Queue
    Properties:
      QueueName: !Sub "${Environment}-orders-queue"
      VisibilityTimeout: 60
      MessageRetentionPeriod: 86400
      KmsMasterKeyId: !Ref OrdersKMSKey
      RedrivePolicy:
        deadLetterTargetArn: !GetAtt OrdersDLQ.Arn
        maxReceiveCount: 3

  # ── IAM Role for Lambda ─────────────────────────────
  OrdersLambdaRole:
    Type: AWS::IAM::Role
    Properties:
      RoleName: !Sub "${Environment}-orders-lambda-role"
      AssumeRolePolicyDocument:
        Version: "2012-10-17"
        Statement:
          - Effect: Allow
            Principal:
              Service: lambda.amazonaws.com
            Action: sts:AssumeRole
      ManagedPolicyArns:
        - arn:aws:iam::aws:policy/service-role/AWSLambdaSQSQueueExecutionRole
        - arn:aws:iam::aws:policy/AWSXRayDaemonWriteAccess
      Policies:
        - PolicyName: OrdersLambdaPolicy
          PolicyDocument:
            Version: "2012-10-17"
            Statement:
              - Effect: Allow
                Action:
                  - dynamodb:PutItem
                  - dynamodb:GetItem
                  - dynamodb:UpdateItem
                  - dynamodb:Query
                Resource:
                  - !GetAtt OrdersTable.Arn
                  - !Sub "${OrdersTable.Arn}/index/*"
              - Effect: Allow
                Action:
                  - kms:Decrypt
                  - kms:GenerateDataKey
                Resource: !GetAtt OrdersKMSKey.Arn
              - Effect: Allow
                Action: secretsmanager:GetSecretValue
                Resource: !Sub "arn:aws:secretsmanager:${AWS::Region}:${AWS::AccountId}:secret:${Environment}/orders/*"

  # ── Lambda Function ──────────────────────────────────
  OrdersLambda:
    Type: AWS::Lambda::Function
    Properties:
      FunctionName: !Sub "${Environment}-orders-processor"
      Runtime: java21
      Handler: com.acme.orders.OrderHandler::handleRequest
      Code:
        S3Bucket: !Sub "${Environment}-deployment-artifacts"
        S3Key: orders-service/latest/orders-processor.jar
      Role: !GetAtt OrdersLambdaRole.Arn
      MemorySize: !Ref LambdaMemoryMB
      Timeout: 60
      Environment:
        Variables:
          TABLE_NAME: !Ref OrdersTable
          ENVIRONMENT: !Ref Environment
      TracingConfig:
        Mode: Active
      ReservedConcurrentExecutions: !If [IsProduction, 100, 10]

  # ── SQS Event Source Mapping ─────────────────────────
  OrdersEventSourceMapping:
    Type: AWS::Lambda::EventSourceMapping
    Properties:
      FunctionName: !GetAtt OrdersLambda.Arn
      EventSourceArn: !GetAtt OrdersQueue.Arn
      BatchSize: 10
      MaximumBatchingWindowInSeconds: 5
      FunctionResponseTypes: [ReportBatchItemFailures]

  # ── Log Group with Retention ─────────────────────────
  OrdersLambdaLogGroup:
    Type: AWS::Logs::LogGroup
    Properties:
      LogGroupName: !Sub "/aws/lambda/${Environment}-orders-processor"
      RetentionInDays: !If [IsProduction, 90, 14]

  # ── KMS Key ──────────────────────────────────────────
  OrdersKMSKey:
    Type: AWS::KMS::Key
    Properties:
      Description: !Sub "${Environment} orders service encryption key"
      EnableKeyRotation: true
      KeyPolicy:
        Version: "2012-10-17"
        Statement:
          - Sid: EnableRootAccess
            Effect: Allow
            Principal:
              AWS: !Sub "arn:aws:iam::${AWS::AccountId}:root"
            Action: kms:*
            Resource: "*"

  OrdersKMSAlias:
    Type: AWS::KMS::Alias
    Properties:
      AliasName: !Sub "alias/${Environment}-orders-key"
      TargetKeyId: !Ref OrdersKMSKey

  # ── CloudWatch Alarm ─────────────────────────────────
  OrdersDLQAlarm:
    Type: AWS::CloudWatch::Alarm
    Condition: IsProduction   # only create in production
    Properties:
      AlarmName: prod-orders-dlq-not-empty
      Namespace: AWS/SQS
      MetricName: ApproximateNumberOfMessagesVisible
      Dimensions:
        - Name: QueueName
          Value: !GetAtt OrdersDLQ.QueueName
      Statistic: Sum
      Period: 60
      EvaluationPeriods: 1
      Threshold: 0
      ComparisonOperator: GreaterThanThreshold
      TreatMissingData: notBreaching
      AlarmActions:
        - !Sub "arn:aws:sns:${AWS::Region}:${AWS::AccountId}:prod-alerts"

Outputs:
  OrdersTableName:
    Description: DynamoDB table name for Orders
    Value: !Ref OrdersTable
    Export:
      Name: !Sub "${Environment}-orders-table-name"

  OrdersQueueUrl:
    Description: SQS queue URL
    Value: !Ref OrdersQueue
    Export:
      Name: !Sub "${Environment}-orders-queue-url"

  OrdersLambdaArn:
    Value: !GetAtt OrdersLambda.Arn

Parameters, Mappings & Conditions

The three mechanisms that make a single template reusable across environments — inputs, lookup tables, and conditional resource creation.

Parameters — Template Inputs

Parameters:
  Environment:
    Type: String
    AllowedValues: [prod, staging, dev]
    Default: dev
    Description: Deployment environment

  DBPassword:
    Type: String
    NoEcho: true   # value hidden in console, logs — for secrets
    MinLength: 8
    Description: Database master password

  SubnetIds:
    Type: List<AWS::EC2::Subnet::Id>   # validates these are real subnet IDs
    Description: List of private subnet IDs

  InstanceType:
    Type: String
    Default: t3.small
    AllowedValues: [t3.small, m7g.large, m7g.xlarge]

# Reference a parameter in the template
# !Ref returns the parameter value directly
MemorySize: !Ref LambdaMemoryMB

Mappings — Static Lookup Tables

Mappings:
  EnvironmentConfig:         # map name
    prod:                     # first key (e.g., environment)
      LambdaMemory: 1024
      LogRetentionDays: 90
      MinCapacity: 10
    staging:
      LambdaMemory: 512
      LogRetentionDays: 30
      MinCapacity: 2
    dev:
      LambdaMemory: 256
      LogRetentionDays: 7
      MinCapacity: 1

  RegionAMI:                  # different AMIs per region
    us-east-1:
      arm64: ami-0abcd1234567890
    eu-west-1:
      arm64: ami-0defg0987654321

# Use !FindInMap to look up values: [MapName, TopLevelKey, SecondLevelKey]
MemorySize: !FindInMap [EnvironmentConfig, !Ref Environment, LambdaMemory]
# If Environment=prod → 1024, if staging → 512, if dev → 256

ImageId: !FindInMap [RegionAMI, !Ref "AWS::Region", arm64]
# Looks up current region's AMI automatically

Conditions — Conditional Resource Creation

Conditions:
  IsProduction: !Equals [!Ref Environment, prod]
  IsNotDev: !Not [!Equals [!Ref Environment, dev]]
  EnableCaching: !Or [!Equals [!Ref Environment, prod], !Equals [!Ref Environment, staging]]
  EnableHA: !And [Condition: IsProduction, !Equals [!Ref EnableMultiAZ, "true"]]

# Use Condition: in a resource to make creation conditional
ElastiCacheCluster:
  Type: AWS::ElastiCache::ReplicationGroup
  Condition: EnableCaching    # only created in prod or staging
  Properties:
    ...

# !If in property values: [Condition, TrueValue, FalseValue]
MultiAZ: !If [IsProduction, true, false]
ReplicaCount: !If [IsProduction, 2, 0]
RetentionInDays: !If [IsProduction, 365, !If [IsNotDev, 30, 7]]
# Nested !If: production=365, staging=30, dev=7

Intrinsic Functions & Pseudo Parameters

CloudFormation's built-in functions let you reference other resources, build strings dynamically, look up values, and access context about the deployment. These are used in nearly every production template.

The Most Important Intrinsic Functions

## !Ref — reference a parameter or resource's primary identifier
TableName: !Ref OrdersTable         # returns the table name
Environment: !Ref Environment       # returns the parameter value

## !GetAtt — get an attribute of a resource (not just its primary ID)
RoleArn: !GetAtt OrdersLambdaRole.Arn
QueueArn: !GetAtt OrdersQueue.Arn
TableStreamArn: !GetAtt OrdersTable.StreamArn
# Every resource type documents what attributes are available

## !Sub — string substitution (most useful function)
FunctionName: !Sub "${Environment}-orders-processor"
# ${Environment} → value of the Environment parameter
# ${AWS::Region} → current region (pseudo parameter)
# ${AWS::AccountId} → account ID
# ${OrdersTable} → Ref of OrdersTable (table name)
# ${OrdersLambdaRole.Arn} → GetAtt of Arn attribute

## Complex !Sub example:
Resource: !Sub "arn:aws:dynamodb:${AWS::Region}:${AWS::AccountId}:table/${OrdersTable}/index/*"

## !Join — join a list of values with a delimiter
Value: !Join
  - ","
  - - !GetAtt Subnet1.SubnetId
    - !GetAtt Subnet2.SubnetId
    - !GetAtt Subnet3.SubnetId
# Returns: "subnet-aaa,subnet-bbb,subnet-ccc"

## !Select — select one item from a list
FirstSubnet: !Select [0, !Ref SubnetIds]
# Returns first element of the SubnetIds parameter list

## !Split — split a string into a list
SubnetList: !Split [",", !Ref SubnetsCommaSeparated]

## !ImportValue — import an exported output from another stack
VpcId: !ImportValue prod-vpc-id
# References the "prod-vpc-id" export from the VPC stack
# Cross-stack references — VPC in one stack, app in another

## !Base64 — encode a string in base64 (used for EC2 UserData)
UserData: !Base64 |
  #!/bin/bash
  yum update -y
  systemctl start myapp

Pseudo Parameters — Context About the Deployment

## These are automatically provided by CloudFormation — no declaration needed

${AWS::Region}         → us-east-1 (the region being deployed to)
${AWS::AccountId}      → 123456789012 (the AWS account number)
${AWS::StackName}      → prod-orders-service (name of this stack)
${AWS::StackId}        → full ARN of the stack
${AWS::NoValue}        → removes a property (used with !If for optional props)
${AWS::Partition}      → aws (or aws-cn, aws-us-gov for other partitions)
${AWS::URLSuffix}      → amazonaws.com

## Practical example — build an S3 bucket name that's globally unique and region-aware
BucketName: !Sub "${AWS::StackName}-artifacts-${AWS::AccountId}-${AWS::Region}"
# → "prod-orders-service-artifacts-123456789012-us-east-1"
# Globally unique, descriptive, no hardcoded values

Stacks, Change Sets & Drift Detection

Change sets let you preview exactly what CloudFormation will do before it does it — essential before updating production. Drift detection finds manual changes made outside CloudFormation.

Change Sets — Preview Before You Deploy

A change set shows you exactly what CloudFormation will create, modify, or delete when you apply a template update — before anything actually changes. This is the "terraform plan" equivalent for CloudFormation.

# Create a change set (doesn't execute — just plans)
aws cloudformation create-change-set \
  --stack-name prod-orders-service \
  --change-set-name update-lambda-memory \
  --template-body file://infrastructure/orders-service.yaml \
  --parameters ParameterKey=Environment,ParameterValue=prod \
               ParameterKey=LambdaMemoryMB,ParameterValue=2048 \
  --capabilities CAPABILITY_NAMED_IAM

# Wait for change set to be created
aws cloudformation wait change-set-create-complete \
  --stack-name prod-orders-service \
  --change-set-name update-lambda-memory

# Describe what the change set will do
aws cloudformation describe-change-set \
  --stack-name prod-orders-service \
  --change-set-name update-lambda-memory \
  --query "Changes[].ResourceChange.{Action:Action,Type:ResourceType,Id:LogicalResourceId,Replace:Replacement}"

# Example output:
# [
#   {"Action": "Modify", "Type": "AWS::Lambda::Function",
#    "Id": "OrdersLambda", "Replace": "False"}
# ]
# Shows: only the Lambda is being modified, not replaced, not deleted.
# Safe to proceed!

# Execute the change set (actually apply the changes)
aws cloudformation execute-change-set \
  --stack-name prod-orders-service \
  --change-set-name update-lambda-memory

# Delete if you decide not to apply
aws cloudformation delete-change-set \
  --stack-name prod-orders-service \
  --change-set-name update-lambda-memory

Replacement vs Modification — Critical Difference

Modify (Replacement: False)
The resource is updated in-place. A Lambda memory change goes from 1024 to 2048MB without downtime. An SQS queue message retention period changes. Safe for production.
Replace (Replacement: True) ⚠️
CloudFormation will delete the old resource and create a new one. A DynamoDB table name change? New table, no data. An RDS instance type change for certain DB engines? New database, data migration required. Always check change sets before deploying to production.

Drift Detection — Finding Manual Changes

# Drift: someone manually changed a resource outside CloudFormation
# e.g., Alice changed a security group in the console during an incident
# Now the live resource differs from what CloudFormation thinks it is

# Start drift detection on a stack
aws cloudformation detect-stack-drift \
  --stack-name prod-orders-service
# Returns: StackDriftDetectionId (asynchronous operation)

# Check drift status
aws cloudformation describe-stack-drift-detection-status \
  --stack-drift-detection-id abc123...
# Status: DETECTION_COMPLETE | DETECTION_IN_PROGRESS | DETECTION_FAILED

# View drifted resources
aws cloudformation describe-stack-resource-drifts \
  --stack-name prod-orders-service \
  --stack-resource-drift-status-filters MODIFIED DELETED

# Example output:
# Resource: OrdersLambdaRole (AWS::IAM::Role)
# DriftStatus: MODIFIED
# PropertyDifferences:
#   - PropertyPath: /Properties/Policies/0/PolicyDocument/Statement/0/Resource
#     ExpectedValue: "arn:aws:dynamodb:us-east-1:123:table/prod-orders"
#     ActualValue:   "arn:aws:dynamodb:us-east-1:123:table/prod-orders/*"
# Alice added /* to the resource ARN during the incident — it's different now

# Resolution: either update the template to match reality, or redeploy to revert

Stack Policies — Protect Production Resources

# Stack policy prevents accidental modification or deletion of specific resources
# Even CloudFormation can't modify these unless the policy is explicitly updated

aws cloudformation set-stack-policy \
  --stack-name prod-orders-service \
  --stack-policy-body '{
    "Statement": [
      {
        "Effect": "Deny",
        "Principal": "*",
        "Action": ["Update:Replace", "Update:Delete"],
        "Resource": "LogicalResourceId/OrdersTable"
      },
      {
        "Effect": "Allow",
        "Principal": "*",
        "Action": "Update:*",
        "Resource": "*"
      }
    ]
  }'
# Now nobody can accidentally delete or replace the DynamoDB table via CloudFormation
# (They'd need to first remove the stack policy)

Nested Stacks & StackSets

Nested stacks let you break large templates into reusable modules. StackSets deploy the same stack across multiple accounts and regions simultaneously.

Nested Stacks — Template Modules

A single CloudFormation template has a 1MB limit and can become unwieldy for complex systems. Nested stacks let you define reusable infrastructure modules (a VPC module, a monitoring module, a database module) and compose them into a root stack.

# Root stack — composes modules
Resources:
  VPCStack:
    Type: AWS::CloudFormation::Stack
    Properties:
      TemplateURL: !Sub "https://s3.amazonaws.com/${TemplatesBucket}/vpc.yaml"
      Parameters:
        Environment: !Ref Environment
        CidrBlock: "10.0.0.0/16"

  DatabaseStack:
    Type: AWS::CloudFormation::Stack
    DependsOn: VPCStack       # create VPC first
    Properties:
      TemplateURL: !Sub "https://s3.amazonaws.com/${TemplatesBucket}/database.yaml"
      Parameters:
        VpcId: !GetAtt VPCStack.Outputs.VpcId
        SubnetIds: !GetAtt VPCStack.Outputs.PrivateSubnetIds
        Environment: !Ref Environment

  AppStack:
    Type: AWS::CloudFormation::Stack
    DependsOn: DatabaseStack
    Properties:
      TemplateURL: !Sub "https://s3.amazonaws.com/${TemplatesBucket}/app.yaml"
      Parameters:
        TableName: !GetAtt DatabaseStack.Outputs.TableName
        Environment: !Ref Environment

# Benefits:
# - Each template stays focused and manageable
# - Reuse vpc.yaml across multiple applications
# - Teams own their templates independently
# - Deploy/update individual components without affecting others

StackSets — Multi-Account, Multi-Region Deployment

# StackSets deploys a template to many accounts/regions at once
# Useful for: security baselines, compliance policies, shared services

# Deploy CloudTrail to all accounts in your AWS Organization
aws cloudformation create-stack-set \
  --stack-set-name organization-cloudtrail \
  --template-url https://s3.amazonaws.com/templates/cloudtrail.yaml \
  --permission-model SERVICE_MANAGED \   # integrates with AWS Organizations
  --auto-deployment Enabled=true,RetainStacksOnAccountRemoval=false

# Deploy to all accounts in specific OUs (Organizational Units)
aws cloudformation create-stack-instances \
  --stack-set-name organization-cloudtrail \
  --deployment-targets OrganizationalUnitIds=ou-root-abc123 \
  --regions us-east-1 eu-west-1 ap-southeast-1 \
  --operation-preferences \
    MaxConcurrentPercentage=30 \      # deploy 30% of accounts simultaneously
    FailureTolerancePercentage=20     # tolerate 20% failure rate

# Common StackSet use cases:
# - CloudTrail trail in every account and region
# - Default VPC deletion (security best practice)
# - Config recorder and rules in every account
# - IAM roles for cross-account access
# - Security baselines (GuardDuty, Security Hub enable)

What is CDK and Why it Exists

AWS Cloud Development Kit (CDK) lets you write infrastructure in your programming language. Instead of YAML, you write Java/TypeScript/Python code that compiles to CloudFormation templates.

The YAML Problem CDK Solves

CloudFormation YAML is verbose and repetitive. Need 5 Lambda functions with identical logging, monitoring, and IAM patterns? You write the same 50-line pattern 5 times. Change one thing? Update all 5. There's no loop, no variable reuse, no abstraction. CDK brings programming language power to infrastructure.

CloudFormation YAML — 80 lines for one Lambda
OrdersLambdaRole:
  Type: AWS::IAM::Role
  Properties:
    RoleName: orders-lambda-role
    AssumeRolePolicyDocument:
      ...20 lines...
    ManagedPolicyArns:
      - arn:aws:iam::aws:policy/...
    Policies:
      - PolicyName: OrdersPolicy
        PolicyDocument:
          ...15 lines...
OrdersLambda:
  Type: AWS::Lambda::Function
  Properties:
    FunctionName: orders
    Runtime: java21
    Handler: ...
    Code: ...
    Role: !GetAtt OrdersLambdaRole.Arn
    ...10 more props...
OrdersLogGroup:
  Type: AWS::Logs::LogGroup
  Properties:
    LogGroupName: /aws/lambda/orders
    RetentionInDays: 90
...repeat for every function...
CDK Java — 10 lines, reusable abstraction
// CDK L2 construct handles Role,
// LogGroup, and permissions automatically

Function.Builder.create(this, "Orders")
  .functionName("orders")
  .runtime(Runtime.JAVA_21)
  .handler("com.acme.OrderHandler")
  .code(Code.fromAsset("target/app.jar"))
  .memorySize(1024)
  .timeout(Duration.seconds(60))
  .tracing(Tracing.ACTIVE)
  .build();

// That's it. IAM role, log group, 
// permissions — all created automatically.
// Add 4 more functions = 4 more lines.

How CDK Works

1
You write CDK code in Java (or TypeScript, Python, C#)
Using the AWS CDK library — Java classes that represent AWS resources. You write your infrastructure logic like application code: loops, conditions, methods, classes.
2
cdk synth compiles your code to CloudFormation
cdk synth runs your code and outputs a CloudFormation template. You can inspect this template — it's what actually gets deployed. CDK is a CloudFormation generator.
3
cdk deploy uploads template and deploys stack
CDK uploads the generated CloudFormation template (and Lambda JARs, etc.) to an S3 bucket, then calls CloudFormation to deploy it. It's CloudFormation doing the actual deployment — CDK is just a better way to write the template.
4
cdk diff shows what will change
Before deploying, cdk diff shows the diff between current deployed state and what your code would produce. Like a CloudFormation change set, but readable.

CDK Constructs — L1, L2, L3

The building blocks of CDK apps. Three abstraction levels — raw CloudFormation, best-practice resources, and complete patterns. Choosing the right level is the key CDK skill.

Three Levels of Constructs

L1 (Level 1) — CloudFormation Resources
Direct mapping to CloudFormation resources. Class names start with Cfn — CfnFunction, CfnQueue, CfnTable. Every CloudFormation property is available. No defaults, no opinions. Full control, maximum verbosity.

Use when: you need a resource property that the L2 construct doesn't expose, or you're migrating an existing CloudFormation template to CDK.
L2 (Level 2) — Curated Constructs
The default CDK constructs. Provide sensible defaults (CloudWatch alarms, IAM roles, encryption) and convenience methods. Most infrastructure is written at L2.

Examples: Function, Queue, Table, Bucket, RestApi. They wrap multiple CloudFormation resources — a CDK Function creates the Lambda + IAM Role + Log Group automatically.
L3 (Level 3) — Patterns
Complete patterns combining multiple L2 constructs. Example: ApplicationLoadBalancedFargateService creates an ECS cluster, Fargate service, ALB, target group, security groups, and IAM roles in one construct.

Use for rapid prototyping. May not have enough flexibility for complex production setups.

L1 vs L2 — Side by Side

import software.amazon.awscdk.services.sqs.*;
import software.amazon.awscdk.services.sqs.CfnQueue;

// ── L1: Direct CloudFormation mapping ─────────────────
// Verbose, every property manually set
var dlq = CfnQueue.Builder.create(this, "L1DLQ")
    .queueName("prod-dlq")
    .messageRetentionPeriod(1209600)
    .build();

var queue = CfnQueue.Builder.create(this, "L1Queue")
    .queueName("prod-queue")
    .visibilityTimeout(60)
    .redrivePolicy(CfnQueue.RedrivePolicy.builder()
        .deadLetterTargetArn(dlq.getAttrArn())
        .maxReceiveCount(3)
        .build())
    .build();

// ── L2: Curated, intelligent defaults ─────────────────
// Cleaner, type-safe, convenience methods
var dlqL2 = Queue.Builder.create(this, "DLQ")
    .queueName("prod-dlq")
    .retentionPeriod(Duration.days(14))   // Duration type, not raw seconds
    .encryption(QueueEncryption.KMS_MANAGED)
    .build();

var queueL2 = Queue.Builder.create(this, "Queue")
    .queueName("prod-queue")
    .visibilityTimeout(Duration.seconds(60))
    .deadLetterQueue(DeadLetterQueue.builder()
        .queue(dlqL2)
        .maxReceiveCount(3)
        .build())
    .encryption(QueueEncryption.KMS_MANAGED)
    .build();

// L2 adds CloudWatch alarm for DLQ automatically if you use addAlarm()
// L2 generates IAM grants with type-safe methods:
queueL2.grantSendMessages(myLambdaFunction);   // adds minimal IAM policy

Custom L3 Construct — Reusable Patterns

// Define your team's standard Lambda+SQS pattern once
public class StandardQueuedLambda extends Construct {

    private final Function function;
    private final Queue queue;
    private final Queue dlq;

    public StandardQueuedLambda(Construct scope, String id, Props props) {
        super(scope, id);

        // Every team Lambda gets: DLQ, Queue with DLQ, Lambda, ESM,
        // CloudWatch alarm on DLQ, X-Ray tracing, structured logging env vars

        this.dlq = Queue.Builder.create(this, "DLQ")
            .retentionPeriod(Duration.days(14)).build();

        this.queue = Queue.Builder.create(this, "Queue")
            .visibilityTimeout(Duration.seconds(props.getTimeoutSeconds() * 6))
            .deadLetterQueue(DeadLetterQueue.builder()
                .queue(this.dlq).maxReceiveCount(3).build())
            .build();

        this.function = Function.Builder.create(this, "Function")
            .handler(props.getHandler())
            .code(props.getCode())
            .runtime(Runtime.JAVA_21)
            .memorySize(props.getMemoryMB())
            .timeout(Duration.seconds(props.getTimeoutSeconds()))
            .tracing(Tracing.ACTIVE)
            .environment(Map.of("SERVICE_NAME", id))
            .build();

        this.function.addEventSource(SqsEventSource.Builder.create(this.queue)
            .batchSize(10)
            .reportBatchItemFailures(true)
            .build());

        // DLQ alarm — automatically created for every instance of this construct
        new Alarm(this, "DLQAlarm", AlarmProps.builder()
            .metric(this.dlq.metricApproximateNumberOfMessagesVisible())
            .threshold(0)
            .evaluationPeriods(1)
            .comparisonOperator(ComparisonOperator.GREATER_THAN_THRESHOLD)
            .alarmName(id + "-dlq-has-messages")
            .build());
    }
}

// Usage: one line creates Lambda + Queue + DLQ + ESM + Alarm
new StandardQueuedLambda(this, "OrderProcessor", Props.builder()
    .handler("com.acme.OrderHandler")
    .code(Code.fromAsset("target/orders.jar"))
    .memoryMB(1024).timeoutSeconds(60).build());

// OrdersServiceStack.java — defines the stack
public class OrdersServiceStack extends Stack {

    public OrdersServiceStack(Construct scope, String id,
                               StackProps props, String env) {
        super(scope, id, props);

        boolean isProd = "prod".equals(env);

        // ── DynamoDB ───────────────────────────────────
        Table ordersTable = Table.Builder.create(this, "OrdersTable")
            .tableName(env + "-orders")
            .partitionKey(Attribute.builder()
                .name("orderId").type(AttributeType.STRING).build())
            .billingMode(BillingMode.PAY_PER_REQUEST)
            .pointInTimeRecovery(true)
            .encryption(TableEncryption.AWS_MANAGED)
            .removalPolicy(isProd ? RemovalPolicy.RETAIN : RemovalPolicy.DESTROY)
            .build();

        // Add GSI
        ordersTable.addGlobalSecondaryIndex(GlobalSecondaryIndexProps.builder()
            .indexName("userId-createdAt-index")
            .partitionKey(Attribute.builder().name("userId").type(AttributeType.STRING).build())
            .sortKey(Attribute.builder().name("createdAt").type(AttributeType.STRING).build())
            .build());

        // ── Lambda ─────────────────────────────────────
        Function ordersLambda = Function.Builder.create(this, "OrdersLambda")
            .functionName(env + "-orders-processor")
            .runtime(Runtime.JAVA_21)
            .handler("com.acme.orders.OrderHandler::handleRequest")
            .code(Code.fromAsset("target/orders-1.0.0.jar"))
            .memorySize(isProd ? 1024 : 512)
            .timeout(Duration.seconds(60))
            .tracing(Tracing.ACTIVE)
            .environment(Map.of(
                "TABLE_NAME", ordersTable.getTableName(),
                "ENVIRONMENT", env
            ))
            .build();

        // Grant Lambda read/write access to DynamoDB (generates minimal IAM policy)
        ordersTable.grantReadWriteData(ordersLambda);

        // ── SQS Queue ──────────────────────────────────
        Queue dlq = Queue.Builder.create(this, "DLQ")
            .queueName(env + "-orders-dlq")
            .retentionPeriod(Duration.days(14))
            .build();

        Queue queue = Queue.Builder.create(this, "Queue")
            .queueName(env + "-orders-queue")
            .visibilityTimeout(Duration.seconds(360))
            .deadLetterQueue(DeadLetterQueue.builder()
                .queue(dlq).maxReceiveCount(3).build())
            .build();

        ordersLambda.addEventSource(SqsEventSource.Builder.create(queue)
            .batchSize(10).reportBatchItemFailures(true).build());

        // ── Outputs ────────────────────────────────────
        new CfnOutput(this, "TableName", CfnOutputProps.builder()
            .value(ordersTable.getTableName())
            .exportName(env + "-orders-table-name").build());
    }
}

SituationUse
Java backend team managing their own infraCDK (Java) — same language as the app, IDE support

IaC in CI/CD Pipelines

Infrastructure changes should go through the same review, testing, and deployment pipeline as application code. This section shows how CloudFormation and CDK fit into a real CI/CD workflow.

The Principle

Infrastructure code lives in Git. Every change goes through: code review (PR) → automated linting/validation → deploy to staging → manual/automated validation → deploy to production. No one should be making manual changes to production infrastructure.

GitHub Actions Pipeline — Deploy CloudFormation on Push

# .github/workflows/infrastructure.yml
name: Deploy Infrastructure

on:
  push:
    branches: [main]
    paths: ['infrastructure/**']  # only run when infra files change
  pull_request:
    paths: ['infrastructure/**']

permissions:
  id-token: write   # for OIDC → IAM role (no long-term credentials!)
  contents: read

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS credentials via OIDC
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/GithubActionsInfraRole
          aws-region: us-east-1

      - name: Lint CloudFormation templates
        run: |
          pip install cfn-lint
          cfn-lint infrastructure/*.yaml

      - name: Validate templates
        run: |
          for template in infrastructure/*.yaml; do
            aws cloudformation validate-template --template-body file://$template
          done

      - name: Create change set for staging
        if: github.event_name == 'pull_request'
        run: |
          aws cloudformation create-change-set \
            --stack-name staging-orders-service \
            --template-body file://infrastructure/orders-service.yaml \
            --parameters ParameterKey=Environment,ParameterValue=staging \
            --change-set-name pr-${{ github.event.number }}-review \
            --capabilities CAPABILITY_NAMED_IAM

      - name: Show change set
        if: github.event_name == 'pull_request'
        run: |
          aws cloudformation describe-change-set \
            --stack-name staging-orders-service \
            --change-set-name pr-${{ github.event.number }}-review \
            --query "Changes[].ResourceChange" > change-set.json
          cat change-set.json
          # Post change set to PR as a comment

  deploy-staging:
    needs: validate
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment: staging    # requires GitHub environment approval
    steps:
      - uses: actions/checkout@v4
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123:role/GithubActionsInfraRole
          aws-region: us-east-1

      - name: Deploy to staging
        run: |
          aws cloudformation deploy \
            --stack-name staging-orders-service \
            --template-file infrastructure/orders-service.yaml \
            --parameter-overrides Environment=staging \
            --capabilities CAPABILITY_NAMED_IAM \
            --no-fail-on-empty-changeset

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment: production   # requires human approval in GitHub
    steps:
      - uses: actions/checkout@v4
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123:role/GithubActionsInfraRole
          aws-region: us-east-1

      - name: Deploy to production
        run: |
          aws cloudformation deploy \
            --stack-name prod-orders-service \
            --template-file infrastructure/orders-service.yaml \
            --parameter-overrides Environment=prod \
            --capabilities CAPABILITY_NAMED_IAM

SAM — Serverless Application Model

SAM is a CloudFormation extension specifically for serverless applications. It adds shorthand resource types for Lambda, API Gateway, DynamoDB, and SQS, and includes local testing tools.

What SAM Is

SAM templates look like CloudFormation but with three special resource types that expand to multiple CloudFormation resources automatically: AWS::Serverless::Function, AWS::Serverless::Api, and AWS::Serverless::SimpleTable. SAM transforms these into full CloudFormation before deployment.

# template.yaml — SAM template (much more concise than raw CloudFormation)
Transform: AWS::Serverless-2016-10-31

Globals:   # apply to ALL functions in template
  Function:
    Runtime: java21
    MemorySize: 1024
    Timeout: 30
    Tracing: Active
    Environment:
      Variables:
        TABLE_NAME: !Ref OrdersTable

Resources:

  GetOrderFunction:
    Type: AWS::Serverless::Function   # shorthand!
    Properties:
      Handler: com.acme.GetOrderHandler::handleRequest
      CodeUri: target/orders.jar
      Events:
        GetOrder:
          Type: Api
          Properties:
            Path: /orders/{orderId}
            Method: get
      Policies:
        - DynamoDBReadPolicy:      # SAM policy template — expands to full IAM
            TableName: !Ref OrdersTable

  CreateOrderFunction:
    Type: AWS::Serverless::Function
    Properties:
      Handler: com.acme.CreateOrderHandler::handleRequest
      CodeUri: target/orders.jar
      Events:
        CreateOrder:
          Type: Api
          Properties:
            Path: /orders
            Method: post
        SQSTrigger:
          Type: SQS
          Properties:
            Queue: !GetAtt OrdersQueue.Arn
            BatchSize: 10
      Policies:
        - DynamoDBCrudPolicy:
            TableName: !Ref OrdersTable
        - SQSPollerPolicy:
            QueueName: !GetAtt OrdersQueue.QueueName

  OrdersTable:
    Type: AWS::Serverless::SimpleTable   # simple single-key DynamoDB table
    Properties:
      PrimaryKey:
        Name: orderId
        Type: String

  OrdersQueue:
    Type: AWS::SQS::Queue    # standard CloudFormation resource (SAM is a superset)

IaC Best Practices

Patterns and habits that separate infrastructure that causes incidents from infrastructure that you can change confidently.

Template Organisation

✓ One stack per bounded context (orders-service, users-service, shared-infra)
  Not one giant stack for everything — changes to one service shouldn't
  require deploying all infrastructure

✓ Separate long-lived infrastructure from short-lived
  Foundation stack: VPC, shared ALB, Route53 zones → changes rarely
  Service stack: Lambda, DynamoDB, SQS → changes frequently
  Foundation is a dependency of service — don't mix them

✓ Use parameters for environment differences, not separate templates
  Bad:  orders-service-prod.yaml, orders-service-dev.yaml (drift risk)
  Good: orders-service.yaml with Environment parameter

✓ Name resources explicitly (not CloudFormation auto-names)
  Bad:  (no name → OrdersTable gets random name like "orders-abc123")
  Good: TableName: !Sub "${Environment}-orders"
  Why:  Auto-named resources get new names on replacement → downstream breakage

Safety Practices

✓ DeletionPolicy: Retain on stateful resources (databases, S3 buckets)
  Prevents accidental data loss when deleting stacks
  DeletionPolicy: Retain
  UpdateReplacePolicy: Retain  ← also set this to prevent replacement deleting data

✓ Stack policies to protect production tables
  "nobody can Replace or Delete OrdersTable" policy on prod stack

✓ Termination protection on production stacks
  aws cloudformation update-termination-protection \
    --stack-name prod-orders-service \
    --enable-termination-protection
  Now 'delete-stack' requires disabling this first — prevents accidents

✓ Use change sets in production pipelines — never raw deploy
  Create change set → review → require explicit approval → execute
  Catches Replacement=True changes before they cause data loss

✓ Validate templates before deploy
  cfn-lint: checks syntax, types, and known anti-patterns
  aws cloudformation validate-template: catches structural errors

Secrets and Sensitive Values

✓ Never hardcode secrets in templates
  Bad:  DBPassword: "xK9#mN2$" (in Git forever)
  Good: DBPassword: "{{resolve:secretsmanager:prod/db:SecretString:password}}"
  
  CloudFormation dynamic references resolve secrets at deploy time:
  {{resolve:secretsmanager:SECRET_ID:SecretString:JSON_KEY}}
  {{resolve:ssm-secure:PARAMETER_NAME}}

✓ Use NoEcho: true for sensitive parameters
  DBPassword:
    Type: String
    NoEcho: true  ← value hidden in console and CloudFormation events

Tagging Strategy

✓ Tag every resource for cost allocation and ownership
  Apply tags at the stack level — propagated to all resources:

  aws cloudformation deploy \
    --tags \
      Environment=prod \
      CostCentre=orders-team \
      Project=e-commerce \
      ManagedBy=cloudformation \
      Owner=backend-team

✓ Use AWS Config rule: required-tags
  Alarms if any resource lacks required tags
  Enforces tagging as a compliance requirement

Interview Questions — Infrastructure as Code

12 questions on CloudFormation, CDK, and IaC patterns.

Beginner

What is Infrastructure as Code and what problems does it solve? ▼
Infrastructure as Code (IaC) means defining your cloud infrastructure — servers, networks, databases, IAM roles — in machine-readable files that live in version control. Problems it solves: (1) Reproducibility: Define your environment once, recreate it identically for dev, staging, prod, disaster recovery. No "I think I configured it this way" uncertainty. (2) Drift prevention: CloudFormation is the source of truth. Detect and resolve manual changes (drift) automatically. (3) Version control: Every infrastructure change is a commit with a message, author, timestamp, and review. "When did this security group rule get added?" — git blame answers it. (4) Code review: Infrastructure changes go through PR review, just like application code. A second person validates that a DynamoDB table deletion doesn't destroy production data. (5) Rollback: Deploy previous version of the template to roll back. (6) Speed: Creating a new staging environment is running one command, not clicking through the console for hours.
What is the difference between a CloudFormation template and a stack? ▼
A template is the YAML or JSON file that declares what infrastructure you want — the recipe. It's stored in Git and S3. It defines resource types, configurations, parameters, and outputs. It has no direct relationship to any deployed infrastructure until you use it to create a stack. A stack is a running instance of a deployed template. When you deploy a template to CloudFormation, it creates a stack — which is the actual set of AWS resources (Lambda functions, DynamoDB tables, IAM roles) that CloudFormation created and manages. Relationship: one template → many stacks. Deploy the same template with different parameter values (Environment=prod, Environment=staging) → two stacks with separate but identically-configured resources. Delete the stack → CloudFormation deletes all the resources it created. The template remains unchanged in Git/S3.
What is a CloudFormation change set and why should you use it? ▼
A change set is a preview of what CloudFormation will do when you apply a template update — without actually doing it. You create the change set, review what resources will be created, modified, or deleted, and then either execute it or discard it. Use it because: (1) It shows the Replacement field — whether a change will delete and recreate a resource vs update in-place. A DynamoDB table name change shows Replacement=True — that would delete your data! A Lambda memory change shows Replacement=False — safe in-place update. (2) It documents every change going into production as a reviewable artifact. (3) It catches mistakes before they happen — you see "DeleteBucket" in the change set and realise you accidentally removed an S3 resource from the template. In production pipelines: always use change sets. Create → post as PR comment for review → require human approval → execute. Never deploy directly to production without reviewing a change set.

Intermediate

What is DeletionPolicy in CloudFormation and when do you use it? ▼
DeletionPolicy controls what happens to a resource when its containing CloudFormation stack is deleted or when the resource is removed from the template. Options: Delete (default): the resource is deleted. For DynamoDB tables, this means all data is gone. Retain: the resource is kept after stack deletion. The data is preserved. The resource becomes unmanaged — CloudFormation no longer tracks it. Snapshot (for RDS, ElastiCache, Redshift): takes a final snapshot before deleting. Always set DeletionPolicy: Retain (and UpdateReplacePolicy: Retain) on stateful resources in production: DynamoDB tables, S3 buckets, RDS databases. The UpdateReplacePolicy is also important — it prevents data loss when CloudFormation needs to replace a resource (e.g., changing the table name). Without it, CloudFormation would delete the old table before creating the new one. With Retain: old table kept, new table created — you migrate data manually and then clean up.
What is CDK and how does it relate to CloudFormation? ▼
AWS CDK (Cloud Development Kit) is a framework that lets you write infrastructure in a programming language (Java, TypeScript, Python, C#) instead of YAML. CDK code is compiled to a CloudFormation template by running cdk synth. The CloudFormation template is then deployed by cdk deploy — using CloudFormation as the deployment engine. So CDK is a CloudFormation template generator. Under the hood, everything deploys the same way — through CloudFormation stacks, with all the same rollback, drift detection, and change set features. CDK adds: (1) Abstractions (L2 constructs handle IAM roles, log groups automatically). (2) Loops and conditions (10 Lambda functions = a loop, not 10 copy-pasted resource blocks). (3) Type safety (compiler catches missing required properties before deploy). (4) Reusable patterns (define a custom L3 construct for your team's standard Lambda configuration, share as a library). CDK generates verbose CloudFormation — use cdk synth to see what it would deploy.
What is CloudFormation drift detection? When would you use it? ▼
Drift detection compares the actual configuration of your AWS resources against what CloudFormation believes they should be based on the last deployed template. Drift occurs when someone modifies a resource outside CloudFormation — changing a security group in the console, modifying an IAM policy via CLI, adjusting Lambda memory from the AWS console. Use drift detection: (1) After an incident where someone made a manual change to fix a production problem — to document what changed and ensure it gets codified in the template. (2) Periodic compliance audits — "verify that no one has manually changed production infrastructure." (3) Before deploying a template update — if there's drift, the update might conflict with or revert the manual change. Running it: aws cloudformation detect-stack-drift → describe-stack-resource-drifts. Resolution options: update the template to match reality (formalise the manual change), or redeploy the template to revert to the declared state.

Describe the IaC pipeline you'd implement for a team with 5 microservices, each needing dev/staging/prod environments. ▼
Repository structure: monorepo with infrastructure/ directory containing one template or CDK stack per service. Pipeline: (1) PR opened: GitHub Actions runs cfn-lint and aws cloudformation validate-template. Creates change sets for both staging and prod stacks. Posts change sets as PR comments — reviewers see exactly what will change. Requires review approval before merge. (2) Merge to main: Automatically deploy to dev (no approval needed — developers should iterate fast here). (3) Deploy to staging: Triggered automatically after dev deploy passes smoke tests. Creates change set, requires the bot to post it to Slack for visibility. Auto-execute after 10 minutes if no objections. (4) Deploy to production: Requires explicit human approval in GitHub Environments. Change set posted for final review. Execute only after approval. Rollback: previous commit re-run via pipeline. Secrets: all credentials via OIDC (no static keys). Templates: use parameters (Environment=dev/staging/prod) so one template serves all envs. DeletionPolicy: Retain on all tables, termination protection on prod stacks. Tagging: all resources tagged with Environment, Service, ManagedBy=cloudformation, Owner.
What are the tradeoffs between using CloudFormation directly vs CDK for a Java backend team? ▼
For a Java team, CDK has significant advantages: Java IDE support (autocomplete, type checking, refactoring), familiar language, write infrastructure alongside application code in the same repo and language. Loops eliminate copy-paste repetition — 5 Lambda functions with the same pattern = a loop, not 5 identical YAML blocks. Custom L3 constructs enforce team standards: "every Lambda must have X-Ray active, structured logging, and a DLQ alarm" — build it once into a construct, use everywhere, and compliance is structural not optional. Testing: test CDK constructs with CDK assertions (fine.Template.fromStack().hasResourceProperties()). CDK disadvantages: Higher toolchain complexity — requires Node.js (CDK CLI is TypeScript even for Java), Maven CDK dependencies, cdk bootstrap per account/region. Generated CloudFormation is verbose and hard to read — debugging CDK deployments sometimes requires inspecting the synth output. Learning curve — CDK API is large and documentation requires understanding both CDK and CloudFormation. Verdict: For a Java team building and maintaining multiple serverless services long-term, CDK's abstraction benefits outweigh the setup cost. For simple one-off infrastructure or compliance templates, CloudFormation YAML is fine and simpler.
