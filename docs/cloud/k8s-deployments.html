Section 1 of 5 · ~6 mins

Deployment Core Mechanics

Deployments manage ReplicaSets which in turn manage Pods. You define desired state and Kubernetes reconciles it.

🧒 ELI5

Robot manager

You tell the robot how many toys you want, and it keeps fixing the shelf if any fall.

🔗 ANALOGY

Factory line

A supervisor ensures the correct number of workers are always on the line.

Technical

What it means in real code

Definition: A Deployment is a controller that ensures a desired number of Pod replicas through ReplicaSets.

Why it matters

  • Declarative updates
  • Automated rollbacks
  • Self-healing pods

Common pitfall

  • Imperative kubectl apply without review
  • No labels/selector hygiene
  • Replica drift
DeploymentReplicaSetPod
Define spec→ReplicaSet created→Pods scheduled→Controller reconciles

Interview answer

“A Deployment defines desired state and manages ReplicaSets to keep Pods healthy and updated.”

ResourcePurpose
DeploymentDesired state
ReplicaSetReplica manager
Basic Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-service
spec:
  replicas: 3
  selector:
    matchLabels:
      app: api
  template:
    metadata:
      labels:
        app: api
    spec:
      containers:
        - name: api
          image: myrepo/api:1.0.0
          ports:
            - containerPort: 8080
Warning

Selector mismatches

If selectors do not match template labels, rollouts break or orphan Pods.

Mini takeaway
  • Deployments drive ReplicaSets.
  • Label consistency matters.
  • Use declarative updates.
Quiz

What does a Deployment directly manage?

Correct answer: B Deployments create and manage ReplicaSets, which create Pods.
Section 2 of 5 · ~6 mins

Rolling Updates & Rollbacks

Rollouts replace old Pods with new ones while maintaining availability through surge and unavailable limits.

🧒 ELI5

Change tires while driving

Replace one tire at a time so the car keeps moving.

🔗 ANALOGY

Shift handoff

New workers come in before the old shift leaves.

Technical

What it means in real code

Definition: Rolling updates use maxSurge and maxUnavailable to control rollout pace and availability.

Why it matters

  • Zero/low downtime deploys
  • Controlled risk
  • Easy rollback

Common pitfall

  • maxUnavailable too high
  • No readiness checks
  • No monitoring during rollout
maxSurgemaxUnavailablerollback
Create new RS→Scale up new Pods→Scale down old Pods→Complete rollout

Interview answer

“Rolling updates balance availability and speed using maxSurge/maxUnavailable; rollbacks revert to previous ReplicaSet.”

SettingEffect
maxSurgeExtra pods allowed
maxUnavailablePods allowed down
Rolling update strategy
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 25%
    maxUnavailable: 25%
Warning

No rollback without history

If you prune ReplicaSet history, rollback targets disappear.

Mini takeaway
  • Tune maxSurge/maxUnavailable.
  • Monitor readiness.
  • Keep rollout history.
Quiz

What controls how many extra Pods can be created during a rollout?

Correct answer: B maxSurge specifies extra Pods above desired replicas.
Section 3 of 5 · ~7 mins

Autoscaling with HPA

Horizontal Pod Autoscaler adjusts replica count based on metrics and resource requests.

🧒 ELI5

More checkout lanes

Open more lanes when the line gets long.

🔗 ANALOGY

Elastic team

You add contractors when demand spikes.

Technical

What it means in real code

Definition: HPA scales replicas based on metrics (CPU, memory, or custom) relative to requested resources.

Why it matters

  • Keeps latency stable
  • Saves cost
  • Responds to demand

Common pitfall

  • No requests/limits
  • Wrong metrics
  • Thundering herd scaling
HPAmetricsrequests
Collect metrics→Compare to target→Adjust replicas→Stabilize

Interview answer

“HPA scales Pods based on metrics, relying on resource requests as baseline.”

MetricUse
CPU %Default metric
CustomApp-specific
HPA example
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-service
  minReplicas: 3
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
Warning

Missing resource requests

Without CPU/memory requests, HPA decisions are unreliable.

Mini takeaway
  • Define requests.
  • Tune targets.
  • Use custom metrics for business KPIs.
Quiz

HPA uses which baseline to calculate utilization?

Correct answer: B Utilization is based on resource requests, not limits.
Section 4 of 5 · ~6 mins

Health Probes & PodDisruptionBudgets

Readiness, liveness, and startup probes gate traffic and restarts. PDBs prevent too many Pods from going down at once.

🧒 ELI5

Doctor checkups

You only let healthy kids go to the playground.

🔗 ANALOGY

Event safety staff

A minimum number of staff must always stay for safety.

Technical

What it means in real code

Definition: Probes signal whether a container is ready/live; PDBs limit voluntary disruptions.

Why it matters

  • Safe rollouts
  • Avoids serving bad pods
  • Controls maintenance impact

Common pitfall

  • No readiness probe
  • Long liveness timeout
  • PDB too strict blocking node drain
readinesslivenessPDB
Startup probe→Readiness gate→Liveness restarts→PDB protects

Interview answer

“Readiness gates traffic, liveness restarts stuck pods, PDBs ensure a minimum number stay available.”

ProbePurpose
ReadinessTraffic gate
LivenessRestart stuck pods
Probe + PDB
readinessProbe:
  httpGet:
    path: /health/ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 10
---
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: api-pdb
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: api
Warning

PDBs can block node drains

If minAvailable is too high, nodes cannot be drained during maintenance.

Mini takeaway
  • Add readiness probes.
  • Use PDBs carefully.
  • Separate startup and liveness checks.
Quiz

Which probe controls when a Pod can receive traffic?

Correct answer: B Readiness determines if the Pod is ready to receive traffic.
Section 5 of 5 · ~7 mins

Requests, Limits & QoS

Resource requests and limits define scheduling and throttling behavior; they also determine QoS classes.

🧒 ELI5

Seat reservations

You reserve seats so everyone fits, and set rules for snacks.

🔗 ANALOGY

Restaurant bookings

Requests guarantee a table; limits stop a party from taking over.

Technical

What it means in real code

Definition: Requests guarantee minimum CPU/memory; limits cap usage and influence QoS (Guaranteed/Burstable/BestEffort).

Why it matters

  • Stable scheduling
  • HPA accuracy
  • Prevents noisy neighbors

Common pitfall

  • No limits for bursty apps
  • Requests too low
  • QoS mismatch
requestslimitsQoS
Set requests→Scheduler places pod→Limits enforce caps→QoS class assigned

Interview answer

“Requests drive scheduling and autoscaling; limits cap usage and define QoS.”

QoSCriteria
GuaranteedRequests = limits
BurstableRequests < limits
Requests and limits
resources:
  requests:
    cpu: "500m"
    memory: "512Mi"
  limits:
    cpu: "1"
    memory: "1Gi"
Warning

Throttling surprises

CPU limits can throttle throughput and increase latency.

Mini takeaway
  • Set realistic requests.
  • Use limits intentionally.
  • Know QoS impact.
Quiz

Which QoS class requires requests to equal limits?

Correct answer: C Guaranteed QoS requires requests equal limits for CPU/memory.
