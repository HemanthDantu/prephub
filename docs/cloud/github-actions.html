Section 1 of 5 · ~6 mins

Workflow Basics

A workflow is a YAML file that defines triggers and jobs.

🧒 ELI5

Recipe card

When you bake, you follow steps in the same order every time.

🔗 ANALOGY

Assembly checklist

A build checklist runs every time parts arrive.

Technical

What it means in real code

Definition: Workflow files in .github/workflows define events and jobs with steps.

Why it matters

  • Consistent CI
  • Automation on PRs
  • Reproducible builds

Common pitfall

  • No status checks
  • Single monolithic job
  • No branch filters
on:jobssteps
Trigger event→Runner starts→Steps execute→Status reported

Interview answer

“A workflow is a YAML pipeline with triggers and jobs running on GitHub-hosted or self-hosted runners.”

TriggerWhen
pushBranch update
pull_requestPR opened/updated
Minimal workflow
name: CI
on:
  pull_request:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm test
Warning

Running on every branch

Without branch filters, you can burn CI minutes fast.

Mini takeaway
  • Define triggers carefully.
  • Split jobs by responsibility.
  • Pin actions versions.
Quiz

Where do you store workflow YAML files?

Correct answer: B GitHub Actions expects workflows in .github/workflows.
Section 2 of 5 · ~6 mins

Jobs, Steps & Runners

Each job runs on a runner. Steps are sequential; jobs can be parallelized.

🧒 ELI5

Teams in different rooms

Each team does their tasks in a room; teams can work at the same time.

🔗 ANALOGY

Restaurant stations

Each station has its own cook and checklist.

Technical

What it means in real code

Definition: Jobs run on a specified runner, with ordered steps; needs defines dependencies.

Why it matters

  • Parallel builds
  • Isolation of stages
  • Deterministic order

Common pitfall

  • Too many sequential steps
  • No caching
  • Needs chain too deep
runs-onneedssteps
Job scheduled→Runner allocated→Steps execute→Artifacts uploaded

Interview answer

“Jobs are the unit of parallelism; steps run sequentially inside a job on a runner.”

FieldMeaning
runs-onRunner type
needsJob dependency
Job dependencies
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - run: npm test
  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - run: npm run build
Warning

Serializing everything

Overusing needs makes pipelines slow.

Mini takeaway
  • Parallelize independent jobs.
  • Use needs sparingly.
  • Choose appropriate runner types.
Quiz

How do you run jobs in parallel?

Correct answer: B Jobs without dependencies run in parallel.
Section 3 of 5 · ~6 mins

Matrix Builds & Caching

Matrix builds run the same job across versions; caching keeps builds fast.

🧒 ELI5

Test all sizes

You try the same recipe in small and big pots.

🔗 ANALOGY

Multi-lane testing

Run the same test in many lanes at once.

Technical

What it means in real code

Definition: Strategy.matrix creates job permutations; actions/cache stores dependencies.

Why it matters

  • Cross-version coverage
  • Faster feedback
  • Reduced cost

Common pitfall

  • Matrix explosion
  • No cache keys
  • Caching build outputs incorrectly
matrixcachekeys
Define matrix→Run permutations→Restore cache→Save cache

Interview answer

“Matrix builds let you test across versions; caching speeds up dependency installs.”

FeatureBenefit
MatrixCoverage
CacheSpeed
Matrix + cache
strategy:
  matrix:
    node: [18, 20]
steps:
  - uses: actions/setup-node@v4
    with:
      node-version: ${{ matrix.node }}
  - uses: actions/cache@v4
    with:
      path: ~/.npm
      key: ${{ runner.os }}-npm-${{ hashFiles('package-lock.json') }}
Warning

Cache key misses

If the key changes too often, the cache never hits.

Mini takeaway
  • Use matrix for versions.
  • Cache dependency dirs.
  • Limit matrix size.
Quiz

What is the main purpose of a matrix?

Correct answer: B Matrix builds run the same job for multiple versions.
Section 4 of 5 · ~6 mins

Secrets, Artifacts & Environments

Secrets keep credentials safe, artifacts share outputs, and environments add approvals and protections.

🧒 ELI5

Locked drawer

Keep your valuables locked and only share keys with trusted people.

🔗 ANALOGY

Passes and badges

Only staff with badges can access restricted areas.

Technical

What it means in real code

Definition: Secrets are masked env vars; artifacts persist files between jobs; environments gate deployments.

Why it matters

  • Protect credentials
  • Enable multi-job pipelines
  • Add human approvals

Common pitfall

  • Secrets in logs
  • Artifacts too big
  • Deploying without approvals
secretsartifactsenvs
Store secret→Inject env→Upload artifact→Download in next job

Interview answer

“Use secrets for credentials, artifacts for build outputs, and environments for approvals.”

ItemUsed for
SecretsCredentials
ArtifactsBuild outputs
Artifacts + env
- uses: actions/upload-artifact@v4
  with:
    name: build
    path: dist/

environment: production
Warning

Accidental secret exposure

Avoid echoing secrets or printing env variables.

Mini takeaway
  • Use environments for deploy gates.
  • Mask secrets.
  • Keep artifacts minimal.
Quiz

What should you use to pass build outputs between jobs?

Correct answer: B Artifacts are used to persist build outputs between jobs.
Section 5 of 5 · ~6 mins

Reusable Workflows & Security

Composite actions and reusable workflows keep pipelines DRY and secure.

🧒 ELI5

Reusable LEGO blocks

Build big things using the same small blocks.

🔗 ANALOGY

Company templates

Teams use the same template to avoid mistakes.

Technical

What it means in real code

Definition: Reusable workflows use workflow_call; composite actions bundle steps and can be versioned.

Why it matters

  • Consistency
  • Less copy/paste
  • Centralized updates

Common pitfall

  • Unpinned third-party actions
  • Broad permissions
  • No OIDC for cloud
workflow_callcompositepermissions
Create reusable workflow→Call from repo→Pin versions→Harden permissions

Interview answer

“Reusable workflows and composite actions standardize pipelines; always pin versions and scope permissions.”

FeatureUse
workflow_callReusable pipeline
OIDCShort-lived auth
Reusable workflow call
jobs:
  release:
    uses: org/.github/.github/workflows/release.yml@v1
    secrets: inherit
Warning

Unpinned actions

Always pin to a tag or SHA to avoid supply-chain issues.

Mini takeaway
  • Reuse workflows.
  • Pin actions.
  • Minimize permissions.
Quiz

What is the secure way to reference actions?

Correct answer: B Pin to a tag or SHA to reduce supply chain risk.

1-line interview answers

Workflow

YAML file defining triggers and jobs.

What to remember

Fast mental model

  • Triggers: events like push, PR, schedule.
  • Jobs: parallelizable units on runners.
  • Matrix: test across versions.
  • Security: secrets + least privilege.
