What is EC2?

Elastic Compute Cloud — virtual machines on demand. The most foundational AWS service. Understanding EC2 well means understanding how most AWS services are built behind the scenes.

EC2 From First Principles

Before EC2, if your company needed to run software you physically owned servers. You bought a Dell PowerEdge, racked it in a data centre, configured it, and ran your app on it. The hardware was dedicated to you — expensive, inflexible, slow to provision.

EC2 (Elastic Compute Cloud) virtualises this. On a single physical host machine, AWS runs a hypervisor — software that divides the physical hardware into many isolated virtual machines. Your EC2 instance is one of those VMs. You get what feels like a dedicated server: your own CPU, RAM, disk, network interface. But underneath, it's sharing physical hardware with hundreds of other customers' instances — completely isolated.

Analogy
EC2 is like renting an apartment vs buying a house

Owning a physical server = buying a house. High upfront cost. You own it but maintain everything. Stuck with the size you bought. EC2 = renting an apartment. Pay monthly (or hourly). Move to a bigger apartment when needed. Building maintenance (hardware, hypervisor) is someone else's problem. Walk away when done. And AWS has 1000s of apartments in the same building — economies of scale mean the cost per room is far lower than building your own house.

What You Actually Get with an EC2 Instance

Compute resources
vCPU: Virtual CPU cores — fraction of a physical CPU's time. A c6i.2xlarge gives you 8 vCPUs.

RAM: Dedicated memory, not shared. A r6i.large gives you 16 GB RAM.

Network: A virtual network interface (ENI) attached to your VPC subnet with a private IP. Optionally a public IP.
Storage options
EBS (default): Network-attached block storage. Persists after instance stop/restart. Like a hard drive you can detach and reattach.

Instance store: Physical disk on the host machine. Extremely fast. BUT: data is LOST when instance stops or fails. Use only for temp data, caches.

EFS/FSx: Shared filesystem mounted by multiple instances.

AWS Nitro — What Powers EC2 Under the Hood

Starting in 2017, AWS replaced its Xen-based hypervisor with the Nitro System — custom hardware AWS designed themselves. This matters because:

Near bare-metal performance
Nitro offloads virtualisation to dedicated hardware cards. The CPU is almost entirely available for your workload — virtualization overhead is near zero (~0.1%).
Enhanced networking
Network I/O is handled by a separate Nitro card — up to 100 Gbps bandwidth on some instance types. Storage IOPS (EBS) is also offloaded to a dedicated controller.
Security isolation
Security group rules are enforced at the Nitro hardware level — not in software that runs in your VM. This means even if someone compromised the guest OS, the network rules can't be bypassed.

Instance Types & Sizing

AWS has 700+ instance types across dozens of families. You don't need to memorise them all — you need to understand the naming system and pick the right family for your workload.

The Naming System — Decode Any Instance Type

Every EC2 instance type name follows a pattern: Family + Generation + Attributes + Size

# Format: {family}{generation}{attributes}.{size}

m7g.2xlarge
│││ └── .2xlarge = 8 vCPU, 32 GB RAM (size multiplier)
││└──── g = Graviton (AWS-designed ARM processor)
│└───── 7 = 7th generation (newer = faster + cheaper)
└────── m = general purpose (the family)

c6i.4xlarge
│││ └── .4xlarge = 16 vCPU, 32 GB RAM
││└──── i = Intel processor (vs g=Graviton, a=AMD, d=local NVMe)
│└───── 6 = 6th generation
└────── c = compute optimised

r6g.xlarge
│││ └── .xlarge = 4 vCPU, 32 GB RAM
││└──── g = Graviton
│└───── 6 = 6th generation
└────── r = memory optimised (high RAM ratio)

Graviton (g) = best value in most cases. AWS's custom ARM processors (Graviton2, Graviton3) deliver 20–40% better price/performance than Intel/AMD equivalents. If your application runs on Linux and is compatible with ARM, choose Graviton. Most modern Java, Python, Go, Node.js apps work perfectly. Always test first — some native libraries may not have ARM builds.

How to Pick the Right Instance Type

1
Start with M-family
For any new workload you don't have data on: start with m6i.large or m7g.large. It's a safe balanced choice. Collect CPU and memory metrics for 1-2 weeks.
2
Analyse CloudWatch metrics
CPU consistently above 70%? Move to C-family (more CPU). Memory constantly high? Move to R-family (more RAM). Both low? Downsize to save money. This is how you right-size.
3
Consider Graviton
Switch m6i to m7g — same price or cheaper, 20-40% better performance. Test your application, run load tests, compare results. Most apps work with no code changes.
4
Use AWS Compute Optimizer
AWS's free service that analyses your actual utilisation and recommends right-sized instances. Catches over-provisioned instances wasting money.

AMIs — Amazon Machine Images

An AMI is the template your EC2 instance boots from. It contains the OS, pre-installed software, configuration, and the initial EBS volume snapshots. Understanding AMIs is essential for automation and scaling.

What an AMI Contains

Think of an AMI as a complete snapshot of a machine at a point in time. When you launch an EC2 instance from an AMI, AWS creates a new EBS volume from the AMI's snapshot and boots from it. Every instance launched from the same AMI starts in the exact same state.

AMI Components
Root volume snapshot: EBS snapshot of the OS and all installed software at the time the AMI was created. This becomes the root EBS volume of your new instance.

Launch permissions: Controls who can use the AMI — private (only your account), shared (specific accounts), or public (anyone on AWS).

Block device mapping: Defines what volumes to attach and how — root volume size, additional data volumes, instance store volumes.

Virtualisation type: HVM (Hardware Virtual Machine) — the modern standard that supports Nitro instances.

Golden AMI Pattern — The Right Way to Scale

The Golden AMI pattern is how professional teams handle EC2 scaling. Instead of booting bare OS instances and installing software on startup (slow, fragile), you pre-bake everything into an AMI. New instances start in seconds because nothing needs to be installed.

❌ Anti-pattern: Install on boot
# User data script runs at boot:
yum install -y java-17
aws s3 cp s3://my-bucket/app.jar .
java -jar app.jar

# Problems:
# - Takes 3-5 minutes to boot
# - yum install fails if internet is slow
# - S3 download fails if IAM not ready
# - S3 bucket might be down
# - 3min before instance is healthy
#   = slow auto scaling response
✅ Golden AMI: Everything pre-baked
# AMI was built with:
# - Java 17 already installed
# - App jar already present
# - systemd service configured

# User data only:
#!/bin/bash
# Pull config from Parameter Store
APP_ENV=$(aws ssm get-parameter ...)
echo "APP_ENV=${APP_ENV}" >> /etc/environment
systemctl start myapp

# Benefits:
# - Boots in 30-60 seconds
# - No external dependencies
# - Reliable, repeatable, fast scaling
# - Rollback = use previous AMI version

Creating a Custom AMI — Full Workflow

# Step 1: Start with a base EC2 instance
aws ec2 run-instances \
  --image-id ami-0c02fb55956c7d316 \  # Amazon Linux 2023
  --instance-type t3.medium \
  --key-name my-key-pair \
  --subnet-id subnet-0pub1a \
  --security-group-ids sg-0base \
  --iam-instance-profile Name=EC2BuildRole

# Step 2: SSH in and install everything
ssh -i my-key.pem ec2-user@54.x.x.x

sudo dnf install -y java-17-amazon-corretto
sudo mkdir -p /opt/myapp
sudo aws s3 cp s3://build-artifacts/myapp-1.2.3.jar /opt/myapp/app.jar

# Create systemd service
sudo tee /etc/systemd/system/myapp.service <<EOF
[Unit]
Description=My Application
After=network.target

[Service]
User=ec2-user
WorkingDirectory=/opt/myapp
ExecStart=/usr/bin/java -Xmx2g -jar app.jar
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl enable myapp  # start on boot
sudo systemctl start myapp   # verify it works

# Clean up before imaging (remove logs, temp files, SSH keys)
sudo rm -rf /var/log/* /tmp/* ~/.bash_history
sudo shred -u ~/.ssh/authorized_keys

# Step 3: Create the AMI
aws ec2 create-image \
  --instance-id i-0abc123def456789 \
  --name "myapp-v1.2.3-$(date +%Y%m%d-%H%M)" \
  --description "MyApp 1.2.3 on Amazon Linux 2023" \
  --no-reboot  # don't reboot (faster but risk of filesystem inconsistency)
               # Remove --no-reboot for critical production images

# AMI ID returned: ami-0newimage123456
# Step 4: Update your Launch Template/Auto Scaling Group to use new AMI
# Step 5: Instance refresh — gradually replace running instances
aws autoscaling start-instance-refresh \
  --auto-scaling-group-name my-app-asg \
  --preferences '{"MinHealthyPercentage":90,"InstanceWarmup":60}'

AMI and Auto Scaling — The Connection

AMIs are central to Auto Scaling. Your Auto Scaling Group uses a Launch Template that specifies the AMI ID. When the ASG scales out, it launches new instances from that AMI. When you deploy a new version, you create a new AMI and update the Launch Template — then trigger an instance refresh to roll out gradually.

EC2 Lifecycle & States

An EC2 instance goes through distinct states from launch to termination. Understanding what happens at each state — especially the billing and data implications — prevents expensive surprises.

The Instance State Machine

Stop vs Terminate — critical difference: Stop = hibernate (EBS data preserved, resume later). Terminate = delete (instance and root EBS gone, all RAM gone, any instance store gone). Most "delete instance" UX errors are actually someone terminating instead of stopping. Enable termination protection on production instances: aws ec2 modify-instance-attribute --instance-id i-xxx --disable-api-termination.

What Happens on Stop and Start

1
Instance stops
OS receives shutdown signal. Processes terminate. Kernel unmounts filesystems. Power off. Billing stops immediately. Instance store data is LOST (NVMe drives wiped). EBS volumes persist.
2
Instance may move to a different physical host
This is important: when you start a stopped instance, it may boot on a completely different physical host. The Nitro hypervisor remounts your EBS volumes to the new host. This is how AWS does hardware maintenance.
3
Public IP changes
Each time you stop and start an instance, it gets a new public IP address (unless you have an Elastic IP). This is why hardcoding public IPs breaks. Use DNS names instead, or assign an Elastic IP for a stable address.
4
Private IP stays the same
The private IP address from the VPC subnet is retained across stop/start. Your internal services referencing this IP keep working.

Instance Metadata — How Your Instance Knows About Itself

The Instance Metadata Service (IMDS) is a local HTTP endpoint available inside every EC2 instance. It provides information about the instance without any authentication. It also provides IAM credentials for the attached role.

# IMDS is available at 169.254.169.254 — only from within the instance
# Using IMDSv2 (required in modern setups — token-based for security)

# Step 1: Get a session token (expires after TTL)
TOKEN=$(curl -X PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")

# Step 2: Use token to query metadata
# Instance ID
curl -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/instance-id
# i-0abc123def456789

# Current AZ (useful for AZ-aware routing in your app)
curl -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/placement/availability-zone
# us-east-1a

# Instance type
curl -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/instance-type
# m7g.2xlarge

# IAM role credentials (this is how the SDK gets temp credentials)
curl -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/iam/security-credentials/MyRole
# Returns: AccessKeyId, SecretAccessKey, Token, Expiration

# User data script (your bootstrap script)
curl -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/user-data

# Private IP address
curl -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/local-ipv4
# 10.0.11.45
IMDSv2 vs v1: IMDSv1 is unauthenticated — any process inside the VM can query metadata. In SSRF (Server-Side Request Forgery) attacks, an attacker tricks your app into fetching metadata and stealing IAM credentials. IMDSv2 requires a session token (PUT request first) which is SSRF-resistant. Enforce IMDSv2 only in production: --metadata-options HttpTokens=required at launch or via policy.

User Data & Bootstrapping

User data is a script that runs exactly once — on the very first boot of a new instance. It's how you automate configuration, software installation, and application startup.

How User Data Works

When EC2 launches an instance, the cloud-init service detects user data in the instance metadata and executes it. Important rules:

Runs once only
User data runs on the first boot after launch. If you stop and start the instance, it does NOT run again (unless you force re-execution with cloud-init config).
Runs as root
The script executes as the root user. No sudo needed. But be careful — mistakes have full system access. Use runuser -l ec2-user -c to run commands as a specific user.
Logs at /var/log
Check /var/log/cloud-init-output.log for all output from user data execution. Check /var/log/cloud-init.log for cloud-init metadata. Essential for debugging startup failures.

Real-World User Data Scripts

Example 1: Spring Boot app server

#!/bin/bash
# Log everything for debugging
exec > >(tee /var/log/user-data.log) 2>&1
set -euxo pipefail   # exit on error, log commands

# 1. System updates
dnf update -y
dnf install -y amazon-cloudwatch-agent

# 2. Fetch config from Systems Manager Parameter Store
# (Instance role must have ssm:GetParameter permission)
ENVIRONMENT=$(aws ssm get-parameter \
  --name "/myapp/environment" \
  --query "Parameter.Value" \
  --output text)

DB_URL=$(aws ssm get-parameter \
  --name "/myapp/$ENVIRONMENT/db-url" \
  --with-decryption \
  --query "Parameter.Value" \
  --output text)

# 3. Configure the application
cat > /opt/myapp/application.properties <<EOF
spring.datasource.url=${DB_URL}
server.port=8080
spring.profiles.active=${ENVIRONMENT}
EOF

# 4. Configure CloudWatch agent (ship logs to CloudWatch)
cat > /opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json <<'CWEOF'
{
  "logs": {
    "logs_collected": {
      "files": {
        "collect_list": [{
          "file_path": "/opt/myapp/logs/application.log",
          "log_group_name": "/myapp/application",
          "log_stream_name": "{instance_id}"
        }]
      }
    }
  }
}
CWEOF
/opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \
  -a fetch-config -m ec2 -s \
  -c file:/opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json

# 5. Start the application
systemctl enable myapp
systemctl start myapp

# 6. Signal Auto Scaling that instance is ready (if using lifecycle hooks)
INSTANCE_ID=$(curl -s http://169.254.169.254/latest/meta-data/instance-id)
ASG_NAME=$(aws ec2 describe-instances --instance-ids $INSTANCE_ID \

  --lifecycle-action-result CONTINUE \
  --instance-id $INSTANCE_ID \
  --auto-scaling-group-name $ASG_NAME \
  --lifecycle-hook-name instance-launching

Launch Templates — Reusable Configuration

Instead of specifying all EC2 options every time you launch an instance (or configure an ASG), a Launch Template stores the configuration: AMI, instance type, security groups, user data, IAM profile, etc. It's versioned — roll back to a previous version if a new AMI causes issues.

# Create a Launch Template
aws ec2 create-launch-template \
  --launch-template-name myapp-lt \
  --version-description "v1.2.3 - Java 21" \
  --launch-template-data '{
    "ImageId": "ami-0new123456",
    "InstanceType": "m7g.large",
    "IamInstanceProfile": {"Name": "MyAppInstanceProfile"},
    "SecurityGroupIds": ["sg-0app123"],
    "UserData": "'$(base64 -w0 user-data.sh)'",
    "MetadataOptions": {"HttpTokens": "required"},
    "BlockDeviceMappings": [{
      "DeviceName": "/dev/xvda",
      "Ebs": {
        "VolumeSize": 30,
        "VolumeType": "gp3",
        "Encrypted": true,
        "DeleteOnTermination": true
      }
    }],
    "TagSpecifications": [{
      "ResourceType": "instance",
      "Tags": [
        {"Key": "Name", "Value": "myapp-server"},
        {"Key": "Environment", "Value": "production"}
      ]
    }]
  }'

# Update ASG to use new Launch Template version
aws autoscaling update-auto-scaling-group \
  --auto-scaling-group-name my-app-asg \
  --launch-template LaunchTemplateName=myapp-lt,Version='$Latest'

# Gradually replace instances with new template (rolling update)
aws autoscaling start-instance-refresh \
  --auto-scaling-group-name my-app-asg \
  --preferences '{"MinHealthyPercentage":80,"InstanceWarmup":120}'

Auto Scaling Groups

Auto Scaling Groups (ASGs) automatically add or remove EC2 instances based on demand. This is how you achieve elasticity — the core cloud benefit of matching capacity to load.

What an ASG Does

An ASG maintains a fleet of EC2 instances between a minimum and maximum count. It monitors health checks and replaces unhealthy instances automatically. It scales out (adds instances) when load increases and scales in (removes instances) when load decreases — following rules you define.

Min / Desired / Max
Min: never go below this (floor). Desired: target count right now. Max: never exceed this (ceiling). Example: min=2, desired=4, max=20. ASG maintains 4 instances. Scaling events change desired between min and max.
Health checks
EC2 health check: is the instance running? ALB health check: is the app responding on /health? If an instance fails health — ASG terminates it and launches a replacement. Self-healing infrastructure.
Multi-AZ spread
Configure ASG with subnets across multiple AZs. ASG automatically balances instances across AZs. If one AZ fails, instances in other AZs keep serving traffic.

Scaling Policies — When to Scale

## 1. Target Tracking (simplest, recommended)
# "Keep average CPU at 60%"
# ASG automatically adds/removes instances to maintain this target
aws autoscaling put-scaling-policy \
  --auto-scaling-group-name my-app-asg \
  --policy-name cpu-target-tracking \
  --policy-type TargetTrackingScaling \
  --target-tracking-configuration '{
    "PredefinedMetricSpecification": {
      "PredefinedMetricType": "ASGAverageCPUUtilization"
    },
    "TargetValue": 60.0,
    "ScaleInCooldown": 300,
    "ScaleOutCooldown": 60
  }'

## 2. Step Scaling (fine-grained)
# CPU 70-80%: add 1 instance; CPU 80-90%: add 2; CPU >90%: add 4
aws autoscaling put-scaling-policy \
  --policy-type StepScaling \
  --adjustment-type ChangeInCapacity \
  --step-adjustments '[
    {"MetricIntervalLowerBound":0,"MetricIntervalUpperBound":10,"ScalingAdjustment":1},
    {"MetricIntervalLowerBound":10,"MetricIntervalUpperBound":20,"ScalingAdjustment":2},
    {"MetricIntervalLowerBound":20,"ScalingAdjustment":4}
  ]'

## 3. Scheduled Scaling (predictable load)
# Scale up every weekday at 8am, scale down at 8pm
aws autoscaling put-scheduled-update-group-action \
  --auto-scaling-group-name my-app-asg \
  --scheduled-action-name scale-up-morning \
  --recurrence "0 8 * * MON-FRI" \
  --desired-capacity 10 --min-size 6

aws autoscaling put-scheduled-update-group-action \
  --scheduled-action-name scale-down-evening \
  --recurrence "0 20 * * MON-FRI" \
  --desired-capacity 4 --min-size 2

Lifecycle Hooks — Taking Action During Scale Events

Lifecycle hooks pause the instance in a state (Pending:Wait or Terminating:Wait) to let you run custom actions before the instance enters service or before it's terminated.

# Scale-out lifecycle hook: wait for app to register with service discovery
# Instance boots → enters Pending:Wait → your Lambda runs registration
# → completes hook → instance enters InService

# Scale-in hook: drain connections before termination
# Scale-in triggered → Terminating:Wait → ALB drains connections
# → your hook completes → instance terminated

aws autoscaling put-lifecycle-hook \
  --auto-scaling-group-name my-app-asg \
  --lifecycle-hook-name launch-hook \
  --lifecycle-transition autoscaling:EC2_INSTANCE_LAUNCHING \
  --default-result CONTINUE \
  --heartbeat-timeout 300  # 5 minutes to complete

aws autoscaling put-lifecycle-hook \
  --lifecycle-hook-name terminate-hook \
  --lifecycle-transition autoscaling:EC2_INSTANCE_TERMINATING \
  --default-result CONTINUE \
  --heartbeat-timeout 60   # 1 min to drain

What is Lambda?

AWS Lambda is serverless compute — you write a function, AWS runs it when something triggers it. No servers to provision, no OS to patch, no capacity to plan. You only pay for execution time to the millisecond.

The Serverless Mental Shift

With EC2 you think: "I need 4 servers running my app, always on, watching for requests." With Lambda you think: "Here's what to run WHEN an event happens. I don't care how many instances run it." The infrastructure scales automatically — from 0 to thousands of concurrent executions.

EC2 Model — Always Provisioned

4 EC2 instances running 24/7. At 3am with 5 users, you're paying for 4 full servers doing nothing. At noon with 50,000 users, you must wait for Auto Scaling to spin up more servers (2-3 minutes). Billing: 24 hours × 4 instances × $0.08/hr = ~$7.68/day regardless of traffic.

Lambda Model — Event-Driven

At 3am with 5 users: 5 Lambda containers running, ~5ms each. At noon with 50,000 simultaneous users: 50,000 Lambda containers, no warmup wait. Billing: you pay for those 5ms × 5 invocations at 3am and those execution-ms × 50,000 at noon. Idle time = $0.

Lambda Execution Model — What Actually Happens

Lambda does NOT run "serverlessly" — it runs in containers (microVMs actually, using AWS Firecracker). You just don't manage them. Understanding the execution model is critical for designing Lambda-based systems.

1
Trigger arrives
An event source (API Gateway, SQS message, S3 event, scheduled cron) sends an event to Lambda. Lambda service receives it.
2
Lambda finds or creates an execution environment
Lambda maintains a pool of "warm" execution environments from recent invocations. If one is available (warm), it reuses it immediately. If not (cold start), it must create a new one — this takes 100ms to several seconds depending on runtime and package size.
3
Function handler runs
Your handler function (handleRequest) is called with the event payload. It executes, calls other services, writes to databases, etc.
4
Environment stays warm (for a while)
After your handler returns, the execution environment stays alive (frozen, not billed) waiting for the next invocation — typically 5-15 minutes. If a new invocation arrives, it reuses this environment with no cold start. This is "warm start."
5
Environment is recycled
After sufficient idle time, AWS recycles the execution environment. The next invocation starts fresh with a cold start.

Cold Starts — Deep Dive

Cold starts are the most misunderstood aspect of Lambda. Understanding exactly what happens, how long each phase takes, and every technique to reduce them is essential for production Lambda systems.

What is a Cold Start — Precisely

A cold start is what happens when Lambda has no warm execution environment available and must create a new one from scratch. It involves downloading your code, initialising the runtime, and running your initialisation code — before even touching your handler function.

The Cold Start Timeline — Phase by Phase

❄️ COLD START (everything below runs)
① Download code package · 50–500ms
Lambda downloads your deployment package or container image. Large packages = longer download. Java 50MB JAR → longer than Node 1MB ZIP.
② Start Firecracker microVM · 10–50ms
AWS Firecracker starts a new microVM for isolation. Very fast — Firecracker boots in ~125ms including Linux kernel.
③ Start language runtime (JVM) · 200ms–2,000ms
THIS IS THE BIG ONE for Java. JVM startup: load JVM, load classes, run bytecode verifier. Java cold starts are 1–3 seconds. Node.js/Python: ~100ms. Go/Rust: ~10ms.
④ Run your init code · 100ms–5,000ms (your fault!)
Code OUTSIDE your handler runs here: Spring context init, DynamoDB client creation, database connection pool, reading SSM parameters. This is where most cold start time hides.
🔥 WARM START (only this runs)
⑤ Your handler function · 1ms–500ms
On warm start: phases ①–④ already done. Execution environment reused. Only your handler code runs. This is why warm starts are so much faster.

Event Sources & Triggers

Lambda is always triggered by an event. Understanding exactly how each event source delivers events — what the payload looks like, how errors are handled, how retries work — determines whether your system is reliable or loses data.

Three Invocation Models

Synchronous (request/response)
Caller waits for Lambda to finish and return a response. If Lambda fails, the error is returned to the caller — no automatic retry.

Sources: API Gateway, ALB, SDK direct invocation, Cognito triggers
Asynchronous (fire and forget)
Event is placed in Lambda's internal queue. Lambda retries up to 2 times on failure. After 3 total failures, event goes to Dead Letter Queue (DLQ).

Sources: S3, SNS, EventBridge, CloudWatch Events, Cognito
Polling (Lambda pulls)
Lambda polls the source for new records. You don't trigger Lambda — Lambda reads from the source continuously.

Sources: SQS, Kinesis, DynamoDB Streams, Kafka (MSK)

SQS → Lambda (Most Common Pattern)

Lambda polls an SQS queue using an Event Source Mapping (ESM). Lambda reads messages in batches, invokes your function with a batch, and deletes the messages from SQS only if your function succeeds. If it fails, messages become visible again after the visibility timeout and will be retried.

# Create the ESM — Lambda polls SQS automatically
aws lambda create-event-source-mapping \
  --function-name OrderProcessor \
  --event-source-arn arn:aws:sqs:us-east-1:123456789012:orders-queue \
  --batch-size 10 \                        # process 10 messages per invocation
  --maximum-batching-window-in-seconds 5 \ # wait up to 5s to fill a batch
  --function-response-types ReportBatchItemFailures  # report partial failures

# Your handler receives a batch of SQS messages
public class OrderHandler implements RequestHandler<SQSEvent, SQSBatchResponse> {

    public SQSBatchResponse handleRequest(SQSEvent event, Context ctx) {
        List<SQSBatchResponse.BatchItemFailure> failures = new ArrayList<>();

        for (SQSEvent.SQSMessage msg : event.getRecords()) {
            try {
                Order order = parseOrder(msg.getBody());
                orderService.process(order);
                // Success — SQS will delete this message
            } catch (Exception e) {
                log.error("Failed to process order: {}", msg.getMessageId(), e);
                // Report this specific message as failed
                // Other messages in batch still succeed (partial batch response)
                failures.add(SQSBatchResponse.BatchItemFailure.builder()
                    .withItemIdentifier(msg.getMessageId())
                    .build());
            }
        }

        // Return failed message IDs → only those messages go back to queue
        // Successful messages are deleted from SQS
        return SQSBatchResponse.builder()
            .withBatchItemFailures(failures)
            .build();
    }
}

Lambda Concurrency Model

Concurrency in Lambda is completely different from threads in a traditional app. Understanding it prevents your functions from throttling each other and helps you design reliable event-driven systems.

What is Lambda Concurrency?

In a traditional server, concurrency = threads running simultaneously inside ONE process. In Lambda, concurrency = number of execution environments running simultaneously, each handling ONE request at a time. Lambda does NOT share an environment between concurrent requests.

Mental Model
Lambda concurrency = number of workers, not threads per worker

Traditional server: 1 server with 100 threads, each handling a request simultaneously. Lambda: 100 separate worker instances (execution environments), each handling 1 request. If 101 requests arrive simultaneously, Lambda spins up a 101st environment (or throttles if you've hit limits).

# Concurrency = simultaneous executions at any given moment
# Formula: Concurrency = (requests per second) × (avg duration in seconds)
# Example:
#   1000 req/sec × 0.1 sec avg = 100 concurrent executions
#   10 req/sec × 5 sec avg = 50 concurrent executions

# Account default limit: 1000 concurrent executions per region (soft limit)
# Can be increased by requesting a service quota increase

# Types of concurrency limits:

# 1. Reserved Concurrency — guarantees AND caps a function
aws lambda put-function-concurrency \
  --function-name CriticalOrderProcessor \
  --reserved-concurrent-executions 200
# This function ALWAYS has 200 concurrency available (reserved from pool)
# It NEVER exceeds 200 (protection against runaway scaling)

# 2. Unreserved Concurrency — shared pool minus reservations
# All functions without reserved concurrency share what's left

# 3. Provisioned Concurrency — pre-warmed environments
# (Covered in Cold Starts section)

Throttling — When Concurrency Limit is Hit

# When a function hits its concurrency limit:
# - Synchronous invocations (API Gateway, ALB): return 429 TooManyRequests immediately
# - Async invocations (S3, SNS): Lambda retries for up to 6 hours
# - Polling (SQS, Kinesis): Lambda stops reading from source until concurrency frees up

# Detect throttles in CloudWatch
aws cloudwatch get-metric-statistics \
  --namespace AWS/Lambda \
  --metric-name Throttles \

  --start-time 2024-01-15T00:00:00 \
  --end-time 2024-01-16T00:00:00 \
  --period 3600 \
  --statistics Sum

# In your code — handle 429 from synchronous invocations
# (SDK handles it with exponential backoff by default for retryable errors)

Reserved Concurrency as a Rate Limiter

Set reserved concurrency to 0 to completely disable a function temporarily without deleting it — useful for emergency incident response.

# Disable a runaway Lambda immediately (incident response)

Fargate vs EC2 Launch Type

ECS can run containers on two backends — Fargate (AWS manages the server) or EC2 (you provide and manage the servers). This is the most fundamental ECS decision.

The Core Difference

EC2 Launch Type — You Manage the Servers

You run an ECS Agent on EC2 instances in your cluster. ECS places containers on those instances. You are responsible for the EC2 fleet: sizing, patching, capacity planning, replacing unhealthy instances, OS updates.

✓ Lower per-unit cost at scale. ✓ Control over instance type. ✓ Spot Instances for cost savings. ✗ Operational overhead. ✗ Must manage cluster capacity.

Fargate — Serverless Containers

You define CPU and memory per task. AWS allocates a dedicated microVM, runs your container, and you pay per task-second. No EC2 instances to manage. No capacity planning. No OS patches.

✓ Zero server management. ✓ Scale to zero when no tasks. ✓ Pay per task (not per idle server). ✗ Higher per-unit cost vs EC2. ✗ Cold start (~30s to start a task).

Fargate — How It Works

With Fargate, you specify CPU and memory in your task definition. When ECS needs to run a task, Fargate allocates a dedicated microVM (using Firecracker, same as Lambda) just for that task. The task runs in complete isolation — no other customer's tasks share that microVM.

Task Definitions & Services

Task definitions are versioned blueprints. Services maintain running counts of tasks, handle deployments, and wire to load balancers. These two together are the heart of ECS.

Task Definition — Everything About How to Run

A Task Definition is a JSON document that describes how to run one or more containers together. It's versioned — each change creates a new revision. Your service runs a specific revision. Rollback = change the service to a previous revision.

# Register a task definition (creates revision 1, or increments revision number)
aws ecs register-task-definition \
  --cli-input-json file://myapp-task-definition.json

# List task definition revisions
aws ecs list-task-definitions --family-prefix myapp-task
# Returns:
# arn:aws:ecs:us-east-1:123:task-definition/myapp-task:1
# arn:aws:ecs:us-east-1:123:task-definition/myapp-task:2
# arn:aws:ecs:us-east-1:123:task-definition/myapp-task:3  ← current

# Rollback: update service to use previous revision
aws ecs update-service \
  --cluster production \
  --service myapp-service \
  --task-definition myapp-task:2   # roll back to revision 2

ECS Service — The Service Manager

An ECS Service ensures that a specified number of task instances are always running. It starts new tasks when existing ones fail, integrates with ALB for load balancing, and controls deployments (rolling updates, blue/green).

# Create a Fargate service with ALB integration
aws ecs create-service \
  --cluster production \
  --service-name myapp-service \
  --task-definition myapp-task:3 \
  --desired-count 6 \
  --launch-type FARGATE \
  --network-configuration "awsvpcConfiguration={
    subnets=[subnet-0app1a,subnet-0app1b,subnet-0app1c],
    securityGroups=[sg-0app123],
    assignPublicIp=DISABLED
  }" \
  --load-balancers "targetGroupArn=arn:aws:elasticloadbalancing:...,
                    containerName=myapp,
                    containerPort=8080" \
  --deployment-configuration '{
    "deploymentCircuitBreaker": {"enable": true, "rollback": true},
    "minimumHealthyPercent": 100,
    "maximumPercent": 200
  }' \
  --health-check-grace-period-seconds 120

# minimumHealthyPercent=100, maximumPercent=200 means:
# During deployment: start new tasks BEFORE removing old ones
# Total tasks may reach 12 (200% of desired=6) during the update
# Ensures no reduction in capacity during deployment

# Deployment Circuit Breaker:
# If new tasks fail health checks → automatically roll back to previous revision
# No manual intervention needed

# Rolling deployment — update to new task definition version
aws ecs update-service \
  --cluster production \
  --service myapp-service \
  --task-definition myapp-task:4 \
  --force-new-deployment
# ECS starts new tasks with myapp-task:4, stops old tasks gradually
# ALB only routes to tasks that pass health checks
# Unhealthy deployment → circuit breaker rolls back to myapp-task:3

How ECS services integrate with load balancers, register/deregister tasks automatically, and scale based on CloudWatch metrics.

ALB + ECS — Dynamic Port Mapping

When ECS tasks use Fargate (awsvpc networking), each task gets its own ENI with a unique private IP. The ALB target group tracks these IPs dynamically — as tasks start, their IPs are registered with the ALB; as tasks stop, they're deregistered. All automatic, no manual configuration.

# Step 1: Create ALB and target group
aws elbv2 create-load-balancer \
  --name myapp-alb \
  --subnets subnet-0pub1a subnet-0pub1b subnet-0pub1c \
  --security-groups sg-0alb123 \
  --scheme internet-facing

aws elbv2 create-target-group \
  --name myapp-targets \
  --protocol HTTP --port 8080 \
  --vpc-id vpc-0abc123def456789 \
  --target-type ip \               # ip (not instance) for Fargate awsvpc
  --health-check-path /actuator/health \
  --health-check-interval-seconds 30 \
  --healthy-threshold-count 2 \
  --unhealthy-threshold-count 3

# Step 2: ECS automatically registers/deregisters task IPs
# When a new task starts:
#   ECS registers its IP:port with the ALB target group
# When a task is stopping:
#   ECS deregisters it → ALB waits for connections to drain (deregistration delay)
#   → ECS stops the container
# This ensures in-flight requests complete before container is killed

# Configure deregistration delay (how long ALB waits for connections to drain)
aws elbv2 modify-target-group-attributes \
  --target-group-arn arn:aws:elasticloadbalancing:...:targetgroup/myapp-targets/xxx \
  --attributes Key=deregistration_delay.timeout_seconds,Value=30

  --cluster $CLUSTER \
  --service $SERVICE \
  --task-definition $NEW_TASK_ARN \
  --force-new-deployment

aws ecs wait services-stable \
  --cluster $CLUSTER \
  --services $SERVICE

Interview Questions — Compute

18 questions across EC2, Lambda, and ECS — from junior developer to senior architect.

EC2 — Beginner to Intermediate

What is the difference between stopping and terminating an EC2 instance? ▼
Stop: Instance is shut down (like pressing power off). EBS volumes are preserved. Instance can be restarted. RAM contents are lost. The instance may move to a different physical host on restart. Public IP changes (unless Elastic IP). You pay only for EBS storage while stopped, not for compute. Terminate: Instance is permanently deleted. Root EBS volume is deleted (by default — this is controlled by the DeleteOnTermination flag). Additional EBS volumes are detached but not deleted by default. Cannot be undone. Enable termination protection on production instances to prevent accidental termination: modify-instance-attribute --disable-api-termination.
What is an AMI and how would you use it for auto scaling? ▼
An AMI (Amazon Machine Image) is a snapshot of an EC2 instance including the OS, installed software, configuration, and EBS volume data. For auto scaling: (1) Build a "golden AMI" — start with a base OS AMI, install all your software, run your app, verify it works, then create a new AMI from that instance. (2) Reference this AMI in a Launch Template. (3) ASG uses the Launch Template to start new instances from your pre-baked AMI. (4) New instances boot in 30-60 seconds (no software to install) and are healthy immediately. For deployments: create a new AMI with the new version, update the Launch Template, trigger an instance refresh to gradually replace running instances.
What EC2 instance type would you choose for each: web server, ML model training, in-memory database? ▼
Web server: M-family (m7g.large or m7g.xlarge) — general purpose, balanced CPU/RAM. Graviton (g suffix) for 20-40% better price/performance on Linux. Start here for any new web workload and right-size after analysing metrics. ML model training: P-family (p4d.24xlarge or p3.16xlarge) — high-performance NVIDIA GPUs specifically designed for deep learning. Or G-family (g5.2xlarge) for less demanding ML workloads. Use Spot Instances to cut cost 70-90% since training jobs are fault-tolerant. In-memory database (Redis, Memcached): R-family (r7g.large or r6i.2xlarge) — memory optimised, highest RAM-to-CPU ratio at the best price. Redis is almost entirely RAM-bound so memory capacity determines your working set size.
What are Spot Instances and when should you use them? What's the risk? ▼
Spot Instances use spare AWS capacity at up to 90% discount vs On-Demand. AWS can reclaim them with 2 minutes notice. Use when: (1) Workload is fault-tolerant — batch jobs, data processing, image rendering, ML training. If interrupted, the job restarts from last checkpoint. (2) Workload is flexible — can run at any time, doesn't need to be continuous. (3) Workload can be distributed — use multiple Spot Instance types across multiple AZs to reduce interruption risk. Never use Spot for: production databases, stateful services, anything where a 2-minute outage is unacceptable. Best practice: use diversified Spot pools (multiple instance types and AZs) and mix 70% Spot with 30% On-Demand for production services.

Lambda — Intermediate

Explain Lambda cold starts in detail. What causes them and how do you reduce them for Java? ▼
A cold start happens when Lambda has no warm execution environment and must create one. Phases: (1) Download code package from S3 (50–500ms — large JAR = longer). (2) Start Firecracker microVM (10–50ms). (3) Start the JVM (200–2,000ms — this is Java's main penalty: class loading, JIT compilation warmup). (4) Run initialisation code outside the handler: SDK client creation, Spring context init, config loading (100ms–5,000ms — this is your code's fault). Total Java cold start: 1–5 seconds. Reductions: (1) Lambda SnapStart: JVM snapshot — restores in ~300ms instead of starting fresh. Best option for Java. (2) Provisioned Concurrency: Pre-warms N environments — zero cold start for those slots. Expensive for high counts. (3) Minimise init code: Only create what's needed outside the handler. No Spring ApplicationContext — use lightweight DI. (4) GraalVM native: Compile to native binary — ~10ms cold start, no JVM. Complex to set up. (5) Smaller packages: Use Lambda Layers for shared deps, reduce JAR size.
What happens to an SQS message if a Lambda function processing it throws an exception? ▼
If your Lambda function throws an exception when processing an SQS message, the message is NOT deleted from SQS. After the function invocation completes (with failure), the messages become visible in SQS again after the visibility timeout expires (e.g., 30 seconds). SQS then redelivers those messages to Lambda for retry. This continues until either: the function succeeds, or the message's maxReceiveCount is reached (configured on the SQS queue's redrive policy). When maxReceiveCount is reached, the message moves to the Dead Letter Queue (DLQ). For partial batch failures: use ReportBatchItemFailures response type — your function returns the IDs of individual failed messages. Only those specific messages are requeued; successfully processed messages are deleted. Without this, if one message in a batch fails, the entire batch is requeued and all messages are retried.
What is the difference between Reserved Concurrency and Provisioned Concurrency in Lambda? ▼
Reserved Concurrency: Reserves a fixed number of concurrency units from the account's pool for a specific function. Two effects: (1) GUARANTEES this function always has N concurrency available — even if other functions are consuming the pool. (2) CAPS the function — it can never exceed N concurrent executions. Use to: protect important functions from being throttled by other functions, limit a function's scale (cost control, downstream service protection). Setting to 0 disables the function.

Provisioned Concurrency: Pre-initialises N execution environments — cold start phases 1-4 run ahead of time. When requests arrive, they hit immediately warm environments. Eliminates cold starts entirely for those N slots. Costs money even when idle (you're paying to keep environments warm). Use for: latency-sensitive APIs where cold start time is unacceptable, functions with heavy init code that causes multi-second cold starts.

ECS — Intermediate to Advanced

What is the difference between ECS Task Role and Task Execution Role? ▼
Task Role: The IAM role that your application code running inside the container assumes. This is what your business logic uses to call AWS services. Example: if your app needs to read from DynamoDB and publish to SNS, the task role has DynamoDB:GetItem, DynamoDB:PutItem, and SNS:Publish permissions. The AWS SDK inside your container automatically fetches credentials from the ECS metadata endpoint (similar to EC2 IMDS).

Task Execution Role: The IAM role that the ECS AGENT uses — not your code. The ECS infrastructure needs this to: pull your Docker image from ECR (ecr:GetAuthorizationToken, ecr:BatchGetImage), write container logs to CloudWatch (logs:CreateLogStream, logs:PutLogEvents), inject secrets from Secrets Manager or SSM Parameter Store into container environment variables at startup. Your app code never uses this role.
How does a zero-downtime deployment work in ECS? ▼
ECS rolling deployment with MinHealthyPercent=100, MaxPercent=200: (1) ECS registers a new task definition with the updated image. (2) ECS starts NEW tasks using the new version (up to MaxPercent=200% of desired count, so if desired=6, up to 12 tasks temporarily). (3) New tasks register with the ALB target group. ALB sends health check requests to new tasks. (4) When new tasks pass ALB health checks (responds 200 OK on /health), ALB starts routing real traffic to them. (5) ECS stops old tasks (one by one). ALB deregisters old tasks — waits for deregistration delay (e.g., 30s) for in-flight requests to complete. (6) Process repeats until all old tasks are replaced. The Deployment Circuit Breaker monitors the deployment: if new tasks repeatedly fail health checks, ECS automatically rolls back to the previous task definition. Result: zero downtime because new tasks handle traffic before old tasks are stopped.
When would you choose ECS Fargate over EC2 launch type? When would you choose EC2? ▼
Choose Fargate when: (1) You want zero server management — no EC2 fleet to maintain, patch, or right-size. (2) Variable workloads that scale to zero — Fargate runs 0 tasks when idle, EC2 launch type still pays for idle EC2 instances. (3) Security isolation is critical — each Fargate task runs in a dedicated microVM. (4) You're starting a new project and don't have enough load to make EC2 cost-effective. Choose EC2 launch type when: (1) High-scale production with consistent load — EC2 costs less per compute unit than Fargate at high utilisation. (2) You need Spot Instances for 70-90% savings on non-critical tasks. (3) GPU workloads — Fargate doesn't support GPUs. (4) You need large memory (>120GB) — Fargate max is 120GB. (5) Windows containers requiring specific EC2 instance types. (6) You're already managing an EC2 fleet and just want to containerise workloads on existing capacity.
Describe how you would architect a Java microservice that handles 10,000 requests per minute with 100ms p99 latency requirement. EC2, Lambda, or ECS? ▼
ECS Fargate is the right choice. Reasoning: (1) 10K RPM = ~167 RPS, steady. Lambda cold starts (1–3s for Java) would occasionally violate 100ms p99 — unless using Provisioned Concurrency (expensive). (2) ECS Fargate runs the JVM continuously — no cold starts after initial boot. JVM JIT compiles hot paths for optimal performance. (3) 100ms p99 with Java is very achievable in ECS — no Lambda overhead. Architecture: Fargate tasks (m-type, java21 runtime, ~1 vCPU / 2GB per task), ALB with 100ms idle timeout, target tracking auto scaling on ALB RequestsCountPerTarget (target ~1000 requests per task). 167 RPS ÷ 1000 = ~0.167 tasks needed at minimum, with headroom set min=3 tasks (for HA across 3 AZs). Java specifics: use G1GC with MaxGCPauseMillis=50 to avoid GC pauses affecting p99, configure JVM heap at 80% container memory, enable JVM tiered compilation. Monitor with CloudWatch Container Insights for CPU/memory, ALB TargetResponseTime for latency, set p99 alarm at 80ms to act before SLA breach.
