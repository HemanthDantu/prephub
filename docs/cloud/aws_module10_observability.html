The Three Pillars of Observability

Observability is your ability to understand what's happening inside your system from its external outputs. Without it, you're flying blind in production. Three distinct types of data give you the full picture.

Why Observability is Different from Monitoring

Monitoring is watching known things: "Alert me if CPU goes above 80%." You pre-define what to watch. Observability is the ability to ask unknown questions about your system: "Why is the p99 latency spiking only for users in the EU, only between 2–3pm, only for requests containing JSON payloads over 10KB?" You can only answer that if you've captured rich enough data from the beginning.

Pillar 1 — Metrics (CloudWatch)
Numeric measurements over time. CPU at 65%, 200 requests/sec, 45ms average latency, 3 errors in the last minute. Good for trending, alerting, dashboards, and capacity planning. Cheap to store, fast to query.

Answers: Is something wrong? How bad is it? Is it getting worse?
Pillar 2 — Logs (CloudWatch Logs)
Timestamped records of discrete events. "User alice placed order ord-123 at 14:32:01." Detailed, verbose, expensive to store at scale. Good for debugging specific incidents — you know something went wrong, now you need to know exactly what.

Answers: What exactly happened? What was the context?
Pillar 3 — Traces (X-Ray)
End-to-end journey of a request through your distributed system. Shows: which services were called, in what order, how long each took, where errors occurred. Essential when a request touches 5+ services.

Answers: Where in the flow is the slowness/error? What's the sequence?

The Debugging Journey — How All Three Work Together

Real Incident
API response time spiked to 5 seconds at 14:30 today
1
Metrics detect the problem
CloudWatch alarm fires: API Gateway p99 latency > 2000ms for 5 consecutive minutes. PagerDuty notifies the on-call engineer at 14:35.
2
Traces narrow down where
X-Ray service map shows the OrderService Lambda has 4.8s average duration. The DynamoDB segment inside it shows 4.6s. Not the Lambda, not the API — it's the DynamoDB call.
3
Logs explain why
CloudWatch Logs Insights query on OrderService logs for that time window: "ProvisionedThroughputExceededException: Exceeded write capacity units on table/Orders/index/status-index." A GSI was throttling due to a hot partition key from a flash sale. Root cause found in 12 minutes.

Metrics — What They Are & How They Work

A metric is a time-ordered set of data points that measures something about your system. Understanding namespaces, dimensions, statistics, and periods is essential for reading and creating alarms correctly.

Metric Anatomy

Every CloudWatch metric has four identifying components:

Metric Components
Namespace: A container that groups related metrics. AWS services each have their own namespace. Example: AWS/EC2, AWS/Lambda, AWS/RDS, AWS/SQS. Your custom metrics go in a namespace you define like MyApp/OrderService.

Metric Name: The thing being measured within a namespace. Example: CPUUtilization, Errors, Duration, ApproximateNumberOfMessagesVisible.

Dimensions: Key-value pairs that identify a specific instance of a metric. FunctionName=OrderProcessor distinguishes this Lambda's Duration from every other Lambda's Duration. Without dimensions, all Lambda functions' Duration metrics would be indistinguishable. One metric can have up to 30 dimensions.

Data point: A value + timestamp. CloudWatch aggregates multiple data points into statistics (Average, Sum, Max, Min, p99) over a period (1 min, 5 min, etc.). 
# A metric is identified by: namespace + name + all dimensions together
# These are THREE DIFFERENT metrics (different dimensions):

AWS/Lambda : Duration : FunctionName=OrderProcessor
AWS/Lambda : Duration : FunctionName=UserService  
AWS/Lambda : Duration : FunctionName=PaymentService

# Get metric statistics via CLI
aws cloudwatch get-metric-statistics \
  --namespace AWS/Lambda \
  --metric-name Duration \
  --dimensions Name=FunctionName,Value=OrderProcessor \
  --start-time 2024-01-15T14:00:00Z \
  --end-time 2024-01-15T15:00:00Z \
  --period 60 \                      # aggregate into 60-second buckets
  --statistics Average Maximum p99    # what to compute per bucket
# Returns: 60 data points (one per minute), each with Average/Max/p99

Statistics — What Each Means

StatisticWhat it computesBest for
AverageSum ÷ count of data points in periodCPU utilisation, latency (overall feel)
SumTotal of all data pointsRequest count, error count, bytes transferred
MaximumHighest value in the periodMemory spikes, latency worst case
MinimumLowest value in the periodDetecting downtime (0 requests means nothing ran)
p50/p90/p99Percentile — value below which X% of data points fallLatency SLAs — p99=500ms means 99% of requests <500ms
SampleCountNumber of data points in the periodDetecting when Lambda wasn't called (count=0)
Average latency lies. If 99 requests take 10ms and 1 request takes 10,000ms, the average is 109ms — which looks fine. The p99 is 10,000ms — a disaster. Always alarm on p99 latency for user-facing APIs, never just average. Average hides the long tail that your slowest users experience.

Metric Resolution — Standard vs High Resolution

## Standard resolution: 1-minute granularity
## High resolution: 1-second granularity (custom metrics only, higher cost)

# Standard metrics: data stored at 1-min resolution, available 15 months
# High-res custom metrics: 1-second, available 3 hours at 1-sec, then 1-min

# Most production metrics don't need sub-minute resolution
# Exception: automated trading, real-time gaming, financial systems

## Retention periods:
# 1-second metrics:  3 hours
# 1-minute metrics:  15 days
# 5-minute metrics:  63 days
# 1-hour metrics:    455 days (15 months)
# After retention: data points deleted — you can't look back further

Built-in Metrics vs Custom Metrics

AWS services automatically publish dozens of metrics. But the most useful metrics for understanding your business — order rate, payment success rate, active users — only you can create.

Key Built-in Metrics to Know Per Service

## Lambda — most important
Invocations      ← how many times called (Sum)
Errors           ← how many failed (Sum) — set alarm here
Throttles        ← how many throttled (Sum) — means concurrency limit hit
Duration         ← execution time (Average/p99) — set alarm here
ConcurrentExecutions  ← simultaneous executions (Maximum)
IteratorAge      ← for Kinesis/DynamoDB stream triggers: how far behind is consumer

## API Gateway
Count            ← total request count
4XXError         ← client errors (Sum) — 400/401/403/404
5XXError         ← server errors (Sum) — 500/502/503 — set alarm here
Latency          ← end-to-end request time (p99)
IntegrationLatency  ← time spent in Lambda (p99)

## ECS/Fargate
CPUUtilization   ← container CPU % (Average per service)
MemoryUtilization  ← container RAM % (Average) — set alarm near limit
RunningTaskCount ← number of running tasks (Minimum)

## RDS / Aurora
CPUUtilization   ← database CPU (average)
DatabaseConnections  ← active connections (Maximum) — alert near max
FreeStorageSpace ← remaining disk (Minimum) — alert at 20%
ReadLatency / WriteLatency  ← query latency (p99)
ReplicaLag       ← read replica lag in seconds

## SQS
ApproximateNumberOfMessagesVisible  ← queue depth — primary scaling signal
ApproximateAgeOfOldestMessage      ← time oldest message has waited (alarm = processing stuck)
NumberOfMessagesSent               ← ingestion rate
NumberOfMessagesDeleted            ← processing rate

## DynamoDB
ConsumedReadCapacityUnits / ConsumedWriteCapacityUnits
ThrottledRequests  ← set alarm — means your app is being throttled
SuccessfulRequestLatency  ← query latency

Custom Metrics — Your Business Metrics

Custom metrics capture business-level signals that AWS has no knowledge of: order placement rate, payment success rate, user login failures, checkout funnel drops.

// Publishing custom metrics from Java Lambda
CloudWatchClient cw = CloudWatchClient.create();

// Track order placement rate
cw.putMetricData(PutMetricDataRequest.builder()
    .namespace("MyApp/Orders")
    .metricData(MetricDatum.builder()
        .metricName("OrdersPlaced")
        .value(1.0)
        .unit(StandardUnit.COUNT)
        .timestamp(Instant.now())
        .dimensions(
            Dimension.builder().name("Environment").value("prod").build(),
            Dimension.builder().name("OrderType").value(order.getType()).build()
        )
        .build())
    .build());

// Track payment success/failure
cw.putMetricData(PutMetricDataRequest.builder()
    .namespace("MyApp/Payments")
    .metricData(
        MetricDatum.builder()
            .metricName("PaymentAttempts").value(1.0)
            .unit(StandardUnit.COUNT).build(),
        MetricDatum.builder()
            .metricName(success ? "PaymentSuccess" : "PaymentFailure")
            .value(1.0).unit(StandardUnit.COUNT).build()
    )
    .build());

// Track latency of a specific operation
long start = System.currentTimeMillis();
inventoryService.reserve(order.getItems());
long duration = System.currentTimeMillis() - start;

cw.putMetricData(PutMetricDataRequest.builder()
    .namespace("MyApp/Orders")
    .metricData(MetricDatum.builder()
        .metricName("InventoryReservationDuration")
        .value((double) duration)
        .unit(StandardUnit.MILLISECONDS)
        .build())
    .build());
Cost of custom metrics: $0.30 per metric per month (first 10,000 metrics). Batch PutMetricData calls together — you can send up to 1,000 metric data points per API call. In high-volume Lambdas, don't call PutMetricData on every invocation — buffer and flush every 30 seconds or at function end.

Alarms — Alerting on Metrics

Alarms watch a metric and trigger actions when it crosses a threshold. The difference between a noisy alarm that pages you at 3am for nothing and a useful alarm that catches real problems is in the configuration.

Alarm States

OK
Metric is within the acceptable range. No action needed. This is the normal state.
ALARM
Metric has breached the threshold for the configured number of consecutive periods. Action triggered (SNS notification, Auto Scaling, Lambda).
INSUFFICIENT_DATA
Not enough data to evaluate. Common when a metric hasn't published yet (Lambda not invoked, service just deployed). Configure how to treat this — usually OK.

Creating Production-Grade Alarms

# Alarm 1: API errors too high (5xx error rate > 1%)
aws cloudwatch put-metric-alarm \
  --alarm-name prod-api-5xx-errors \
  --alarm-description "API 5xx error rate above 1% for 2 consecutive minutes" \
  --namespace AWS/ApiGateway \
  --metric-name 5XXError \
  --dimensions Name=ApiName,Value=users-api Name=Stage,Value=prod \
  --statistic Sum \
  --period 60 \                    # evaluate every 60 seconds
  --evaluation-periods 2 \         # must breach for 2 consecutive periods
  --threshold 5 \                  # absolute threshold: > 5 errors per minute
  --comparison-operator GreaterThanThreshold \
  --treat-missing-data notBreaching \  # no data = OK (API just not called)
  --alarm-actions arn:aws:sns:us-east-1:123:pagerduty-alerts \
  --ok-actions arn:aws:sns:us-east-1:123:pagerduty-alerts

# Alarm 2: Lambda p99 latency above SLA
aws cloudwatch put-metric-alarm \
  --alarm-name prod-orders-lambda-latency \
  --namespace AWS/Lambda \
  --metric-name Duration \
  --dimensions Name=FunctionName,Value=OrderProcessor \
  --extended-statistic p99 \       # p99, not average
  --period 300 \                   # 5-minute windows
  --evaluation-periods 3 \         # 3 windows = 15 minutes sustained
  --threshold 2000 \               # 2000ms = 2 seconds p99 SLA
  --comparison-operator GreaterThanThreshold \
  --alarm-actions arn:aws:sns:us-east-1:123:engineering-slack

# Alarm 3: SQS DLQ not empty (alert immediately)
aws cloudwatch put-metric-alarm \
  --alarm-name prod-orders-dlq-has-messages \
  --namespace AWS/SQS \
  --metric-name ApproximateNumberOfMessagesVisible \
  --dimensions Name=QueueName,Value=orders-processing-dlq \
  --statistic Sum \
  --period 60 \
  --evaluation-periods 1 \         # alarm on first occurrence
  --threshold 0 \
  --comparison-operator GreaterThanThreshold \
  --treat-missing-data notBreaching \
  --alarm-actions arn:aws:sns:us-east-1:123:pagerduty-critical

# Alarm 4: RDS storage running low
aws cloudwatch put-metric-alarm \
  --alarm-name prod-rds-low-storage \
  --namespace AWS/RDS \
  --metric-name FreeStorageSpace \
  --dimensions Name=DBInstanceIdentifier,Value=prod-postgres-primary \
  --statistic Average \
  --period 300 \
  --evaluation-periods 2 \
  --threshold 10737418240 \        # 10GB in bytes (10 * 1024^3)
  --comparison-operator LessThanThreshold \
  --alarm-actions arn:aws:sns:us-east-1:123:engineering-slack

Composite Alarms — Combine Multiple Alarms

# Reduce alert noise: only page when BOTH latency is high AND error rate is high
# (latency spike alone might be a batch job, not customer impact)

aws cloudwatch put-composite-alarm \
  --alarm-name prod-api-critical \
  --alarm-rule "ALARM(prod-api-5xx-errors) AND ALARM(prod-api-p99-latency)" \
  --alarm-actions arn:aws:sns:us-east-1:123:pagerduty-critical

# OR: alert if ANY component in a service is alarming
aws cloudwatch put-composite-alarm \
  --alarm-name prod-orders-service-degraded \
  --alarm-rule "ALARM(prod-orders-lambda-errors) OR ALARM(prod-orders-dlq-has-messages) OR ALARM(prod-orders-latency)"

CloudWatch Dashboards

A dashboard puts the most important metrics for a service or system in one view. Good dashboards let you assess system health in seconds, not minutes of navigating through individual metric pages.

What Makes a Good Dashboard

Bad dashboard habits
Showing 40+ metrics at once with no hierarchy. Only showing CPU/memory (infrastructure) without business metrics. No annotations for deployments. Mixing different time scales. No alarm state indicators.
Good dashboard design
Top row: health indicators (RED: Requests, Errors, Duration). Middle: key business metrics (orders/min, payment success rate). Bottom: infrastructure details (CPU, memory, DB connections). Add vertical annotations for deployments. Show alarm states.

Log Groups, Streams & Retention

CloudWatch Logs is where all AWS service logs land. Understanding the hierarchy and retention settings prevents two common problems: paying for logs you'll never read, and discovering you can't look back because logs expired.

The Log Hierarchy

Log Groups → Log Streams → Log Events
/aws/lambda/OrderProcessor ← Log Group (one per function)
  └── 2024/01/15/[$LATEST]abc123def456 ← Log Stream (one per instance/cold start)
      └── 14:30:01.123 START RequestId: req-1 ... ← Log Event
      └── 14:30:01.456 Order processed: ord-123
      └── 14:30:01.789 END RequestId: req-1
      └── 14:30:01.789 REPORT RequestId: req-1 Duration: 330ms ...

Retention — The Setting Everyone Forgets

By default, log groups have no retention policy — logs are kept forever and you pay storage costs indefinitely. Most logs are only useful for 7-90 days. Set retention on every log group.

# Set retention on a Lambda log group (must exist first)
aws logs put-retention-policy \
  --log-group-name /aws/lambda/OrderProcessor \
  --retention-in-days 90          # 1, 3, 5, 7, 14, 30, 60, 90, 120, 150, 180,
                                   # 365, 400, 545, 731, 1827, 2192, 2557, 2922,
                                   # 3288, or 3653

# Apply retention to all Lambda log groups at once (use org-wide automation)
aws logs describe-log-groups --query "logGroups[?starts_with(logGroupName, '/aws/lambda/')].logGroupName" \
  --output text | xargs -I{} aws logs put-retention-policy --log-group-name {} --retention-in-days 90

# Create log group with retention from the start
aws logs create-log-group \
  --log-group-name /myapp/orders-service \
  --kms-key-id alias/prod-logs-key      # encrypt sensitive logs

aws logs put-retention-policy \
  --log-group-name /myapp/orders-service \
  --retention-in-days 30

# Retention by environment:
# Production security/audit logs: 365+ days (compliance)
# Production application logs:    90 days
# Staging/dev logs:               14 days
# Lambda default logs:            30 days

Lambda Log Format — What AWS Automatically Logs

# Every Lambda invocation automatically logs these 4 lines:

# 1. START — when function begins
START RequestId: abc-123-def Version: $LATEST

# 2. Your application logs (System.out.println or your logging framework)
2024-01-15T14:30:01.456Z abc-123-def INFO Processing order ord-456 for user alice

# 3. END — when function returns
END RequestId: abc-123-def

# 4. REPORT — performance summary (always logged, very useful)
REPORT RequestId: abc-123-def  Duration: 330.45 ms  Billed Duration: 331 ms
       Memory Size: 1024 MB   Max Memory Used: 312 MB  Init Duration: 1250.23 ms

# Init Duration appears only on COLD STARTS — tells you cold start cost
# Max Memory Used — use this to right-size memory allocation
# If Max Memory Used consistently < 50% of Memory Size → reduce allocation

Logs Insights — Query Your Logs

CloudWatch Logs Insights is a query engine for your log data. Instead of grepping through files, you write SQL-like queries that scan across all log streams in a group. Essential for production debugging.

The Query Language

Logs Insights uses a pipe-based query language. Each command is separated by |. Commands are applied left-to-right — filter first for performance, then parse, then aggregate.

## Basic query structure:
fields @timestamp, @message           ← select which fields to show
| filter @message like "ERROR"        ← filter rows matching condition
| sort @timestamp desc                ← order results
| limit 50                            ← limit number of results

## Useful built-in fields:
# @timestamp  - event time
# @message    - raw log line
# @logStream  - which log stream (which Lambda instance)
# @requestId  - Lambda request ID (extracted automatically)
# @duration   - Lambda duration (from REPORT lines)
# @billedDuration, @memorySize, @maxMemoryUsed (from REPORT lines)
# @initDuration - cold start time (only on cold starts)

Real-World Queries for Production Debugging

## 1. Find all errors in the last hour
fields @timestamp, @message
| filter @message like /ERROR|EXCEPTION|Exception/
| sort @timestamp desc
| limit 100

## 2. Find slow Lambda invocations (cold starts over 2 seconds)
filter @type = "REPORT"
| parse @message "Init Duration: * ms" as initDuration
| filter ispresent(initDuration)
| filter initDuration > 2000
| sort initDuration desc
| stats count(), avg(initDuration), max(initDuration)

## 3. Lambda performance summary — p99 latency and error rate
filter @type = "REPORT"
| parse @message "Duration: * ms" as duration
| stats count() as invocations,
        avg(duration) as avgMs,
        pct(duration, 99) as p99Ms,
        max(duration) as maxMs
        by bin(5min)

## 4. Find which requestIds had errors (then look up those in traces)
fields @requestId, @timestamp
| filter @message like "EXCEPTION"
| sort @timestamp desc
| limit 20

## 5. Parse structured JSON logs — extract fields from JSON body
# If your Lambda logs JSON: {"level":"ERROR","orderId":"ord-123","msg":"DB timeout"}
fields @timestamp
| parse @message '{"level":"*","orderId":"*","msg":"*"' as level, orderId, msg
| filter level = "ERROR"
| stats count() by orderId
| sort count desc

## 6. Memory right-sizing — find over-provisioned Lambdas
filter @type = "REPORT"
| parse @message "Memory Size: * MB	Max Memory Used: * MB" as allocated, used
| stats max(used) as peakUsed, avg(allocated) as alloc
     by @log
| sort peakUsed asc
| filter alloc / peakUsed > 2   # allocated more than 2x what was actually used

## 7. 5xx error rate from API Gateway logs
fields @timestamp, status, ip, requestId
| filter status >= 500
| stats count() by bin(1min)
| sort @timestamp desc

## 8. Find slow database queries in application logs
# Assumes your app logs: "Query executed in Xms: SELECT * FROM..."
fields @timestamp, @message
| parse @message "Query executed in *ms: *" as queryMs, queryText
| filter queryMs > 500
| sort queryMs desc
| limit 20

Querying Multiple Log Groups at Once

# Query all Lambda functions at once (useful for distributed debugging)
# Use log group prefix pattern in the query

# In console: select "Query multiple log groups" and pick a pattern
# Pattern: /aws/lambda/* matches all Lambda log groups

# CLI with multiple groups:
aws logs start-query \
  --log-group-names /aws/lambda/OrderProcessor /aws/lambda/PaymentProcessor /aws/lambda/UserService \
  --start-time $(date -d '1 hour ago' +%s) \
  --end-time $(date +%s) \
  --query-string 'fields @timestamp, @logStream, @message
| filter @message like "ERROR"
| sort @timestamp desc
| limit 50'

Metric Filters — Logs to Metrics

Metric filters watch your log streams in real time, extract values matching a pattern, and increment a CloudWatch metric. Turns log data into something you can alarm on without changing application code.

What Metric Filters Do

Your Lambda logs "ERROR: PaymentFailed for order ord-123." You want to alarm when payment failures exceed 5 per minute. Without metric filters, you'd need to emit a custom metric in code. With metric filters, you create a pattern that matches "ERROR: PaymentFailed" in your logs and increments a counter metric every time it matches — no code change.

# Create metric filter: count ERROR lines in Lambda logs
aws logs put-metric-filter \
  --log-group-name /aws/lambda/PaymentProcessor \
  --filter-name PaymentFailures \
  --filter-pattern '"ERROR: PaymentFailed"' \
  --metric-transformations \
    metricName=PaymentFailures,\
    metricNamespace=MyApp/Payments,\
    metricValue=1,\
    defaultValue=0     # 0 when no match in period — prevents INSUFFICIENT_DATA

# Now alarm on that metric
aws cloudwatch put-metric-alarm \
  --alarm-name prod-payment-failures \
  --namespace MyApp/Payments \
  --metric-name PaymentFailures \
  --statistic Sum \
  --period 60 \
  --evaluation-periods 2 \
  --threshold 5 \
  --comparison-operator GreaterThanThreshold \
  --alarm-actions arn:aws:sns:us-east-1:123:payment-alerts

# Extract a numeric value from log line
# Log: "Order processed in 4532ms for user alice"
aws logs put-metric-filter \
  --log-group-name /aws/lambda/OrderProcessor \
  --filter-name OrderProcessingTime \
  --filter-pattern '[prefix="Order processed in", time, unit="ms"...]' \
  --metric-transformations \
    metricName=OrderProcessingTime,\
    metricNamespace=MyApp/Orders,\
    metricValue='$time'   # extract the numeric value from pattern

# Filter patterns:
# Simple text: "ERROR"
# Quoted phrase: '"payment failed"'
# JSON field: '{$.level = "ERROR"}'
# JSON numeric: '{$.duration > 1000}'
# Multiple terms: '"ORDER" "FAILED"'
# Exclude: '?"NOT THIS"'

Structured Logging Best Practices

Plain-text logs are hard to query. JSON structured logs are easily parsed by Logs Insights, can be filtered by any field, and work with log aggregation tools. This is the modern approach.

Unstructured vs Structured

Unstructured — hard to query
2024-01-15 14:30:01 ERROR Order ord-123
failed for user alice. Payment
declined. Retry 2/3.

# To find all payment failures:
grep "Payment declined"
# To find failure rate: grep + wc -l
# To find slowest orders: manually parse
Structured JSON — queryable by any field

// logback.xml — output JSON to stdout
<configuration>
  <appender name="STDOUT" class="ch.qos.logback.core.ConsoleAppender">

  <root level="INFO"><appender-ref ref="STDOUT"/></root>
</configuration>
public class OrderHandler {
    private static final Logger log = LoggerFactory.getLogger(OrderHandler.class);

    public void handleRequest(SQSEvent event, Context ctx) {
        // Set correlation IDs in MDC (Mapped Diagnostic Context)
        // These appear in every log line for this invocation
        MDC.put("requestId", ctx.getAwsRequestId());

        for (var msg : event.getRecords()) {
            Order order = parseOrder(msg.getBody());
            MDC.put("orderId", order.getId());
            MDC.put("userId", order.getUserId());

            long start = System.currentTimeMillis();
            try {
                orderService.process(order);

                // Structured log event — all context from MDC + explicit fields
                log.info("order_processed",
                    kv("event", "order_processed"),
                    kv("durationMs", System.currentTimeMillis() - start),
                    kv("totalAmount", order.getTotal())
                );
                // Produces: {"level":"INFO","event":"order_processed","orderId":"ord-123",
                //            "userId":"alice","durationMs":234,"totalAmount":99.99,...}

            } catch (Exception e) {
                log.error("order_processing_failed",
                    kv("event", "order_processing_failed"),
                    kv("errorType", e.getClass().getSimpleName()),
                    kv("errorMessage", e.getMessage()),
                    e
                );
            } finally {
                MDC.clear();
            }
        }
    }
}

Why Tracing? The Distributed Request Problem

When a single user request touches 5 services, metrics tell you something is slow and logs tell you what happened in each service — but neither tells you how those services connected. Tracing fills this gap.

The Problem Tracing Solves

A user reports that checkout is slow. Your metrics show Lambda p99 is elevated but don't tell you which Lambda in the chain. You grep through logs across 5 different Lambda log groups — but which log events are from the same user request? How do you correlate events in OrderService with events in InventoryService with events in PaymentService?

Without Tracing — The Debugging Nightmare
Finding a slow request across 5 services

API Gateway received request ID req-abc. Passed to OrderLambda as a new invocation with request ID req-xyz (different ID). OrderLambda called InventoryLambda — which got invocation ID req-123. Three different log groups. Three different request IDs. No way to correlate them without manually instrumenting correlation IDs across every service call.

With X-Ray: one trace ID propagated through the entire call chain automatically. One view showing all five services, exact timing of each hop, where 4.8 seconds of the 5-second request was spent.

X-Ray Concepts — Traces, Segments & Subsegments

Understanding X-Ray's data model is essential before you instrument your code. A small number of concepts cover everything you'll encounter.

The Data Model

Trace
One end-to-end request flow from the first service to the last. Has a globally unique Trace ID that's propagated in HTTP headers (X-Amzn-Trace-Id) through every service call. All segments with the same Trace ID are assembled into the trace waterfall.
Segment
One service's contribution to a trace. Lambda creates one segment per invocation. Contains: service name, timing (start/end), HTTP data, metadata, annotations, and any subsegments.
Subsegment
A named block of work within a segment — a DynamoDB call, an HTTP call to another service, a database query. Created by the X-Ray SDK automatically for AWS SDK calls and HTTP clients, or manually in your code.

Annotations vs Metadata

Annotations — Indexed, Searchable
Key-value pairs (string, number, boolean) that X-Ray indexes. You can search/filter traces by annotation values. Use for: things you'll want to search by (userId, orderId, environment, version).

Max 50 annotations per trace. Keys must be alphanumeric.
Metadata — Not Indexed, Just Context
Arbitrary JSON objects. Not searchable but visible in the trace details. Use for: large payloads, debug context you want to inspect manually but don't need to search on (request body, response payload, configuration values).

The Trace Header

# X-Ray propagates the trace context via HTTP header:
# X-Amzn-Trace-Id: Root=1-5759e988-bd862e3fe1be46a994272793;Parent=53995c3f42cd8ad8;Sampled=1

# Root=1-{epoch-hex}-{16-hex-chars}  ← unique trace ID
# Parent={16-hex-chars}              ← parent segment ID (so X-Ray knows the tree structure)
# Sampled=1                          ← this request is sampled (being traced)
# Sampled=0                          ← not sampled (no trace data sent)

# When your Lambda calls another service (DynamoDB SDK, HTTP to another Lambda):
# SDK automatically includes this header → X-Ray connects the segments

Sampling — Not Every Request Gets Traced

## At high traffic, tracing every request would be:
## - Expensive ($5 per million traces recorded)
## - High overhead in your Lambda (some CPU for trace data)
## Sampling: record a representative subset

## Default rule: 1 request per second + 5% of additional requests
## Custom rules:
{
  "version": 2,
  "default": {
    "fixed_target": 1,      // always trace this many per second
    "rate": 0.05            // trace 5% of remaining traffic

    {

Adding X-Ray SDK to Java Lambda

<!-- Maven dependency -->
<dependency>
    <groupId>com.amazonaws</groupId>
    <artifactId>aws-xray-recorder-sdk-core</artifactId>
    <version>2.14.0</version>
</dependency>
<dependency>
    <groupId>com.amazonaws</groupId>
    <artifactId>aws-xray-recorder-sdk-aws-sdk-v2-instrumentor</artifactId>
    <version>2.14.0</version>
</dependency>
public class OrderHandler implements RequestHandler<SQSEvent, Void> {

    // SDK clients created with X-Ray instrumentation
    // aws-xray-recorder-sdk-aws-sdk-v2-instrumentor patches SDK v2 automatically
    // Every DynamoDB, SQS, SNS, S3, etc call creates a subsegment automatically
    private static final DynamoDbClient ddb = DynamoDbClient.create();
    private static final SnsClient sns = SnsClient.create();

    public Void handleRequest(SQSEvent event, Context ctx) {

        // Add annotations — searchable in X-Ray
        AWSXRay.getCurrentSegment().putAnnotation("environment", "production");
        AWSXRay.getCurrentSegment().putAnnotation("functionVersion", "v1.2.3");

        for (var msg : event.getRecords()) {
            Order order = parseOrder(msg.getBody());

            // Add per-order annotation
            AWSXRay.getCurrentSegment().putAnnotation("orderId", order.getId());
            AWSXRay.getCurrentSegment().putAnnotation("userId", order.getUserId());

            // Manual subsegment for a custom operation (external HTTP call)
            AWSXRay.beginSubsegment("stripe-payment-charge");
            try {
                PaymentResult result = stripeClient.charge(order.getTotal());
                AWSXRay.getCurrentSubsegment().putMetadata("paymentId", result.getId());
            } catch (Exception e) {
                AWSXRay.getCurrentSubsegment().addException(e);
                throw e;
            } finally {
                AWSXRay.endSubsegment();
            }

            // DynamoDB call — automatically creates subsegment "DynamoDB"
            // Subsegment includes: TableName, Operation, request/response data
            ddb.putItem(PutItemRequest.builder()
                .tableName("Orders")
                .item(toAttributeMap(order))
                .build());
        }

        return null;
    }
}

Interview Questions — Observability

12 questions covering CloudWatch metrics, logs, and X-Ray tracing.

Beginner

What are the three pillars of observability and what does each tell you? ▼
Metrics: Numeric measurements over time. CPU at 65%, 200 requests/second, p99 latency of 450ms. Tell you what is happening and how bad it is. Good for alerting thresholds and capacity planning. Cheap to store, fast to query. Answer: "Is something wrong?"

Logs: Timestamped records of discrete events. "Order ord-123 failed with DuplicateKeyException at 14:30:01." Tell you what exactly happened with full context. More expensive than metrics, more useful for debugging a specific incident. Answer: "What happened and why?"

Traces: End-to-end journey of a single request through multiple services. Shows each service in the call chain, how long it took, where errors occurred. Unique to distributed systems — you can't get this from metrics or logs alone. Answer: "Where in the flow is the problem?"

In practice: metrics detect an incident, traces narrow down which service/dependency is responsible, logs explain the specific error details.
What is the difference between Average and p99 latency? Why does p99 matter more for user-facing APIs? ▼
Average latency is the sum of all response times divided by the count. It's easily skewed by a small number of fast requests hiding a long tail. p99 latency is the 99th percentile — the value that 99% of requests completed faster than. If p99 is 2 seconds, 1% of your users wait more than 2 seconds per request. Example: if 99 requests take 20ms and 1 takes 10,000ms, average = 119ms (looks fine). p99 = 10,000ms (disaster for that user). For user-facing APIs, alarm on p99 because: (1) At 1,000 requests/second, 1% = 10 users/second experiencing the slow path. (2) SLAs are usually defined as percentile guarantees ("p99 < 500ms"), not average guarantees. (3) Average can improve (more fast requests) while p99 stays bad — you'd miss the problem. Always set latency alarms on p99. Use Average only for internal capacity planning.
What does CloudWatch Logs Insights do and how is it different from just reading logs? ▼
CloudWatch Logs Insights is a query engine that scans your log data and returns aggregated results. Difference from reading logs: (1) Scale: A busy Lambda might generate millions of log lines per day across thousands of log streams. You can't grep through all of them. Logs Insights scans them all in seconds. (2) Aggregation: "How many PaymentFailed errors occurred per minute in the last hour? Which userId had the most errors?" These are aggregation queries that are impossible with grep. (3) Structured query: If you log JSON, Insights automatically parses the JSON fields. You can write filter event = "payment_failed" | stats count() by errorCode. (4) Cross-stream: A Lambda function might have 500 concurrent log streams (one per warm instance). Logs Insights queries all of them together. Cost: $0.005 per GB of log data scanned — so write targeted queries with good filters.

Intermediate

What is X-Ray and why can't you solve the same problem with just logs? ▼
X-Ray is a distributed tracing service that follows a single request through your entire microservices architecture, recording timing and error information for each service hop. Why logs can't solve this: (1) Correlation: When API Gateway invokes Lambda, Lambda invokes DynamoDB, each generates its own log events with different IDs. There's no common identifier linking them unless you manually propagate one. X-Ray auto-propagates a trace ID via HTTP headers. (2) Causality: Logs show what happened in each service independently. Traces show the causal chain — Service A called Service B which called Service C, and Service C was slow, causing Service B to be slow, causing the user to see a slow response. You can't reconstruct this from independent logs. (3) Timing: Logs have timestamps but no way to know if 50ms in Service A and 50ms in Service B were concurrent or sequential. Traces show the exact temporal relationship. (4) Service Map: X-Ray automatically generates a graph of how your services connect, with error rates on each edge — impossible to derive from logs.
What is a metric filter in CloudWatch? When would you use it? ▼
A metric filter watches a CloudWatch Logs log group in real time, matches log lines against a pattern, and increments a CloudWatch metric each time the pattern matches. Use cases: (1) No code change needed: Your application logs "ERROR: payment_failed" and you want to alarm when payment failures spike. Instead of adding custom metric code to your Lambda, create a metric filter with pattern "ERROR: payment_failed" → creates a metric that increments each match → create a CloudWatch alarm on that metric. (2) Extract numeric values: If your app logs "Order processed in 4532ms" you can extract 4532 as the metric value. Now you have a latency metric without changing application code. (3) Legacy applications: Applications you can't easily modify. (4) Limitation: metric filters have slight delay (seconds) vs real-time metrics from the app. They're also harder to maintain — pattern changes require careful testing. For new code, emitting explicit custom metrics from the application is cleaner and more flexible.
A Lambda function is showing elevated p99 latency. Walk through how you'd debug this. ▼
(1) CloudWatch Metrics: Check Duration p99 trend — is it a sudden spike or gradual increase? Check Errors and Throttles — eliminate those as the cause. Check ConcurrentExecutions — are you hitting the concurrency limit? (2) Lambda REPORT lines (Logs Insights): filter @type = "REPORT" | parse @message "Duration: * ms\tBilled Duration" as dur | stats pct(dur, 99) by bin(5min) — see exactly when latency spiked. Also check: filter @type = "REPORT" | parse @message "Init Duration: * ms" as init | filter ispresent(init) | stats count(), avg(init) — are cold starts driving the p99? (3) X-Ray traces: Open X-Ray console, find the Lambda in the Service Map (may appear yellow/red). Filter traces from the high-latency period. Sort by Duration. Click the slowest trace. Look for the red/slow subsegment — is it DynamoDB? An external HTTP call? The handler itself? (4) Application logs (Logs Insights): For the requestId of a slow trace, find all log lines. Look for: long-running loops, large payloads being parsed, retried calls, lock contention. (5) Check if a deployment happened around the same time (check dashboard annotation or CloudTrail for recent UpdateFunctionCode/UpdateFunctionConfiguration).
What is X-Ray sampling and why doesn't it trace every request by default? ▼
X-Ray sampling means only a fraction of requests are traced. The default rule: always trace 1 request per second, then trace 5% of additional requests. Reasons not to trace everything: (1) Cost: X-Ray charges $5 per million traces recorded and $0.50 per million traces scanned. At 1,000 requests/second, 100% sampling = 86.4 million traces/day = $432/day in recording alone. With 5% sampling = ~4.3 million/day = ~$21/day. (2) Performance overhead: The X-Ray SDK adds slight overhead to each traced request (serialising trace data, sending to daemon). At high throughput, this adds up. (3) Storage: Traces are retained for 30 days (cannot be extended). Tracing everything would hit storage limits. The 5% sample is statistically sufficient to identify systemic issues — if DynamoDB is throttling 50% of requests, it will show up clearly in a 5% sample. For debugging specific requests (like a customer complaint), you can tag those requests with a specific annotation and create a custom sampling rule to trace 100% of requests with that annotation.

Advanced

Design the observability strategy for a new microservices platform (10 services, Lambda + ECS, API Gateway frontend, RDS + DynamoDB). ▼
Strategy across all three pillars: Metrics + Alarms: Per Lambda: Error rate (Sum, alarm >0), Duration p99 (alarm at SLA), Throttles (alarm >0), cold start rate. Per ECS service: CPU/Memory utilisation (alarm at 80%), RunningTaskCount vs desired (alarm if < desired), ALB 5XX rate and latency. Per DynamoDB table: ThrottledRequests (alarm >0), ConsumedCapacity vs provisioned. Per RDS: CPU, DatabaseConnections (alarm at 80% max), FreeStorageSpace, ReadLatency/WriteLatency p99. All services: custom business metrics (order rate, payment success rate). Composite alarms for "service degraded" that combine component alarms. All alarms → SNS → PagerDuty (critical) or Slack (warning). Logs: All 10 services log structured JSON to CloudWatch Logs. Common fields across all services: requestId (X-Ray trace ID), userId, serviceName, environment, version. Retention: 90 days prod, 14 days dev. Log group per service. Metric filters for error patterns → metrics. Logs Insights saved queries for common debugging patterns. Traces: X-Ray Active on all Lambdas. X-Ray daemon sidecar on all ECS tasks. Common annotations: environment, version, userId, entityId (orderId, productId, etc.). Custom subsegments for all external calls (Stripe, email service). Sampling: 5% default + 100% error sampling. Service Map shows complete request flow. Dashboards: One dashboard per service (RED metrics + business metrics). One org-wide dashboard (all services health at a glance). Deployment annotations in CI/CD pipeline.
What is structured logging and why is it better than plain text logs? ▼
Structured logging outputs log events as machine-readable JSON (or another structured format) instead of free-form text strings. Plain text: ERROR: Order ord-123 failed for user alice after 1250ms. Structured JSON: {"level":"ERROR","event":"order_failed","orderId":"ord-123","userId":"alice","durationMs":1250,"timestamp":"..."}. Benefits: (1) Queryable by any field: Logs Insights can parse JSON automatically. filter event = "order_failed" | stats count() by userId — instantly find which users are experiencing failures. With plain text you'd need complex regex that breaks when the message format changes. (2) Consistent structure: Every developer on the team logs the same fields the same way. No more "you log userId as 'user', I log it as 'userId', she uses 'uid'." (3) Metric filters: Create metric filters on exact JSON field values. (4) Correlation: Common fields like requestId, userId propagated via MDC appear in every log line — trivially find all logs for a specific request or user. (5) Tooling integration: Structured logs integrate with log aggregation tools (Datadog, Splunk, OpenSearch) that auto-index JSON fields. Implementation: add logstash-logback-encoder to Java, use SLF4J structured logging with key-value pairs, set common MDC context (requestId, userId) at the start of each handler.
