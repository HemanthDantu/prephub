Start Here — GET vs POST for Excel Download

From your notes: "In the UI there is a button — clicking it should show an Excel sheet. Why GET, why not POST?" This question tests REST principles, security awareness, and practical HTTP knowledge.

🤔 What Does the Question Actually Test?

The interviewer is not just asking about HTTP methods. They are testing whether you understand:

  • HTTP semantics — what GET and POST are designed for
  • Security implications — what goes in URL vs request body
  • Practical constraints — URL length limits, caching, bookmarking
  • Real-world patterns — how large exports are handled in production

There is no single correct answer — the right choice depends on the size and sensitivity of your filter parameters. Showing you understand the trade-offs is what impresses interviewers.

⚙️ Three Real-World Export Patterns

Pattern 1 — Synchronous GET (simple reports, under 10 seconds): User clicks button → GET /api/export?fromDate=2024-01-01&toDate=2024-01-31 → backend builds and streams Excel → browser downloads immediately. Works well for small-to-medium exports. User sees a spinner. The browser shows a download dialog when bytes arrive.

Pattern 2 — Asynchronous POST (large reports): User clicks button → POST /api/export with filter body → backend returns {"jobId": "abc-123"} immediately (202 Accepted) → frontend polls GET /api/export/status/abc-123 every 2 seconds → when status=READY → download the file from GET /api/export/download/abc-123. User sees a progress indicator. This is how Salesforce, JIRA, and most enterprise apps handle large exports.

Pattern 3 — Streaming POST (very large, real-time): POST starts, backend streams Excel bytes using chunked transfer encoding as rows are produced. No temp file needed. Browser receives the download progressively. Best for very large datasets where you cannot wait for the full file to be built before sending bytes.

// Pattern 1: Synchronous GET — streams directly to response @GetMapping("/api/orders/export") public void exportGet( @RequestParam String fromDate, @RequestParam String toDate, HttpServletResponse response) throws IOException { response.setContentType( "application/vnd.openxmlformats-officedocument.spreadsheetml.sheet"); response.setHeader(HttpHeaders.CONTENT_DISPOSITION, "attachment; filename=orders-export.xlsx"); // Stream directly to response — no byte[] in memory excelExportService.streamOrders(fromDate, toDate, response.getOutputStream()); } // Pattern 2: Async POST — returns job ID immediately @PostMapping("/api/orders/export") public ResponseEntity<Map<String, String>> requestExport( @RequestBody ExportFilterRequest filter) { String jobId = exportJobService.scheduleExport(filter); // async return ResponseEntity.accepted() .body(Map.of( "jobId", jobId, "statusUrl", "/api/orders/export/status/" + jobId, "downloadUrl", "/api/orders/export/download/" + jobId )); } // Frontend: trigger GET download without page navigation const downloadExcel = () => { const params = new URLSearchParams({fromDate, toDate}); const a = document.createElement('a'); a.href = '/api/orders/export?' + params.toString(); a.download = 'orders.xlsx'; document.body.appendChild(a); a.click(); document.body.removeChild(a); };

DB Fetch Design for Excel Export

The data is in the DB. How do you get it out for 50,000 rows without running out of memory?

The Memory Bomb — The Most Common Mistake

The most dangerous line of code in a data export:

List<Order> orders = orderRepository.findAll();

On a table with 100,000 rows, this loads every row, every column, every association into Java heap memory simultaneously. At approximately 2KB per JPA entity (with associations), that is 200MB. With 10 concurrent export requests, you hit 2GB and your pod OOMKills. K8s restarts the pod. Users lose their downloads. The export button appears broken.

The fix is streaming — process one row at a time, writing it directly to the Excel output stream. Memory stays flat at roughly the size of one batch regardless of dataset size.

❌ findAll() into List — Memory Bomb
Row 1 loaded into heap
Row 1000 loaded
Row 10,000 loaded
Row 50,000 loaded — OOMKill!
All rows in memory simultaneously
✅ JPA Stream — Flat Memory
Processing row 1
Processing row 1000
Processing row 10,000
Processing row 50,000
Only one batch in memory at any time
⚙️ Three Correct Approaches — Compared

Approach 1 — JPA Streaming with @QueryHints: Spring Data JPA returns a Stream<Entity> backed by a database cursor. JDBC fetches rows in small batches (e.g. 500 at a time) rather than all at once. You process each row and discard it before the next batch arrives. Memory stays constant. Must be inside @Transactional(readOnly=true) — the JPA session must stay open for the cursor to remain open.

Approach 2 — Paginated processing: Fetch 1,000 rows at a time using Pageable. Process each page, write to Excel, clear from memory. Simple to implement. Slightly less efficient than streaming (N database round trips). Avoid OFFSET pagination at large page numbers — use keyset instead.

Approach 3 — Native JDBC ResultSet streaming: Use JdbcTemplate with ResultSetExtractor or Spring's RowCallbackHandler. On MySQL, set fetchSize = Integer.MIN_VALUE to enable streaming mode. Maximum control, maximum efficiency, but more verbose code.

// ── Approach 1: JPA Streaming ──────────────────────── @Repository public interface OrderRepository extends JpaRepository<Order, Long> { // fetchSize hint: JDBC fetches 500 rows at a time from cursor @QueryHints(@QueryHint(name = HINT_FETCH_SIZE, value = "500")) @Transactional(readOnly = true) Stream<Order> streamByCreatedAtBetween(LocalDate from, LocalDate to); } @Service public class OrderExportService { @Transactional(readOnly = true) // MUST be transactional — cursor needs open session public void exportToExcel(LocalDate from, LocalDate to, OutputStream out) throws IOException { // SXSSFWorkbook keeps only 100 rows in memory — rest flushed to disk try (SXSSFWorkbook wb = new SXSSFWorkbook(100); Stream<Order> stream = orderRepo.streamByCreatedAtBetween(from, to)) { Sheet sheet = wb.createSheet("Orders"); AtomicInteger rowNum = new AtomicInteger(0); stream.forEach(order -> { Row row = sheet.createRow(rowNum.getAndIncrement()); row.createCell(0).setCellValue(order.getId().toString()); row.createCell(1).setCellValue(order.getCustomerName()); row.createCell(2).setCellValue(order.getTotal().doubleValue()); row.createCell(3).setCellValue(order.getStatus().name()); // row is written and no longer referenced — GC collects it }); wb.write(out); } } } // ── Approach 2: Keyset Paginated Processing ────────── @Transactional(readOnly = true) public void exportPaginated(OutputStream out) throws IOException { try (SXSSFWorkbook wb = new SXSSFWorkbook(500)) { Sheet sheet = wb.createSheet("Orders"); int rowNum = 0; Long lastId = 0L; while (true) { // Keyset pagination: always fast, no OFFSET scan List<Order> batch = orderRepo.findNextBatch(lastId, 1000); if (batch.isEmpty()) break; for (Order o : batch) { Row row = sheet.createRow(rowNum++); row.createCell(0).setCellValue(o.getId().toString()); // ... more cells ... } lastId = batch.get(batch.size() - 1).getId(); // remember last ID } wb.write(out); } }

ORM Choice and Query Optimization

The N+1 problem, how to fix it, when to abandon JPA, and the full query optimization toolkit.

🤔 The N+1 Problem — The Most Common JPA Performance Bug

You load a list of 100 orders. For each order, your code accesses order.getCustomer().getName(). This is a lazy-loaded association — JPA does not load the Customer until you access it. So for 100 orders, JPA issues 100 separate SELECT queries for the customer. Plus the 1 initial query to load orders. That is 101 queries instead of 1.

At 100 orders and 5ms per query, this takes 500ms just for database calls. At 1,000 orders, it takes 5 seconds. This is called the N+1 problem — you intended 1 query but got N+1.

Real Production Story — Page That Went From 200ms to 45 Seconds

A team built an order listing page. In development with 50 orders it loaded in 200ms. In production with 5,000 orders the page took 45 seconds. Root cause: their template accessed order.getCustomer().getName() for every order in the list. 5,001 queries × 9ms average = 45 seconds.

The fix: add one JOIN FETCH to the repository query. New query time: 280ms. The only change was one line in the repository interface.

// ── The N+1 Problem ────────────────────────────────── List<Order> orders = orderRepo.findAll(); // 1 SELECT query orders.forEach(o -> { System.out.println(o.getCustomer().getName()); // N SELECT queries! // Each getCustomer() triggers: SELECT * FROM customers WHERE id=? }); // Total: 1 + N queries. For 1000 orders: 1001 queries. BAD. // ── Fix 1: JOIN FETCH in JPQL ──────────────────────── @Query("SELECT o FROM Order o JOIN FETCH o.customer WHERE o.status = :status") List<Order> findWithCustomer(@Param("status") OrderStatus status); // Generates: SELECT o.*, c.* FROM orders o INNER JOIN customers c ON o.customer_id = c.id // Total: 1 query with a JOIN. Always. // ── Fix 2: @EntityGraph (declarative, no JPQL needed) ─ @EntityGraph(attributePaths = {"customer", "items", "items.product"}) List<Order> findByStatus(OrderStatus status); // Fetches order + customer + items + each item's product in ONE query // ── Fix 3: DTO Projection — only fetch what you need ── // Instead of loading full Order entities with all associations, // fetch only the specific columns you actually display @Query("SELECT new com.example.dto.OrderSummary(o.id, o.total, o.status, c.name) " + "FROM Order o JOIN o.customer c WHERE o.createdAt > :since") List<OrderSummary> findSummariesSince(@Param("since") LocalDate since); // Selects 4 columns instead of loading 20-column entity + associations // OrderSummary is a plain POJO — not managed by Hibernate, no dirty checking // ── Bulk Update — never load-and-save 50k entities ──── // BAD: loads 50k entities, dirty checks all, generates 50k individual UPDATEs orderRepo.findByStatus(PENDING).forEach(o -> { o.setStatus(ARCHIVED); orderRepo.save(o); }); // GOOD: one SQL UPDATE for all 50k rows @Modifying @Query("UPDATE Order o SET o.status = 'ARCHIVED' WHERE o.createdAt < :cutoff") int archiveOldOrders(@Param("cutoff") LocalDate cutoff); // ── When to abandon JPA entirely ───────────────────── // Use native SQL for: complex aggregations, window functions, CTEs, // full-text search, bulk operations, anything JPA makes harder than SQL @Query( value = "SELECT DATE_TRUNC('month', created_at) AS month, " + "SUM(total) AS revenue, COUNT(*) AS order_count " + "FROM orders GROUP BY 1 ORDER BY 1", nativeQuery = true ) List<MonthlyRevenue> getMonthlyRevenue();

Scaling from 10k to 50k Records

Performance degrades going from 10k to 50k records. Here is the systematic approach — always diagnose first, then fix the actual bottleneck.

🤔 The First Rule — Profile Before You Optimise

Most developers jump to "we need Redis caching" or "we need more servers" when performance degrades. This is usually wrong. At 50k records the bottleneck is almost always one of: a missing database index, loading too much data at once, N+1 queries, or connection pool exhaustion. These are all free to fix.

Before adding any infrastructure, run EXPLAIN ANALYZE on your slow queries. It will tell you exactly what the database is doing and where the time is going.

🔍 How to Read EXPLAIN ANALYZE

Run EXPLAIN ANALYZE your_query_here in your database console. Look for these two patterns:

Seq Scan (bad): The database is reading every single row in the table and checking each one against your WHERE condition. At 50,000 rows this means 50,000 comparisons. At 5,000,000 rows this means 5,000,000 comparisons. Performance degrades linearly with table size.

Index Scan (good): The database jumps directly to the relevant rows using the index — like looking up a word in a book's index rather than reading the whole book. Performance is O(log n) — almost constant regardless of table size.

If you see "Seq Scan" on a large table — add an index on the columns in your WHERE clause. This single change can turn a 3-second query into a 2ms query.

Step 1
Add Missing Indexes — Free 10x–1000x Speedup
Index every column used in WHERE, ORDER BY, and JOIN conditions. A composite index on (user_id, status) is faster than two separate indexes when your query filters on both. Use EXPLAIN ANALYZE to confirm the index is being used.
Step 2
Fix N+1 Queries — Often Bigger Impact Than Indexes
Profile your requests. If an endpoint makes 50+ database queries, it has N+1. Add JOIN FETCH or @EntityGraph. This often reduces query count from 500 to 1 — a 500x reduction in database round trips.
Step 3
Switch to Keyset Pagination — For Large Tables
OFFSET pagination scans and discards all preceding rows. At page 500 with 100 rows per page, the database scans 50,000 rows to find your 100. Keyset pagination always uses an index — equally fast on page 1 or page 5,000.
Step 4
Tune Connection Pool — Prevent Thread Starvation
If your API times out under load but individual queries are fast, check HikariCP metrics. "connection timeout" errors mean threads are waiting for a DB connection — increase maximum-pool-size. But do not exceed your DB's max_connections limit.
Step 5
Add Redis Caching — For Repeated Hot Reads
Cache data that is read frequently but updated rarely: product catalog, user roles, configuration. Cache misses still hit the DB, but cache hits return in under 1ms. Only add caching after you have fixed queries — caching a slow query just caches slowness.
Step 6
Read Replicas — Route Heavy Reads Away from Primary
If the primary DB CPU is the bottleneck after all optimisations, add a read replica. Route reporting, export, and search queries to the replica. Writes go to primary. Use Spring's routing DataSource to do this transparently based on @Transactional(readOnly=true).
// ── Keyset pagination — always fast ────────────────── // BAD: OFFSET scans preceding rows — slows at large offsets @Query("SELECT o FROM Order o ORDER BY o.id LIMIT 100 OFFSET :offset") List<Order> findPage(@Param("offset") int offset); // At offset 49,000 → scans 49,000 rows to skip them. Slow! // GOOD: keyset — jumps straight to index entry for lastId @Query("SELECT o FROM Order o WHERE o.id > :lastId ORDER BY o.id LIMIT 100") List<Order> findNextPage(@Param("lastId") Long lastId); // Always O(log n) — equally fast at any position in the table // First call: lastId=0, next call: lastId = last id from previous batch // ── Database Indexes ────────────────────────────────── -- Single column index: good for queries filtering on one column CREATE INDEX idx_orders_user_id ON orders(user_id); -- Composite index: better when you always filter on both columns together CREATE INDEX idx_orders_user_status ON orders(user_id, status); -- Query: WHERE user_id = 123 AND status = 'PENDING' → uses this index perfectly -- Query: WHERE user_id = 123 alone → also uses this index (leftmost prefix rule) -- Query: WHERE status = 'PENDING' alone → does NOT use this composite index -- Covering index: index contains all columns the query needs -- No heap access needed — query satisfied entirely from index CREATE INDEX idx_orders_covering ON orders(user_id, status, created_at, total); -- EXPLAIN ANALYZE to check if index is being used EXPLAIN ANALYZE SELECT id, total, status FROM orders WHERE user_id = 123 AND status = 'PENDING' ORDER BY created_at DESC; -- Look for: "Index Scan using idx_orders_user_status" = good -- Look for: "Seq Scan on orders" = bad, add an index // ── HikariCP Tuning ─────────────────────────────────── spring: datasource: hikari: maximum-pool-size: 20 # increase if seeing connection timeouts minimum-idle: 5 connection-timeout: 30000 # 30s max wait for connection from pool idle-timeout: 600000 # release idle connections after 10 min max-lifetime: 1800000 # replace connections after 30 min

Writing Large Excel Files Efficiently with Apache POI

The right way to generate Excel files with 50,000+ rows without running out of memory.

📄
HSSFWorkbook
Old .xls format (Excel 97-2003). Max 65,535 rows. Entire file in memory. Only use if client requires legacy .xls format.
📋
XSSFWorkbook
New .xlsx format. Max 1,048,576 rows. Entire file in memory. Crashes with OOM on large datasets. Only safe for <10,000 rows.
⚡
SXSSFWorkbook
Streaming .xlsx format. Max 1,048,576 rows. Only N rows in memory. Rest flushed to temp disk file. Use for all production exports.
⚙️ How SXSSFWorkbook Works — The Sliding Window

new SXSSFWorkbook(100) means: keep only the last 100 rows in memory. When you create row 101, row 1 is flushed to a temporary file on disk. You can no longer access or modify row 1 — it is gone from memory. Rows are written strictly sequentially.

When you call wb.write(outputStream), POI streams from the temp file to your output stream. The entire file never exists in memory at once. Memory usage is flat at approximately 100 rows × row_size, regardless of whether you write 100 or 1,000,000 rows.

wb.setCompressTempFiles(true) compresses the temp file with GZIP, reducing disk usage. Worth enabling for large exports.

Three Critical Gotchas That Cause Real Production Bugs

Gotcha 1 — Cell styles created in a loop: Calling wb.createCellStyle() inside your row loop (once per row) creates a new style object for every row. Excel has a hard limit of 64,000 unique cell styles. At 64,001 rows, you get an exception and your export crashes. Fix: create all cell styles ONCE before the loop and reuse the same style object on every row.

Gotcha 2 — trackColumnForAutoSizing called after writing rows: In SXSSF, rows that have been flushed to disk cannot be re-read for column width calculation. You must call sheet.trackColumnForAutoSizing(columnIndex) BEFORE writing any rows. If you call it after, column widths will be wrong or an exception will be thrown.

Gotcha 3 — wb.dispose() not called: SXSSFWorkbook creates temporary files in the system temp directory. They are only cleaned up when you call wb.dispose(). Without it, temp files accumulate over time and fill the disk. Always call dispose() in a finally block or use try-with-resources. Note: wb.close() does NOT call dispose() — you must call both.

@Service public class ExcelExportService { @Transactional(readOnly = true) public void streamOrders(LocalDate from, LocalDate to, OutputStream out) throws IOException { SXSSFWorkbook wb = new SXSSFWorkbook(500); // 500-row sliding window wb.setCompressTempFiles(true); try { SXSSFSheet sheet = wb.createSheet("Orders"); // Track columns BEFORE writing any rows for (int i = 0; i < 5; i++) sheet.trackColumnForAutoSizing(i); // Create ALL styles ONCE — never inside the row loop CellStyle headerStyle = createHeaderStyle(wb); CellStyle dateStyle = createDateStyle(wb); CellStyle moneyStyle = createMoneyStyle(wb); // Header row String[] headers = {"ID", "Customer", "Total", "Date", "Status"}; Row headerRow = sheet.createRow(0); for (int i = 0; i < headers.length; i++) { Cell cell = headerRow.createCell(i); cell.setCellValue(headers[i]); cell.setCellStyle(headerStyle); // REUSE — not createCellStyle() } // Data rows — stream from DB, write row by row try (Stream<Order> stream = orderRepo.streamByCreatedAtBetween(from, to)) { AtomicInteger rowNum = new AtomicInteger(1); stream.forEach(order -> { Row row = sheet.createRow(rowNum.getAndIncrement()); row.createCell(0).setCellValue(order.getId().toString()); row.createCell(1).setCellValue(order.getCustomerName()); Cell totalCell = row.createCell(2); totalCell.setCellValue(order.getTotal().doubleValue()); totalCell.setCellStyle(moneyStyle); // REUSE Cell dateCell = row.createCell(3); dateCell.setCellValue(order.getCreatedAt()); dateCell.setCellStyle(dateStyle); // REUSE row.createCell(4).setCellValue(order.getStatus().name()); }); } // Auto-size after writing (works because we tracked above) for (int i = 0; i < 5; i++) sheet.autoSizeColumn(i); wb.write(out); // stream to response output stream } finally { wb.dispose(); // CRITICAL: clean up temp files on disk wb.close(); } } private CellStyle createHeaderStyle(Workbook wb) { CellStyle style = wb.createCellStyle(); Font font = wb.createFont(); font.setBold(true); font.setFontHeightInPoints((short) 11); style.setFont(font); style.setFillForegroundColor(IndexedColors.GREY_25_PERCENT.getIndex()); style.setFillPattern(FillPatternType.SOLID_FOREGROUND); style.setBorderBottom(BorderStyle.THIN); return style; } private CellStyle createDateStyle(Workbook wb) { CellStyle style = wb.createCellStyle(); style.setDataFormat(wb.createDataFormat().getFormat("dd/mm/yyyy")); return style; } private CellStyle createMoneyStyle(Workbook wb) { CellStyle style = wb.createCellStyle(); style.setDataFormat(wb.createDataFormat().getFormat("#,##0.00")); return style; } }

Follow-Up Questions & Full Answers
Q1: How do you show a progress bar for a large async export? +

For the async POST pattern, the frontend polls a status endpoint while the export is generating. Here is the complete implementation:

Backend — export job with progress: When the export job starts, store a job record in Redis with status=PROCESSING and progress=0. As rows are written (using a counter in the stream forEach), periodically update Redis: at every 1,000 rows, update progress=(rowsWritten/totalRows)*100. When complete, set status=READY and store the file path or S3 URL.

Backend — status endpoint: GET /api/export/status/{jobId} reads the Redis record and returns: {"status": "PROCESSING", "progress": 45, "rowsWritten": 22500, "totalRows": 50000}.

Frontend — polling: After POST /api/export returns the jobId, start polling GET /api/export/status/{jobId} every 2 seconds. Update a progress bar using the progress field. When status=READY, show a "Download" button pointing to GET /api/export/download/{jobId}. If status=FAILED, show an error message.

Alternative — Server-Sent Events (SSE): Instead of polling, the backend pushes progress updates via SSE. More efficient (no polling overhead), real-time updates. Spring supports SSE via SseEmitter. The frontend uses the EventSource API. Better user experience but more complex to implement.

Q2: The data is growing and queries are getting slow. What is your exact diagnostic process? +

Follow this exact sequence — do not skip to solutions before diagnosing:

Step 1 — Enable slow query logging: In MySQL: SET GLOBAL slow_query_log=1; SET GLOBAL long_query_time=1;. Queries taking more than 1 second appear in the slow query log. This tells you WHICH queries are slow before you look at any code.

Step 2 — Run EXPLAIN ANALYZE on the slow queries: Paste each slow query into your DB console with EXPLAIN ANALYZE prepended. Look for Seq Scan on large tables. Look at the estimated vs actual row counts — a large discrepancy means your table statistics are stale (run ANALYZE).

Step 3 — Check for N+1 queries: Add Datasource-proxy or p6spy to your Spring Boot project. These libraries count queries per HTTP request and log them. Any endpoint making 50+ queries has N+1. This is often the biggest issue.

Step 4 — Check connection pool metrics: Look at HikariCP metrics (exposed via Spring Boot Actuator). If you see "connection timeout" errors or the pool is consistently at maximum size — you have connection pool exhaustion. Either increase the pool size or reduce query time so connections are released faster.

Step 5 — Check what the query plan is doing: If EXPLAIN ANALYZE shows an index exists but is not being used — the DB might have decided a full table scan is faster (at low cardinality). Consider forcing the index or reconsidering the query structure. Sometimes a slightly rewritten query uses the index better.

Step 6 — Only after the above — consider infrastructure changes: Redis caching for hot data. Read replicas for read-heavy workloads. Query result caching at the application layer. These are optimisations on top of correct queries, not replacements for correct queries.

Q3: What is a covering index and when does it help? +

A covering index is an index that contains all the columns a query needs to return its result. When a covering index exists, the database can satisfy the query entirely from the index without accessing the actual table rows (the "heap"). This is called an "index-only scan" in PostgreSQL or a "covering index scan" in MySQL.

Why it matters: When a regular index is used, the database: (1) searches the index to find matching entries, (2) uses each index entry to look up the actual row in the table heap, (3) reads the columns it needs from the table row. Step 2 and 3 add I/O overhead — potentially one random disk access per matching row.

With a covering index: the database searches the index and the index already contains every column the query needs. No heap access. For read-heavy queries on large tables, this can be 2-10x faster.

Example: Query: SELECT id, status, created_at FROM orders WHERE user_id = 123 ORDER BY created_at. Regular index on (user_id) — uses index to find rows, then reads heap for status and created_at. Covering index on (user_id, status, created_at) — index contains all needed columns, no heap access.

Trade-off: Covering indexes are larger (more columns) and slower to update (more data to maintain when rows are inserted/updated). Use them on high-read, low-write tables, or for your most critical and frequently-run queries. Do not create covering indexes for every query — the maintenance overhead adds up.

Q4: What is the difference between xlsx and csv for large exports — which is faster? +

This is a practical question that comes up when optimising export performance.

CSV is significantly faster to generate and much smaller in file size:

  • CSV is plain text — no binary encoding, no XML, no ZIP compression to manage
  • A CSV row is just values separated by commas and terminated by a newline — trivial to write
  • 100,000 rows in CSV: typically 5-20MB and generates in under 1 second
  • 100,000 rows in xlsx: typically 20-80MB and generates in 5-20 seconds

xlsx has features CSV cannot match:

  • Multiple sheets in one file
  • Cell formatting (bold headers, date formats, currency symbols, colours)
  • Column widths and row heights
  • Formulas and pivot tables
  • Type awareness — numbers stay numbers, dates stay dates (CSV is all strings)

Recommendation: If the user needs formatting, charts, or multiple sheets — xlsx. If they just need the raw data quickly and will process it programmatically — CSV. Consider offering both options — a "Quick CSV Export" and a "Formatted Excel Export" button. Most data engineers prefer CSV; most business users prefer Excel.

For large datasets where speed matters most: generate CSV in under a second by streaming directly to the response as a text/csv content type. No Apache POI needed — just a StringBuilder or BufferedWriter writing comma-separated values.

Q5: How do you handle the case where data changes while the export is running? +

This is a subtle consistency problem that many developers do not consider. A large export takes 30 seconds to generate. During those 30 seconds, other users are creating, updating, and deleting records. The exported data may be inconsistent — some rows reflect the state at the start, others at the end.

Option 1 — Accept inconsistency (most common, usually acceptable): For most business reports, data that is "as of roughly this moment" is fine. A sales report where some orders were captured at 10:00:01 and others at 10:00:29 is still useful. Document the report as a point-in-time snapshot. This is what most applications do.

Option 2 — Snapshot isolation (database-level): Use REPEATABLE READ or SERIALIZABLE isolation. The entire transaction sees a consistent snapshot of the database as it existed when the transaction started. New commits during the export are invisible to it. Spring: @Transactional(isolation = Isolation.REPEATABLE_READ, readOnly = true). Works perfectly but holds read locks longer, increasing the chance of deadlocks in high-write environments.

Option 3 — Export from a read replica: The replica has a slight lag behind the primary (usually milliseconds to seconds). For export purposes, this is acceptable — the export reflects the replica's state at the time it started. No impact on the primary at all. This is the best option for large, frequent exports in high-traffic systems.

Option 4 — Materialised snapshot: Before exporting, create a snapshot table: INSERT INTO export_snapshot_xyz SELECT * FROM orders WHERE .... Export from the snapshot. Delete the snapshot after export. Fully consistent export, no long-running transaction holding locks. Requires extra disk space for the snapshot.

Q6: When would you completely abandon JPA and use JDBC directly? +

JPA is excellent for standard CRUD operations on your domain entities. But it is the wrong tool for several common scenarios:

1. Bulk inserts of thousands of rows: JPA's save() method for each row means individual INSERT statements (or at best small batches). JDBC's PreparedStatement batch mode can insert 10,000 rows with one database round trip. For importing large datasets, JDBC is 50-100x faster.

2. Complex analytical queries with window functions, CTEs, GROUPING SETS: JPA's JPQL cannot express these. You can use nativeQuery=true in @Query, but at that point you are writing SQL anyway — use JdbcTemplate for full control and readability. Fighting JPA to produce the right SQL takes longer than writing the SQL directly.

3. Reporting and aggregation: Queries like "top 10 products by revenue this month with comparison to last month" require self-joins, CASE expressions, and complex GROUP BY. The JPA-generated SQL for these is verbose, hard to tune, and often inefficient. Write native SQL.

4. Full-text search: JPA has no support for full-text search (MATCH AGAINST in MySQL, tsvector in PostgreSQL). Use nativeQuery=true or switch to Elasticsearch for full-text capabilities.

5. Database-specific features: Postgres-specific features like JSONB queries, array operations, or custom types. MySQL-specific features like full-text indexes or spatial queries. JPA abstracts these away — use native SQL when you need them.

The rule of thumb: Use JPA for your core domain entities — the Order, User, Product, Payment classes that represent your business model. Use JDBC directly for everything that feels like a query problem rather than an object problem — reports, analytics, batch operations, imports, exports.

Interview Tip — Answering the Full Excel Export Question

The interviewer starts with "In the UI there is a button, clicking it should download an Excel file." They will keep asking follow-up questions to see how deep your knowledge goes. Here is the complete answer chain:

"For a simple export with date range parameters, I use GET — the parameters fit cleanly in the URL, it is bookmarkable, and the browser handles the download naturally with Content-Disposition: attachment header."

They ask: "What if it's 50,000 rows?"

"I switch to an async POST pattern. The POST schedules a background job and returns a job ID immediately. The frontend polls for status and shows a progress bar. When ready, the user downloads via a pre-signed URL."

They ask: "How do you fetch 50,000 rows from the DB without running out of memory?"

"I use JPA streaming — Stream<Entity> with @QueryHints setting the fetch size to 500. This keeps a DB cursor open and fetches 500 rows at a time into memory, processes them, then fetches the next 500. The service method must be @Transactional(readOnly=true) to keep the session open. Memory stays flat regardless of row count."

They ask: "How do you write those 50,000 rows to Excel without OOM?"

"SXSSFWorkbook with a 500-row sliding window. Rows beyond 500 are flushed to a temp file on disk. I create all cell styles once before the loop. I always call wb.dispose() in a finally block to clean up temp files."
