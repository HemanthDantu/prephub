Microservice Design Patterns

Start Here — What Is a Design Pattern?

Before diving into specific patterns, understand what a pattern actually is and why they exist.

Formal Definition

A design pattern is a reusable, proven solution to a commonly occurring problem in software design. It is not code you copy and paste — it is a blueprint, a way of thinking and structuring your solution so that it works reliably at scale.

Patterns emerged because developers across companies kept facing the same problems, solving them differently each time, with inconsistent results. Patterns captured the best solutions so everyone could benefit.

Everyday Analogy

Think of architectural blueprints for houses. A "blueprint for a two-storey home with load-bearing walls" is not a specific house — it is a proven pattern that builders reuse. You adapt it to your land and requirements, but the core structural solution is trusted and tested.

Software design patterns are exactly this — trusted structural solutions you adapt to your specific system.

Why Do Microservices Specifically Need Patterns?

In a monolith (one big application), many problems solve themselves automatically:

  • One database = one transaction = if anything fails, everything rolls back automatically
  • One codebase = you call any function directly, no network involved
  • One deployment = no version mismatches between parts of the system

When you split into microservices, all of that disappears:

  • Multiple databases = no single transaction spans all of them
  • Network calls between services = failures, timeouts, partial results
  • Independent deployments = your service might call a service running a different version

Design patterns are the systematic answers to these new problems. Without them, every team invents their own inconsistent solutions — creating systems that are hard to understand, debug, and maintain.

Pattern 1 — Saga Pattern

The most commonly asked pattern in senior interviews. Understand this one deeply.

Formal Definition

The Saga pattern is a design pattern for managing data consistency across multiple microservices in distributed transactions. A saga is a sequence of local transactions. Each local transaction updates one service's database and triggers the next step. If any step fails, the saga executes compensating transactions to undo the previous steps.

🤔 The Problem It Solves — Start Here

Imagine you are building an e-commerce order system. Placing one order involves 4 services:

  • OrderService — creates the order record
  • PaymentService — charges the customer's card
  • InventoryService — reserves the items
  • ShippingService — schedules delivery

In a monolith with one database, you wrap all 4 in one SQL transaction. If InventoryService step fails, the database rolls back the charge too — automatically. Perfect.

In microservices, each service has its own database. There is no shared transaction. If you charge the card in PaymentService (step 2) and then InventoryService fails (step 3) — you have charged the customer for an order that cannot be fulfilled. No automatic rollback. The customer has lost money.

This is the distributed transaction problem. The Saga pattern is the solution.

Real World Analogy

Think about booking a holiday package: flight + hotel + car rental. If all three succeed, great. But if the car rental is unavailable after you have already booked the flight and hotel, you need to cancel the hotel and cancel the flight — those cancellations are the compensating transactions. The travel agency (saga orchestrator) manages this for you.

📋 Happy Path vs Failure Path

Happy path — everything succeeds:

1
OrderService — Create Order
Saves order with status PENDING to its own database. Triggers next step.
↓
2
PaymentService — Charge Card
Charges the customer's card. Records payment. Triggers next step.
↓
3
InventoryService — Reserve Stock
Reduces stock count by the ordered quantity. Triggers next step.
↓
4
ShippingService — Schedule Delivery
Books a delivery slot. Triggers final step.
↓
5
OrderService — Mark CONFIRMED
Updates order status to CONFIRMED. Sends confirmation email.
❌ Failure Path — Step 3 Fails (Inventory Out of Stock)

Steps 1 and 2 already committed. Step 3 fails. We must undo steps 1 and 2 using compensating transactions:

3
InventoryService — FAILS (out of stock)
Cannot reserve stock. Triggers compensation chain in reverse order.
↓
C2
PaymentService — Compensate: Refund Card
Issues a refund for the charge made in step 2. This is the compensating transaction for step 2.
↓
C1
OrderService — Compensate: Mark FAILED
Updates order status to FAILED. Notifies the customer. This is the compensating transaction for step 1.
Critical Understanding — Compensating Transactions Are NOT Rollbacks

A database rollback means the operation never happened from the database's perspective. No trace in the records.

A compensating transaction is a brand new business operation that logically reverses a previous one. The original charge DID happen and IS recorded. The refund also happens and is recorded. Both appear in audit logs, transaction history, and customer receipts.

This means you must design and implement compensating transactions deliberately for every step of your saga. You cannot assume they will be automatic. Some operations cannot be compensated — a sent email cannot be unsent. For those, you design "countermeasures" (send a cancellation email instead).

🎭
Two Flavours of Saga
🎵 Choreography-Based Saga
  • No central coordinator
  • Each service reacts to events from the previous service
  • Services talk via a message broker (Kafka)
  • Simple to set up initially
  • Good for 2-3 steps
  • Hard to debug at scale
🎼 Orchestration-Based Saga
  • Central Saga Orchestrator class
  • Orchestrator explicitly calls each service in order
  • One place to see the entire flow
  • More complex to set up
  • Best for 4+ steps or complex flows
  • Easy to debug — one log shows everything
🎵 Choreography — How It Actually Works

Imagine a WhatsApp group where each team posts an update and the next team acts on it — with no project manager involved:

  • OrderService saves order → publishes ORDER_CREATED event to Kafka
  • PaymentService listens for ORDER_CREATED → charges card → publishes PAYMENT_COMPLETED
  • InventoryService listens for PAYMENT_COMPLETED → reserves stock → publishes STOCK_RESERVED
  • ShippingService listens for STOCK_RESERVED → schedules delivery

No single service knows about the full flow. Each only knows: "when I see THIS event, I do MY job and emit THAT event."

Choreography Analogy

A relay race. Each runner knows exactly what to do when they receive the baton. There is no coach standing in the middle of the track directing every hand-off. The runners just react to receiving the baton and pass it on. If one runner drops it (failure), you need a protocol for who picks it up — this is the compensating transaction design challenge in choreography.

🎼 Orchestration — How It Actually Works

A single Saga Orchestrator class (a Spring @Service) explicitly calls each service and manages the flow:

  • Orchestrator calls PaymentService.charge() → if fails: saga ends with FAILED
  • Orchestrator calls InventoryService.reserve() → if fails: calls PaymentService.refund(), saga FAILED
  • Orchestrator calls ShippingService.schedule() → if fails: calls InventoryService.release(), calls PaymentService.refund(), saga FAILED
  • All succeed → Orchestrator marks order CONFIRMED

The orchestrator also persists its state (in a saga_state database table) after every step. If the application crashes, the next startup reads the table and knows exactly which step to resume from.

Orchestration Analogy

An air traffic controller. Planes (services) do not talk to each other directly — they would crash into each other. The controller (orchestrator) tells each plane when to take off, where to fly, and when to land. If one plane has a problem, the controller manages the response — no other planes need to know about it.

✅ Saga Pattern — Advantages
  • No distributed lock needed — no performance bottleneck
  • Each service stays autonomous and independent
  • Works across completely different database technologies
  • Resilient — individual steps can retry independently
  • Scales horizontally without constraint
  • Full audit trail — every step is recorded
❌ Saga Pattern — Disadvantages
  • Eventual consistency — data temporarily inconsistent between steps
  • Compensating transactions are hard to design and test correctly
  • Compensation can also fail — needs its own retry logic
  • Choreography becomes "event spaghetti" with many services
  • Difficult to debug production failures mid-saga
  • Cannot compensate some operations (sent emails, printed documents)
🔄 Saga vs Two-Phase Commit (2PC) — Why Saga Won

Two-Phase Commit (2PC) is the old approach to distributed transactions. A transaction coordinator asks all participants to "prepare" (phase 1), then commits them all (phase 2). It seems perfect — but:

  • Performance killer: All participants hold database locks during both phases. At scale with thousands of transactions per second, this creates massive lock contention.
  • Single point of failure: If the coordinator crashes between phase 1 and phase 2, all participants are stuck holding locks forever, waiting for a coordinator that is never coming back.
  • Tight coupling: All services must use the same transaction protocol. Incompatible with polyglot persistence (different DB types per service).

Saga is preferred because it uses local transactions only — no locks held across services, no coordinator bottleneck, works with any database type.

Pattern 2 — CQRS

Command Query Responsibility Segregation — separating reads from writes for performance and scalability.

Formal Definition

CQRS (Command Query Responsibility Segregation) is an architectural pattern that separates the model used to update information (Command — writes) from the model used to read information (Query — reads). Instead of one unified data model for everything, you maintain two separate models, each optimised for its specific purpose.

🤔 The Problem It Solves

In most real systems, reading and writing data have completely different characteristics:

  • Writes are less frequent, need strict validation, follow business rules, require normalised data structure
  • Reads are much more frequent, need to be fast, often join many tables, return data shaped exactly for the UI

When you force one model to serve both, you get compromises everywhere. The normalised write model forces expensive JOIN operations on every read. Denormalising for reads makes writes complex and error-prone.

Real example: Your order placement is fast and simple — one INSERT to the orders table. But your admin dashboard needs to show orders with customer names, product details, payment status, and delivery tracking — that is 5 JOINs on a single query, taking 2 seconds on a large table.

⚙️ How CQRS Works — Step by Step

Command Side (Write Model):

  • Handles all state changes: CreateOrder, UpdateStatus, CancelOrder
  • Uses a normalised relational model — strict integrity, foreign keys, validation
  • After a successful command, publishes a domain event to Kafka
  • Optimised for consistency and correctness, not for query speed

Query Side (Read Model):

  • A completely separate data store — can be same DB different table, or Redis, or Elasticsearch
  • Pre-joined, pre-aggregated, denormalised — shaped exactly like what the UI needs
  • Updated asynchronously by a consumer that listens to events from the command side
  • Multiple read models can exist for different consumers — mobile gets slim data, dashboard gets rich data

The flow: User places order → command side validates and saves normalised data → publishes ORDER_PLACED event → read model consumer updates the dashboard view table → dashboard query hits the pre-joined table and returns in 5ms instead of 2 seconds.

Real World Example — LinkedIn

When you post an update on LinkedIn, the write operation saves to a normalised write database. Your 500 followers seeing your post in their feed read from a completely separate pre-computed feed model.

Without CQRS, every feed page load would require: find all people this user follows, find their recent posts, rank by engagement, merge timelines, paginate. At LinkedIn scale (900 million users) — this is impossible to compute in real time for every page load.

With CQRS: the read model is pre-computed and updated as new posts arrive. Feed loads are near-instant because the data is already shaped exactly for the feed UI.

⚠️ Eventual Consistency — The Trade-Off You Must Accept

Because the read model is updated asynchronously after the write, there is a small window (milliseconds to a second) where the read model is behind the write model. A user who just placed an order might not immediately see it in their order list.

Solutions:

  • Optimistic UI update: Frontend adds the order to the list immediately without waiting for the server, then confirms when the event propagates. This is what most modern apps do — it feels instant.
  • Read your own writes: After a command, route that specific user's immediate read to the write model (bypassing the eventual consistency window), then switch back to the read model after a few seconds.
  • Inform the user: Show "Your order is being processed" for a moment. Most users do not notice a 500ms lag between submitting and seeing their data.
✅ CQRS — Advantages
  • Read and write sides scale independently
  • Read model is optimised purely for query performance
  • Multiple read models for different clients
  • Writes stay clean — no query optimisation concerns
  • Enables event sourcing naturally
  • Dashboard queries go from seconds to milliseconds
❌ CQRS — Disadvantages
  • Eventual consistency — read model slightly behind write
  • Significantly more complex than simple CRUD
  • Two models to maintain — bug fixes touch both sides
  • Event pipeline is a new failure point
  • Over-engineering for simple CRUD applications
  • Team must understand and accept eventual consistency
When NOT to Use CQRS

CQRS is one of the most over-applied patterns. Do not use it when your read and write shapes are essentially the same, when your application is simple CRUD with low traffic, when your team is small and the complexity will slow you down, or when you genuinely need immediate consistency between reads and writes.

A good rule: if your queries are fast and not causing problems, CQRS is premature optimisation. Apply it when you have evidence of a read/write mismatch causing real problems.

Pattern 3 — BFF (Backend For Frontend)

One API for all clients causes compromises everywhere. BFF gives each client its own tailored backend.

Formal Definition

The Backend for Frontend (BFF) pattern creates a dedicated backend service for each type of frontend application. Instead of one general-purpose API that tries to serve every client simultaneously, each client has its own tailored API layer that aggregates, filters, and shapes data specifically for that client's needs, network characteristics, and interaction model.

🤔 The Problem It Solves

Without BFF, a single general API must satisfy every client at once:

  • Over-fetching: Mobile app needs 5 fields to show a card. The API returns 40 fields. Mobile wastes bandwidth and battery downloading 35 fields it does not use.
  • Under-fetching (chatty client): Mobile needs data from 3 services. It makes 3 sequential API calls. On a mobile network with 200ms latency each, that is 600ms of waiting before rendering anything.
  • Change paralysis: Adding a mobile-specific feature means modifying the shared API — which risks breaking the web app and any third-party integrations.
  • One-size-fits-none: The API ends up being too complex for mobile (too much data), too simple for the web dashboard (not enough data), and too coupled for third parties.
⚙️ How BFF Works

Each client type gets its own BFF service. The BFF sits between the client and the downstream microservices:

  • MobileBFF: Calls UserService, OrderService, ProductService → aggregates → returns one slim, compressed response with only the fields the mobile app needs → one network call instead of three
  • WebBFF: Calls the same services plus AnalyticsService → returns richer data with more fields → can include data the mobile app would never need
  • PublicAPI: A rate-limited, versioned API for third-party developers → exposes only a safe subset of functionality

Each BFF is owned by the team that owns the corresponding frontend. The mobile team controls their BFF — they can add a new field without asking the web team or the API team for permission.

Real World Example — Netflix

Netflix has separate backend services for mobile phones, tablets, smart TVs, game consoles, PlayStation, and web browsers. Each is optimised for that device's constraints:

  • TV BFF: Returns large, high-resolution images and pre-fetches more content because TV users browse longer and have fast internet connections
  • Mobile BFF: Returns compressed images, fewer recommendations, and less metadata to save mobile data and battery
  • Smart TV BFF: Returns simplified navigation-friendly data because smart TVs use remote controls, not mice

Without BFF, one API would either waste TV bandwidth with compressed images, or drain mobile batteries with large images. BFF lets each client be served perfectly.

✅ BFF — Advantages
  • Each client gets exactly the data it needs
  • Client teams work independently — no cross-team dependencies
  • Fewer network calls from client — BFF aggregates
  • Client-specific caching, auth, and rate limiting strategies
  • Easier to evolve — changes affect only one client
❌ BFF — Disadvantages
  • Code duplication — multiple BFFs may call the same services
  • More services to deploy, monitor, and maintain
  • Risk of business logic leaking into BFF — it should only aggregate
  • Requires discipline to keep BFFs thin over time
  • Overkill if you only have one type of client
🔄 Alternative to BFF — GraphQL

GraphQL is a popular alternative. Instead of multiple BFF services, you have one endpoint. The client specifies exactly which fields it needs in the query itself — no over-fetching, no under-fetching.

BFF is better when: Clients have very different requirements and you want each team to have full ownership and autonomy over their API layer.

GraphQL is better when: You want one flexible endpoint that multiple clients can query differently without maintaining separate services. This is what GitHub, Twitter, and Shopify use for their public APIs.

Pattern 4 — Sidecar Pattern
Formal Definition

The Sidecar pattern attaches a helper component — the "sidecar" — alongside a primary service component. The sidecar runs in the same execution context (same Kubernetes pod) as the main service. It provides supporting infrastructure features without the main service needing to implement them or even know about them.

🤔 The Problem It Solves

Every microservice needs: logging, distributed tracing, metrics collection, TLS encryption, service discovery, circuit breaking. If every service implements all of this:

  • Every codebase has boilerplate infrastructure code that clutters business logic
  • Updating the logging library requires redeploying every single service
  • Different teams implement the same concerns differently — inconsistency
  • A bug in the shared infrastructure library affects every service simultaneously

The Sidecar moves all infrastructure concerns OUT of the service code entirely, into a separate helper container.

Real World Example — Istio Service Mesh

Istio automatically injects an Envoy proxy container into every pod in your Kubernetes cluster. Your service code makes zero changes. The Envoy sidecar intercepts all inbound and outbound network traffic and transparently provides:

  • Mutual TLS (mTLS): Encrypts all service-to-service communication automatically — no SSL code in your service
  • Distributed tracing: Propagates trace headers automatically — no OpenTelemetry setup in your code
  • Circuit breaking: Configured via Kubernetes config files — no Resilience4j annotations needed
  • Retries and timeouts: Configured externally — no retry logic in your service code
  • Canary deployments: Route 5% of traffic to new version — zero application code changes

Your service code becomes purely business logic. Infrastructure is the sidecar's job.

Pattern 5 — Strangler Fig Pattern
Formal Definition

The Strangler Fig pattern is a migration strategy for incrementally replacing a legacy system by gradually routing individual features to new microservices one at a time, until the legacy system handles no traffic and can be safely decommissioned.

🤔 The Problem It Solves — Why Not Rewrite Everything?

The temptation when dealing with a large legacy monolith is to rewrite it from scratch. This is called a "big bang rewrite" and it almost always fails or significantly overruns:

  • The new system takes months or years to build
  • The business keeps running on the old system the entire time
  • The new system must replicate ALL functionality before it can go live
  • One deployment switches everything — if anything is wrong, everything breaks
  • The new team rarely understands all the edge cases encoded in the old system

Strangler Fig eliminates big-bang risk by making the migration incremental — each feature is extracted and replaced independently, with the ability to roll back instantly if something goes wrong.

⚙️ How It Works
  1. Put a routing layer in front of the monolith: An API gateway or reverse proxy sits in front of everything. Initially, 100% of traffic goes to the monolith. Nothing changes for users.
  2. Extract one feature: Build a new microservice for one specific feature — say, the user profile page. Test it thoroughly.
  3. Update routing: Change the routing layer to send /profile/* to the new service instead of the monolith. Everything else still goes to the monolith.
  4. Monitor: Watch metrics and error rates. If anything is wrong, update routing to send /profile/* back to the monolith. Zero user impact.
  5. Repeat: Extract the next feature. And the next. Over months, the monolith handles less and less.
  6. Decommission: When the monolith handles zero traffic, shut it down.
Why It Is Called Strangler Fig

The strangler fig tree starts as a seed in the canopy of a host tree. It grows roots down around the host tree's trunk. Over years, the fig's roots completely surround the host. The host tree eventually dies and rots away. The fig stands on its own, having used the host as scaffolding. The old monolith is the host tree. Your new microservices are the fig.

Follow-Up Questions & Full Answers

These are the exact questions interviewers ask after you explain the patterns. Click each to reveal the full answer.

Q1: What is the difference between Choreography and Orchestration Saga? When would you choose each? +

Choreography has no central coordinator. Services communicate entirely through events on a message broker like Kafka. Each service knows what to do when it receives a specific event. There is no single point of failure because there is no coordinator. However, when something goes wrong, you must trace events across multiple service logs to understand what happened — there is no single place that shows the full picture. As the number of steps grows, the event dependencies become hard to visualise and maintain.

Orchestration has a central Saga Orchestrator class that explicitly calls each service in sequence and handles failures. The entire flow is visible in one class. Debugging is straightforward — the orchestrator's logs show every step. Adding conditional logic (if user is premium, skip step 2) is easy in the orchestrator but complex in choreography (every service would need to know about premium users).

Choose choreography when: 2-3 steps, simple linear flow, one team owns all services, you want zero central dependency.

Choose orchestration when: 4 or more steps, complex conditional flows, multiple teams, you need an audit trail, debugging production issues is a concern.

Q2: What happens if a compensating transaction itself fails? +

This is a real and serious production problem that many candidates overlook. If the refund to the payment provider fails because PaymentService is down, the customer has been charged and no refund was issued. This is a genuine data inconsistency that requires careful handling.

Solution 1 — Retry with exponential backoff: Compensating transactions must be retried with the same resilience patterns as normal transactions. They should not be attempted once and abandoned.

Solution 2 — Persist the compensation state: The saga state table records not just which step the saga is on, but also which compensations are pending. If the application restarts mid-compensation, the saga resumes and retries the pending compensation.

Solution 3 — Human escalation: After a maximum number of retries, the saga moves to a COMPENSATION_FAILED state and triggers a human alert. A support team member manually resolves the inconsistency. This is unavoidable in extreme cases and must be designed into the system — you need tooling to view and resolve stuck sagas.

This is why robust saga implementations always include a saga_state table, a maximum retry count per compensation, and an alerting mechanism for stuck sagas.

Q3: What is eventual consistency in CQRS and how do you handle it in the UI? +

Eventual consistency means the read model is slightly behind the write model. There is an asynchronous event pipeline between the write side (where the command was processed) and the read side (the pre-computed view). This pipeline takes milliseconds to seconds. During that window, a user who just placed an order might not see it in their order list because the read model has not yet received and processed the ORDER_PLACED event.

Handling it in the UI — Option 1 (Optimistic UI Update): When the user submits an order, the frontend immediately adds it to the order list without waiting for the server event to propagate. The frontend is "optimistic" that the write will succeed. If the write fails, the frontend removes it. This is what most modern applications do — users never notice the lag because the UI updates instantly.

Handling it in the UI — Option 2 (Read Your Own Writes): After a successful command, route that specific user's immediate reads to the write model for a short window (5-10 seconds), then switch back to the read model. The user always sees their own latest changes immediately, but other users' queries still hit the fast read model.

Handling it in the UI — Option 3 (Version Token): The command response includes a version number. The read request includes that version number as a minimum. The read side waits (up to a timeout) until its data reaches that version. More complex but guarantees consistency for the specific user.

Q4: How do you ensure the Saga Orchestrator does not become a single point of failure? +

The orchestrator becoming a single point of failure is the most common criticism of the orchestration approach. Three techniques address this:

Make the orchestrator stateless: The orchestrator itself holds no saga state in memory. All state is persisted to a saga_state database table after every step. Any instance of the orchestrator can pick up any saga by reading the table. If one instance crashes, another picks up from exactly where it left off.

Run multiple instances: Deploy the orchestrator as a K8s Deployment with multiple replicas. All instances share the same saga_state table. Use optimistic locking or distributed locking (Redis lock) on the saga record to prevent two instances from processing the same saga simultaneously.

Use a dedicated saga framework: Tools like Axon Framework, Eventuate Tram, or AWS Step Functions manage orchestrator state, retries, and fault tolerance for you. They are designed specifically for this problem and handle the edge cases that are hard to implement correctly from scratch.

Q5: Can you use Saga without Kafka? What are the alternatives? +

Yes, absolutely. Kafka is a popular choice but not required.

For Orchestration-based Saga: Kafka is not required at all. The orchestrator can call downstream services directly via REST or gRPC (synchronous calls). The orchestrator decides what to call next — no event needed. This is actually simpler and works well when you control all the services and latency is acceptable.

For Choreography-based Saga: You need some messaging system, but it does not have to be Kafka. Options:

  • RabbitMQ: Simpler to set up and operate than Kafka, good for enterprise applications with moderate volume. Message delivery guarantees are strong. Not as durable or replayable as Kafka.
  • AWS SQS/SNS: Fully managed, no infrastructure to operate, good for cloud-native applications. SNS fans out to multiple SQS queues — naturally supports multiple consumers.
  • Azure Service Bus: Microsoft's managed messaging service, similar to SQS/SNS.
  • ActiveMQ: The Java EE standard messaging system, still widely used in enterprise environments.

Choose Kafka when you need: very high throughput (millions of events/day), message replay capability (consumers can re-read old events), long retention (keep events for days/weeks), or multiple independent consumer groups reading the same topic.

Choose RabbitMQ when you need: simpler setup, lower volume, and you do not need replay capability.

Q6: What is Event Sourcing and how does it relate to CQRS? +

Event Sourcing is the practice of storing every state change as an immutable event in an append-only event log, rather than storing only the current state of the entity.

Traditional approach: You have an orders table. A row gets updated as the order progresses: PENDING → CONFIRMED → SHIPPED → DELIVERED. At any point, you only know the current state. You cannot answer "what was the order status yesterday at 3pm?"

Event sourcing approach: You have an order_events table. Every change is a new row: ORDER_CREATED, PAYMENT_COMPLETED, STOCK_RESERVED, ORDER_SHIPPED, ORDER_DELIVERED. You never update a row — you only append. The current state is derived by replaying all events for that order. You can answer "what was the status at any point in time" by replaying only events up to that timestamp.

Relationship with CQRS: They are separate patterns that are often used together but are completely independent. CQRS says "separate your read and write models." Event sourcing says "store state as events, not as current values." When combined: the write side stores events (event sourcing), and the read side subscribes to those events to build pre-computed projections (CQRS). The event log is the source of truth. You can rebuild any read model at any time by replaying the events from the beginning. This is extremely powerful for debugging, audit requirements, and building new views of historical data.

Downside: Deriving current state by replaying many events can be slow. You use "snapshots" — periodically save the current state so replay starts from the most recent snapshot, not from event 1.

Q7: What pattern would you use if two microservices need to share data — do they share a database? +

This is one of the most important microservice design decisions and the answer is almost always: no, they should not share a database.

Sharing a database between two services creates tight coupling at the data layer — the worst kind of coupling. If Service A changes the schema of a shared table, Service B breaks. If Service A's queries are slow and cause DB locks, Service B's queries are also affected. Both services can now only be deployed together, defeating the purpose of microservices.

The correct alternatives:

  • API calls (synchronous): Service A calls Service B's API to get the data it needs. Service B owns the data. Clean, but adds network latency and creates a runtime dependency.
  • Event-driven data replication: Service B publishes events when its data changes. Service A subscribes and maintains its own local copy of the data it needs. No runtime dependency — Service A reads its local copy. This is called the "shared nothing" architecture.
  • Saga for write operations: When a write spans both services, use the Saga pattern instead of a shared transaction.

The key principle is: each service owns its data. Other services access that data only through the owning service's API or through events it publishes. Never through direct database access.

Interview Tip — How to Answer "What Patterns Did You Use?"

Never just list pattern names. Connect them to a real problem you solved:

Strong answer: "In our order management system, we used the Orchestration Saga because we had 5 services involved and needed clear visibility into which step failed when things went wrong in production. We also applied CQRS for our reporting dashboard — our write model was a normalised schema with 8 tables, but our dashboard queries needed pre-aggregated data. By maintaining a separate denormalised read model, dashboard queries went from 3 seconds to under 50ms."

Then be ready for the follow-up: "What were the trade-offs?" Always acknowledge the downsides — eventual consistency for CQRS, the complexity of compensating transactions for Saga. Interviewers trust candidates who acknowledge trade-offs far more than those who only highlight benefits.
