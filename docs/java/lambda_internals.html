Lambdas Are NOT Anonymous Inner Classes
invokedynamic · bytecode comparison · what the JVM actually does
The core idea

You know how when someone asks you to do something, you can either hire a whole new employee (train them, give them a desk, fill out paperwork) — or just hand them a sticky note with the instructions? Before Java 8, when you wanted to pass behaviour around, the JVM hired a full employee (anonymous inner class — its own .class file, its own object on the heap). Lambdas are sticky notes. Lightweight, no extra class file, and the JVM figures out at runtime the most efficient way to use them.

The contractor analogy

An anonymous inner class is like hiring a full-time contractor: you sign a contract, set up payroll, give them a badge, assign a desk — even if they'll only do one task once. An lambda is like texting a friend "hey can you grab milk?" — no contract, no desk, just the task itself passed around as a value.

The difference matters at the JVM level because anonymous inner classes create a real .class file on disk and a real object on the heap. Lambdas use a special JVM instruction called invokedynamic that defers how the lambda is represented until the first time it runs — and often avoids creating an object entirely.

What the Compiler Actually Produces

Anonymous inner class — old way (pre Java 8)
// You write this:
Runnable r = new Runnable() {
    @Override
    public void run() {
        System.out.println("Hello");
    }
};

// Compiler produces TWO .class files:
//   MyClass.class
//   MyClass$1.class   ← a brand-new class just for this one Runnable
// At runtime: new MyClass$1() → a real object on the heap, every time
Lambda — new way (Java 8+)
// You write this:
Runnable r = () -> System.out.println("Hello");

// Compiler produces ONE .class file (MyClass.class).
// The lambda body is compiled as a private static method inside MyClass:
//   private static void lambda$0() { System.out.println("Hello"); }
//
// The call-site gets an invokedynamic instruction. On first call,
// the JVM's LambdaMetafactory wires it up. On subsequent calls?
// Often just a method call — no heap allocation at all.

This is the key insight: invokedynamic defers the "how" to runtime. The JVM can choose the most efficient strategy — sometimes it allocates an object, sometimes it doesn't (for non-capturing lambdas it can reuse a singleton). The compiler doesn't lock in the representation at compile time.

Non-capturing vs capturing — memory difference
// NON-CAPTURING — no variables from outer scope used
Runnable r1 = () -> System.out.println("Hello");
// JVM can use a SINGLETON — same object reused every time. Zero heap cost.

// CAPTURING — uses 'name' from outer scope
String name = "Hemanth";
Runnable r2 = () -> System.out.println("Hello, " + name);
// JVM must create a new object each time to hold the captured 'name'.
// Still cheaper than anonymous inner class (no separate .class file),
// but not free.

Section 2 of 5 · ~7 mins
Variable Capture & Effectively Final
Why lambdas can't capture mutable variables · Stack vs Heap connection
The core idea

Imagine you hand a friend a printed recipe (a lambda). The recipe says "add 2 cups of sugar." That number is frozen in print — your friend can't know if you changed your mind to 3 cups after printing. If the recipe said "add as much sugar as the bowl currently has" — that's a reference to something that can change, and your friend might get a different answer depending on when they look. To keep things safe and predictable, Java says: you can only print variables into the recipe that you promise won't change. That promise is called "effectively final."

The photograph analogy

A lambda is like taking a photograph of a value. Once the photo is taken, it doesn't update if the real thing changes. If you allowed capturing a mutable variable, the photo and the real thing would go out of sync — especially problematic when the lambda runs on a different thread long after the original method's stack frame is gone.

Java's rule ("effectively final") is really about this: the stack frame that owns the variable might not exist by the time the lambda runs. The lambda needs its own stable copy, not a live reference to a potentially-dead stack slot.

When you capture orderId in a lambda, Java copies its value into the lambda object on the heap. The original stack variable is gone when the method returns — but the lambda (and its copy) live on. If Java allowed capturing mutable variables, you'd have a race: the stack changes the value at the same time the lambda reads its copy. Chaos.

What "effectively final" means in code
void processOrder(String orderId) {
    // ✅ effectively final — never reassigned after initialization
    String prefix = "ORDER-";
    Runnable r = () -> System.out.println(prefix + orderId);

    // ❌ NOT effectively final — reassigned later
    int count = 0;
    count++;  // This single line makes 'count' not effectively final
    Runnable r2 = () -> System.out.println(count);
    // Compiler error: "Variable used in lambda should be effectively final"
}

// The "final" keyword is optional — the compiler just checks that
// you never reassign it. These two are identical:
final String a = "hello";   // explicitly final
String b = "hello";          // effectively final (never reassigned)

The Common Workaround

When you need a "mutable counter" inside a lambda
// ❌ This won't compile:
int count = 0;
list.forEach(item -> count++);  // ERROR

// ✅ Wrap it in an array (1-element array is effectively final)
int[] count = {0};
list.forEach(item -> count[0]++);  // array ref is final, content isn't

// ✅ Better: use AtomicInteger for thread safety
AtomicInteger count = new AtomicInteger(0);
list.forEach(item -> count.incrementAndGet());

// ✅ Best for most cases: use streams + reduce/collect
long count = list.stream().filter(item -> item.isActive()).count();
⚠️ Watch out: captured object references ARE allowed — but mutation through them is dangerous
// The LIST reference is effectively final (never reassigned)
// but you can still MUTATE the list — Java won't stop you!
List<String> names = new ArrayList<>();
Runnable r = () -> names.add("Hemanth");  // compiles fine
// But if 'r' runs on another thread concurrently → race condition.
// "Effectively final" protects the REFERENCE, not the object's contents.

Functional Interfaces & SAM
Function · Predicate · Consumer · Supplier · BiFunction · @FunctionalInterface
The core idea

A lambda is just a behaviour — but Java is strongly typed, so every behaviour needs a type. A functional interface is the "shape mold" that gives the lambda a type. It's like a form with exactly one blank line: "Do one thing here: ___". The lambda fills in that blank. The form is the interface, the blank line is the single abstract method (SAM).

The vending machine analogy

Think of functional interfaces as different types of vending machines:

  • 🍎 Supplier — a machine that gives you something with no input. Just press the button: "give me a value."
  • 🗑️ Consumer — a machine that takes something and does work with it. You put something in and it processes it — no output back to you.
  • 🔀 Function — a machine that takes one thing and gives back a different thing. Input → transform → output.
  • ❓ Predicate — a machine that takes something and answers yes or no. "Is this item expired?"
  • 🔀🔀 BiFunction — a Function machine but it takes two inputs and produces one output.

What is a SAM interface?

Single Abstract Method — the rule behind lambdas

A SAM interface is an interface with exactly one abstract method. That's it. The lambda's body becomes the implementation of that one method. Java needs this constraint because otherwise the compiler wouldn't know which method the lambda implements.

// This is a SAM interface:
@FunctionalInterface
interface Transformer<T, R> {
    R transform(T input);  // ← the ONE abstract method

    // default methods are fine — they're not abstract
    default Transformer<T, R> andLog() {
        return input -> {
            R result = transform(input);
            System.out.println("Transformed: " + result);
            return result;
        };
    }
}

// Using it with a lambda:
Transformer<String, Integer> lengthOf = s -> s.length();
System.out.println(lengthOf.transform("Hemanth"));  // 7
Why does @FunctionalInterface matter?

It's a compile-time guard. If you accidentally add a second abstract method to the interface, the compiler immediately fails with a clear error instead of silently breaking all the lambdas using that interface.

@FunctionalInterface
interface MyFunc {
    void execute();
    void cleanup();  // ← Compiler ERROR: "multiple non-overriding abstract methods"
    // @FunctionalInterface caught this immediately.
    // Without it, the error only surfaces when you try to use a lambda.
}

All five in one real example
// Scenario: processing order list in a Spring Boot service

Supplier<List<Order>>   fetcher   = () -> orderRepo.findAll();
Predicate<Order>         isActive  = order -> order.getStatus() == ACTIVE;
Function<Order, String>  toId      = order -> order.getId();
Consumer<String>         logger    = id -> log.info("Processing: {}", id);
BiFunction<String,String,String> combine = (a, b) -> a + "-" + b;

fetcher.get()          // Supplier: fetch list
    .stream()
    .filter(isActive)    // Predicate: keep only active
    .map(toId)           // Function: Order → String id
    .forEach(logger);    // Consumer: log each id

// BiFunction used separately:
String tag = combine.apply("ORDER", "42");  // "ORDER-42"

Method References
4 types · when to use each · common gotchas
The core idea

A method reference is just a lambda with a shorter name. Instead of writing x -> Math.abs(x), you can write Math::abs. You're saying: "take this existing method and use it as a behaviour." It's like giving someone directions to a destination versus just handing them a pin on a map — same result, less words.

The speed dial analogy

A lambda is like dialing a full phone number: x -> x.toUpperCase(). A method reference is a speed-dial button labeled String::toUpperCase. Same call, one tap. The four types of method references are just four different ways to pre-program that button: with a static function, with a specific object, with any object of a class, or with a constructor.

The 4 Types

① Static Method
ClassName::staticMethod
The method doesn't belong to any instance. The lambda's argument becomes the method's argument.
② Instance (bound)
instance::method
References a method on a specific, already-known object. The object is "bound" — captured just like a lambda variable.
③ Instance (unbound)
ClassName::instanceMethod
The first argument to the lambda becomes this — the receiver. Used when iterating over a collection of objects of that type.
④ Constructor
ClassName::new
Creates a new object. The lambda's arguments become constructor arguments. Used as factories.
All 4 types in code — side by side
// ① STATIC — ClassName::staticMethod
// Lambda:           x -> Integer.parseInt(x)
Function<String, Integer> parse = Integer::parseInt;
Function<Integer, String> abs   = Math::abs;   // wait — Math.abs returns int
// Use: parse.apply("42") → 42

// ② INSTANCE (bound) — instance::method
// Lambda:           s -> myPrefix.concat(s)
String myPrefix = "ORDER-";
Function<String, String> addPrefix = myPrefix::concat;
// Use: addPrefix.apply("42") → "ORDER-42"
// 'myPrefix' is the bound/captured object

// ③ INSTANCE (unbound) — ClassName::instanceMethod
// Lambda:           s -> s.toUpperCase()
Function<String, String> upper = String::toUpperCase;
List<String> names = List.of("alice", "bob");
names.stream().map(String::toUpperCase).toList();
// The String in the stream becomes 'this' for toUpperCase()

// ④ CONSTRUCTOR — ClassName::new
// Lambda:           name -> new Order(name)
Function<String, Order> factory = Order::new;
List<Order> orders = List.of("ORD-1", "ORD-2")
    .stream()
    .map(Order::new)
    .toList();

The Tricky One: Unbound vs Bound

⚠️ Common confusion — unbound looks like static but isn't
// These LOOK similar but are very different:

// Static — Integer.parseInt is static, takes String, returns int
Function<String, Integer> f1 = Integer::parseInt;
//   signature: (String) → Integer ✓

// Unbound — String.toUpperCase is an instance method
// The Function's input becomes the receiver (this)
Function<String, String> f2 = String::toUpperCase;
//   signature: (String) → String ✓  ← the String is 'this', not an arg

// How to tell the difference?
// Check if the method is static in the class.
// If it's static → Type ① (static)
// If it's an instance method on ClassName → Type ③ (unbound)
Real-world usage in a Spring Boot service
@Service
public class OrderService {

    private final OrderRepository repo;
    private final OrderMapper     mapper;

    public List<OrderDto> getActiveOrders() {
        return repo.findAll()
            .stream()
            .filter(Order::isActive)          // ③ unbound — order.isActive()
            .map(mapper::toDto)              // ② bound — mapper.toDto(order)
            .sorted(Comparator.comparing(OrderDto::getCreatedAt)) // ③ unbound
            .toList();
    }

    public List<Order> buildOrders(List<String> ids) {
        return ids.stream()
            .map(Order::new)               // ④ constructor ref
            .toList();
    }
}

Putting It All Together
Composing lambdas · Interview-ready mental model · Common pitfalls
The big picture

You now have all the pieces: lambdas compile to invokedynamic (not inner classes), they can only capture effectively final variables (because stack frames die), they need a SAM interface as their type, the standard library gives you 5 ready-made shapes (Function, Predicate, Consumer, Supplier, BiFunction), and method references are just shorthand lambdas in 4 flavors. All of this combines to make Java's functional-style stream pipelines work.

Function Composition

Chaining Functions with andThen() and compose()
Function<String, String> trim   = String::trim;         // ③ unbound
Function<String, String> upper  = String::toUpperCase;  // ③ unbound
Function<String, Integer> len   = String::length;       // ③ unbound

// andThen: left runs first, then right
Function<String, Integer> pipeline = trim.andThen(upper).andThen(len);
pipeline.apply("  hello  ");  // "  hello  " → "hello" → "HELLO" → 5

// Predicate composition
Predicate<String> notNull  = s -> s != null;
Predicate<String> notBlank = s -> !s.isBlank();
Predicate<String> valid    = notNull.and(notBlank);
Predicate<String> invalid  = valid.negate();

List<String> input = List.of("hello", "", null, "world");
input.stream().filter(valid).toList();  // ["hello", "world"]

Key difference from anonymous inner class: no .class file on disk, no guaranteed heap allocation (JVM decides). Capturing a variable copies its value to the heap into the lambda object. Only effectively-final values allowed because the original stack frame may be gone by the time the lambda runs. The lambda's type is always a SAM interface — @FunctionalInterface enforces the one-method contract at compile time.

Common Pitfalls to Know

⚠️ 1. Checked exceptions in lambdas
// Standard Function doesn't declare checked exceptions:
Function<String, String> read = path -> Files.readString(Path.of(path));
// ❌ Compile error: Unhandled IOException

// Workaround: wrap in a helper or use a custom functional interface
@FunctionalInterface
interface ThrowingFunction<T, R> {
    R apply(T t) throws Exception;
}

// Or wrap inline:
Function<String, String> safeRead = path -> {
    try { return Files.readString(Path.of(path)); }
    catch (IOException e) { throw new RuntimeException(e); }
};
⚠️ 2. Serialization and lambdas
// Lambdas are NOT serializable by default.
// If you store a lambda in a serializable object — watch out.
// Workaround (use sparingly — lambdas generally shouldn't be serialized):
Serializable & Runnable r = () -> System.out.println("ok");
// Cast to both interfaces simultaneously — valid in Java
⚠️ 3. this inside a lambda vs anonymous inner class
class OrderProcessor {
    String name = "Hemanth";

    void process() {
        // Anonymous inner class: 'this' = the anonymous class instance
        Runnable anon = new Runnable() {
            public void run() { System.out.println(this); } // Runnable instance
        };

        // Lambda: 'this' = the enclosing OrderProcessor instance
        Runnable lambda = () -> System.out.println(this.name); // "Hemanth"
        // Lambdas don't introduce a new 'this' scope. Very useful!
    }
}
Primitives — avoid boxing with specialised interfaces
// Function<Integer, Integer> boxes/unboxes — avoid in hot paths
Function<Integer, Integer> boxed   = x -> x * 2;  // Integer → int → Integer

// Prefer primitive specialisations from java.util.function:
IntUnaryOperator           noBox   = x -> x * 2;  // int → int, zero boxing
IntToLongFunction          widen   = x -> x * 2L; // int → long
ToIntFunction<String>      strlen  = String::length;
IntSupplier                rand    = () -> 42;
// Rule: any time you see Function<Integer,...> in a hot loop,
// reach for the Int*/Long*/Double* variants instead.

Cheat Sheet
⚡ Lambdas ≠ anonymous inner classes — no .class file on disk; compiled to a private static method + invokedynamic call-site
🔗 invokedynamic — defers the "how" to runtime; LambdaMetafactory bootstraps once; non-capturing lambdas can be singletons
📸 Effectively final — captured variables are copied to heap; stack frame may be gone when lambda runs; mutation would cause unsynchronized copies
🧩 SAM = Single Abstract Method — any interface with exactly one abstract method is a functional interface; @FunctionalInterface enforces this at compile time
🔧 The Big 5: Function<T,R> (transform) · Predicate<T> (test) · Consumer<T> (use, no return) · Supplier<T> (produce, no input) · BiFunction<T,U,R> (two-input transform)
🔀 Method refs: ① Class::staticMethod ② obj::method (bound) ③ Class::instanceMethod (unbound, first arg = this) ④ Class::new (constructor)
⚠️ Pitfalls: checked exceptions need wrapping · this = enclosing class (not the lambda) · use primitive specializations (IntFunction etc.) to avoid boxing
