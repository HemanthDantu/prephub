Creational Patterns (Singleton, Factory, Builder)

Creational patterns help you control how objects are made, especially when setup is complex or should be consistent across the app.

🧒 ELI5

One toy, one rulebook

Imagine a toy store where certain toys must always be built the same way. You use a simple rulebook so every toy comes out correct.

🔗 ANALOGY

Restaurant prep station

Instead of every cook mixing ingredients differently, a prep station creates the ingredients the same way so the kitchen stays consistent.

Technical

What it means in real code

Definition: Creational patterns centralize and control object creation so callers do not need to know construction details.

Why it matters

  • Keeps object setup consistent
  • Hides complex construction
  • Improves testability

Common smell

  • Newing up many variants in one class
  • Constructor parameters exploding
  • Duplicate setup logic
Factory for variantsBuilder for optional fieldsSingleton with care
Need object→Choose creator→Build safely→Return ready instance

Interview answer

“I use Factory when I need multiple implementations behind a stable interface, and Builder when a constructor would become unreadable or error-prone.”

Best fitAvoid when
Many variantsOnly one type forever
Optional paramsRigid constructor
Shared configHidden global state
Factory + Builder example
public class UserBuilder {
  private String name;
  private String role;
  public UserBuilder name(String name){ this.name = name; return this; }
  public UserBuilder role(String role){ this.role = role; return this; }
  public User build(){ return new User(name, role); }
}

public class UserFactory {
  public static UserBuilder builder(){ return new UserBuilder(); }
}

Callers get a clean API and never touch the messy construction details.

Warning

Singletons hide global state

Singletons are easy to overuse. Prefer DI-managed singletons so you can test and swap implementations.

Mini takeaway
  • Factory hides creation choices.
  • Builder makes complex construction readable.
  • Singletons should be rare and explicit.
Quiz

Which pattern best fits a class with 8 optional constructor fields?

Correct answer: B Builder is ideal when many optional fields would make a constructor unreadable.
Section 2 of 5 · ~7 mins

Behavioral Patterns (Strategy, Observer)

Behavioral patterns define how objects collaborate and how behavior can change without rewriting core logic.

🧒 ELI5

Pick your game plan

If you play soccer or basketball, you pick a different strategy. You still have the same team, but the plan changes.

🔗 ANALOGY

News subscription

A news app sends updates to all subscribers. Each subscriber can react differently without changing the app.

Technical

What it means in real code

Definition: Behavioral patterns separate actions and communication so the same system can behave differently without big rewrites.

Why it matters

  • Swap behavior at runtime
  • Reduce giant if/else trees
  • Decouple event producers from consumers

Common smell

  • Large conditional blocks
  • Tight coupling between sender and receiver
  • Copy/pasted behavior
Strategy for algorithmsObserver for eventsHard-coded branching
Need behavior→Select strategy→Execute→Notify observers

Interview answer

“Strategy lets me plug different algorithms behind one interface, and Observer lets producers broadcast events without knowing consumers.”

PatternWhen to use
StrategyRuntime algorithm swap
ObserverEvent fan-out
Switch chainsGrowing brittle logic
Strategy example
public interface PricingStrategy {
  double price(Order order);
}

public class SurgePricing implements PricingStrategy {
  public double price(Order order){ return order.base() * 1.2; }
}

Adding a new strategy never changes the caller — you just inject a new implementation.

Warning

Don’t over-abstract early

Use Strategy when you actually have variation. One-off behavior does not need a pattern.

Mini takeaway
  • Strategy swaps algorithms cleanly.
  • Observer decouples sender and listeners.
  • Patterns are tools, not mandatory.
Quiz

You need to notify multiple systems when an order ships. Best fit?

Correct answer: B Observer handles one-to-many event notifications without tight coupling.
Section 3 of 5 · ~7 mins

Structural Patterns (Decorator)

Structural patterns focus on how objects are composed so features can be added without exploding class hierarchies.

🧒 ELI5

Layer your outfit

If it’s cold, you add a jacket, not a whole new body. You stack layers instead of replacing the base.

🔗 ANALOGY

Filters on a photo

You can apply multiple filters to a photo without changing the original. Each filter wraps the previous one.

Technical

What it means in real code

Definition: Decorator wraps an object to add behavior dynamically without modifying the original class.

Why it matters

  • Avoid subclass explosion
  • Mix features at runtime
  • Keep core class simple

Common smell

  • Dozens of subclasses for combos
  • Base class growing flags
  • Hardcoded feature mix
Composable featuresRuntime wrappingSubclass explosion
Base component→Wrap feature A→Wrap feature B→Use composed result

Interview answer

“Decorator lets me add behavior by wrapping objects instead of creating a subclass for every feature combination.”

ApproachTrade-off
DecoratorFlexible composition
Subclass per comboExponential growth
Flags in baseHard to maintain
Decorator example
public interface Notifier {
  void send(String msg);
}

public class EmailNotifier implements Notifier {
  public void send(String msg){ /* email */ }
}

public class SmsDecorator implements Notifier {
  private final Notifier base;
  public SmsDecorator(Notifier base){ this.base = base; }
  public void send(String msg){ base.send(msg); /* sms */ }
}

You can stack decorators to add SMS, Slack, or audit logging without changing EmailNotifier.

Warning

Too many wrappers can confuse

Decorator chains can be hard to trace. Keep names clear and avoid deep nesting.

Mini takeaway
  • Decorator composes features.
  • Avoid subclass explosion.
  • Wrap when you need runtime flexibility.
Quiz

Which technique avoids creating 12 subclasses for feature combinations?

Correct answer: A Decorator composes features by wrapping objects instead of subclassing.
Section 4 of 5 · ~7 mins

Pattern Selection & Trade-offs

Interviewers often care more about why you chose a pattern than the pattern name itself.

🧒 ELI5

Use tools that fit the job

You wouldn’t use a hammer for every task. Pick the tool that solves the problem with the least fuss.

🔗 ANALOGY

Choosing a route

Sometimes a highway is faster, sometimes a side street is better. The context decides, not the label.

Technical

What it means in real code

Definition: Pattern choice is a trade-off between flexibility, complexity, and risk of change.

Why it matters

  • Explains design intent
  • Avoids over-engineering
  • Helps teammates reason quickly

Common smell

  • Pattern everywhere
  • Adding layers without variation
  • No clear extension point
Problem-drivenPremature abstractionPattern for resume
Define problem→Check variation→Pick pattern→Review trade-off

Interview answer

“I choose patterns when they reduce change risk or clarify intent. If no variation exists, I keep it simple.”

QuestionAnswer
Do we expect variants?Yes → Strategy/Factory
Need runtime composition?Decorator
No variation?Keep it simple
Simple decision stub
public Pattern pick(boolean variants, boolean compose){
  if (compose) return Pattern.DECORATOR;
  if (variants) return Pattern.STRATEGY;
  return Pattern.NONE;
}

Showing a decision framework makes your interview answer stronger than just naming patterns.

Warning

Avoid pattern bingo

Interviewers dislike forced patterns. Ground your answer in the problem and trade-offs.

Mini takeaway
  • Patterns communicate intent.
  • Trade-offs matter more than labels.
  • Keep solutions minimal when possible.
Quiz

What is the strongest reason to use a pattern?

Correct answer: B Patterns are valuable when they reduce risk and clarify extension points.
Section 5 of 5 · ~7 mins

Deep Dive: Choosing Patterns Under Constraints

Map the problem to variation points, then pick the lightest pattern that reduces change risk.

🧒 ELI5

Pick the simplest tool

If you just need to cut paper, you do not pull out a chainsaw. You use scissors.

🔗 ANALOGY

Kitchen utensils

Use a whisk for eggs, not a power drill. The right tool is the smallest one that solves the job.

Technical

What it means in real code

Definition: Patterns are trade-offs. Choose them based on where change is likely, not because the pattern exists.

Why it matters

  • Keeps design flexible at the right seams
  • Avoids over-engineering
  • Improves maintainability

Common pitfall

  • Pattern for every class
  • Inheritance explosions
  • Tight coupling to concrete types
Variation pointsMinimal abstractionAvoid overdesign
Identify change→Select seam→Pick light pattern→Validate trade-off

Interview answer

“I pick patterns where I expect change: Strategy for algorithm swaps, Decorator for optional behavior, Factory for creation complexity.”

SignalPattern
Many algorithm variantsStrategy
Optional behavior layersDecorator
Complex constructionBuilder/Factory
Strategy injected through DI
interface PricingStrategy { int price(int base); }

class HolidayPricing implements PricingStrategy {
  public int price(int base){ return base - 20; }
}

class CheckoutService {
  private final PricingStrategy strategy;
  CheckoutService(PricingStrategy strategy){ this.strategy = strategy; }
  int total(int base){ return strategy.price(base); }
}
Warning

Beware pattern gravity

Once you add a pattern, teams tend to expand it. Start small and refactor when the variation is real.

Mini takeaway
  • Patterns model change, not complexity.
  • Prefer composition over inheritance.
  • Leave escape hatches for future changes.
Quiz

What is the best reason to introduce a pattern?

Correct answer: B Patterns are justified when they reduce a concrete change risk or clarify extension points.

1-line interview answers

Creational

Choose Singleton/Factory/Builder to control creation and remove messy constructors.

Behavioral

Use Strategy/Observer to swap behavior and broadcast events cleanly.

Structural

Decorator composes features without subclass explosion.

Trade-offs

Patterns reduce change risk but add indirection — justify them.

What to remember

Fast mental model

  • Creational: control object creation.
  • Behavioral: coordinate behavior and communication.
  • Structural: compose objects safely.
  • Interview move: name the pattern and the trade-off.
