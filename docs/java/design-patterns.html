Creational Patterns (Singleton, Factory, Builder)

Creational patterns answer one question: how do we construct objects safely when creation logic becomes repetitive, conditional, or easy to misuse?

🧒 ELI5

A school with one admissions desk and standard forms

Imagine a school where every student must be registered using the right form, the right defaults, and the right classroom assignment. You would not let every teacher invent the process from scratch. A single admissions desk gives consistency, and a checklist helps build the final student record correctly. Creational patterns solve that kind of problem in code.

🔗 ANALOGY

A car factory versus a backyard garage

A factory knows exactly how to build different car variants from the same platform. Buyers ask for a type, and the factory handles the wiring, engine choice, and safety defaults. Without that central creation step, assembly logic gets scattered through the codebase and becomes painful to change later.

Technical

Factory, Builder, and Singleton in real systems

Definition: Creational patterns centralize object construction so callers do not own complex setup details.

Use Factory when the caller wants “the right implementation” without knowing which concrete class should be built. Use Builder when an object has many optional fields or construction steps that would make constructors unreadable. Use Singleton only when one logical instance is truly required and hidden global state is acceptable.

In Spring applications, DI often plays the role people used to solve with Singletons manually. That is usually safer because Spring-managed singletons can still be mocked and replaced in tests.

Why it matters in production

Centralized construction reduces duplication, prevents invalid object states, and makes later changes localized instead of scattered.

Trade-off

Extra abstraction adds files and indirection. Use it when creation complexity is real, not imagined.

PatternBest use
FactoryChoose the correct implementation at runtime.
BuilderCreate complex objects with optional data and validation.
SingletonOne shared instance, but be careful with hidden global state.

💬 Model Interview Answer

Creational patterns are about controlling object creation. I reach for Builder when constructors become unreadable, Factory when callers should not know which concrete class to instantiate, and I avoid manual Singletons unless there is a genuine one-instance requirement. In Spring, DI-managed beans often remove the need for handwritten Singleton code. The trade-off is added abstraction, so I only use these patterns when construction complexity is real.

❌ Wrong: callers own every construction detail
NotificationClient client = new NotificationClient(
    "smtp.example.com",
    587,
    true,
    Duration.ofSeconds(2),
    Duration.ofSeconds(5),
    new RetryPolicy(3)
);
✅ Right: builder validates and keeps construction readable
NotificationClient client = NotificationClient.builder()
    .host("smtp.example.com")
    .port(587)
    .tlsEnabled(true)
    .connectTimeout(Duration.ofSeconds(2))
    .readTimeout(Duration.ofSeconds(5))
    .retryPolicy(new RetryPolicy(3))
    .build();
Warning

Manual Singleton is often a smell in DI-based apps

A handwritten Singleton can hide global mutable state, complicate tests, and quietly create coupling across the codebase. In Spring, prefer constructor injection into a bean with singleton scope unless you have a very specific reason not to.

Quiz

Which pattern is the best fit for constructing an object with many optional fields and validation rules?

AnswerB is correct because Builder makes complex construction readable and gives one place for validation. Observer is about notifications, Decorator is for wrapping behavior, and Strategy is for swapping algorithms. The interview memory hook is: “Builder handles optional construction complexity.”
Section 2 of 5 · ~7 mins

Behavioral Patterns (Strategy, Observer)

Behavioral patterns organize how objects choose behavior and react to events without turning core flows into giant conditionals.

🧒 ELI5

Choosing a different game plan for each match

A team might defend in one match and attack in another. The players are the same, but the plan changes depending on the opponent. That is what Strategy does. Observer is different: it is like fans, reporters, and scoreboards all reacting when the referee blows the whistle.

🔗 ANALOGY

A fire alarm connected to many listeners

One sensor detects smoke, but many things react: alarms sound, doors unlock, elevators stop, and the fire department is notified. The detector should not hard-code every one of those behaviors. Observer exists so the event producer stays simple while listeners can change independently.

Technical

Separating decisions from core workflow

Definition: Behavioral patterns control how objects interact and how behavior changes over time without rewriting the main flow.

Strategy is for swapping algorithms behind a stable interface, such as pricing, validation, fraud checks, or routing rules. Observer is for publishing events to interested listeners without coupling the producer to every consumer.

These patterns matter when the codebase is expected to grow. A checkout service with discount logic hard-coded in one switch statement will become brittle quickly. A publisher that directly invokes email, analytics, billing, and audit code will be painful to extend or test.

When Strategy helps

When you know variants will grow and you want to add new behavior without touching stable code.

When Observer helps

When one event should trigger many independent reactions such as notifications, metrics, and workflows.

PatternTypical backend example
StrategyDifferent discount or fraud-scoring algorithms.
ObserverOrder placed event fan-out to email, metrics, and inventory listeners.

💬 Model Interview Answer

Strategy lets me plug in different algorithms behind one interface, which is great for rules that evolve over time. Observer lets one event trigger many independent reactions without tightly coupling the producer to its consumers. The main trade-off is indirection — the code becomes easier to extend, but you need clear naming and tracing so the behavior stays discoverable.

✅ Strategy example
interface PricingStrategy {
    Money price(Order order);
}

final class HolidayPricingStrategy implements PricingStrategy {
    @Override
    public Money price(Order order) {
        return order.basePrice().minus(Money.of(20));
    }
}

final class CheckoutService {
    private final PricingStrategy pricingStrategy;

    CheckoutService(PricingStrategy pricingStrategy) {
        this.pricingStrategy = pricingStrategy;
    }

    Money totalFor(Order order) {
        return pricingStrategy.price(order);
    }
}
✅ Observer-style event publishing
record OrderPlacedEvent(UUID orderId) {}

interface OrderPlacedListener {
    void onOrderPlaced(OrderPlacedEvent event);
}

final class MetricsListener implements OrderPlacedListener {
    public void onOrderPlaced(OrderPlacedEvent event) {
        System.out.println("Increment metric for " + event.orderId());
    }
}
Warning

Do not create a pattern when you have only one behavior and no evidence of change

Patterns should reduce change risk, not inflate class count for no reason. If there is genuinely only one algorithm and no sign of future variants, a simple method may be the better design today.

Quiz

A service must support multiple pricing algorithms chosen at runtime. Which pattern fits best?

AnswerA is correct because Strategy cleanly swaps algorithms behind one interface. Singleton is about one instance, Builder is about object construction, and Adapter changes interfaces. Interview takeaway: if behavior varies, think Strategy.
Section 3 of 5 · ~7 mins

Structural Patterns (Decorator)

Structural patterns are about composition. Decorator is especially important because it adds behavior without exploding subclass hierarchies.

🧒 ELI5

Putting layers on a winter outfit

You start with regular clothes, then add a jacket, scarf, and gloves depending on the weather. You do not create a brand-new human for every weather combination. Decorator works the same way: keep a simple base object, then wrap extra behavior around it as needed.

🔗 ANALOGY

Applying filters to a photo one by one

A photo editor can add brightness, contrast, blur, and watermark filters in sequence. Each filter wraps the result of the previous one. That is easier than creating a separate image class for every possible combination of features.

Technical

Why Decorator beats subclass explosion

Definition: Decorator wraps an object that implements a shared interface and adds behavior before or after delegating to the wrapped instance.

Decorator is valuable when optional features can be combined in many ways. Logging, caching, retries, rate limiting, encryption, and compression are classic examples. Without Decorator, teams often create a subclass per combination, which grows out of control.

The trade-off is discoverability: once wrappers stack, debugging requires understanding the order of decoration. That means naming and composition need to be explicit.

Why it matters

Lets you add behavior dynamically while keeping the base component simple and stable.

Main trade-off

Too many wrappers can make the execution path harder to trace unless composition is clear.

ApproachOutcome
Subclass per feature comboClass explosion and duplication.
Flags inside one classLarge conditional logic and feature tangling.
DecoratorComposable behavior with stable interfaces.

💬 Model Interview Answer

Decorator is useful when behavior is optional or combinable, like logging plus retries plus metrics around a client. Instead of subclassing for every combination, I wrap a shared interface and compose behavior at runtime. The trade-off is that deep wrapper chains can be harder to trace, so I keep the composition explicit and well named.

✅ Decorator example
interface FileStore {
    byte[] read(String key);
}

final class S3FileStore implements FileStore {
    public byte[] read(String key) {
        return key.getBytes();
    }
}

final class LoggingFileStore implements FileStore {
    private final FileStore delegate;

    LoggingFileStore(FileStore delegate) {
        this.delegate = delegate;
    }

    public byte[] read(String key) {
        long start = System.nanoTime();
        try {
            return delegate.read(key);
        } finally {
            System.out.println("read(" + key + ") took " + (System.nanoTime() - start));
        }
    }
}
Warning

Decorator is not a license to hide complexity

If wrappers are assembled in many places with different orderings, behavior becomes unpredictable. Centralize composition, document wrapper order, and prefer integration tests that exercise the final decorated chain.

Quiz

Why is Decorator usually preferred over many subclasses for optional features?

AnswerC is correct because Decorator uses composition to add optional behavior around a common interface. A and B are false, and D is not guaranteed. The real benefit is maintainability: fewer rigid subclasses and more flexible combinations.
Section 4 of 5 · ~7 mins

Pattern Selection & Trade-offs

Senior interviews care less about whether you memorized pattern names and more about whether you can justify a design choice under changing requirements.

🧒 ELI5

Picking the right tool from a toolbox

You do not use a chainsaw to open a cardboard box. Good builders choose the simplest tool that handles the job safely. Pattern selection is the same: use the lightest abstraction that solves the real problem, not the fanciest one.

🔗 ANALOGY

Choosing the right route for traffic conditions

Sometimes the highway is best, sometimes the local road is better. The right answer depends on traffic, not on a favorite road. Patterns should be selected based on likely change, risk, and complexity — not on personal preference.

Technical

Start with the change you expect

Definition: Pattern selection is the design decision of choosing the smallest abstraction that reduces future change cost and improves clarity.

A good pattern answer begins with the problem: “I expect multiple algorithm variants,” or “construction rules are spreading,” or “one event should trigger multiple independent side effects.” The pattern is just the response to that pressure.

That is why over-engineering is dangerous. Adding Strategy, Factory, and Observer to code with no real variation only adds files and indirection. Refactor into a pattern when change becomes likely or duplication becomes costly.

Healthy signal

The pattern reduces duplication, creates a clear extension point, and keeps the stable code mostly unchanged.

Bad signal

The pattern exists only to “look advanced,” and nobody can explain the concrete variation it protects against.

QuestionDesign hint
Will behavior vary?Consider Strategy or Observer.
Is creation messy?Consider Builder or Factory.
Need optional feature layers?Consider Decorator.
No real variation yet?Keep it simple.

💬 Model Interview Answer

I choose patterns based on expected change, not on memorization. If I expect algorithm variants, I lean toward Strategy; if construction logic is spreading, I use Builder or Factory; if behavior is layered, Decorator helps. If none of those pressures are real yet, I keep the design simple. Interviewers usually like hearing the problem-to-pattern mapping more than the pattern name alone.

Warning

Pattern bingo is a red flag

If you answer every design question with a named pattern before describing the problem, it sounds memorized. Lead with the change risk, the design pressure, and the trade-off. Then mention the pattern as the mechanism, not the headline.

Quiz

What is the strongest reason to choose a design pattern?

AnswerC is correct because patterns are worthwhile when they reduce future change cost and make extension points explicit. A and B are superficial, and D is wrong because abstractions still require tests. For interviews, tie patterns to maintainability and change isolation.
Section 5 of 5 · ~7 mins

Deep Dive: Choosing Patterns Under Constraints

Great design is rarely about applying a textbook pattern in pure form. It is about adapting the idea to constraints like deadlines, team familiarity, testability, and system growth.

🧒 ELI5

Packing for a short trip versus a year abroad

If you are leaving for one night, you pack lightly. If you are moving for a year, you plan much more carefully. Software design works the same way: the right amount of abstraction depends on how much future change and complexity you genuinely expect.

🔗 ANALOGY

Building a temporary stage versus a permanent theater

A temporary stage can be simple and fast to assemble. A permanent theater needs stronger structure, better access paths, and room for future upgrades. Patterns should scale with the expected lifespan and change rate of the code.

Technical

Balancing elegance with team reality

Definition: Constraints-driven design means choosing patterns that fit not just the code problem but also the operational and human context around the code.

Sometimes the technically pure pattern is not the best immediate choice. A team unfamiliar with a complicated abstraction may move slower and introduce bugs, while a simpler design could be more effective in the short term.

A senior engineer should be able to say, “I could use Strategy here, but today there are only two stable cases, so I will keep a simple branch and refactor when the third variant appears.” That is not weakness — it is judgment.

Use a stronger pattern when

Change is likely, duplication is already appearing, tests need isolation, or multiple teams will extend the logic.

Delay the pattern when

The requirement is stable, the variation is hypothetical, or the team would pay too much complexity for too little return.

ConstraintEffect on choice
Deadline pressurePrefer simpler designs unless risk is high.
Frequent new variantsCreate extension points earlier.
Team unfamiliarityBias toward readability and explicitness.
Heavy testing needsPatterns that isolate dependencies become more valuable.

💬 Model Interview Answer

When I choose patterns, I also think about constraints beyond the code itself: how often the logic will change, how testable the solution needs to be, how familiar the team is with the abstraction, and how costly future refactors would be. A senior answer is not “always use the textbook pattern.” It is “pick the smallest abstraction that protects against the most likely change.”

Warning

The wrong abstraction can lock a team into complexity

Once a pattern spreads, it becomes hard to remove because other code starts depending on it. That is why “start simple, refactor when variation becomes real” is often the safest path.

Quiz

Which answer sounds most like strong senior judgment?

AnswerC is correct because it shows balance, change awareness, and restraint. A over-engineers, B underestimates abstraction, and D is not engineering judgment at all. In interviews, thoughtful trade-off language is often more impressive than the pattern catalog itself.

🎉 Design Patterns Cheat Sheet

Use this page as a fast interview recap before mock rounds or senior backend interviews.

✅

What to remember

Creational

Factory chooses implementations, Builder manages complex construction, and Singleton should be used carefully.

Behavioral

Strategy swaps algorithms; Observer fans out events without tight coupling.

Structural

Decorator adds optional layered behavior without subclass explosion.

Selection rule

Start with the design pressure or expected change, then choose the smallest useful abstraction.

Senior framing

Explain the problem, the risk of change, and the trade-off — then name the pattern.

Most common confusion points

Where people slip

“Patterns are always good.”

No — they are tools. Extra abstraction without real change pressure is just complexity.

“Singleton is the standard way to share state.”

In DI-based applications, manual Singleton often hides global state and hurts testability.

“The pattern name is the most important part of the answer.”

Usually the design pressure and trade-off matter more than the label.
