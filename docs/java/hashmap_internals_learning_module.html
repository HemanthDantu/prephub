Section 1 of 5 · ~6 mins

Hashing and bucket placement

HashMap is fast because it turns a key into a bucket location instead of scanning all entries.

🧒 ELI5

Mailboxes with number rules

Imagine every letter gets a number from its address, and that number tells you which mailbox to put it in. You do not search every mailbox. You use the number to jump near the right place.

🔗 ANALOGY

Spread things out before you store them

A good hashing function spreads keys across buckets so traffic is balanced. If too many keys land in the same bucket, the fast path disappears and the bucket gets crowded.

Technical

From key to array index

HashMap stores entries in an array of buckets. For a key, Java computes a hash, mixes high bits into low bits, then derives a bucket index. In many implementations this is effectively based on (n - 1) & hash when capacity is a power of two.

Fast path

  • Compute hash from key
  • Map hash to bucket index
  • Search only inside that bucket

Why mixing matters

  • Low bits alone may cluster badly
  • Bit spreading improves distribution
  • Better distribution means fewer collisions
Interview answer

HashMap uses hashing to convert a key into a bucket index in an internal array. A good distribution keeps most lookups near O(1) because only one bucket, not the whole map, needs to be checked.

Java example

Map<String, Integer> scores = new HashMap<>();
scores.put("alice", 91);
scores.put("bob", 87);

Integer value = scores.get("alice");

// Conceptually:
// 1) hash("alice")
// 2) map hash to bucket index
// 3) search only that bucket for the matching key

The actual JDK code is optimized and more nuanced, but this 3-step story is the mental model interviewers expect.

Quiz

Why does HashMap need hashing at all?

Why: Hashing is the shortcut that helps HashMap jump to a bucket instead of scanning every entry. It reduces work, but it does not eliminate collisions.
Mini takeaway

What to remember

  • HashMap uses an internal bucket array.
  • A key becomes a hash, then a bucket index.
  • Good distribution is the foundation of average O(1) behavior.
Section 2 of 5 · ~6 mins

Collision handling

Different keys can still land in the same bucket. That is a collision, and handling collisions well is what keeps the map practical.

🧒 ELI5

Two people share the same mailbox slot

Sometimes two letters get sent to the same mailbox number. You do not throw one away. You keep a small list inside that mailbox and check which letter belongs to which person.

🔗 ANALOGY

A parking lot row, not a single spot

A bucket is not one parking spot. It is more like a row. If several cars hash to the same row, you inspect only that row to find the right car.

Technical

What happens when two keys collide

A collision means two keys map to the same bucket index. HashMap stores multiple entries in that bucket and then compares hashes and keys to locate the exact match.

Lookup inside a bucket

  • Check entry hash first
  • Then compare keys with equals()
  • Continue until match or end

Why equals and hashCode both matter

  • Equal objects must produce same hashCode
  • hashCode narrows the search
  • equals confirms exact identity
Interview answer

Collisions are expected. HashMap handles them by storing multiple entries in the same bucket and then resolving the exact key with hash and equals checks.

Java example

@Override
public int hashCode() {
    return accountId.hashCode();
}

@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof UserKey other)) return false;
    return accountId.equals(other.accountId);
}

A bad hashCode/equals contract can make lookups incorrect or slow. Interviewers often probe this immediately after asking about collisions.

Warning

Bad key design hurts maps badly

If a custom key has a poor hashCode or broken equals implementation, you can get heavy collisions, failed lookups, or subtle bugs.

Collision does not mean failure. It means multiple entries share one bucket and must be searched locally.

Quiz

When two different keys land in the same bucket, what does HashMap do?

Why: Collisions are normal. The map stores multiple entries in the same bucket and then checks hashes and equals to find the correct key.
Mini takeaway

What to remember

  • Collisions are expected, not exceptional.
  • hashCode narrows, equals confirms.
  • Good custom key implementations matter.
Section 3 of 5 · ~6 mins

Load factor and capacity

HashMap cannot keep stuffing entries into a small table forever. Load factor is the knob that decides when the map should grow.

🧒 ELI5

How crowded can the shelves get?

If too many items are stuffed into too few bins, every bin becomes messy. So you set a crowding rule: once the shelves are too full, bring in a bigger shelf.

🔗 ANALOGY

Traffic density

The more cars per lane, the more likely traffic jams become. Load factor is a density threshold. It balances memory usage against collision frequency.

Technical

Threshold before resize

Load factor is typically 0.75 by default. Threshold is roughly capacity × loadFactor. When size exceeds that threshold, the table resizes.

Higher load factor

  • Better memory utilization
  • More crowded buckets
  • Potentially slower lookups

Lower load factor

  • More space overhead
  • Fewer collisions
  • Potentially faster lookups
Interview answer

Load factor controls how full the hash table is allowed to get before resizing. Java's default of 0.75 is a balance between memory overhead and lookup performance.

Java example

// initial capacity = 16, load factor = 0.75
Map<String, Integer> map =
    new HashMap<>(16, 0.75f);

// threshold ≈ 16 * 0.75 = 12
// when size grows beyond threshold, resize is triggered

If you know the map will hold many entries, choosing a better initial capacity can reduce costly resizes.

Mini takeaway

What to remember

  • Load factor balances memory and collision rate.
  • Default 0.75 is a practical trade-off.
  • Initial capacity matters when you expect many entries.
Section 4 of 5 · ~6 mins

Resizing and rehashing

Resize is one of the most important hidden costs in HashMap. Average O(1) is real, but some puts can still be expensive when growth happens.

🧒 ELI5

Moving to a bigger shelf

When the shelf gets too full, you cannot just bolt on one extra slot in the middle. You often need a bigger shelf and then move many items over.

🔗 ANALOGY

A warehouse expansion day

Most days are routine. But on expansion day, inventory gets redistributed into a bigger warehouse layout. That one day costs more, but it keeps future operations efficient.

Technical

Why resize is expensive but acceptable

When threshold is exceeded, HashMap allocates a larger table, usually doubling capacity, and redistributes existing entries into new buckets. Because this does not happen on every insert, put remains amortized O(1) on average.

What changes

  • Capacity usually doubles
  • Entries move to old index or old index + old capacity
  • Future collisions reduce due to more buckets

Why interviewers care

  • Explains amortized behavior
  • Shows why initial capacity tuning helps
  • Separates average case from worst case
Interview answer

Resize is the expensive maintenance step. HashMap occasionally grows the bucket array and redistributes entries, so individual puts can spike, but overall average insertion remains amortized O(1).

Java example

Map<Integer, String> map = new HashMap<>();

for (int i = 0; i < 100000; i++) {
    map.put(i, "v" + i); // most puts are cheap; some trigger resize
}

The average is still good, but hotspots can appear if you constantly grow a map from a tiny starting capacity.

Quiz

Why can put() still be called amortized O(1) even though resize is expensive?

Why: Amortized analysis spreads the occasional expensive resize across many cheap insertions, which is why average put performance remains O(1).
Mini takeaway

What to remember

  • Resizing is occasional but expensive.
  • Average insertion is amortized O(1), not always constant in every single call.
  • Pre-sizing helps for large expected maps.
Section 5 of 5 · ~6 mins

Java 8 tree bins

Long collision chains can degrade lookups. Java 8 introduced tree bins to protect against worst-case bucket behavior.

🧒 ELI5

A messy line becomes a mini map

If one mailbox row gets too crowded, instead of keeping one long line, you organize that row into a smarter structure so finding a letter becomes faster.

🔗 ANALOGY

From a long waiting line to a branching directory

A long linked list in one bucket is like a single-file line. A tree is like a branching directory. It is not free, but it avoids the worst pain of huge collision chains.

Technical

Why tree bins exist

In Java 8+, when a bucket becomes too large under certain conditions, HashMap can transform that bucket's linked list into a balanced tree structure. This improves worst-case lookup within that bucket from O(n) to O(log n).

Benefits

  • Protects against pathological collision patterns
  • Improves worst-case bucket lookup
  • Makes denial-of-service style collision attacks harder

Nuance

  • Treeification does not happen for every small chain
  • Capacity and threshold conditions matter
  • Average case is still usually simple bucket lookup
Interview answer

Java 8 added tree bins so that heavily collided buckets do not stay as long linked lists forever. Under the right threshold and capacity conditions, the bucket becomes a tree, improving worst-case lookup to O(log n).

Java example

// Conceptual only:
// bucket[5]: Node -> Node -> Node -> ...
// if chain gets too long and table is large enough:
// bucket[5]: TreeNode(root)

// You do not trigger this manually in normal code.
// It is an internal protection mechanism.

You usually mention tree bins when an interviewer asks about worst-case behavior or Java 8 improvements.

Mini takeaway

What to remember

  • Java 8 tree bins are an internal worst-case optimization.
  • They matter when discussing collision-heavy scenarios.
  • Average case is still based on well-distributed hashes and buckets.

HashMap Internals Cheat Sheet

You now have the story interviewers expect: how a key becomes a bucket index, what happens on collisions, when the map resizes, and why Java 8 sometimes turns long bucket chains into trees.

Cheat sheet

Fast scan summary

  • Hashing: A key becomes a hash, then a bucket index in an internal array.
    HashMap uses hashing to jump near the right storage location instead of scanning every entry.
  • Collisions: Different keys can land in the same bucket and are resolved locally.
    HashMap stores multiple entries in one bucket and uses hash plus equals to find the exact key.
  • Load factor: Controls how full the table gets before growth.
    Default load factor 0.75 balances memory cost and collision risk.
  • Resizing: Occasional expensive maintenance that redistributes entries.
    Resize makes single puts occasionally expensive, but average insertion is still amortized O(1).
  • Tree bins: Java 8 turns very crowded buckets into trees under certain conditions.
    That improves worst-case bucket lookup from O(n) to O(log n).
Common confusion

Separate the look-alikes

Average vs worst case

Average lookup is O(1) with good distribution. Worst case can degrade if collisions are bad, though Java 8 tree bins help.

hashCode vs equals

hashCode helps place and narrow candidates. equals confirms the actual matching key.

Resize vs rehashing story

You do not need to recite JDK source code. Explain threshold, larger table, redistributed entries, and amortized O(1).
