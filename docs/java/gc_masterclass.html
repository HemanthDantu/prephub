What Is Garbage Collection?
Heap anatomy · Object lifecycle · Why GC exists
The core idea

Imagine your bedroom. As you play, you create wrappers, cups, toys all over the floor. Eventually it gets so messy you can't find anything. So periodically, someone (the GC) walks in, picks up everything you're no longer touching, and throws it away.

In Java — every new MyObject() you write puts a "toy" on the floor (the heap). When no part of your program holds a reference to that object anymore, it becomes garbage. The JVM's garbage collector finds and reclaims that memory automatically, so you don't have to call free() like in C.

The apartment building analogy

The JVM heap is like an apartment building with three wings:

  • 🌱 Eden (Young Gen) — New tenant apartments. Most objects are born here. High turnover — most die quickly.
  • 🏠 Survivor Spaces (S0, S1) — Objects that survived one cleanup get moved here temporarily.
  • 🏛️ Old Gen (Tenured) — Long-term residents. Objects that survived many cleanups live here. Less frequent cleanup.

There's also a Metaspace — the building's blueprint room (class metadata, method bytecode). It lives in native memory, outside the heap.

Minor GC vs Major GC vs Full GC
Triggers · Scope · Cost · What gets collected
The core idea

Back to your bedroom. Sometimes you just quickly clean your desk (Minor GC — fast, frequent, small area). Sometimes you clean your whole room (Major GC — slower, bigger area). And sometimes your parents make you clean your entire home including the garage and attic (Full GC — everything stops, takes forever, everyone's annoyed).

GC Type What it cleans Triggered by Speed STW?
Minor GC Young Generation only (Eden + Survivors) Eden space is full Fast (ms) Short STW
Major GC Old Generation (sometimes overlaps with Full GC in docs) Old Gen fills up; promotion failure Slower (10x+) Longer STW
Full GC Entire heap + Metaspace + code cache Old Gen full, System.gc(), Metaspace full, promotion failure, explicit trigger Slow (seconds) Long STW ⚠️
Why Minor GC is so much faster

The JVM leverages the Weak Generational Hypothesis: most objects die young. Think about it — in a web request, you create dozens of short-lived objects (request bodies, DTOs, local variables). They're garbage before the response is even sent.

Because Eden is small and most of it is dead, Minor GC has very few live objects to copy out — it mostly just wipes the slate clean. That's why it's fast: less work to do.

What Triggers a Full GC? (Production-critical)

⚠️ Full GC triggers to watch for
  • Old Gen exhaustion — Objects promoted from Young Gen fill Old Gen completely
  • Promotion failure — Young Gen survivor objects can't fit into Old Gen during Minor GC
  • Metaspace full — Too many class definitions loaded (common in OSGi, reflection-heavy apps)
  • System.gc() call — Someone called it explicitly (almost always a bug)
  • Concurrent Mode Failure (CMS) — Background collection can't keep up with allocation rate
  • Humongous object allocation failure (G1) — Very large objects that don't fit in a region

A Real-World Scenario

What happens in a typical Spring Boot request
// During a POST /api/orders request...
// Each of these creates short-lived objects in Eden:
OrderRequest req = objectMapper.readValue(body, OrderRequest.class); // born in Eden
Order order = orderService.create(req);          // born in Eden
OrderResponse response = mapper.toResponse(order); // born in Eden
// After response is sent, all 3 objects become garbage
// → Next Minor GC wipes them. Old Gen untouched.

// BUT: if your app holds a static cache that keeps growing:
static Map<String, Order> cache = new HashMap<>(); // objects promoted to Old Gen
// Eventually Old Gen fills → Full GC → your app freezes for 5 seconds

Section 3 of 5 · ~8 mins
GC Algorithms
Mark-and-Sweep · Mark-Compact · Copying
The core idea

There are different strategies for cleaning your room. Some people pick up only the trash and leave everything in place (Mark-and-Sweep). Some people pick up trash and rearrange everything neatly on one side (Mark-Compact). And some people just move everything good to a brand new room and abandon the messy one (Copying). Each has trade-offs.

1. Mark-and-Sweep

Phase 1: Mark
→
Phase 2: Sweep
How it works

Mark: Starting from GC Roots (stack variables, static fields, JNI refs), traverse all reachable objects and mark them as "alive".

Sweep: Walk the entire heap. Any object not marked = garbage. Free that memory.

The Problem

After sweeping, free memory is scattered in fragments throughout the heap — like holes in Swiss cheese. A large new object might not fit even though total free memory is enough. This is memory fragmentation.

// After Mark-and-Sweep — fragmented memory:
// [OBJ-A][FREE][OBJ-C][FREE][FREE][OBJ-F][FREE][FREE]
// Need 3 contiguous blocks? Can't allocate even though 4 blocks free!

2. Mark-Compact

Phase 1: Mark
→
Phase 2: Compact
How it works

Mark (same as above). Then Compact: slide all live objects to one end of the heap, filling gaps. Now free memory is one contiguous chunk at the end.

// Before:  [OBJ-A][dead][OBJ-C][dead][dead][OBJ-F]
// After:   [OBJ-A][OBJ-C][OBJ-F][FREE FREE FREE FREE]

Pro: No fragmentation. New allocations are just a pointer bump (super fast). Con: Compaction is expensive — you have to move objects and update all references pointing to them.

Used in: Old Generation of many collectors (Serial, Parallel GC, G1 for Old Gen cleanup).

3. Copying (Semi-Space)

Phase 1: Find live objects in "From"
→
Phase 2: Copy to "To" space
→
Phase 3: Swap From/To
How it works

Split space into two halves: From and To. Only allocate in From. When it fills, copy all live objects to To (compacted automatically). Wipe From. Swap names.

// Eden is "From":  [OBJ-A][dead][dead][OBJ-D][dead][dead][OBJ-G]
// After Minor GC → Survivor "To": [OBJ-A][OBJ-D][OBJ-G] + bump pointer at end
// Eden wiped clean. Allocation in next Eden is now a single pointer increment!

Pro: Extremely fast allocation (pointer bump). Automatically compact. Con: Wastes 50% of the space (one half always empty). Fine for Young Gen (small), terrible for Old Gen (large).

Used in: Young Generation (Eden → Survivor is exactly the Copying algorithm). S0 and S1 alternate as From/To.

Section 4 of 5 · ~7 mins
G1GC — Garbage First
Regions · Why it's the default · How it achieves predictable pauses
The core idea

Old collectors had fixed neighborhoods: Young Gen was always on the left, Old Gen always on the right. G1 threw that out. Instead, it divides the heap into equal little regions, like a grid of city blocks. Each block can dynamically become "Eden," "Old," or whatever the JVM needs. And when cleaning, G1 is smart — it cleans the most garbage-filled blocks first (hence "Garbage First"). This gives much more predictable pause times.

The Region Grid

G1 heap: ~2048 equal-sized regions (1MB – 32MB each)
Hover over a region to see its type:
E = Eden
S = Survivor
O = Old
H = Humongous (≥half region)
_ = Free

Key insight: There's no hard boundary between Young and Old Gen anymore. The JVM assigns region roles dynamically based on what's needed. This avoids the "fixed size" problems of Serial/Parallel GC.

Mixed GC is where "Garbage First" shines: G1 picks the Old Gen regions with the most garbage (highest return on investment) and collects only those. You never have to collect the entire Old Gen at once.

Why is G1 the Default Since Java 9?

G1 vs Parallel GC (the old default)
FeatureParallel GC (Java 8 default)G1GC (Java 9+ default)

Most modern JVM apps run with 8GB–64GB heaps. Parallel GC was designed for smaller heaps. G1 scales far better and gives operators a way to bound worst-case pause time — critical for latency-sensitive services.

JVM Flags & Stop-the-World Pauses
-Xmx · -Xms · -XX:+UseG1GC · MaxGCPauseMillis · STW explained
Stop-the-World — The core idea

Imagine you're trying to count people in a busy train station while they're all moving. Impossible, right? So the JVM hits a giant pause button — all your application threads freeze in place. The GC thread then safely walks the heap, marks live objects, and moves things around. When done, everyone unfreezes. That freeze is called a Stop-the-World (STW) pause.

In production, this means: while GC is running, your API handles zero requests. A 2-second Full GC pause on a payment service = 2 seconds of timeouts and unhappy users.

What Causes STW?

Root causes
  • Root scanning — Must freeze to get a consistent view of GC roots (stack, statics)
  • Object relocation — When objects move (Copying / Compact), all references to them must be updated atomically — you can't do this while threads are reading those references
  • Write barriers — Some STW is needed to finalize concurrent marking (Remark phase in G1)

Modern collectors like G1 minimize STW by doing as much work concurrently as possible (marking happens while your app runs), but they can't eliminate it entirely for the relocation phase.

The Essential JVM Flags

-Xms<size>
Initial heap size. The JVM starts with this amount of memory.
-Xms512m or -Xms2g. Setting equal to -Xmx avoids heap resizing overhead at startup — common in production.
-Xmx<size>
Maximum heap size. The JVM will never grow beyond this.
-Xmx4g. If your app needs more, it throws OutOfMemoryError. Too high → long GC pauses. Too low → frequent GC.
-XX:+UseG1GC
Explicitly enable G1GC. (Default since Java 9, but useful to be explicit in config, or to force it on Java 8).
Pair with -XX:+PrintGCDetails -XX:+PrintGCDateStamps to log GC events.
-XX:MaxGCPauseMillis=
<ms>
G1's pause time target. Default is 200ms. G1 will try to keep STW pauses under this — it's a goal, not a hard guarantee.
-XX:MaxGCPauseMillis=100 for latency-sensitive services. Lowering this makes G1 collect in smaller increments more frequently.

Practical Example: Tuning for a Spring Boot Service

A real startup command
# Production-grade JVM flags for a Spring Boot microservice
java \
  -Xms2g -Xmx2g                       # Fixed heap size (no resizing overhead)
  -XX:+UseG1GC                          # Explicit G1 (good habit even on Java 17)
  -XX:MaxGCPauseMillis=200              # Target max pause time
  -XX:+UseStringDeduplication           # G1 bonus: dedup String objects
  -XX:+PrintGCDetails                   # Log GC events (send to Datadog/CloudWatch)
  -XX:+HeapDumpOnOutOfMemoryError       # Dump on OOM for analysis
  -XX:HeapDumpPath=/var/log/heapdump.hprof \
  -jar app.jar
⚠️ STW in production — things to watch for
  • GC pause > 500ms → health checks fail → load balancer removes instance from rotation
  • GC pause > database query timeout → cascading failures
  • Frequent Full GC = memory leak or undersized heap — always investigate
  • Monitor: jstat -gc <pid> 1000 or expose GC metrics via Micrometer/Actuator in Spring Boot

⚡ Minor GC = fast (Young only) · Full GC = expensive (entire heap)

🧹 Mark-Sweep (fragments) · Mark-Compact (slow, no frag) · Copying (fast, 50% waste)

🚩 -Xms -Xmx (heap size) · -XX:+UseG1GC · -XX:MaxGCPauseMillis (pause target)

⏸️ STW = all app threads frozen. Caused by root scanning + object relocation. Minimize by watching Full GC frequency.
