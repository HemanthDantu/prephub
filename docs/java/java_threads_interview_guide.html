☕ Java Threads & Multithreading

Interview Prep Guide — Core Concepts to Executors

1. Core Concepts

What is a Thread?

A thread is the smallest unit of execution within a process. Think of it like a worker in Kafka's consumer group—each worker (thread) can independently process messages (tasks) from a queue, and multiple workers can run in parallel.

🔄 Work Analogy: Imagine your SR Path funnel processing user signups. With 1 thread, you process one signup at a time. With 10 threads, you can handle 10 signups concurrently. But if they all try to increment the same counter without coordination, chaos ensues (race condition).

Thread vs Process

Aspect Process Thread Memory Separate address space Shared memory (same process) Creation Cost Very expensive Cheaper than processes Communication IPC (complex) Shared heap (careful!) Isolation Complete isolation Share code, heap, open files

CPU-Bound vs I/O-Bound Work

CPU-Bound: Pure computation. Threads can't help if limited by CPU cores. (Limit threads ≈ CPU cores)

I/O-Bound: Waiting for network, disk, database. More threads = parallelism. (Can have many more threads than cores)
CPU-Bound Example
long sum = 0; for (long i = 0; i < 1_000_000_000; i++) { sum += i; // Pure computation, CPU keeps working } // Adding more threads won't help (still limited by CPU)
I/O-Bound Example
// Waiting for database query User user = userRepository.findById(123); // Thread sleeps here // CPU is idle—other threads can work // So more threads = higher throughput

Context Switching & Overhead

When you have more threads than CPU cores, the OS rapidly switches between them. This is powerful but has cost:

  • Cache invalidation: CPU cache loses data when switching threads
  • Memory overhead: Each thread needs stack memory (~1MB)
  • Diminishing returns: Too many threads = thrashing, not parallelism
🎯 Work Analogy: More Kafka consumer threads = good for I/O. But if you spin up 1000 threads on 4 CPU cores, context switching becomes the bottleneck, not the work itself.

2. Thread Lifecycle & Basics

Creating Threads: 3 Approaches

Approach 1: Extend Thread (old, less flexible)

Extended Thread
class MyThread extends Thread { @Override public void run() { System.out.println("Thread running"); } } // Usage MyThread t = new MyThread(); t.start(); // CORRECT: calls run() in new thread // t.run(); // WRONG: runs in current thread!

Approach 2: Implement Runnable (preferred)

Runnable Implementation
class MyRunnable implements Runnable { @Override public void run() { System.out.println("Runnable task"); } } // Usage Thread t = new Thread(new MyRunnable()); t.start(); // Or with lambda (Java 8+) Thread t = new Thread(() → { System.out.println("Lambda task"); }); t.start();

Approach 3: Callable (returns result, throws checked exceptions)

Callable Implementation
class MyCallable implements Callable<Integer> { @Override public Integer call() throws Exception { return 42; } } // Use with ExecutorService (section 5) ExecutorService executor = Executors.newFixedThreadPool(2); Future<Integer> future = executor.submit(new MyCallable()); Integer result = future.get(); // Blocks until done

start() vs run() — The Most Common Mistake

⚠️ CRITICAL: Always call start(), NOT run()

start() → JVM creates new thread → calls run() in that thread
run() → executes in CURRENT thread (no parallelism!)

Key Thread Methods

Method What it does Blocking? start() Creates new thread, calls run() No join() Wait for thread to finish Yes sleep(millis) Sleep for N milliseconds Yes interrupt() Request thread to stop (cooperative) No isAlive() Check if thread still running No

join() Example: Waiting for Threads

Using join()
Thread t1 = new Thread(() → { System.out.println("T1 working..."); Thread.sleep(2000); }); t1.start(); System.out.println("Main continues immediately"); t1.join(); // Main BLOCKS here until t1 finishes System.out.println("T1 finished!");

Interrupt Handling (Cooperative Cancellation)

Interrupts don't force a thread to stop—they request it. The thread must check and respond.

Interrupt Handling
Thread worker = new Thread(() → { try { while (!Thread.currentThread().isInterrupted()) { do work... Thread.sleep(1000); } } catch (InterruptedException e) { // Restore interrupt status and exit Thread.currentThread().interrupt(); } }); worker.start(); worker.interrupt(); // Request stop

3. Race Conditions & Visibility

What is a Race Condition?

Two or more threads access shared data, at least one modifies it, and the result depends on timing. The outcome "races" to completion—unpredictable.

🏁 Work Analogy: Two Kafka consumers try to increment the same counter without coordination. Consumer A reads (value = 5), Consumer B reads (value = 5). Both add 1. Both write back 6. Result: 6 instead of 7. Data lost!

Classic Example: Unsynchronized Counter

Race Condition (BROKEN)
class Counter { private int count = 0; public void increment() { count++; // NOT atomic! 3 steps: // 1. READ count // 2. ADD 1 // 3. WRITE count } public int getCount() { return count; } } // Race condition demo Counter counter = new Counter(); for (int i = 0; i < 1000; i++) { new Thread(() → counter.increment()).start(); } // Expected: 1000, Actual: 950-1000 (varies!)

Visibility vs Atomicity

Atomicity: An operation completes without interruption. count++ is NOT atomic.

Visibility: Changes made by one thread are seen by others. Without synchronization, changes may stay in thread-local caches.
Problem Cause Solution Lost update Non-atomic read-modify-write synchronized, AtomicInteger Stale reads No visibility guarantee volatile, synchronization Out-of-order execution Compiler/CPU reordering Synchronization, volatile

volatile: Guarantees Visibility, NOT Atomicity

volatile ensures all threads see the latest value, but doesn't prevent race conditions.

volatile Example
class Flag { private volatile boolean running = true; public void stop() { running = false; // All threads see this immediately } public void loop() { while (running) { // Always reads fresh value do work... } } } // But for counters, volatile still fails (not atomic): private volatile int count; count++; // STILL NOT SAFE!

Happens-Before: Memory Visibility Guarantees

The Java Memory Model (JMM) defines happens-before relationships:

  • Synchronization: Lock release → lock acquire
  • volatile write → volatile read
  • Thread.start() → code in that thread
  • Thread code → Thread.join() return
Key insight: If your critical section is inside a synchronized block or guarded by a lock, all visibility issues are handled automatically.

4. Synchronization Primitives

synchronized Blocks & Methods

The simplest way to ensure atomicity and visibility. Acquiring a lock (monitor) prevents other threads from entering the synchronized block simultaneously.

synchronized Method
class Counter { private int count = 0; public synchronized void increment() { count++; // Now atomic! Lock held by one thread } public synchronized int getCount() { return count; } } // Now safe: count will be exactly 1000 after 1000 increments
synchronized Block (fine-grained locking)
class BankAccount { private int balance = 100; private int transactions = 0; public void withdraw(int amount) { synchronized (this) { if (balance >= amount) { balance -= amount; transactions++; // Both protected } } } } // Or synchronize on a different object (lock striping) private Object balanceLock = new Object(); synchronized (balanceLock) { balance -= amount; }

Intrinsic Locks & Monitor Concept

Every Java object has an intrinsic lock (monitor). When you use synchronized on an object, you're acquiring its lock.

🔐 Work Analogy: The lock is like a badge that lets only one worker access the shared counter at a time. Other workers wait in a queue. When the badge holder exits, the next worker in queue gets the badge.

Lock Interface (More Flexible)

synchronized is simple but inflexible. Lock provides more control:

ReentrantLock
import java.util.concurrent.locks.*; class Counter { private int count = 0; private Lock lock = new ReentrantLock(); public void increment() { lock.lock(); try { count++; } finally { lock.unlock(); // ALWAYS unlock in finally } } public int getCount() { return count; // Unprotected read (risky!) } } // Advantages over synchronized: // - tryLock(timeout): non-blocking attempt // - Multiple Condition objects for waiting // - Fair locking option

Deadlock, Livelock, Starvation

Deadlock: Two threads hold locks and wait for each other. Program hangs forever.

Example: Thread A holds lock1, needs lock2. Thread B holds lock2, needs lock1. Both wait infinitely.
Livelock: Threads are active but making no progress. They keep changing state in response to each other.

Example: Both threads keep releasing locks and retrying, but one always "gives way" to the other.
Starvation: A thread never gets access to a resource due to unfair scheduling.

Example: High-priority threads continuously run, low-priority threads never get CPU time.

How to Avoid Deadlock

  • Lock order: Always acquire locks in the same order
  • Minimize critical sections: Don't hold locks longer than needed
  • Use timeouts: lock.tryLock(1, TimeUnit.SECONDS)
  • Single lock: When possible, use one lock instead of multiple

5. Executors & Thread Pools

Why Thread Pools?

Creating a new thread for every task is expensive. Thread pools reuse threads, reducing overhead dramatically. Think of it as a Kafka consumer group with a fixed number of workers.

⚙️ Work Analogy: Instead of hiring a new worker for each signup, you hire 10 workers once. They wait for tasks in a queue. When a signup arrives, any idle worker picks it up. Much cheaper!

ExecutorService: The Main Interface

Basic ExecutorService Usage
ExecutorService executor = Executors.newFixedThreadPool(5); // Submit Runnable (no return value) executor.execute(() → System.out.println("Task")); // Submit Callable (with return value) Future<Integer> future = executor.submit(() → 42); Integer result = future.get(); // Blocks until ready // Shutdown executor.shutdown(); // No new tasks, wait for existing if (!executor.awaitTermination(10, TimeUnit.SECONDS)) { executor.shutdownNow(); // Force kill }

Common Executor Types

Type Use Case Pool Size newFixedThreadPool(n) General purpose, known workload Fixed n threads newCachedThreadPool() Many short-lived tasks Unbounded (⚠️ can exhaust memory) newSingleThreadExecutor() Sequential execution, 1 thread 1 thread newScheduledThreadPool(n) Delayed/periodic tasks Fixed n threads

ThreadPoolExecutor (Under the Hood)

ExecutorService is built on ThreadPoolExecutor. Understanding it helps with tuning.

Creating a Tuned ThreadPoolExecutor
ThreadPoolExecutor executor = new ThreadPoolExecutor( 5, // corePoolSize: threads kept alive 10, // maxPoolSize: max threads 60, TimeUnit.SECONDS, // keepAliveTime new LinkedBlockingQueue<>(100), // task queue new ThreadPoolExecutor.CallerRunsPolicy() // if queue full ); // Rejection policies: // - AbortPolicy: throw exception (default) // - CallerRunsPolicy: caller thread runs task // - DiscardPolicy: silently drop task // - DiscardOldestPolicy: drop oldest, add new

Pool Sizing Guidelines

  • I/O-bound (network, database): Use more threads. Formula: (tasks * expected blocking ratio) / CPU cores. Often 2-3x cores.
  • CPU-bound (computation): Use ~CPU cores. More threads = context switching overhead.
  • Web servers: Often 10-100 threads, tune based on latency/throughput tradeoffs.
For Kafka consumers (I/O-bound): More threads = more parallelism. But watch memory: each thread = ~1MB stack. 1000 threads = ~1GB just for stacks.

Future & Timeouts

Using Future with Timeout
ExecutorService executor = Executors.newFixedThreadPool(2); Future<String> future = executor.submit(() → { Thread.sleep(5000); return "Done"; }); try { String result = future.get(2, TimeUnit.SECONDS); // 2 second timeout } catch (TimeoutException e) { System.out.println("Timeout! Cancelling..."); future.cancel(true); // Interrupt if running }

6. Interview Patterns

Pattern 1: Producer-Consumer (Thread-Safe Queue)

Problem
Producer threads add items to a queue, consumer threads process them. Handle blocking when queue is empty or full.
Producer-Consumer with BlockingQueue
BlockingQueue<String> queue = new LinkedBlockingQueue<>(10); // Producer new Thread(() → { try { for (int i = 0; i < 100; i++) { queue.put("item-" + i); // Blocks if full Thread.sleep(100); } } catch (InterruptedException e) { Thread.currentThread().interrupt(); } }).start(); // Consumer new Thread(() → { try { while (true) { String item = queue.take(); // Blocks if empty System.out.println("Processing: " + item); } } catch (InterruptedException e) { Thread.currentThread().interrupt(); } }).start();

Pattern 2: Rate Limiter (Semaphore)

Problem
Limit concurrent access to a resource (e.g., max 5 concurrent DB queries, max 10 API calls/second).
Rate Limiter with Semaphore
Semaphore semaphore = new Semaphore(5); // Max 5 concurrent public void queryDatabase() { try { semaphore.acquire(); // Decrement, block if 0 System.out.println("Query running"); Thread.sleep(2000); // Simulate work } catch (InterruptedException e) { Thread.currentThread().interrupt(); } finally { semaphore.release(); // Increment, wake waiting thread } } // Usage: spawn 20 threads, only 5 run concurrently for (int i = 0; i < 20; i++) { new Thread(this::queryDatabase).start(); }

Pattern 3: Barrier (CountDownLatch / CyclicBarrier)

Problem
Wait for multiple threads to reach a point before continuing (e.g., all workers ready before processing starts).
CyclicBarrier (Reusable)
CyclicBarrier barrier = new CyclicBarrier(3, () → System.out.println("All workers ready! Starting...") ); for (int i = 0; i < 3; i++) { new Thread("Worker-" + i) { public void run() { System.out.println(Thread.currentThread().getName() + " ready"); try { barrier.await(); // Wait for all 3 System.out.println(Thread.currentThread().getName() + " processing"); } catch (Exception e) {} } }.start(); }

Pattern 4: Safe Singleton (Double-Checked Locking)

Lazy Singleton (Thread-Safe)
class Singleton { private static volatile Singleton instance; private Singleton() {} public static Singleton getInstance() { if (instance == null) { synchronized (Singleton.class) { if (instance == null) { instance = new Singleton(); // Create once } } } return instance; } } // Better: use eager initialization or bill Pugh class Singleton2 { private static class SingletonHelper { private static final Singleton2 INSTANCE = new Singleton2(); } public static Singleton2 getInstance() { return SingletonHelper.INSTANCE; } }

7. Advanced Topics: Virtual Threads, Memory, & Spring

A. Virtual Threads (Java 21+)

A game-changer for I/O-bound applications. Virtual Threads allow millions of lightweight threads instead of thousands.

What are Virtual Threads? Lightweight threads managed by the JVM, not the OS. They live on the Heap (a few hundred bytes) instead of requiring 1MB of native stack memory each.

Platform Threads vs Virtual Threads

Aspect Platform Threads (Standard) Virtual Threads (Java 21+) Memory per thread ~1 MB (native stack) ~250 bytes (on Heap) Concurrency limit ~2,000–5,000 (RAM constraint) 1,000,000+ (virtually unlimited) Context switching OS kernel involved (slow) JVM user-space (fast, ~1000x) Scheduler OS scheduler JVM scheduler (Project Loom) Blocking cost Thread idles, wastes 1MB Thread unmounts, core reused immediately

How Virtual Threads Work

Carrier Threads: Virtual threads don't run directly on CPU cores. Instead, they run on top of a small number of "Carrier Threads" (typically one per CPU core).

🚀 Work Analogy: Imagine a Kafka consumer group. Instead of 200 dedicated workers (Platform Threads), you have 10 workers (Carrier Threads = cores). Each worker can grab 1000s of tasks (Virtual Threads) and batch them. When a task blocks (I/O), the worker instantly grabs another ready task. The same 10 workers handle millions of tasks efficiently.
Creating Virtual Threads (Java 21+)
// Platform Thread (old way) Thread t = new Thread(() → { System.out.println("Task"); }); t.start(); // Virtual Thread (new way) Thread vt = Thread.ofVirtual() .name("worker") .start(() → { System.out.println("Virtual task"); }); // Or use ExecutorService ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor(); executor.execute(() → { System.out.println("Unlimited VT tasks"); });

Virtual Threads in Spring Boot 3.2+

Enable Virtual Threads in application.properties
# Spring Boot 3.2+ spring.threads.virtual.enabled=true

When enabled, Tomcat's thread pool switches to virtual threads. You can now safely increase the pool size to thousands.

Memory Calculation for Thread Pools

Memory for Platform Threads: threads × 1 MB = native memory

200 threads × 1 MB = 200 MB just for stacks (before any objects!)

Memory for Virtual Threads: threads × 250 bytes = manageable

10,000 virtual threads × 250 bytes = ~2.5 MB (vs 10 GB for platform threads!)
💾 Work Analogy: Each Kafka consumer worker needs a desk (stack). 200 workers need 200 desks (200 MB). Virtual threads are digital—they don't need desks. 10,000 digital workers consume almost no space.

C. OS Scheduler & Context Switching

How the OS Schedules Threads

The OS Scheduler decides which thread runs on which CPU core at any given moment. This happens automatically via time-slicing.

Timeline of OS Scheduling:

OS Scheduler Flow
1. Timer Interrupt (hardware timer fires every ~10ms) ↓ 2. CPU pauses current thread (Thread A on Core 0) ↓ 3. OS saves context (CPU registers, stack pointer, etc.) ↓ 4. OS picks next thread from Ready Queue (e.g., Thread C) ↓ 5. OS restores context of Thread C ↓ 6. Thread C runs for ~10ms on Core 0 ↓ 7. Repeat...

Context Switching Cost

What happens during a context switch:

  • OS enters Kernel Mode (privileged CPU state)
  • CPU registers (~15-20) saved to memory
  • Instruction pointer, stack pointer updated
  • CPU caches flushed (loses performance data)
  • New thread's context loaded into registers
  • CPU exits Kernel Mode, resumes user code

This is expensive. With Platform Threads, if you have 200 threads on 10 cores, the OS is constantly context switching (thrashing). With Virtual Threads, the JVM switches them in user-space—just a pointer move, no Kernel Mode, no cache flush. ~1000x faster.

Scheduler Algorithms

OS Scheduler Algorithm How it works Linux CFS (Completely Fair Scheduler) Each thread gets a fair share of CPU time based on nice priority Windows Priority-based Round Robin Threads run in priority order; same priority = round-robin time slices JVM (Virtual Threads) Work Stealing Scheduler Carrier threads steal work from each other's queues (load balancing)

D. Spring Boot Threading Model

Thread-Per-Request Model (Default)

Spring's embedded Tomcat uses a simple model:

Spring Boot Request Flow (Synchronous)
1. HTTP Request arrives ↓ 2. Tomcat borrows idle thread from pool (max 200 by default) ↓ 3. Thread executes: filters → interceptors → controller logic ↓ 4. If database call: thread BLOCKS (idle, wastes 1MB stack) ↓ 5. Response sent → thread returns to pool ↓ 6. Repeat
🏪 Work Analogy: You have 200 checkout cashiers (threads). Customer arrives (request), cashier is assigned. If customer goes to fetch an item (DB query), cashier waits idle. No checkout happens while waiting. If you have 500 customers but only 200 cashiers, 300 wait in line.

Default Configuration

Tomcat Thread Pool Defaults
// application.properties server.tomcat.threads.max=200 // Max worker threads server.tomcat.threads.min-spare=10 // Min kept alive server.tomcat.accept-count=100 // Queue size if all threads busy server.tomcat.max-connections=10000 // Max TCP connections

Non-Blocking Alternative: Spring WebFlux

For ultra-high concurrency, use Spring WebFlux instead of Spring MVC:

Feature Spring MVC (Sync) Spring WebFlux (Async) Threading Thread per request Event loop (small thread pool) Max concurrent ~200 (before 21), millions (with VT) ~10,000+ (small fixed pool) Blocking I/O Natural (synchronous code) Requires async APIs (CompletableFuture, Reactor) Code complexity Simple, familiar Steeper learning curve (reactive streams) Best for Most web apps Real-time, high-concurrency systems
Modern recommendation: Use Spring MVC with Virtual Threads (Java 21+) instead of WebFlux for most cases. You get async performance with synchronous, readable code.

E. Enhanced Thread Pool Sizing Formula

The Magic Formula

Optimal Threads = CPU Cores / (1 - Blocking Ratio)

Where Blocking Ratio = (time waiting for I/O) / (total request time)

Real-World Example

Example: Service with 90% I/O Wait
Request timeline: - CPU processing: 10ms (10%) - Database I/O: 90ms (90%) - Total: 100ms blocking_ratio = 90 / 100 = 0.9 On 10-core machine: optimal_threads = 10 / (1 - 0.9) = 10 / 0.1 = 100 threads Why? While 1 thread waits on DB (90ms), other 99 threads can execute their own work. 10 cores stay 100% utilized with 100 threads.

Formula Examples

Workload Type Blocking Ratio On 8 Cores Rationale CPU-bound (image processing) 0% (no I/O) 8 threads 8 / (1 - 0) = 8. Adding more = context switching overhead Mixed (50% I/O) 0.5 16 threads 8 / 0.5 = 16. Moderate I/O, moderate threads I/O-heavy (API calls, DB) 0.9 80 threads 8 / 0.1 = 80. High I/O = many threads needed Kafka consumer (pure I/O) 0.95+ 150+ threads Almost all time waiting for messages

How to Measure Blocking Ratio

Capture Request Metrics
// Use monitoring: Micrometer, Spring Cloud Sleuth, or logs Timer timer = Timer.start(); // CPU work long cpuStart = System.nanoTime(); // ... processing ... long cpuTime = System.nanoTime() - cpuStart; // Blocking I/O long ioStart = System.nanoTime(); response = userRepository.findById(123); // Database call long ioTime = System.nanoTime() - ioStart; double blockingRatio = (double) ioTime / (cpuTime + ioTime); System.out.println("Blocking ratio: " + blockingRatio);
