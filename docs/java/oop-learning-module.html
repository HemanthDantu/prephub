What is Object-Oriented Programming?
The mental model that shapes most of the Java you'll ever write.
🧒 ELI5

Imagine you're playing with LEGO. You have different types of LEGO bricks — a wheel, a door, a window. Each type knows what it looks like and what it can do. You can make many copies of the same brick type — 10 wheels from one mold. OOP is that — you define a mold (called a class), and stamp out as many bricks (objects) as you need.

Before OOP, code was more like a recipe written on a scroll — just a long list of steps, top to bottom. OOP lets you think in things instead of steps.

🔗 ANALOGY — Blueprint vs. Building

A class is like an architectural blueprint. The blueprint for a house describes how many rooms there are, where the wiring goes, what the walls look like. But you can't live in a blueprint.

An object is an actual house built from that blueprint. You can build 50 houses from the same blueprint — each house is independent, but they all share the same design rules.

⚙ Technical — Classes & Objects in Java

The Core Vocabulary

  • Class — A user-defined type. It's a template that bundles state (fields) and behavior (methods).
  • Object — A runtime instance of a class, allocated on the heap.
  • Field — A variable that belongs to an object (instance field) or the class (static field).
  • Method — A function that belongs to a class. It can read/modify the object's state.
  • Constructor — A special method called when you do new. Initializes the object.
public class Car {
    // Fields — the object's STATE
    private String brand;
    private int speed;

    // Constructor — runs when you say `new Car(...)`
    public Car(String brand) {
        this.brand = brand;
        this.speed = 0;
    }

    // Method — the object's BEHAVIOR
    public void accelerate(int amount) {
        speed += amount;
    }

    public String getStatus() {
        return brand + " going " + speed + "mph";
    }
}

// Creating objects (instances)
Car myCar = new Car("Tesla");   // object #1
Car yourCar = new Car("BMW");    // object #2 — separate state
myCar.accelerate(60);
System.out.println(myCar.getStatus()); // Tesla going 60mph
System.out.println(yourCar.getStatus()); // BMW going 0mph

Each Car object has its own copy of brand and speed. Calling accelerate on myCar has zero effect on yourCar.

🧪 Quiz — Section 1
Q1. What is the relationship between a class and an object in Java?
✅ A class defines the shape (fields + methods) and lives in your source code. Objects are instantiated at runtime with new. Many independent objects can share the same class definition, each with their own state.
Q2. In the code Car c1 = new Car("Tesla"); Car c2 = new Car("BMW"); c1.accelerate(50); — what is c2.speed?
✅ Instance fields belong to each object independently. c1.speed becomes 50, but c2 was created separately and its speed stays at 0.
Q3. Which statement BEST describes why OOP was invented over procedural programming?
✅ OOP shines at scale — grouping state and behavior into cohesive units reduces cognitive load, makes code easier to test in isolation, and lets teams work on separate classes without constant conflict.
Section 2 of 5 · ~6 mins
🔒 Encapsulation
Bundle data with behavior, then guard the gates.
🧒 ELI5

Think of a piggy bank. You can put coins in through the slot, and shake it to hear how much is in it — but you can't just reach inside and grab whatever you want. The bank controls how you interact with the money inside it.

Encapsulation is the same idea: your object hides its internal data and only lets the outside world interact with it through approved doors (methods). If a bug sneaks in, you know exactly where to look.

🔗 ANALOGY — ATM Machine

You don't interact with the bank's raw database to check your balance. You use the ATM's interface — insert card, enter PIN, press "Check Balance." The ATM encapsulates all the complex logic of talking to the bank's servers. You get a clean, controlled interface.

If the bank changes its backend system, the ATM buttons stay exactly the same. Callers are insulated from internal changes — that's the power of encapsulation.

⚙ Technical — Access Modifiers & Getters/Setters

The two parts of Encapsulation

  • Bundling — keeping related fields + methods inside one class.
  • Access control — using modifiers to restrict who can touch what.

Classic pattern: private fields + public getters/setters

public class BankAccount {
    // 🔒 private — no direct outside access
    private double balance;
    private String owner;

    public BankAccount(String owner, double initial) {
        this.owner = owner;
        this.balance = initial;
    }

    // ✅ Controlled write — validation lives HERE
    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Must be positive");
        balance += amount;
    }

    // ✅ Controlled read — returns a copy, not the real field
    public double getBalance() { return balance; }

    // ❌ If balance were public, anyone could do:
    //    account.balance = -999999;  ← no validation, game over
}
⚠ Common Pitfall — Getters/Setters aren't always "real" encapsulation

Generating a getter and setter for every field is a reflex, but if you expose both freely, you haven't really encapsulated anything — you've just added boilerplate. Real encapsulation means asking: does the outside world need to know or change this? If not, don't expose it at all.

Also: returning a mutable object (like a List) from a getter hands the outside world a live reference to your internals. Return a defensive copy instead: return Collections.unmodifiableList(items);

🧪 Quiz — Section 2: Encapsulation
Q1. A field marked protected in class Animal is accessible from:
✅ protected = own class + same package + subclasses. It's a middle ground between private (locked to one class) and public (fully open). It's commonly used for fields or methods you want subclasses to inherit but don't need the entire world to see.
Q2. What is the main BENEFIT of making fields private and providing getter methods?
✅ This is the real win. If balance is public, any caller can set it to anything. Behind a method, you can add constraints, logging, events — and you can even change the underlying data structure (e.g., storing balance in cents internally) without changing the method signature your callers see.
Q3. Your class has a private List<String> tags field. Which getter is SAFEST?
✅ Returning the raw tags reference leaks a live handle — anyone can call getTags().clear() and corrupt your internal state. Wrapping it in Collections.unmodifiableList() returns a view that throws UnsupportedOperationException if modified. Alternatively, return new ArrayList<>(tags) for a true defensive copy.
Section 3 of 5 · ~6 mins
🎭 Abstraction
Show what something does. Hide how it does it.
🧒 ELI5

When you drive a car, you use the steering wheel, accelerator, and brakes. You don't need to know how the fuel injection system works, or what the differential gear ratio is. The car hides all that complexity — you just get a simple set of controls.

Abstraction means: expose what matters, hide what doesn't. Users of your code get a clean, simple surface. All the messy details live underneath, safely hidden.

🔗 ANALOGY — TV Remote

A TV remote gives you an interface: Channel+, Channel−, Volume+, Mute. You don't know whether the remote sends IR or Bluetooth signals, or what frequency it uses. You just press buttons.

In code, an interface or abstract class is that remote. It declares what operations exist. Each TV brand (Samsung, LG, Sony) can implement those operations differently under the hood, but you use the same remote with any of them.

// ── INTERFACE ── (pure "what", no "how")
public interface Payable {
    double calculatePay();  // must implement this

    default String paymentSummary() {  // optional override
        return "Pay: $" + calculatePay();
    }
}

// ── ABSTRACT CLASS ── (partial implementation)
public abstract class Employee implements Payable {
    protected String name;
    protected double baseSalary;

    public Employee(String name, double base) {
        this.name = name;
        this.baseSalary = base;
    }

    // Subclasses MUST provide this — "I don't know yet how"
    public abstract double calculatePay();

    // Shared behavior all employees get for free
    public String getName() { return name; }
}

// ── CONCRETE CLASSES ── (fill in the gaps)
public class FullTimeEmployee extends Employee {
    public FullTimeEmployee(String name, double salary) {
        super(name, salary);
    }
    @Override
    public double calculatePay() { return baseSalary / 12; }
}

public class Contractor extends Employee {
    private int hoursWorked;
    public Contractor(String name, double hourly, int hours) {
        super(name, hourly);
        this.hoursWorked = hours;
    }
    @Override
    public double calculatePay() { return baseSalary * hoursWorked; }
}
💡 INTERVIEW ANSWER — Abstraction in one sentence

"Abstraction means exposing only the relevant operations of an object while hiding the implementation details. In Java it's achieved via interfaces and abstract classes — callers depend on a contract, not on a specific implementation."

⚠ Encapsulation vs. Abstraction — Don't confuse them

Encapsulation is about data hiding — protecting internal state using access modifiers. It's a mechanism.

Abstraction is about complexity hiding — simplifying what callers see. It's about design at a higher level. A method can be abstract without its class being abstract (via an interface). You can encapsulate data in a fully concrete class with no abstraction at all.

Think of it this way: a microwave has a button panel (abstraction — you don't see the magnetron) and a locked chassis (encapsulation — you can't touch the internal components).

🧪 Quiz — Section 3: Abstraction
Q1. You're designing a payment system that will support PayPal, Stripe, and bank transfers. What's the BEST starting point in Java?
✅ An interface defines the contract (what operations exist) without locking in how they work. Each provider implements it independently. Callers use PaymentProcessor — swapping providers doesn't break callers. This also enables easy mocking in tests.
Q2. When should you prefer an abstract class over an interface?
✅ Use an abstract class when you have shared code (e.g., a constructor that sets a common field, or a method that all subclasses need identically). If there's no shared implementation, a pure interface is cleaner and more flexible since a class can implement many interfaces.
Q3. Can a Java abstract class have a constructor?
✅ You can't call new AbstractEmployee() directly, but when a subclass is instantiated, Java walks up the chain and calls the abstract class's constructor too. This is how the abstract class initializes its own fields (like name, baseSalary) that subclasses inherit.

🧬 Inheritance
Reuse what you have. Extend what you need. Build hierarchies of behavior.
🧒 ELI5

Think about how dogs, cats, and birds are all animals. All animals breathe, eat, and sleep. You don't have to re-teach a dog what breathing is — it just inherits that from being an animal. But a dog also barks, which cats don't do.

Inheritance in code works the same way: a child class automatically gets everything the parent class has, and then adds or changes what it needs. You write shared code once, in the parent.

🔗 ANALOGY — Biological Taxonomy

Biologists group life into a hierarchy: Kingdom → Phylum → Class → Order → Family → Genus → Species. A Golden Retriever is a Dog, which is a Mammal, which is an Animal. Each level inherits all the traits of everything above it.

Code inheritance builds the same kind of hierarchy. At the top: general traits shared by everything. At the bottom: very specific behavior unique to one type. The hierarchy is always an IS-A relationship — not HAS-A.

⚙ Technical — extends, super, and Method Overriding

Key keywords

  • extends — declares the parent class. Java allows only ONE parent (single inheritance).
  • super — refers to the parent. super() calls the parent constructor; super.method() calls an overridden parent method.
  • @Override — annotation telling the compiler "I intend to override a parent method." If the signature doesn't match, the compiler errors — safety net.
  • final on a class — prevents subclassing. final on a method — prevents overriding.
public class Animal {
    protected String name;

    public Animal(String name) { this.name = name; }

    public String sound() { return "..."; }

    public void describe() {
        System.out.println(name + " says: " + sound());
    }
}

public class Dog extends Animal {
    private String breed;

    public Dog(String name, String breed) {
        super(name);  // MUST call parent constructor first
        this.breed = breed;
    }

    @Override
    public String sound() { return "Woof!"; }  // replace parent behavior

    public void fetch() { System.out.println(name + " fetches!"); }
}

public class GuardDog extends Dog {
    public GuardDog(String name) { super(name, "Mixed"); }

    @Override
    public String sound() {
        return super.sound() + " WOOF WOOF!"; // extend parent behavior
    }
}

// Usage
Dog rex = new Dog("Rex", "Lab");
rex.describe();  // Rex says: Woof!  (inherited describe(), overridden sound())
rex.fetch();     // Rex fetches!   (Dog-specific method)

Java always calls the chain top-to-bottom. If you don't explicitly call super(), the compiler inserts a no-arg super() call — if the parent doesn't have a no-arg constructor, you'll get a compile error.

⚠ The "IS-A" Test — when NOT to use inheritance

Inheritance models a true IS-A relationship. A Dog IS-A Animal. But don't misuse it for code reuse alone.

Bad example: class Stack extends ArrayList. A Stack IS NOT an ArrayList — it just uses one internally. When you inherit, you expose the entire parent's API. Now callers can call stack.add(0, item) and bypass the stack contract. This famously broke Java's own java.util.Stack.

Rule of thumb: If you can say "X is a Y" and it makes real-world sense in every context → inheritance. If it just happens to share some methods → use composition (HAS-A) instead.

🧪 Quiz — Section 4: Inheritance
Q1. In Java, if class C extends B extends A, and you call new C(), in what order are constructors executed?
✅ Java ensures the parent is fully initialized before the child. C() calls super() → B() calls super() → A() runs first. Then B's body runs, then C's body. This guarantees inherited fields are ready before subclass code tries to use them.
Q2. What does the @Override annotation actually DO at runtime?
✅ @Override is purely a compiler hint. If the annotated method doesn't match any parent method signature exactly (e.g., you typo'd the name), you get a compile error instead of silently creating a new method that never gets called. At runtime, it's a no-op — the JVM doesn't see it.
Q3. Why is class Stack extends ArrayList considered a bad design in Java?
✅ A Stack should only expose push/pop/peek. By extending ArrayList, you inherit add(), remove(), set(), etc. — callers can manipulate the middle of the stack, destroying the LIFO contract. The right design is composition: class Stack { private List<T> list; } — you control what's exposed.
Section 5 of 5 · ~7 mins
🔀 Polymorphism
One interface. Many forms. The JVM figures out who answers the call.
🧒 ELI5

Imagine you shout "Speak!" in a room full of a dog, a cat, and a parrot. Each animal responds differently — woof, meow, squawk — but you used the exact same command.

Polymorphism means: the same method call produces different behavior depending on what object you actually have. You write code that talks to a general "Animal" type, and each specific animal handles it in its own way.

🔗 ANALOGY — The Universal Remote

You have a universal remote with one "Play" button. Point it at a TV — it plays a show. Point it at a Blu-ray player — it plays a disc. Point it at a music player — it plays a song. Same button press. Completely different result depending on what's receiving the signal.

That "Play" button is your method call. The device is your object. The JVM acts as the remote's signal — it routes the call to the right handler at runtime.

// ══ COMPILE-TIME: Overloading ══
public class Printer {
    public void print(String s)  { System.out.println(s); }
    public void print(int n)     { System.out.println(n); }
    public void print(String s, int times) {
        for (int i = 0; i < times; i++) System.out.println(s);
    }
    // Compiler picks the right method based on argument types at compile time
}

// ══ RUNTIME: Overriding (Dynamic Dispatch) ══
public class Animal {
    public String sound() { return "..."; }
}
public class Dog extends Animal {
    @Override
    public String sound() { return "Woof!"; }
}
public class Cat extends Animal {
    @Override
    public String sound() { return "Meow!"; }
}

// The power — write code against the PARENT type:
List<Animal> animals = List.of(new Dog(), new Cat(), new Dog());

for (Animal a : animals) {
    System.out.println(a.sound());  // JVM resolves the REAL type at runtime
    // Output: Woof! / Meow! / Woof!
}
// Add a new Bird class tomorrow — this loop still works with ZERO changes

Why this matters — the Open/Closed Principle

That loop over List<Animal> never needs to change even if you add 10 new animal types. You're open for extension (add new subclasses) but closed for modification (loop code stays the same). This is OCP — one of the SOLID principles. Polymorphism is what makes it possible.

💡 INTERVIEW ANSWER — Polymorphism in two sentences

"Polymorphism lets a single reference type invoke different behaviors depending on the actual object it points to. Java achieves compile-time polymorphism via method overloading and runtime polymorphism via method overriding + dynamic dispatch — where the JVM resolves the actual method to call based on the concrete type at runtime."

⚠ Overloading vs. Overriding — Easy to Confuse

Overloading: Same class, same name, different parameters. Resolved by the compiler. Not polymorphism in the OOP sense — it's just syntactic convenience.

Overriding: Subclass redefines a parent's method with the same signature. Resolved at runtime by the JVM. This IS the "true" runtime polymorphism that enables the open/closed principle.

Common gotcha: if you call an overloaded method through a parent reference, Java picks the overload at compile time based on the declared type — not the runtime type. Overriding always wins at runtime.

🧪 Quiz — Section 5: Polymorphism
Q1. Given: Animal a = new Dog(); — When you call a.sound(), which sound() runs?
✅ The declared type (Animal) controls what methods are visible at compile time. The runtime type (Dog) controls which implementation runs. This is dynamic dispatch — the JVM checks the actual object's class at runtime and routes to Dog's overriding method.
Q2. Which of these is an example of COMPILE-TIME polymorphism?
✅ Three add() methods with different parameter types is overloading — the compiler resolves which method to call based on the argument types you pass in. No runtime information is needed. The other options all involve runtime dispatch through inheritance or interfaces.
Q3. You add a new Elephant extends Animal class with an overriding sound(). Which part of existing code needs to change?
✅ This is the Open/Closed Principle in action. Code written against the Animal type doesn't care what the concrete type is — the JVM figures it out at runtime. You can add 100 new animal subclasses and your iteration loop stays identical. This is why polymorphism is considered the most powerful OOP pillar for code extensibility.
