The Problem Lambdas Solve
Why Java was verbose · Passing behaviour as a value · The "before" world
Start here

Before Java 8, if you wanted to pass a behaviour — like "sort these names alphabetically" — to a method, you couldn't just pass the idea. You had to create an entire class just to wrap that one idea and hand it over. It was like being asked to pass someone a recipe, but the rules said you had to build a whole restaurant first.

Lambdas let you just hand over the recipe — a small, anonymous piece of behaviour, right inline, no extra class needed.

The callback analogy

Imagine telling a taxi driver: "Call me when you arrive." In real life, you just hand them your phone number — simple. In pre-Java-8, you'd have to create a PhoneNumber class, instantiate it, pass it to them, and they'd call a method named ring() on it. Same result, ten times the paperwork.

A lambda is the phone number — just the behaviour, passed directly, without ceremony.

The "Before" vs "After"

Let's say we want to sort a list of names. Watch how the code shrinks:

Java 1–7
// You had to write a whole separate class class NameComparator implements Comparator<String> { public int compare(String a, String b) { return a.compareTo(b); } } Collections.sort(names, new NameComparator());
anonymous inner class — still a class, just inline
Java 7
// Slightly better — inline, but still a LOT of boilerplate Collections.sort(names, new Comparator<String>() { public int compare(String a, String b) { return a.compareTo(b); } });
Java 8 lambdas — same intent, fraction of the code
Java 8+
// Lambda — expresses the SAME idea in one line Collections.sort(names, (a, b) -> a.compareTo(b)); // Even shorter with a method reference: names.sort(String::compareTo);

All three examples do the exact same thing. The lambda version just cuts out the noise and expresses the intent directly.

The key insight

In Java, a lambda is a way to treat behaviour as a value — something you can store in a variable, pass to a method, or return from a method. Just like you pass a String or an int, you can now pass a piece of logic.

// Storing behaviour in a variable (we'll explain the type in Section 3)
Runnable greet = () -> System.out.println("Hello, Hemanth!");

// Passing behaviour to a method
new Thread(greet).start();  // prints "Hello, Hemanth!" on a new thread

Lambda Syntax — Reading & Writing
The arrow operator · Parameters · Body forms · Reading lambdas out loud
The anatomy

A lambda has just two parts separated by an arrow ->: what goes in (parameters), and what happens (the body). Read it as: "given [these inputs], do [this]."

Lambda anatomy — labeled
(String name, int age)  ->  { System.out.println(name + " is " + age); }
⬆️ Parameters
input values — types optional if inferable
⬆️
"given … do"
⬆️ Body
can be one expr or a full block {}

All the Valid Syntax Forms

These are all valid lambdas
// ── PARAMETERS ──────────────────────────────────────────────

// No parameters: use empty parentheses
() -> System.out.println("Hello")

// One parameter: parentheses optional
name -> System.out.println(name)
(name) -> System.out.println(name)   // same thing

// Multiple parameters: parentheses required
(a, b) -> a + b

// With explicit types (usually not needed — Java infers them)
(String a, String b) -> a.compareTo(b)

// ── BODY ────────────────────────────────────────────────────

// Single expression — no braces, no return keyword needed
x -> x * x

// Multi-line block — needs braces AND explicit return
x -> {
    int result = x * x;
    System.out.println("Squared: " + result);
    return result;
}

// Returns nothing (void) — just has side effects
name -> System.out.println(name)

Reading Lambdas Out Loud

How to mentally parse a lambda you've never seen before

Left side of -> = "given ___". Right side = "return ___" or "do ___".

// Read: "given x, return x times x"
x -> x * x

// Read: "given name, print name"
name -> System.out.println(name)

// Read: "given a and b, return a.compareTo(b)"
(a, b) -> a.compareTo(b)

// Read: "given nothing, print Hello"
() -> System.out.println("Hello")

Lambdas in Common Situations

Places you'll immediately see lambdas in real Java code
// 1. Sorting a list of strings
List<String> names = new ArrayList<>(List.of("Charlie", "Alice", "Bob"));
names.sort((a, b) -> a.compareTo(b));
// names is now ["Alice", "Bob", "Charlie"]

// 2. Printing every element of a list
names.forEach(name -> System.out.println(name));

// 3. Running something on a new thread
new Thread(() -> System.out.println("Running on new thread!")).start();

// 4. Filtering a list (we'll go deeper in Streams section)
List<String> shortNames = names.stream()
    .filter(name -> name.length() <= 4)
    .toList();
// shortNames = ["Bob"]

Section 3 of 6 · ~7 mins
Functional Interfaces — The Type of a Lambda
What type is a lambda? · The Big 5 · When to use each
Every value needs a type — so what's a lambda's type?

In Java, every variable has a type. String name, int age. When you write a lambda, Java still needs to know its type. A functional interface is the type of a lambda — it's just an interface with one method, and the lambda fills in that method's body.

Think of it as a slot: the interface defines the shape of the slot (how many inputs, what type comes out), and the lambda is the piece that fits in.

The socket analogy

Different countries have different socket shapes. A functional interface is the socket — it has a specific shape (signature). A lambda is the plug that fits that shape. Runnable is a socket for "no input, no output" — so () -> doSomething() plugs in. Comparator<T> is a socket for "two inputs, returns int" — so (a,b) -> a.compareTo(b) plugs in.

A functional interface has exactly ONE abstract method
// This is a functional interface — one abstract method
@FunctionalInterface  // optional, but good practice
interface Greeter {
    void greet(String name);  // ← the ONE method
}

// A lambda can be used wherever a Greeter is expected:
Greeter g = name -> System.out.println("Hello, " + name);
g.greet("Hemanth");  // prints: Hello, Hemanth

The Built-in Big 5 — You Don't Need to Write Your Own

Java ships with a whole toolbox of ready-made functional interfaces in java.util.function. These 5 cover 90% of use cases:

Function<T, R>
R apply(T t)
Takes something in, gives something different back. Use for transformations.
s -> s.toUpperCase()
n -> n * 2
Predicate<T>
boolean test(T t)
Takes something in, answers yes or no. Use for filtering and checking.
s -> s.isEmpty()
n -> n > 18
Consumer<T>
void accept(T t)
Takes something in, returns nothing. Use when you just want side-effects.
s -> System.out.println(s)
order -> save(order)
Supplier<T>
T get()
Takes nothing in, gives something back. Use for lazy creation / factories.
() -> new Order()
() -> "default"
BiFunction<T, U, R>
R apply(T t, U u)
Like Function, but takes two inputs and returns one output.
(first, last) -> first + " " + last  |  (price, qty) -> price * qty
Using the built-in types — no custom interface needed
// Function — transform a String to its length
Function<String, Integer> getLength = s -> s.length();
System.out.println(getLength.apply("Hemanth"));  // 7

// Predicate — check if a name is short
Predicate<String> isShort = name -> name.length() <= 4;
System.out.println(isShort.test("Bob"));   // true
System.out.println(isShort.test("Alice")); // false

// Consumer — just print something
Consumer<String> printer = s -> System.out.println(">> " + s);
printer.accept("Hello");  // >> Hello

// Supplier — provide a default value lazily
Supplier<String> defaultName = () -> "Anonymous";
System.out.println(defaultName.get());  // Anonymous

// BiFunction — combine two strings
BiFunction<String, String, String> fullName = (first, last) -> first + " " + last;
System.out.println(fullName.apply("Hemanth", "Kumar"));  // Hemanth Kumar
Quick reference — which one to reach for
  • Something → Something else → Function<T, R>
  • Something → true/false → Predicate<T>
  • Something → (nothing, just do work) → Consumer<T>
  • (nothing) → Something → Supplier<T>
  • Two things → Something else → BiFunction<T, U, R>

Section 4 of 6 · ~7 mins
Lambdas with Collections
forEach · sort · removeIf · replaceAll · Writing real code right now
Where you'll use lambdas every single day

The most immediate place you'll encounter lambdas is with Java collections — Lists, Maps, Sets. Methods like forEach, sort, removeIf all take a lambda. Instead of writing a loop, you hand the collection a behaviour and say "do this for each element."

forEach — replace your for-loop
List<String> names = List.of("Alice", "Bob", "Charlie");

// Old way:
for (String name : names) {
    System.out.println(name);
}

// Lambda way — same result, cleaner intent:
names.forEach(name -> System.out.println(name));

// Even shorter with method reference (we cover these next section):
names.forEach(System.out::println);
sort — custom ordering
List<String> names = new ArrayList<>(List.of("Charlie", "Alice", "Bob"));

// Alphabetical (natural order):
names.sort((a, b) -> a.compareTo(b));
// ["Alice", "Bob", "Charlie"]

// Sort by length (shortest first):
names.sort((a, b) -> a.length() - b.length());
// ["Bob", "Alice", "Charlie"]

// Sort in reverse alphabetical:
names.sort((a, b) -> b.compareTo(a));
// ["Charlie", "Bob", "Alice"]

// Cleaner: Comparator.comparing() — reads like English
names.sort(Comparator.comparing(String::length));
names.sort(Comparator.comparing(String::length).reversed());
removeIf and replaceAll
List<String> names = new ArrayList<>(List.of("Alice", "Bob", "", "Charlie", ""));

// removeIf — remove all elements that match a condition
names.removeIf(name -> name.isEmpty());
// ["Alice", "Bob", "Charlie"]

// replaceAll — transform every element in place
names.replaceAll(name -> name.toUpperCase());
// ["ALICE", "BOB", "CHARLIE"]
Map — iterating over key-value pairs
Map<String, Integer> scores = new HashMap<>();
scores.put("Alice", 90);
scores.put("Bob", 75);
scores.put("Charlie", 85);

// forEach on a Map gives you key AND value
scores.forEach((name, score) -> System.out.println(name + ": " + score));

// getOrDefault with Supplier — provide value if key missing
scores.computeIfAbsent("Dave", key -> 0);  // Dave gets 0 if not in map

Method References — Shorthand Lambdas
:: syntax · 4 types · When to use them · Reading fluently
The core idea

Sometimes your lambda does nothing but call an existing method: name -> System.out.println(name). That's just calling println with one argument. Java lets you shortcut that to System.out::println. The double-colon :: means "use this existing method as my lambda." They're not new — just shorthand.

Speed dial analogy

A lambda is like dialing a full number: name -> System.out.println(name). A method reference is a speed-dial contact already saved: System.out::println. Same call, one tap.

The 4 types — with plain English explanations
// ① Static method: ClassName::staticMethod
// Lambda:  s -> Integer.parseInt(s)
// Ref:     Integer::parseInt
List.of("1", "2", "3").stream().map(Integer::parseInt).toList();
// [1, 2, 3] as Integers

// ② Bound instance: specificObject::method
// Lambda:  s -> System.out.println(s)   (System.out is the specific object)
// Ref:     System.out::println
List.of("Alice", "Bob").forEach(System.out::println);

// ③ Unbound instance: ClassName::instanceMethod
// Lambda:  s -> s.toUpperCase()   (each s becomes 'this')
// Ref:     String::toUpperCase
List.of("alice", "bob").stream().map(String::toUpperCase).toList();
// ["ALICE", "BOB"]

// ④ Constructor: ClassName::new
// Lambda:  name -> new StringBuilder(name)
// Ref:     StringBuilder::new
List.of("Alice", "Bob").stream().map(StringBuilder::new).toList();
When to use a method reference vs a lambda

Simple rule: if the lambda does nothing but delegate to one method call, use a reference. If it does anything else (combines calls, has conditions, does math), keep it as a lambda.

// ✅ Use method reference — just delegating
names.forEach(System.out::println);
names.stream().map(String::trim)

// ✅ Keep as lambda — doing more than one thing
names.forEach(name -> System.out.println("Hello, " + name));
names.stream().map(name -> name.trim().toLowerCase())

Streams — Where Lambdas Truly Shine
What is a Stream? · filter · map · collect · Reading a pipeline
The core idea

A Stream is like an assembly line for data. Raw parts (your List) go in one end, pass through a series of operations (each powered by a lambda), and a finished product comes out the other end. Each operation on the line does one focused thing — filter out defects, transform a part, count what's left.

The beauty: you describe what you want done, not how to loop through doing it.

The assembly line analogy in detail

Imagine a factory: raw orders come in, they're filtered (reject invalid ones), transformed (convert to shipment format), and finally collected into a box ready to ship. Each station on the line is a lambda. The stream is the conveyor belt. You never need to write "move to next station" — it's automatic.

The Three Types of Stream Operations

📦
Source start
list.stream() · Set.stream() · Stream.of(...) · IntStream.range(0, 10)
🔍
filter(Predicate) intermediate
Keep only elements where the predicate returns true. Lazy — no work until terminal.
🔀
map(Function) intermediate
Transform each element. Input type can differ from output type. Returns a new Stream.
🧺
collect / toList / count / forEach terminal
Triggers the whole pipeline. Produces a result or side-effect. Stream can't be reused after.
A complete stream pipeline — step by step
List<String> names = List.of("Alice", "Bob", "Charlie", "Dave", "Eve");

List<String> result = names
    .stream()                              // 1. Create a stream from the list
    .filter(name -> name.length() > 3)  // 2. Keep only names longer than 3 chars
    .map(String::toUpperCase)             // 3. Uppercase every remaining name
    .sorted()                              // 4. Sort alphabetically
    .toList();                             // 5. Collect into a new List (terminal)

// result = ["ALICE", "CHARLIE", "DAVE"]
// "Bob" filtered out (length 3), "Eve" filtered out (length 3)
More useful terminal operations
List<Integer> nums = List.of(3, 1, 4, 1, 5, 9, 2, 6);

// count() — how many elements pass the filter?
long bigCount = nums.stream().filter(n -> n > 4).count();  // 3

// anyMatch / allMatch / noneMatch — boolean questions
boolean hasNine  = nums.stream().anyMatch(n -> n == 9);   // true
boolean allSmall = nums.stream().allMatch(n -> n < 10);  // true

// min / max
Optional<Integer> max = nums.stream().max(Integer::compareTo);  // 9

// joining strings
String result = List.of("Alice", "Bob").stream()
    .collect(Collectors.joining(", "));  // "Alice, Bob"
⚠️ One important rule: streams are one-use
Stream<String> s = names.stream();
s.forEach(System.out::println);  // ✅ works
s.count();                         // ❌ IllegalStateException: stream already closed
// Solution: call .stream() again to create a fresh one

Cheat Sheet
🎯 Lambdas = behaviour as a value. Pass them to methods instead of writing one-off classes.
✍️ Syntax: (params) -> body. One param? No parens needed. One expression? No braces or return needed.
🧩 Functional interface = the type of a lambda. One abstract method. @FunctionalInterface enforces it.
🔧 The Big 5: Function (T→R) · Predicate (T→boolean) · Consumer (T→void) · Supplier (→T) · BiFunction (T,U→R)
📚 Collections: forEach · sort · removeIf · replaceAll — all take lambdas.
⚡ Method refs Class::method — shorthand when lambda just delegates to one method call.
🏭 Streams: source → intermediate ops (filter/map, lazy) → terminal op (collect/count, triggers execution). One-use only.
