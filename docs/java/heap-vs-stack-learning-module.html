Section 1 of 5 · ~6 mins

Two kinds of memory: what lives where?

Start with the mental model first. The stack is about active work per thread. The heap is about shared objects that outlive a single method call. That one sentence alone clears up a lot of confusion.

🧒 ELI5

Think of a chef during a busy dinner service

The chef keeps a tiny scratchpad in hand for the current order: table number, "2 soups", "1 salad", "fire steak". That scratchpad is like the stack: fast, small, and only for what is happening right now.

The kitchen pantry and fridge hold the actual ingredients and dishes. That is the heap: the bigger shared storage where the real stuff lives.

🔗 ANALOGY

Backpack vs storage room

Your backpack carries quick-access things for the current trip: wallet, keys, a sticky note, a map address. That is the stack.

Your storage room holds the actual larger items: suitcases, winter coats, tools. That is the heap.

The backpack can hold an address to the storage room shelf, but not the full shelf itself. That is why a reference can be on the stack while the object is on the heap.

Technical

Precise model

  • Stack is thread-local memory.
  • Each method call creates a stack frame.
  • Frames contain local primitives and local references.
  • Heap is shared across threads.
  • All normal Java objects and arrays are allocated on the heap.

Section 2 of 5 · ~6 mins

The stack in detail: frames, locals, and references

Now zoom in on the stack. The biggest mistake people make is thinking an object variable is the object. In most cases, the local variable is only a reference stored in the current frame.

🧒 ELI5

Stack frames are like sticky notes for each task

Every time you start a new tiny task, you place a fresh sticky note on top of the pile. When the task finishes, you remove the top note. That top-down last-in-first-out behavior is exactly why it is called a stack.

🔗 ANALOGY

A phone call stack

You call one person, they transfer you to someone else, who then transfers you again. Each call is stacked on top of the previous one. The current active call is on top. When a call ends, you return to the previous one.

That is like method invocation. Each method call pushes a new frame.

Technical

What a stack frame contains

  • Parameters passed into the method
  • Local primitive variables like int, boolean
  • Local reference variables like User u
  • Bookkeeping data for returning to the caller

Important: the frame contains the reference value, not the heap object's full contents.

Java example

One heap object, two stack references

User u1 = new User();
User u2 = u1;

u1.name = "Ava";
System.out.println(u2.name);  // prints Ava

Why? Because u1 and u2 are two separate references in stack frames, both pointing to the same heap object.

Section 3 of 5 · ~6 mins

The heap in detail: Young Gen, Old Gen, and object lifetime

The heap is not one flat bucket. Generational GC is based on a practical observation: most objects die young. That is why the heap is organized into regions that reflect object age and survival.

🧒 ELI5

A school with new students and long-term teachers

Most new students show up and leave quickly. Only a smaller number stick around for years. So the school keeps a fast temporary area for new arrivals and a more stable area for long-term people.

That is like the heap: many new objects are born in the Young Generation; only survivors get promoted to the Old Generation.

🔗 ANALOGY

Airport terminal → lounge → long-stay hotel

Eden is the arrivals hall where new objects first land. If they survive garbage collection, they move through Survivor spaces. Objects that keep surviving are promoted to Old Gen, like guests who stay for the long haul.

Technical

Precise terminology

  • Eden: default area for most new object allocations.
  • Survivor spaces: intermediate regions for objects that survive minor collection.
  • Old Gen: region for long-lived objects, usually collected less frequently.
  • Minor GC: focuses mostly on young generation cleanup.
  • Major/Old collection: deals with old generation pressure.
Java example

Short-lived vs long-lived objects

public String formatName(User user) {
    StringBuilder sb = new StringBuilder();
    sb.append(user.firstName).append(" ").append(user.lastName);
    return sb.toString();
}

The StringBuilder is often short-lived and dies young. A cached singleton service object may survive long enough to reach Old Gen.

Section 4 of 5 · ~6 mins

Metaspace: class metadata, interned strings, and what changed in Java 8

Metaspace is commonly misunderstood because it is not where normal objects live. Think of it as the JVM's area for describing classes, not for storing your business objects.

🧒 ELI5

The instruction manual shelf

You have toys in the room, but you also have the instruction manuals that explain how each toy is built and what pieces it has. The manuals are not the toys. They are descriptions.

Metaspace is that shelf of descriptions for loaded classes.

🔗 ANALOGY

Blueprint archive vs actual buildings

The city archive stores blueprints, property records, and structural definitions. Those documents are like class metadata. The actual buildings in the city are like heap objects created from those class definitions.

Technical

What goes in Metaspace?

  • Loaded class metadata
  • Method metadata
  • Field metadata
  • Runtime constant pool data associated with classes
  • Classloader-related metadata

Metaspace uses native memory, unlike the heap which is JVM heap memory.

Warning

About interned Strings post-Java 8

For interview purposes, the most useful simplification is:

  • String objects themselves, including interned strings, are associated with the heap in modern Java.
  • The class metadata about String and other classes is in Metaspace.

So if somebody says “interned Strings are in Metaspace,” treat that as a trap. The clearer answer is: class metadata is in Metaspace; String objects are heap objects.

Section 5 of 5 · ~6 mins

Why StackOverflowError and OutOfMemoryError happen differently

These failures look similar because both mention memory, but they come from different memory regions, different resource limits, and different root causes. This distinction matters a lot in debugging and interviews.

🧒 ELI5

Two different ways a house can fail

One failure is stacking too many boxes on a narrow staircase until no one can move. That is StackOverflowError.

The other is filling the storage areas until there is simply no room left. That is OutOfMemoryError.

🔗 ANALOGY

Phone menu loop vs warehouse overflow

If customer support keeps transferring you in a loop, the call chain grows until it breaks. That is recursion causing stack overflow.

If a warehouse keeps receiving boxes faster than it can remove them, space runs out. That is heap pressure. If the blueprint archive keeps growing due to endless new building plans, that is metaspace pressure.

Technical

StackOverflowError

  • Usually caused by deep or infinite recursion.
  • Each method call pushes another stack frame.
  • When the thread's stack limit is exhausted, JVM throws StackOverflowError.
void recurse() {
    recurse();
}
Technical

OutOfMemoryError: Java heap space

  • Happens when the JVM cannot allocate more heap objects.
  • Typical causes: leaks, unbounded collections, too-small heap, large burst allocations.
List<byte[]> list = new ArrayList<>();
while (true) {
    list.add(new byte[1024 * 1024]);
}
Warning

OutOfMemoryError: Metaspace

  • This is different from heap exhaustion.
  • The JVM cannot reserve enough native memory for more class metadata.
  • Often linked to many dynamically generated classes or classloader leaks.

Cheat sheet

Core ideas

  • Stack is thread-local and stores method frames, local primitives, and local references.
  • Heap is shared and stores objects and arrays.
  • Young Gen contains Eden and Survivor spaces for new and recently surviving objects.
  • Old Gen stores longer-lived objects.
  • Metaspace stores class metadata using native memory.
Cheat sheet

Error patterns

  • StackOverflowError usually means runaway recursion or very deep call depth.
  • Heap OOME means the JVM cannot allocate enough object memory on the heap.
  • Metaspace OOME usually means too many loaded/generated classes or classloader leaks.
  • In modern Java, class metadata is in Metaspace; String objects are heap objects.
🔗 ANALOGY

One-line memory analogy you can remember

Stack = active sticky notes for the current work. Heap = shared storage room for real objects. Metaspace = library of class blueprints.

  • Local primitive? Usually stack frame.
  • Local object variable? Usually a stack reference.
  • Actual object created with new? Heap.
  • User.class metadata? Metaspace.
  • Infinite recursion? Think stack overflow first.
  • Growing collection? Think heap pressure first.
  • Dynamic proxies and classloader leaks? Think metaspace pressure.
