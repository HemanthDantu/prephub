Thread lifecycle

Before coordination primitives make sense, you need to know what a thread is doing at different moments of its life.

🧒 ELI5

A worker changing states

A worker can be ready to start, actively working, waiting for a door to open, taking a nap until a timer rings, or finished for the day. Threads move through similar states.

🔗 ANALOGY

A train moving through stations

A thread does not just go from start to finish in one uninterrupted line. It can be scheduled, paused, blocked, resumed, and finally terminated.

Technical

Lifecycle states give debugging clues

A Java thread moves through states such as NEW, RUNNABLE, BLOCKED, WAITING, TIMED_WAITING, and TERMINATED. These states help explain why a program is progressing, stalling, or hanging.

Key states

  • NEW: created but not started
  • RUNNABLE: eligible to run / may be running
  • BLOCKED: waiting for a monitor lock
  • WAITING or TIMED_WAITING: waiting for another event
  • TERMINATED: finished execution

Why this matters

  • Useful in thread dumps
  • Helps separate lock contention from waiting logic
  • Common first step in diagnosing hangs
Interview answer

A Java thread typically starts in NEW, becomes RUNNABLE when started, may move into BLOCKED or WAITING states depending on locks or coordination, and ends in TERMINATED when run finishes.

Java example

Thread t = new Thread(() -> doWork());

System.out.println(t.getState()); // NEW

t.start();
System.out.println(t.getState()); // often RUNNABLE

t.join();
System.out.println(t.getState()); // TERMINATED

Thread state snapshots are timing-dependent, but the lifecycle vocabulary matters a lot in interviews and debugging.

Quiz

Which thread state specifically means waiting to acquire a monitor lock?

Why: BLOCKED is the state for a thread waiting to acquire a monitor lock, such as entering a synchronized block already held by another thread.
Mini takeaway

What to remember

  • Know the lifecycle vocabulary cold.
  • BLOCKED usually points to lock contention.
  • WAITING and TIMED_WAITING usually point to coordination logic or sleep/join/wait.
Section 2 of 5 · ~9 mins

The synchronized keyword

synchronized is the classic Java tool for mutual exclusion and visibility around shared mutable state.

🧒 ELI5

One person in the room at a time

If two people try to write on the same whiteboard at once, the result gets messy. So the room gets one key. Only one person can enter and use the board at a time.

🔗 ANALOGY

A lock around a shared notebook

The notebook is shared state. synchronized is the key that ensures only one thread edits it at a time, and that the next reader sees a consistent version after the lock is released.

Technical

Mutual exclusion plus memory visibility

synchronized uses an intrinsic monitor lock. It guarantees mutual exclusion for the protected block and also establishes visibility guarantees when a thread exits and another thread later acquires the same monitor.

What it gives you

  • Only one thread executes the critical section at a time
  • Prevents race conditions on shared mutable state
  • Provides visibility via monitor enter/exit semantics

Common traps

  • Locking too much code hurts throughput
  • Locking on the wrong object gives fake safety
  • Mixing synchronized and unsynchronized access can reintroduce bugs
Interview answer

synchronized is used to protect shared mutable state. It gives mutual exclusion and also visibility, because writes before releasing the monitor become visible to a thread that later acquires the same monitor.

Java example

class Counter {
    private int count = 0;

    public synchronized void increment() {
        count++;
    }

    public synchronized int getCount() {
        return count;
    }
}

Without synchronization, two threads could interleave increment logic and lose updates.

Warning

synchronized is not just about blocking

A lot of people only say 'one thread at a time.' A stronger answer also mentions visibility and ordering guarantees associated with monitor acquisition and release.

What to remember

  • synchronized protects critical sections.
  • It gives both mutual exclusion and visibility.
  • You must synchronize consistently on the same shared state.
Section 3 of 5 · ~8 mins

volatile and visibility

volatile is often misunderstood. It is not a lightweight synchronized replacement for every concurrency problem.

🧒 ELI5

Always read the latest sticky note

Imagine one whiteboard in the hallway. If the note is volatile, everyone must read the latest note from the board, not a stale copy in their pocket.

🔗 ANALOGY

Freshness, not full coordination

volatile is like forcing threads to read the latest published version of a value. It helps with visibility, but it does not put a lock around compound actions.

Technical

volatile = visibility and ordering, not atomicity

A volatile field tells the JVM and CPU that reads and writes to that variable must not be hidden in stale caches or reordered across certain boundaries. It improves visibility between threads, but operations like count++ are still not atomic.

Good use cases

  • Stop flags and shutdown signals
  • Publishing a simple state value
  • One writer / many readers for a simple variable

Bad assumptions

  • volatile does not make increments atomic
  • volatile does not replace locking for compound invariants
  • volatile alone does not prevent race conditions on multiple fields
Interview answer

volatile guarantees visibility of the latest value across threads and restricts harmful reordering around that variable, but it does not provide mutual exclusion or atomic compound updates.

Java example

class Worker {
    private volatile boolean running = true;

    public void stop() {
        running = false;
    }

    public void runLoop() {
        while (running) {
            doUnitOfWork();
        }
    }
}

This is a classic volatile use case. A stop flag needs freshness, not a long critical section.

Why: volatile is about seeing the latest value and preventing certain reorderings. It does not provide locking or atomic compound updates.
Mini takeaway

What to remember

  • volatile helps with visibility, not mutual exclusion.
  • It is good for simple state flags and publication.
  • count++ still needs stronger coordination like synchronization or atomics.

Deadlocks

A deadlock is not just 'the app is stuck.' It is a very specific circular waiting pattern.

🧒 ELI5

Two people holding each other’s keys

Person A has key 1 and wants key 2. Person B has key 2 and wants key 1. Neither can move forward, so both wait forever.

🔗 ANALOGY

Circular waiting between locked resources

Deadlocks happen when threads acquire locks in inconsistent order and end up waiting on each other. The work is not slow. It is frozen.

Technical

How deadlock forms

Deadlock happens when threads hold resources while waiting for other resources in a cycle. In Java, a common example is two threads acquiring the same pair of locks in opposite order.

Classic causes

  • Inconsistent lock ordering
  • Nested synchronized blocks across shared resources
  • Multiple locks with no global acquisition rule

How to prevent

  • Use consistent lock ordering
  • Reduce lock scope and number of locks
  • Use timed locks or higher-level concurrency tools where appropriate
Interview answer

Deadlock is a circular wait condition. Thread A holds lock 1 and waits for lock 2 while Thread B holds lock 2 and waits for lock 1. The standard prevention technique is consistent lock ordering.

Java example

Object lockA = new Object();
Object lockB = new Object();

synchronized (lockA) {
    synchronized (lockB) {
        doWork();
    }
}

// Elsewhere, if another thread does lockB then lockA,
// you have a deadlock risk.

The bug is not the nested lock itself. The bug is acquiring shared locks in different orders across threads.

Warning

Slow is not the same as deadlocked

High contention can make a program slow, but deadlock means circular waiting with no forward progress. Use thread dumps to distinguish them.

What is the most standard prevention technique for deadlocks involving multiple locks?

Why: Consistent lock ordering breaks the circular wait pattern that causes many deadlocks.
Mini takeaway

What to remember

  • Deadlock = circular waiting, not just slowness.
  • Inconsistent lock ordering is a classic cause.
  • Consistent acquisition order is the standard prevention answer.
Section 5 of 5 · ~10 mins

Happens-before

This is the deeper memory-model concept that ties together synchronized, volatile, and thread coordination.

🧒 ELI5

One event must happen before another can trust it

If one person writes the answer on a board, and only after that rings a bell, the next person who hears the bell knows the board is ready to read.

🔗 ANALOGY

A guaranteed publish-before-read relationship

Happens-before is the promise that one action's effects are visible to another action. Without that relationship, one thread may not reliably see what another thread already did.

Technical

The language of visibility and ordering

A happens-before relationship means memory writes by one thread become visible to another thread in a defined order. Examples include monitor release/acquire with synchronized, volatile write/read, thread start, and thread join.

Examples

  • Exiting a synchronized block happens-before another thread enters the same monitor
  • A volatile write happens-before a later volatile read of the same variable
  • Actions before thread.start() happen-before the started thread sees them
  • Actions in a thread happen-before another thread successfully returns from join()

Why interviewers ask it

  • Shows you understand visibility, not just syntax
  • Explains why synchronized and volatile work
  • Connects code to the Java Memory Model
Interview answer

Happens-before is the visibility and ordering guarantee in the Java Memory Model. If action A happens-before action B, then B is guaranteed to see the effects of A. synchronized, volatile, start, and join create these guarantees.

Java example

private volatile boolean ready = false;
private int data;

public void writer() {
    data = 42;
    ready = true; // volatile write
}

public void reader() {
    if (ready) {       // volatile read
        use(data);       // guaranteed to see 42 here
    }
}

The volatile write/read establishes the happens-before edge that makes data visible in the intended order.

Quiz

What does a happens-before relationship guarantee?

Why: Happens-before is about visibility and ordering guarantees in the Java Memory Model. It does not mean all operations are atomic or deadlock-free.
Mini takeaway

What to remember

  • Happens-before is the core visibility concept.
  • synchronized and volatile are important because they create happens-before edges.
  • A strong interview answer names start, join, monitor release/acquire, and volatile read/write.

Threads & Synchronization Cheat Sheet

You now have the interview-ready concurrency story: what a thread does over its lifetime, what synchronized and volatile really guarantee, how deadlocks form, and why happens-before is the language of visibility and ordering.

Cheat sheet

Fast scan summary

  • Lifecycle: Threads move through states like NEW, RUNNABLE, BLOCKED, WAITING, and TERMINATED.
    Use thread states to reason about contention, waiting, and completion.
  • synchronized: Protects critical sections with mutual exclusion and visibility.
    A good answer mentions both locking and monitor-based visibility guarantees.
  • volatile: Freshness and ordering for a variable, not full atomicity.
    Use volatile for simple state publication like stop flags, not for compound updates like count++.
  • Deadlock: Circular waiting between locks or resources.
    The classic prevention technique is consistent lock ordering.
  • Happens-before: The memory-model guarantee behind visibility and ordering.
    If A happens-before B, then B must see A's effects in the defined order.
Common confusion

Separate the look-alikes

volatile vs synchronized

volatile gives visibility for a variable. synchronized gives mutual exclusion plus visibility around a monitor-protected critical section.

Slow threads vs deadlock

A slow system may still be progressing. Deadlock means circular waiting with no forward progress.

Atomicity vs visibility

Seeing the latest value does not mean a multi-step update is atomic. That distinction is central in concurrency interviews.
