Single Responsibility Principle

SRP is about keeping one class focused on one reason to change. It sounds simple, but this is usually where messy service classes, giant controllers, and impossible-to-test code start.

🧒 ELI5

One person, one main job

Think of a restaurant. A chef should cook food. A cashier should take payments. A cleaner should clean tables. If one person tries to do all three, things become slow, confusing, and error-prone. In code, a class should also have one clear job.

🔗 ANALOGY

Your class should not be the whole team

A class that validates input, talks to the database, formats PDFs, sends emails, and writes logs is like asking one engineer to be backend, frontend, QA, DevOps, and product manager at the same time. You might ship something, but it will be painful to change later.

Technical

What it means in real code

Definition: A class should have only one reason to change. That means it should own one cohesive responsibility, not several unrelated ones.

Why it matters

  • Smaller blast radius when requirements change
  • Cleaner unit tests and easier mocking
  • Better readability and ownership boundaries

Common smell

  • Classes named *Manager or *Service doing everything
  • Methods mixing business rules and I/O
  • Frequent changes for unrelated reasons
Focused class Cohesion over class count God object
Validate→ Persist→ Notify

Each step can be its own collaborator instead of living inside one overloaded class.

Interview answer

“Single Responsibility means a class should do one cohesive thing and therefore have one main reason to change. I use it to keep business logic, persistence, formatting, and side effects separated so the code is easier to test and modify.”

Ask yourselfSignal
Why would this class change?If the answer is several unrelated reasons, SRP is weak.
Can I describe the class in one sentence?If not, responsibility is probably mixed.
Can I unit test core logic without I/O?If not, concerns may be tangled.
Responsibility breakdown

Bad: UserService

Bad example
public class UserService {
  public void register(User user) {
    // validation
    if (user.getEmail() == null) {
      throw new IllegalArgumentException("email required");
    }

    // persistence
    saveToDatabase(user);

    // side effect
    sendWelcomeEmail(user);
  }
}
Better example
public class UserRegistrationService {
  private final UserValidator validator;
  private final UserRepository repository;
  private final EmailNotifier notifier;

  public UserRegistrationService(
      UserValidator validator,
      UserRepository repository,
      EmailNotifier notifier) {
    this.validator = validator;
    this.repository = repository;
    this.notifier = notifier;
  }

  public void register(User user) {
    validator.validate(user);
    repository.save(user);
    notifier.sendWelcome(user);
  }
}

The second version still coordinates the workflow, but the responsibilities are separated. That makes testing and future changes much safer.

Warning

SRP is not “make everything tiny”

Do not split classes just to satisfy a slogan. The real goal is high cohesion, not maximum fragmentation. One class can have multiple methods, as long as they all serve the same responsibility.

Quiz

Which change most clearly violates SRP?

A class named InvoiceService now calculates totals, saves invoices, sends emails, and renders PDF templates.

Correct answer: B The problem is not the number of methods or dependencies. The issue is that business calculation, persistence, communication, and document rendering are separate concerns that change for different reasons.
Mini takeaway
  • SRP asks for one cohesive responsibility, not one method.
  • Different reasons to change usually mean different classes or collaborators.
  • Mixed I/O and business rules are a common production smell.
  • SRP often makes testing dramatically easier.

Open/Closed Principle

OCP is about designing code so you can add new behavior without repeatedly editing stable, already-working logic. In interviews, this usually shows up as replacing giant if/else blocks with polymorphism or strategy objects.

🧒 ELI5

Add a new toy without breaking the box

Imagine a toy box with slots that let you add a new toy type without sawing the whole box open every time. Good software should let you plug in new behavior instead of reopening old code and risking bugs.

🔗 ANALOGY

Like adding a new payment method

If every new payment type forces you to keep editing the checkout core, you are increasing risk in the most sensitive path. A better design lets you add a new payment strategy while the checkout flow stays stable.

Technical

What it means in real code

Definition: Software entities should be open for extension but closed for modification. You should be able to add behavior with new code instead of constantly editing tested core logic.

Why it matters

  • Reduces regressions in stable code paths
  • Makes features easier to add predictably
  • Improves extensibility for evolving domains

Common smell

  • Growing switch statements by type
  • Repeated edits to the same class for every feature
  • Feature flags and branching everywhere
Extend with new class Use interfaces/strategies Edit central branch logic

Interview answer

“Open/Closed means I try to design stable core flows so new behavior is added through extension points like interfaces, strategies, or polymorphism rather than repeatedly modifying a big conditional block.”

Bad example
public class PaymentService {
  public void pay(String type) {
    if ("CARD".equals(type)) {
      processCard();
    } else if ("PAYPAL".equals(type)) {
      processPayPal();
    } else if ("UPI".equals(type)) {
      processUpi();
    }
  }
}
Better example
public interface PaymentProcessor {
  void process();
}

public class CardProcessor implements PaymentProcessor {
  public void process() { /* card logic */ }
}

public class PaymentService {
  private final Map<String, PaymentProcessor> processors;

  public PaymentService(Map<String, PaymentProcessor> processors) {
    this.processors = processors;
  }

  public void pay(String type) {
    processors.get(type).process();
  }
}

The second design is easier to extend. Adding a new payment type mainly means adding a new implementation and wiring it in, instead of rewriting the core service.

Warning

Do not over-abstract too early

OCP is not a command to create ten interfaces for a feature that only has one implementation and no variation. It pays off when you actually expect a family of behaviors to grow.

Quiz

Which design best supports Open/Closed?

Correct answer: C OCP is strongest when stable orchestration depends on an extension point, and new behavior is added by introducing new implementations instead of modifying central conditional logic.
Mini takeaway
  • OCP tries to keep stable code stable.
  • Look for repeated edits to the same branching logic.
  • Interfaces, strategies, and polymorphism are common tools.
  • Use OCP where variation is expected, not everywhere by default.

Liskov Substitution Principle

LSP is the one people often memorize but struggle to explain. The practical meaning: if code works with a parent type, it should still work correctly when given any valid subtype.

🧒 ELI5

A replacement should still behave like the thing it replaces

If a TV remote takes AA batteries, you can swap one AA brand for another and it should still work. But if someone gives you a fake “AA” battery that does not actually power the remote, it is not a proper replacement. Subclasses should behave like real substitutes.

🔗 ANALOGY

Like a new driver in the same lane rules

If your system says “anything implementing this contract can go through this workflow,” then every implementation must obey the same expectations. If one subtype silently throws, refuses, or changes the meaning, the whole abstraction becomes unreliable.

Technical

What it means in real code

Definition: Objects of a subtype should be replaceable for objects of the base type without breaking correctness. Subtypes must honor the contract of the parent type.

Violation signals

  • Subclass throws UnsupportedOperationException for parent behavior
  • Subclass changes expected output semantics
  • Caller needs subtype checks to stay safe

Why it matters

  • Makes polymorphism trustworthy
  • Avoids surprise runtime failures
  • Keeps contracts meaningful
Subtype honors contract Subtype breaks assumptions Inheritance is not always the right reuse tool

Interview answer

“Liskov Substitution means a subclass must be usable anywhere the parent type is expected without breaking behavior. If callers need special checks for certain subclasses, the inheritance contract is probably wrong.”

Bad example
class Bird {
  void fly() {}
}

class Penguin extends Bird {
  @Override
  void fly() {
    throw new UnsupportedOperationException();
  }
}
Better example
interface Bird {
  void move();
}

interface FlyingBird extends Bird {
  void fly();
}

class Penguin implements Bird {
  public void move() { /* swim/walk */ }
}

class Sparrow implements FlyingBird {
  public void move() { /* hop */ }
  public void fly() { /* fly */ }
}

The fix is not “make Penguin try harder.” The fix is designing the abstraction so the promises it makes are actually true for every subtype.

Warning

LSP is about behavior, not just method signatures

A subclass can compile perfectly and still violate LSP if it breaks expectations around invariants, side effects, exceptions, or return semantics.

Quiz

Which example most clearly breaks LSP?

Correct answer: A If the parent contract says callers can invoke a method, and one subtype cannot honor that contract, substitution breaks. That is a classic LSP violation.
Mini takeaway
  • LSP is about preserving behavioral contracts.
  • If callers need subtype checks, your abstraction may be lying.
  • Unsupported operations in subclasses are a major red flag.
  • Composition is often safer than forcing inheritance.
Section 4 of 5 · ~6 mins

Interface Segregation Principle

ISP says clients should not be forced to depend on methods they do not use. This usually means smaller, role-based interfaces instead of one oversized contract.

🧒 ELI5

Do not hand everyone a giant remote

If someone only needs the volume button, do not force them to carry a giant remote full of buttons for streaming, gaming, camera control, and air conditioning. In code, give classes only the methods they actually need.

🔗 ANALOGY

Like team permissions

A finance user should not receive admin, HR, and engineering controls in one mega portal. Likewise, a class that only prints should not be forced to implement scan, fax, upload, and archive methods.

Technical

What it means in real code

Definition: No client should be forced to depend on methods it does not use. Prefer narrow, role-focused interfaces over one fat interface.

Common smell

  • Interface with many unrelated methods
  • Implementations throwing unsupported exceptions
  • Classes depending on methods they never call

Why it matters

  • Reduces coupling
  • Improves test setup and mocking
  • Makes contracts easier to understand
Small role-based interfaces Client-specific contracts Fat interfaces

Interview answer

“Interface Segregation means I prefer smaller, focused interfaces so a class only depends on the operations it actually needs. It avoids fat contracts and reduces the chance of unsupported or irrelevant methods.”

interface Machine {
  void print();
  void scan();
  void fax();
}

class BasicPrinter implements Machine {
  public void print() {}
  public void scan() { throw new UnsupportedOperationException(); }
  public void fax() { throw new UnsupportedOperationException(); }
}
Better example
interface Printer {
  void print();
}

interface Scanner {
  void scan();
}

class BasicPrinter implements Printer {
  public void print() {}
}

class OfficeMachine implements Printer, Scanner {
  public void print() {}
  public void scan() {}
}

When interfaces are focused, implementations are simpler and clients are not forced to carry behavior they do not need.

Warning

ISP and SRP are related, but not the same

SRP focuses on class responsibility. ISP focuses on client-facing contracts. A class can violate SRP without using interfaces at all, and an interface can violate ISP even if implementations are otherwise clean.

Quiz

Which change best applies ISP?

Correct answer: D ISP prefers smaller, role-focused interfaces so clients depend only on what they use. That reduces unnecessary coupling and avoids unsupported behaviors.
Mini takeaway
  • Do not force clients to depend on methods they never use.
  • Fat interfaces often lead to unsupported operations.
  • Small interfaces make mocks, tests, and contracts cleaner.
  • Think in roles: printer, scanner, notifier, exporter, validator.

Dependency Inversion Principle

DIP is about depending on abstractions instead of concrete low-level details. This is the principle behind constructor injection, testable services, and flexible architecture layers.

🧒 ELI5

Do not build the house around one exact plug

If your lamp only works with one weird wall socket in one room, moving it becomes painful. Better design uses a standard plug so different rooms can still power it. In code, high-level logic should not be tightly glued to one concrete implementation.

🔗 ANALOGY

Business logic should not marry infrastructure

Your order workflow should care that a message was sent, not whether it was sent through Twilio, SES, Kafka, or a fake test adapter. DIP keeps business rules stable while infrastructure stays swappable.

Technical

What it means in real code

Definition: High-level modules should not depend on low-level modules; both should depend on abstractions. Details should depend on policies, not the other way around.

Why it matters

  • Makes high-level services easier to test
  • Allows replacing infrastructure implementations
  • Supports cleaner layering and architecture boundaries

Common smell

  • new inside business services for external clients
  • Business rules importing framework or vendor code directly
  • Hard-to-mock code paths
Depend on interface Inject collaborator Hard-coded concrete dependency

Interview answer

“Dependency Inversion means my core business logic depends on abstractions, not concrete infrastructure classes. That lets me inject implementations, test the core behavior cleanly, and swap details without changing the policy layer.”

Bad example
public class OrderService {
  private final EmailSender emailSender = new EmailSender();

  public void placeOrder(Order order) {
    // business logic
    emailSender.send(order.getCustomerEmail(), "order placed");
  }

Better example
public interface ReceiptSender {
  void sendReceipt(Order order);
}

public class OrderService {
  private final ReceiptSender receiptSender;

  public OrderService(ReceiptSender receiptSender) {
    this.receiptSender = receiptSender;
  }

  public void placeOrder(Order order) {
    // business logic
    receiptSender.sendReceipt(order);
  }
}

This is why constructor injection is so common in Java applications. It supports DIP by making dependencies explicit and replaceable.

Warning

DIP is not just “use interfaces everywhere”

The point is architectural direction. High-level policy should not know concrete infrastructure details. Sometimes the abstraction can even be a small domain-facing port interface with only one implementation today.

Quiz

Which design best follows DIP?

Correct answer: B DIP is strongest when business policy points to an abstraction, while concrete infrastructure plugs in from the outside. That keeps the core testable and less coupled to implementation details.
Mini takeaway
  • High-level policy should depend on abstractions, not vendor details.
  • Constructor injection is a common practical expression of DIP.
  • DIP improves testability, layering, and replaceability.
  • Think in ports and adapters, not direct framework coupling.

1-line interview answers

SRP

A class should have one cohesive responsibility and one main reason to change.

OCP

Code should be open to extension but closed to repeated modification of stable logic.

LSP

Subtypes must be safely usable anywhere the parent type is expected, without breaking behavior.

ISP

Clients should not be forced to depend on methods they do not use.

DIP

High-level modules should depend on abstractions, not concrete low-level details.

What to remember

Fast mental model

  • SRP: keep responsibilities cohesive.
  • OCP: add new behavior without rewriting the core.
  • LSP: do not lie with inheritance.
  • ISP: give clients only the contract they need.
  • DIP: keep business logic pointed at abstractions.

Common confusion

SRP vs ISP

SRP is about the responsibility of a class. ISP is about the size and relevance of a client-facing interface.

OCP vs DIP

OCP is about extending behavior cleanly. DIP is about directing dependencies toward abstractions so high-level logic stays decoupled.

LSP vs ISP

LSP is about subtype behavior staying valid. ISP is about not forcing extra methods into contracts.
