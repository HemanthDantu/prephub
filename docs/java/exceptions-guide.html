What is an Exception?

Understanding the fundamentals — what goes wrong, when, and why Java needs a dedicated mechanism for it.

The Simple Definition

An exception is an unexpected event that occurs during the execution of a program and disrupts the normal flow of instructions. Think of it as Java's way of saying: "Something went wrong — here's all the information about what happened and where."

Real World Analogy
ATM Machine

You insert your card and try to withdraw £500. Several things could go wrong: the network is down (IOException), your balance is only £200 (InsufficientFundsException), you entered the wrong PIN 3 times (AccountLockedException). Each of these is a different "exception" — the machine stops the normal flow (dispensing cash) and handles each problem differently.

Without Exception Handling — What Would Happen?

// Without exception handling, the JVM just CRASHES the entire program
int[] arr = new int[5];
arr[10] = 42;  // Index 10 doesn't exist!
// Result: Program terminates, user sees a cryptic error, data may be lost

With Exception Handling — Graceful Recovery

try {
    int[] arr = new int[5];
    arr[10] = 42;
} catch (ArrayIndexOutOfBoundsException e) {
    // We caught it! Now we can:
    System.out.println("Invalid index: " + e.getMessage());
    // Log it, show a user-friendly message, retry, etc.
}
// Program continues running normally after this block

The Three Types of Problems in Java

Syntax Errors
Caught at compile time. Missing semicolons, wrong types. The code won't even run.
Exceptions
Occur at runtime. The code compiled fine but something went wrong during execution. Recoverable.
Errors
Serious JVM problems like OutOfMemoryError. Usually unrecoverable. Don't catch these.

What Information Does an Exception Carry?

try {
    String s = null;
    s.length();
} catch (NullPointerException e) {
    e.getMessage();      // "Cannot invoke String.length() because s is null"
    e.getClass();        // class java.lang.NullPointerException
    e.getStackTrace();   // Array of StackTraceElement showing exact line numbers
    e.getCause();        // The exception that caused THIS exception (null if root cause)
    e.toString();        // "java.lang.NullPointerException: ..."
    e.printStackTrace(); // Prints full stack trace to stderr (avoid in production)
}

The Exception Lifecycle

1
Something goes wrong
e.g. You try to parse "abc" as an integer — Integer.parseInt("abc")
2
JVM creates an Exception object
A NumberFormatException object is instantiated with the message, stack trace, and cause.
3
Exception is "thrown"
Java unwinds the call stack, looking for a matching catch block. It checks the current method first, then the caller, then the caller's caller, etc.
4
Caught or propagated
If a matching catch is found, it handles it. If no catch is found all the way up to main(), the JVM prints the stack trace and terminates the thread.
Key insight: Exceptions don't crash programs — unhandled exceptions crash programs. The whole point of exception handling is to give you a chance to respond gracefully.

Exception Hierarchy

The full class tree — understanding what inherits from what, and why it matters for catch blocks.

java.lang.Object └── java.lang.Throwable ├── java.lang.Error ← JVM-level, DO NOT catch │ ├── OutOfMemoryError │ ├── StackOverflowError │ ├── VirtualMachineError │ └── AssertionError └── java.lang.Exception ← Application-level ├── IOException ← CHECKED │ ├── FileNotFoundException │ ├── SocketException │ └── MalformedURLException ├── SQLException ← CHECKED ├── ClassNotFoundException ← CHECKED ├── CloneNotSupportedException ← CHECKED └── RuntimeException ← UNCHECKED ├── NullPointerException ├── ArrayIndexOutOfBoundsException ├── StringIndexOutOfBoundsException ├── ClassCastException ├── ArithmeticException ├── NumberFormatException ├── IllegalArgumentException ├── IllegalStateException ├── UnsupportedOperationException └── ConcurrentModificationException

Why Does the Hierarchy Matter?

Java's catch block uses polymorphism. When you catch Exception, you catch everything that inherits from it — which means ALL checked exceptions AND all RuntimeExceptions. This is why catch-block order matters.

// Catch order MATTERS — most specific FIRST, most general LAST
try {
    readFile("data.csv");
} catch (FileNotFoundException e) {
    // Specific — file doesn't exist
    createDefaultFile();
} catch (IOException e) {
    // Broader — any other IO problem
    log.error("IO failed", e);
} catch (Exception e) {
    // Widest net — catches everything else
    log.error("Unexpected error", e);
}

// ❌ This WON'T COMPILE — IOException is parent of FileNotFoundException
// The FileNotFoundException catch is unreachable:
catch (IOException e) { }
catch (FileNotFoundException e) { }  // Compile error!

Checked vs Unchecked Exceptions

One of Java's most debated design decisions — understanding when to use each and why.

The Core Difference

Checked Exceptions — Deep Dive

// The compiler FORCES you to handle or declare checked exceptions
public String readFile(String path) throws IOException {
    // IOException is checked — you must either:
    // 1. Catch it here (handle it)
    // 2. Declare 'throws IOException' (propagate it)
    return Files.readString(Path.of(path)); // throws IOException
}

// Real-world: Reading a config file at startup
public class AppConfig {
    public void load() {
        try {
            Properties props = new Properties();
            props.load(new FileInputStream("app.properties")); // throws FileNotFoundException (checked)
        } catch (FileNotFoundException e) {
            // Meaningful recovery: use defaults
            log.warn("Config not found, using defaults");
            loadDefaults();
        } catch (IOException e) {
            // Can't recover from corrupt config — rethrow
            throw new RuntimeException("Failed to load config", e);
        }
    }
}

Unchecked Exceptions — Deep Dive

// Unchecked — compiler doesn't force you to handle it
public User findUser(Long id) {
    if (id == null) {
        throw new IllegalArgumentException("User ID cannot be null");
        // No 'throws' needed in signature
        // This is a PROGRAMMING ERROR — caller passed null
    }
    if (id <= 0) {
        throw new IllegalArgumentException("User ID must be positive, got: " + id);
    }
    return userRepository.findById(id)
        .orElseThrow(() -> new NoSuchElementException("User not found: " + id));
}

// Real-world: Service validation
public Order createOrder(Cart cart) {
    Objects.requireNonNull(cart, "Cart cannot be null"); // throws NPE if null
    if (cart.isEmpty()) {
        throw new IllegalStateException("Cannot create order from empty cart");
    }
    return orderRepository.save(new Order(cart));
}
✓ Checked Exceptions — Pros
  • Compiler enforces handling — you can't forget
  • Self-documenting API — callers know what can fail
  • Forces you to think about failure scenarios upfront
  • Great for external dependencies (files, network, DB)
  • Easier to audit error handling in code review
✗ Checked Exceptions — Cons
  • Verbose — forces try-catch or throws everywhere
  • Pollutes method signatures up the call chain
  • Can lead to exception swallowing: catch(e) {}
  • Doesn't work with lambdas/streams natively
  • Tightly couples callers to implementation details
✓ Unchecked Exceptions — Pros
  • Cleaner code — no forced try-catch boilerplate
  • Works naturally with lambdas and streams
  • Used by most modern frameworks (Spring, Hibernate)
  • Doesn't pollute method signatures
  • Preferred by Effective Java (Joshua Bloch)
✗ Unchecked Exceptions — Cons
  • Easy to forget to handle them
  • No compiler safety net
  • Callers can't tell from the signature what might fail
  • Requires good Javadoc discipline
  • Can silently bubble up to the top if missed

The Golden Rule — When to Use Which

SituationUseWhy
External resource fails (file, network, DB)CheckedCaller can realistically recover
Invalid method argument passedUncheckedCaller made a programming error
Business rule violatedUncheckedModern approach — Spring style
Object in invalid stateUncheckedIllegalStateException pattern
Method not implementedUncheckedUnsupportedOperationException
Wrapping checked exceptionsUncheckedAt service boundaries, translate lower-level exceptions
Industry trend: Modern Java (Spring, Jakarta EE) overwhelmingly uses unchecked exceptions. Checked exceptions are mostly used in low-level I/O. If in doubt, go unchecked and document with @throws Javadoc.

try-catch-finally

The backbone of exception handling — every edge case, gotcha, and best practice you need to know.

Basic Structure

try {
    // Code that might throw an exception
    String result = riskyOperation();
    processResult(result);

} catch (SpecificException e) {
    // Handles SpecificException AND its subclasses
    // 'e' gives you full exception info
    log.error("Specific failure: {}", e.getMessage(), e);

} catch (AnotherException e) {
    // Multiple catch blocks — order matters! Most specific first!
    handleAnotherCase(e);

} finally {
    // ALWAYS executes — success, exception, or even return
    // Perfect for cleanup: close connections, release locks, etc.
    cleanup();
}

The finally Block — All the Tricky Cases

// Case 1: finally ALWAYS runs — even when there's a return in try
static int test1() {
    try {
        return 1;       // Return is "prepared" but finally runs first
    } finally {
        System.out.println("finally ran!"); // This prints
    }
}  // Returns 1 — finally ran before the actual return

// Case 2: finally return OVERRIDES try return
static int test2() {
    try {
        return 1;
    } finally {
        return 2;  // ⚠️ This OVERRIDES the try's return — returns 2!
    }
}  // Returns 2 — NEVER do this, very confusing

// Case 3: finally SUPPRESSES try's exception if it throws
static void test3() {
    try {
        throw new RuntimeException("from try");
    } finally {
        throw new RuntimeException("from finally"); // "from try" is LOST!
    }
}

// Case 4: Only way to skip finally — System.exit()
static void test4() {
    try {
        System.exit(0); // JVM shutdown — finally never runs
    } finally {
        System.out.println("This NEVER prints");
    }
}

Real-World: Database Connection Handling

// BEFORE try-with-resources — classic finally pattern
public User findUserById(long id) {
    Connection conn = null;
    PreparedStatement ps = null;
    ResultSet rs = null;
    try {
        conn = dataSource.getConnection();
        ps = conn.prepareStatement("SELECT * FROM users WHERE id = ?");
        ps.setLong(1, id);
        rs = ps.executeQuery();
        if (rs.next()) {
            return mapToUser(rs);
        }
        return null;
    } catch (SQLException e) {
        throw new DataAccessException("Failed to find user: " + id, e);
    } finally {
        // Must close in reverse order, each in its own try-catch
        if (rs   != null) try { rs.close();   } catch (SQLException ignored) {}
        if (ps   != null) try { ps.close();   } catch (SQLException ignored) {}
        if (conn != null) try { conn.close(); } catch (SQLException ignored) {}
    }
}
Why is cleanup so complex? If rs.close() throws, without the individual try-catch, ps and conn would never be closed — causing a resource leak. This is exactly why try-with-resources was invented in Java 7.

try Without catch — Valid!

// try + finally without catch — let exceptions propagate, but guarantee cleanup
public void transfer(Account from, Account to, double amount) {
    lock.lock();
    try {
        // Don't catch — if transfer fails, caller should handle it
        // But we MUST release the lock no matter what happens
        from.debit(amount);
        to.credit(amount);
    } finally {
        lock.unlock(); // GUARANTEED to run
    }
}

throw vs throws

Two keywords, completely different purposes — a common source of confusion for beginners.

throw — The Action
Used inside a method body to explicitly throw an exception instance.

Think of it as: "I am now throwing this exception."

Always followed by an exception instance.
throws — The Declaration
Used in a method signature to declare what checked exceptions a method might propagate.

Think of it as: "Caller, be aware — I might throw these."

Always followed by exception class names. 
// 'throws' in the METHOD SIGNATURE — just a warning to callers
public Order processPayment(Payment payment)
        throws PaymentException, InsufficientFundsException {

    // Validate input
    if (payment == null) {
        throw new IllegalArgumentException("Payment cannot be null"); // 'throw' the action
    }

    if (payment.getAmount() <= 0) {
        throw new IllegalArgumentException(
            "Payment amount must be positive: " + payment.getAmount());
    }

    Account account = accountRepo.find(payment.getAccountId());
    if (account.getBalance() < payment.getAmount()) {
        throw new InsufficientFundsException(        // 'throw' the action
            "Balance: " + account.getBalance() +
            ", Required: " + payment.getAmount()
        );
    }

    try {
        return paymentGateway.charge(payment);
    } catch (GatewayException e) {
        throw new PaymentException("Gateway rejected payment", e); // wrap and rethrow
    }
}

throws With Multiple Exceptions

// Declaring multiple checked exceptions
public void importData(String filePath)
        throws IOException, ParseException, ValidationException {
    // Each exception signals a different failure mode
    String content = Files.readString(Path.of(filePath));  // throws IOException
    List<Record> records = parser.parse(content);         // throws ParseException
    validator.validate(records);                          // throws ValidationException
}

throws Exception — The Nuclear Option

// ❌ BAD — 'throws Exception' tells callers NOTHING useful
public User getUser(long id) throws Exception {
    // Laziness — caller has no idea what might go wrong
}

// ✅ GOOD — specific and informative
public User getUser(long id) throws UserNotFoundException, DataAccessException {
    // Caller knows exactly what to handle
}
throws does NOT mean the exception will be thrown! It's a declaration of possibility. A method that declares throws IOException might never actually throw it — it's just telling callers to be prepared.
Java 7+

try-with-resources

Automatic resource management — Java's answer to resource leaks. Cleaner, safer, and handles suppressed exceptions.

The Problem It Solves

Before Java 7, properly closing resources required complex, nested finally blocks. Any exception during close would swallow the original exception. Resources would leak if you forgot a single null check.

❌ Before Java 7 — Error Prone
InputStream in = null;
try {
    in = new FileInputStream(file);
    // use stream...
} finally {
    if (in != null) {
        try { in.close(); }
        catch (IOException ignored) {}
    }
}
✅ Java 7+ — Clean
try (InputStream in =
    new FileInputStream(file)) {

    // use stream...

} // auto-closed here, always
// even on exception

How It Works Internally

The resource must implement AutoCloseable (or its subinterface Closeable). The compiler generates the try-finally code for you, including the suppressed exception handling.

// Multiple resources — closed in REVERSE order of declaration
try (
    Connection conn = dataSource.getConnection();         // closed 3rd
    PreparedStatement ps = conn.prepareStatement(sql);   // closed 2nd
    ResultSet rs = ps.executeQuery()                     // closed 1st
) {
    while (rs.next()) {
        // process rows
    }
} catch (SQLException e) {
    throw new DataAccessException("Query failed", e);
}
// All three resources guaranteed closed, even if processRows() throws

Suppressed Exceptions — The Hidden Gem

// What if BOTH the body AND close() throw?
// The body exception is PRIMARY, close() exception is SUPPRESSED
try (Connection conn = dataSource.getConnection()) {
    conn.execute("bad SQL"); // throws SQLException A (primary)
    // conn.close() also throws SQLException B (suppressed)
} catch (SQLException e) {
    // e = SQLException A (the original body exception)
    Throwable[] suppressed = e.getSuppressed(); // [SQLException B]
    for (Throwable s : suppressed) {
        log.warn("Suppressed during close: {}", s.getMessage());
    }
}

Creating Your Own AutoCloseable Resources

// Real-world: Connection pool wrapper, lock, timer, etc.
public class DatabaseTransaction implements AutoCloseable {
    private final Connection conn;
    private boolean committed = false;

    public DatabaseTransaction(DataSource ds) throws SQLException {
        conn = ds.getConnection();
        conn.setAutoCommit(false);
    }

    public Connection getConnection() { return conn; }

    public void commit() throws SQLException {
        conn.commit();
        committed = true;
    }

    @Override
    public void close() throws SQLException {
        try {
            if (!committed) conn.rollback(); // auto-rollback if not committed
        } finally {
            conn.close();
        }
    }
}

// Usage — clean, safe, guaranteed rollback on exception
try (DatabaseTransaction tx = new DatabaseTransaction(dataSource)) {
    userRepo.save(tx.getConnection(), user);
    auditLog.write(tx.getConnection(), event);
    tx.commit(); // only committed if BOTH succeed
} // auto-rollback if any exception thrown before commit

AutoCloseable vs Closeable

InterfaceMethodThrowsUse For
AutoCloseableclose()throws ExceptionGeneral resources — broadest contract
Closeableclose()throws IOExceptionI/O resources — extends AutoCloseable
Best practice: If implementing AutoCloseable, prefer making close() idempotent (safe to call multiple times) and document whether it can throw.

Multi-catch & Precise Rethrow

Catching multiple exceptions in one block, and how Java 7 improved rethrow semantics.

Multi-catch Syntax

// Before Java 7 — repetitive code
try {
    processRequest(req);
} catch (IOException e) {
    log.error("Request failed", e);
    throw new ServiceException("Request failed", e);
} catch (ParseException e) {
    log.error("Request failed", e);  // duplicate!
    throw new ServiceException("Request failed", e);
}

// Java 7+ multi-catch — clean and DRY
try {
    processRequest(req);
} catch (IOException | ParseException e) {
    // 'e' is effectively final — cannot be reassigned
    log.error("Request failed", e);
    throw new ServiceException("Request failed", e);
}

Rules and Gotchas

// ❌ Cannot catch related types (parent + child) in multi-catch
// IOException is parent of FileNotFoundException — compile error
catch (IOException | FileNotFoundException e) { } // COMPILE ERROR

// ❌ Cannot reassign 'e' in multi-catch — it's effectively final
catch (IOException | ParseException e) {
    e = new IOException("replaced"); // COMPILE ERROR
}

// ✅ Can call common methods since both types share Throwable methods
catch (IOException | ParseException e) {
    e.getMessage();     // OK — from Throwable
    e.getClass();       // OK — from Object
    e.printStackTrace(); // OK — from Throwable
}

Real-World: API Client with Multiple Failure Modes

public UserProfile fetchProfile(String userId) {
    try {
        HttpResponse response = httpClient.get("/api/users/" + userId);
        return objectMapper.readValue(response.getBody(), UserProfile.class);

    } catch (ConnectTimeoutException | SocketTimeoutException e) {
        // Network timeout — retryable
        throw new RetryableException("Network timeout fetching user: " + userId, e);

    } catch (JsonParseException | JsonMappingException e) {
        // Bad response format — not retryable
        throw new DataFormatException("Invalid user profile response", e);

    } catch (IOException e) {
        // Any other IO problem
        throw new ServiceException("Failed to fetch user profile", e);
    }
}

Precise Rethrow (Java 7)

// Before Java 7 — had to declare 'throws Exception' which is too broad
public void rethrowOld() throws Exception {
    try {
        operationThatThrowsAOrB();
    } catch (Exception e) {
        log.error("Failed", e);
        throw e; // Before Java 7: inferred as 'Exception', too broad
    }
}

// Java 7+ Precise Rethrow — compiler infers the actual thrown types
public void rethrowNew() throws IOException, ParseException {
    try {
        operationThatThrowsIOOrParse(); // throws IOException or ParseException
    } catch (Exception e) {
        log.error("Failed", e);
        throw e; // Java 7+: precisely inferred as IOException | ParseException
    }
    // Can declare precise types in throws clause despite catching 'Exception'!
}

Exception Chaining

Preserving the root cause through layers — essential for debugging production issues.

Why It Matters

In a layered architecture (Controller → Service → Repository → DB), exceptions cross multiple layers. Without chaining, you lose the original cause — making it nearly impossible to debug production issues.

❌ Root Cause Lost
catch (SQLException e) {
  throw new ServiceException(
    "Failed"
    // e is GONE! No cause!
  );
}
// Log shows:
// ServiceException: Failed
// (no idea WHY it failed)
✅ Full Chain Preserved
catch (SQLException e) {
  throw new ServiceException(
    "Failed", e // chain it!
  );
}
// Log shows:
// ServiceException: Failed
//   Caused by: SQLException: ...
//   Caused by: Connection reset

Complete Real-World Example — E-Commerce

// Layer 1: Repository
public class OrderRepository {
    public Order save(Order order) {
        try {
            return em.persist(order);
        } catch (PersistenceException e) {
            // Translate JPA exception to domain exception, preserve cause
            if (isDuplicateKey(e)) {
                throw new DuplicateOrderException("Order already exists: " + order.getId(), e);
            }
            throw new DataAccessException("Failed to persist order", e);
        }
    }
}

// Layer 2: Service
public class OrderService {
    public OrderConfirmation placeOrder(Cart cart) {
        try {
            Order order = buildOrder(cart);
            orderRepository.save(order);
            paymentService.charge(cart.getPaymentMethod(), order.getTotal());
            return new OrderConfirmation(order);
        } catch (DuplicateOrderException e) {
            throw e; // let it propagate as-is
        } catch (DataAccessException | PaymentException e) {
            throw new OrderPlacementException(
                "Failed to place order for cart: " + cart.getId(), e); // chain
        }
    }
}

// Caused by: PersistenceException: could not execute statement
//   at ...

Navigating the Chain Programmatically

public static <T extends Throwable> Optional<T> findCause(
        Throwable ex, Class<T> type) {
    Throwable current = ex;
    while (current != null) {
        if (type.isInstance(current)) {
            return Optional.of(type.cast(current));
        }
        current = current.getCause();
    }
    return Optional.empty();
}

// Usage: check if a DataAccessException was caused by a constraint violation
findCause(ex, SQLIntegrityConstraintViolationException.class)
    .ifPresent(e -> log.warn("Constraint violation: {}", e.getMessage()));

Custom Exceptions

Building meaningful, informative exceptions that make your domain model expressive and debugging easy.

Why Create Custom Exceptions?

Semantic Clarity
InsufficientFundsException tells you exactly what happened. RuntimeException tells you nothing.
Carry Domain Data
A custom exception can carry structured fields: error code, accountId, required amount — not just a string message.
Targeted Handling
Callers can catch UserNotFoundException specifically without catching all RuntimeExceptions.

The Full Template — Production-Grade Custom Exception

public class InsufficientFundsException extends RuntimeException {

    // Required for serialization
    private static final long serialVersionUID = 1L;

    // Domain-specific fields — carry context, not just a message
    private final String accountId;
    private final BigDecimal available;
    private final BigDecimal required;
    private final ErrorCode errorCode;

    // Constructor with domain data
    public InsufficientFundsException(
            String accountId, BigDecimal available, BigDecimal required) {
        super(String.format(
            "Account %s: available=%.2f, required=%.2f",
            accountId, available, required));
        this.accountId = accountId;
        this.available = available;
        this.required = required;
        this.errorCode = ErrorCode.INSUFFICIENT_FUNDS;
    }

    // Constructor with cause (for chaining)
    public InsufficientFundsException(
            String accountId, BigDecimal available,
            BigDecimal required, Throwable cause) {
        super(String.format(
            "Account %s: available=%.2f, required=%.2f",
            accountId, available, required), cause);
        this.accountId = accountId;
        this.available = available;
        this.required = required;
        this.errorCode = ErrorCode.INSUFFICIENT_FUNDS;
    }

    // Getters — immutable, no setters
    public String     getAccountId() { return accountId; }
    public BigDecimal getAvailable()  { return available; }
    public BigDecimal getRequired()   { return required; }
    public ErrorCode  getErrorCode()  { return errorCode; }
}

// Usage — informative and type-safe
throw new InsufficientFundsException(account.getId(), balance, amount);

// Catching — access domain data directly
catch (InsufficientFundsException e) {
    return ApiResponse.error(422,
        "Insufficient funds",
        Map.of("available", e.getAvailable(), "required", e.getRequired())
    );
}

Exception Anti-Patterns

The most common mistakes — what not to do and why, with real-world consequences.

1. Swallowing Exceptions (The Worst)

// ❌ TERRIBLE — exception is silently swallowed
try {
    processPayment(order);
} catch (Exception e) {
    // Do nothing — "it'll probably be fine"
}
// Payment failed. Nobody knows. Customer charged but order not placed.
// This causes data corruption, financial loss, and silent failures in production.

// ✅ Minimum acceptable — at least log it
catch (Exception e) {
    log.error("Payment processing failed for order {}", order.getId(), e);
    throw new PaymentException("Payment failed", e);
}

2. Catching Exception Too Broadly

// ❌ Catches EVERYTHING including NullPointerException, bugs, etc.
catch (Exception e) {
    return "Something went wrong";
}
// A NullPointerException caused by YOUR BUG will silently become
// "Something went wrong" — impossible to debug!

// ✅ Catch what you expect, let others propagate
catch (UserNotFoundException e) {
    return ResponseEntity.notFound().build();
} catch (ValidationException e) {
    return ResponseEntity.badRequest().body(e.getErrors());
}
// NullPointerExceptions will now surface properly

3. Using Exceptions for Flow Control

// ❌ BAD — exceptions are expensive; this is a logic problem
public boolean userExists(Long id) {
    try {
        userRepository.findById(id); // throws if not found
        return true;
    } catch (UserNotFoundException e) {
        return false; // exception as if-else — terrible!
    }
}

// ✅ GOOD — use Optional or boolean return
public boolean userExists(Long id) {
    return userRepository.existsById(id);
}
// Or:
// return userRepository.findById(id).isPresent();

4. Losing Stack Trace on Rethrow

// ❌ Stack trace is RESET — you lose the original location
catch (SQLException e) {
    throw new DataException(e.getMessage()); // Message only, no cause!
}

// ✅ Chain it — preserve the original stack trace
catch (SQLException e) {
    throw new DataException("Query failed", e); // e is the cause
}

5. Throwing Exception or Throwable

// ❌ Forces every caller to handle the broadest possible thing
public void processData() throws Exception { }
public void doWork() throws Throwable { }

// ✅ Be specific
public void processData() throws DataProcessingException { }

6. Not Closing Resources (Resource Leaks)

// ❌ If readLine() throws, stream is never closed — CONNECTION LEAK
BufferedReader reader = new BufferedReader(new FileReader(file));
String line = reader.readLine();
reader.close();

// ✅ Always use try-with-resources
try (BufferedReader reader = new BufferedReader(new FileReader(file))) {
    String line = reader.readLine();
}

7. Logging AND Rethrowing

// ❌ Logs the same exception multiple times — pollutes logs
catch (SQLException e) {
    log.error("DB error", e);          // logged here
    throw new DataException("DB error", e); // AND will be logged upstream too!
}

// ✅ Log OR rethrow — not both
// Rethrow without logging — let the top-level handler log it once
catch (SQLException e) {
    throw new DataException("DB error", e); // logged once at the top
}

// OR log and recover without rethrowing
catch (SQLException e) {
    log.error("DB error, returning empty list", e);
    return Collections.emptyList(); // handled — not rethrown
}

public class SafeThreadPoolExecutor extends ThreadPoolExecutor {

    public SafeThreadPoolExecutor(int coreSize, int maxSize) {
        super(coreSize, maxSize, 60L, TimeUnit.SECONDS,
              new LinkedBlockingQueue<>(),
              new ThreadFactoryBuilder()
                  .setNameFormat("safe-worker-%d")
                  .setUncaughtExceptionHandler((t, e) ->
                      log.error("Worker {} died", t.getName(), e))
                  .build());
    }

    @Override
    protected void afterExecute(Runnable r, Throwable t) {
        super.afterExecute(r, t);
        // If task was submitted via submit(), t is null but exception is in Future
        if (t == null && r instanceof Future<?> f) {
            try {
                if (f.isDone()) f.get();
            } catch (ExecutionException e) {
                t = e.getCause();
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
        if (t != null) {
            log.error("Task failed with exception", t);
            metrics.increment("thread_pool.task_failures");
        }
    }
}

Exceptions in Streams & Lambdas

One of Java 8's biggest pain points — checked exceptions and functional interfaces don't mix. Here's every solution.

The Core Problem

// ❌ This WON'T COMPILE — Files.readString() throws checked IOException
// but Function doesn't declare any checked exceptions
List<String> contents = paths.stream()
    .map(p -> Files.readString(p))  // COMPILE ERROR: Unhandled IOException
    .collect(Collectors.toList());

Solution 1: Wrap in Unchecked (Quick & Dirty)

// Wrap the checked exception at the point of use
List<String> contents = paths.stream()
    .map(p -> {
        try {
            return Files.readString(p);
        } catch (IOException e) {
            throw new UncheckedIOException(e); // wrap in unchecked
        }
    })
    .collect(Collectors.toList());
// Problem: ONE bad file aborts the ENTIRE stream

Solution 2: Throwing Function Wrapper (Reusable)

// Generic utility that wraps any throwing function
@FunctionalInterface
public interface ThrowingFunction<T, R> {
    R apply(T t) throws Exception;

    static <T, R> Function<T, R> wrap(ThrowingFunction<T, R> fn) {
        return t -> {
            try { return fn.apply(t); }
            catch (Exception e) { throw new RuntimeException(e); }
        };
    }
}

// Clean usage
List<String> contents = paths.stream()
    .map(ThrowingFunction.wrap(Files::readString))
    .collect(Collectors.toList());
// Compiles! Checked exception wrapped automatically.

Solution 3: Try-Result Pattern (Best for Production)

// Process ALL files, collect successes and failures separately
public record Try<T>(T value, Throwable error) {
    public boolean isSuccess() { return error == null; }
    public boolean isFailure() { return error != null; }

    public static <T> Try<T> of(Callable<T> supplier) {
        try { return new Try<>(supplier.call(), null); }
        catch (Exception e) { return new Try<>(null, e); }
    }
}

// Map each path, capturing success OR failure
List<Try<String>> results = paths.stream()
    .map(p -> Try.of(() -> Files.readString(p)))
    .collect(Collectors.toList());

// Process results
List<String> successes = results.stream()
    .filter(Try::isSuccess)
    .map(Try::value)
    .collect(Collectors.toList());

results.stream()
    .filter(Try::isFailure)
    .forEach(t -> log.error("Failed to read file", t.error()));

// No files are skipped silently — all failures are captured and logged
Best choice depends on context: If ANY failure should abort the entire operation, use Solution 1. If failures are expected and you want partial results, use Solution 3. If you're writing reusable library code, use Solution 2.

    return CompletableFuture.completedFuture(
        PaymentResult.pending("Payment queued for processing"));
}

Saga Pattern — Compensating Transactions

// When a multi-step distributed transaction partially fails,
// compensate (undo) the steps that already succeeded
public OrderConfirmation placeOrder(Order order) {
    String paymentId = null;
    String inventoryReservationId = null;

    try {
        // Step 1: Reserve inventory
        inventoryReservationId = inventoryService.reserve(order.getItems());

        // Step 2: Charge payment
        paymentId = paymentService.charge(order.getPayment());

        // Step 3: Create shipment
        shippingService.schedule(order);

        return OrderConfirmation.success(order);

    } catch (ShippingException e) {
        // Step 3 failed — compensate steps 1 and 2
        safeCancel(() -> paymentService.refund(paymentId));
        safeCancel(() -> inventoryService.release(inventoryReservationId));
        throw new OrderFailedException("Shipping unavailable", e);

    } catch (PaymentException e) {
        // Step 2 failed — compensate step 1 only
        safeCancel(() -> inventoryService.release(inventoryReservationId));
        throw new OrderFailedException("Payment failed", e);
    }
}

private void safeCancel(Runnable compensation) {
    try { compensation.run(); }
    catch (Exception e) {
        // Compensation failed! Alert ops — manual intervention needed
        log.error("CRITICAL: Compensation failed — manual intervention required", e);
        alertOps.critical("Saga compensation failure", e);
    }
}

Exception Performance Impact

Stack trace creation is expensive — understanding the cost and when to optimise.

Why Exceptions Are Expensive

When an exception is created, the JVM calls fillInStackTrace() which walks the entire call stack and captures every frame. This is the expensive part — not the throw/catch itself.

OperationRelative CostNotes
Normal method call~1xBaseline
new Exception()~100-1000xfillInStackTrace() — stack depth matters
throw / catch~1-2xThe throw/catch itself is cheap
Exception with deep stackUp to 10,000x100-deep call stack = huge overhead

Overriding fillInStackTrace() for High-Frequency Exceptions

// When you throw the same exception thousands of times per second
// (e.g. validation in a hot path), skip stack trace collection
public class FastValidationException extends RuntimeException {

    private static final long serialVersionUID = 1L;

    public FastValidationException(String message) {
        super(message, null, true, false); // writableStackTrace = false
    }

    // OR override directly:
    @Override
    public synchronized Throwable fillInStackTrace() {
        return this; // Do nothing — massive performance win
    }
}
// Trade-off: You lose stack trace info — only use for EXPECTED, frequent exceptions
// where you already know the location (e.g. input validation)

Static Exception Instances (Use with Extreme Care)

// Pre-created exception — zero allocation cost
public class CacheException extends RuntimeException {
    // Singleton for cache miss — thrown millions of times
    public static final CacheException CACHE_MISS =
        new CacheException("Cache miss");

    private CacheException(String msg) {
        super(msg, null, true, false);
    }
}
// WARNING: No stack trace. No context. Only suitable for very
// specific, well-understood hot paths. Not for general use.

JIT Optimisation of Exception-Free Paths

// The JVM's JIT compiler optimises code that never throws.
// If your hot path throws exceptions 99% of the time, JIT can't optimise it.
// Profile BEFORE optimising — exceptions are rarely the bottleneck.

// ❌ Using exception for expected flow — prevents JIT optimisation
try {
    return cache.get(key); // throws CacheException on miss
} catch (CacheException e) {
    return db.fetch(key);
}

// ✅ Use null or Optional — no exception, JIT can optimise freely
String cached = cache.getOrNull(key);
return cached != null ? cached : db.fetch(key);
Rule of thumb: Exceptions should be exceptional. If you're throwing more than a few hundred per second on a hot path, that's a code smell — not a performance tuning opportunity. Fix the logic first.

Interview Q&A — All Levels

50 questions from beginner to senior. Click any question to reveal the answer.

Beginner

What is the difference between Error and Exception? ▼
Exception is for recoverable application-level problems (file not found, invalid input). Error is for unrecoverable JVM-level problems (OutOfMemoryError, StackOverflowError). Both extend Throwable. You should handle Exceptions; you should generally not catch Errors.
What is the difference between checked and unchecked exceptions? ▼
Checked exceptions extend Exception (not RuntimeException) and must be either caught or declared in the method signature with throws. The compiler enforces this. Examples: IOException, SQLException.

Unchecked exceptions extend RuntimeException. No compiler enforcement — you can ignore them. Examples: NullPointerException, IllegalArgumentException.

Rule: Use checked for external/recoverable failures. Use unchecked for programming errors and business rules.
Does finally always execute? ▼
Almost always — yes. The only exceptions (no pun intended) are: System.exit() being called, the JVM crashing, or the thread being forcibly killed. Even if the try block throws an exception, even if there's a return statement — finally runs.
What is the output of this code? try { return 1; } finally { return 2; } ▼
The output is 2. The finally block runs before the actual return, and its return 2 overrides the return 1 from the try block. This is considered very bad practice and should never appear in real code — it's confusing and hides the original return value.
What is the difference between throw and throws? ▼
throw is an action inside a method body — it creates and throws an exception instance: throw new IllegalArgumentException("bad input").

throws is a declaration in the method signature — it tells callers what checked exceptions might propagate: public void read() throws IOException.
Can you have try without catch? ▼
Yes! Two valid forms: try { } finally { } (no catch, but must have finally) and try-with-resources try (Resource r = ...) { } (no catch or finally required). What's NOT valid: try { } alone with no catch or finally.
What happens if a catch block throws an exception? ▼
The new exception from the catch block propagates up the call stack. The original exception that triggered the catch block is lost (unless you chain them with the cause constructor). The finally block still runs first before propagation. This is why you should almost never throw from within catch without chaining the original exception.

Intermediate

What are suppressed exceptions? ▼
In try-with-resources, if both the body and the resource's close() method throw exceptions, the body exception is the primary one and the close exception is suppressed — attached to the primary exception. You can access them via e.getSuppressed(). Before Java 7, the close exception would silently swallow the body exception.
What is exception chaining and why is it important? ▼
Exception chaining means wrapping one exception inside another using the cause constructor parameter: throw new ServiceException("msg", originalException). It's critical in layered architectures because it preserves the entire failure history. Without chaining, you lose the root cause — making production debugging nearly impossible. The full chain appears in logs as "Caused by:" sections.
Can you catch multiple exceptions in one catch block? ▼
Yes, since Java 7 using the multi-catch syntax: catch (IOException | ParseException e) { }. Key rules: (1) the exception variable e is effectively final — you can't reassign it; (2) you cannot list related types (parent and child) in the same multi-catch — that's a compile error; (3) any method both types share from Throwable can be called.
What is the AutoCloseable interface and how does it relate to try-with-resources? ▼
Any class implementing AutoCloseable (which has a single close() method) can be used in try-with-resources. The compiler automatically generates the finally block to call close(). If close() throws while an exception is already in flight, the close exception is added as suppressed (not lost). Closeable extends AutoCloseable and is for I/O specifically.
When should you create a custom exception? ▼
Create custom exceptions when: (1) you need to carry domain-specific context beyond a message string (userId, errorCode, available balance); (2) callers need to distinguish this failure mode from others — they can't do that if you throw a generic RuntimeException; (3) you want semantic clarity in code — InsufficientFundsException is far more readable than RuntimeException("insufficient").
What is the problem with using exceptions for control flow? ▼
Three problems: (1) Performance — constructing an exception requires filling the stack trace which is expensive (100-1000x normal method call); (2) Readability — exceptions signal exceptional conditions; using them for "user not found" pollutes the semantics; (3) JIT pessimisation — the JVM can't optimise code paths that routinely throw. Use Optional, null, or boolean returns for expected conditions.
Why should you never swallow exceptions? ▼
Swallowing exceptions (catch (Exception e) { }) is one of the worst practices because: failures become invisible — you have no idea the code failed; subsequent code runs in an invalid state potentially causing data corruption; debugging becomes nearly impossible since there's no error signal; what looks like a rare bug might actually be a 100% failure rate masked by empty catch blocks.

Advanced

How do you handle checked exceptions in Java 8 streams and lambdas? ▼
Three approaches: (1) Wrap inline: wrap with try-catch inside the lambda and throw an unchecked exception — aborts the stream on first failure; (2) ThrowingFunction wrapper: create a generic functional interface that wraps checked exceptions; (3) Try-Result pattern: map each element to a Try<T> that holds either a value or error — processes all elements, collects failures separately. Choice depends on whether partial success is acceptable.
How are exceptions handled in CompletableFuture? ▼
Exceptions are captured in the future itself. get() throws checked ExecutionException wrapping the real cause; join() throws unchecked CompletionException. Handle without blocking using: exceptionally() for providing fallback values, handle() for processing both success and failure, whenComplete() for side effects (logging/metrics) without changing the result. Always unwrap with e.getCause() to get the original exception.
What happens to exceptions thrown in background threads? ▼
By default, they're silently swallowed — the thread dies, nobody knows. Solutions: (1) Set Thread.setDefaultUncaughtExceptionHandler() for all threads; (2) For ExecutorService.submit(), the exception is captured in the Future — you get it when calling future.get(); (3) For ExecutorService.execute(), it goes to UncaughtExceptionHandler; (4) Override afterExecute() in custom ThreadPoolExecutor to catch both.
What is @ControllerAdvice and how does exception handling priority work? ▼
@ControllerAdvice (or @RestControllerAdvice) is a global exception handler for all controllers. Priority rules: (1) most specific exception type wins over more general; (2) handler in the controller class itself takes precedence over @ControllerAdvice; (3) among multiple @ControllerAdvice classes, order them with @Order or implement Ordered; (4) @ExceptionHandler can declare multiple exception types.
What is Spring's exception translation for @Repository? ▼
The @Repository annotation activates Spring's PersistenceExceptionTranslationPostProcessor. This AOP proxy intercepts JPA/JDBC exceptions thrown from @Repository beans and translates them to Spring's DataAccessException hierarchy. This makes your persistence layer vendor-neutral — the service layer catches DataAccessException, not Hibernate-specific exceptions. Without this, JPA implementation details leak into the service layer.
How does OptimisticLocking work and what exception does it throw? ▼
JPA's @Version field enables optimistic locking. When you read an entity (version=1) and try to save it, JPA checks the DB still has version=1. If another thread saved it (version=2), the update fails and throws OptimisticLockException (JPA) / ObjectOptimisticLockingFailureException (Spring). The standard response is to reload the entity and retry. This prevents lost updates in concurrent scenarios without using database-level locks.

Senior / Expert

Why is creating exceptions expensive and how can you mitigate it? ▼
Expense comes from fillInStackTrace() — the JVM walks the entire call stack capturing every frame. On a 50-frame stack this can be 100-1000x more expensive than a normal method call. Mitigations: (1) Override fillInStackTrace() to return this without walking the stack — use for high-frequency expected exceptions; (2) Use the constructor: super(msg, null, true, false) where the last boolean disables stack trace; (3) Use pre-allocated static exception instances (only for stateless, very hot paths). Never optimise prematurely — benchmark first.
Why should you never swallow InterruptedException? ▼
InterruptedException signals that a thread was requested to stop — it's a cooperative shutdown mechanism. Swallowing it clears the interrupt flag on the thread, breaking the entire shutdown chain. Thread pools won't shut down cleanly, awaitTermination() blocks forever, and other code checking Thread.isInterrupted() will get wrong answers. The correct options: (1) re-throw InterruptedException, or (2) call Thread.currentThread().interrupt() to restore the interrupt status before catching it.
What is the RFC 7807 ProblemDetail standard and how does Spring Boot 3 support it? ▼
RFC 7807 defines a standard JSON format for HTTP API error responses with fields: type (URI), title, status, detail, instance. Spring Boot 3 / Spring 6 has native ProblemDetail class. Configure with spring.mvc.problemdetails.enabled=true to auto-translate Spring MVC exceptions, or return ProblemDetail from @ExceptionHandler methods. Content-Type is application/problem+json. Allows adding custom properties via setProperty().
How do you design exception handling for microservices with Feign clients? ▼
Implement a custom ErrorDecoder for each Feign client. Parse the structured error response body, map errorCode fields to specific exception types, and distinguish retryable (5xx, timeout) from non-retryable (4xx) errors. For retryable errors, throw a RetryableException which Feign knows to retry. For non-retryable errors, throw domain exceptions. Combine with Resilience4j @CircuitBreaker and @Retry annotations, with separate ignoreExceptions/retryExceptions configuration for each error class.
What is the Saga pattern and how does exception handling work in it? ▼
Saga manages distributed transactions as a sequence of local transactions with compensating actions for rollback. When step N fails, you execute compensation transactions for steps 1 through N-1 in reverse order. Exception handling is critical: compensations must be wrapped in their own try-catch (safe cancellation) because compensation failure is worse than original failure — it leaves the system in an inconsistent state. Failed compensations must trigger alerts for manual intervention. Each compensation should be idempotent for retryability.
What are the best practices for logging exceptions in production? ▼
Key rules: (1) Always pass the exception as the LAST argument to SLF4J — log.error("msg", ex) — this appends the full stack trace; (2) Include rich context: orderId, userId, what operation was attempted; (3) Use MDC for correlation IDs so all log lines for a request are traceable; (4) Log OR rethrow — not both (avoid duplicate log entries); (5) Never log sensitive data (passwords, card numbers, tokens); (6) Choose the right level: ERROR for unexpected, WARN for expected failures worth monitoring, INFO for normal 404s/validation; (7) Use structured JSON logging for searchability.
Tricky: What is "precise rethrow" introduced in Java 7? ▼
Before Java 7, if you caught Exception and re-threw it, the compiler treated the throw as type Exception — forcing you to declare throws Exception (too broad). Java 7 introduced precise rethrow: the compiler now analyses what can actually be thrown in the try block. So you can catch Exception, rethrow it, and still declare only the specific subtypes. Example: a try block that only throws IOException and ParseException allows throws IOException, ParseException even though you catch and rethrow as bare Exception.
How does Spring AOP interact with exception handling, and what is the pitfall? ▼
Spring AOP proxies wrap bean methods. If an exception is thrown inside a @Transactional method and caught by AOP advice, the transaction may not roll back correctly. Key rules: (1) @Transactional only rolls back on RuntimeException and Error by default — checked exceptions commit; (2) If you catch an exception inside a @Transactional method and don't rethrow, the transaction commits — it doesn't know an error occurred; (3) Set rollbackFor = Exception.class to rollback on checked exceptions too; (4) Self-invocation (calling a @Transactional method on this) bypasses the proxy and won't trigger rollback.
What is serialVersionUID and why does it matter for exceptions? ▼
serialVersionUID is a version marker for Java serialization. It matters for exceptions in distributed systems where exceptions are serialized and sent over the wire (RMI, some messaging systems). If sender and receiver have different class versions without explicit serialVersionUID, Java generates one from the class structure — any change (adding a field) produces a different ID, causing InvalidClassException on deserialization. Always declare private static final long serialVersionUID = 1L in custom exceptions. Increment it when making incompatible changes.
