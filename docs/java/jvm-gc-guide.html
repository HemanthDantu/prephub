What is Garbage Collection?

Automatic memory management — why Java doesn't need free() and what the JVM does behind the scenes.

The Problem GC Solves

In languages like C/C++, you manually allocate memory with malloc() and free it with free(). Forget to free and you get a memory leak. Free twice and you get a crash. Free the wrong pointer and you corrupt memory. Java's GC eliminates this entire class of bugs by automatically reclaiming memory that is no longer needed.

Real World Analogy
A Restaurant Kitchen

Imagine every object is a plate. When a chef is done with a plate, they put it down. In C, the chef must manually return each plate to the shelf — forget one and the kitchen runs out of plates (memory leak). With Java's GC, a kitchen assistant (the garbage collector) watches which plates are actively being used and periodically collects all abandoned plates and puts them back — automatically.

The Core Concept — Reachability

GC doesn't track whether you're "done" with an object — it tracks whether the object is reachable. An object is alive if any live part of the program holds a reference to it, directly or transitively. An object is garbage when nothing in the running program can possibly reach it anymore.

// Object lifecycle example
public void processOrder() {
    Order order = new Order();      // Object created — reachable via 'order' variable
    order.setItems(getItems());
    save(order);                     // Still reachable
    // method returns — 'order' variable goes out of scope
    // Order object is now UNREACHABLE — eligible for GC
    // GC will reclaim it at some future point (not immediately)
}

// Making an object unreachable explicitly
List<User> users = loadAllUsers(); // holds reference to large list
process(users);
users = null;                       // Dereference — list now eligible for GC
// (but often unnecessary — just let the variable go out of scope)

What GC Does NOT Guarantee

Timing
GC runs when the JVM decides — not when an object becomes unreachable. You cannot force it with System.gc() (it's only a hint).
Immediacy
An unreachable object can live in memory for seconds or minutes before being collected. Your heap size determines how long this matters.
Zero Pauses
GC needs to stop or slow the application to work safely. Even modern collectors like ZGC have some pause, just very short.

GC Phases at a High Level

1
Mark
Starting from GC roots (stack variables, static fields, etc.), traverse all object references and mark every reachable object as "alive".
2
Sweep
Scan the heap. Any object NOT marked as alive is garbage — reclaim its memory.
3
Compact (optional)
Move surviving objects together to eliminate fragmentation — now allocation is just a pointer bump.
Key insight: Java GC is a tracing collector — it traces live objects, not dead ones. It never looks at garbage; it marks everything that's alive, and everything it didn't mark is garbage by definition.

PermGen vs Metaspace (Java 7 vs Java 8)

Java 7 — PermGen (Obsolete)

Fixed-size region on the heap. If you loaded many classes (frameworks, dynamic proxies), you'd get java.lang.OutOfMemoryError: PermGen space. Common in old app servers. You had to tune -XX:MaxPermSize.

Java 8+ — Metaspace (Current)

Moved to native memory (outside heap). Grows dynamically — no fixed limit by default. No more PermGen OOM. Can still be bounded with -XX:MaxMetaspaceSize. Much better for framework-heavy apps.

TLAB — Thread-Local Allocation Buffers

Each thread gets its own small private chunk of Eden space called a TLAB. Object allocation is just a pointer bump within the TLAB — no synchronisation needed, making allocation extremely fast (nanoseconds). When the TLAB fills up, the thread requests a new one.

// JVM flag to see TLAB usage
-XX:+PrintTLAB                  // print TLAB statistics
-XX:TLABSize=512k               // set TLAB size
-XX:-UseTLAB                    // disable TLAB (not recommended)

Object Lifecycle & Aging

How objects are born, age, get promoted, and eventually die — the full journey through heap generations.

The Complete Object Journey

1
Allocation in Eden
New object created via new. Allocated in the thread's TLAB within Eden space. Age = 0. If the object is large (>threshold), it goes directly to Old Gen (humongous objects in G1).
2
Eden fills up → Minor GC triggered
JVM pauses application (Stop-The-World). All live objects in Eden are copied to Survivor S0. Dead objects in Eden are wiped. Age of surviving objects incremented to 1.
3
Survivor bouncing
On next Minor GC, live objects from Eden + S0 are copied to S1 (always the empty survivor). Dead objects in S0 are wiped. Age incremented to 2. S0 and S1 roles alternate each GC.
4
Promotion to Old Gen
When an object's age reaches the tenuring threshold (default 15, configurable), it's promoted to Old Generation. Also promoted early if Survivor space is full.
5
Major / Full GC
Old Gen fills up → Major GC collects Old Gen. If heap is fully exhausted and GC can't free enough → OutOfMemoryError.

Object Age & Promotion

// Object header contains 4 bits for GC age (0-15)
-XX:MaxTenuringThreshold=15        // default — max age before promotion
-XX:InitialTenuringThreshold=7     // starting threshold (adaptive)

// Dynamic tenuring: JVM adjusts threshold based on survivor space usage
// If 50% of survivor space would be occupied by objects of age N+,
// threshold is set to N, promoting older objects early
-XX:TargetSurvivorRatio=50         // default 50% — tune if you see early promotion

// Large object bypass — goes directly to Old Gen
-XX:PretenureSizeThreshold=1m      // objects > 1MB bypass Young Gen entirely

Premature Promotion — A Common Problem

When Survivor spaces are too small, short-lived objects that would normally die in Young Gen get promoted to Old Gen prematurely — polluting Old Gen with garbage and causing unnecessary Major GCs.

// Symptom: Old Gen filling up fast despite mostly short-lived objects
// Diagnosis: check GC logs for "Desired survivor size" vs actual

// Fix 1: Increase Young Gen size
-XX:NewRatio=2       // Old:Young ratio — default 2 (Old is 2x Young)
-XX:NewSize=512m     // explicit Young Gen size
-XX:SurvivorRatio=8  // Eden:Survivor ratio — default 8 (Eden is 8x each Survivor)

// Fix 2: Reduce tenuring threshold to promote objects faster
-XX:MaxTenuringThreshold=4  // promote after 4 GC cycles, not 15

Generational Hypothesis

The foundational observation that makes generational GC so efficient — and when it breaks down.

The Weak Generational Hypothesis

The entire generational GC design is built on one empirical observation: most objects die young. Studies across many languages and applications consistently show that 80-98% of objects become unreachable within milliseconds of creation.

Real World Example — Web Application
Per-Request Objects in a Spring Application

Every HTTP request creates: an HttpServletRequest wrapper, DTO objects, Optional wrappers, Stream intermediaries, StringBuilder buffers, Jackson deserialization objects. All of these are created for a single request and are garbage the moment the response is sent — typically in under 100ms. They never need to outlive the request.

Meanwhile, Spring beans (ApplicationContext, repositories, services), database connection pools, thread pools, and caches are created once and live for the lifetime of the application — they're long-lived by design.

Why Generational GC is Efficient

Minor GC — Fast & Frequent
Only scans Young Gen (~10-20% of heap)
Most objects are already dead — very few survive
Copying survivors is proportional to live objects (few)
Typical pause: 1-100ms
Frequency: every few seconds under load
Major/Full GC — Slow & Rare
Must scan entire heap
Old Gen mostly contains live objects (expensive to scan)
Compaction is expensive
Typical pause: 100ms - several seconds
Frequency: minutes to hours (ideally)

When the Hypothesis Breaks Down

Some workloads violate the generational hypothesis — causing GC to perform poorly:

ViolationExampleSymptomFix
Long-lived temp objectsLarge in-memory caches, session stateOld Gen fills fastOff-heap caching (Redis, Ehcache off-heap)
Objects referencing young onesOld object holds reference to new objectCard table overheadRemember sets / write barriers handle this
Very large allocation rateStreaming data processingConstant Minor GCsZGC/Shenandoah, object pooling
Humongous objectsAllocating large byte arraysDirect Old Gen pressureBuffer pooling, streaming I/O
Card Table & Write Barrier: When an Old Gen object holds a reference to a Young Gen object, the GC must know about it during Minor GC. The JVM maintains a "card table" — a bitmap of 512-byte card regions. When any reference in a card changes (write barrier), that card is marked "dirty". Minor GC only scans dirty Old Gen cards, not the entire Old Gen.

GC Roots & Reachability

The starting points of liveness tracing — understanding exactly what keeps objects alive.

What are GC Roots?

GC starts its mark phase from a set of definitive starting points called GC roots. Any object reachable from a GC root (directly or through a chain of references) is considered alive. Everything else is garbage.

The GC Root Set
Thread stacks — local variables and method parameters in all active thread frames
Static fields — all static fields of loaded classes
JNI references — references held by native code via JNI
Synchronization monitors — objects used as locks currently held
JVM internals — classes loaded by bootstrap classloader, exception classes, interned strings
Common Memory Leak Roots
Static collections — adding to a static HashMap/List and never removing
ThreadLocal — storing objects in ThreadLocal without remove()
Listeners/Callbacks — registering event listeners and never deregistering
Caches — unbounded caches that grow forever
Inner class references — non-static inner class holds implicit reference to outer class

Reference Chains — How GC Traces

// GC traces the entire object graph from roots
static Map<String, Session> sessions = new HashMap<>();  // GC ROOT (static field)
//   └── HashMap object (ALIVE — reachable from static field)
//       └── Session object (ALIVE — reachable from HashMap)
//           └── User object (ALIVE — reachable from Session)
//               └── List<Order> (ALIVE — reachable from User)
//                   └── Order objects (ALIVE)
// ALL of these stay in memory as long as sessions map holds them!

// After removing from the map:
sessions.remove("session123");
// Session and everything it referenced MAY now be unreachable
// (unless something else also references them)

The ThreadLocal Leak — Classic Senior Interview Topic

public class RequestContextHolder {
    // ThreadLocal backed by Thread → ThreadLocalMap → (WeakRef key, value)
    private static final ThreadLocal<RequestContext> context =
        new ThreadLocal<>();

    public static void set(RequestContext ctx) { context.set(ctx); }
    public static RequestContext get() { return context.get(); }

    // ❌ LEAK: in a thread pool, threads are reused.
    // ThreadLocal value stays in the thread's ThreadLocalMap FOREVER
    // if you don't remove it. RequestContext (and all it references)
    // is never GC'd even though the request is done.

    // ✅ ALWAYS call remove() when done — especially with thread pools
    public static void clear() { context.remove(); }
}

// In Spring, use a Filter to ensure cleanup:
public class RequestContextFilter implements Filter {
    public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain) {
        try {
            RequestContextHolder.set(new RequestContext(req));
            chain.doFilter(req, res);
        } finally {
            RequestContextHolder.clear(); // ALWAYS clean up
        }
    }
}

Mark-Sweep-Compact Algorithm

The fundamental algorithm all GC collectors are built upon — and why each phase has its trade-offs.

Phase 1: Mark

Mark Phase STW in simple collectors

Starting from GC roots, traverse every reachable object reference recursively. Set a "marked" bit in each reachable object's header. Objects not marked after traversal are garbage.

// Conceptual traversal (actual JVM uses efficient bitwise marking)
void mark(Object obj) {
    if (obj == null || isMarked(obj)) return;
    setMarked(obj);
    for (Object ref : getReferences(obj)) {
        mark(ref);  // DFS traversal of object graph
    }
}

Phase 2: Sweep

Sweep Phase STW

Linear scan of the entire heap. For each object: if marked → clear the mark bit (reset for next GC). If not marked → reclaim the memory (add to free list). Problem: creates fragmentation — gaps between live objects.

Phase 3: Compact

Compact Phase STW — most expensive

Move all live objects to one end of the heap. Update all references to point to new locations. The result is a contiguous block of free space — allocation becomes a simple pointer bump again. Expensive but eliminates fragmentation.

Why Compaction Matters

Without Compaction — Fragmented

Free space scattered in small chunks. Allocating a 1MB object fails even if 10MB total is free (but in 100 × 100KB fragments). Forces a GC even though enough memory exists. Performance degrades over time.

With Compaction — Clean

All free space in one contiguous block. Allocation is a simple pointer increment — near-zero cost. GC can be deferred longer. Consistent performance.

Copying Collection (Used in Young Gen)

An alternative to compact — instead of moving objects in-place, copy live objects to a fresh "to" space. The "from" space is then entirely empty. Faster than compacting because you only touch live objects (which are few in Young Gen), and the entire from-space is freed atomically.

// Why Young Gen uses copying — most efficient when few objects survive
// Cost is proportional to LIVE objects, not total space
// If 99% of Eden objects are dead, copying 1% is very fast
// Eden → S0 (live objects), Eden is then clean — no sweep needed

Tri-Color Marking (Concurrent GC)

Concurrent collectors (G1, ZGC) use tri-color marking to allow GC to run while the application continues:

⬜ White
Not yet visited. Either unreachable (garbage) or not yet reached during marking.
🟡 Grey
Visited but references not yet processed. In the marking worklist.
⬛ Black
Fully processed — object and all its references are scanned and marked.

The challenge: the application can modify references while marking is in progress. Write barriers detect these changes and re-mark affected objects (SATB — Snapshot At The Beginning, used in G1).

Serial GC & Parallel GC

The original collectors — Serial for single-core/small apps, Parallel for throughput-oriented batch workloads.

Serial GC — Single-Threaded

The simplest collector. Uses a single thread for all GC work. During GC, all application threads stop (full Stop-The-World). After GC, all application threads resume.

-XX:+UseSerialGC    // Enable Serial GC

// Young Gen: Serial copying collector
// Old Gen: Serial mark-sweep-compact
// All phases: single-threaded, full STW
✓ Serial GC — When to Use
  • Single-CPU machines or VMs with 1-2 cores
  • Very small heap (<100MB)
  • Client-side desktop applications
  • Microservices with tiny heaps in containers
  • When overhead of multi-thread coordination is worse than serial
  • Embedded systems with limited resources
✗ Serial GC — Avoid When
  • Multi-core server with >1GB heap
  • Latency-sensitive applications
  • High-throughput web services
  • Any production server workload
  • Pause times grow linearly with heap size

Parallel GC (Throughput Collector)

Same algorithm as Serial but uses multiple threads for Young Gen collection (parallel copying). Old Gen uses a single-threaded mark-sweep-compact. Goal: maximise throughput — spend as little total CPU time on GC as possible, even if individual pauses are long.

-XX:+UseParallelGC                  // Enable Parallel GC (default Java 8)
-XX:ParallelGCThreads=8             // number of GC threads (default: CPU count)
-XX:MaxGCPauseMillis=200            // goal: max 200ms pause (best-effort)
-XX:GCTimeRatio=99                  // goal: <1% time in GC (99:1 app:gc)
-XX:+UseAdaptiveSizePolicy          // JVM auto-adjusts heap sizes (default ON)
✓ Parallel GC — When to Use
  • Batch processing — throughput matters more than latency
  • ETL pipelines, analytics, overnight batch jobs
  • CPU-bound workloads that can tolerate longer pauses
  • Multi-core servers where concurrent GC overhead isn't worth it
  • When maximising total work done is the priority
✗ Parallel GC — Avoid When
  • Interactive applications or APIs with latency SLAs
  • Pause times of seconds are unacceptable
  • Large heaps (>4GB) — pauses scale with heap size
  • Any app where users notice GC pauses

Parallel Old GC

// Java 7+ — Parallel Old: both Young and Old Gen are parallel
-XX:+UseParallelOldGC   // parallel multi-threaded Old Gen collection
// This is the default with UseParallelGC in Java 7+
// Reduces Full GC pause significantly on multi-core machines

CMS — Concurrent Mark Sweep

The first low-latency collector — revolutionary in its time, deprecated in Java 9, removed in Java 14. Understanding it is still key to interviews.

CMS Goal

CMS aimed to minimise Old Gen pause times by doing most of the marking work concurrently — while the application continues running. Only two short STW phases required.

CMS Phases

1. Initial Mark STW — very short

Mark only the objects directly reachable from GC roots. Very fast — just one level deep.

2. Concurrent Mark Concurrent

Traverse the object graph from initially marked objects. Application runs simultaneously — this is the long phase but doesn't pause the app.

3. Concurrent Preclean Concurrent

Scan card table for objects modified by the app during concurrent mark. Reduces work needed in Remark.

4. Remark STW — short

Re-scan objects modified during concurrent mark. Must pause app for correctness. The pause is much shorter than a Full GC.

5. Concurrent Sweep Concurrent

Reclaim unmarked objects. Application runs. No compaction — free space added to free list.

6. Concurrent Reset Concurrent

Reset data structures for next CMS cycle.

CMS Fatal Flaws — Why It Was Deprecated

No Compaction
CMS never compacts — it uses free-list allocation. Over time, fragmentation grows until it can't allocate large objects, triggering a full compacting STW collection — the dreaded "concurrent mode failure" pause that can last seconds.
Concurrent Mode Failure
If Old Gen fills up while concurrent sweep is still running, CMS falls back to a Serial full STW compacting GC — often worse than if CMS had never run. Happens when allocation rate is too high or CMS wasn't started early enough.
High CPU Usage
Concurrent marking uses CPU cycles while the application runs — steals throughput. Applications with high CPU utilisation suffer under CMS.
Complex Tuning
Requires extensive tuning: CMSInitiatingOccupancyFraction, CMSParallelRemarkEnabled, CMSTriggerRatio. Getting it right for a specific workload required significant expertise.
// CMS flags (for reference — deprecated/removed in Java 14)
-XX:+UseConcMarkSweepGC
-XX:CMSInitiatingOccupancyFraction=75  // start CMS when OldGen is 75% full
-XX:+UseCMSInitiatingOccupancyOnly     // don't use adaptive triggering
-XX:+CMSParallelRemarkEnabled          // parallel remark phase
-XX:+CMSScavengeBeforeRemark           // minor GC before remark to reduce work
Interview note: CMS was removed in Java 14. If you're still using CMS, you need to migrate to G1GC or ZGC. Interviewers often ask you to compare CMS vs G1 — the key difference is G1 does compaction (in regions) which eliminates concurrent mode failures.

The default collector since Java 9 — designed for large heaps (>4GB) with predictable pause time goals.

The Core Innovation — Regions

G1 divides the heap into equal-sized regions (typically 1-32MB each). Instead of fixed Young/Old generations, any region can be designated as Eden, Survivor, Old, or Humongous. This enables incremental collection — G1 collects the regions with the most garbage first ("Garbage First").

G1 Collection Phases

Young-Only GC (Evacuation Pause) STW

Parallel collection of Eden and Survivor regions. Live objects copied to new Survivor/Old regions. Collected regions become free. Happens frequently. G1 tries to meet the pause time goal by limiting the number of regions collected per pause.

Concurrent Marking Cycle 2 short STW pauses

Triggered when heap occupancy exceeds InitiatingHeapOccupancyPercent (default 45%). Phases: Initial Mark (STW, piggybacks on Young GC) → Root Region Scan (concurrent) → Concurrent Mark (concurrent) → Remark (STW, SATB) → Cleanup (STW for accounting, concurrent for freeing empty regions).

Mixed GC STW

Collects Young regions PLUS selected Old Gen regions with most garbage. "Garbage First" — G1 picks Old regions by garbage density to maximise reclamation per pause. Multiple mixed GC rounds follow a marking cycle.

Full GC (Emergency) Full STW — avoid at all costs

Serial full heap collection triggered when G1 can't keep up with allocation rate. In Java 10+, Full GC is parallel. Indicates G1 tuning is needed — increase heap, adjust IHOP, reduce allocation rate.

Key G1 Tuning Flags

-XX:+UseG1GC                          // enable G1 (default Java 9+)

Humongous Objects — G1 Special Case

// Objects > half a region size are "humongous"
// They're allocated directly in Old Gen (contiguous regions)
// They bypass Young Gen entirely — NEVER collected by Minor GC
// Humongous objects are only collected during Concurrent Marking or Full GC
// ❌ This is a common performance problem in G1!
byte[] data = new byte[10 * 1024 * 1024]; // 10MB — humongous if region=16MB
// Fix: buffer pooling, streaming, or increase G1HeapRegionSize
✓ G1GC Strengths
  • Predictable pause times (pause goal-oriented)
  • Handles large heaps well (4GB-100GB+)
  • Concurrent marking reduces Old Gen pressure
  • Does compact (no fragmentation like CMS)
  • Adaptive — self-tuning based on pause goal
  • Best general-purpose collector for most applications
✗ G1GC Limitations
  • More CPU overhead than Parallel GC (lower throughput)
  • Humongous object handling can cause issues
  • Complex to tune (many flags)
  • Write barriers have per-write overhead
  • Not suited for ultra-low-latency (<10ms) — use ZGC
  • Full GC still single-threaded until Java 10
Java 15+ Production

ZGC — Ultra Low Latency

Sub-millisecond pause times regardless of heap size — designed for terabyte heaps and microsecond SLAs.

ZGC Goals

Pause Target
<1ms pause times (often sub-millisecond), regardless of heap size or live set size.
Heap Scale
Works efficiently from 8MB to 16TB heap. Pause time does NOT scale with heap size.
Throughput
Typically within 5-15% of throughput compared to Parallel GC — much better than CMS.

How ZGC Achieves Sub-Millisecond Pauses

1
Load Barriers (not Write Barriers)
ZGC inserts a load barrier — code that runs when any reference is loaded from memory. This allows ZGC to intercept reference reads and handle relocation on the fly, without needing STW for relocation.
2
Colored Pointers (Java 15+)
ZGC uses unused bits in 64-bit pointers to store metadata (marked0, marked1, remapped, finalizable). This lets GC track object state by checking the pointer itself — no need to access the object header.
3
Concurrent Relocation
Objects are relocated (moved) while the application runs. Load barriers ensure that when app code reads a reference to a relocated object, it gets the new address transparently — no STW needed for relocation.
4
Extremely Short STW Phases
Only 3 very short STW pauses: Pause Mark Start (scan GC roots), Pause Mark End (flush marking), Pause Relocate Start (start relocation). All typically <1ms.

ZGC Phases

Pause Mark Start STW ~0.5ms

Scan GC roots. Mark them. Very short — just the roots, not the whole graph.

Concurrent Mark Concurrent

Traverse entire object graph. Application runs. Load barrier handles any objects created during this phase.

Pause Mark End STW ~0.5ms

Flush remaining marking work. Handle weak references.

Concurrent Prepare for Relocation Concurrent

Identify which regions to relocate (relocation set). Sort by garbage density.

Pause Relocate Start STW ~0.5ms

Scan GC roots for the relocation set.

Concurrent Relocate Concurrent

Move objects. Maintain forwarding table. Load barriers redirect stale references. Application runs throughout.

✓ ZGC — When to Use
  • Ultra-low-latency: real-time trading, gaming, ad-tech
  • Very large heaps (16GB-16TB)
  • P99/P999 latency SLAs — microsecond requirements
  • Java 21+ with Generational ZGC for best results
  • When G1 pause times are too long despite tuning
✗ ZGC — Trade-offs
  • Load barrier overhead on every reference read (~5-15% throughput)
  • Needs more heap headroom than G1 (ZGC needs space to relocate)
  • 64-bit platforms only (no 32-bit support)
  • Higher memory usage for GC metadata
  • Generational ZGC only stable in Java 21+
OpenJDK / RedHat

Shenandoah GC

Red Hat's low-latency collector — concurrent compaction using Brooks pointers, available in OpenJDK 12+.

Shenandoah's Key Innovation — Brooks Pointers

Where ZGC uses colored pointers (metadata in the pointer), Shenandoah uses forwarding pointers — an extra word in each object header pointing to the object itself (or its new location during relocation). This allows concurrent compaction without STW.

Shenandoah Phases

Init Mark STW — short

Scan GC roots. Very brief.

Concurrent Mark Concurrent

Mark all live objects. Application runs.

Final Mark STW — short

Drain remaining mark queues, handle weak refs.

Concurrent Cleanup Concurrent

Reclaim completely empty regions.

Concurrent Evacuation Concurrent

Copy live objects from collection set to new regions. Brooks pointers redirect reads/writes to new location. This is Shenandoah's unique feature.

Init Update Refs STW — very short

Ensure evacuation finished before updating references.

Concurrent Update Refs Concurrent

Update all references to point to new object locations.

Final Update Refs STW — very short

Update GC root references. Release old regions.

-XX:+UseShenandoahGC                     // enable Shenandoah
-XX:ShenandoahGCMode=normal              // normal, compact, passive
-XX:ShenandoahAllocationThreshold=10    // trigger GC at 10% free
-XX:ShenandoahInitFreeThreshold=70      // initial free threshold

Minor, Major & Full GC

The three types of GC events — causes, costs, and how to minimise the expensive ones.

Minor GC

Minor GC — Young Generation Collection

Trigger: Eden space is full. Scope: Young Gen only (Eden + Survivor). Cost: 1-100ms, proportional to live objects in Young Gen (usually very few). Frequency: Every few seconds under load. Most objects die here — this is normal and efficient.

// GC log — Minor GC (Java 9+ unified GC logging)
[0.234s][info][gc] GC(1) Pause Young (Normal) (G1 Evacuation Pause)
                     256M->24M(512M) 8.234ms
// Heap: 256MB → 24MB (heap capacity 512MB), pause: 8.2ms
// 232MB of Young Gen garbage collected in 8ms — excellent

Major GC

Major GC — Old Generation Collection

Trigger: Old Gen reaches occupancy threshold (IHOP in G1, CMSInitiatingOccupancyFraction in CMS). Scope: Old Gen (may include Young Gen). Cost: 100ms-several seconds. Frequency: Should be rare — minutes to hours. Frequent Major GCs indicate a problem.

Full GC — The Worst Case

Full GC — Entire Heap Full STW — avoid at all costs

Trigger: Heap exhausted, concurrent GC can't keep up, G1 evacuation failure, explicit System.gc(). Scope: Entire heap — Young + Old + Metaspace. Cost: Seconds to minutes. Impact: Application completely frozen during this time.

Stop-The-World Pauses

What happens during an STW pause, why it's necessary, and how to measure and minimise it.

What Happens During an STW Pause

1
Safepoint request
JVM sets a flag requesting all threads to reach a safepoint — a point in execution where the thread's state is fully known (no partially completed bytecode, no holding partial references).
2
Threads park
Each thread checks for the safepoint flag at regular intervals (back edges of loops, method calls). They suspend themselves and signal "I'm at a safepoint." JVM waits for ALL threads to park.
3
GC work runs
With all application threads stopped, GC runs its STW phase(s) safely. The heap is in a consistent, non-changing state — GC can read and modify it safely.
4
Threads resume
JVM signals all threads to resume. They continue execution from where they paused. Application resumes as if nothing happened (but time passed).

Safepoint Bias — The Hidden Latency

Reported GC pause times are measured from when all threads reach safepoint — but Time To Safepoint (TTSP) — the time for all threads to reach safepoint — is often not included. A thread executing a counted loop may not check for safepoints until the loop completes, causing 50-100ms TTSP even for a 5ms GC.

// Diagnose safepoint issues

-XX:PrintSafepointStatisticsCount=1     // every safepoint
-Xlog:safepoint=info                    // Java 9+ unified logging

// ❌ This loop won't check safepoints in old JVMs (int loop, JIT-optimised)

    process(data[i]);
}
// ✅ Use long loop — long loops always have safepoint checks

    process(data[(int)i]);
}

SoftReference — Memory-Sensitive Cache

public class ImageCache {
    private final Map<String, SoftReference<BufferedImage>> cache = new HashMap<>();

    public BufferedImage get(String key) {
        SoftReference<BufferedImage> ref = cache.get(key);
        if (ref == null) return null;
        BufferedImage img = ref.get(); // null if GC cleared it
        if (img == null) {
            cache.remove(key); // clean up stale entry
        }
        return img;
    }

    public void put(String key, BufferedImage img) {
        cache.put(key, new SoftReference<>(img));
    }
    // JVM will clear images before throwing OOM
    // Cache shrinks automatically under memory pressure
}

WeakReference — WeakHashMap Pattern

// WeakHashMap — keys are weakly referenced
// When a key is no longer strongly referenced elsewhere, entry is removed
WeakHashMap<Widget, WidgetMetadata> metadata = new WeakHashMap<>();

// Real-world use: attaching metadata to objects you don't own
// without preventing their GC
Widget w = new Widget();
metadata.put(w, new WidgetMetadata());
// When 'w' is no longer referenced anywhere (strong ref),
// the WeakHashMap entry is automatically removed — no memory leak!

// Manual WeakReference
WeakReference<User> weakUser = new WeakReference<>(user);
User u = weakUser.get(); // null if GC already collected it
if (u != null) {
    process(u); // safe — we checked it's still alive

PhantomReference & Cleaner — Post-GC Cleanup

// Modern way: java.lang.ref.Cleaner (Java 9+)
// Register cleanup action to run after object is GC'd
public class NativeBuffer implements AutoCloseable {
    private static final Cleaner CLEANER = Cleaner.create();

    private final long nativePtr;
    private final Cleaner.Cleanable cleanable;

    public NativeBuffer(int size) {
        this.nativePtr = allocateNative(size);
        // Register cleanup — runs when this object is GC'd
        // Uses a separate Runnable to avoid capturing 'this' (would prevent GC)
        this.cleanable = CLEANER.register(this, new Cleanup(nativePtr));
    }

    private record Cleanup(long ptr) implements Runnable {
        public void run() { freeNative(ptr); } // called after GC
    }

    @Override
    public void close() { cleanable.clean(); } // explicit cleanup if possible
}
// Cleaner is the recommended replacement for finalize()

Finalization & Cleaners

finalize() is deprecated — why it's broken and what to use instead.

finalize() — What It Was

// finalize() runs before an object is GC'd — in theory
public class LegacyResource {
    @Override
    @Deprecated // deprecated since Java 9, scheduled for removal
    protected void finalize() throws Throwable {
        try {
            releaseNativeResource();
        } finally {
            super.finalize();
        }
    }
}

Why finalize() is Broken

Unpredictable Timing
No guarantee WHEN finalize() runs. Could be seconds, minutes, or never if the JVM exits. Can't rely on it for timely resource release.
Resurrection Risk
finalize() can make an object reachable again by storing this somewhere. JVM must check this — objects with finalize() require two GC cycles to collect.
Performance Penalty
Objects with finalizers are moved to a special finalization queue. They survive their first GC — always promoted to Old Gen. Massive GC overhead.
Exception Swallowing
Exceptions thrown from finalize() are silently ignored. Bugs in finalize() vanish without a trace.

The Right Way — Cleaner or AutoCloseable

// Best practice: AutoCloseable + try-with-resources (explicit, deterministic)
public class DatabaseConnection implements AutoCloseable {
    private Connection conn;

    public DatabaseConnection() { conn = openConnection(); }

    @Override
    public void close() {
        if (conn != null) {
            conn.close();
            conn = null;
        }
    }
}
// Usage: deterministic, timely resource release
try (var db = new DatabaseConnection()) {
    db.query("SELECT 1");
} // close() called immediately — no GC needed

Memory Leaks — Detection & Prevention

Java can have memory leaks despite GC — objects that are technically reachable but should have been released.

Java Memory Leak Definition

In Java, a memory leak is when objects are logically unused but remain strongly reachable — GC cannot collect them because something still holds a reference. The objects are "alive" to GC but "dead" to the application's logic.

The 6 Most Common Memory Leaks

1. Unbounded Static Collections

// ❌ Classic leak — static collection grows forever
public class RequestTracker {
    private static final List<Request> ALL_REQUESTS = new ArrayList<>();

    public static void track(Request r) {
        ALL_REQUESTS.add(r); // Added forever, never removed
    }
}
// Every request adds to the list. Static = GC Root. Never collected.
// After 24h of traffic: GBs of old request objects in memory.

// ✅ Fix: use size-bounded collection or evict old entries
private static final Deque<Request> RECENT = new ArrayDeque<>();
public static synchronized void track(Request r) {
    RECENT.addLast(r);
    if (RECENT.size() > 1000) RECENT.removeFirst(); // bounded
}

2. Event Listener Accumulation

// ❌ Listener registered but never removed
public class UserSessionView {
    public UserSessionView(EventBus bus) {
        bus.register(this); // EventBus holds strong ref to 'this'
    }
    // When the view is "closed", it's removed from UI
    // But EventBus still holds the reference — UserSessionView never GC'd
    // All its fields (user data, session) remain in memory
}

// ✅ Fix: always deregister
public void destroy() {
    bus.unregister(this); // remove before discarding
}

3. ThreadLocal Without Remove

// ❌ In a thread pool — threads are reused, ThreadLocals accumulate
public class RequestHandler {
    static ThreadLocal<LargeContext> ctx = new ThreadLocal<>();

    public void handle(Request req) {
        ctx.set(new LargeContext(req));
        process();
        // ❌ No ctx.remove() — LargeContext lives in thread pool thread forever
    }
}
// ✅ Fix:
public void handle(Request req) {
    ctx.set(new LargeContext(req));
    try { process(); } finally { ctx.remove(); } // ALWAYS remove
}

4. Inner Class Holding Outer Reference

// ❌ Non-static inner class implicitly holds reference to outer class
public class DataProcessor {
    private byte[] largeBuffer = new byte[100 * 1024 * 1024]; // 100MB

    class Task implements Runnable { // non-static inner class
        public void run() { /* only uses a few fields */ }
    }
}
// If Task is submitted to a long-lived executor, it holds a reference to
// DataProcessor, which holds 100MB buffer — even if DataProcessor is "done"

// ✅ Fix: use static nested class (no implicit outer reference)
static class Task implements Runnable {
    private final String data; // only capture what you need
    Task(String d) { this.data = d; }
}

How to Find Memory Leaks

1
Monitor heap over time
Use JVM metrics (Micrometer, JMX) to plot heap after each Full GC. If heap-after-GC grows consistently over hours/days — you have a leak.
2
Take heap dump
jmap -dump:format=b,file=heap.hprof <pid> or configure -XX:+HeapDumpOnOutOfMemoryError. Take two dumps 30 minutes apart.
3
Analyse with MAT or VisualVM
Eclipse MAT (Memory Analyser Tool): open heap dump → "Leak Suspects" report → shows largest object trees and what's holding them alive.
4
Compare two dumps
Look at what grew between the two dumps. Objects that are increasing but shouldn't be are your leak. Follow the reference chain to the GC root that's holding them.

for (Request r : requests) {

// ✅ Pool direct buffers (Netty PooledByteBufAllocator does this)
private final BlockingDeque<ByteBuffer> pool = new LinkedBlockingDeque<>();

ByteBuffer acquire() {

    return buf != null ? buf.clear() : ByteBuffer.allocateDirect(64*1024);
}
void release(ByteBuffer buf) { pool.offer(buf); }

GC Anti-Patterns

Common mistakes that hurt GC performance — what not to do and why.

1. Calling System.gc()

// ❌ NEVER do this in production code
System.gc(); // Hint to JVM to run GC — but causes Full GC on most JVMs
// Common in code that "wants to be helpful" — actually causes:
// 1. Full STW pause (seconds)
// 2. Disrupts GC's own adaptive scheduling
// 3. RMI/Java serialization calls this every 60s by default!

// Fix: disable it entirely
-XX:+DisableExplicitGC              // treat System.gc() as no-op
-Dsun.rmi.dgc.client.gcInterval=3600000 // RMI: GC every hour instead of 60s

2. Excessive Object Creation (Allocation Pressure)

// ❌ Creating objects inside hot loops
for (String line : lines) {
    String[] parts = line.split(",");         // allocates String array + Strings
    BigDecimal value = new BigDecimal(parts[2]); // allocates BigDecimal
    DateTimeFormatter fmt = DateTimeFormatter.ofPattern("yyyy-MM-dd"); // !!!
    // DateTimeFormatter creation is expensive AND creates garbage per iteration
}

// ✅ Hoist allocations out of loops
private static final DateTimeFormatter FMT = DateTimeFormatter.ofPattern("yyyy-MM-dd");
for (String line : lines) {
    int idx1 = line.indexOf(',');            // avoid split() allocation
    int idx2 = line.indexOf(',', idx1+1);
    LocalDate date = LocalDate.parse(line, FMT);
}

3. Finalizer-Dependent Code

// ❌ Relying on finalize() for resource cleanup
class Connection {
    protected void finalize() { socket.close(); } // NEVER rely on this
}
// Finalizers run in an unpredictable thread, may never run before OOM,
// and keep objects alive an extra GC cycle. Recipe for connection exhaustion.

// ✅ AutoCloseable + try-with-resources or Cleaner

4. Keeping References Longer Than Needed

// ❌ Holding reference in stack variable unnecessarily
public void process() {
    byte[] largeData = loadLargeFile(); // 500MB
    String result = extract(largeData);
    // largeData still referenced until end of method!
    doOtherWork(result); // this allocates lots of memory — OOM risk
}

// ✅ Null out early to allow GC
public void process() {
    String result;
    {
        byte[] largeData = loadLargeFile();
        result = extract(largeData);
    } // largeData out of scope here — eligible for GC during doOtherWork
    doOtherWork(result);
}

5. Mismatched Heap Size for GC Type

// ❌ Using Serial GC on a 32GB heap
-XX:+UseSerialGC -Xmx32g    // Full GC will take MINUTES — single-threaded

// ❌ Using Parallel GC for a latency-sensitive API
-XX:+UseParallelGC -Xmx8g  // Pauses will be hundreds of ms — users notice

// ✅ Match collector to heap size and latency requirements
-XX:+UseG1GC -Xmx8g -XX:MaxGCPauseMillis=200    // API servers
-XX:+UseZGC -Xmx64g                             // huge heap, latency-critical
-XX:+UseParallelGC -Xmx4g                       // batch jobs

Container-specific GC considerations — memory limits, CPU throttling, and tuning for small heaps.

Container Memory Awareness

// Java 8u191+ / Java 10+: JVM is container-aware by default
// UseContainerSupport is ON — JVM reads cgroup limits

// Docker/K8s: set memory request and limit
# kubernetes deployment.yaml
resources:
  requests:
    memory: "1Gi"
  limits:
    memory: "2Gi"    # JVM sees 2GB as available memory

// JVM flags for containers
-XX:MaxRAMPercentage=75.0    // use 75% of container memory for heap
-XX:InitialRAMPercentage=50.0 // start at 50%
-XX:MinRAMPercentage=25.0    // minimum for very small containers
// Result in 2GB container: ~1.5GB heap, ~500MB for stack, Metaspace, JIT code

CPU Throttling — GC Thread Impact

// Problem: G1GC defaults concurrent threads to CPU_COUNT / 4
// In a container with 2 CPU limit, G1 gets 0.5 concurrent threads → 1 thread
// But if the node has 32 CPUs, JVM might think it has 32 and spawn 8 GC threads
// Then CPU throttling kills performance

-XX:ActiveProcessorCount=4          // explicitly set CPU count for GC thread calc
-XX:ParallelGCThreads=4             // explicit STW GC threads
-XX:ConcGCThreads=2                 // explicit concurrent GC threads

// Java 11+: JVM respects container CPU quotas (UseContainerSupport)
// Still good to set explicitly for predictability

GC Tuning for Small Containers (256MB-512MB heap)

// Small heaps = frequent GC = different tuning needed

// Option 1: Serial GC — surprisingly good for very small heaps
-XX:+UseSerialGC -Xmx256m
// Single-threaded GC overhead is lower than G1's region tracking
// Works well for small stateless microservices

// Option 2: G1GC with small region size
-XX:+UseG1GC -Xmx256m -XX:G1HeapRegionSize=1m
// Small regions for small heap — avoid huge humongous region overhead

// Option 3: ZGC (Java 21 generational ZGC is much better for small heaps)
-XX:+UseZGC -XX:+ZGenerational -Xmx512m

// GraalVM Native Image — ultimate for containers
// No JVM warmup, no GC pause variance, tiny memory footprint
// Trade-off: no JIT optimisation after startup

Startup Time Optimisation

// JVM JIT compilation causes GC pressure at startup
// Tier 1-3 JIT compilers run, creating code objects, discarding intermediates

-XX:+TieredCompilation            // enabled by default — keep it
-XX:ReservedCodeCacheSize=256m    // larger code cache prevents deopt under load
-XX:+UseStringDeduplication       // G1: deduplicate identical String contents
// String dedup can reduce heap by 10-30% in typical web apps
// Background thread identifies Strings with same content, keeps one char[]

// AppCDS — precompile to shared archive, faster startup + smaller heap
java -XX:DumpLoadedClassList=classes.lst -cp app.jar Main    // step 1
java -XX:SharedArchiveFile=app.jsa -XX:SharedClassListFile=classes.lst ...  // step 2
java -XX:SharedArchiveFile=app.jsa -cp app.jar Main          // use archive

Beginner

What is garbage collection and why does Java need it? ▼
GC is automatic memory management — the JVM tracks which objects are reachable from the running program and reclaims memory for objects that are no longer reachable. Java needs it because manually managing memory (as in C/C++) leads to memory leaks (forgetting to free) and crashes (double-free, use-after-free). GC eliminates this entire class of bugs, making Java safer and more productive.
What is the difference between heap and stack memory? ▼
Stack: per-thread memory holding method call frames, local variables (primitives and references), return addresses. Automatically freed when the method returns. Fast allocation (pointer decrement). Fixed max size (configurable with -Xss).

Heap: shared memory where all objects live. GC-managed. Dynamic size. Objects survive method returns as long as references to them exist. Where garbage collection happens. Configured with -Xms/-Xmx.
Can you force garbage collection in Java? ▼
System.gc() and Runtime.getRuntime().gc() are hints — the JVM may ignore them. On HotSpot JVM, they typically do trigger a Full GC, which is undesirable in production. Best practice is to never call them in application code. Use -XX:+DisableExplicitGC to make them true no-ops.
What causes an OutOfMemoryError? ▼
OOM occurs when JVM cannot allocate an object because there's not enough memory and GC cannot free enough. Variants: Java heap space (heap full — GC overhead limit exceeded or memory leak), Metaspace (too many classes loaded, ClassLoader leak), GC overhead limit exceeded (GC spending >98% of time, freeing <2%), Direct buffer memory (off-heap direct buffers exhausted), Unable to create native thread (OS-level thread limit).
What is the Young Generation and why does it exist? ▼
Young Generation holds newly created objects. It exists because of the Weak Generational Hypothesis — empirically, 80-98% of objects die within milliseconds of creation. By keeping young objects in a small, frequently-collected space, GC can reclaim most garbage quickly (Minor GC collects only Young Gen — fast, cheap, frequent). Only the small fraction of long-lived objects ever reaches Old Gen.

Intermediate

What is Stop-The-World and why is it necessary? ▼
Stop-The-World (STW) pauses all application threads to allow GC to work on a consistent, non-changing heap. It's necessary because: concurrent object graph traversal is unsafe — if the application modifies references while GC is marking, GC might miss live objects (causing data corruption) or mark dead objects as live. STW ensures heap consistency. Modern GCs minimise STW using concurrent algorithms — only very short phases (root scanning) require STW.
What is the difference between Minor GC, Major GC, and Full GC? ▼
Minor GC: collects Young Generation only. Triggered when Eden is full. Fast (1-100ms), frequent. Normal and expected.
Major GC: collects Old Generation. Triggered when Old Gen reaches occupancy threshold. Slower (100ms-seconds). Should be infrequent.
Full GC: collects entire heap — Young + Old + Metaspace. Usually serial. Can pause for seconds. Indicates a problem (memory leak, under-tuned heap, G1 falling back). Avoid in production.
What happens to an object during a Minor GC? ▼
Live objects in Eden are copied to the empty Survivor space (say S0). Dead objects in Eden are simply abandoned — no explicit deallocation, Eden is cleared atomically. Existing survivors in S1 with age <threshold are copied to S0 with age incremented. Objects reaching tenuring threshold are promoted to Old Gen. S0 and S1 roles swap. The entire Eden space becomes free for new allocations.
What is card table and write barrier? ▼
The card table is a bitmap where each bit represents a 512-byte "card" of Old Gen heap. A write barrier is code inserted by the JVM compiler before every reference write — when an Old Gen object's field is updated to point to a Young Gen object, the barrier marks that card as "dirty." During Minor GC, GC only scans dirty Old Gen cards (not the entire Old Gen) to find cross-generational references. This makes Minor GC efficient even with Old Gen references to Young Gen objects.
What is the difference between SoftReference, WeakReference, and PhantomReference? ▼
SoftReference: cleared by GC only when JVM is low on memory (before throwing OOM). Use for memory-sensitive caches — cache entries are kept if memory is available, cleared if needed.
WeakReference: cleared on any GC cycle when no strong references exist. Use for canonicalising maps (WeakHashMap), listener registries, object metadata you don't want to prevent GC.
PhantomReference: always returns null from get(). Enqueued AFTER finalization. Use for post-GC cleanup actions (modern replacement: Cleaner API).
Why is finalize() deprecated and what should you use instead? ▼
finalize() is broken because: unpredictable timing (may never run before OOM), objects with finalizers require two GC cycles to collect (survive first GC), exceptions are silently swallowed, can resurrect objects, and runs in an unpredictable thread. Use instead: AutoCloseable + try-with-resources for explicit deterministic cleanup, or java.lang.ref.Cleaner (Java 9+) for safety-net cleanup that runs after GC (without finalize()'s problems).

Advanced

How does G1GC achieve predictable pause times? ▼
G1 achieves predictable pauses through region-based collection. It divides the heap into equal regions and tracks the amount of garbage in each region. Given a pause time goal (MaxGCPauseMillis), G1 selects only as many regions as can be collected within that time budget. It prioritises regions with the most garbage ("Garbage First"). Concurrent marking runs alongside the application to identify garbage density per region. This enables incremental collection — G1 never has to collect the whole heap in one pause.
What is a humongous object in G1GC and why is it a problem? ▼
A humongous object is any object larger than half a G1 region size. It's allocated directly into Old Generation (spanning one or more contiguous regions), bypassing Young Gen entirely. Problems: (1) only collected during concurrent marking cycles or Full GC — not during regular Young GC; (2) contiguous regions required — can cause premature Full GC due to fragmentation; (3) can trigger humongous allocation failure. Fix: increase G1HeapRegionSize (up to 32MB), use buffer pools for large byte arrays, or switch to streaming I/O to avoid large allocations.
How does ZGC achieve sub-millisecond pause times? ▼
ZGC uses three techniques: (1) Load barriers — code injected at every reference read that can handle relocation on the fly, allowing concurrent object moving; (2) Colored pointers — metadata bits in the 64-bit pointer indicate object state (marked, remapped, finalizable), enabling GC state tracking without accessing the object; (3) Concurrent relocation — objects are moved while the application runs; load barriers redirect reads/writes to the new location transparently. Only 3 very short STW phases exist (scanning GC roots), each typically <1ms regardless of heap size.
What is a safepoint and what is safepoint bias? ▼
A safepoint is a point in thread execution where the thread's state is fully known (no partial operations). The JVM requests all threads to reach a safepoint before STW GC begins. Threads check for safepoint requests at back edges of loops and method calls.

Safepoint bias: GC pause reporting measures from when all threads reach safepoint, not from when GC was requested. A thread stuck in a long counted loop may not check for safepoints until the loop completes — causing 50-200ms "Time To Safepoint" that isn't counted in the GC pause time but still affects application latency.
What is tri-color marking and why does it matter for concurrent GC? ▼
Tri-color marking assigns each object one of three states: White (unvisited — possibly garbage), Grey (visited but references not yet processed), Black (fully processed — object and all references scanned). GC marks grey objects until none remain; white objects are garbage. In concurrent GC, the application can modify references while marking — this could cause a black object to reference a white object (a reference the mutator just created). Write barriers (G1 uses SATB) detect these changes and ensure new references are also marked, maintaining correctness.

Senior / Expert

How do you diagnose and fix a GC-related performance issue in production? ▼
Systematic approach: (1) Identify symptoms: latency spikes (correlate with GC pauses in logs), high CPU usage, memory growing. (2) Collect GC logs: enable -Xlog:gc*:file=gc.log:time,uptime. (3) Analyse logs with GCEasy or GCViewer — identify: pause frequency/duration, Full GC occurrences, heap growth after GC. (4) Heap profiling: jmap histogram, heap dump with MAT for leak detection. (5) JFR: allocation hot spots, allocation rate. (6) Tune based on findings: memory leak → fix code, short-lived objects in Old Gen → increase Young Gen / reduce tenuring threshold, humongous objects → increase region size / pooling, Full GCs → increase IHOP, heap size.
Explain the GC overhead limit exceeded error and how to handle it. ▼
GC overhead limit exceeded is thrown when GC is spending >98% of CPU time but recovering <2% of heap — the JVM determines the application can't make progress. Root causes: memory leak (heap never gets free enough), undersized heap for the workload, massive allocation rate. Temporary fix: -XX:-UseGCOverheadLimit (NOT recommended — just delays OOM). Real fix: (1) take heap dump and analyse for leaks; (2) increase heap size; (3) reduce allocation rate; (4) profile and fix allocation hot spots with JFR/async-profiler.
How does string interning interact with GC? ▼
Interned strings are stored in the JVM's string pool — in Java 7+, this is in the heap (previously PermGen). The pool holds WeakReferences to strings in Java 8+, so interned strings ARE GC-eligible when no other references exist (unlike old PermGen which was never GC'd). However, String.intern() is expensive (hash table lookup/insert) and the pool can grow large. G1's String Deduplication (-XX:+UseStringDeduplication) is different — it doesn't use the pool but finds Strings with identical chars and makes them share the same char array, reducing memory without affecting semantics.
What is NUMA-aware GC and when does it matter? ▼
Non-Uniform Memory Access (NUMA) systems have multiple memory banks, each with a CPU socket. Accessing local memory (same NUMA node) is faster than remote. Without NUMA awareness, GC might allocate objects on a remote node, degrading performance.

-XX:+UseNUMA enables NUMA-aware allocation: each thread allocates from its local NUMA node's memory. Young Gen is partitioned across NUMA nodes. Improves GC performance significantly on NUMA systems (typically large multi-socket servers). Check with numactl --hardware if relevant.
How do you tune GC for a container environment with limited CPU and memory? ▼
Key considerations: (1) UseContainerSupport must be enabled (default Java 11+) — JVM reads cgroup memory limits instead of host memory; (2) Use -XX:MaxRAMPercentage=75 instead of fixed -Xmx for portability; (3) Set -XX:ActiveProcessorCount equal to CPU limit to prevent overprovisioning GC threads; (4) For small heaps (<512MB): consider SerialGC (lower overhead) or G1 with 1MB regions; (5) Generational ZGC (Java 21) is increasingly viable even for small containers; (6) Set -XX:+ExitOnOutOfMemoryError so k8s can restart the pod immediately instead of limping along; (7) Configure liveness probes to detect GC-induced hangs.
What is Epsilon GC and when would you use it? ▼
Epsilon GC is a "no-op" garbage collector — it allocates memory but never frees any. When the heap is exhausted, the JVM exits with OOM. Use cases: (1) Performance benchmarking — eliminate GC variance from benchmark results to see true algorithm performance; (2) Short-lived jobs — batch jobs that finish quickly before exhausting heap, avoiding any GC overhead; (3) Memory usage analysis — allocate until OOM to see exact memory consumption; (4) GraalVM native image testing. Enable with -XX:+UseEpsilonGC. Never use in production services!
Describe G1GC concurrent mode failure and evacuation failure — what causes them and how to fix? ▼
Evacuation failure (G1-specific): during a Young GC, live objects can't be copied to Survivor/Old regions because there isn't enough free space. G1 falls back to a Full GC (serial in older Java, parallel from Java 10). Causes: heap too small, IHOP too high (marking starts too late), high allocation rate, humongous object avalanche. Fix: increase heap, lower IHOP (start marking earlier), increase region size, fix humongous allocations.

Concurrent mode failure (CMS-specific, deprecated): Old Gen fills up during concurrent sweep. CMS triggers a serial compacting Full GC. Causes: allocation rate exceeds concurrent sweep speed, CMSInitiatingOccupancyFraction too high. Fix: lower CMSInitiatingOccupancyFraction (or migrate to G1/ZGC).
