Section 1 of 5 · ~6 mins

The OOP mental model: objects, state, behavior, and responsibility

Before the four famous principles, you need a stable picture of what OOP is trying to optimize: modeling a system as collaborating objects with clear responsibilities instead of one giant pile of procedures and shared data.

By the end of this section

  • You’ll separate class, object, and instance state cleanly.
  • You’ll see why OOP is really about responsibility boundaries.
  • You’ll know why “just putting data in classes” is not enough.

🧒 ELI5 · A busy restaurant

Imagine a restaurant. The cashier takes orders, the kitchen cooks, and the delivery person takes food out. Each person has their own job and their own little area to manage. Nobody needs to know every tiny detail of how the others work.

That’s the basic idea of OOP. Instead of one giant person doing everything, you split the work across “objects.” Each object keeps track of its own stuff and knows how to do its own job.

🔗 ANALOGY · A team, not a spreadsheet

A spreadsheet model says: “Here is all the data; now go write formulas over it.” An OOP model says: “Here are actors with responsibilities.”

Customer places order → Cashier validates → Kitchen prepares → Driver delivers

Good OOP feels like routing work to the right role, not like dumping everything into a util class.

Technical card · What OOP is really buying you

  • Objects bundle state and behavior together.
  • Classes are blueprints; objects are runtime instances created from those blueprints.
  • Good object models reduce coupling and increase cohesion: each object owns related data and logic.
  • In mature codebases, OOP is less about “using classes” and more about assigning responsibilities well.
class BankAccount {
    private String accountNumber;
    private double balance;

    public BankAccount(String accountNumber, double openingBalance) {
        this.accountNumber = accountNumber;
        this.balance = openingBalance;
    }

    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("amount must be positive");
        balance += amount;
    }

    public double currentBalance() {
        return balance;
    }
}

Notice the point: the account doesn’t just store data. It also owns the rules around that data.

Concept Meaning Why it matters Quick read
Class Blueprint for creating objects Defines structure and behavior Design-time
Object Runtime instance with identity + state Actually receives messages and does work Runtime
Responsibility What this object should know or do Prevents God classes and scattered logic Core idea

Warning card · OOP can be faked

Code is not “object-oriented” just because it uses classes. If your classes are only bags of getters/setters and all real logic lives elsewhere, you often end up with procedural code wearing OOP clothing.

Section 2 of 5 · ~6 mins

Encapsulation: protect invariants, not just fields

Encapsulation is often taught as “make fields private.” That is true, but incomplete. The real value is that an object controls how its own state can change, so invalid states become harder to create.

Key ideas here

  • Encapsulation protects invariants.
  • It’s about exposing safe behavior, not hiding everything.
  • Good encapsulation makes change easier because callers depend on stable operations, not internals.

🧒 ELI5 · A medicine bottle with a child lock

You can still use the medicine bottle, but you cannot randomly mess with what is inside. There’s a safe way to open it and a safe way to use it.

Encapsulation is similar. The object says, “You can ask me to do certain things, but you don’t get to reach in and change my insides however you want.”

🔗 ANALOGY · A vending machine interface

You press buttons and insert money. You do not grab the internal motor or manually spin the shelf. The machine gives you a small public surface and hides the internal moving parts.

Insert money → Select item → Machine validates → Dispense or reject

Technical card · Encapsulation in real Java code

  • Use private fields to prevent arbitrary mutation.
  • Expose behavioral methods such as deposit(), changeAddress(), approve() instead of raw setters when business rules matter.
  • Preserve invariants: rules that must always stay true, such as “balance cannot go negative” or “email must be valid.”
  • Encapsulation reduces breakage because callers rely on what the object does, not how it stores data.
class ShoppingCart {
    private final List<Item> items = new ArrayList<>();
    private double total;

    public void addItem(Item item) {
        if (item == null) throw new IllegalArgumentException("item required");
        items.add(item);
        total += item.price();
    }

    public void removeItem(Item item) {
        if (items.remove(item)) {
            total -= item.price();
        }
    }

    public List<Item> itemsView() {
        return List.copyOf(items);
    }

    public double total() {
        return total;
    }
}

Returning List.copyOf(items) is part of encapsulation too. It stops callers from mutating internal state behind your back.

Approach What caller can do Risk Read
Public mutable field Anything State corruption, no validation High risk
Private field + raw setter Change state through a method Still weak if setter skips rules Medium
Private field + behavior methods Request safe operations Object can enforce invariants Strong

Warning card · Getters/setters are not automatic good design

A class with 20 private fields and 40 getters/setters might be technically “encapsulated,” but it can still leak its internal model everywhere. The question is: does the object control its rules, or do callers control them?

Section 3 of 5 · ~6 mins

Abstraction: show the contract, hide the machinery

Abstraction reduces the amount a caller must understand. Instead of forcing everyone to know internal details, you expose a useful contract: what operations exist and what guarantees they provide.

Focus points

  • Abstraction lets code depend on what, not how.
  • Interfaces and abstract classes are tools for expressing contracts.
  • It improves extensibility because implementations can vary behind the same API.

🧒 ELI5 · Driving a car without knowing the engine

You know the steering wheel turns the car, the brake stops it, and the accelerator makes it go. You do not need to know how fuel injection works before driving to the store.

That is abstraction. It gives you a useful set of controls without forcing you to study the hidden parts first.

🔗 ANALOGY · A remote control, many devices

The button says Power. One TV might use infrared, another Bluetooth, another Wi‑Fi. The user presses one concept; the implementation behind it varies.

Client calls play() → Contract decides allowed action → Concrete implementation runs

Technical card · Interfaces, abstract classes, and API design

  • Abstraction means emphasizing essential behavior while hiding irrelevant detail.
  • In Java, common tools are interfaces and abstract classes.
  • Code that depends on abstractions is typically easier to test, extend, and replace.
  • This is why dependency injection often targets interfaces: callers care about a capability, not a concrete class.
interface PaymentProcessor {
    Receipt charge(Money amount);
}

class StripePaymentProcessor implements PaymentProcessor {
    @Override
    public Receipt charge(Money amount) {
        // call Stripe API, map response, return receipt
        return new Receipt("stripe-123");
    }
}

class CheckoutService {
    private final PaymentProcessor paymentProcessor;

    public CheckoutService(PaymentProcessor paymentProcessor) {
        this.paymentProcessor = paymentProcessor;
    }

    public Receipt checkout(Money total) {
        return paymentProcessor.charge(total);
    }
}

CheckoutService does not care whether the implementation is Stripe, PayPal, or a fake test double. That is the payoff of abstraction.

Tool Best for Can carry state? Read
Interface Pure capability / contract Not instance state Flexible
Abstract class Shared base behavior + partial template Yes Use with care
Concrete class dependency Simple direct usage Yes More coupled

Warning card · Don’t abstract too early

Abstraction is powerful, but premature abstraction creates extra layers with no real variability behind them. If you only have one stable implementation and no likely change axis, a concrete class may be enough for now.

Section 4 of 5 · ~6 mins

Inheritance: reuse with hierarchy — and with sharp edges

Inheritance lets one type derive from another. It can model real subtype relationships and share behavior, but it also creates tight coupling and can age badly when the hierarchy no longer matches the domain.

You should leave with

  • A clean sense of is-a versus has-a.
  • An intuition for when inheritance helps and when it traps you.
  • A practical bias toward modeling behavior, not just hierarchy.

🧒 ELI5 · Family traits

A child may inherit some traits from a parent, like eye color, but the child is still its own person. In code, a child class can reuse some behavior from a parent class and also add or change some of its own behavior.

🔗 ANALOGY · Tools in the same family

Imagine a base tool called Tool with common things like weight and brand. A Hammer and a Screwdriver are both tools, but they behave differently. That makes sense as a hierarchy because the “is-a” relationship is natural.

Base type defines common contract → Subtype inherits → Subtype extends or overrides

Technical card · The real tradeoff in inheritance

  • Inheritance expresses an is-a relationship: a Dog is an Animal.
  • It enables reuse of fields and methods from a superclass.
  • Overriding lets subclasses provide specialized behavior.
  • But inheritance couples subclass and superclass tightly. Changes high in the tree can ripple downward.
abstract class Notification {
    protected final String recipient;

    protected Notification(String recipient) {
        this.recipient = recipient;
    }

    public abstract void send();
}

class EmailNotification extends Notification {
    public EmailNotification(String recipient) {
        super(recipient);
    }

    @Override
    public void send() {
        System.out.println("Sending email to " + recipient);
    }
}

class SmsNotification extends Notification {
    public SmsNotification(String recipient) {
        super(recipient);
    }

    @Override
    public void send() {
        System.out.println("Sending SMS to " + recipient);
    }
}
Question If answer is yes… If answer is no…
Is the subtype truly an “is-a” of the parent? Inheritance may fit Avoid hierarchy
Do subclasses need the same core contract? Shared base can help Prefer composition
Will parent changes likely break children? Coupling risk More stable

Warning card · The “is-a” test matters

A classic mistake is inheriting for code reuse when the relationship is not a true subtype. A Stack is not really a general List just because both contain items. Reuse alone is not enough reason to inherit.

Section 5 of 5 · ~6 mins

Polymorphism: same message, different behavior — plus when to favor composition

Polymorphism lets you write code against a common type while different concrete objects respond in their own way. This is where abstraction and inheritance become useful in practice. It is also why composition often outlives inheritance in real systems.

What you’ll lock in

  • How dynamic dispatch works at a practical level.
  • Why polymorphism removes branching pressure from callers.
  • Why composition over inheritance is a healthy default in many codebases.

🧒 ELI5 · Same button, different toy

You press the same “start” button on two different toys. One spins. One lights up. The button is the same idea, but each toy knows how to respond in its own way.

That is polymorphism: same request, different object-specific behavior.

🔗 ANALOGY · A music playlist with many speaker types

You call play() on a phone speaker, a Bluetooth speaker, and a car stereo. Your code says one thing. The actual output path differs per device.

Caller sees Speaker → Calls play() → Runtime picks concrete implementation

Technical card · Dynamic dispatch and composition

  • Polymorphism allows one interface/reference type to point to many concrete implementations.
  • With runtime dispatch, the method that actually runs depends on the object’s concrete type, not the reference variable’s type.
  • This helps remove repetitive if/else or switch logic from callers.
  • Composition means building behavior by assembling collaborating objects, which is often more flexible than deep inheritance trees.
interface Notifier {
    void notify(String message);
}

class EmailNotifier implements Notifier {
    @Override
    public void notify(String message) {
        System.out.println("EMAIL: " + message);
    }
}

class SlackNotifier implements Notifier {
    @Override
    public void notify(String message) {
        System.out.println("SLACK: " + message);
    }
}

class AlertService {
    private final Notifier notifier;

    public AlertService(Notifier notifier) {
        this.notifier = notifier;
    }

    public void sendAlert(String message) {
        notifier.notify(message);
    }
}

AlertService can work with any Notifier. That flexibility comes from polymorphism. The fact that AlertService holds a Notifier is composition.

Design choice What it does well Main risk Rule of thumb
Polymorphism Swap behavior behind a common contract Too many tiny abstractions if overused Great for variation
Inheritance Shared base behavior and subtype modeling Tight coupling Use selectively
Composition Assemble behavior from smaller objects More objects to wire together Healthy default

Warning card · Polymorphism is not “magic reuse”

If every behavior difference is tiny and artificial, you may be creating an abstraction zoo. Use polymorphism when you have a real axis of variation that callers should not need to branch on themselves.

Here’s the compressed cheat sheet. Read this once more before an interview, then try explaining each bullet out loud in your own words. If you can say the idea, the tradeoff, and a Java example, you’re in strong shape.

State + behavior together Protect invariants Depend on contracts Use real subtype relationships Same message, different runtime behavior Composition is often the safer default

1) OOP mental model

  • A class is a blueprint; an object is a runtime instance.
  • Good OOP assigns responsibilities to the right objects.
  • It improves cohesion and reduces scattered logic.

2) Encapsulation

  • Not just private fields — it’s about controlling valid state changes.
  • Expose meaningful behavior methods, not blind mutation everywhere.
  • Protect internal collections and invariants.

3) Abstraction

  • Expose essential behavior, hide implementation detail.
  • Interfaces express contracts; abstract classes can share partial behavior.
  • Depending on abstractions improves extensibility and testing.

4) Inheritance

  • Use it for true is-a relationships, not just code reuse.
  • It can share behavior, but it creates tight coupling.
  • If the hierarchy feels forced, it probably is.

5) Polymorphism

  • One interface/reference can work with many implementations.
  • Runtime dispatch chooses the concrete overridden method.
  • It reduces caller-side branching and centralizes behavior differences.

6) Composition over inheritance

  • Prefer assembling behavior from collaborators when possible.
  • It is usually more flexible and easier to evolve than deep hierarchies.
  • Think has-a before you jump to is-a.
