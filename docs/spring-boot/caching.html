Caching Basics with @Cacheable

Spring cache abstraction wraps method calls and stores results by key.

← Back to Spring Boot
🧒 ELI5

Keep snacks nearby

Instead of walking to the pantry every time, you keep snacks on your desk.

🔗 ANALOGY

Sticky note reminders

If you already looked up a fact, you keep the sticky note so you do not search again.

Technical

What it means in real code

Definition: @Cacheable checks the cache before running the method, and stores the result by key.

@Cacheable implements cache-aside: the first request loads from DB and stores the result, later requests hit the cache.

Keys must uniquely represent the query; ambiguous keys cause collisions and subtle bugs.

Why it matters

  • Faster responses
  • Less database load
  • Simple read optimization

Common pitfall

  • Repeated heavy queries
  • Expensive API calls
  • No memoization
Cache hitCache missKey
Call method→Check cache→Return cached→Or compute + store

Interview answer

“@Cacheable wraps a method so it returns cached data when a key exists, otherwise it computes and stores.”

StepResult
HitFast return
MissCompute then store
EvictRemove stale
Cacheable example
@Cacheable(value = "users", key = "#id")
public User getUser(String id) {
  return userRepo.findById(id).orElseThrow();
}
Warning

Cache keys must be stable

If keys are inconsistent, you can miss caches and waste memory.

Mini takeaway
  • @Cacheable caches method results.
  • Keys define cache entries.
  • Enable caching once per app.
Quiz

What happens on a cache hit with @Cacheable?

Correct answer: C On a cache hit, the cached value is returned and the method body is skipped.
Section 2 of 5 · ~7 mins

Writing, Updating, and Evicting

Use @CachePut to update caches and @CacheEvict to remove stale entries.

🧒 ELI5

Replace old notes

If the phone number changes, you update the sticky note so it stays correct.

🔗 ANALOGY

Fridge cleanup

When food expires, you throw it away so no one eats it later.

Technical

What it means in real code

Definition: @CachePut always updates the cache after method execution, while @CacheEvict removes entries.

When data changes, you must update or evict cached entries to avoid stale reads.

@CachePut writes through the cache, while @CacheEvict removes entries to force a refresh on next read.

Why it matters

  • Keeps cache aligned with writes
  • Prevents stale reads
  • Supports explicit invalidation

Common pitfall

  • Never evicting
  • Updates without cache updates
  • Multiple caches out of sync
@CachePut@CacheEvictEviction
Write request→Persist change→Update cache→Evict stale entries

Interview answer

“I use @CachePut for write-through updates and @CacheEvict for invalidation when data changes.”

AnnotationBehavior
@CacheableRead-through
@CachePutWrite-through
@CacheEvictInvalidate
Evict after update
@CacheEvict(value = "users", key = "#user.id")
public User update(User user) {
  return userRepo.save(user);
}
Warning

Do not forget evictions

If you update data without evicting caches, readers will see stale results.

Mini takeaway
  • Use @CachePut for updates.
  • Evict after mutations.
  • Keep cache coherent with DB.
Quiz

Which annotation forces the cache to update even on a cache hit?

Correct answer: B @CachePut always executes the method and updates the cache.

Cache Managers and Eviction Strategies

Choose the right cache manager and eviction policy for your latency and consistency goals.

🧒 ELI5

Different fridges

A small fridge keeps food close, but you might need a big shared fridge for the whole family.

🔗 ANALOGY

Local notes vs shared notebook

Personal sticky notes are fast, but a shared notebook keeps everyone in sync.

Technical

What it means in real code

Definition: Spring uses CacheManager to abstract cache providers like Caffeine, Redis, or Ehcache.

Cache managers define storage and eviction policy; in-memory caches are fast while Redis scales across instances.

Eviction strategies like TTL or LRU should match data churn and consistency requirements.

Why it matters

  • Pick memory vs distributed
  • Control TTL and LRU
  • Match scalability needs

Common pitfall

  • Using local cache in a distributed system
  • No eviction strategy
  • Huge cache growth
CaffeineRedisTTL/LRU
Choose provider→Configure TTL→Bind CacheManager→Monitor hit ratio

Section 4 of 5 · ~7 mins

Consistency and Pitfalls

Caching introduces consistency challenges like stampedes, stale data, and self-invocation issues.

🧒 ELI5

Outdated map

If your map is old, you can take the wrong road and get lost.

🔗 ANALOGY

Library checkout list

If the list is not updated, you might think a book is available when it is not.

Technical

What it means in real code

Definition: Caches can return stale data or cause thundering herd problems if not managed carefully.

Cache consistency is hard: stampedes, stale reads, and race conditions are common in high-traffic systems.

Mitigate with TTLs, locking, and idempotent writes, and monitor hit rate to ensure caching adds value.

Why it matters

  • Avoids stale reads
  • Protects backing stores
  • Ensures consistency

Common pitfall

  • Cache stampedes
  • No TTLs
  • Self-invocation bypassing cache
StampedeStale dataConsistency
Cache miss→Many requests→DB overload→Slow response

Interview answer

“Caching speeds reads but I mitigate stampedes, handle eviction, and avoid self-invocation bypassing the proxy.”

RiskMitigation
StampedeRequest coalescing
Stale dataTTL + evict
Self-callUse proxy aware calls
Self-invocation pitfall
public class UserService {
  @Cacheable("users")
  public User getUser(String id) { ... }

  public User getUserInternal(String id) {
    return getUser(id); // bypasses proxy
  }
}
Warning

Cache bypass in self-calls

Calling a @Cacheable method from the same class bypasses the proxy and skips caching.

Mini takeaway
  • Caching needs eviction strategy.
  • Avoid stampedes.
  • Proxy rules still apply.
Quiz

Why might @Cacheable not work on a self-invocation?

Correct answer: B Spring cache uses proxies, and self-invocation bypasses them.

Edge cases

Edge Cases & Tradeoffs

  • Stale data if eviction is forgotten.
  • Cache stampede under high load.
  • Incorrect keys lead to collisions.
  • Serialization costs can dominate.
  • Caching nulls can mask data fixes.
Checklist

Production Checklist

  • Define cache TTLs and eviction strategy.
  • Use @CacheEvict on writes.
  • Include relevant fields in cache keys.
  • Protect hot keys with locking or refresh.
  • Monitor hit ratio and latency.
