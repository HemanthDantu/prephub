Section 1 of 5 · ~7 mins

Authn, Authz & the Security Filter Chain

Spring Security is not “just an annotation library.” It is a request pipeline that decides identity first, authorization second, and only then lets your controller run.

🧒 ELI5

The office lobby model

Picture an office building. At the front desk, someone checks who you are. At the elevator, your badge decides which floor you can reach. Only after both checks do you enter the room where real work happens.

That is what Spring Security does for HTTP requests. The filter chain acts like the lobby and badge system before the request ever reaches your business logic.

🔗 ANALOGY

Airport security plus boarding gate

At the airport, there are two distinct questions: “Are you a real traveler with valid ID?” and “Are you allowed onto this specific flight?” Security and the gate agent answer different questions at different moments.

Authentication is the ID check. Authorization is the boarding decision. Mixing those together is one of the most common interview mistakes.

Technical

What really happens on a request

Definition: Spring Security wraps each request in a filter chain that can load or establish an Authentication, store it in the SecurityContext, and enforce authorization rules before controller code runs.

The key mental model is request interception. Filters run before your MVC controller, so security remains centralized instead of being copied into every endpoint.

In a JWT-based API, one filter parses the bearer token, delegates validation to a decoder, creates an Authentication object, and stores it in the security context. Later authorization logic reads that context to decide whether the request may continue.

What authentication answers

  • Who is calling?
  • Was identity established by a trusted mechanism?
  • What authorities or scopes came with that identity?

What authorization answers

  • Can this identity call this endpoint?
  • Does it have the required role or scope?
  • Is access denied even though the user is valid?
HTTP request arrives → Security filters run → Authentication created → Authorization checked → Controller executes

Interview answer

“I explain Spring Security as a filter-chain based system. Authentication establishes identity and authorities, usually before controller code. Authorization then checks whether that authenticated principal is allowed to call the endpoint. I avoid burying security decisions in controllers because the filter chain keeps policy centralized and consistent.”

❌ Wrong: controller-centric security
@GetMapping("/admin/reports")
public ReportResponse reports(Authentication authentication) {
    if (authentication == null) {
        throw new ResponseStatusException(HttpStatus.UNAUTHORIZED);
    }
    boolean admin = authentication.getAuthorities().stream()
        .anyMatch(a -> a.getAuthority().equals("ROLE_ADMIN"));
    if (!admin) {
        throw new ResponseStatusException(HttpStatus.FORBIDDEN);
    }
    return service.loadReports();
}
✅ Right: centralize in the chain
@Bean
SecurityFilterChain security(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/actuator/health").permitAll()
            .requestMatchers("/admin/**").hasRole("ADMIN")
            .anyRequest().authenticated()
        )
        .oauth2ResourceServer(oauth2 -> oauth2.jwt());

    return http.build();
}

A strong answer does not stop at “use annotations.” It explains where the decision is enforced and why central enforcement is safer than ad hoc checks.

Warning

permitAll() is not the same as ignoring the path

permitAll() still lets the request travel through the security chain, which means logging, headers, and other protections can still apply. Completely ignoring a path can bypass useful behavior and should be rare.

Mini takeaway
  • Authentication answers “who are you?”
  • Authorization answers “what may you do?”
  • The filter chain is the enforcement boundary, not your controller body.
Quiz

Why is the security filter chain usually a better place for authorization rules than controller code?

Correct answer: A A is right because the filter chain is the real enforcement boundary. Once you push role checks into controllers, teams start duplicating security logic, forgetting edge cases, and returning inconsistent 401/403 behavior.
Section 2 of 5 · ~7 mins

JWT Structure, Claims, and Safe Validation

A JWT that “looks readable” is not a trusted token. Good systems validate signature, issuer, audience, expiry, and intended use before accepting any claim.

🧒 ELI5

The tamper-evident visitor pass

Imagine a visitor pass that lists your name, the building, and an expiration time. Just because the text is easy to read does not mean the pass is real. You still need to check whether the pass was issued by the real front desk and whether it is still valid.

A JWT is like that. Reading claims is easy. Trusting them safely is the hard part.

🔗 ANALOGY

A concert ticket with a digital signature

A concert ticket can contain the event name, seat number, and date. But the venue does not accept any random PDF with those fields. It checks the issuer, the signature, and whether the ticket was actually meant for this event.

JWT validation is the same idea: readable payload is not the same thing as trustworthy authorization data.

Technical

What a JWT really is

Definition: Most application JWTs are signed JSON Web Signatures (JWS): header, payload, and signature. The payload is usually base64url-encoded, not encrypted.

That means anyone holding the token can often decode the claims locally. Confidentiality is not the point. Integrity and trust are the point.

A resource server should validate at least the signature, iss, exp, and usually aud. Depending on the provider, you may also validate scopes, token type, custom claims, or authorized party.

Claims you should understand

  • sub – subject / principal identifier
  • iss – issuer
  • aud – intended audience
  • exp and nbf – validity window

Common production mistakes

  • Decoding without validating
  • Accepting any token signed by the provider, even if it is for another API
  • Putting sensitive data in claims because “it is just base64”
Validate issuer Validate audience JWT is usually not encrypted Never trust decoded text alone

Interview answer

“I treat JWTs as signed claims, not as proof just because they parse. In Spring Boot I prefer a JwtDecoder or resource-server support so signature validation, issuer checks, expiry checks, and custom validators are centralized instead of hand-written in filters.”

StepWhy it matters

❌ Wrong: trusting the payload because it parses
String[] parts = token.split("\\.");
String payloadJson = new String(Base64.getUrlDecoder().decode(parts[1]), StandardCharsets.UTF_8);

JsonNode claims = objectMapper.readTree(payloadJson);
String userId = claims.get("sub").asText();

// Dangerous: no signature verification, no issuer check, no audience check
return new UsernamePasswordAuthenticationToken(userId, token, List.of());
✅ Right: delegate validation to Spring Security
@Bean
JwtDecoder jwtDecoder() {
    NimbusJwtDecoder decoder = NimbusJwtDecoder.withJwkSetUri(jwkSetUri).build();

    OAuth2TokenValidator<Jwt> withIssuer =
        JwtValidators.createDefaultWithIssuer("https://auth.example.com");

    OAuth2TokenValidator<Jwt> withAudience = jwt -> {
        List<String> audience = jwt.getAudience();
        return audience.contains("sr-path-api")
            ? OAuth2TokenValidatorResult.success()
            : OAuth2TokenValidatorResult.failure(
                new OAuth2Error("invalid_token", "Missing required audience", null)
            );
    };

    decoder.setJwtValidator(new DelegatingOAuth2TokenValidator<>(withIssuer, withAudience));
    return decoder;
}
Warning

JWT is not a free revocation system

Self-contained access tokens are great for stateless APIs, but revoking them early is harder than invalidating a server-side session. That is why short lifetimes and careful refresh-token handling matter.

Mini takeaway
  • A JWT payload is often readable by design.
  • Readable claims are not trusted claims until validation succeeds.
  • Audience and issuer checks prevent “valid but wrong token” bugs.
Quiz

Why is checking audience important even if the JWT signature is valid?

Correct answer: B B is correct because signature validity only tells you the token was signed by a trusted key. It does not prove your service is the intended recipient. Audience checks close that gap.
Section 3 of 5 · ~7 mins

Access Tokens, Refresh Tokens, and Session Reality

Good token design is about lifetime, blast radius, rotation, and revocation—not just “make everything stateless.”

🧒 ELI5

Day pass vs re-entry slip

A day pass gets you into the park right now, but it expires soon. A re-entry slip lets you go back to the desk and get a fresh pass without proving your identity from scratch every minute.

Access tokens are the day pass. Refresh tokens are the thing you guard more carefully because they can mint new day passes.

🔗 ANALOGY

Hotel room key vs reservation record

Your room key lets you open the door. The reservation record at the front desk is stronger; it can produce a new key if the old one expires or gets deactivated.

You do not walk around handing the front-desk reservation record to every door in the building. That is exactly why refresh tokens should not be sent to resource servers.

Technical

How the two-token model works

Definition: Access tokens are short-lived bearer credentials used by APIs. Refresh tokens are longer-lived credentials used only with the authorization server to obtain new access tokens.

The main design reason for short-lived access tokens is blast radius. If an access token leaks, short expiry limits damage. Refresh tokens are more powerful, so they need tighter storage and revocation strategy.

Modern providers often support refresh token rotation, where each refresh returns a new refresh token and invalidates the old one. Reuse detection helps catch theft.

Good practice

  • Keep access tokens short-lived
  • Use rotation for refresh tokens when available
  • Store refresh tokens more carefully than access tokens

What “stateless” does not mean

  • It does not mean logout is free
  • It does not mean revocation disappears
  • It does not mean refresh tokens belong in every API call
User authenticates → Client gets access + refresh token → API receives only access token → Client refreshes when access expires

Interview answer

“I keep access tokens short-lived because they are bearer credentials that may leak through logs, browser bugs, or proxies. Refresh tokens are stronger and should stay inside the refresh flow with the authorization server. If the provider supports rotation and revocation, I use them because they reduce replay risk.”

TokenWhere it belongs
Access tokenSent to resource servers
Refresh tokenSent only to auth server refresh endpoint
ID tokenIdentity info for client, not API authorization

// Browser stores both tokens in JavaScript-visible storage
// and sends whichever one is available to downstream APIs.

fetch("/api/orders", {
    headers: {
        Authorization: "Bearer " + refreshToken
    }

✅ Right: keep refresh in the refresh flow
// Client calls APIs with access token only.
// When the access token expires, client talks to the authorization server
// (or a backend-for-frontend) to exchange the refresh token for a new access token.

POST /oauth/token

Warning

Logout is harder with self-contained tokens

If access tokens are already issued and self-validating, you cannot instantly “erase” them the way you can delete a server-side session. That is one reason short expiry is such an important design tool.

Mini takeaway
  • Access tokens are for APIs.
  • Refresh tokens are for the authorization server.
  • Short lifetimes and rotation are security controls, not just implementation details.
Quiz

Why should refresh tokens usually not be accepted by resource servers?

Correct answer: C C is the key design distinction. Refresh tokens have a different trust boundary and a different purpose. Treating them like bearer API credentials widens blast radius and breaks the separation between authorization server and resource server.
Section 4 of 5 · ~7 mins

Token Storage, Cookies, CSRF, and XSS

Security design changes when the browser automatically sends credentials. This section is about threat modeling, not one magical “best storage” answer.

🧒 ELI5

Where you keep the spare key matters

If you keep a spare key in your pocket, someone who picks your pocket can steal it. If you keep it in a lockbox attached to the house, random visitors cannot see it—but someone may still trick the lockbox into opening automatically under the wrong conditions.

That is the browser trade-off. JavaScript-visible storage has XSS risk. Auto-sent cookies create CSRF considerations.

🔗 ANALOGY

Wallet vs office badge turnstile

A card in your wallet is visible to you and anything that can reach your wallet. A badge on an automatic turnstile is not manually typed each time, but the turnstile may accept it whenever you approach.

Local storage and HttpOnly cookies create different failure modes. Good engineers stop asking “which one is perfect?” and start asking “which threat am I optimizing against?”

Technical

The real browser trade-off

Definition: If tokens live in JavaScript-visible storage such as localStorage, XSS can expose them. If credentials live in cookies and the browser sends them automatically, CSRF becomes relevant unless mitigations are in place.

That is why architecture matters. A pure SPA using bearer tokens in memory has different trade-offs from a server-rendered app using session cookies. A BFF pattern changes the boundary again by keeping tokens server-side and issuing the browser only a same-site session cookie.

When cookies are attractive

  • HttpOnly can hide token contents from JavaScript
  • Secure restricts them to HTTPS
  • SameSite reduces some cross-site request risk

When bearer headers are attractive

  • APIs stay explicit about credential sending
  • CSRF risk can be lower because browser does not auto-attach the header
  • You still need strong XSS discipline
PatternMain risk to think about
localStorage tokenXSS can exfiltrate it
HttpOnly cookieBrowser auto-sends it, so think about CSRF
BFF/session cookieOperational complexity, but often simpler browser security model

Interview answer

“I do not say ‘always use localStorage’ or ‘always use cookies.’ I explain the threat model. JavaScript-visible storage raises XSS risk. Cookies with automatic sending raise CSRF considerations. In browser-heavy apps I often prefer a BFF or carefully designed cookie strategy because it keeps long-lived tokens away from frontend JavaScript.”

Think in threat models CSRF matters when browser auto-sends credentials Never say one storage choice solves everything
❌ Wrong: disable CSRF because “we use JWT”
@Bean
SecurityFilterChain security(HttpSecurity http) throws Exception {
    http
        .csrf(csrf -> csrf.disable())
        .authorizeHttpRequests(auth -> auth.anyRequest().authenticated());
    return http.build();
}

// This can be wrong if the browser is automatically sending cookies.
✅ Better: align CSRF choice with credential transport
@Bean
SecurityFilterChain security(HttpSecurity http) throws Exception {
    http
        .csrf(csrf -> csrf.csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse()))
        .authorizeHttpRequests(auth -> auth.anyRequest().authenticated())
        .formLogin(Customizer.withDefaults());

    return http.build();
}

// If credentials ride automatically in cookies, protect state-changing requests.
// If you use bearer headers instead, explain why CSRF risk changes.
Warning

“JWT means no CSRF” is not universally true

CSRF depends on how the browser sends credentials. If a browser auto-attaches cookies containing a session or token, CSRF is back in the conversation even if your backend uses JWT under the hood.

Mini takeaway
  • localStorage vs cookies is a threat-model decision.
  • XSS and CSRF are different risks.
  • BFF patterns exist because browser token handling is hard.
Quiz

When is CSRF most relevant in a browser-based application?

Correct answer: B B is correct because CSRF exploits the browser’s habit of automatically attaching credentials. That is the core condition. The token format itself is not the deciding factor.
Section 5 of 5 · ~7 mins

Deep Dive: Spring Boot Resource Server Design

This section connects the framework mechanics to production decisions: authority mapping, custom validators, exception handling, and why built-in support beats hand-rolled filters in most cases.

🧒 ELI5

Use the building’s real badge system

If your office already has a reliable badge system, you do not invent a handwritten guest list at the side door. You use the existing system and configure it for your building’s rules.

That is what resource-server support gives you. It is the official badge desk for bearer tokens, not a one-off filter written during a sprint crunch.

🔗 ANALOGY

Adapter, not reinvention

Think of Spring Security as a mature security appliance. Your job is usually to configure it well—issuer, audience, authority mapping, public paths, error responses—not to rebuild token parsing by hand.

Technical

What senior engineers actually tune

Definition: A solid resource-server setup usually configures endpoint rules, token decoding, claim validation, authority mapping, and consistent 401/403 handling.

Real systems often need claim translation. For example, an IdP may emit scope, scp, roles, or realm-specific claims. You may need a custom JwtAuthenticationConverter so authorization rules stay readable.

You also often need operational decisions: how JWK rotation is handled, what to log on auth failures, and how much detail error responses should expose.

What to configure explicitly

  • Public vs protected paths
  • Decoder and validators
  • Authority mapping from claims
  • 401 vs 403 handling

Red flags

  • Custom filter parses JWT manually
  • Controllers inspect claims directly for roles
  • Random claims become authorities without normalization

Interview answer

“In Spring Boot I usually start with resource-server support instead of a custom filter. Then I add explicit endpoint rules, issuer and audience validators, and a JwtAuthenticationConverter if claim-to-authority mapping needs customization. That gives me consistent behavior, easier testing, and fewer subtle bugs around token parsing.”

Advanced Spring Boot example
@Bean
SecurityFilterChain security(HttpSecurity http) throws Exception {
    http
        .csrf(csrf -> csrf.disable())
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/actuator/health", "/docs/**").permitAll()
            .requestMatchers(HttpMethod.GET, "/orders/**").hasAuthority("SCOPE_orders.read")
            .requestMatchers(HttpMethod.POST, "/orders/**").hasAuthority("SCOPE_orders.write")
            .anyRequest().authenticated()
        )
        .oauth2ResourceServer(oauth2 -> oauth2
            .jwt(jwt -> jwt.jwtAuthenticationConverter(jwtAuthenticationConverter()))
        );

    return http.build();
}

@Bean
Converter<Jwt, ? extends AbstractAuthenticationToken> jwtAuthenticationConverter() {
    JwtGrantedAuthoritiesConverter scopes = new JwtGrantedAuthoritiesConverter();
    scopes.setAuthorityPrefix("SCOPE_");
    scopes.setAuthoritiesClaimName("scope");

    JwtAuthenticationConverter converter = new JwtAuthenticationConverter();
    converter.setJwtGrantedAuthoritiesConverter(scopes);
    return converter;
}
Custom audience validator
@Bean
JwtDecoder jwtDecoder() {
    NimbusJwtDecoder decoder = NimbusJwtDecoder.withJwkSetUri(jwkSetUri).build();

    OAuth2TokenValidator<Jwt> issuerValidator =
        JwtValidators.createDefaultWithIssuer("https://auth.example.com");

    OAuth2TokenValidator<Jwt> audienceValidator = jwt -> {
        return jwt.getAudience().contains("orders-api")
            ? OAuth2TokenValidatorResult.success()
            : OAuth2TokenValidatorResult.failure(
                new OAuth2Error("invalid_token", "Audience orders-api is required", null)
            );
    };

    decoder.setJwtValidator(new DelegatingOAuth2TokenValidator<>(issuerValidator, audienceValidator));
    return decoder;
}
Warning

Custom filters age badly

Teams often start with a quick custom filter “just to parse the JWT.” Months later, they have reimplemented validation poorly, forgotten issuer checks, and scattered authority mapping logic across services. Start with the framework unless you can clearly justify the deviation.

Checklist

Production checklist

  • Know exactly which claims drive authorization.
  • Validate issuer and audience explicitly.
  • Map provider-specific claims to stable internal authorities.
  • Log authentication failures without leaking token contents.
  • Prefer built-in resource-server support over ad hoc parsing.
Quiz

What is the best reason to prefer Spring’s built-in resource-server support over a homegrown JWT filter?

Correct answer: A A is correct because mature framework support reduces custom parsing bugs and keeps the security boundary consistent. The other choices over-promise guarantees that still require design and configuration.

Fast review

Common traps interviewers expect you to avoid

Trap 1: “JWT means secure by default.”

No. A JWT is only useful if you validate the right claims and understand who issued it and who it is for.

Trap 2: “Authentication and authorization are basically the same.”

They are separate decisions. Good answers keep those concepts clean.

Trap 3: “Stateless means logout/revocation no longer matter.”

Token lifetime and refresh design still matter a lot, especially after compromise.
