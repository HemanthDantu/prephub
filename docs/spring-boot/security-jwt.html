Authn, Authz & Filter Chain

Spring Security runs requests through a filter chain to authenticate and authorize.

🧒 ELI5

Club entry rules

You show your ID at the door, then the bouncer checks if you can enter the VIP area.

🔗 ANALOGY

Airport security

You pass identity checks first, then your boarding permissions decide where you go.

Technical

What it means in real code

Definition: Authentication proves who you are; authorization decides what you can access.

Spring Security runs a filter chain per request that builds the SecurityContext and applies access rules. Authentication verifies identity; authorization decides access.

Separating those two concerns keeps code clear. A common beginner mistake is mixing them and losing track of where access decisions happen.

Filters are ordered, so mistakes in ordering can break authentication or allow requests through unexpectedly.

Why it matters

  • Clear security boundaries
  • Defense in depth
  • Consistent access control

Common smell

  • Mixing authn and authz
  • Skipping filters for some routes
  • Hard-coded roles

Request→Authenticate→Authorize→Handle response

“Spring Security authenticates first, then authorizes through the filter chain based on roles and rules.”

Avoid public endpoints by accident

Mini takeaway
  • Authn before authz.
  • Filter chain is the pipeline.
  • Least privilege.

Section 2 of 5 · ~7 mins

JWT Structure & Validation

JWTs are signed tokens that carry claims; validation must happen on every request.

🧒 ELI5

Signed permission slip

A signed note says you’re allowed to do something, and the signature proves it’s real.

🔗 ANALOGY

Stamped passport

Border officers verify the stamp to ensure it’s authentic.

Technical

What it means in real code

Definition: JWTs have header, payload, signature; servers verify signature and expiry.

A JWT is made of header, payload, and signature. The signature proves the token came from a trusted issuer.

Validation must check signature, expiration, issuer, and audience. Simply decoding the payload without verification is insecure.

If key rotation is supported, make sure you validate against JWKS and cache keys carefully.

Why it matters

  • Stateless authentication
  • Portable claims
  • Scalable APIs

Common smell

  • Skipping signature checks
  • Long-lived access tokens
  • No clock skew handling
Verify signatureShort-lived tokensTrust payload blindly
Receive token→Verify signature→Check exp→Build auth

Interview answer

“I verify JWT signature and expiry on every request; I never trust payload without validation.”

JWT validation (conceptual)
if (!jwt.verifySignature(key)) throw new SecurityException();

Validation happens on every request; no server session required.

Warning

Never trust unsigned tokens

A token without a verified signature is not trustworthy.

Mini takeaway
  • Validate signature & expiry.
  • JWTs are stateless.
  • Claims are only trusted after verification.

Access vs Refresh Tokens

Access tokens should be short-lived; refresh tokens extend sessions safely.

🧒 ELI5

Short pass + long ticket

You use a short pass for entry and a longer ticket to get a new pass.

🔗 ANALOGY

Concert wristband

Your wristband gets you in, but you can replace it at the booth with your ticket.

Technical

What it means in real code

Definition: Access tokens expire quickly; refresh tokens issue new access tokens.

Access tokens are short-lived and used on API requests, while refresh tokens are long-lived and used only to obtain new access tokens.

Rotate refresh tokens and revoke them on suspicious activity to reduce replay risk.

A safe beginner pattern is short access tokens plus refresh token rotation with server-side revocation checks.

Why it matters

  • Limit damage if stolen
  • Allow session continuity
  • Support rotation

Common smell

  • Long-lived access tokens
  • No rotation
  • Storing refresh in localStorage

Section 4 of 5 · ~7 mins

Token Storage & CSRF

Storage decisions affect XSS/CSRF risks and overall security.

🧒 ELI5

Where you keep your key

If you leave a key on the table, anyone can grab it. If you lock it, it’s safer.

🔗 ANALOGY

Mailbox vs pocket

A mailbox is protected; a note left outside isn’t.

Technical

What it means in real code

Definition: Authorization headers avoid CSRF, while cookies require CSRF protection.

Token storage matters. Cookies require CSRF defenses, while in-memory storage avoids CSRF but loses tokens on refresh.

A safe beginner-friendly pattern is short-lived access tokens plus secure storage, combined with SameSite cookies when cookies are used.

Never put sensitive data inside the JWT payload; it is only base64-encoded, not encrypted.

Why it matters

  • Reduce XSS/CSRF risk
  • Prevent token theft
  • Comply with best practices

Common smell

  • Storing access token in localStorage
  • Disabling CSRF blindly
  • No secure cookie flags
HttpOnly cookiesAuthorization headerlocalStorage tokens
Token stored→Browser sends→Server validates→Allow/deny

“If I use cookies, I keep CSRF enabled. If I use Authorization headers, CSRF is less relevant.”

Deep Dive: Token Validation & Filter Chains

JWT security relies on correct validation, key rotation, and filter ordering.

🧒 ELI5

Check the stamp

You only accept a ticket if the stamp is real and not expired.

🔗 ANALOGY

Airport security

Multiple checkpoints verify your identity and boarding pass.

Technical

What it means in real code

Definition: JWT validation must verify signature, issuer, audience, and expiry; filters enforce auth and authz.

Why it matters

  • Prevents token spoofing
  • Protects APIs
  • Enforces least privilege

Common pitfall

  • Accepting unsigned tokens
  • Overlapping filter chains

Edge cases

Edge Cases & Tradeoffs

  • Clock skew causes token rejection.
  • JWT is not encrypted; do not store secrets inside.
  • Key rotation must be supported (kid + JWKS).
  • Refresh tokens need secure storage.
  • CSRF still matters if tokens are in cookies.
Checklist

Production Checklist

  • Validate issuer, audience, and signature.
  • Use short-lived access tokens.
  • Rotate signing keys and cache JWKS.
  • Use httpOnly cookies or secure storage.
  • Log auth failures with correlation IDs.
