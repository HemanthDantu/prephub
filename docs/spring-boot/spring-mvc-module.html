DispatcherServlet — The Front Controller
🍅 Pomodoro · 25 min
🧒 ELI5 — The Hotel Concierge

Imagine a large hotel. Every single guest walks in through one main entrance — the front desk. The concierge (DispatcherServlet) greets every single person. They look at the request ("I need room service" / "I need a taxi") and route it to the right department (restaurant, transport). The departments don't deal with guests directly — all communication goes through the concierge.

This is the Front Controller pattern: one entry point, centralized routing.

⚙️ The 5 Components of DispatcherServlet
1. DispatcherServlet
Receives ALL HTTP requests. Registered at "/" by default in Spring Boot.
↓
2. HandlerMapping
Finds which @Controller method handles this URL. Built from @RequestMapping at startup.
↓
3. HandlerAdapter
Invokes the found method. Resolves method args (@PathVariable, @RequestBody, etc.).
↓
4. MessageConverter
Serializes return value to JSON/XML. Jackson's MappingJackson2HttpMessageConverter handles JSON.
↓
5. HTTP Response
Written back to client with body, status, and headers.

HandlerMapping — How URLs Get Mapped at Startup

RequestMappingHandlerMapping scans all @Controller / @RestController beans at startup and builds a lookup map: URL + HTTP method → HandlerMethod. This is why adding a new @GetMapping requires a restart — the map is built once at startup.

// Conceptually what gets built:
Map<RequestMappingInfo, HandlerMethod> mappings = {
    GET "/accounts/{id}" → AccountController.getAccount(),
    POST "/accounts"    → AccountController.createAccount(),
    DELETE "/accounts/{id}" → AccountController.deleteAccount()
};

HttpMessageConverter — How @ResponseBody Works

When a method is annotated with @ResponseBody (or the class with @RestController), the HandlerAdapter asks the registered HttpMessageConverter list: "can you convert this object to JSON?" MappingJackson2HttpMessageConverter says yes for any object when Jackson is on the classpath.

// Spring has a list of converters — first matching one wins:
List<HttpMessageConverter<?>> converters = [
    StringHttpMessageConverter,       // String → text/plain
    MappingJackson2HttpMessageConverter, // Object → application/json
    Jaxb2RootElementHttpMessageConverter // Object → application/xml
    ...
]

Content Negotiation

When a client sends Accept: application/xml, Spring uses Content Negotiation to pick the right converter. The ContentNegotiationManager inspects:

  1. The Accept header (highest priority)
  2. URL extension (/accounts/1.json) — deprecated
  3. Request parameter (?format=json)
@GetMapping(value = "/accounts/{id}",
            produces = {"application/json", "application/xml"})
public Account getAccount(@PathVariable long id) { ... }
// produces= restricts which types this endpoint can return
💬 Interview Answer

"DispatcherServlet is a Front Controller — all requests enter through it. It delegates to HandlerMapping (which URL maps to which method), then HandlerAdapter (which invokes the method and resolves arguments), then HttpMessageConverter (which serializes the return value to JSON/XML based on the Accept header via ContentNegotiation). RequestMappingHandlerMapping builds the URL-to-method map at startup by scanning @RequestMapping annotations."

🧠 Quiz — Section 1
At what point does RequestMappingHandlerMapping build the URL → method map?
✅ RequestMappingHandlerMapping implements InitializingBean, so its afterPropertiesSet() runs during ApplicationContext startup. It scans all beans for @RequestMapping annotations and builds the complete routing table at that moment. This is why changes to mappings require a restart.
Section 2 of 5 · Spring MVC
Request Lifecycle — Full Trace
🍅 Pomodoro · 30 min
🧒 ELI5 — A Letter Through the Post Office

Imagine mailing a letter. It goes through: security scan (Filter chain — check the envelope), sorting center (DispatcherServlet — who handles this?), department routing (HandlerMapping), pre-checks (HandlerInterceptor.preHandle — is the sender authorized?), actual delivery (your Controller method), quality check on the reply (HandlerInterceptor.postHandle), package the reply (MessageConverter), then send it back (Response). Filters touch the raw envelope. Interceptors see the sorted, decoded letter.

⚙️ The Complete Pipeline
HTTP Request arrives
Raw bytes from the network
↓
Filter Chain (Servlet Filters)
Runs BEFORE DispatcherServlet. Sees raw HttpServletRequest/Response. Auth, CORS, logging, compression.
↓
DispatcherServlet
Receives the request. Delegates to HandlerMapping to find the handler.
↓
HandlerMapping
Returns a HandlerExecutionChain (handler + list of interceptors)
↓
HandlerInterceptor.preHandle()
Runs before the controller. Return false to abort the request. Auth checks, rate limiting.
↓
Controller Method Executes
Your @RestController method runs. Arguments resolved by HandlerMethodArgumentResolver.
↓
HandlerInterceptor.postHandle()
Runs AFTER controller, BEFORE response is written. Can modify ModelAndView. Only called if preHandle returned true.
↓
MessageConverter → Response body written
Jackson serializes the return value to JSON based on Accept header.
↓
HandlerInterceptor.afterCompletion()
Always called (even on exception). Cleanup, logging, metrics. Exception object available.
↓
HTTP Response sent to client
Status + headers + JSON body

How @RequestBody Works — Jackson Deserialization

@PostMapping("/accounts")
public Account create(@RequestBody CreateAccountRequest req) { ... }
// 1. Client sends: Content-Type: application/json + JSON body
// 2. MappingJackson2HttpMessageConverter is selected
// 3. Jackson reads the JSON stream
// 4. Calls CreateAccountRequest() constructor or setters
// 5. Your method receives the populated Java object

Internally, HandlerMethodArgumentResolver is what Spring calls to figure out how to populate each parameter. RequestResponseBodyMethodProcessor handles @RequestBody specifically — it reads the request stream and delegates to the selected HttpMessageConverter.

⚙️ HandlerMethodArgumentResolver — Custom Parameter Binding

Spring uses a chain of HandlerMethodArgumentResolver implementations to populate each controller method parameter. Built-ins handle @PathVariable, @RequestParam, @RequestBody, HttpServletRequest, etc. You can register custom ones:

public class CurrentUserResolver implements HandlerMethodArgumentResolver {

    @Override
    public boolean supportsParameter(MethodParameter param) {
        return param.hasParameterAnnotation(CurrentUser.class);
    }

    @Override
    public Object resolveArgument(MethodParameter param, ..., NativeWebRequest request) {
        // Extract user from JWT token in request header
        String token = request.getHeader("Authorization");
        return jwtService.extractUser(token);
    }
}

// Register it:
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addArgumentResolvers(List<HandlerMethodArgumentResolver> resolvers) {
        resolvers.add(new CurrentUserResolver(jwtService));
    }
}

// Then in your controller:
@GetMapping("/me")
public Profile getProfile(@CurrentUser User user) {
    return profileService.getProfile(user.getId());
}
💬 Interview Answer

"The full request lifecycle: Filter chain runs first (before DispatcherServlet — sees raw request/response), then DispatcherServlet delegates to HandlerMapping (returns handler + interceptors), then HandlerInterceptor.preHandle (can abort the request), then the controller method executes with arguments resolved by HandlerMethodArgumentResolver, then postHandle, then MessageConverter writes the response, then afterCompletion (always called, even on exception — good for cleanup). Filters are Servlet-level; Interceptors are Spring MVC-level."

🧠 Quiz — Section 2
You want to log every request's execution time including cleanup. Which lifecycle hook do you use?
✅ afterCompletion() is the right choice because it's always called — even if the controller throws an exception, even if postHandle was skipped. It also receives the exception object. This makes it ideal for metrics, logging, and resource cleanup. postHandle() is NOT called when the controller throws an exception.
Section 3 of 5 · Spring MVC
Parameter Binding — @PathVariable, @RequestParam, @RequestBody
🍅 Pomodoro · 25 min
🧒 ELI5 — Three Ways to Pass a Note
  • @PathVariable — the info is written in the address on the envelope: /users/42. It identifies the resource.
  • @RequestParam — the info is on a sticky note attached to the outside: ?sort=asc&page=2. It modifies or filters.
  • @RequestBody — the info is inside the envelope as a full letter. The body contains rich structured data (JSON). Used for creates/updates.
⚙️ @PathVariable — Resource Identifier in the URL
// URL: GET /accounts/42/transactions/99
@GetMapping("/accounts/{accountId}/transactions/{txId}")
public Transaction getTransaction(
        @PathVariable long accountId,
        @PathVariable long txId) {
    return service.find(accountId, txId);
}

// Optional path variable (path must also be optional):
@GetMapping({"/accounts/{id}", "/accounts"})
public List<Account> get(@PathVariable(required = false) Long id) { ... }

// Name mismatch (variable name ≠ path placeholder):
@GetMapping("/accounts/{account-id}")
public Account get(@PathVariable("account-id") long id) { ... }
⚙️ @RequestParam — Query String Parameters
// URL: GET /accounts?page=0&size=10&sort=createdAt&active=true
@GetMapping("/accounts")
public Page<Account> list(
        @RequestParam(defaultValue = "0") int page,
        @RequestParam(defaultValue = "10") int size,
        @RequestParam(required = false) String sort,
        @RequestParam(defaultValue = "true") boolean active) {
    return service.findAll(page, size, sort, active);
}

// Get all params as a map:
@GetMapping("/search")
public List<Account> search(@RequestParam Map<String,String> params) { ... }

// Multi-value param: ?tags=java&tags=spring
@GetMapping("/accounts")
public List<Account> filter(@RequestParam List<String> tags) { ... }
⚙️ @RequestBody — Rich JSON Payload
// POST /accounts with body: {"name":"John","balance":1000.00}
@PostMapping("/accounts")
@ResponseStatus(HttpStatus.CREATED)
public Account create(@RequestBody CreateAccountRequest req) {
    return service.create(req);
}

// With validation (see Section 4):
@PostMapping("/accounts")
public Account create(@Valid @RequestBody CreateAccountRequest req) { ... }

Comparison: When to Use What

AnnotationLocation in RequestHTTP MethodsUse For
@PathVariable URL path segment All Resource ID — "which resource?"
@RequestParam Query string Mostly GET Filtering, pagination, sorting — "how to query?"
@RequestBody Request body (JSON) POST/PUT/PATCH Create or update — structured data
@RequestHeader HTTP headers All Authorization, correlation IDs, custom metadata
@CookieValue Cookie All Session tokens, preferences
💬 Interview Answer

"@PathVariable extracts from the URL path — used for resource identity. @RequestParam extracts from query string — used for filtering and pagination. @RequestBody deserializes the JSON request body via Jackson — used for POST/PUT with structured data. At the framework level, HandlerMethodArgumentResolver implementations handle each of these — RequestResponseBodyMethodProcessor for @RequestBody, PathVariableMethodArgumentResolver for @PathVariable, etc."

🧠 Quiz — Section 3
For the endpoint GET /accounts?status=active&page=2, how should the parameters be bound?
✅ Query string parameters (?key=value) are bound with @RequestParam. @PathVariable is for path segments like /accounts/{id}. @RequestBody reads the HTTP body (JSON) — GET requests typically have no body. For this endpoint: @RequestParam String status, @RequestParam int page.

@Valid + BindingResult — Validation Pipeline
🍅 Pomodoro · 25 min
🧒 ELI5 — The Bouncer at the Door

Imagine a club. A bouncer (@Valid) checks every person at the door before they're let in to the party (your controller method). The bouncer has a checklist: "Must be 21+, must have ID, shoes required." If someone fails any check, the bouncer turns them away with a specific reason — they never enter the club.

BindingResult is like the bouncer's clipboard — it records every rejection reason so you can inspect them and send a nice error message back.

⚙️ Where Validation Fits in the Pipeline
Request arrives
→
@RequestBody deserialized
→
@Valid triggers validation
→
Fails → MethodArgumentNotValidException
→
Passes → Controller runs

Validation runs in the HandlerAdapter phase, after deserialization and before the controller method body executes. It's part of argument resolution.

Setting Up Bean Validation

// DTO with constraints:
public class CreateAccountRequest {

    @NotBlank(message = "Name is required")
    @Size(min = 2, max = 100)
    private String name;

    @NotNull
    @DecimalMin(value = "0.0", inclusive = false)
    private BigDecimal initialBalance;

    @Email
    private String email;

    @Pattern(regexp = "^[A-Z]{3}$", message = "Must be 3-letter currency code")
    private String currency;
}

Style 1: Let Spring Auto-Throw (Recommended)

// Validation failure → Spring throws MethodArgumentNotValidException automatically
// Catch it in @RestControllerAdvice (see Bridge module Section 2)
@PostMapping("/accounts")
public Account create(@Valid @RequestBody CreateAccountRequest req) {
    // Only reaches here if ALL validations pass ✓
    return service.create(req);
}

Style 2: Manual Check with BindingResult

// BindingResult MUST immediately follow the validated parameter
@PostMapping("/accounts")
public ResponseEntity<?> create(
        @Valid @RequestBody CreateAccountRequest req,
        BindingResult result) {

    if (result.hasErrors()) {
        Map<String, String> errors = result.getFieldErrors().stream()
            .collect(Collectors.toMap(
                FieldError::getField,
                FieldError::getDefaultMessage
            ));
        return ResponseEntity.badRequest().body(errors);
    }

    return ResponseEntity.status(201).body(service.create(req));
}

When BindingResult follows the validated parameter, Spring does NOT throw automatically — it populates the result object and lets you decide what to do.

@Validated vs @Valid

// @Valid = standard JSR-380 annotation (javax.validation)
// @Validated = Spring's annotation — adds group validation support

public interface OnCreate {}
public interface OnUpdate {}

public class AccountRequest {
    @NotNull(groups = OnCreate.class) // only validated on create
    private String name;

    @NotNull(groups = OnUpdate.class) // only on update
    private Long id;
}

@PostMapping
public Account create(@Validated(OnCreate.class) @RequestBody AccountRequest req) { ... }

@PutMapping
public Account update(@Validated(OnUpdate.class) @RequestBody AccountRequest req) { ... }
💬 Interview Answer

"@Valid triggers Bean Validation (JSR-380) during argument resolution in the HandlerAdapter phase — after JSON deserialization, before the controller method runs. Failure throws MethodArgumentNotValidException, which a @RestControllerAdvice can catch globally. If BindingResult immediately follows the validated parameter, Spring suppresses the exception and you get manual control. @Validated (Spring's version) adds validation group support — useful for different rules on create vs update endpoints."

🧠 Quiz — Section 4
You annotate a String field with @Size(min=2, max=50). The client sends a null value. What happens?
✅ Most Bean Validation constraints (including @Size, @Min, @Max, @Email, @Pattern) treat null as valid and skip the check. This follows the principle that null and "absent" should be handled by @NotNull or @NotBlank separately from format/range checks. To require a non-empty string, you typically combine: @NotBlank @Size(max=50).
Section 5 of 5 · Spring MVC
Filters vs Interceptors
🍅 Pomodoro · 25 min
🧒 ELI5 — Airport vs Hotel

Filters are like airport security — they check every single person entering the country, regardless of which hotel they're going to. They work at the border (Servlet container level) and don't know anything about your destination.

Interceptors are like the hotel's own staff — they only engage with guests who are specifically staying at that hotel (Spring MVC), and they know which room (controller method) the guest is going to.

🔵 Servlet Filter

  • Javax/Jakarta Servlet API
  • Runs BEFORE DispatcherServlet
  • Sees raw HttpServletRequest / Response
  • Can intercept ANY request (static files, actuator, etc.)
  • No access to Spring beans directly (unless you grab from context)
  • Can modify request body / headers
  • Use for: CORS, auth tokens, compression, logging at network level

🟣 HandlerInterceptor

  • Spring MVC specific
  • Runs INSIDE DispatcherServlet
  • Has access to HandlerMethod (knows which controller runs)
  • Only intercepts Spring MVC requests
  • Full Spring bean injection
  • preHandle / postHandle / afterCompletion hooks
  • Use for: auth, audit logging, rate limiting at controller level
⚙️ Implementing a Filter
@Component
public class RequestLoggingFilter implements Filter {

    @Override
    public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain)
            throws IOException, ServletException {

        HttpServletRequest request = (HttpServletRequest) req;
        long start = System.currentTimeMillis();

        // Pre-processing (before DispatcherServlet)
        log.info("→ {} {}", request.getMethod(), request.getRequestURI());

        chain.doFilter(req, res); // pass to next filter / DispatcherServlet

        // Post-processing (after full response)
        log.info("← {} ms", System.currentTimeMillis() - start);
    }
}

// Or with ordering + URL pattern:
@Bean
public FilterRegistrationBean<RequestLoggingFilter> loggingFilter() {
    var reg = new FilterRegistrationBean<>(new RequestLoggingFilter());
    reg.addUrlPatterns("/api/*");
    reg.setOrder(1); // lower = runs first
    return reg;
}
⚙️ Implementing a HandlerInterceptor
@Component
public class AuthInterceptor implements HandlerInterceptor {

    @Autowired
    private JwtService jwtService; // Spring beans available here

    @Override
    public boolean preHandle(HttpServletRequest req, HttpServletResponse res,
                               Object handler) {
        if (handler instanceof HandlerMethod hm) {
            // Can inspect the target controller method
            if (hm.hasMethodAnnotation(RequiresAuth.class)) {
                String token = req.getHeader("Authorization");
                if (!jwtService.isValid(token)) {
                    res.setStatus(401);
                    return false; // abort — controller never runs
                }
            }
        }
        return true; // continue
    }

    @Override
    public void afterCompletion(HttpServletRequest req, HttpServletResponse res,
                                  Object handler, Exception ex) {
        // Cleanup — always runs
    }
}

// Register it:
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Autowired AuthInterceptor authInterceptor;

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(authInterceptor)
                .addPathPatterns("/api/**")
                .excludePathPatterns("/api/health", "/api/auth/login");
    }
}

💬 Interview Answer

"Filters are Servlet-level — they run before DispatcherServlet, see raw HttpServletRequest/Response, and intercept everything including static files. They can't easily access Spring beans. Interceptors are Spring MVC-level — they run inside DispatcherServlet, have access to the HandlerMethod (so they know which controller and method), and support full Spring injection. Use Filters for cross-cutting network concerns (CORS, compression, request decoding); use Interceptors when you need Spring context or controller-level awareness (auth based on annotations, audit logging per endpoint)."

🧠 Quiz — Section 5
You need to implement JWT auth that skips endpoints annotated with @Public. Which mechanism do you choose and why?
✅ HandlerInterceptor is the right choice here. In preHandle(), you receive the handler as a HandlerMethod object, which lets you call hm.hasMethodAnnotation(Public.class) or check class-level annotations. A Servlet Filter doesn't have access to controller metadata — it only sees the raw URL and headers, not Spring annotations on methods.
