Roles and Tokens

OAuth2 defines roles for clients, authorization servers, and resource servers, while OIDC adds identity.

← Back to Spring Boot
🧒 ELI5

Club wristbands

The bouncer gives you a wristband to access certain rooms.

🔗 ANALOGY

Event tickets

A ticket grants entry; it is checked at the door.

Technical

What it means in real code

Definition: OAuth2 uses access tokens for resource access; OIDC adds ID tokens to assert user identity.

OAuth2 defines roles and scopes that grant access, while OIDC adds identity so you can authenticate users and get claims.

Understanding the difference prevents misuse of ID tokens for API access.

As a beginner, remember: OAuth2 is about access, OIDC is about who the user is.

Why it matters

  • Delegated auth
  • Single sign-on
  • Standardized token handling

Common pitfall

  • Confusing ID vs access token
  • No issuer validation
  • Custom auth flows

OAuth2 Flows

Authorization Code with PKCE is the standard flow for browser and mobile clients.

🧒 ELI5

Ticket exchange

You get a receipt, then trade it for the actual ticket at the counter.

🔗 ANALOGY

Locker key

You receive a claim ticket and exchange it for the item.

Technical

What it means in real code

Definition: Auth Code + PKCE prevents token interception by using a code challenge and verifier.

Authorization code flow with PKCE is the modern standard for browser and mobile clients.

Client credentials flow is for machine-to-machine access with no user context.

Implicit flow is deprecated; do not use it for new systems.

Why it matters

  • Secure browser/mobile auth
  • Prevents code theft
  • Standard flow

Common pitfall

  • Implicit flow in SPAs
  • No PKCE
  • Tokens in URLs

“Auth Code with PKCE is the recommended flow for public clients like SPAs.”

Technical

What it means in real code

Definition: Boot auto-configures OAuth2 client and resource server using properties and SecurityFilterChain.

Spring Boot uses client registrations and provider metadata to configure OAuth and OIDC quickly.

Resource servers validate JWTs on every request and map scopes to authorities.

If you see 401s, verify issuer, audience, and clock skew first before debugging app code.

Why it matters

  • Faster setup
  • Standardized configuration
  • Secure defaults

Common pitfall

  • Custom token parsing
  • No JWT validation
  • Manual filter chains

Interview answer

“I use the OAuth2 resource server starter and configure issuer-uri so Boot validates JWTs automatically.”

Property sources are ordered, and later sources override earlier ones. Profiles and environment variables are the safe way to separate dev/test/prod behavior.

SecurityFilterChain example
@Bean
SecurityFilterChain security(HttpSecurity http) throws Exception {
  http.authorizeHttpRequests(auth -> auth
        .requestMatchers("/actuator/**").permitAll()
        .anyRequest().authenticated())
      .oauth2ResourceServer(oauth2 -> oauth2.jwt());
  return http.build();
}
Warning

Do not skip token validation

If you disable validation, any JWT-like token could be accepted.

Mini takeaway
  • Use the resource server starter.
  • Configure issuer-uri.
  • Let Boot validate tokens.

Security Pitfalls and Validation

Validate issuer, audience, and clock skew to avoid accepting invalid tokens.

🧒 ELI5

Check the wristband

You look closely to ensure it is real before letting someone enter.

🔗 ANALOGY

Inspect the ticket

You check the date and hologram to make sure the ticket is real.

Technical

What it means in real code

Definition: Token validation checks signature, issuer, audience, and expiration; OIDC adds user claims.

Common pitfalls include skipping audience checks, trusting unsigned tokens, or storing tokens in localStorage.

Always validate issuer and audience, rotate keys, and use secure storage with CSRF protections when cookies are involved.

For beginners, the safest default is short-lived access tokens with secure cookie storage and CSRF protection.

Why it matters

  • Prevents token replay
  • Avoids accepting wrong tokens
  • Improves security

Common pitfall

  • No audience check
  • Large clock skew
  • Trusting unsigned tokens
IssuerAudienceExpiry

Deep Dive: Token Lifetimes & Refresh

Correct token lifetimes and refresh strategies prevent security gaps and poor UX.

🧒 ELI5

Short passes

Short passes expire quickly so you check tickets again.

🔗 ANALOGY

Temporary badges

Badges expire and must be renewed.

Technical

What it means in real code

Definition: Access tokens should be short-lived; refresh tokens renew them securely.

Why it matters

  • Limits token leakage impact
  • Improves security
  • Supports long sessions

Common pitfall

  • Long-lived access tokens
  • Refresh tokens in browser storage
  • No revocation

Edge cases

Edge Cases & Tradeoffs

  • Auth code flow should use PKCE for public clients.
  • Refresh token rotation prevents replay.
  • ID token is for identity, not API access.
  • JWKS caching needs eviction on key rotation.
  • Scopes and claims should be minimal.
Checklist

Production Checklist

  • Validate issuer, audience, and signature.
  • Use OIDC discovery endpoints.
  • Separate access and ID token usage.
  • Store tokens securely and avoid localStorage for SPAs.
  • Handle logout and token revocation.
