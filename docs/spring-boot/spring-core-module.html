What is a Spring Bean?
🍅 Pomodoro · 25 min
🧒 ELI5 — What is a Bean?

Imagine you run a restaurant. Every morning, you personally hire every single cook, dishwasher, and waiter — call each one yourself, tell them where to stand, make sure they know each other. Exhausting, right?

Now imagine you hire a manager (Spring). You just tell the manager "I need a chef, a server, and a cashier" — the manager handles hiring, introduces them to each other, and tells you when they're ready. That manager is Spring. The staff members are Beans.

⚙️ Technical

A Spring Bean is simply a Java object whose lifecycle is managed by the Spring IoC (Inversion of Control) container — not by you manually using new.

Without Spring

// YOU manage every dependency manually
DataSource ds = new DataSource();
AccountRepo repo = new AccountRepo(ds);
TransactionService svc = new TransactionService(repo);

With Spring (IoC)

@Component
public class TransactionService {
    private final AccountRepo repo;

    public TransactionService(AccountRepo repo) {
        this.repo = repo; // Spring injects this
    }
}

You declare what you need; Spring builds the dependency graph and wires everything. This is Inversion of Control (IoC).

The ApplicationContext

Beans live in the ApplicationContext — think of it as a giant Map<String, Object> mapping bean names to instances:

// Conceptually:
Map<String, Object> context = {
    "transactionService" → TransactionService@1a2b3c,
    "accountRepo"        → AccountRepo@4d5e6f,
    "dataSource"         → DataSource@7g8h9i
};

By default, Spring creates exactly one instance of each bean per ApplicationContext — this is Singleton scope.

💬 Interview Answer

"A Spring Bean is any object whose lifecycle — creation, dependency injection, initialization, and destruction — is managed by the Spring IoC container. Instead of manually constructing object graphs with new, you declare beans and their dependencies, and the container wires them together. This is Inversion of Control."

🔗 Analogy — The Vending Machine

Without Spring: you have a pantry — you go find every ingredient yourself and cook. With Spring: you have a vending machine. You press the button ("give me a TransactionService") and it appears — fully assembled, dependencies included. Spring is the vending machine.

🧠 Quiz — Section 1
What does "Inversion of Control" mean in the context of Spring?
✅ IoC means control is inverted — instead of your code calling new to create and wire dependencies, you hand that responsibility to the Spring container. Your code just declares what it needs.

@Component, @Service, @Repository, @Controller
🍅 Pomodoro · 25 min
🧒 ELI5 — The Name Tags

Think of a school where everyone wears a name tag. All four annotations are basically the same name tag that says "Spring, manage this object!" But some name tags have a subtitle: Chef, Librarian, Receptionist. The subtitle doesn't change what the school does with you — it just tells everyone at a glance what your role is.

⚙️ Technical — Component Hierarchy

All four annotations are ultimately meta-annotated with @Component. They all get picked up by Spring's component scan. The differences are mostly semantic — but @Repository has one real behavioral difference.

@Repository — The Real Behavioral Difference

@Repository enables exception translation. Spring wraps raw JDBC/JPA exceptions (like SQLException) into Spring's DataAccessException hierarchy — so your service layer isn't coupled to the DB technology.

@Repository
public class UserRepository {
    // If this throws SQLException,
    // Spring translates it to DataAccessException
    public User findById(long id) { ... }
}

Why Use Semantic Annotations?

  • Readability — tells humans what layer a class belongs to
  • Tooling — IDEs and static analyzers can warn you about architectural violations
  • Future-proofing — Spring can add layer-specific behavior later without you changing code
  • AOP targeting — you can write aspects that apply only to @Service or @Repository beans
💬 Interview Answer

"All four are @Component specializations — they all register the class as a Spring bean. The difference is semantic layering: @Service signals business logic, @Repository signals data access and adds exception translation (SQLException → DataAccessException), and @Controller marks a class for Spring MVC handler mapping. Using the right annotation makes architecture intent explicit and allows Spring or AOP to target specific layers."

⚠️ Common Mistake

Using @Component everywhere — technically works, but you lose the architectural signal and @Repository's exception translation. Always use the most specific annotation for the layer.

🧠 Quiz — Section 2
Which annotation has a real behavioral difference beyond just being a semantic marker?
✅ @Repository is the only one with real extra behavior: it activates Spring's PersistenceExceptionTranslationPostProcessor, which wraps technology-specific DB exceptions into Spring's DataAccessException hierarchy. @Service is purely semantic. @Controller needs @ResponseBody to serialize JSON (that's @RestController).
Section 3 of 7 · Spring Core
Dependency Injection — 3 Styles
🍅 Pomodoro · 30 min
🧒 ELI5 — Getting Your Tools

You're a carpenter. You need a hammer. Three ways to get it:

  1. Constructor injection: You say "I won't start work unless you give me a hammer upfront." It's locked in. Non-negotiable. You always have it.
  2. Setter injection: You start work, and someone can drop off a hammer (or not) later. Good for optional tools.
  3. Field injection: Someone secretly puts a hammer in your toolbox while you're not looking. Convenient — but you can't see it or test without them.
⚙️ Style 1: Constructor Injection (Recommended)
@Service
public class OrderService {

    private final PaymentGateway gateway;
    private final InventoryClient inventory;

    // @Autowired optional since Spring 4.3 (single constructor)
    public OrderService(PaymentGateway gateway, InventoryClient inventory) {
        this.gateway = gateway;
        this.inventory = inventory;
    }
}

Why it's best:

  • Dependencies are final — immutable after construction
  • Testable without Spring — just pass mocks in the constructor
  • NPE-safe — if a dep is missing, the app fails fast at startup, not at runtime
  • Makes the dependency graph explicit and visible
⚙️ Style 2: Setter Injection
@Service
public class ReportService {

    private MailSender mailSender;

    @Autowired(required = false) // optional dep
    public void setMailSender(MailSender mailSender) {
        this.mailSender = mailSender;
    }
}

Use setter injection for optional dependencies — the bean can exist and function without the dependency being set. Note: @Autowired(required=false) — Spring won't throw if no matching bean is found.

⚙️ Style 3: Field Injection (Avoid in Production Code)
@Service
public class BadService {

    @Autowired // Spring injects via reflection
    private UserRepository userRepo;
}

Problems:

  • Hidden dependencies — you can't tell what a class needs without reading the body
  • Breaks testing — can't pass mocks without a Spring context or Mockito reflection hacks
  • Field can't be final — allows mutable state
  • Spring team and IntelliJ both warn against it
⚙️ @Autowired(required=false) — What Does It Do?

When you mark a dependency as required=false, Spring will inject the bean if one exists, but won't throw a NoSuchBeanDefinitionException if it doesn't. The field/setter is simply left null (or not called).

@Autowired(required = false)
private CacheManager cacheManager; // null if no cache configured

public void doWork() {
    if (cacheManager != null) {
        // use cache
    }
    // otherwise proceed without cache
}

The modern alternative is Optional<CacheManager> injection or using @ConditionalOnBean.

StyleTestabilityImmutabilityRecommended
Constructor ✓ Easy — just pass mocks ✓ Can use final ✓ Yes
Setter Requires setter call No final Optional deps only
Field ✗ Needs Spring/Reflection ✗ No final ✗ Avoid
💬 Interview Answer

"Constructor injection is preferred because dependencies become final (immutable), the class is testable without a Spring context — you just pass mocks directly — and missing dependencies cause a startup failure rather than a runtime NPE. Field injection is convenient but hides dependencies, prevents final fields, and requires Spring or reflection hacks to test. Setter injection is appropriate for optional dependencies."

🧠 Quiz — Section 3
Why does field injection break unit testing?
✅ With field injection, Spring uses reflection to set the private field. In a pure unit test (no Spring context), that injection never happens — the field stays null. You either need @SpringBootTest (slow, integration-style), Mockito's @InjectMocks (which also uses reflection), or you refactor to constructor injection — which is the right answer.

Bean Scopes
🍅 Pomodoro · 30 min
🧒 ELI5 — The Coffee Shop

Imagine four types of items at a coffee shop:

  • Singleton: The espresso machine. One machine for the entire shop, used by everyone. Always the same one.
  • Prototype: A paper cup. Every customer gets a new one. Once they're done, the shop doesn't care what happens to it.
  • Request: Your order slip. Created when you walk in, gone when your order is complete.
  • Session: Your loyalty card. Exists as long as you're a regular customer (your "session"), then expires.

@Singleton (default)

One instance per ApplicationContext. Shared across all threads. Spring manages full lifecycle.

@Prototype

New instance every time bean is requested. Spring does NOT manage its destruction — you are responsible for cleanup.

@Request

One instance per HTTP request. Only valid in a web context. Gone when the request ends.

@Session

One instance per HTTP session. Persists across requests from the same user. Only valid in web context.

⚙️ Declaring Scopes
@Component
@Scope("singleton") // default — same as omitting @Scope
public class SharedService { }

@Component
@Scope("prototype")
public class StatefulProcessor { }

// Web scopes — only valid with WebApplicationContext
@Component
@Scope(value = "request", proxyMode = ScopedProxyMode.TARGET_CLASS)
public class RequestContext { }

@Component
@Scope(value = "session", proxyMode = ScopedProxyMode.TARGET_CLASS)
public class UserSessionData { }
⚠️ The Stale Prototype Problem — Critical Interview Topic

What happens when you inject a Prototype bean into a Singleton?

@Component
public class MySingletonService {

    @Autowired
    private MyPrototypeBean protoBean; // ← PROBLEM!

    public void doWork() {
        protoBean.process(); // Always the SAME instance!
    }
}

The Singleton is created once. At that moment, Spring injects one Prototype instance. That instance gets stuck in the Singleton forever — it never gets a new Prototype. The "prototype" behavior is lost.

Fix Option 1: ApplicationContext.getBean()

@Component
public class MySingletonService implements ApplicationContextAware {

    private ApplicationContext ctx;

    @Override
    public void setApplicationContext(ApplicationContext ctx) {
        this.ctx = ctx;
    }

    public void doWork() {
        MyPrototypeBean fresh = ctx.getBean(MyPrototypeBean.class);
        fresh.process(); // new instance each time ✓
    }
}

Fix Option 2: @Lookup (Cleaner)

@Component
public abstract class MySingletonService {

    @Lookup
    public abstract MyPrototypeBean getPrototypeBean();

    public void doWork() {
        getPrototypeBean().process(); // Spring overrides this method
    }
}

@Lookup tells Spring to CGLIB-subclass this method and make it return a fresh prototype bean every time. Much cleaner than injecting the ApplicationContext.

💬 Interview Answer

"The stale prototype problem occurs when a prototype bean is injected into a singleton. Since the singleton is created once, the prototype instance is captured at that moment and never refreshed. The fixes are: calling applicationContext.getBean() at method invocation time, or using @Lookup which lets Spring override the method via CGLIB to return a fresh prototype on every call."

🧠 Quiz — Section 4
You have a Prototype-scoped bean injected into a Singleton via constructor injection. How many instances of the Prototype bean will exist after 100 requests?
✅ This is the classic stale prototype trap. When the Singleton is created, Spring injects exactly one Prototype instance into the constructor. That instance is held for the entire lifetime of the Singleton — you never get a fresh one. To get a new Prototype per call, use @Lookup or fetch from ApplicationContext.getBean().
Section 5 of 7 · Spring Core
IoC Container — BeanFactory vs ApplicationContext
🍅 Pomodoro · 25 min
🧒 ELI5 — Basic vs Premium Manager

BeanFactory is a basic manager: "You ask for something, I'll make it." Minimal, lazy, nothing extra.

ApplicationContext is the premium manager: same basics, but also does announcements (events), handles multiple languages (i18n), runs background tasks (AOP), and has everything set up before you arrive (eager initialization). Spring Boot uses ApplicationContext.

⚙️ BeanFactory

BeanFactory is the root interface — the minimal IoC container. It's lazy: beans are only created when explicitly requested via getBean(). You almost never use it directly in modern Spring.

BeanFactory factory = new XmlBeanFactory(new ClassPathResource("beans.xml"));
MyService svc = factory.getBean("myService", MyService.class);
// Bean created only NOW — not at startup
⚙️ ApplicationContext — What Spring Boot Uses

ApplicationContext extends BeanFactory and adds enterprise features:

  • Eager initialization — all Singleton beans are created at startup
  • Events — ApplicationEvent / @EventListener support
  • i18n — MessageSource for internationalization
  • AOP — BeanPostProcessor integration (proxies are created here)
  • Resource loading — classpath, filesystem, URL resources
  • Environment abstraction — @Value, profiles, property sources

Two Common Implementations

// Annotation-based (what Spring Boot auto-creates)
AnnotationConfigApplicationContext ctx =
    new AnnotationConfigApplicationContext(AppConfig.class);

// XML-based (legacy)
ClassPathXmlApplicationContext ctx =
    new ClassPathXmlApplicationContext("applicationContext.xml");
⚠️ Concurrency Gotcha — Singleton + Mutable State

Singleton beans are shared across ALL threads. If your @Service has mutable instance fields, you have a race condition:

@Service
public class BadService {
    private int counter = 0; // ← DANGER: shared mutable state

    public void increment() {
        counter++; // NOT thread-safe!
    }
}

Rule of thumb: Singleton beans should be stateless. If you need state, use Prototype scope, ThreadLocal, or synchronize properly.

⚙️ ApplicationContext Refresh Lifecycle

When does Spring instantiate beans? During the refresh() call:

Load Config
→
Parse Bean Defs
→
refresh()
→

After refresh() completes, all Singleton beans are ready. If any bean fails to initialize, refresh() throws and the app doesn't start — a key advantage of eager initialization.

💬 Interview Answer

"BeanFactory is the base IoC container — lazy, minimal, rarely used directly. ApplicationContext extends it with eager singleton initialization, event publishing, AOP integration, i18n, and profile/environment support. Spring Boot uses a AnnotationConfigServletWebServerApplicationContext. The key advantage of ApplicationContext's eager loading is fail-fast at startup — misconfigured beans are caught before serving any traffic."

🧠 Quiz — Section 5
Your Spring Boot app starts up and immediately crashes before handling any request. What's the most likely advantage this demonstrates over BeanFactory?
✅ This is the fail-fast advantage of ApplicationContext. BeanFactory is lazy — it only creates beans when requested, so a bad bean configuration would silently sit there until a request triggers getBean() and then crashes in production. ApplicationContext's eager initialization surfaces these problems at startup, which is far safer.

Bean Lifecycle — Every Phase
🍅 Pomodoro · 30 min
🧒 ELI5 — Hiring a New Employee

Think of a Spring bean as a new employee at your company:

  1. Hired (instantiation) — they show up
  2. Given their tools (DI) — here's your laptop, badge, desk
  3. Filled out HR forms (Aware callbacks) — "what's your name? what department?"
  4. Pre-training review (BeanPostProcessor.before) — HR checks their paperwork
  5. Orientation (@PostConstruct) — they run their own setup routine
  6. Final HR signoff (BeanPostProcessor.after) — AOP wraps them in a proxy vest 🪄
  7. Working — doing their job
  8. Exit interview (@PreDestroy) — cleanup before they leave

⚙️ Phase 5 & 8 — What You'll Actually Use
@Component
public class ConnectionPool {

    private DataSource dataSource;

    @PostConstruct // Phase 5 — runs after DI is complete
    public void init() {
        // Safe to use injected deps here
        dataSource.setMaxPoolSize(20);
        System.out.println("Pool initialized");
    }

    @PreDestroy // Phase 8 — runs before context shuts down
    public void cleanup() {
        dataSource.close(); // release connections
        System.out.println("Pool closed");
    }
}

Why @PostConstruct vs constructor?

At constructor time, DI hasn't happened yet — injected fields are still null. @PostConstruct runs after Spring has set all dependencies, so it's safe to use them there.

⚙️ Phase 6 — BeanPostProcessor & AOP Proxies

This phase is where the magic happens. postProcessAfterInitialization() is called for every bean, and Spring's AOP infrastructure uses this hook to wrap your bean in a dynamic proxy.

// This is conceptually what Spring does with @Transactional:
// Your real bean ↓
MyService real = new MyService();
// Spring wraps it in a proxy during postProcessAfterInitialization:
MyService proxy = Proxy.create(real, new TransactionInterceptor());
// The ApplicationContext stores the PROXY, not the real bean

This is why calling a @Transactional method from within the same class doesn't work — you're calling on this (the real object), bypassing the proxy.

@Conditional & @ConditionalOnProperty

Before a bean is even instantiated, Spring evaluates @Conditional annotations to decide whether to register the bean definition at all. This is the backbone of Spring Boot's autoconfiguration:

@Configuration
@ConditionalOnProperty(name = "feature.cache.enabled", havingValue = "true")
public class CacheConfig {
    // Only loaded if feature.cache.enabled=true in properties
}
💬 Interview Answer

"The bean lifecycle has 8 phases: instantiation, DI, Aware callbacks, BeanPostProcessor.before, @PostConstruct/afterPropertiesSet, BeanPostProcessor.after — where AOP proxies are created — bean ready for use, and @PreDestroy on shutdown. Phase 6 is critical: that's where @Transactional, @Cacheable, and other AOP proxies are woven in. This is also why self-invocation of @Transactional methods doesn't work — you bypass the proxy."

🧠 Quiz — Section 6
At which lifecycle phase are AOP proxies (like @Transactional) created?
✅ AOP proxies are created in Phase 6 — postProcessAfterInitialization(). Spring's AbstractAutoProxyCreator implements BeanPostProcessor and in this method, it checks if the bean needs proxying (due to @Transactional, @Cacheable, etc.) and wraps it. The ApplicationContext then stores the proxy reference, not the original bean.
Section 7 of 7 · Spring Core
@SpringBootApplication + CGLIB
🍅 Pomodoro · 25 min
🧒 ELI5 — The Master Key

@SpringBootApplication is like a single master key that actually unlocks three doors: the configuration door (tell Spring your settings), the scanning door (go find all our staff), and the autoconfiguration door (set up all standard equipment automatically). One annotation, three jobs.

And CGLIB? Imagine you hired a stunt double for your configuration class. When someone calls your dataSource() method twice, the stunt double intercepts and says "hold on, we already have one of those" and returns the cached copy instead of making a new one. That's CGLIB proxying.

⚙️ @SpringBootApplication Unpacked
// @SpringBootApplication is literally a composite of:
@Configuration      // Marks this class as a bean definition source
@EnableAutoConfiguration  // Activates Spring Boot's auto-config
@ComponentScan      // Scans this package and sub-packages for @Component
public class MyApp {
    public static void main(String[] args) {
        SpringApplication.run(MyApp.class, args);
    }
}

@EnableAutoConfiguration — How it Works

This tells Spring Boot to read from:

  • Modern (Boot 2.7+): META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
  • Legacy: META-INF/spring.factories

These files list hundreds of @Configuration classes. Spring Boot loads them all, but each one is guarded by @ConditionalOnClass, @ConditionalOnProperty, etc. — so only what's relevant to your classpath gets activated.

// SpringFactoriesLoader reads files like:
// org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
//   ...DataSourceAutoConfiguration,\
//   ...JpaRepositoriesAutoConfiguration,\
//   ...WebMvcAutoConfiguration,\
//   ... (100+ more)
⚙️ CGLIB Proxying of @Configuration

When you annotate a class with @Configuration, Spring uses CGLIB (Code Generation Library) to subclass it at runtime and intercept all @Bean method calls.

@Configuration
public class AppConfig {

    @Bean
    public DataSource dataSource() {
        return new HikariDataSource(); // creates a new instance
    }

    @Bean
    public UserRepository userRepository() {
        return new UserRepository(dataSource()); // calls dataSource()
    }

    @Bean
    public OrderRepository orderRepository() {
        return new OrderRepository(dataSource()); // calls dataSource() again
    }
}

You'd expect dataSource() called twice to create two HikariDataSource instances. But CGLIB intercepts the call and returns the same singleton from the ApplicationContext both times. ✅

⚠️ The Interview Trap — @Component + @Bean

If you put @Bean methods inside a @Component class (not @Configuration), CGLIB does NOT subclass it. Each direct call to dataSource() creates a brand new instance:

@Component // ← NOT @Configuration — no CGLIB proxy!
public class LiteConfig {

    @Bean
    public DataSource dataSource() {
        return new HikariDataSource();
    }

    @Bean
    public UserRepository userRepository() {
        // Creates a NEW DataSource — NOT the singleton one!
        return new UserRepository(dataSource());
    }
}

This is called "Lite Mode" — @Bean methods in @Component classes are not intercepted. Cross-references between @Bean methods will create new instances instead of reusing singletons.

Rule: If your @Bean methods call each other — always use @Configuration.

proxyBeanMethods=false — When Would You Use It?

@Configuration(proxyBeanMethods = false)
public class FastConfig { ... }

Setting proxyBeanMethods=false disables CGLIB proxying — the class runs in Lite Mode. Use this when:

  • Your @Bean methods never call each other (no cross-references)
  • You want faster startup (CGLIB subclassing adds overhead)
  • You're building a library or autoconfiguration class

Spring Boot's own autoconfiguration classes use proxyBeanMethods=false for exactly this reason — they're designed to never cross-reference beans.

💬 Interview Answer

"@SpringBootApplication is a meta-annotation combining @Configuration, @EnableAutoConfiguration, and @ComponentScan. @EnableAutoConfiguration uses SpringFactoriesLoader to discover and conditionally load auto-configuration classes. CGLIB proxies @Configuration classes so that calling a @Bean method returns the same singleton from the context — without CGLIB (i.e., @Component with @Bean), each call creates a new instance. proxyBeanMethods=false opts into Lite Mode — useful for autoconfiguration classes that have no inter-@Bean dependencies, giving faster startup."

🧠 Quiz — Section 7
You have a @Component class with two @Bean methods where one calls the other. What happens at runtime?
✅ This is the classic CGLIB interview trap. In Lite Mode (@Component with @Bean), Spring does NOT subclass the config class via CGLIB. Method calls execute as regular Java — each call to dataSource() runs the method body and creates a fresh object. You end up with multiple instances of what should be a singleton. Always use @Configuration when your @Bean methods have dependencies on each other.
