Section 1 · Core concept

@Transactional Basics

Learn what a transaction boundary actually does, why Spring implements it through proxies, and why the annotation belongs on business operations instead of being sprinkled randomly.

🧒 ELI5

One shopping trip, one receipt

Imagine you are at a grocery store with a single receipt for the whole trip. If the cashier scans all your items and then the card machine fails at the very end, you do not want half the groceries paid for and half left hanging in the system.

The store either finishes the whole checkout or cancels the whole thing, because partial completion creates confusion for both the customer and the store. That is the core idea here: treat related work as one all-or-nothing unit.

It matters because business actions like transferring money, placing an order, or reserving inventory stop making sense when only part of the work succeeds. The “single receipt” mirrors the single success-or-fail boundary in your application.

🔗 ANALOGY

A movie shoot with one director call

Think of a film scene where lighting, camera, actors, and sound all have to start together when the director says “action.” If only some people follow the call, the take is useless even though pieces of work happened.

This concept exists so one coordinating boundary can decide whether the full scene is usable. If you ignore that boundary, you end up debugging ghost bugs where one database change is visible but its matching change is missing.

Technical

What it means under the hood

Definition: @Transactional tells Spring to open a database transaction before the method, commit on success, and roll back on failure according to configured rules.

Spring usually applies the annotation by wrapping your bean in a proxy. When another bean calls that proxied method, the proxy starts the transaction, delegates to your method, then commits or rolls back afterward.

Inside the transaction, Hibernate or JDBC work is attached to the same thread-bound transaction context. That is why multiple repository calls can behave like one logical unit even if they are separate Java method calls.

The mental model is not “annotation magic.” The mental model is “a proxy surrounds my method with begin / commit / rollback behavior.” Once you see that, self-invocation bugs suddenly make sense.

Why it matters in production

  • Prevents half-written business operations
  • Coordinates repository calls into one unit
  • Keeps consistency rules close to service-layer logic

Common smell

  • Annotating private methods and expecting it to work
  • Calling a transactional method from the same class
  • Keeping the transaction open during slow remote calls
Atomic writesProxy boundarySelf-invocation
Client calls service→Proxy opens transaction→Repositories participate→Commit or rollback

💬 Model Interview Answer

“I think of @Transactional as a service-layer consistency boundary. Spring usually enforces it with a proxy, so it only works when the call crosses the proxy. The common gotcha is self-invocation: a method inside the same bean can skip the proxy entirely, which means no transaction ever starts.”

@Service
public class TransferService {

    @Autowired
    private AccountRepository accountRepository;

    public void moveMoney(Long fromId, Long toId, BigDecimal amount) {
        // Self-invocation: this call does not go through Spring's proxy
        saveTransfer(fromId, toId, amount);

    @Transactional

        accountRepository.debit(fromId, amount);
        accountRepository.credit(toId, amount);

✅ Right: public service boundary

    public TransferService(AccountRepository accountRepository) {
        this.accountRepository = accountRepository;
    }

    @Transactional
    public void moveMoney(Long fromId, Long toId, BigDecimal amount) {
        // Both statements now run inside the same transaction
        accountRepository.debit(fromId, amount);
        accountRepository.credit(toId, amount);
    }
}

Put the annotation on a public service method that represents one business action. Keep the transaction short and centered on database work, not on slow network chatter.

The safest mental model is that Spring gives you a framework feature only when your call crosses the Spring-managed boundary and the surrounding infrastructure can participate.

Warning

Self-invocation bypasses the proxy

The most misunderstood part of Spring transactions is that the annotation is not woven directly into your method body. It is usually applied by a separate proxy object placed in front of your bean.

When one method in the same class calls another, that internal call does not leave the object, so the proxy never gets a chance to start the transaction. The fix is to move the boundary to the externally called public method, or split behavior into another bean.

Broken pattern
@Service
public class ReportService {

    public void generateDailyReport() {
        // Looks fine, but no proxy is involved here
        loadAndStore();
    }

    @Transactional
    public void loadAndStore() {
        // No transaction actually starts for the internal call above
    }
}
Fixed pattern
@Service
public class ReportWriter {

    @Transactional
    public void loadAndStore() {
        // Called from another bean, so the proxy can open the transaction
    }
}

@Service
public class ReportService {

    private final ReportWriter reportWriter;

    public ReportService(ReportWriter reportWriter) {
        this.reportWriter = reportWriter;
    }

    public void generateDailyReport() {
        reportWriter.loadAndStore();
    }
}
Mini takeaway
  • A transaction is a business boundary, not decoration
  • Proxy crossing matters as much as the annotation itself
  • Short transactions are safer than long “do everything” transactions
Quiz

Why might @Transactional appear to do nothing on a method?

Correct answer: BSpring starts most transactions through a proxy, not by rewriting your method body. Option B is correct because a same-class call can bypass that proxy and therefore skip the begin/commit/rollback logic entirely. Option A is false because service-layer methods are the normal place for transactions, option C is false because Spring transactions are not limited to JPA, and option D is false because checked exceptions affect rollback rules, not whether the transaction infrastructure starts at all.
Section 2 · Control flow

Propagation

Propagation decides whether a method joins an existing transaction or starts a brand new one. It becomes important when one business action calls another and you need to control how failure should spread.

🧒 ELI5

Joining a dinner bill or starting your own tab

Picture friends at a restaurant. Sometimes you join the group bill because the whole dinner is one shared activity. Other times you open a separate tab because your dessert should not fail just because someone else’s card gets declined.

That choice changes how mistakes are handled. If everyone is on one bill, one problem can block the whole checkout. If you split tabs, one person can finish even if the rest of the table has trouble.

Propagation is that choice for method calls inside your application: join the current unit of work or carve out a separate one with different consequences.

🔗 ANALOGY

Shipping boxes inside a larger shipment

A warehouse may ship one large order in a single truck, but it can also put a sensitive item in a separate insulated box with its own handling rules. That extra box exists because not every piece of work should share the exact same failure boundary.

If you ignore propagation and default everything blindly, you can accidentally make audit logging roll back with business data, or worse, commit side effects that should have stayed coupled to the main action.

Technical

How propagation really works

Definition: Propagation tells Spring what to do when a transactional method is called while another transaction may already be active.

REQUIRED is the default: join the current transaction if one exists, otherwise create a new one. REQUIRES_NEW suspends the current transaction and starts another independent one.

SUPPORTS, NOT_SUPPORTED, NEVER, and MANDATORY are not interview trivia; they are rules about whether code should require, reuse, avoid, or reject an ambient transaction context.

The senior-engineer mental model is: “When my call stack gets deeper, which operations must succeed or fail together, and which ones deserve their own commit boundary?”

Why it matters in production

  • Lets you separate audit work from main writes
  • Prevents accidental coupling between unrelated side effects
  • Makes nested service calls predictable

Common smell

  • Using REQUIRES_NEW everywhere to “be safe”
  • Assuming REQUIRED always means “new transaction”
  • Forgetting suspended transactions still resume later
REQUIRED defaultSuspend/resumeSeparate audit writes
Enter outer service→Check current transaction→Join or suspend/create→Commit according to each boundary

💬 Model Interview Answer

“Propagation is about how nested service calls participate in transactions. REQUIRED joins the current one, while REQUIRES_NEW creates an independent boundary and suspends the old one. The trade-off is that REQUIRES_NEW can keep audit or notification work durable, but it can also hide failures if you use it carelessly.”

ScenarioWhat to watch
REQUIREDJoin existing or create one
REQUIRES_NEWAlways independent commit/rollback
MANDATORYFail fast if no transaction exists
❌ Wrong: audit log tied to outer outcome by accident

public class OrderService {

    private final AuditService auditService;

    public OrderService(AuditService auditService, OrderRepository orderRepository) {
        this.auditService = auditService;
        this.orderRepository = orderRepository;
    }

    @Transactional
    public void placeOrder(Order order) {
        orderRepository.save(order);
        auditService.record("ORDER_CREATED"); // Uses REQUIRED by default
        throw new IllegalStateException("Payment failed");

public class AuditService {

        // Rolls back with the outer transaction

}
✅ Right: independent audit commit when appropriate
@Service
public class AuditService {

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void record(String eventName) {
        // This can commit even if the caller later rolls back
    }
}

Use separate propagation only when the business requirement is explicit. Audit trails, idempotency records, or failure diagnostics are common reasons; routine domain writes usually belong in the same REQUIRED boundary.

The safest mental model is that Spring gives you a framework feature only when your call crosses the Spring-managed boundary and the surrounding infrastructure can participate.

Warning

REQUIRES_NEW can hide design mistakes

Developers sometimes reach for REQUIRES_NEW because it “fixes” rollback behavior quickly. The danger is that it can commit records that no longer match the final business outcome.

If the outer operation fails, your independent inner transaction still commits. That is correct only when the business genuinely wants independent durability, such as an audit trail or outbox entry.

Broken pattern
@Transactional
public void registerUser(User user) {
    userRepository.save(user);
    emailPreferenceService.saveDefaults(user.getId()); // REQUIRES_NEW added to "fix" a bug
    throw new RuntimeException("validation later failed");
}
Fixed pattern
@Transactional
public void registerUser(User user) {
    userRepository.save(user);
    emailPreferenceService.saveDefaults(user.getId()); // REQUIRED: stays coupled to user creation
}

@Transactional(propagation = Propagation.REQUIRES_NEW)
public void writeAuditEvent(String message) {
    auditRepository.save(new AuditEvent(message));
}
Mini takeaway
  • REQUIRED means “same unit of work when possible”
  • REQUIRES_NEW is for intentionally independent durability
  • Choose propagation from business semantics, not from debugging frustration
Quiz

Which propagation always creates a completely new transaction boundary?

Correct answer: CREQUIRES_NEW is correct because Spring suspends any current transaction and starts a separate one for that method. REQUIRED may create a transaction only when none exists, SUPPORTS simply joins if available and otherwise runs without one, and MANDATORY does not create anything at all—it throws if there is no ambient transaction. In interviews, emphasize the business consequence: REQUIRES_NEW can commit even if the outer operation later rolls back.
Section 3 · Consistency rules

Isolation Levels

Isolation levels control how one transaction can see changes made by another. They are about visibility and interference, not about whether a transaction exists in the first place.

🧒 ELI5

Reading a whiteboard while someone else keeps erasing and rewriting it

Imagine you are copying a schedule from a whiteboard while another person keeps changing times in the middle of your note-taking. If you see half-finished edits, your copy can become nonsense even though you worked carefully.

Some workplaces solve that by saying nobody reads the board until an edit is fully finished. Stronger rules might even stop new items from being added while you are reviewing the whole schedule.

Isolation levels are those rules for concurrent work. They decide how much unfinished or newly arrived data your transaction is allowed to see.

🔗 ANALOGY

A library with different reading-room rules

One library might let staff rearrange books while readers are browsing shelves. Another might require readers to wait until reshelving finishes. A stricter room may even freeze the shelf layout until a catalog audit completes.

These rules exist because concurrency is a trade-off. If you ignore isolation, you get weird anomalies. If you crank it to the maximum everywhere, throughput drops because more work waits on more locks.

Technical

How isolation affects visibility

Definition: Isolation level defines which read anomalies a transaction may experience when other transactions are concurrently updating related data.

READ_COMMITTED prevents dirty reads, meaning you cannot observe another transaction’s uncommitted changes. REPEATABLE_READ and SERIALIZABLE add stronger guarantees, but at higher locking or coordination cost.

The database, not Spring alone, ultimately enforces the isolation semantics. Spring passes the requested level to the transaction manager, and the underlying database interprets it according to its own engine behavior.

The mental model is: “What inconsistent view can this reader tolerate while others are writing?” That answer should come from the use case, not from cargo-cult defaults.

Why it matters in production

  • Prevents subtle concurrency bugs in money, inventory, and reporting flows
  • Lets you balance correctness and throughput explicitly
  • Makes anomalies discussable in design reviews

Common smell

  • Setting SERIALIZABLE everywhere out of fear
  • Assuming databases implement every level identically
  • Confusing isolation with optimistic locking
Correctness vs throughputDB-enforced semanticsVendor differences
Start transaction→Read rows→Other transaction writes→Isolation decides what becomes visible

💬 Model Interview Answer

“Isolation levels answer a concurrency question: what can this transaction see while others are changing data? READ_COMMITTED blocks dirty reads, stronger levels reduce more anomalies, and the trade-off is reduced concurrency. In practice I pick the weakest level that still protects the business invariant, then I combine it with locking or optimistic version checks where needed.”

ScenarioWhat to watch
READ_COMMITTEDNo dirty reads
REPEATABLE_READMore stable repeated reads

❌ Wrong: assume repeated reads are automatically stable
@Transactional(isolation = Isolation.READ_COMMITTED)

✅ Right: choose the guarantee or lock explicitly

    Product freshProduct = productRepository.findById(productId).orElseThrow();
    return new InventorySnapshot(product.getStock(), freshProduct.getStock());
}

Isolation is about visibility, but business correctness often still needs row locking, unique constraints, or optimistic version columns. Do not expect one switch to solve every concurrency problem.

The safest mental model is that Spring gives you a framework feature only when your call crosses the Spring-managed boundary and the surrounding infrastructure can participate.

Warning

Higher isolation has a cost

A stronger isolation level can reduce anomalies, but it may also increase lock contention, retries, and latency under load. That is why “use the strongest level” is not a mature answer.

The right fix may be narrower: lock one row, add a uniqueness constraint, or use optimistic locking instead of globally tightening the whole transaction.

Broken pattern
@Transactional(isolation = Isolation.SERIALIZABLE)
public void generateDailyAnalytics() {
    // Heavy long-running reporting query under the strictest level
    // Can throttle other writers and create unnecessary contention
}
Fixed pattern
@Transactional(readOnly = true, isolation = Isolation.READ_COMMITTED)
public ReportPage fetchReportPage() {
    // Use the weakest level that still serves the reporting use case
    // Apply explicit locking only where a business invariant truly needs it
}
Mini takeaway
  • Isolation is about what reads can observe under concurrency
  • Database engines ultimately enforce the semantics
  • Strongest isolation is not always the best engineering choice

Section 4 · Failure behavior

Rollback Rules & Timeouts

A transaction boundary is only safe if you also understand what actually triggers rollback and how long the boundary is allowed to stay open. Defaults help, but defaults are not the same as explicit design.

🧒 ELI5

Undo rules on a board game timer

Imagine a long board game where some mistakes let you rewind the turn, but other mistakes do not because the rules treat them differently. Now add a chess clock: if you hold the table too long, your turn must end before everyone else gets blocked.

That combination is what rollback rules and timeouts do. They decide which kinds of failure undo the work and how long the system will tolerate a transaction holding resources.

Without clear rules, you think your work will be undone when it will not, or you leave database locks open so long that unrelated requests slow down behind you.

🔗 ANALOGY

A refund policy plus room reservation timer

A hotel may allow a full refund for some cancellation reasons but not others. At the same time, it cannot let a guest hold a room forever without paying, so there is also an expiration timer.

Rollback rules exist because not all exceptions mean the same thing, and timeouts exist because correctness without throughput is still a production failure. Ignoring either one leads to systems that are surprising under load.

Technical

What Spring does by default

Definition: Spring rolls back by default for unchecked exceptions and errors, while checked exceptions require explicit rollback configuration if you want the transaction to undo work.

That default surprises many engineers because business code often throws checked exceptions such as IOException or custom checked domain exceptions. If you expect rollback for those, use rollbackFor explicitly.

Timeouts protect the system from transactions that sit open too long, holding locks or connection-pool resources. They are especially important when complex workflows accidentally mix database work with slow operations.

The mature mental model is: “What failures should undo this work, and what is the maximum safe time I want this transaction to monopolize resources?”

Why it matters in production

  • Makes rollback behavior explicit instead of accidental
  • Protects the database and pool from long-running work
  • Prevents “it committed even though there was an error” surprises

Common smell

  • Catching exceptions and swallowing them inside the transaction
  • Relying on checked exceptions without rollbackFor
  • Performing remote API calls before commit with no timeout plan
Explicit rollbackForTimeoutsSwallowed exceptions
Method throws→Spring checks rollback rules→Transaction commits or rolls back→Resources are released

💬 Model Interview Answer

“By default Spring rolls back for RuntimeException and Error, not for checked exceptions. So if my domain flow throws a checked exception that should cancel the write, I configure rollbackFor explicitly. I also keep a timeout on important write paths because an open transaction is holding scarce resources even while my code is waiting.”

ScenarioWhat to watch
RuntimeExceptionRollback by default

❌ Wrong: checked exception but no rollback rule

✅ Right: explicit rollback rule and timeout

    importLogRepository.markFinished(path.toString());

Defaults are useful, but they are not a substitute for business intent. Whenever the failure type matters to correctness, encode that intent directly on the transaction boundary.

The safest mental model is that Spring gives you a framework feature only when your call crosses the Spring-managed boundary and the surrounding infrastructure can participate.

Warning

Catching an exception can accidentally force a commit

Spring can only react to an exception that escapes the method or to a manually marked rollback-only status. If you catch the exception, log it, and keep going, the framework may happily commit.

That is why “catch and continue” inside a transactional method is dangerous unless you also decide whether the transaction should be marked for rollback.

Broken pattern
@Transactional
public void reserveSeat(Long seatId) {
    try {
        seatRepository.lockAndReserve(seatId);
        paymentGateway.charge(seatId); // Assume this throws RuntimeException
    } catch (RuntimeException ex) {
        log.warn("Payment failed", ex);
        // Method returns normally, so the transaction may commit seat reservation
    }
}
Fixed pattern
@Transactional
public void reserveSeat(Long seatId) {
    try {
        seatRepository.lockAndReserve(seatId);
        paymentGateway.charge(seatId);
    } catch (RuntimeException ex) {
        TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
        throw ex;
    }
}
Mini takeaway
  • Rollback defaults cover unchecked exceptions only
  • Timeouts are a resource-protection feature, not just a nice-to-have
  • Never swallow an exception unless you also decide the transaction outcome
Quiz

Which exception category triggers rollback by default in Spring transactions?

Correct answer: BRuntimeException is the correct answer because Spring’s default rule rolls back on unchecked exceptions and Error. Checked exceptions such as IOException do not trigger rollback unless you configure rollbackFor, so A and C are wrong. D is also wrong because Spring does not roll back for every possible Exception type by default. In interviews, say the default aloud—many candidates vaguely remember “exceptions roll back” without knowing the checked-vs-unchecked nuance.
Section 5 · Interview deep dive

Deep Dive: Propagation & Rollback Rules

This section puts the pieces together: nested service calls, explicit rollback semantics, and the difference between domain writes that must stay coupled versus side effects that can commit independently.

🧒 ELI5

One wedding planner, many vendors

Imagine a wedding planner coordinating venue, food, music, and photography. Some work must stay together: if the venue booking fails, the main event should not be considered fully arranged.

But some records still need to survive failure, like the planner’s own notes about what went wrong or a message sent to the support team. That is the same tension engineers manage with nested transactions and rollback rules.

The hard part is not knowing the words REQUIRED and REQUIRES_NEW. The hard part is deciding which pieces of work really belong to the same promise to the user.

🔗 ANALOGY

A main suitcase with small sealed pouches

When you pack for a long trip, most clothes belong in the main suitcase because they are part of one trip outcome. But medicine or documents might go into sealed pouches because losing the big bag should not destroy every critical item.

Propagation exists for that packaging decision. If you ignore it, you either over-couple independent work or scatter commits so widely that your database no longer reflects one believable business story.

Technical

How senior engineers reason about nested boundaries

Definition: The deep-dive question is not “Which propagation type exists?” but “Which data must share one atomic story, and which records are allowed to live outside it?”

An outer service usually represents the user-visible business action. Inner services should normally join that same transaction with REQUIRED unless you have a very explicit reason to separate durability.

REQUIRES_NEW is appropriate for independent audit records, outbox messages, retry markers, or operational diagnostics. It is dangerous when used to mask modeling mistakes, because the database may then reflect multiple inconsistent stories about the same event.

Rollback rules belong to that same conversation. If an inner service catches a checked or unchecked exception and hides it, the outer layer may commit data that should have been abandoned.

What senior engineers optimize for

  • Start from business invariants, not framework defaults
  • Use independent transactions only for explicitly independent records
  • Keep transaction boundaries around database work, not around network waiting

Trade-offs

  • Too much REQUIRES_NEW creates fragmented outcomes
  • Too much catching/logging hides rollback intent
  • Long boundaries multiply lock contention and retry pain
Invariant-driven designOutbox/audit candidatesFragmented commits
Outer service begins→Inner services join or split→Failure occurs→Rollback rules decide final story

💬 Model Interview Answer

“My rule is that the outer service method should model the user’s main business action, and inner methods should usually join it with REQUIRED. I reserve REQUIRES_NEW for deliberately independent records like audit or outbox writes. The gotcha is that catching exceptions or overusing new transactions can leave the database in a state that is technically committed but semantically inconsistent.”

Warning

Catching exceptions without deciding the final transaction outcome

If an inner REQUIRED method marks the transaction rollback-only, the outer method may later hit UnexpectedRollbackException when it tries to commit. That is not Spring being weird; it is Spring protecting consistency after a deeper layer already decided the unit of work must fail.

Developers often swallow an exception “just for logging” and accidentally convert a rollback into a commit. If the business action should fail, let the exception escape or mark the transaction rollback-only explicitly.

Broken pattern
@Transactional
public void checkout(Order order) {
    try {
        paymentService.charge(order);
    } catch (RuntimeException ex) {
        log.warn("charge failed", ex);
        // execution continues, commit may still happen
    }
    orderRepository.markPaid(order.getId());
}
Fixed pattern
@Transactional
public void checkout(Order order) {
    paymentService.charge(order);
    orderRepository.markPaid(order.getId());
}
Edge cases

Edge Cases & Tradeoffs

  • UnexpectedRollbackException usually means some inner layer already marked rollback-only.
  • saveAndFlush sends SQL early, but commit still happens only at transaction end.
  • Constraints and optimistic locking still matter alongside propagation.
  • Long-running remote calls inside transactions amplify lock contention.
  • An outbox table is often better than one giant cross-system transaction.
Checklist

Production Checklist

  • Define the outermost business action first
  • Default inner domain writes to REQUIRED
  • Use REQUIRES_NEW only for intentionally independent durability
  • Set rollbackFor on checked exceptions that should cancel writes
  • Keep remote I/O outside the main database transaction when possible
Interview

Interview Hooks

  • Be ready to explain self-invocation and proxy boundaries
  • Know why checked exceptions do not roll back by default
  • Describe one justified REQUIRES_NEW use and one smell

Advanced order + outbox pattern example

Advanced example
@Service
public class OrderPlacementService {

    private final OrderRepository orderRepository;
    private final OutboxService outboxService;

    public OrderPlacementService(OrderRepository orderRepository, OutboxService outboxService) {
        this.orderRepository = orderRepository;
        this.outboxService = outboxService;
    }

    @Transactional
    public Long place(Order order) {
        Order saved = orderRepository.save(order);
        outboxService.enqueueOrderCreated(saved.getId());
        return saved.getId();
    }
}

@Service
public class OutboxService {

    private final OutboxRepository outboxRepository;

    public OutboxService(OutboxRepository outboxRepository) {
        this.outboxRepository = outboxRepository;
    }

    @Transactional(propagation = Propagation.REQUIRED)
    public void enqueueOrderCreated(Long orderId) {
        outboxRepository.save(new OutboxEvent("ORDER_CREATED", orderId.toString()));
    }
}

  • Transaction: one believable business story in the database.
  • Propagation: decide whether nested work shares that story.
  • Isolation: decide what concurrent edits this story is allowed to see.
  • Rollback: make failure semantics explicit, not assumed.
