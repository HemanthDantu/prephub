@Transactional Basics

Spring transactions are proxy-based and wrap method calls with begin/commit/rollback.

🧒 ELI5

A safe envelope

You put multiple actions into one sealed envelope. Either all go through or none do.

🔗 ANALOGY

Bank transfer ledger

The bank records all steps together so money never disappears mid-transfer.

Technical

What it means in real code

Definition: @Transactional creates a transaction boundary with commit or rollback.

@Transactional defines a unit of work that either commits or rolls back as a whole. It is usually applied at the service layer where business rules live.

Because it is proxy-based, calls inside the same class bypass the proxy. That is why self-invocation does not start a transaction.

A simple mental model is that a transaction is a safety boundary: either everything succeeds, or everything is undone.

Why it matters

  • Keeps data consistent
  • Groups multiple DB operations
  • Supports rollback on failure

Common smell

  • Transactional methods called internally
  • Missing @Transactional on write paths
  • Long transactions
Clear boundariesCommit/rollbackSelf-invocation
Begin→Do work→Commit→Rollback on error

Interview answer

“@Transactional wraps a method in a transaction boundary; it’s proxy-based, so self-invocation won’t apply.”

ScenarioEffect
Runtime exceptionRollback
Checked exceptionNo rollback
Self-callNo proxy
@Transactional usage
@Transactional
public void transfer(){
  debit();
  credit();
}

Place @Transactional on public service methods called from outside the bean.

Transactions are enforced via proxies, so propagation and rollback rules only apply when calls cross the proxy boundary. Always call transactional methods through Spring-managed beans, and be explicit about checked-exception rollback behavior.

Warning

Self-invocation bypasses proxy

Calling a transactional method inside the same class won’t start a transaction.

Mini takeaway
  • Transactions are proxy-based.
  • Define clear boundaries.
  • Handle failures.

Section 2 of 5 · ~7 mins

Propagation

Propagation defines how a method behaves if a transaction already exists.

🧒 ELI5

Joining or starting a new game

Sometimes you join an existing game; sometimes you start a new one.

🔗 ANALOGY

Nested receipts

You can add items to the same receipt or open a new receipt for a separate purchase.

Technical

What it means in real code

Definition: REQUIRED joins existing; REQUIRES_NEW suspends and starts new; NESTED uses savepoints.

Propagation determines how nested calls behave. REQUIRED joins an existing transaction, while REQUIRES_NEW suspends it and starts a new one.

REQUIRES_NEW is useful for side effects like audit logs that must persist even if the main transaction fails.

NESTED can be used for partial rollbacks with savepoints, but it depends on database support.

Why it matters

  • Control transaction boundaries
  • Isolate side effects
  • Support audit logging

Common smell

  • Using REQUIRES_NEW everywhere
  • No understanding of join behavior
  • Unexpected rollbacks
REQUIRED defaultREQUIRES_NEW isolateNESTED savepoint

Interview answer

“REQUIRED joins an existing transaction, REQUIRES_NEW starts a new one for isolated work.”

Propagation example

Use REQUIRES_NEW for side effects that must commit independently.

As you study this, focus on what Spring/Java is doing on your behalf versus what you must configure explicitly. That mental split makes the API feel predictable instead of magical.

Warning

REQUIRES_NEW can hide failures

If used everywhere, it can mask rollback behavior and create inconsistencies.

Mini takeaway
  • REQUIRED is default.
  • REQUIRES_NEW isolates work.
  • Use propagation intentionally.

Section 3 of 5 · ~7 mins

Isolation Levels

Isolation controls how concurrent transactions see each other’s data.

🧒 ELI5

Reading a book while someone edits it

Do you want to see edits live or only after the author finishes? That’s isolation.

🔗 ANALOGY

Shared document

Strong isolation is like getting your own snapshot copy while others edit.

Technical

What it means in real code

Definition: Isolation prevents anomalies like dirty reads, non-repeatable reads, and phantoms.

Isolation levels trade performance for consistency. Higher isolation prevents anomalies but reduces concurrency due to locking.

Pick the lowest isolation that still meets correctness so you avoid unnecessary contention in busy systems.

If you do not know which to pick, start with READ_COMMITTED and adjust only when you observe anomalies.

Why it matters

  • Correctness under concurrency
  • Predictable reads
  • Protect invariants

Common smell

  • Using SERIALIZABLE everywhere
  • Ignoring anomalies
  • Deadlocks from over-isolation

Section 4 of 5 · ~7 mins

Rollback Rules & Timeouts

Rollback behavior defaults to unchecked exceptions, but you can override it.

🧒 ELI5

Undoing a move

If the move fails, you undo it. Some mistakes are considered serious enough to undo automatically.

🔗 ANALOGY

Refund policy

Certain failures trigger automatic refunds; others require manual review.

Technical

What it means in real code

Definition: By default, Spring rolls back on RuntimeException; use rollbackFor for checked exceptions.

By default, only unchecked exceptions trigger rollback. If you expect checked exceptions to roll back, specify rollbackFor explicitly.

Timeouts and readOnly flags make transactional behavior predictable and help avoid long-running locks.

It is also common to split long operations so you avoid holding locks for the entire workflow.

Why it matters

  • Ensure data integrity
  • Control business exceptions
  • Avoid partial writes

Common smell

  • Checked exception without rollbackFor
  • No timeout on long tx
  • Swallowing errors
rollbackFortimeoutSilent failures

“Spring rolls back on unchecked exceptions. I use rollbackFor for checked exceptions that should undo work.”

As you study this, focus on what Spring/Java is doing on your behalf versus what you must configure explicitly. That mental split makes the API feel predictable instead of magical.

Warning

Unchecked isn’t always enough

If a checked exception should roll back, you must declare it.

Mini takeaway
  • Rollback defaults to runtime exceptions.
  • Use rollbackFor when needed.
  • Set timeouts.

Deep Dive: Propagation & Rollback Rules

Understanding propagation and rollback behavior prevents subtle data bugs.

🧒 ELI5

Shared piggy bank

If everyone uses the same bank, you need clear rules about when money goes in or out.

🔗 ANALOGY

Nested boxes

A box inside a box may open with the outer one or separately depending on the rule.

Technical

What it means in real code

Definition: @Transactional propagation decides whether methods join existing transactions or start new ones.

Why it matters

  • Avoids partial updates
  • Controls rollback behavior
  • Improves data consistency

Common pitfall

  • Unexpected commits
  • Catching exceptions and swallowing rollback
  • Self-invocation

Warning

Catching exceptions can commit

If you catch and swallow exceptions, Spring may commit instead of rolling back.

Mini takeaway
  • Know propagation types.
  • Configure rollback rules explicitly.
  • Avoid self-invocation issues.
Edge cases

Edge Cases & Tradeoffs

  • Self-invocation bypasses @Transactional.
  • Checked exceptions do not trigger rollback by default.
  • Long transactions hold locks and reduce throughput.
  • Isolation level mismatch causes anomalies.
  • Nested transactions need explicit propagation.
Checklist

Production Checklist

  • Annotate service-layer methods, not controllers.
  • Pick propagation that matches the use case.
  • Set timeouts for long-running work.
  • Use readOnly for query-only paths.
  • Document rollback rules explicitly.
