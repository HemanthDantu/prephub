Section 1 · Core concept

IoC & Dependency Injection

Dependency Injection is Spring’s way of constructing objects and wiring collaborators from the outside, so your classes focus on business logic instead of manual object creation.

🧒 ELI5

A restaurant kitchen where chefs do not buy their own stoves

Imagine every chef in a restaurant being told to leave cooking and run out to buy a stove, pans, knives, and ingredients before starting dinner service. The kitchen would be chaotic because each person would build their own little world instead of using the shared setup.

A well-run kitchen works the opposite way: the restaurant provides the equipment and ingredients, then chefs just do their actual jobs. That is what this concept does for code—objects receive what they need instead of constructing everything themselves.

It matters because when classes build their own dependencies, testing gets painful, configuration gets duplicated, and swapping implementations becomes expensive.

🔗 ANALOGY

A movie cast assembled by a production team

Actors do not personally hire the camera crew, lighting team, and editor every morning. A production team assembles the right people first, then each person shows up already paired with the right collaborators.

This exists because object graphs become too complex to manage manually at scale. If you ignore it, your classes become tightly coupled to concrete implementations and your codebase resists change.

Technical

What Spring is doing for you

Definition: Inversion of Control means the framework owns object creation, and Dependency Injection means it supplies collaborators into those objects instead of letting them create collaborators manually.

At startup, Spring scans configuration and bean definitions, builds an application context, chooses constructors, creates objects, and then wires dependencies based on type and qualifiers.

Constructor injection is usually the best default because it makes required dependencies explicit and makes the object impossible to create in an invalid state.

The mental model is: “Spring builds the graph; my class declares what it needs.” Once you adopt that model, testing and refactoring get dramatically cleaner.

Why it matters in production

  • Makes dependencies explicit
  • Improves testability and replacement
  • Keeps construction logic centralized

Common smell

  • new-ing collaborators inside services
  • Field injection hiding required dependencies
  • Classes with too many dependencies signaling design issues
Constructor injectionApplicationContext builds graphHidden dependencies
Read bean definitions→Create objects→Resolve constructor deps→Expose ready-to-use graph

“Dependency Injection means my class declares collaborators and Spring supplies them. In practice I prefer constructor injection because it makes required dependencies explicit, is test-friendly, and prevents half-initialized objects. A common smell is when a service suddenly needs six or seven dependencies—that usually signals the class is doing too much.”

Let Spring construct the graph and keep your classes focused on behavior. Constructor injection also makes unit tests simple because you can pass mocks directly without spinning up a container.

The safest mental model is that Spring gives you a framework feature only when your call crosses the Spring-managed boundary and the surrounding infrastructure can participate.

Warning

Field injection hides required dependencies

Field injection looks short, but it hides what the class truly needs and makes immutability harder. It also encourages objects that can exist in a partially configured state outside the container.

Use constructor injection for mandatory dependencies. Reserve setter-style injection for truly optional collaborators, and even then use it intentionally.

Broken pattern
@Service
public class PaymentFacade {

    @Autowired
    private FraudClient fraudClient;

    @Autowired
    private LedgerClient ledgerClient;
}
Fixed pattern
@Service
public class PaymentFacade {

    private final FraudClient fraudClient;
    private final LedgerClient ledgerClient;

    public PaymentFacade(FraudClient fraudClient, LedgerClient ledgerClient) {
        this.fraudClient = fraudClient;
        this.ledgerClient = ledgerClient;
    }
}
Mini takeaway
  • Spring owns object creation; your class declares requirements
  • Constructor injection is the safest default
  • Too many dependencies usually means too many responsibilities

Section 2 · Resolution rules

Bean Resolution (@Qualifier, @Primary)

When multiple beans satisfy the same interface, Spring needs help deciding which one to inject. That is where @Primary, @Qualifier, and explicit configuration become important.

🧒 ELI5

Choosing the right driver for the same ride

Suppose a company app lets you request a ride, but there are three drivers already waiting in the same parking lot. Saying “send a driver” is not enough if one is for VIP rides, one is for airport trips, and one is for standard pickups.

Someone has to add the rule that says which driver is the default and when a specific one must be requested. Without that rule, the dispatcher cannot choose safely.

That is what happens when Spring sees multiple beans of the same type. The container needs either one default or a precise instruction.

🔗 ANALOGY

Multiple power outlets with one labeled adapter

Think of a travel bag with several adapters that all plug into the wall but serve different countries. A “primary” adapter is the one you reach for by default, while a labeled pouch helps when you need the special one.

This concept exists because large systems often have multiple implementations of the same contract. Ignore it and you get ambiguous wiring errors or worse, a silently wrong implementation injected into production paths.

Technical

How Spring picks one bean

Definition: Spring resolves dependencies primarily by type; if multiple beans match, it then needs qualifiers, a primary bean, or a more explicit configuration path.

@Primary defines the default bean of a type when no narrower choice is made. @Qualifier selects a specific bean by name or custom qualifier.

Good teams do not rely on accidental bean names or implicit wiring when the distinction matters. They make intent obvious at the injection point.

The mental model is simple: type narrows the candidate set, then qualifiers and primary rules break ties.

Why it matters in production

  • Supports multiple implementations cleanly
  • Makes environment-specific or strategy-specific wiring explicit
  • Prevents accidental ambiguity

Common smell

  • Relying on bean names without documenting intent
  • Marking everything @Primary
  • Using string qualifiers when a custom qualifier would be clearer
Explicit strategy choiceType first, qualifier secondOne default only
Find beans by interface→See multiple matches→Apply qualifier/primary rule→Inject chosen implementation

💬 Model Interview Answer

“Spring resolves dependencies by type first, then needs help if there are multiple candidates. I use @Primary for the sensible default and @Qualifier when a call site genuinely needs a specific strategy. The gotcha is that if several implementations exist and the choice has business meaning, the injection point should say so explicitly instead of relying on lucky bean names.”

Warning

Too many primaries create confusion

Developers sometimes slap @Primary on whichever bean fixes the current startup error. That turns dependency resolution into a game of whack-a-mole.

Use one clear default per interface when it makes sense. When the choice changes by use case, prefer explicit qualifiers or a factory abstraction instead of container guesswork.

Mini takeaway
  • Type matching comes first, tie-breakers come second
  • Use @Primary for the default, @Qualifier for the exception
  • Injection points should reveal important strategy choices
Quiz

When two beans implement the same interface, which annotation most directly selects a specific one at the injection point?

Correct answer: CC is correct because @Qualifier narrows the candidate set at the exact injection point. @Primary helps define the default bean when no narrower choice is provided, but it does not express a specific call-site preference the way a qualifier does. @Transactional and @Profile solve unrelated concerns—transaction boundaries and environment-based activation.
Section 3 · Lifecycle behavior

Scopes & Lifecycle

Bean scope controls how many instances Spring creates and how long they live. The default singleton scope is usually correct, but request, prototype, and other scopes matter in the right cases.

🧒 ELI5

Shared kitchen appliances vs disposable paper cups

A restaurant keeps one industrial oven for everyone because it is expensive and shared. But it uses a fresh paper cup for each guest because reusing that object across people would be messy and unsafe.

Some things are naturally shared, while other things should exist only for one short-lived interaction. The key is matching lifespan to responsibility.

That is exactly what scope solves. If you use the wrong lifetime, state leaks across users or you create unnecessary objects on every request.

🔗 ANALOGY

Hotel rooms vs lobby furniture

Lobby furniture is shared by all guests, but each room belongs to one booking window. You would not treat towels like lobby chairs, and you would not try to assign one sofa to every room.

This concept exists because not every object should be global. If you ignore scope, mutable state in a singleton becomes a production bug factory.

Technical

What scope changes in practice

Definition: Bean scope tells Spring whether to create one shared instance, one per request, one per prototype lookup, or another lifecycle-managed variant.

Singleton is the default and is ideal for stateless services, repositories, and configuration helpers. Request scope is useful for request-specific state, and prototype is usually rare in typical Spring apps.

The tricky part is when a long-lived singleton needs access to a shorter-lived bean. In that case you need a proxy, ObjectProvider, or another lookup mechanism so the shorter-lived instance is resolved at the right time.

The mental model is: “Who owns this state, and for how long should it exist?” Answer that before choosing a scope.

Why it matters in production

  • Prevents state leakage across requests
  • Improves memory and concurrency behavior
  • Makes request-specific state explicit

Common smell

  • Mutable fields inside singleton services
  • Injecting request-scoped beans directly into singletons without a proxy
  • Using prototype to avoid fixing poor design
Stateless singletonsRequest scope for request stateMutable shared state
Container sees scope→Creates bean at right lifecycle→Injects or proxies reference→Disposes at scope end

💬 Model Interview Answer

“Most Spring beans should stay singleton and stateless. Scope becomes interesting when state is tied to a request, session, or short-lived workflow. The common gotcha is injecting a short-lived bean into a singleton—if you need that, use a scoped proxy or provider so the correct instance is resolved at runtime.”

Scope should match ownership of state. Stateless services are ideal singletons; stateful per-request data belongs in request-bound objects or explicit method parameters.

The safest mental model is that Spring gives you a framework feature only when your call crosses the Spring-managed boundary and the surrounding infrastructure can participate.

Warning

Singleton + mutable state is a concurrency bug

A singleton bean is shared across threads. If it stores mutable user-specific data, one request can overwrite another and you get cross-request leakage.

The fix is usually to remove the mutable field entirely and pass data as method arguments. If true request state is needed, use request scope or a dedicated context abstraction.

Broken pattern
@Service
public class GreetingService {

    private Locale locale;

    public String greet(User user) {
        this.locale = user.getLocale();
        return locale.getLanguage();
    }
}
Fixed pattern
@Service
public class GreetingService {

    public String greet(User user) {
        Locale locale = user.getLocale();
        return locale.getLanguage();
    }
}
Mini takeaway
  • Singletons should normally be stateless
  • Request scope is about state ownership, not convenience
  • Passing state through method arguments is often better than changing scope
Quiz

What is the biggest risk of putting mutable request-specific data inside a singleton bean?

Correct answer: CC is correct because singleton beans are shared across threads, so mutable user-specific state can bleed from one request into another. A, B, and D describe unrelated concerns. In interviews, this is a great place to mention that stateless singleton services are safe, while shared mutable state is the real danger.
Section 4 · Environment-specific wiring

Profiles & Configuration

Profiles and configuration classes let you activate different beans for different environments without scattering if/else environment checks throughout business code.

🧒 ELI5

Different uniforms for different work shifts

A warehouse might have one set of workers for daytime loading and another for overnight cleaning. You do not want each worker deciding at runtime which uniform to wear by reading a sticky note on the wall.

The company sets up the right team before the shift starts. Then each worker just does the job for that environment.

Profiles do that for your application. They let the environment decide which beans exist so your code stays clean.

🔗 ANALOGY

Plugging the right appliance into the right country voltage

You do not keep opening the toaster to ask whether you are in the U.S. or Europe. You prepare the right adapter and power setup before turning it on.

Profiles exist because environment-specific wiring is a bootstrapping concern, not business logic. If you ignore that separation, production conditions leak into every service class.

Technical

How profiles fit into the container

Definition: Profiles are conditional bean-activation rules that let Spring include or exclude beans and configuration classes based on the current environment.

A @Configuration class can define beans for local development, production, test, or feature-specific setups. Spring evaluates active profiles at startup and builds the bean graph accordingly.

This is more maintainable than sprinkling environment checks across services. Business classes should depend on interfaces and receive whichever implementation the container selected.

The mental model is: “Profiles shape the application graph before runtime work begins.” They are about assembly, not day-to-day business branching.

Why it matters in production

  • Keeps environment rules out of domain logic
  • Makes test vs prod dependencies replaceable
  • Supports safer local/dev/prod wiring

Common smell

  • Using profiles for business logic instead of environment wiring
  • Creating too many fine-grained profiles that are hard to reason about
  • Reading env vars directly in every service
Assembly-time decisionConfiguration classesAvoid profile explosion
Read active profiles→Load matching config→Create matching beans→Inject selected implementations

💬 Model Interview Answer

“I use profiles to change wiring by environment, not to encode domain behavior. The container should decide whether I get a fake client, local stub, or production implementation before my services run. The trade-off is that too many profiles become hard to reason about, so I keep them purposeful and well named.”

Section 5 · Interview deep dive

Deep Dive: Qualifiers, Scopes, and Injection Patterns

This deep dive ties together the container’s resolution rules, lifecycle rules, and design signals so you can explain not just how Spring injects beans, but how to structure code around those mechanics.

🧒 ELI5

A hotel manager assigning rooms, staff, and equipment

A hotel manager does three jobs at once: deciding which staff member handles a guest, deciding whether an item is shared or personal, and making sure the right equipment is available in the right place.

That is the same coordination Spring performs when it decides which implementation to inject, how long it should live, and how objects should get access to it.

Senior-level reasoning starts when you stop treating these as separate annotations and start seeing them as one object-graph design problem.

🔗 ANALOGY

A toolbox with default tools and special attachments

Most jobs use the default hammer from the toolbox, but a few jobs need a specialty attachment. Some tools stay in the common workshop, while others are issued per technician for one task.

If you do not label those decisions clearly, people grab the wrong tool or share something that was supposed to be personal. That is exactly what ambiguous injection and wrong scope do in code.

Technical

How to reason like a designer, not just a framework user

Definition: The real design question is how to keep dependencies explicit, strategies intentional, and state ownership unambiguous as the object graph grows.

Constructor injection communicates required collaborators. Factories or strategy registries help when the choice between implementations is dynamic and business-driven.

Request state should usually travel as method data, but when a request-bound context abstraction is useful, keep it clearly scoped and avoid leaking it into generic services.

Custom qualifiers can be better than raw string qualifiers when the distinction has long-term meaning, because they reduce typo risk and make intent easier to read.

What senior engineers optimize for

  • Prefer constructor injection for mandatory deps
  • Use factories when implementation choice is dynamic
  • Treat mutable state ownership as a first-class design decision

Trade-offs

  • Field injection hides graph complexity
  • Overusing request scope to avoid parameter passing
  • Container rules cannot rescue poor boundaries
Explicit graphFactories for dynamic choiceHidden state
Declare collaborators→Resolve defaults or qualifiers→Respect scope lifetime→Use through clear boundaries

💬 Model Interview Answer

“At a senior level I think about DI as graph design, not annotation memorization. Constructor injection makes required collaborators obvious, qualifiers make strategy choices explicit, and scope decisions protect state ownership. If a class needs too many collaborators or weird scope tricks, that is often a design signal that the class boundary should be reconsidered.”

✅ Right: inject a strategy registry cleanly
public interface PricingStrategy {
    Money price(Order order);
    PricingTier tier();
}

@Service
public class PricingStrategyFactory {

    private final Map<PricingTier, PricingStrategy> strategies;

    public PricingStrategyFactory(List<PricingStrategy> strategies) {
        this.strategies = strategies.stream()
                .collect(Collectors.toMap(PricingStrategy::tier, s -> s));
    }

    public PricingStrategy forTier(PricingTier tier) {
        return strategies.get(tier);
    }
}
Warning

The service locator anti-pattern hides dependencies

If you inject ApplicationContext into arbitrary business services just to pull out beans manually, you are often sliding back toward a service locator anti-pattern.

A request-scoped bean inside a singleton is not automatically wrong, but it should raise the question of whether the data should simply be passed as an argument.

Broken pattern
@Service
public class PricingService {

    @Autowired
    private ApplicationContext applicationContext;

    public Money price(Customer customer, Order order) {
        PricingStrategy strategy = applicationContext.getBean(customer.isPremium()
                ? "premiumPricingStrategy"
                : "standardPricingStrategy", PricingStrategy.class);
        return strategy.price(order);
    }
}
Fixed pattern
@Service
public class PricingService {

    private final PricingStrategyFactory pricingStrategyFactory;

    public PricingService(PricingStrategyFactory pricingStrategyFactory) {
        this.pricingStrategyFactory = pricingStrategyFactory;
    }

    public Money price(Customer customer, Order order) {
        return pricingStrategyFactory.forTier(customer.getTier()).price(order);
    }
}
Edge cases

Edge Cases & Tradeoffs

  • Circular dependencies are usually a design smell, not a wiring challenge to outsmart.
  • Custom qualifiers are useful when the distinction is semantic and stable, like @InternalClient vs @ExternalClient.
  • Bean lifecycle callbacks are helpful, but business logic should not depend on container timing in surprising ways.
  • A request-scoped bean inside a singleton may need a proxy or provider.
  • Factories are cleaner than long chains of qualifiers when selection becomes dynamic.
Checklist

Production Checklist

  • Prefer constructor injection everywhere by default
  • Use @Primary for the sensible default, @Qualifier for deliberate exceptions
  • Reach for factories when implementation choice is dynamic
  • Keep singletons stateless unless there is a very good reason
  • Treat circular dependencies as a redesign signal
Interview

Interview Hooks

  • Explain type resolution order: type, then qualifier/primary
  • Explain why field injection is discouraged
  • Mention that mutable state in singletons is a concurrency risk
