Entity Mapping & Relationships

JPA mappings define how objects translate to tables and how relationships are stored.

🧒 ELI5

Friends list in a notebook

You write a friend’s name once and reference it instead of copying it everywhere.

🔗 ANALOGY

Library catalog

A book references an author entry instead of repeating the author data in every row.

Technical

What it means in real code

Definition: Relationships like @ManyToOne and @OneToMany map foreign keys between tables.

Why it matters

  • Clear domain modeling
  • Avoid duplicate data
  • Supports navigation in code

Common smell

  • No owning side defined
  • Both sides controlling FK
  • Confusing cascades
Owning sideMappedByCascade with care
Define entity→Map relation→Set owning side→Persist

Interview answer

“The owning side owns the foreign key. I use mappedBy on the inverse side.”

Section 2 of 4 · ~7 mins

Lazy vs Eager & N+1

Fetching strategy drives query count and performance.

🧒 ELI5

Order now or later

You can bring all groceries home at once (eager) or fetch them when needed (lazy).

🔗 ANALOGY

Streaming vs download

Streaming loads data on demand; downloading grabs everything upfront.

Technical

What it means in real code

Definition: Lazy loads related data when accessed; N+1 happens when each access triggers a query.

Why it matters

  • Avoid unnecessary queries
  • Control performance
  • Reduce memory usage

Common smell

  • Multiple queries per row
  • Eager everywhere
  • Slow list screens
Fetch joinEntity graphN+1
Load list→Access relation→Extra query per row→N+1

Interview answer

“I prefer lazy defaults and fix N+1 with fetch joins or entity graphs.”

Fetch joins load relationships in a single query.

Warning

Eager can explode

Eager loading can multiply rows and memory usage.

Mini takeaway
  • Lazy by default.
  • Use fetch joins for N+1.
  • Measure query counts.
Quiz

What is the N+1 problem?

Correct answer: B N+1 means 1 query for list plus N queries for each relation.
Section 3 of 4 · ~7 mins

Persistence Context & Dirty Checking

Hibernate tracks entity changes and flushes them at transaction boundaries.

🧒 ELI5

A smart notebook

You write notes, and the notebook remembers what changed so it can update later.

🔗 ANALOGY

Auto-save document

Changes are tracked and saved when you click save, not after every keystroke.

Technical

What it means in real code

Definition: Managed entities are tracked in the persistence context; changes are flushed on commit.

Why it matters

  • Automatic updates
  • Consistent unit of work
  • Less manual SQL

Common smell

  • Detached entity updates lost
  • Unexpected updates
  • Entity outside transaction
Managed entityDirty checkingDetach risk
Load entity→Modify→Commit→Flush SQL

Interview answer

“Hibernate tracks managed entities and writes changes on flush/commit, so I keep updates inside transactions.”

  • Changes flush at commit.

When does dirty checking write SQL?

Section 4 of 4 · ~7 mins

Performance Tuning

JPA performance improves with batching, pagination, and DTO projections.

🧒 ELI5

Carry groceries in batches

You bring groceries in bags, not one item at a time.

🔗 ANALOGY

Bulk shipping

Shipping in bulk is cheaper and faster than many tiny shipments.

Technical

What it means in real code

Definition: Use batch size, pagination, and projections to reduce SQL and memory overhead.

Why it matters

  • Less memory usage
  • Fewer SQL statements
  • Predictable performance

Common smell

  • Loading huge lists
  • N+1 everywhere
  • Unbounded fetch
PaginationDTO projectionLoad everything
Query page→Fetch join (safe)→Map DTO→Return

Interview answer

“I paginate, use fetch joins carefully, and project DTOs for read-heavy screens.”

Mini takeaway
  • Paginate large queries.
  • Use DTOs for read screens.
  • Tune batch size.
Quiz

Which technique reduces memory for list screens?
