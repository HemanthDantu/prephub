Property Sources and Precedence

Spring Boot merges multiple property sources, with higher precedence sources overriding lower ones.

← Back to Spring Boot
🧒 ELI5

The last instruction wins

If your parent says bedtime is 8, but later says 9, you go with the latest rule.

🔗 ANALOGY

Manager overrides the checklist

The checklist sets defaults, but a manager can override them for a special case.

Technical

What it means in real code

Definition: Boot resolves properties from many sources and applies a strict precedence order for overrides.

Spring Boot loads configuration from multiple sources in a defined order, and later sources override earlier ones.

Understanding precedence is critical because it explains why an env var can silently override your application.yml.

When debugging config issues, print the effective configuration or enable the /actuator/env endpoint to see source origins.

Why it matters

  • Environment-specific overrides
  • Secure secrets via env vars
  • Simple defaults in code

Common pitfall

  • Assuming one config file
  • Hidden overrides in env
  • Unclear deployment flags

Defaults→application.yml→Profile file→Env vars→Command-line

Interview answer

“Boot loads multiple property sources and applies a fixed precedence, so higher priority sources override lower ones.”

As you study this, focus on what Spring/Java is doing on your behalf versus what you must configure explicitly. That mental split makes the API feel predictable instead of magical.

Typed Binding with @ConfigurationProperties

Use @ConfigurationProperties to map config into a typed object that is validated and easy to test.

🧒 ELI5

Label your drawers

You label each drawer so you always know where socks and shirts go.

🔗 ANALOGY

A labeled toolbox

Instead of loose tools, everything is in a labeled slot and easy to find.

Technical

What it means in real code

Definition: @ConfigurationProperties binds structured config into POJOs and can be validated with JSR-303.

@ConfigurationProperties binds related settings into a typed object, which is easier to validate and test than scattered @Value strings.

Validation fails fast on bad config, which is safer than letting invalid values slip into production.

Typed config also improves IDE support and documentation because fields are discoverable and centralized.

Why it matters

  • Type safety
  • Validation at startup
  • Clear ownership of config

Common pitfall

  • Magic strings everywhere
  • No validation

Profiles and Environment Overrides

Profiles let you swap config or beans depending on the environment without changing code.

🧒 ELI5

Different outfits

You wear a raincoat on rainy days and a t-shirt on sunny days.

🔗 ANALOGY

Technical

What it means in real code

Definition: Profiles activate different configuration files and beans using spring.profiles.active or activation rules.

Profiles let you isolate environment-specific settings while keeping shared defaults in application.yml.

Profile groups can activate multiple profiles together, which is helpful for layered environments like prod + observability.

Be explicit about which profiles are active in each environment so you avoid hidden behavior differences.

Why it matters

  • Clean dev/test/prod separation
  • Swap implementations safely
  • Less manual toggling

Common pitfall

  • Manual bean switches
  • Config sprawl

Secrets and Safe Overrides

Externalized config keeps secrets out of repos and makes deployments safer and repeatable.

🧒 ELI5

A lockbox for keys

You keep the house key in a lockbox, not taped to the front door.

🔗 ANALOGY

Bank vault

The vault stores the real valuables, and the teller only grabs what is needed.

Technical

What it means in real code

Definition: Sensitive values should come from secure sources like environment variables, vaults, or secret managers.

Secrets should live outside the repo in env vars or a secret manager, not in config files.

If you need runtime config changes, use a refresh mechanism or restart strategy rather than ad-hoc overrides.

A simple rule is: if leaking it would hurt you, it belongs in a secret manager, not in Git.

Why it matters

  • Avoids leaking secrets
  • Supports rotation
  • Keeps repos clean

Common pitfall

  • Passwords in Git
  • Plaintext secrets in YAML
  • No rotation strategy
Env varsVaultRotation
Secret store→Inject into env→Boot binds→App uses securely

Interview answer

“I keep secrets out of code by using env vars or a vault and let Boot bind them like any other property.”

Edge cases

Edge Cases & Tradeoffs
