Circuit Breakers

Circuit breakers stop calls to failing dependencies to protect the system and allow recovery.

← Back to Spring Boot
🧒 ELI5

A safety fuse

If the power surges, the fuse trips to protect the house.

🔗 ANALOGY

Traffic barrier

If a road is closed, you stop cars from entering so it can recover.

Technical

What it means in real code

Definition: A circuit breaker tracks failures and opens when a threshold is reached, then half-opens to test recovery.

Circuit breakers stop repeated calls to failing dependencies by opening after a threshold of errors.

They protect the system from cascading failures and give downstreams time to recover.

As a beginner, imagine a fuse that trips when a service is unhealthy, preventing more damage.

Why it matters

  • Prevents cascading failures
  • Fails fast
  • Gives dependencies time to recover

Common pitfall

  • Repeated timeouts
  • Cascading failures
  • No fallback logic
ClosedOpenHalf-open
Call service→Failures exceed threshold→Open circuit→Half-open trial

Interview answer

“Circuit breakers fail fast when a dependency is unhealthy and periodically test recovery.”

StateBehavior
ClosedCalls allowed
OpenCalls blocked
Half-openTrial requests
Resilience4j annotation
@CircuitBreaker(name = "inventory", fallbackMethod = "fallback")
public Inventory check(String sku) { ... }
Warning

No fallback = poor UX

If the circuit opens, you need a fallback or you will fail every request.

Mini takeaway
  • Circuit breakers fail fast.
  • Use fallback methods.
  • Monitor breaker events.
Quiz

What is the purpose of the half-open state?

Correct answer: B Half-open allows limited calls to test recovery.
Section 2 of 5 · ~7 mins

Retries and Timeouts

Retries handle transient errors, while timeouts prevent hanging calls.

🧒 ELI5

Try again, but not forever

If a toy does not work, you try again a couple of times then move on.

🔗 ANALOGY

Redial with patience

You redial after a short wait instead of hanging on forever.

Technical

What it means in real code

Definition: Retries should use backoff and be applied only to idempotent operations, with timeouts to cap latency.

Retries can fix transient errors but must be paired with timeouts to avoid infinite waits.

Use exponential backoff with jitter to avoid retry storms during outages.

A simple rule is to retry only on safe, idempotent calls and keep retry counts small.

Why it matters

  • Improves resilience
  • Avoids long waits
  • Handles transient faults

Common pitfall

  • Retry storms
  • No timeouts
  • Retrying writes
BackoffTimeoutIdempotent
Call→Timeout or error→Retry with backoff→Fail or succeed

Interview answer

“Retries with backoff work for transient errors, but only for idempotent calls and with timeouts.”

RuleReason
Use backoffReduce load
Limit attemptsAvoid storms
Idempotent onlyPrevent duplicates
Retry config
resilience4j:
  retry:
    instances:
      payments:
        max-attempts: 3
        wait-duration: 200ms
Warning

Retries can amplify failures

If every caller retries, a failing service can get overwhelmed.

Mini takeaway
  • Retry only safe calls.
  • Add backoff.
  • Always use timeouts.
Quiz

Why use exponential backoff on retries?

Correct answer: B Backoff reduces pressure on a failing service.

Bulkheads and Rate Limiting

Bulkheads isolate resources so one failure does not take down the whole system.

🧒 ELI5

Separate rooms

If one room is noisy, the rest of the house stays quiet.

🔗 ANALOGY

Lifeboats on a ship

Compartments prevent water from flooding the entire ship.

Technical

What it means in real code

Definition: Bulkheads limit concurrent calls via separate thread pools or semaphores, and rate limiters cap traffic.

Bulkheads limit resource usage so failures in one area do not starve others.

Rate limiting protects downstreams and enforces fairness under load.

Think of bulkheads as separate compartments in a ship: one leak should not sink everything.

Why it matters

  • Prevents resource exhaustion
  • Keeps critical paths alive
  • Controls load

Common pitfall

  • Shared thread pool for everything
  • No rate limits
  • Cascading failures
BulkheadSemaphoreRate limit
Request enters→Check capacity→Execute or reject→Protect resources

Interview answer

“Bulkheads isolate expensive calls with their own pools, while rate limiting caps total load.”

PatternGoal
BulkheadIsolation
Rate limiterTraffic control
TimeoutBound latency
Bulkhead annotation
@Bulkhead(name = "inventory", type = Bulkhead.Type.SEMAPHORE)
public Inventory check(String sku) { ... }
Warning

Shared pools cause cascades

If all calls share one pool, a slow dependency can exhaust all threads.

Mini takeaway
  • Bulkheads isolate load.
  • Rate limit protects services.
  • Separate pools for critical paths.
Quiz

What does a bulkhead protect against?

Correct answer: B Bulkheads limit resource usage so one dependency cannot exhaust all capacity.

Spring Integration and Observability

Resilience4j integrates with Spring Boot through annotations and configuration, with event metrics.

🧒 ELI5

Plug-in safety kit

You add a safety kit to your bike so brakes and lights are always there.

🔗 ANALOGY

Airbag system

Airbags are built in and activate when needed without manual effort.

Technical

What it means in real code

Definition: Resilience4j exposes annotations, config, and events that integrate with Boot and Micrometer.

Spring integrates Resilience4j via annotations and configuration, making patterns easy to apply consistently.

Expose metrics for breaker state, retries, and latency so you can tune in production.

If you cannot observe the breaker state, you cannot trust it; monitoring is part of the pattern.

Why it matters

  • Centralized configuration
  • Consistent behavior
  • Metrics on resilience events

Common pitfall

  • Hard-coded resilience logic
  • No monitoring
  • Inconsistent retries

Deep Dive: Fallbacks and Failure Budgets

Resilience patterns must be paired with sensible fallbacks and SLO budgets.

🧒 ELI5

Have a backup plan

If the toy breaks, you have another game to play.

🔗 ANALOGY

Emergency exits

Buildings must have exits in case a path fails.

Technical

What it means in real code

Definition: Fallbacks provide degraded responses while circuit breakers and retries protect systems.

Why it matters

  • Improves user experience
  • Avoids full outages
  • Supports graceful degradation

Common pitfall

  • No fallback
  • Fallback slower than main path
  • Retry storms
FallbackBudgetGraceful
Call fails→Breaker opens→Fallback returns→Track budget

Interview answer

“I combine circuit breakers with safe fallbacks so the system degrades instead of failing hard.”

FallbackExample
Cached dataStale ok
Partial responseDegraded
Error messageWorst-case
Fallback method
public Inventory fallback(String sku, Throwable ex) {
  return Inventory.unavailable();
}
Warning

Fallbacks can hide outages

If you never alert on fallback use, you might miss incidents.

Mini takeaway
  • Provide safe fallbacks.
  • Monitor fallback rate.
  • Tune retry budgets.
Edge cases

Edge Cases & Tradeoffs

  • Retry storms amplify outages.
  • Circuit breakers must be tuned per dependency.
  • Fallbacks can hide real failures.
  • Bulkhead limits can starve critical traffic.
  • Timeouts should be lower than upstream SLAs.
Checklist

Production Checklist

  • Set timeouts before retries.
  • Use exponential backoff with jitter.
  • Add circuit breakers for unstable deps.
  • Limit concurrency with bulkheads.
  • Export metrics for breaker state.
