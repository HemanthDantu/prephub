WebClient Fundamentals

WebClient is the modern non-blocking HTTP client in Spring for reactive or async usage.

← Back to Spring Boot
🧒 ELI5

Send a text message

You send a message and keep doing other things while waiting for the reply.

🔗 ANALOGY

Courier service

You hand off a package to a courier and do not wait at the door.

Technical

What it means in real code

Definition: WebClient builds reactive HTTP requests using Reactor and supports filters and codecs.

WebClient is non-blocking and designed for reactive pipelines, returning Mono or Flux for async composition.

It shines in high-concurrency IO because threads are not blocked on network waits.

Why it matters

  • Non-blocking I/O
  • Composable filters
  • Consistent with reactive stack

Common pitfall

  • Using RestTemplate in reactive app
  • Blocking calls on event loop
  • No central client config
ReactiveFiltersBackpressure
Build request→Apply filters→Send→Receive response

Interview answer

“WebClient is the reactive HTTP client; I configure a shared builder with timeouts and filters.”

ClientBest for
RestTemplateLegacy blocking
WebClientReactive + async

Basic WebClient
WebClient client = WebClient.builder()
  .baseUrl("https://api.acme.com")
  .build();

Mono<User> user = client.get()
  .uri("/users/{id}", id)
  .retrieve()
  .bodyToMono(User.class);
Warning

Do not block

Calling block() inside a reactive pipeline can deadlock or reduce throughput.

Mini takeaway
  • Use WebClient for non-blocking HTTP.
  • Create a shared builder.
  • Avoid blocking calls.
Quiz

What is the primary advantage of WebClient over RestTemplate?

Correct answer: B WebClient is built for non-blocking reactive calls.
Section 2 of 5 · ~7 mins

retrieve vs exchangeToMono

retrieve is simple for success responses, while exchangeToMono provides full control over status handling.

🧒 ELI5

Fast checkout vs manual checkout

Fast checkout is quick when things go well; manual checkout handles special cases.

🔗 ANALOGY

Autopilot vs manual driving

Autopilot works on highways, but manual driving handles tricky roads.

Technical

What it means in real code

Definition: retrieve() assumes a 2xx success path, while exchangeToMono gives access to status codes and headers.

retrieve() is convenient for happy-path responses, while exchangeToMono gives access to status codes and headers.

Use exchangeToMono for custom error mapping or conditional handling based on response metadata.

Why it matters

  • Simpler happy path
  • Custom error mapping
  • Access to headers

Common pitfall

  • Ignoring non-2xx responses
  • No error mapping
  • Hidden retries
retrieve()exchangeToMono()Status handling
Send request→Inspect status→Map body→Handle errors

Interview answer

“I use retrieve for simple calls and exchangeToMono when I need custom status handling or headers.”

MethodUse case
retrieve()Simple 2xx flow
exchangeToMono()Custom status handling
onStatus()Error mapping
exchangeToMono example
client.get().uri("/users/{id}", id)
  .exchangeToMono(resp -> {
    if (resp.statusCode().is2xxSuccessful()) {
      return resp.bodyToMono(User.class);
    }
    return resp.createException().flatMap(Mono::error);
  });
Warning

Error handling is explicit

If you use retrieve without onStatus, you may miss useful error details.

Mini takeaway
  • retrieve is simpler.
  • exchangeToMono gives control.
  • Always map errors.
Quiz

When should you prefer exchangeToMono?

Correct answer: B exchangeToMono provides full response access for custom handling.
Section 3 of 5 · ~7 mins

Timeouts and Retries

Production clients must enforce timeouts and define safe retry policies.

🧒 ELI5

Stop waiting forever

If no one answers the door, you stop waiting and try later.

🔗 ANALOGY

Phone call retry

You call again after a short wait, but not forever.

Technical

What it means in real code

Definition: Timeouts cut off slow calls, and retries with backoff handle transient failures safely.

Timeouts prevent slow dependencies from tying up resources, and retries should be bounded with backoff.

Only retry idempotent requests; otherwise you can duplicate side effects under failure.

Why it matters

  • Avoid hanging threads
  • Reduce tail latency
  • Improve resilience

Common pitfall

  • No timeouts
  • Aggressive retries
  • Retrying POST/PUT
TimeoutsBackoffIdempotency
Send request→Wait timeout→Retry with backoff→Fail gracefully

Interview answer

“I set connect/response timeouts and retry only idempotent operations with backoff.”

PolicyNotes
TimeoutProtects threads
RetryTransient errors
Circuit breakerStops overload
Timeout configuration
HttpClient httpClient = HttpClient.create()
  .responseTimeout(Duration.ofSeconds(2));

WebClient client = WebClient.builder()
  .clientConnector(new ReactorClientHttpConnector(httpClient))
  .build();
Warning

Do not retry non-idempotent calls

Retrying POST or charge operations can cause duplicate side effects.

Mini takeaway
  • Always set timeouts.
  • Retry safely with backoff.
  • Avoid retrying non-idempotent calls.
Quiz

Why avoid retrying POST requests?

Correct answer: B POST is not idempotent and retries can duplicate side effects.

Connection Pools and Observability

Tune the HTTP client connection pool and add filters for logging, metrics, and tracing.

🧒 ELI5

Shared highway lanes

If every car makes a new lane, traffic explodes. A shared highway uses lanes efficiently.

🔗 ANALOGY

Call center lines

A call center reuses phone lines rather than creating new lines per call.

Technical

What it means in real code

Definition: Reactor Netty provides connection pooling, while filters add logging, metrics, and tracing to WebClient.

Connection pooling reduces socket churn and improves latency, especially under high throughput.

Enable metrics and logs to observe latency, error rates, and pool saturation.

Why it matters

  • Reuse connections
  • Reduce latency
  • Consistent observability

Common pitfall

  • Creating new clients per request
  • No pooling
  • Missing trace headers
PoolingFiltersTracing
Acquire connection→Send request→Record metrics→Release connection

Interview answer

“I configure connection pooling and add filters for logging and metrics to standardize clients.”

ToolBenefit
Connection poolLower latency
ExchangeFilterFunctionCross-cutting concerns
MicrometerMetrics visibility
Add a logging filter
ExchangeFilterFunction logFilter = (req, next) -> {
  return next.exchange(req).doOnNext(resp -> {
    log.info("{} {} -> {}", req.method(), req.url(), resp.statusCode());
  });
};

WebClient client = WebClient.builder()
  .filter(logFilter)
  .build();
Warning

Log carefully

Logging full payloads can leak sensitive data and slow down requests.

Mini takeaway
  • Reuse WebClient instances.
  • Enable pooling.
  • Add filters for observability.
Quiz

Why create a shared WebClient bean?

Correct answer: B A shared WebClient reuses pooled connections and keeps configuration consistent.

Deep Dive: Error Mapping & Resilience

Production clients must map errors, apply retries safely, and propagate context.

🧒 ELI5

Know why it failed

If a call fails, you want to know if it was busy or broken.

🔗 ANALOGY

Customer support scripts

Different errors need different responses.

Technical

What it means in real code

Definition: Map HTTP status to domain errors, apply retries only when safe, and propagate tracing headers.

Why it matters

  • Clearer failures
  • Safer retries
  • Consistent observability

Common pitfall

  • Throwing generic errors
  • Retrying 4xx
  • No tracing context
Error mappingRetry policyTracing
Send request→Check status→Map error→Retry if safe

Interview answer

“I map non-2xx to domain errors and retry only transient 5xx for idempotent requests.”

StatusAction
4xxDo not retry
5xxRetry with backoff
TimeoutCircuit breaker
Map error status
client.get().uri("/orders/{id}", id)
  .retrieve()
  .onStatus(HttpStatus::is4xxClientError, resp ->
      resp.bodyToMono(String.class)
          .map(msg -> new ClientError(msg)))
  .bodyToMono(Order.class);
Warning

Retrying 4xx is wasteful

Most client errors will not succeed on retry.

Mini takeaway
  • Map errors explicitly.
  • Retry only safe calls.
  • Propagate trace headers.
Edge cases

Edge Cases & Tradeoffs

  • Default timeouts can be too lenient.
  • Blocking in reactive chains causes deadlocks.
  • Retries can amplify load during outages.
  • Large payloads need memory limits.
  • Error handling differs between retrieve and exchange.
Checklist

Production Checklist

  • Configure connect/read timeouts.
  • Use exchangeToMono for full response control.
  • Add retry with backoff and circuit breaker.
  • Propagate correlation IDs.
  • Avoid blocking calls in reactive flows.
Interview

Interview Hooks

  • Use Reactor retry with jitter.
  • Use connection pooling to avoid socket churn.
