Why AOP Exists

AOP solves one problem: applying the same behavior across many methods without copy-pasting it into each one. Understanding why it exists is what lets you explain @Transactional, @Cacheable, and @Async confidently in interviews.

🧒 ELI5

The automatic security checkpoint

Imagine a large office building. You need security checks at every door — log who entered, verify their badge, record the time. One approach: hire a guard to stand at every single door. Another: build a single central checkpoint that every visitor must pass through, and put your security logic there once.

Without AOP, your code is the first approach: you write log.info("entering method"), start a timer, check authentication, and open a transaction in every single service method. The method body that actually does the business work drowns in infrastructure noise. With AOP, you define the checkpoint once — an Aspect — and it intercepts all the relevant method calls automatically. Your business methods stay clean; the cross-cutting concern lives in exactly one place.

This also means when you need to change the logging format, you change it in one file, not 80.

🔗 ANALOGY

Highway toll booths vs. tax stickers on every car window

One approach to collecting highway tolls: give every car manufacturer a sticker to put on each new car. When the car passes a road, it somehow pays. Absurd — the manufacturer has nothing to do with tolling.

The real approach: toll booths at specific points that every car passes through. The car doesn't know the booth exists. The booth applies the toll logic consistently. AOP is the toll booth. Your service methods are the cars. The aspect intercepts them at a well-defined point and applies cross-cutting logic (timing, logging, transactions, auth) without the method knowing it's being intercepted.

The cost of not using AOP for cross-cutting concerns: when the security policy changes, you update 80 methods. When you miss one, that's a production security hole.

Technical

AOP Terminology — the four concepts

Aspect — the class containing the cross-cutting logic (e.g., TimingAspect, AuditAspect).

Advice — the action to perform: @Before, @AfterReturning, @AfterThrowing, @After, or @Around (most powerful — wraps the entire call).

Pointcut — the expression that defines which methods to intercept. Can target packages, annotations, or method signatures.

Join Point — the specific method execution being intercepted. In Spring AOP, every join point is a method call.

Caller invokes method → Spring Proxy intercepts → Advice runs (@Before) → Real method executes → Advice runs (@After)

Built-in Spring AOP uses

  • @Transactional → TransactionInterceptor
  • @Cacheable → CacheInterceptor
  • @Async → AsyncExecutionInterceptor
  • @PreAuthorize → MethodSecurityInterceptor
  • @Retryable → RetryOperationsInterceptor

Custom AOP use cases

  • Method execution timing / metrics
  • Audit logging (who called what, when)
  • Rate limiting at method level
  • Distributed tracing (MDC correlation IDs)
  • Feature flag enforcement
💬 Model Interview Answer

"AOP lets me apply cross-cutting concerns — logging, transactions, security, timing — once in a central Aspect rather than repeating them in every method. Spring implements it through proxies: when you annotate a bean method with @Transactional or @Cacheable, Spring wraps the bean in a proxy that intercepts calls and applies the advice. The key limitation is that it only works on calls that enter through the proxy — internal this.method() calls bypass the proxy entirely, which is why @Transactional on a private or internally-called method silently does nothing."

✅ Right — AOP centralizes the concern in one place
// One Aspect replaces all that boilerplate in all 80 methods
@Aspect
@Component
@RequiredArgsConstructor
public class AuditTimingAspect {

    private final MeterRegistry meterRegistry;

    // Pointcut: any method in any class under the service package
    @Pointcut("execution(* com.example.service..*(..))")
    public void serviceLayer() {}

    @Around("serviceLayer()")
    public Object auditAndTime(ProceedingJoinPoint pjp) throws Throwable {
        String methodName = pjp.getSignature().toShortString();
        String user = SecurityContextHolder.getContext().getAuthentication().getName();
        long start = System.currentTimeMillis();

        log.info("AUDIT user={} method={} args={}", user, methodName, pjp.getArgs());
        try {
            Object result = pjp.proceed();  // call the real method
            log.info("AUDIT method={} status=OK duration={}ms", methodName,
                     System.currentTimeMillis() - start);
            return result;
        } catch (Throwable ex) {
            log.error("AUDIT method={} status=ERROR duration={}ms error={}", methodName,
                      System.currentTimeMillis() - start, ex.getMessage());
            throw ex;  // always re-throw — don't swallow exceptions in aspects!
        } finally {
            meterRegistry.timer("service.method.duration", "method", methodName)
                         .record(System.currentTimeMillis() - start, TimeUnit.MILLISECONDS);
        }
    }
}

// OrderService is now clean — zero infrastructure noise
@Service
public class OrderService {
    public Order findById(Long id) { return repository.findById(id).orElseThrow(); }
    public Order save(Order order)  { return repository.save(order); }
}
Warning

Overly broad pointcuts advise Spring internals — and break everything

Writing execution(* *.*(..)) applies your aspect to every method call in the JVM, including Spring's own infrastructure beans. This causes infinite recursion, massively slow startup, or subtle breakage of Spring's transaction and security machinery.

Always scope pointcuts to your own packages: execution(* com.yourcompany..*(..)). Even better, use annotation-based pointcuts: @annotation(com.yourcompany.annotation.Audited) — which only fires for methods you explicitly opt into, giving you precise control with zero risk of advising Spring internals.

Quiz

Which advice type gives you the most control — the ability to modify the return value, suppress exceptions, or even skip calling the target method?

✅ Correct: C — @Around @Around is the most powerful advice type. It receives a ProceedingJoinPoint — you decide whether to call pjp.proceed() (invoke the real method), return a cached value instead, modify the arguments before passing them, change the return value, or swallow/translate exceptions. @Before can abort by throwing but cannot modify return values. @AfterReturning can see the return value but cannot replace it. @After runs like finally — no control over the outcome. All other advice types (@Transactional, @Cacheable, @Async) are implemented internally using @Around.

Proxy Types: JDK Dynamic vs CGLIB

Spring creates a proxy — a wrapper object — around your bean to intercept method calls. The proxy type determines what Spring can and cannot intercept, and knowing this explains many mysterious "why didn't @Transactional work?" bugs.

🧒 ELI5

Stand-in actor vs. a clone of the original actor

Imagine your actor (your bean) has a contract. JDK Dynamic Proxy is hiring a different actor who reads the same script (implements the same interface). They can play the same role — they're a valid substitute — but they're fundamentally a different person. The director (Spring) directs everything through the understudy.

CGLIB is like creating an exact genetic clone of your actor and teaching the clone some extra behaviours. The clone is your actor — same DNA, same mannerisms, just with additional capabilities layered on top. This is subclassing at runtime: CGLIB generates a subclass of your concrete class and overrides methods to inject advice.

The practical implication: if your actor has final methods, the clone inherits them but cannot override them — the extra behaviours cannot be injected into those methods. That's why @Transactional on a final method silently does nothing.

🔗 ANALOGY

Outlet adapter vs. extension cable with surge protection

JDK Proxy = a plug adapter. It works because both the original and the adapter implement the same standard (the interface). Plug the adapter in anywhere the original would fit. But you cannot use an adapter if there is no standard plug shape (no interface).

CGLIB = a surge-protected extension cable wired directly into the wall. It carries all the same electricity (behaviour) from the original outlet (class), adds protection (advice), and works even if there's no standard adapter shape (no interface). But it still can't modify a circuit that was sealed at the factory — a final method or final class.

Technical

How each proxy works under the hood

JDK Dynamic Proxy: Generated at runtime using java.lang.reflect.Proxy. Implements the same interfaces as the target bean. All interface method calls are intercepted; any non-interface methods are invisible to the proxy. Spring was the default before Boot 2.x.

CGLIB Proxy: Generates a subclass of the target class at runtime (using the CGLIB bytecode library, now bundled in Spring Core). Overrides all non-final, non-private methods to inject advice. Spring Boot 2.x made this the default — even when an interface exists.

JDK Proxy constraints

  • Target MUST implement at least one interface
  • Only interface methods are intercepted
  • Slightly faster to create
  • Enable: spring.aop.proxy-target-class=false

CGLIB constraints

  • Target class must NOT be final
  • Intercepted methods must NOT be final
  • Requires a no-arg constructor (or Objenesis)
  • Default in Spring Boot 2.x+
CGLIB: default in Spring Boot JDK: set proxy-target-class=false final method: advice silently skipped final class: cannot proxy at all
💬 Model Interview Answer

"Spring Boot defaults to CGLIB proxies — it subclasses the concrete class at runtime and overrides methods to inject advice. JDK dynamic proxies work by implementing the target's interface and are used when you set proxy-target-class=false. The critical limitation with CGLIB: it cannot override final methods, so @Transactional on a final method is silently ignored — no transaction is started, no error is thrown. @Configuration classes always use CGLIB regardless of this setting, because Spring intercepts @Bean method calls to enforce singleton semantics."

Warning

final method + @Transactional = silent failure, no error thrown

This is one of the hardest production bugs to diagnose because there is absolutely no error. Spring silently skips proxying the final method. Your code runs — but without a transaction. If the method throws midway, nothing rolls back. The data is left in a partially-written state.

❌ Silent failure — @Transactional on final method
@Service
public class PaymentService {

    @Transactional  // ← completely ignored by CGLIB — no warning, no error
    public final void processPayment(Payment payment) {
        accountRepo.debit(payment.getFrom(), payment.getAmount());
        // If this throws, debit() is NOT rolled back — the method is final,
        // CGLIB couldn't override it, so the proxy never wraps this call.
        ledgerRepo.record(payment);
    }
}

// Also broken: final class — entire class cannot be proxied
@Service
public final class InvoiceService {  // ← CGLIB cannot subclass this
    @Transactional                   // ← silently ignored for ALL methods
    public void generate() { ... }
}
✅ Remove final — allow CGLIB to override
@Service
public class PaymentService {        // non-final class ✓

    @Transactional                   // CGLIB overrides this — transaction works ✓
    public void processPayment(Payment payment) {
        accountRepo.debit(payment.getFrom(), payment.getAmount());
        ledgerRepo.record(payment);  // exception here → debit() rolled back ✓
    }
}
Quiz

You annotate a method with @Transactional and make the class final for "security reasons." In a Spring Boot application, what actually happens at runtime?

✅ Correct: B — @Transactional is silently ignored CGLIB works by generating a subclass. Java's final keyword prevents subclassing — CGLIB cannot create the proxy. Spring Boot does not throw an error; it falls back to using the original, unproxied bean. @Transactional on its methods is therefore completely inert. Option A is wrong — Spring registers the bean fine, it just can't proxy it. Option C is wrong — switching to JDK dynamic proxy requires an interface AND proxy-target-class=false; Spring doesn't switch automatically. Option D is wrong — Spring uses runtime proxy-based AOP (not AspectJ compile-time weaving) by default, which has no solution for final classes.

Self-invocation & AOP Limitations

The single most common AOP bug in production codebases. When a method calls another method on the same object, the proxy is bypassed — @Transactional, @Cacheable, @Async, @Retryable all silently do nothing.

🧒 ELI5

The security guard who only checks people coming through the front door

Your office building has a security guard at the front entrance. The guard checks every visitor's badge before letting them in. But there's a side corridor inside the building — once you're already inside, you can walk freely from room to room without ever passing through the guard again.

Spring's proxy is that guard. It intercepts calls coming from outside your class — other beans calling your method. But if you're already inside your class and call this.method(), you're using the side corridor. The guard (proxy) is never triggered. @Transactional on that method? Never activated. @Cacheable? Never checked. This is not a bug — it's exactly how proxy-based AOP works by design.

🔗 ANALOGY

Speed camera on the motorway — only works at the entry ramp

A speed camera measures cars entering the motorway at a specific ramp. Cars already on the motorway switching lanes internally are invisible to it. The camera is the proxy — it only sees traffic that enters through the monitored ramp (external method calls crossing a proxy boundary).

A self-invocation call is a car that's already on the motorway moving between lanes — the speed camera (proxy) at the ramp never sees it. You cannot fix this by adding more cameras on the ramp — you need a camera further along the motorway (a separate Spring bean with its own proxy), so traffic that goes through that bean's entry point gets measured.

Technical

Why it happens — the call path

When an external bean calls your service, the call hits the proxy object first. The proxy applies the advice (opens a transaction, checks cache, etc.) and then delegates to the real target object.

When your service calls this.anotherMethod(), it calls directly on the target object — bypassing the proxy entirely. The reference this inside your class is always the real object, never the proxy.

External call → Proxy → Advice applied ✓ → Target.method()
this.method() (internal) → Target directly ✗ → No advice!

Annotations affected

  • @Transactional — no transaction started
  • @Cacheable — cache never checked
  • @Async — runs synchronously in same thread
  • @Retryable — no retry on exception
  • @PreAuthorize — security check skipped

Fix options (ranked)

  • 1. Extract method to a separate @Service bean
  • 2. Inject self via @Autowired @Lazy
  • 3. Use ApplicationContext.getBean()
  • 4. Refactor to avoid the self-call entirely
  • 5. AspectJ compile-time weaving (complex)
💬 Model Interview Answer

"Self-invocation bypasses Spring's AOP proxy because the proxy only intercepts calls that arrive from external beans. When you call this.method() inside the same class, you're calling on the real object — the proxy is never in the call path. The cleanest fix is extracting the annotated method to a separate Spring-managed bean, so the call crosses a proxy boundary. In a pinch, injecting the bean into itself via @Autowired @Lazy works, but it's a code smell that usually signals the class is doing too much."

Warning

The @Transactional self-call is the #1 undetected production data integrity bug

Unit tests usually mock the repository layer, so they pass regardless of whether the transaction actually started. Integration tests rarely verify transaction propagation specifically. This means the bug ships to production, where two concurrent requests corrupt the same data because one of them wasn't actually running in an isolated transaction. The failure appears as a random data inconsistency weeks after deployment, with no obvious cause.

Always review code for the pattern: a public @Transactional method calling another method on this that also has @Transactional. That inner annotation is dead code. Add a code review checklist item for this pattern — it pays for itself quickly.

Quiz

A service method annotated with @Async calls this.sendNotification() internally. sendNotification() is also in the same class. What happens when the outer method is invoked?

✅ Correct: D — runs synchronously, @Async is bypassed @Async works via a Spring AOP proxy that intercepts the method call and submits it to a TaskExecutor thread pool instead of running it in the caller's thread. When you call this.sendNotification(), you invoke it directly on the real object, bypassing the AsyncExecutionInterceptor proxy entirely. The method runs synchronously in the calling thread — the caller blocks. No new thread is created. No exception is thrown. This is particularly nasty because it's invisible: the feature "works" (the notification is sent) but loses the non-blocking property entirely, causing unexpected latency in the calling request. Fix: move sendNotification() to a separate @Service bean.

Common Spring AOP Use Cases

AOP powers most of Spring's declarative programming model. Understanding which annotations use proxies lets you debug them correctly and build your own aspects with confidence.

🧒 ELI5

Invisible assistants — Spring annotations are just named proxies

Every time you write @Transactional, @Cacheable, or @Async, you're not using magic — you're asking Spring to assign an invisible assistant to that method. When the method is called, the assistant runs first (opens a transaction, checks the cache), lets your method do its work, then cleans up (commits, stores in cache).

Understanding that these are all just proxy-based aspects demystifies them completely. You can reason about their limitations (self-invocation, final methods), predict their ordering (which assistant goes first when multiple annotations apply?), and build your own assistants using the same mechanism.

Technical — Custom Annotation + Aspect Pattern

Building your own AOP annotation

The same mechanism Spring uses for @Transactional is available to you. Define a custom annotation, write an @Aspect that targets it, and any method annotated with your annotation automatically gets the cross-cutting behaviour.

@Around — most common

  • Full control: proceed or not
  • Modify args before calling target
  • Modify/replace return value
  • Catch and translate exceptions

Aspect ordering (@Order)

  • Lower number = outer wrapper
  • Security: @Order(0) — outermost
  • Transactions: @Order(1)
  • Your custom: @Order(2)
  • No @Order = undefined order
💬 Model Interview Answer

"I've used custom AOP aspects for audit logging — a @AuditLog annotation on service methods that captures the caller's identity, method name, arguments, and outcome in a single centralised aspect. The key design decisions: use @annotation() pointcut so only explicitly opted-in methods are advised, always re-throw exceptions from @Around advice, and set @Order explicitly so the audit aspect runs after security (you want the authenticated user available in the audit record)."

Custom @AuditLog annotation + aspect — production pattern
// Step 1: Define the annotation
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)  // must be RUNTIME — not CLASS or SOURCE
public @interface AuditLog {
    String action();                 // e.g., "TRANSFER_FUNDS", "DELETE_ACCOUNT"
    boolean logArgs = true;          // opt out for sensitive data like passwords
}

// Step 2: Write the aspect
@Aspect
@Component
@Order(2)  // runs inside security (@Order(0)) and transaction (@Order(1)) aspects
@RequiredArgsConstructor
public class AuditLogAspect {

    private final AuditRepository auditRepo;

    // @annotation() pointcut — only fires for explicitly annotated methods
    @Around("@annotation(auditLog)")
    public Object audit(ProceedingJoinPoint pjp, AuditLog auditLog) throws Throwable {
        String principal = SecurityContextHolder.getContext()
                                               .getAuthentication().getName();
        String method = pjp.getSignature().toShortString();
        Object[] args = auditLog.logArgs() ? pjp.getArgs() : new Object[]{"[redacted]"};

        try {
            Object result = pjp.proceed();  // call the real method
            // Success: save the audit record (in a REQUIRES_NEW tx — must persist
            // even if the outer transaction rolls back)
            auditRepo.save(AuditEntry.success(principal, auditLog.action(), method, args));
            return result;
        } catch (Throwable ex) {
            auditRepo.save(AuditEntry.failure(principal, auditLog.action(), method,
                                              args, ex.getMessage()));
            throw ex;  // ALWAYS re-throw — aspects must not silently swallow exceptions
        }
    }
}

// Step 3: Use it — clean business method, audit handled automatically
@Service
public class AccountService {

    @AuditLog(action = "TRANSFER_FUNDS")
    @Transactional
    public TransferResult transfer(Long from, Long to, BigDecimal amount) {
        // No audit boilerplate here — the aspect handles it
        accountRepo.debit(from, amount);
        accountRepo.credit(to, amount);
        return new TransferResult(from, to, amount);
    }

    @AuditLog(action = "CHANGE_PASSWORD", logArgs = false)  // don't log the password
    public void changePassword(Long userId, String newPassword) {
        userRepo.updatePassword(userId, passwordEncoder.encode(newPassword));
    }
}

// Conceptual nesting when all three apply to the same method:
// @Order(0): SecurityAspect.before()  — check if user has permission FIRST
//   @Order(1): TransactionAspect.begin() — open DB tx
//     @Order(2): AuditLogAspect.before() — record audit with authenticated user ✓
//       Target method executes
//     @Order(2): AuditLogAspect.after() — record outcome
//   @Order(1): TransactionAspect.commit() / rollback()
// @Order(0): SecurityAspect.after()

// If security ran INSIDE transaction (@Order(1) < @Order(0)):
// Transaction opens → Security check → may rollback tx — inefficient
// Correct ordering: Security outermost → fail fast before DB resources allocated

// Named pointcut composition for reuse:
@Aspect
@Component
public class ServiceAspects {

    @Pointcut("execution(* com.example.service..*(..))")
    public void inServiceLayer() {}

    @Pointcut("@annotation(org.springframework.transaction.annotation.Transactional)")
    public void isTransactional() {}

    @Pointcut("inServiceLayer() && isTransactional()")
    public void transactionalServiceMethod() {}

    // Use composed pointcut
    @Before("transactionalServiceMethod()")
    public void logTransactionalServiceEntry(JoinPoint jp) {
        log.debug("Entering transactional service method: {}", jp.getSignature());
    }
}
Warning

Never silently swallow exceptions inside @Around advice

The most dangerous @Around mistake: catching an exception and returning null or a default value instead of re-throwing. The caller has no idea the method failed. Transactions don't roll back (RuntimeException is what triggers rollback). Callers receive corrupt or empty data and proceed as if everything worked.

❌ Never do this — silently swallowing exceptions
@Around("execution(* com.example.service..*(..))")
public Object dangerousAround(ProceedingJoinPoint pjp) {
    try {
        return pjp.proceed();
    } catch (Throwable ex) {
        log.error("Error in {}: {}", pjp.getSignature(), ex.getMessage());
        return null;  // ← SILENT FAILURE: caller gets null, thinks it worked
        // @Transactional never sees the exception — no rollback happens
        // Database is left in a partially-written state
    }
}
✅ Always re-throw (or wrap in a more specific exception)
@Around("execution(* com.example.service..*(..))")
public Object safeAround(ProceedingJoinPoint pjp) throws Throwable {
    try {
        return pjp.proceed();
    } catch (Throwable ex) {
        log.error("Error in {}: {}", pjp.getSignature(), ex.getMessage());
        // Option 1: re-throw as-is
        throw ex;
        // Option 2: wrap in domain exception (preserve stacktrace)
        // throw new ServiceException("Failed in " + pjp.getSignature(), ex);
    }
}
Quiz

You have two aspects on the same method: SecurityAspect (@Order(1)) and TransactionAspect (@Order(2)). In what order do they run?

✅ Correct: A — lower @Order number = outer wrapper Spring AOP nesting follows the @Order convention: lower numbers run as the outer wrapper. @Order(1) SecurityAspect wraps @Order(2) TransactionAspect, which wraps the target method. For @Before advice this means SecurityAspect.before() runs first, then TransactionAspect.before(), then the method. For @After it's reversed: TransactionAspect.after(), then SecurityAspect.after(). This is intentional for security: you want security enforcement outermost so a rejected call fails fast without opening a database transaction. Option D is wrong — @EnableAspectJAutoProxy enables AOP support but the 'order=true' attribute doesn't exist; @Order on the Aspect class is what controls nesting.

Deep Dive: Proxy Boundaries in Practice

The patterns senior engineers reach for when vanilla AOP hits its limits — pointcut composition, @Transactional + AOP interaction, and knowing when Spring AOP is not enough.

🧒 ELI5

Knowing where the walls of your room are

Spring AOP is powerful but has clear walls: it only works across proxy boundaries, only on Spring-managed beans, and only on non-final, non-private methods. A senior engineer knows exactly where those walls are — not from memorising rules, but from understanding the proxy mechanism. When something's not working, they can immediately diagnose: "Is this a self-invocation? Is the class or method final? Is this bean even Spring-managed?"

This section is about building that diagnostic muscle and knowing what to reach for when you hit a wall.

Technical — AOP Limitations Diagnostic Checklist

When your aspect or annotation silently fails — run this checklist

Check 1: Proxy boundary

  • Is the calling code external to the bean?
  • Or is it this.method() (self-invocation)?
  • Fix: extract to separate @Service bean

Check 2: final modifier

  • Is the class final?
  • Is the method final?
  • Fix: remove final

Check 3: Bean registration

  • Is the class a Spring bean (@Service/@Component)?
  • Instantiated via new()? → not proxied
  • Fix: let Spring create it

Check 4: Method visibility

  • Is the method private?
  • Spring AOP cannot intercept private methods
  • Fix: make public or protected
💬 Model Interview Answer

"When an AOP annotation silently fails, I run through four checks in order: self-invocation (this.method() bypasses the proxy), final class or method (CGLIB can't subclass or override), not a Spring bean (instantiated via new()), or a private method (invisible to proxies). For the rare case where I genuinely need to advise private methods or self-invocations — like advising domain objects not managed by Spring — I'd consider AspectJ compile-time or load-time weaving, but that's a significantly more complex setup I'd only add if the design truly requires it."

@Configuration CGLIB — why calling @Bean methods works correctly
// @Configuration classes ALWAYS use CGLIB — this is why:
@Configuration
public class AppConfig {

    @Bean
    public DataSource dataSource() {
        return new HikariDataSource(hikariConfig());
    }

    @Bean
    public UserRepository userRepository() {
        return new UserRepositoryImpl(dataSource()); // calls dataSource() directly!
    }

    @Bean
    public OrderRepository orderRepository() {
        return new OrderRepositoryImpl(dataSource()); // also calls dataSource()!
    }
}

// Without CGLIB: dataSource() is called THREE TIMES → THREE HikariDataSource instances.
// That's a connection pool catastrophe: 3 pools instead of 1.

// With CGLIB (@Configuration's default): Spring subclasses AppConfig.
// Every @Bean method call is intercepted. The CGLIB proxy checks: "Has dataSource()
// already been created?" If yes, return the cached singleton from ApplicationContext.
// If no, call the real method and cache the result.
// Result: dataSource() method body runs once — one HikariDataSource — regardless
// of how many times other @Bean methods call dataSource().

// @Component + @Bean (Lite Mode) — NO CGLIB interception:
@Component  // ← NOT @Configuration
public class LiteConfig {

    @Bean
    public DataSource dataSource() { return new HikariDataSource(hikariConfig()); }

    @Bean
    public UserRepository userRepository() {
        return new UserRepositoryImpl(dataSource()); // ← REAL Java call, no interception
        // Creates a SECOND HikariDataSource! This is the lite-mode trap.
    }
}

// Rule: if @Bean methods call each other, use @Configuration (never @Component).
Spring AOP vs AspectJ — when to consider the difference
// Spring AOP: proxy-based, runtime, only Spring beans, only method calls
// ✓ Zero configuration overhead
// ✓ Works with 99% of production use cases
// ✗ Self-invocation not advised
// ✗ Private methods not advised
// ✗ Non-Spring-managed objects not advised

// AspectJ: bytecode weaving, compile-time or load-time
// ✓ Advises ANY Java code, including private methods and self-calls
// ✓ Works on objects not managed by Spring
// ✗ Requires AspectJ compiler plugin (compile-time) or Java agent (load-time)
// ✗ Significantly more complex setup and debugging

// When would you choose AspectJ in Spring?
// 1. You need to advise domain objects not in the Spring context
// 2. You must intercept self-invocations and extracting to a separate bean
//    would fundamentally break the design
// 3. You need to advise constructors (Spring AOP cannot do this)

// Enable load-time weaving in Spring:
// @EnableLoadTimeWeaving // in a @Configuration class
// + -javaagent:spring-instrument.jar JVM arg
// + META-INF/aop.xml for AspectJ configuration
Warning

Objects created with new() are NEVER proxied — even if the class is @Service

The @Service/@Component annotation on a class tells Spring to create and manage an instance of that class. It does not affect instances you create yourself with new. An object created via new MyService() is just a plain Java object — no proxy, no DI, no AOP, even if the class has @Transactional or @Cacheable annotations.

❌ new() bypasses Spring entirely
@Service
public class ReportController {

    public void generateReport() {
        // Creating via new() — this is NOT the Spring-managed bean!
        // No proxy, no @Transactional, no @Cacheable, no DI injection.
        ReportService svc = new ReportService();  // plain Java object
        svc.generate();  // @Transactional on generate() does NOTHING here
    }
}

// ✅ Always use DI — let Spring create and inject the proxied bean
@Service
@RequiredArgsConstructor
public class ReportController {

    private final ReportService reportService;  // Spring-injected, proxied ✓

    public void generateReport() {
        reportService.generate();  // goes through proxy → @Transactional works ✓
    }
}
Quiz

In a @Configuration class, method A() is annotated @Bean and calls method B() which is also @Bean. How many instances of B's bean are created if A() and one other @Bean method both call B()?

✅ Correct: A — one instance, thanks to CGLIB interception @Configuration classes are always CGLIB-proxied. Every @Bean method call on the config object is intercepted by the proxy. The proxy checks whether that bean already exists in the ApplicationContext. If yes — return the existing singleton. If no — call the real method body and register the result. This is why calling dataSource() three times in a @Configuration class creates exactly one DataSource. If you change @Configuration to @Component (Lite Mode), CGLIB is not used, each call to dataSource() executes the method body — creating three separate DataSource objects. This is a famous Spring interview trap.

🎉 AOP & Proxies Cheat Sheet

Everything you need to recall in interviews and code reviews — the mechanisms, the traps, and the fixes.

Cheat Sheet
⚙️

Why AOP exists

Apply cross-cutting concerns (transactions, logging, security, timing) to many methods without repeating the code in each one. One Aspect, zero duplication.

JDK vs CGLIB proxy

CGLIB is the Spring Boot default — subclasses your class at runtime. JDK proxy implements your interface. Both fail silently on final methods. CGLIB requires non-final class + non-final methods.

Self-invocation trap

this.method() bypasses the proxy. @Transactional, @Cacheable, @Async, @Retryable — all silently do nothing. Fix: extract the annotated method to a separate @Service bean.

Advice types

@Around = full control (proceed or not, modify return). @Before = pre-execution. @AfterReturning = post-success. @AfterThrowing = on exception. @After = always (finally).

@Order = nesting depth

Lower @Order number = outer wrapper. Security @Order(0) wraps Transactions @Order(1) wraps your custom aspect. Always set @Order explicitly on production aspects.

@Configuration vs @Component + @Bean

@Configuration uses CGLIB — @Bean method calls return the ApplicationContext singleton. @Component (Lite Mode) has no CGLIB — each @Bean call creates a new instance. Never use @Component with cross-referencing @Bean methods.

Common traps

Mistakes that trip up senior engineers

@Transactional / @Async on final method

CGLIB cannot override final methods — the annotation is silently ignored. No error, no warning. The feature simply doesn't work. Remove final.

Self-invocation (this.method())

The proxy is bypassed. @Transactional on the called method does nothing. This is the #1 undetected transaction bug. Always call through a separate injected bean.

Swallowing exceptions in @Around advice

Catching Throwable and returning null means callers get corrupt data, transactions don't rollback, and failures are invisible. Always re-throw from @Around advice.

Overly broad pointcut (execution(* *.*(..)))

Advises Spring's own beans — causes infinite loops, slow startup, or subtle infrastructure breakage. Always scope to your own packages or use @annotation() pointcuts.

new MyService() vs @Autowired

Objects created with new() are not Spring-managed — no proxy, no DI, no AOP. Always let Spring inject the proxied instance.
