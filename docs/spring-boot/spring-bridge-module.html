@Transactional — Deep Dive
🍅 Pomodoro · 30 min
🧒 ELI5 — The ATM Transaction

You're at an ATM transferring $100 from checking to savings. Two things happen: (1) subtract $100 from checking, (2) add $100 to savings. If the machine crashes between step 1 and 2 — your money is gone. A transaction says: either BOTH steps happen, or NEITHER does. If anything goes wrong, we undo everything back to the starting state. That "undo" is called a rollback.

@Transactional is the annotation that tells Spring: "wrap this method in that ATM guarantee."

⚙️ How It Works Under the Hood

When Spring sees @Transactional on a bean, it creates an AOP proxy (via CGLIB or JDK proxy) that wraps the method. Before your code runs, the proxy opens a DB transaction. After your code, it commits. If an exception escapes, it rolls back.

// What Spring does at runtime (conceptually):
try {
    txManager.beginTransaction();
    yourService.actualMethod(); // your real code
    txManager.commit();
} catch (RuntimeException e) {
    txManager.rollback();
    throw e;
}

Propagation — The 2 You Must Know Cold

PropagationBehaviorUse Case
REQUIRED (default) Join existing tx if one exists; create new one if not 99% of your service methods
REQUIRES_NEW Always create a new tx; suspend the current one Audit logging — must persist even if outer tx rolls back
SUPPORTS Join if exists; run non-transactionally if not Read-only methods that can work either way
NOT_SUPPORTED Suspend existing tx; run non-transactionally Batch ops that shouldn't hold a tx open
NEVER Throw if a tx exists Methods that must never run in a transaction
MANDATORY Throw if no tx exists Methods that require a caller-managed tx 
@Service
public class TransferService {

    @Transactional // REQUIRED — joins outer tx or creates one
    public void transfer(long fromId, long toId, BigDecimal amount) {
        accountRepo.debit(fromId, amount);
        auditService.log(fromId, toId, amount); // see below
        accountRepo.credit(toId, amount);
    }
}

@Service
public class AuditService {

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void log(long fromId, long toId, BigDecimal amount) {
        // Runs in its OWN transaction
        // Even if transfer() rolls back, this audit row is saved
        auditRepo.save(new AuditLog(fromId, toId, amount));
    }
}

Rollback Rules

By default, Spring only rolls back on unchecked exceptions (RuntimeException and its subclasses) and Errors. Checked exceptions do NOT trigger rollback unless you configure it.

// Default: rollback on RuntimeException only
@Transactional
public void doWork() throws IOException {
    // IOException is CHECKED — NO rollback by default!
}

// Explicit rollback for checked exception:
@Transactional(rollbackFor = IOException.class)
public void doWork() throws IOException { ... }

// Prevent rollback even for RuntimeException:
@Transactional(noRollbackFor = OptimisticLockException.class)
public void doWork() { ... }
⚠️ The Self-Invocation Trap — Most Common @Transactional Bug

If you call a @Transactional method from within the same class, the proxy is bypassed — no transaction is started.

@Service
public class OrderService {

    public void placeOrder(Order o) {
        // Calls saveOrder() on 'this' — bypasses the proxy!
        saveOrder(o); // ← NO transaction!
    }

    @Transactional
    public void saveOrder(Order o) { ... }
}

Fix: Either merge the methods, or inject self and call through the proxy:

@Service
public class OrderService {

    @Autowired
    private OrderService self; // inject proxy of self

    public void placeOrder(Order o) {
        self.saveOrder(o); // goes through proxy ✓
    }

    @Transactional
    public void saveOrder(Order o) { ... }
}
⚙️ readOnly=true — Performance Hint
@Transactional(readOnly = true)
public List<Account> findAll() {
    return accountRepo.findAll();
}

With readOnly=true, Spring hints to the DB driver and Hibernate that no writes will happen. Hibernate can skip dirty-checking on all loaded entities — measurable speedup for large result sets.

💬 Interview Answer

"@Transactional works via an AOP proxy — Spring wraps the bean in a proxy that opens a DB transaction before the method and commits on success, rolls back on unchecked exceptions. Key gotchas: self-invocation bypasses the proxy so no transaction starts; checked exceptions don't trigger rollback by default — you need rollbackFor; and REQUIRES_NEW creates an independent transaction useful for audit logs that must persist even if the outer transaction rolls back."

🧠 Quiz — Section 1
A @Service method with @Transactional throws a checked IOException. What happens to the transaction by default?
✅ Spring's default rollback policy is: roll back on RuntimeException and Error only. Checked exceptions (like IOException, SQLException) are committed unless you explicitly configure rollbackFor = IOException.class. This trips up many developers who expect "any exception = rollback."

@ExceptionHandler + @RestControllerAdvice
🍅 Pomodoro · 25 min
🧒 ELI5 — The Customer Service Desk

Imagine every department in a store just throws their problems into the hallway when something goes wrong. Chaos. Instead, you have a central customer service desk at the entrance — all complaints go there, get sorted by type, and receive a professional response.

@RestControllerAdvice is that central desk. @ExceptionHandler is the staff member trained for a specific type of complaint. No matter which controller throws the exception, it lands at that desk.

⚙️ Without Global Handling — The Problem
@RestController
public class AccountController {

    @GetMapping("/accounts/{id}")
    public Account getAccount(@PathVariable long id) {
        return service.findById(id); // throws AccountNotFoundException
        // Without handling: Spring returns ugly 500 white-label page
    }
}

Without error handling, Spring returns a generic 500 response. Every controller would need its own try-catch — tons of duplication.

⚙️ The Solution: Global Exception Handler
// 1. Custom exception
public class AccountNotFoundException extends RuntimeException {
    public AccountNotFoundException(long id) {
        super("Account not found: " + id);
    }
}

// 2. Standard error response body
public record ErrorResponse(String error, String message, int status) {}

// 3. Global handler — applies to ALL @RestControllers
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(AccountNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ErrorResponse handleNotFound(AccountNotFoundException ex) {
        return new ErrorResponse("NOT_FOUND", ex.getMessage(), 404);
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public ErrorResponse handleValidation(MethodArgumentNotValidException ex) {
        String msg = ex.getBindingResult().getFieldErrors().stream()
            .map(f -> f.getField() + ": " + f.getDefaultMessage())
            .collect(Collectors.joining(", "));
        return new ErrorResponse("VALIDATION_FAILED", msg, 400);
    }

    @ExceptionHandler(Exception.class) // catch-all fallback
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    public ErrorResponse handleAll(Exception ex) {
        return new ErrorResponse("INTERNAL_ERROR", "Unexpected error", 500);
    }
}

Using ResponseEntity for Full Control

@ExceptionHandler(AccountNotFoundException.class)
public ResponseEntity<ErrorResponse> handleNotFound(AccountNotFoundException ex) {
    var body = new ErrorResponse("NOT_FOUND", ex.getMessage(), 404);
    return ResponseEntity
        .status(HttpStatus.NOT_FOUND)
        .header("X-Error-Code", "ACC-001")
        .body(body);
}

Use ResponseEntity when you need to control response headers, not just body and status.

@RestControllerAdvice vs @ControllerAdvice

AnnotationEquivalent ToResponse Type
@ControllerAdvice Base annotation Returns view names by default
@RestControllerAdvice @ControllerAdvice + @ResponseBody Serializes return value to JSON/XML directly

Scope Control — Limit to Specific Packages

// Only handle exceptions from these controllers:
@RestControllerAdvice(basePackages = "com.myapp.api")
public class ApiExceptionHandler { ... }

// Or specific controller classes:
@RestControllerAdvice(assignableTypes = {AccountController.class})
public class AccountExceptionHandler { ... }
💬 Interview Answer

"@RestControllerAdvice is a global exception interceptor — it's @ControllerAdvice + @ResponseBody, applied across all @RestControllers. @ExceptionHandler methods inside it map specific exception types to HTTP responses. This eliminates try-catch duplication across controllers and gives you a single place to define error contracts. For full response control including headers, return ResponseEntity instead of using @ResponseStatus."

⚠️ Exception Handler Priority

When multiple handlers match an exception, Spring picks the most specific type. If you have handlers for both RuntimeException and AccountNotFoundException extends RuntimeException, the AccountNotFoundException handler wins.

The catch-all @ExceptionHandler(Exception.class) should always be last-resort — it catches everything including Spring internals you might not want to intercept.

🧠 Quiz — Section 2
Your @RestControllerAdvice has handlers for both RuntimeException and IllegalArgumentException (which extends RuntimeException). A controller throws IllegalArgumentException. Which handler runs?
✅ Spring resolves exception handlers by specificity — the most specific matching type wins. IllegalArgumentException is a more specific match than RuntimeException, so that handler is chosen. This lets you have fine-grained control per exception type with a broader fallback.

@Value, @ConfigurationProperties & Profiles
🍅 Pomodoro · 20 min
🧒 ELI5 — The Settings File

Think of a game with a settings.cfg file — resolution, sound volume, difficulty. Your game code doesn't hard-code those values; it reads from the file. That way you can change behavior without recompiling. Spring's property system does the same: application.properties is your settings file, @Value is how you read one setting, and @ConfigurationProperties is how you read a whole group of related settings at once.

Profiles are like having separate settings files per environment: settings-dev.cfg, settings-prod.cfg.

⚙️ @Value — Single Property Injection
# application.properties
app.max-retries=3
app.api-key=abc-123
app.timeout-ms=5000
@Service
public class PaymentService {

    @Value("${app.max-retries}")
    private int maxRetries;

    @Value("${app.api-key}")
    private String apiKey;

    // Default value if property not found:
    @Value("${app.timeout-ms:3000}")
    private int timeoutMs; // 3000 if not in properties

    // SpEL expression:
    @Value("#{${app.max-retries} * 2}")
    private int doubleRetries;
}

Note: @Value uses ${...} for property placeholders and #{...} for SpEL (Spring Expression Language). Don't mix them up in interviews.

⚙️ @ConfigurationProperties — The Better Way for Groups

For more than 2-3 related properties, @ConfigurationProperties is strongly preferred over multiple @Value annotations. It's type-safe, validates, and is IDE-friendly.

# application.properties
payment.gateway.url=https://api.stripe.com
payment.gateway.api-key=sk_live_xxx
payment.gateway.timeout-ms=5000
payment.gateway.max-retries=3
@ConfigurationProperties(prefix = "payment.gateway")
@Component // or use @EnableConfigurationProperties in config class
public class PaymentGatewayProperties {

    private String url;
    private String apiKey;
    private int timeoutMs;
    private int maxRetries;

    // getters + setters (or use record / @ConstructorBinding)
}

// Usage:
@Service
public class PaymentService {

    private final PaymentGatewayProperties props;

    public PaymentService(PaymentGatewayProperties props) {
        this.props = props;
    }
}

Adding Validation to Properties

@ConfigurationProperties(prefix = "payment.gateway")
@Validated
public class PaymentGatewayProperties {

    @NotNull
    private String url;

    @Min(1) @Max(10)
    private int maxRetries;
    // App fails at startup if constraints violated — fail fast!
}

⚙️ Profiles — Environment-Specific Config
# application.properties (base config)
spring.application.name=bank-service

# application-dev.properties (overrides for dev)
spring.datasource.url=jdbc:h2:mem:devdb
app.debug-mode=true

# application-prod.properties (overrides for prod)
spring.datasource.url=jdbc:postgresql://prod-host:5432/bank
app.debug-mode=false
// Activate profile:
// 1. In application.properties:
spring.profiles.active=dev

// 2. JVM arg:
// -Dspring.profiles.active=prod

// 3. Environment variable:
// SPRING_PROFILES_ACTIVE=prod

@Profile on Beans

@Configuration
public class DataSourceConfig {

    @Bean
    @Profile("dev")
    public DataSource h2DataSource() {
        return new EmbeddedDatabaseBuilder().setType(H2).build();
    }

    @Bean
    @Profile("prod")
    public DataSource postgresDataSource() {
        return new HikariDataSource(...);
    }

    @Bean
    @Profile("!test") // negation: active when NOT test
    public SchedulerService scheduler() { ... }
}
💬 Interview Answer

"@Value injects individual properties using ${placeholder} syntax and supports SpEL with #{expression}. For groups of related properties, @ConfigurationProperties is preferred — it's type-safe, supports validation with @Validated, and IDE autocomplete. Profiles allow environment-specific configuration: application-dev.properties overrides base properties when the dev profile is active. Beans can be conditionally registered with @Profile, enabling different implementations per environment."

🧠 Quiz — Section 3
You have 8 related payment config properties. Which approach is best and why?
✅ @ConfigurationProperties wins for groups of related properties. It binds all matching prefixed properties to a typed POJO, supports @Validated constraints (failing fast at startup if invalid), offers IDE autocomplete via spring-boot-configuration-processor, and handles relaxed binding — payment.gateway.api-key maps to apiKey automatically.
