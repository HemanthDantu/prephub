Scheduling Basics

Spring scheduling runs annotated methods on a timer using fixed rate, fixed delay, or cron expressions.

← Back to Spring Boot
🧒 ELI5

Alarm clocks

You set alarms to go off at a certain time or every few minutes.

🔗 ANALOGY

Train timetable

Trains leave at fixed times so everyone knows the schedule.

Technical

What it means in real code

Definition: @Scheduled runs methods at intervals or cron expressions using a TaskScheduler.

@Scheduled supports fixedRate, fixedDelay, and cron, each with different timing semantics.

Use cron for predictable schedules and fixedDelay when you must wait for completion before the next run.

Beginners often choose fixedRate by default, but fixedDelay is safer when tasks can run longer than expected.

Why it matters

  • Automate recurring work
  • Clean timing semantics
  • Simple annotations

Common pitfall

  • Manual timers
  • Hard-coded loops
  • No visibility into schedule

Warning

Cron timezones

Cron defaults to server timezone. Set it explicitly if your app spans regions.

Mini takeaway
  • Pick the right schedule type.
  • Cron for calendars.
  • Understand rate vs delay.
Quiz

Correct answer: B fixedDelay schedules the next run after completion.
Section 2 of 5 · ~7 mins

Thread Pools and Execution

By default, scheduled tasks run on a single thread. Configure a TaskScheduler for concurrency.

🧒 ELI5

Multiple alarm clocks

If one alarm is busy, another clock should still ring.

🔗 ANALOGY

More kitchen staff

With more cooks, multiple dishes can be prepared at once.

Technical

What it means in real code

Definition: TaskScheduler controls the thread pool used for @Scheduled tasks and avoids blocking.

By default, Spring uses a single-threaded scheduler, so long jobs block other tasks.

A dedicated TaskScheduler with a thread pool allows parallel jobs and isolates slow tasks.

If you see one job delaying another, that is a signal to add a pool and separate workloads.

Why it matters

  • Prevents backlog
  • Supports concurrency
  • Improves reliability

Common pitfall

  • Single-thread bottleneck
  • Long-running tasks
  • Unbounded delays
TaskSchedulerPool sizeConcurrency
Schedule task→Dispatch to pool→Execute→Return thread

Interview answer

“I configure ThreadPoolTaskScheduler so long tasks do not block other scheduled jobs.”

SettingEffect
poolSizeParallel tasks
threadNamePrefixDebugging
awaitTerminationGraceful shutdown
Scheduler configuration
@Configuration
@EnableScheduling
public class SchedulingConfig {
  @Bean
  public TaskScheduler taskScheduler() {
    ThreadPoolTaskScheduler scheduler = new ThreadPoolTaskScheduler();
    scheduler.setPoolSize(4);
    scheduler.setThreadNamePrefix("scheduler-");
    return scheduler;
  }
}
Warning

Blocking tasks block the pool

If scheduled tasks block on I/O, you may exhaust scheduler threads.

Mini takeaway
  • Configure a pool for concurrency.
  • Name threads for debugging.
  • Avoid blocking in schedules.
Quiz

Why configure a ThreadPoolTaskScheduler?

Correct answer: B A thread pool prevents one task from blocking all others.
Section 3 of 5 · ~7 mins

Reliability and Overlap

Long-running jobs can overlap or drift. Design schedules to avoid duplicate work.

🧒 ELI5

Do not double-clean

If the floor is still being mopped, do not start another mop on top of it.

🔗 ANALOGY

Assembly line timing

If one station is slow, you need to adjust the whole line to avoid pileups.

Technical

What it means in real code

Definition: Overlapping schedules can cause duplicate work; use locks or adjust delay to fit task duration.

Overlapping runs can corrupt state or overwhelm systems; guard with locks or conditional checks.

Always catch exceptions inside jobs so a single failure does not stop future executions.

Monitoring job duration helps you detect drift and adjust schedules before failures happen.

Why it matters

  • Avoid double processing
  • Protect shared resources
  • Reduce contention

Common pitfall

  • Overlapping runs
  • No guardrails
  • Repeated failures
OverlapLocksBackoff

Overlaps cause duplicate side effects

Overlapping jobs can double-send emails, double-charge, or corrupt state.

Mini takeaway
  • Avoid overlapping jobs.
  • Use fixedDelay for long work.
  • Guard with locks if needed.

Section 4 of 5 · ~7 mins

Distributed Scheduling

In multi-instance deployments, you need to ensure only one node runs a scheduled job.

🧒 ELI5

One chef per kitchen

Only one chef should start the soup; two chefs will spoil it.

🔗 ANALOGY

Passing the baton

Runners take turns; only one holds the baton at a time.

Technical

What it means in real code

Definition: Distributed locks or external schedulers ensure a job runs once across a cluster.

In distributed deployments, each instance will run the same job unless you coordinate.

Leader election or libraries like ShedLock ensure only one instance runs a scheduled task.

Design scheduled work to be idempotent so duplicates are safe if coordination fails.

Why it matters

  • Avoid duplicate runs
  • Support scaling
  • Centralized scheduling

Common pitfall

  • Same job running on every node
  • No locking mechanism
  • Inconsistent results
ShedLockQuartzExternal scheduler

ShedLock example
@SchedulerLock(name = "cleanupJob", lockAtMostFor = "10m")

Warning

No lock means duplicate work

Without coordination, every node will run the job and you will see duplicates.

Mini takeaway
  • Use locks in clusters.
  • Avoid duplicate execution.
  • Pick a scheduler appropriate to complexity.
Quiz

How do you prevent a scheduled job from running on every node?

Correct answer: B A distributed lock or external scheduler ensures only one node runs the job.

Edge cases

Edge Cases & Tradeoffs

  • Long tasks can overlap without safeguards.
  • Default scheduler is single-threaded.
  • Cron runs in server time zone by default.
  • Exceptions can stop future executions.
  • Clock drift affects fixedRate scheduling.
Checklist

Production Checklist

  • Use a dedicated TaskScheduler pool.
  • Guard with locks for single-instance jobs.
  • Set zone in cron expressions.
  • Handle exceptions inside tasks.
  • Monitor execution duration.
