Test Types & Slices

Choose the right test type to balance speed and confidence.

🧒 ELI5

Check one part vs whole toy

Sometimes you test a wheel, sometimes you test the whole toy car.

🔗 ANALOGY

Kitchen tasting

You taste the sauce alone, then the full dish later.

Technical

What it means in real code

Definition: Slice tests load only a layer (like web or data), while integration tests load more context.

Test slices verify one layer at a time and keep tests fast. For example, @WebMvcTest focuses on controllers without loading the full context.

Integration tests should be fewer but high value, covering the main user flows and cross-layer behavior.

As a beginner, aim for many small unit/slice tests and a handful of full-stack tests to catch wiring issues.

Why it matters

  • Fast feedback
  • Isolate failures
  • Confidence at different layers

Common smell

  • Only slow integration tests
  • Only unit tests
  • No coverage for wiring
Slice testsIntegration testsBalance needed
Unit test→Slice test→Integration test→Confidence

Interview answer

“I use unit tests for logic, slice tests for layers, and a few integration tests for wiring.”

TestScope
UnitPure logic
@WebMvcTestWeb layer
@SpringBootTestFull context
Web slice
@WebMvcTest(UserController.class)
class UserControllerTest {
  @Autowired MockMvc mvc;
}

Slice tests are fast and isolate one layer.

As you study this, focus on what Spring/Java is doing on your behalf versus what you must configure explicitly. That mental split makes the API feel predictable instead of magical.

Warning

Don’t skip integration tests

You still need a few tests to confirm wiring and config.

Mini takeaway
  • Use multiple test layers.
  • Slice tests are fast.
  • Integration for wiring.

@Mock vs @MockBean

Mockito has two common modes in Spring tests: plain mocks and Spring-managed mocks.

🧒 ELI5

Pretend tool vs real tool in the toolbox

You can pretend a tool exists, or you can place a fake tool in the toolbox everyone uses.

🔗 ANALOGY

Stand-in actor

Sometimes you just need a stand-in actor in your scene; sometimes the whole set uses them.

Technical

What it means in real code

Definition: @Mock is a plain Mockito mock; @MockBean replaces a Spring bean in the context.

@Mock is for plain unit tests, while @MockBean replaces a Spring-managed bean within a test context.

Use @MockBean when you want Spring wiring but need to isolate an external dependency like a database or API client.

A simple rule: if Spring is creating the object, use @MockBean; otherwise use @Mock.

Why it matters

  • Mock dependencies cleanly
  • Control behavior
  • Isolate the layer under test

Common smell

  • Confusing @Mock with @MockBean
  • Mocking too much
  • Mocking the class under test
@MockBean in Spring@Mock for unit testsMock everything
Create mock→Inject into bean→Run test→Verify

Interview answer

“@Mock is plain Mockito; @MockBean replaces a Spring bean in the application context.”

AnnotationEffect
@MockNo Spring
@MockBeanReplaces bean
@InjectMocksMockito injection
@MockBean example
@WebMvcTest(UserController.class)
class UserControllerTest {
  @MockBean UserService service;
}

@MockBean lets Spring inject the mock into the test context.

DI keeps construction separate from behavior. Constructor injection is safest, qualifiers resolve ambiguity, and scopes decide lifecycle and caching.

Warning

Avoid mocking everything

Over-mocking makes tests brittle and detached from reality.

Mini takeaway
  • @MockBean replaces Spring beans.
  • @Mock is for unit tests.
  • Mock strategically.
Quiz

Correct answer: B @MockBean replaces a bean in the Spring context.

MockMvc Patterns

MockMvc lets you test controllers without starting a full server.

🧒 ELI5

Practice match

You practice plays without the full stadium. It’s faster but still realistic.

🔗 ANALOGY

Flight simulator

You test the controls without flying a real plane.

Technical

What it means in real code

Definition: MockMvc executes controller endpoints in-memory, letting you assert responses.

MockMvc tests controllers without running a server, which makes them faster and easier to maintain.

Use JSON path assertions to validate key fields and avoid brittle full-response comparisons.

When endpoints are secured, include authentication in MockMvc tests so you validate both security and behavior.

Why it matters

  • Fast HTTP tests
  • Validates routing and JSON
  • Avoids full server startup

Common smell

  • Testing only service logic
  • No controller tests
  • Overusing integration tests

Stubbing & Verification

Mockito stubbing defines behavior; verification confirms interactions.

🧒 ELI5

Telling a toy how to respond

You decide how the toy reacts when a button is pressed, then check if it was pressed.

🔗 ANALOGY

Stage rehearsal

You practice expected lines, then check if the actor said them.

Technical

What it means in real code

Definition: Stub with when/thenReturn; verify with verify() to confirm interactions.

Stubbing defines behavior, verification checks interactions. Over-verification can make tests fragile by locking in implementation details.

Use ArgumentCaptor when you need to inspect complex inputs rather than duplicating the production logic in test assertions.

A good mindset is to verify outcomes first, and only verify interactions when the interaction itself is the contract.

Why it matters

  • Control dependencies
  • Assert collaboration
  • Avoid flaky tests

Common smell

  • Verifying every method
  • Overly strict order
  • Testing implementation details
Stub only neededVerify key callsVerify everything
Stub behavior→Run test→Verify critical calls→Assert output

Interview answer

“I stub external dependencies and verify the key interactions that matter to behavior.”

MockitoUse
when().thenReturn()Stub
verify()Interaction
verifyNoMore()Use sparingly
Stub + verify
when(repo.findById(1L)).thenReturn(Optional.of(user));
service.loadUser(1L);
verify(repo).findById(1L);

Focus verification on the interactions that matter to behavior.

As you study this, focus on what Spring/Java is doing on your behalf versus what you must configure explicitly. That mental split makes the API feel predictable instead of magical.

Warning

Don’t over-verify

Tests become brittle if they assert every internal call.

Mini takeaway
  • Stub to control behavior.
  • Verify key interactions only.
  • Avoid over-asserting.

Deep Dive: Test Slices & Mocking Strategy

Pick the smallest Spring test slice that still validates the behavior you care about.

🧒 ELI5

Test one room

You don’t test the entire house when checking a light switch.

🔗 ANALOGY

Car diagnostics

Check the brakes without rebuilding the entire car.

Technical

What it means in real code

Definition: Use @WebMvcTest for controllers, @DataJpaTest for repositories, and @SpringBootTest for full integration.

Why it matters

  • Faster tests
  • Clearer intent
  • Less flaky infra

Common pitfall

  • Using @SpringBootTest everywhere
  • Mocking too much
  • No assertion of interactions
Test sliceMock boundariesIntegration
Pick slice→Mock dependencies→Run assertions→Verify interactions

Interview answer

“I use narrow slices for unit-style tests and @SpringBootTest only for end-to-end behavior.”

Resilience patterns protect downstreams and your own service. Always pair retries with timeouts and circuit breakers to avoid cascading failure.

AnnotationScope
@WebMvcTestController only
@DataJpaTestJPA layer
@SpringBootTestFull context
Controller slice with MockMvc
@WebMvcTest(OrderController.class)
class OrderControllerTest {
  @Autowired MockMvc mvc;
  @MockBean OrderService service;

  @Test void listOrders() throws Exception {
    when(service.list()).thenReturn(List.of());
    mvc.perform(get("/orders")).andExpect(status().isOk());
  }
}
Warning

Too many mocks hide real bugs

If you mock everything, you can miss wiring or serialization issues.

Mini takeaway
  • Use the smallest slice.
  • Mock external boundaries.
  • Add a few full integration tests.
Edge cases

Edge Cases & Tradeoffs

  • Over-mocking hides real integration bugs.
  • Stubbing unused interactions can mask regressions.
  • Final classes require inline mocking or PowerMockito.
  • Argument matchers mixed with raw values cause errors.
  • Async code needs deterministic scheduling.
Checklist

Production Checklist

  • Mock external boundaries, not core logic.
  • Use @ExtendWith(MockitoExtension).
  • Prefer verify() for behavior, not internal calls.
  • Use ArgumentCaptor for complex args.
  • Keep tests fast and focused.
Interview

Interview Hooks

Show you know when to mock and when to integrate.

  • Mock vs spy vs fake tradeoffs.
  • Stubbing with when/thenReturn and BDDMockito.
  • Avoid testing implementation details.

Captors validate arguments without over-specifying behavior.
