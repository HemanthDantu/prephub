Test Types & Slices

Spring Boot's test slices let you boot only the layer you're testing — not the entire application context. Choosing the right test type is what separates a 2-second test suite from a 2-minute one.

🧒 ELI5

The kitchen inspection analogy

Imagine a restaurant health inspector. They don't need to visit every room in the building to check if the kitchen is clean — they go straight to the kitchen. If they inspected the whole restaurant every time (dining room, bathrooms, parking lot), the inspection would take 10x longer and most of what they checked wouldn't tell them anything useful about the kitchen.

@SpringBootTest is inspecting the whole restaurant — it loads everything: security, database, message queues, web layer, all beans. That's sometimes necessary, but incredibly slow for a focused check. Test slices are like giving the inspector a kitchen-only pass — they boot only the relevant layer, giving fast, meaningful feedback exactly where you need it.

🔗 ANALOGY

Circuit breaker panel — test only the circuit you care about

A building has one main breaker and many individual circuit breakers. To test if the kitchen outlet works, you don't need to throw every breaker in the building — just isolate the kitchen circuit. That's a test slice: it throws only the breakers (beans, configuration) relevant to one layer.

The cost of not doing this: developers run @SpringBootTest everywhere, each test takes 8–15 seconds to start up, 300 tests take 40 minutes in CI. The cost of slices: 5–10x faster test suite, tighter feedback loops, and tests that fail for exactly the right reason.

Technical

The three test tiers

Unit tests (no Spring context): Instantiate the class yourself, inject mocks via constructor. Fastest — sub-millisecond. Best for service/domain logic.

Slice tests (partial Spring context): Boot only the specific layer. @WebMvcTest loads only MVC components; @DataJpaTest loads only JPA/Hibernate + H2. ~1–3 seconds.

Integration tests (@SpringBootTest): Boot the full application context. Use sparingly — for testing cross-layer wiring, real HTTP calls via TestRestTemplate, or end-to-end contract verification.

Key slice annotations

  • @WebMvcTest — controllers, filters, MVC config only
  • @DataJpaTest — JPA, Hibernate, H2, repositories only
  • @JsonTest — Jackson serialization/deserialization only
  • @WebFluxTest — reactive controllers only
  • @RestClientTest — RestTemplate/WebClient only

What each slice excludes

  • @WebMvcTest: no JPA, no @Service beans (mock them)
  • @DataJpaTest: no controllers, no @Service
  • @JsonTest: no web layer, no DB, no services
  • All slices: no full autoconfiguration loading
Unit: <1ms Slice: ~1-3s @SpringBootTest: 8-15s @SpringBootTest everywhere
Unit Test→ Slice Test→ @SpringBootTest→ Use sparingly

Test pyramid principle

Write many unit tests, some slice tests, few full integration tests. The pyramid collapses when every test is @SpringBootTest — you end up with an "ice cream cone" anti-pattern: slow, expensive, hard to debug.

Warning

@SpringBootTest context caching — don't break it

Spring caches the ApplicationContext between tests that use the same configuration. If one test uses @MockBean or @SpyBean, it forces a new context to be created — blowing the cache for all subsequent tests that share that context. Each @MockBean introduced in a @SpringBootTest can cost you 8–15 extra seconds per unique context configuration.

Solution: consolidate @MockBean declarations in a base test class that all integration tests extend, or use slice tests (@WebMvcTest) where mocking is expected and the context is intentionally narrow.

Quiz

Which test annotation loads only the Spring MVC components (controllers, filters, MVC config) without loading JPA or service beans?

✅ Correct: B — @WebMvcTest @WebMvcTest is a Spring Boot slice annotation that activates only the web layer: @Controller, @ControllerAdvice, @JsonComponent, Filters, WebMvcConfigurer, and HandlerMethodArgumentResolver. It does NOT load @Service, @Repository, or @Component beans — you must declare any service dependencies with @MockBean. @SpringBootTest (A) loads everything and is 3-10x slower. @DataJpaTest (C) loads JPA/Hibernate components, not controllers. @ContextConfiguration (D) is a lower-level Spring Test annotation for custom context loading, not a slice.

@Mock vs @MockBean

Two annotations that look similar but operate at completely different levels — one is a plain Mockito object, the other reaches into the Spring ApplicationContext and replaces a registered bean.

🧒 ELI5

Pretend tool vs. swapping the real tool in the toolbox

@Mock is like handing a carpenter a pretend hammer. The carpenter knows it's fake and uses it for their own practice — Spring is not involved at all. No toolbox (ApplicationContext) is touched.

@MockBean is like going into the shared workshop toolbox and replacing the real hammer with a fake one. Now every worker in the workshop (every Spring bean that gets injected with that hammer) gets the fake version — Spring is fully aware of the swap and wires everything accordingly.

Use @Mock when you instantiate the class yourself in a plain unit test. Use @MockBean when Spring needs to inject the mock into a bean it manages — like in @WebMvcTest where the controller is Spring-managed and needs its service dependency replaced.

🔗 ANALOGY

Stand-in actor vs. the casting department's official replacement

@Mock = you hire a stand-in for your own private rehearsal. The production company (Spring) doesn't know the stand-in exists — your rehearsal is completely independent.

@MockBean = you notify the casting department and they officially swap the actor in all scheduled scenes. Every other actor on set who has scenes with that role now works with the replacement — the production system is updated. This is exactly what happens with Spring's context: @MockBean registers the mock in the ApplicationContext and resets it after each test method.

Technical

The mechanism behind each

@Mock (Mockito only): Creates a Mockito proxy object. No Spring involvement. Works with @ExtendWith(MockitoExtension.class). The class under test must receive it via constructor injection or @InjectMocks.

@MockBean (Spring Boot Test): Creates a Mockito mock AND registers it in the Spring ApplicationContext, replacing any existing bean of that type. Works with @SpringBootTest, @WebMvcTest, @DataJpaTest, etc. Spring resets the mock's interactions after each test method automatically.

Critical: using @MockBean in a @SpringBootTest invalidates the context cache. Each unique combination of @MockBean declarations causes Spring to create a separate ApplicationContext. This is why consolidating mocks in a base class matters for performance.

@Mock — when to use

  • Plain unit test — no Spring context
  • Testing a service with constructor injection
  • Fast — no context startup overhead
  • Pair with @InjectMocks or pass to constructor

@MockBean — when to use

  • Slice test (@WebMvcTest, @DataJpaTest)
  • Bean is injected by Spring, not by you
  • Need mock registered in ApplicationContext
  • Reset automatically between tests
💬 Model Interview Answer

"@Mock is a plain Mockito mock used in unit tests where I instantiate the class under test myself — no Spring context involved. @MockBean is for Spring Boot tests: it creates the mock AND registers it in the ApplicationContext so Spring can inject it into beans it manages, like a controller in @WebMvcTest. The common mistake is using @Mock in a @WebMvcTest — the controller never receives the mock because Spring wires its own bean, not your Mockito object. The other gotcha: each unique @MockBean combo in @SpringBootTest creates a new context, which kills context caching and slows your test suite."

Warning

@SpyBean — the partial mock trap

@SpyBean is the Spring Boot version of Mockito's @Spy: it wraps a real Spring bean and lets you stub specific methods while calling through to the real implementation for everything else. It looks convenient but carries the same context-cache problem as @MockBean, plus an additional risk: unstubbed methods hit real external services, real databases, real queues. Use it deliberately, not as a shortcut around writing a proper unit test.

@SpyBean — real bean with selective stubbing
@SpringBootTest
class AuditServiceTest {

    @SpyBean  // Real AuditService from context — but we can stub specific methods
    AuditService auditService;

    @Test
    void processOrder_logsAudit_butSkipsSlowNotification() {
        // Only stub the slow/external method — rest of AuditService runs REAL code
        doNothing().when(auditService).sendExternalAuditNotification(any());

        orderService.process(testOrder);

        // Verify the real log() method was called (not stubbed — it ran for real)
        verify(auditService).log(testOrder.getId(), "PROCESSED");
    }
}
Quiz

You add @MockBean EmailService to a @SpringBootTest that previously had no mocks. Ten other @SpringBootTest classes share the same context. What happens?

✅ Correct: C — a new ApplicationContext is created for this test class Spring Boot's test context caching is keyed by the context configuration, which includes the set of @MockBean/@SpyBean declarations. Adding @MockBean EmailService changes the configuration key — so this test class gets its own fresh context (8-15s startup penalty). The other ten classes that don't have @MockBean EmailService still share their original cached context. This is why teams with many @SpringBootTest + @MockBean combinations see test suites that take 20+ minutes — each unique mock combination spawns a new context. Fix: extract all @MockBean declarations to a shared base class so context configurations are identical.

MockMvc Patterns

MockMvc simulates the full Spring MVC request pipeline — routing, argument resolution, validation, serialization, exception handling — without starting an HTTP server. It tests your HTTP contract, not just your Java method.

🧒 ELI5

A flight simulator for your controller

Real pilots train in a flight simulator before flying a real plane. The simulator is not a real aircraft — there's no actual altitude, no real passengers — but it exercises the exact same controls, instruments, and procedures as the real thing. Crucially, it is infinitely safer and faster to test dangerous scenarios: engine failure, crosswind landings, system malfunctions.

MockMvc is your controller's flight simulator. It exercises the full MVC pipeline — URL routing, parameter binding, @Valid validation, Jackson serialization, @ExceptionHandler error responses — without spinning up a real HTTP server. You test that POST /users with a missing name field returns 400 Bad Request with a specific error body, without actually making an HTTP call.

🔗 ANALOGY

Testing a door's lock mechanism vs. the key's metal composition

Calling a controller method directly (controller.createUser(request)) is like testing whether the key is made of the right metal — it ignores the lock entirely. MockMvc tests the whole door-locking mechanism: does the key fit (routing)? Does the lock turn correctly (validation)? Does the door open (response body)? Does it refuse a wrong key (auth)?

If you skip MockMvc and call controller methods directly, you miss: routing mistakes (/user vs /users), validation not triggering, Jackson not serializing as expected, @ExceptionHandler not converting exceptions to proper HTTP responses.

Technical

What MockMvc actually tests

MockMvc drives the full DispatcherServlet pipeline without a network: URL matching → argument resolution → @Valid → method execution → MessageConverter → @ExceptionHandler → response writing.

perform()→ DispatcherServlet→ HandlerMapping→ Your Controller→ andExpect()

What MockMvc catches

  • Wrong URL mappings (@GetMapping typos)
  • @Valid not triggering on @RequestBody
  • Jackson serialization field name mismatches
  • @ExceptionHandler not returning correct status
  • Missing/wrong Content-Type headers
  • Security filter chain rejecting requests

What MockMvc doesn't test

  • Real database queries (use @DataJpaTest)
  • Actual HTTP connection handling
  • Full integration between layers (use @SpringBootTest)
  • Real security tokens (mock the SecurityContext)
💬 Model Interview Answer

"MockMvc exercises the full Spring MVC pipeline — routing, validation, serialization, exception handling — without a real HTTP server. I use it in @WebMvcTest to verify the HTTP contract of my controllers: status codes, response body shape, and error responses. The key thing it catches that calling a controller method directly misses is @Valid validation triggering and Jackson serialization behavior. For security-protected endpoints I use @WithMockUser to set the SecurityContext without a real auth server."

Warning

Testing Jackson serialization separately with @JsonTest

If your DTO has complex Jackson config (@JsonProperty, @JsonIgnore, @JsonFormat for dates, custom serializers), don't test serialization through MockMvc — the response JSON is one big string and errors are hard to pinpoint. Use @JsonTest which loads only Jackson, giving you direct access to JacksonTester for precise assertion on serialization and deserialization.

@JsonTest — precise Jackson serialization testing
@JsonTest
class UserDtoJsonTest {

    @Autowired JacksonTester<UserDto> json;

    @Test
    void serialize_formatsDateAsISO8601() throws Exception {
        var dto = new UserDto(1L, "Hemanth", Instant.parse("2024-01-15T10:30:00Z"));

        assertThat(json.write(dto))
            .hasJsonPathStringValue("$.createdAt")
            .extractingJsonPathStringValue("$.createdAt")
            .isEqualTo("2024-01-15T10:30:00Z");  // verifies @JsonFormat behavior
    }

    @Test
    void deserialize_acceptsISOString() throws Exception {
        String input = """{"id":1,"name":"Hemanth","createdAt":"2024-01-15T10:30:00Z"}""";
        assertThat(json.parse(input)).isEqualTo(new UserDto(1L, "Hemanth", ...));
    }
}
Quiz

You write a MockMvc test that POSTs invalid JSON (missing a required field) to a @Valid-annotated endpoint. The test asserts status 400 but gets 200 instead. What is the most likely cause?

✅ Correct: D — BindingResult suppresses automatic 400 rejection When you declare a BindingResult parameter immediately after the @Valid parameter (e.g., @Valid @RequestBody CreateRequest req, BindingResult errors), Spring does NOT automatically throw MethodArgumentNotValidException on validation failure. Instead it populates the BindingResult and lets your method handle it. If your method then returns 200 without checking errors.hasErrors(), the client receives a 200 for invalid input. The fix: either remove BindingResult and let Spring auto-throw (caught by @RestControllerAdvice), or explicitly check errors.hasErrors() at the start of the method body. Option A is wrong — @Valid works fine in MockMvc. Option C would produce a 400 from Jackson itself, not a 200.

Stubbing & Verification

Stubbing controls what collaborators return; verification confirms how they were used. Knowing when to use each — and when NOT to verify — is what separates brittle tests from resilient ones.

🧒 ELI5

Training a new employee with a script and a checklist

Stubbing is giving a script to the support team member who plays the "bank system" in your training exercise: "When the trainee asks for account balance for ID 42, say £1,000." You control exactly what the collaborator returns so you can test how the trainee (your service) reacts to different scenarios — an empty account, a frozen account, a server timeout.

Verification is checking after the exercise that the trainee followed the correct protocol: "Did you call the fraud department when the transfer exceeded £10,000? Did you record the transaction ID?" Verification confirms specific, meaningful interactions happened — not every single button press.

🔗 ANALOGY

Wiring a circuit and testing continuity

Stubbing is like feeding a known voltage into one end of a circuit. You control the input so you can predict what the output should be — isolating the component you're actually testing. Verification is like using a continuity tester at specific junctions to confirm the signal passed through the right path — not checking every single wire, just the ones that matter for correctness.

Over-verification (checking every wire) makes tests brittle — they break when you refactor the implementation even if the behaviour is correct. Under-stubbing means your tests don't cover edge cases (what happens when the input voltage is zero?). The skill is knowing exactly which interactions matter.

Technical

Stubbing — controlling collaborator behaviour

Use BDDMockito's given().willReturn() style (preferred over Mockito's when().thenReturn()) for readability with JUnit 5. Stubs define the state the test assumes about collaborators.

What to stub

  • Repository findById — happy path + empty Optional
  • External service responses — success + failure + timeout
  • Checked exceptions from I/O
  • Time-dependent behaviour (use Clock mock)

What to verify

  • A notification was sent once (not twice)
  • A delete was called only when conditions met
  • A dangerous op was NOT called on invalid input
  • verifyNoInteractions() on side-effect services
given().willReturn() verify() meaningful interactions verify() every call Over-specifying internals
💬 Model Interview Answer

"I stub collaborator behaviour to control the inputs my class under test sees — repository returning empty Optional, external service throwing timeout, etc. I verify interactions sparingly: only when the call itself is the observable behaviour I care about, like 'send exactly one welcome email' or 'never call delete if the user is an admin.' Over-verifying implementation details (how many times an internal method was called) makes tests brittle — they break on refactors even when the behaviour is unchanged. ArgumentCaptor is my tool when I need to assert on what was passed to a collaborator, not just that it was called."

    @Mock EmailService emailService;
    @InjectMocks UserService userService;

    // Test: happy path — verify rename propagates + no email for rename
    @Test
    void rename_updatesUserName_andDoesNotSendEmail() {
        var user = new User(1L, "OldName", "h@example.com");
        given(userRepo.findById(1L)).willReturn(Optional.of(user));
        given(userRepo.save(any())).willAnswer(inv -> inv.getArgument(0));

        UserDto result = userService.rename(1L, "NewName");

        assertThat(result.getName()).isEqualTo("NewName");
        verifyNoInteractions(emailService);  // rename should NOT send email
    }

    // Test: exception on missing user — verify no side effects
    @Test
    void rename_throwsNotFound_whenUserDoesNotExist() {
        given(userRepo.findById(99L)).willReturn(Optional.empty());

        assertThatThrownBy(() -> userService.rename(99L, "Name"))
            .isInstanceOf(UserNotFoundException.class)
            .hasMessageContaining("99");

        verify(userRepo, never()).save(any());  // repo.save must NEVER be called
        verifyNoInteractions(emailService);
    }

    // Test: ArgumentCaptor — assert WHAT was saved, not just THAT save was called
    @Test
    void createUser_savesWithHashedPassword() {
        given(userRepo.save(any())).willAnswer(inv -> {
            User u = inv.getArgument(0);
            u.setId(1L);
            return u;
        });

        userService.create(new CreateUserRequest("Hemanth", "plaintext123"));

        // Capture what was actually passed to save()
        ArgumentCaptor<User> captor = ArgumentCaptor.forClass(User.class);
        verify(userRepo).save(captor.capture());

        User saved = captor.getValue();
        assertThat(saved.getPassword()).isNotEqualTo("plaintext123"); // must be hashed
        assertThat(saved.getPassword()).startsWith("$2a$"); // bcrypt prefix
        assertThat(saved.getName()).isEqualTo("Hemanth");
    }

    // Test: thenAnswer for dynamic responses
    @Test
    void bulkCreate_savesAllUsers() {
        given(userRepo.save(any())).willAnswer(inv -> {
            User u = inv.getArgument(0);
            u.setId(ThreadLocalRandom.current().nextLong(1, 100));
            return u;
        });

        List<UserDto> results = userService.bulkCreate(List.of(
            new CreateUserRequest("Alice", "a@example.com"),
            new CreateUserRequest("Bob",   "b@example.com")
        ));

        assertThat(results).hasSize(2).allMatch(u -> u.getId() > 0);
        verify(userRepo, times(2)).save(any(User.class));
        verify(emailService, times(2)).sendWelcome(anyString());
    }
}
Warning

UnnecessaryStubbingException — Mockito's lint check

Mockito's strict stubbing mode (default with @ExtendWith(MockitoExtension.class)) throws UnnecessaryStubbingException if you declare a stub that is never used during the test. This is intentional — unused stubs are a sign of copy-paste test code or dead test paths. Don't suppress it with @MockitoSettings(strictness = LENIENT) — fix the root cause: remove the unused stub or move it to the specific test that needs it.

Quiz

You want to assert that userRepository.save() was called with a User object whose password field is BCrypt-hashed, not the plaintext input. Which Mockito tool do you use?

✅ Correct: C — ArgumentCaptor ArgumentCaptor lets you capture the exact object that was passed to a mock method, then make detailed assertions on its fields. Option A (exact object match) relies on equals() which may not be implemented or may match on ID before it's set. Option B (any()) confirms the type but tells you nothing about whether hashing actually happened. Option D (willAnswer()) happens during stubbing setup — the argument inspection runs when the method is called, but you need to use a local variable to capture and assert outside the answer lambda, which is messy. ArgumentCaptor.forClass(User.class) + verify(repo).save(captor.capture()) + captor.getValue() is the clean, intent-revealing pattern.

Deep Dive: Test Slices & Mocking Strategy

How to design a test suite that is fast, trustworthy, and survives refactoring — the decisions that matter at the senior level.

🧒 ELI5

The three-ring circus — each ring has different performers

A circus has three rings. Ring 1 (unit tests) has solo performers who practise fast, independent acts — no one else on stage. Ring 2 (slice tests) has small troupes: the acrobats work with the safety net (the MVC layer and its mocked services), but the jugglers (database, queues) stay backstage. Ring 3 (integration tests) has the full company — every performer, every prop, every animal — but runs once a day, not 50 times during development.

A senior engineer designs the programme so Ring 1 runs every save (seconds), Ring 2 runs every commit (minutes), Ring 3 runs every PR (before merge). Each ring earns its cost through the confidence it provides.

Technical

@DataJpaTest in depth

@DataJpaTest replaces your real DataSource with an embedded H2, wraps each test in a transaction that rolls back after the test (so no test data leaks), and loads only JPA-related beans. Use TestEntityManager to set up test data directly without going through your service layer.

@DataJpaTest — testing custom JPQL queries
@DataJpaTest
class OrderRepositoryTest {

    @Autowired TestEntityManager em;
    @Autowired OrderRepository orderRepo;

    @Test
    void findByCustomerId_returnsOnlyThatCustomersOrders() {
        // Arrange: set up data directly in H2 via TestEntityManager
        Customer c1 = em.persist(new Customer(null, "Alice"));
        Customer c2 = em.persist(new Customer(null, "Bob"));
        em.persist(new Order(null, c1, "PENDING", BigDecimal.TEN));
        em.persist(new Order(null, c1, "SHIPPED", BigDecimal.ONE));
        em.persist(new Order(null, c2, "PENDING", BigDecimal.TEN));
        em.flush();

        // Act: test the JPQL query
        List<Order> alicesOrders = orderRepo.findByCustomerId(c1.getId());

        // Assert
        assertThat(alicesOrders).hasSize(2)
            .allMatch(o -> o.getCustomer().getId().equals(c1.getId()));
    }

    @Test
    void findPendingOrdersTotalByCustomer_returnsAggregatedTotal() {
        // Test a @Query with aggregation / GROUP BY
        Customer c = em.persist(new Customer(null, "Hemanth"));
        em.persist(new Order(null, c, "PENDING", new BigDecimal("100.00")));
        em.persist(new Order(null, c, "PENDING", new BigDecimal("50.00")));
        em.persist(new Order(null, c, "SHIPPED", new BigDecimal("200.00"))); // excluded
        em.flush();

        Optional<BigDecimal> total = orderRepo.sumPendingByCustomer(c.getId());

        assertThat(total).isPresent().get().isEqualByComparingTo("150.00");
    }
}

Choosing your test strategy

💬 Model Interview Answer

"My rule: test each layer at its natural boundary. Business logic in plain unit tests with @Mock — no Spring, fast feedback. HTTP contract and MVC wiring with @WebMvcTest + @MockBean. Repository queries with @DataJpaTest against H2 — this catches JPQL mistakes that mocks would miss. Full @SpringBootTest only for critical integration smoke tests. I keep a base test class for @SpringBootTest to consolidate @MockBean declarations and preserve the Spring context cache."

// ✅ Consolidate all @MockBean in one base class — Spring reuses the same context
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
@ActiveProfiles("test")
public abstract class IntegrationTestBase {

    @MockBean EmailService emailService;    // all integration tests share this mock setup
    @MockBean PaymentGateway paymentGateway;
    @MockBean ExternalAuditClient auditClient;

    @Autowired protected TestRestTemplate restTemplate;

    @BeforeEach
    void resetMocks() {
        // BDDMockito.reset() if you want clean stubs on each test
        reset(emailService, paymentGateway, auditClient);
    }
}

// Concrete test class — inherits context, no new @MockBean = no new context created
class UserIntegrationTest extends IntegrationTestBase {

    @Test
    void createUser_endToEnd_sendsWelcomeEmail() {
        // Act: real HTTP call → real controller → real service → real DB (H2/test DB)
        ResponseEntity<UserDto> resp = restTemplate.postForEntity(
            "/api/users",
            new CreateUserRequest("Hemanth", "h@example.com"),
            UserDto.class
        );

        assertThat(resp.getStatusCode()).isEqualTo(HttpStatus.CREATED);
        assertThat(resp.getBody().getName()).isEqualTo("Hemanth");
        // Verify the mocked email service was called by the real service layer
        verify(emailService).sendWelcome("h@example.com");
    }
}
Warning

H2 compatibility — some PostgreSQL features don't work

@DataJpaTest uses H2 by default, which doesn't support all PostgreSQL-specific features: ON CONFLICT DO UPDATE, JSON operators (->>), partial indexes, and some native query dialects. If your repository uses native queries with PostgreSQL-specific syntax, use @AutoConfigureTestDatabase(replace = NONE) with a real PostgreSQL test container instead of H2.

@DataJpaTest with real PostgreSQL via Testcontainers
@DataJpaTest
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
@Testcontainers
class OrderRepositoryPostgresTest {

    @Container
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16")
        .withDatabaseName("testdb").withUsername("test").withPassword("test");

    @DynamicPropertySource
    static void configureProperties(DynamicPropertyRegistry reg) {
        reg.add("spring.datasource.url", postgres::getJdbcUrl);
        reg.add("spring.datasource.username", postgres::getUsername);
        reg.add("spring.datasource.password", postgres::getPassword);
    }

    @Autowired OrderRepository orderRepo;

    @Test
    void upsertOrder_usesOnConflictDoUpdate() {
        // Test PostgreSQL-specific ON CONFLICT behavior
        orderRepo.upsert(new Order(1L, "PENDING"));
        orderRepo.upsert(new Order(1L, "SHIPPED")); // same ID — should update
        assertThat(orderRepo.findById(1L)).isPresent()
            .get().extracting(Order::getStatus).isEqualTo("SHIPPED");
    }
}
Quiz

Your @DataJpaTest passes locally with H2 but a custom @Query using PostgreSQL JSON operators fails in CI (real PostgreSQL). What is the cleanest fix?

✅ Correct: B — @AutoConfigureTestDatabase(replace=NONE) + Testcontainers This is the only fix that guarantees your test runs against the exact same database engine as production. Testcontainers spins up a real PostgreSQL Docker container for the test run and tears it down after. Option A (rewrite to JPQL) may not be possible — JSON operators and UPSERT patterns have no JPQL equivalent. Option C is wrong in principle — repository tests catch real bugs that service/controller tests mock away. Option D (H2 PostgreSQL compatibility mode) helps with some syntax but does not support all PostgreSQL features like JSON operators, materialized views, or partial indexes, so CI still breaks for those.

🎉 Testing with Mockito Cheat Sheet

Key concepts for interviews and for reviewing PRs in your team.

Cheat Sheet
✅

Test Pyramid

Many unit tests (<1ms each) → some slice tests (~2-3s) → few @SpringBootTest (~10s). Never invert this pyramid.

@WebMvcTest

Loads only MVC layer. Services must be @MockBean. Tests routing, @Valid, serialization, @ExceptionHandler — without a real server.

@DataJpaTest

Loads JPA + H2, no services or controllers. Each test runs in a rolled-back transaction. Use TestEntityManager for setup data.

@Mock vs @MockBean

@Mock = pure Mockito, no Spring. @MockBean = registered in Spring context, auto-reset, breaks context cache in @SpringBootTest.

ArgumentCaptor

Use when you need to assert on WHAT was passed to a mock, not just THAT it was called. Capture → verify → getValue() → assert fields.

Verification discipline

Verify meaningful interactions (send email once, never call delete). Don't verify internal implementation details — they make tests brittle.

Common traps

Mistakes that will fail your PR review

@Mock in @WebMvcTest

Spring doesn't see it. The controller's injected dependency is still the real Spring bean, not your mock. Always use @MockBean inside slice/integration tests.

@MockBean proliferation in @SpringBootTest

Each unique @MockBean combination creates a new ApplicationContext. Consolidate @MockBean in a shared base class to preserve context caching.

Over-verification (testing how, not what)

Verifying logger.debug(), mapper.toDto(), or the exact number of repository.findById() calls ties tests to implementation. Refactors break tests without changing behaviour — that's the definition of a brittle test.

H2 for PostgreSQL-specific queries

H2 in PostgreSQL compat mode doesn't cover JSON operators, ON CONFLICT, or partial indexes. Use Testcontainers for native query tests.

Calling controller methods directly (not MockMvc)

Skips @Valid, Jackson serialization, @ExceptionHandler, and security filters. You get a unit test of a Java method — not a test of an HTTP endpoint.

Unit test service logic @WebMvcTest for HTTP contract @DataJpaTest for queries @SpringBootTest everywhere @Mock inside slice tests Verify behaviour, not internals
