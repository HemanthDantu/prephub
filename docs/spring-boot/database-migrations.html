Why Migrations Exist

Migrations keep schema changes consistent across environments and teams.

← Back to Spring Boot
🧒 ELI5

Recipe history

You keep a record of every recipe change so everyone cooks the same dish.

🔗 ANALOGY

Blueprint revisions

Every building change is recorded so crews stay in sync.

Technical

What it means in real code

Definition: Migrations are ordered, versioned schema changes applied automatically at startup or deploy time.

Migrations give you a versioned history of schema changes so every environment is predictable.

They also provide a reviewable audit trail of how the database evolved over time.

Why it matters

  • Consistent schema
  • Repeatable deployments
  • Auditable history

Common pitfall

  • Manual SQL in prod
  • Missing history
  • Environment drift
VersionedRepeatableAuditable
Detect version→Apply migrations→Update history→App starts

Flyway Basics

Flyway runs SQL migrations in version order and tracks them in a schema history table.

🧒 ELI5

Numbered steps

You follow steps 1, 2, 3 so the project is built in the right order.

🔗 ANALOGY

Staircase

You climb one step at a time; skipping steps makes you fall.

Technical

What it means in real code

Definition: Flyway uses versioned files like V1__init.sql and records checksums in flyway_schema_history.

Flyway uses versioned SQL scripts with naming conventions and records applied versions and checksums.

Because checksums are enforced, you must never edit applied scripts in production.

Why it matters

  • Simple SQL
  • Automatic tracking
  • Fast startup

Common pitfall

  • Renaming applied files
  • Editing old migrations
  • No baseline

Never edit applied migrations

Changing old migrations breaks checksums and risks production drift.

Mini takeaway
  • Use versioned SQL files.
  • Do not change applied migrations.
  • Let Flyway track history.

Liquibase Basics

Liquibase uses changelogs with change sets that can be rolled back and audited.

🧒 ELI5

Change log book

You write every change in a notebook so you can undo or reapply it.

🔗 ANALOGY

Cookbook revisions

Each change to the recipe is recorded with a note so you can revert.

Technical

What it means in real code

Definition: Liquibase uses XML/YAML/JSON changelogs with changesets, checksums, and optional rollback rules.

Liquibase uses changelogs with IDs and authors, allowing rollback definitions and structured changes.

It is more verbose than Flyway but useful for teams that need explicit rollback support.

Why it matters

  • Structured changes
  • Rollback support
  • Supports multiple DBs

Common pitfall

  • Editing applied changesets
  • No rollback plan
  • One massive changeset
ChangeSetRollbackChangelog

Safe Migration Practices

Production migrations need review, backups, and careful rollout strategies.

🧒 ELI5

Safety checks

🔗 ANALOGY

Technical

What it means in real code

Definition: Safe migrations include validation, backups, and release sequencing to reduce risk.

Safe migrations require discipline: backups, staging tests, and small, reversible changes.

Plan for locks and downtime when altering large tables or hot indexes.

Why it matters

  • Prevent downtime
  • Enable rollback

Common pitfall

Deep Dive: Zero‑Downtime Migrations

Production migrations must avoid locking and support rolling deployments.

🧒 ELI5

Fix the road while cars drive

You repair one lane at a time so traffic keeps moving.

🔗 ANALOGY

Hotel renovation

You renovate floor by floor so guests can still stay.

Technical

What it means in real code

Definition: Use expand/contract patterns: add new columns, deploy code, backfill, then remove old columns.

Why it matters

  • Avoids downtime
  • Supports rollbacks
  • Safe deployments

Common pitfall

  • Dropping columns immediately
  • Large table locks
  • Big-bang migrations
Expand/contractBackfillLock risk
Add column→Deploy code→Backfill data→Remove old

“I use expand/contract so schema changes don’t break running services.”

Warning

Large locks cause outages

Avoid schema changes that lock tables for long periods.

Mini takeaway
  • Use expand/contract.
  • Backfill safely.
  • Monitor locks.
Edge cases

Edge Cases & Tradeoffs

  • Changing old migrations breaks checksums.
  • Out-of-order migrations can surprise prod.
  • Rollback support is limited in Flyway.
  • Prod drift happens without validation.
  • Large migrations can lock tables.
Checklist

Production Checklist

  • Never edit applied migrations.
  • Use repeatable migrations for views/functions.
  • Validate on startup in CI/CD.
  • Use baseline for legacy databases.
  • Split large migrations to reduce locks.
Interview

Interview Hooks

  • Flyway uses checksums to detect drift.
  • Liquibase supports rollback more naturally.
  • Order migrations by version and test in staging.
