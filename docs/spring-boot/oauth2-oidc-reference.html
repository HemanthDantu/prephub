// 01 — foundations

What is OAuth2?

OAuth2 is a way to let third-party apps access your data without sharing your actual password. Think "Sign in with Google" — the website never sees your Google password. Google verifies who you are and gives the website permission to access specific things about you.

The Four Actors

👤
Resource Owner
You — the person whose data is being protected. You grant permission.
🔐
Authorization Server
Google (or Keycloak). Verifies your identity and issues tokens.
💻
Client
The app trying to access your data. Never sees your password.
🗄️
Resource Server
Holds your actual data (email, profile). Validates tokens on each request.
Note: Authorization Server and Resource Server can be the same (Google) or different entities. In enterprise setups like Angi, Keycloak is the Authorization Server while Angi's own APIs are the Resource Servers.
// 02 — the flow

Authorization Code Flow

This is the most common and secure OAuth2 flow. Here's exactly what happens when you click "Sign in with Google":

1
User clicks "Sign in with Google"
The client app redirects you to Google with its client_id in the URL, telling Google which app is asking for access.
2
Google verifies your identity
If you're not logged in, Google asks you to. Then shows a consent screen: "This app wants access to your email and profile."
3
You click Yes — Google sends Authorization Code
Google redirects you back to the client app with a temporary authorization code in the URL. NOT the token — just a short-lived code.
4
Client exchanges code for token (backend)
The client's backend secretly sends the auth code + client secret to Google's backend. This happens server-to-server, invisible to you. URLs can be logged/intercepted — so the valuable token stays out of the URL.
5
Google returns Access Token + Refresh Token
Client stores these securely. The access token is used for all subsequent API calls. The refresh token is used to get new access tokens when they expire.
// 03 — tokens

Types of Tokens

OAuth2 gives you multiple tokens, each with a specific purpose and lifetime:

Access Token
Purpose: Authorize API calls
Lifetime: Short (minutes–1 hour)
Stored: Client backend / memory
Used by: Client on every API request
Refresh Token
Purpose: Get new access tokens
Lifetime: Long (days–weeks)
Stored: Client backend only (secret)
Used by: Client when access token expires
ID Token (OIDC)
Purpose: Prove who the user is
Lifetime: Short
Format: Always a JWT
Used by: Client to know user identity
Authorization Code
Purpose: Temporary exchange voucher
Lifetime: Very short (seconds)
Used: Once only, then discarded
Travels via: URL redirect
Token Refresh Flow: When access token expires → client uses refresh token → asks auth server for new access token → user doesn't notice anything. You only re-login when the refresh token itself expires.
Revoking Access: Go to Google settings → disconnect the app → Google deletes the refresh token → client can't get new access tokens. Old access tokens may still work until they expire naturally.
// 04 — jwt

JSON Web Tokens (JWT)

JWT is the format of the token — not a protocol. It's a structured, self-contained token with three base64-encoded parts separated by dots:

// JWT structure: header.payload.signature
eyJhbGciOiJSUzI1NiJ9 ← Header (algorithm used)
.eyJ1c2VySWQiOiIxMjM0In0 ← Payload (your data/claims)
.SflKxwRJSMeKKF2QT4fw ← Signature (proves it's real)

Why JWTs are powerful

The resource server can verify a JWT without calling the auth server. The signature proves Google issued it. The payload contains user info. Any service with Google's public key can verify it independently — this is why it's scalable.

Self-contained
User ID, email, roles — all inside the token. No database lookup needed.
Signed
Cryptographically signed. Tamper-proof. Anyone can verify but only the issuer can create.
Stateless
Server doesn't need to store tokens. Just verify the signature on each request.
// 05 — oidc

OpenID Connect (OIDC)

OIDC is built on top of OAuth2. OAuth2 handles authorization (can this app access your data?). OIDC adds authentication (who is this person?). Most "Sign in with" flows use OIDC.

Key Difference

OAuth2
Authorization. "This app has permission to access Hemanth's email." Doesn't confirm identity.
OIDC
Authentication + Authorization. "I've verified this is actually Hemanth, AND the app has permission."
OIDC gives you an extra ID Token (always a JWT) on top of the access token. This ID token is specifically for proving identity — who the user is.
// 06 — relationships

How OAuth2, OIDC, Keycloak & JWT Relate

Think of it as layers — each builds on the one below:

JWT — the token format (the currency)
↑ uses
Keycloak — software that implements the specs (the building)
↑ implements
OIDC — authentication layer on top of OAuth2 (blueprint + identity)
↑ extends
OAuth2 — the authorization framework (the foundation blueprint)

When you log in: Keycloak runs the OIDC flow → which uses OAuth2 underneath → and issues you a JWT token that proves who you are and what you can access.

What is Keycloak?

Keycloak is an open-source identity provider maintained by Red Hat and the community. Companies like Angi self-host it instead of paying for Auth0 or Okta. They get full control over their auth infrastructure but take on the responsibility of maintaining it.

// 07 — spring boot config

Spring Boot OAuth2 Setup

Dependencies (pom.xml)

<!-- Turn on Spring Security -->
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-security</artifactId>
</dependency>

<!-- OAuth2 client support -->
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-oauth2-client</artifactId>
</dependency>

<!-- For resource servers validating JWTs -->
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-oauth2-resource-server</artifactId>
</dependency>

application.yml config

spring:
  security:
    oauth2:
      client:
        registration:
          google:
            client-id: ${GOOGLE_CLIENT_ID} # from Secrets Manager
            client-secret: ${GOOGLE_CLIENT_SECRET} # never hardcode!
            scope: openid, email, profile

Security Config Class

@Configuration
@EnableWebSecurity
public class SecurityConfig {

  @Bean
  public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
      .authorizeHttpRequests(auth -> auth
        .requestMatchers("/login", "/public/**").permitAll()
        .anyRequest().authenticated() // all others need auth
      )
      .oauth2Login(); // Spring handles redirect + callback
    return http.build();
  }
}
@EnableWebSecurity turns on Spring Security globally. The SecurityFilterChain bean defines which endpoints are public and which need authentication. Spring handles all redirect and token exchange logic automatically.
Never hardcode client ID or secret. Always pull from environment variables or AWS Secrets Manager / HashiCorp Vault. Spring reads these automatically at startup.

// 09 — real world

Angi's OAuth Implementation

Angi

From inspecting the browser dev tools on id.angi.com, here's what Angi's auth stack looks like:

Identity Provider
Keycloak, self-hosted at id.angi.com. Evidence: /realms/ads-user/ in the URL — "realms" is a Keycloak-specific concept.
Protocol
OIDC on top of OAuth2. client_id=web-consumer visible in the auth URL.
MFA
OTP (6-digit email code) layered on top of the OIDC flow for extra security.
Keycloak Cookies
KC_RESTART, AUTH_SESSION_ID, AUTH_SESSION_ID_LEGACY — KC_ prefix = Keycloak session management cookies.

How Angi APIs are protected

Keycloak issues JWTs on login. Angi's Spring Boot backend services are configured as OAuth2 Resource Servers using spring-boot-starter-oauth2-resource-server. They validate the JWT on every incoming request — checking the signature and expiry — without calling Keycloak on each call. Keycloak's job is issuing tokens. APIs' job is validating them.

Concert wristband analogy: Security checks your wristband at the gate (login). After that, staff inside just glance at it (JWT validation) — they don't call the ticket company every time.
// 10 — interview prep

Interview Q&A

Click each question to reveal a natural, experience-grounded answer:

How was authentication and authorization handled at Angi?
"Angi used Keycloak as their identity provider, running on id.angi.com. It implemented OIDC on top of OAuth2, so when users logged in, Keycloak handled the authentication and issued JWT access tokens and refresh tokens. Our backend Spring Boot services were configured as OAuth2 resource servers using spring-boot-starter-oauth2-resource-server, so they'd validate incoming JWTs on every request — verifying the signature and checking expiry — without hitting Keycloak on each call. Keycloak's job was issuing tokens, and our services' job was validating them independently."
Have you worked with JWT tokens directly?
"Yes — in my analytics instrumentation work, I needed to extract user identity from JWT claims to associate Segment events with specific users. So I was reading claims like user ID and email directly from the token to enrich our analytics events with user context."
How did you protect API endpoints in Spring Boot?
"We used Spring Security's security filter chain to define which endpoints required authentication and which are public. Protected endpoints would automatically validate the JWT token from the Authorization header. If the token was missing, expired, or invalid, Spring Security rejected the request before it even hit the controller."
What is the difference between OAuth2 and OIDC?
"OAuth2 is the authorization framework — it defines how to securely grant access without sharing passwords. OIDC is built on top of OAuth2 and adds identity verification — so you know not just that someone has permission, but who they actually are. Angi used OIDC via Keycloak, which gave us both authentication and authorization in one flow."
What is PKCE and when would you use it?
"PKCE — Proof Key for Code Exchange — is an extra security layer for OAuth2 flows where you can't safely store a client secret, like mobile apps or React SPAs. Before redirecting to the auth server, the app generates a random code verifier and sends a hashed version (the code challenge) to the auth server. When exchanging the authorization code for a token, the app sends the original verifier. The auth server hashes it and checks it matches — proving it's really the same app, even without a secret. For Spring Boot backends where we have a client secret, PKCE is less critical, but for frontend apps it's essential."
What is Keycloak and why do enterprises use it?
"Keycloak is an open-source identity and access management solution maintained by Red Hat. Enterprises use it as a self-hosted alternative to managed services like Auth0 or Okta. At Angi, Keycloak ran at id.angi.com and handled all authentication — including OIDC flows, session management, and MFA via OTP codes. The benefit is full control over your auth infrastructure and user data without depending on a third party. The tradeoff is you own the responsibility of running and maintaining it."
What are the main advantages of OAuth2 over building your own auth?
"A few big ones. First, you don't store passwords — huge security win since you can't leak what you don't have. Second, you skip building login forms, password reset flows, hashing logic — all that plumbing is handled for you. Third, you get single sign-on — if a user is already logged into Google they don't need to re-enter credentials. Fourth, users trust it more — 'Sign in with Google' feels safer than creating yet another account. And fifth, you delegate security responsibility to a battle-tested system that's way more resourced than most apps can afford."
