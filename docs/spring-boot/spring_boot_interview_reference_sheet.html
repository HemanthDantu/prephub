Spring MVC and REST APIs

4 questions

DispatcherServlet is Spring MVC’s front controller. It receives incoming HTTP requests, delegates to handler mappings and adapters, invokes controllers, and coordinates the response rendering or JSON serialization.

@Controller is used for MVC controllers that often return view names. @RestController is effectively @Controller plus @ResponseBody, so return values are written directly to the HTTP response body, typically as JSON.

java
@RestController
@RequestMapping("/users")
class UserController {
  @GetMapping("/{id}")
  User getUser(@PathVariable String id) {
    return service.getUser(id);
  }
}

Use Bean Validation annotations on request DTOs and annotate the controller parameter with @Valid. Invalid payloads raise binding or validation exceptions, which are usually handled globally with @RestControllerAdvice.

java
public record CreateUserRequest(
    @NotBlank String name,
    @Email String email
) { }

@PostMapping
public ResponseEntity<Void> create(@Valid @RequestBody CreateUserRequest request) {
  service.create(request);
  return ResponseEntity.ok().build();
}

RestTemplate is the traditional synchronous HTTP client. WebClient supports reactive, asynchronous, and streaming use cases and is the better fit when you need high concurrency or non-blocking I/O; for many existing synchronous codebases, RestTemplate is still common.

BeanFactory is the minimal IoC container focused on bean creation and lookup. ApplicationContext builds on it and adds enterprise features such as event publishing, AOP integration, message resolution, and resource loading. In Spring Boot applications, ApplicationContext is what you typically use.

Spring commonly creates proxies for cross-cutting concerns such as transactions, security, caching, and async execution. IoC is the broader principle where the container controls object creation and wiring. Dependency injection is the common Spring mechanism for implementing IoC by providing collaborators through constructors, setters, or fields.

@Autowired asks Spring to resolve a dependency from the container. In interviews and real code, constructor injection is preferred because dependencies are explicit, mandatory collaborators are enforced, and the class is easier to test.

All three register beans with the container, but they communicate intent. @Component is generic, @Service marks business-layer logic, and @Repository marks persistence code and also enables exception translation for data-access exceptions.

    return new ObjectMapper();
  }
}

Spring AOP applies cross-cutting behavior around method execution using proxies. It is commonly used for transactions, method security, caching, logging, metrics, and retry behavior.

By default, Spring rolls back the transaction on unchecked exceptions. For checked exceptions, configure rollback rules explicitly if needed. For transient failures such as deadlocks or timeouts, add retries. Avoid long transactions and avoid calling external systems while holding database locks.

Optimistic locking prevents lost updates by using a version column. When two transactions update the same row, Hibernate checks the version during update and fails one of them instead of silently overwriting data. It works well when conflicts are rare and you prefer retries over database locks.

  class="tok-kw">private Long id;

Use Flyway or Liquibase with versioned migration scripts stored in source control. Migrations should run consistently across environments and be designed for safe deployment, especially when dealing with large tables, backfills, or zero-downtime release steps.

Spring Data JPA is a repository abstraction built on top of JPA that reduces boilerplate for common persistence operations. Hibernate is an ORM implementation and JPA provider. In many Spring apps, Spring Data JPA gives the repository layer and Hibernate does the actual ORM work underneath.

CrudRepository gives basic CRUD methods. JpaRepository extends it with JPA-oriented capabilities such as paging, sorting, flush operations, and additional convenience methods, so it is the more common choice in Spring Data JPA applications.

LAZY loads an association only when it is accessed, while EAGER loads it immediately with the entity. A common rule of thumb is to be intentional and prefer explicit fetching in queries; in JPA, to-many associations default to LAZY, while many-to-one and one-to-one are EAGER by default.

java
@Entity
class Order {
  @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
  private List<OrderItem> items;
}

It happens when lazy-loaded data is accessed after the persistence context is closed. The usual fixes are to fetch what you need inside the service transaction, use join fetch or entity graphs, or map to DTOs before leaving the transactional boundary.

Cascade settings propagate operations such as persist, merge, remove, refresh, or detach from a parent to its children. orphanRemoval is different: it deletes a child entity when it is removed from the parent’s relationship, which is useful for true parent-owned child collections.

java
@OneToMany(mappedBy = "order", cascade = CascadeType.ALL, orphanRemoval = true)
private List<OrderItem> items = new ArrayList<>();

Propagation controls how a transactional method behaves when it is called inside another transaction. REQUIRED joins the current transaction or creates one if none exists, and it is the default. REQUIRES_NEW always starts a new transaction and suspends the outer one.

java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void audit(AuditEvent event) {
  auditRepository.save(event);
}

Isolation controls what one transaction can see from concurrent transactions, such as dirty reads or non-repeatable reads. Typical levels are READ_UNCOMMITTED, READ_COMMITTED, REPEATABLE_READ, and SERIALIZABLE; higher isolation improves consistency but can reduce concurrency.

java
@Transactional(isolation = Isolation.READ_COMMITTED)
public void transferMoney(...) {
  // business logic
}

The main states are transient, managed or persistent, detached, and removed. Knowing these states helps explain behaviors like dirty checking, merge, and why updates may or may not be written automatically.

The N+1 problem happens when one query loads parent rows and then additional queries are fired for each child association. Typical fixes are join fetch, entity graphs, batch fetching, DTO projections, and designing queries around the data actually needed by the API.

Spring Security authenticates requests through its filter chain. Depending on the setup, it can validate username/password credentials, sessions, OAuth2 logins, or bearer tokens such as JWTs. Once authenticated, it stores the principal and authorities in the security context for downstream authorization checks.

Authentication verifies identity using mechanisms such as JWTs, sessions, OAuth2, or API keys. Authorization determines what the caller is allowed to do using roles, scopes, or permissions. In Spring, endpoint rules are typically defined in SecurityFilterChain and method rules with annotations such as @PreAuthorize.

CSRF is an attack where a browser is tricked into sending an unwanted authenticated request. It mainly matters for cookie-based session authentication in browser apps. For stateless REST APIs that use bearer tokens in headers, CSRF protection is often disabled because the browser is not automatically attaching those tokens the same way it does cookies.

Spring Security is the framework Spring uses for authentication, authorization, session management, CSRF protection, and request filtering. Integration usually means adding the security starter, configuring SecurityFilterChain, selecting an auth strategy such as form login or JWT, and restricting endpoints by role or scope.

SecurityFilterChain defines how Spring Security processes incoming requests before they reach controllers. It runs in the servlet filter layer and handles concerns such as authentication, authorization, CSRF, CORS integration, and exception translation.

java
@Bean
SecurityFilterChain security(HttpSecurity http) throws Exception {
  return http
      .authorizeHttpRequests(auth -> auth.anyRequest().authenticated())
      .build();
}

Session-based auth stores user session state on the server and usually sends a session cookie to the browser. JWT-based auth is commonly used for stateless APIs, where the client sends a signed token on each request and the server validates it without looking up session state.

Passwords should never be stored in plain text. Spring Security’s PasswordEncoder hashes passwords using a one-way adaptive algorithm such as bcrypt so verification is possible, but the original password is not recoverable and brute-force attacks become more expensive.

java
@Bean
PasswordEncoder passwordEncoder() {
  return new BCryptPasswordEncoder();
}

CORS controls which browser-based cross-origin requests are allowed to access your backend. In Spring, you can configure it globally, per controller, or through Spring Security; preflight requests must be handled correctly before security rules block them.

java

  CorsConfiguration config = new CorsConfiguration();
  config.setAllowedOrigins(List.of("https://app.example.com"));
  config.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE"));

  UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
  source.registerCorsConfiguration("/**", config);
  return source;
}

Use a layered strategy. Unit tests cover business logic with JUnit and mocks. Slice tests such as @WebMvcTest validate controller behavior. @DataJpaTest covers repository behavior. Full integration tests use @SpringBootTest, and Testcontainers is a strong choice when you want realistic databases or infrastructure dependencies.

  class="tok-ann">@Autowired
  class="tok-kw">private MockMvc mockMvc;
}

@SpringBootTest loads the full application context and is used for broader integration testing. @WebMvcTest loads only the MVC slice, so it is faster and focused on controller behavior, request mapping, validation, and serialization.

MockMvc tests the MVC layer without starting a real server, which makes it fast and good for controller testing. TestRestTemplate or WebTestClient are closer to real HTTP integration testing and are better when you want to exercise the application through actual HTTP calls.
